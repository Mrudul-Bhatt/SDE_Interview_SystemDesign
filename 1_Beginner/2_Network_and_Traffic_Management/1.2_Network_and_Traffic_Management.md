# Load Balancers — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine a busy hospital with 10 doctors. Patients walk in through one door. If every patient rushes to Doctor 1, she's overwhelmed and makes mistakes. Doctors 2–10 sit idle.

A **receptionist** at the front desk looks at all patients coming in and says:
- "You go to Doctor 3 — she's free."
- "You go to Doctor 7 — he has the shortest queue."
- "You go to Doctor 1 — this is your regular doctor."

That receptionist **is** a load balancer. One entry point. Smart distribution behind it. Patients don't know or care which doctor they get — they just want to be seen quickly.

---

## What a Load Balancer Actually Does

Without a load balancer:

```
User → Server 1's IP directly
         (what if Server 1 dies?)
         (what if 100K users all hit it at once?)
```

With a load balancer:

```
User → Load Balancer IP (the only IP users know)
            │
    ┌───────┼───────┐
    ▼       ▼       ▼
 Server1  Server2  Server3
```

The load balancer:
1. **Receives** every incoming request
2. **Decides** which backend server gets it
3. **Forwards** the request there
4. **Returns** the response to the user

Users only ever talk to one IP — the load balancer. The entire server fleet is invisible to them.

---

## Why Load Balancers Exist — The 3 Core Problems They Solve

### Problem 1: One server can't handle infinite traffic

```
Without LB:
10,000 req/s → Server 1 → melts at 2,000 req/s → 💥

With LB (5 servers):
10,000 req/s → LB → 2,000 req/s each → each server comfortable ✅
```

### Problem 2: Servers die — you need zero downtime

```
Without LB:
Server 1 crashes → your entire product is down
Users see 502 errors → they leave → you lose money

With LB:
Server 1 crashes → LB detects it via health check
                 → removes Server 1 from pool
                 → routes traffic to Servers 2, 3, 4
                 → users notice nothing ✅
```

### Problem 3: Deploying new code without downtime

```
Old way (no LB): take entire system down → deploy → bring back up
                 → users see downtime

With LB (rolling deploy):
Step 1: Remove Server 1 from LB pool
Step 2: Deploy new code to Server 1
Step 3: Add Server 1 back to pool
Step 4: Repeat for Server 2, 3, 4...
→ Users always have servers available ✅
```

---

## Load Balancing Algorithms — The Core of the Topic

This is what interviewers dig into. There are 6 algorithms you need to know.

---

### Algorithm 1: Round Robin

The simplest. Requests go to servers in order, cycling endlessly.

```
Request 1  → Server A
Request 2  → Server B
Request 3  → Server C
Request 4  → Server A  (back to start)
Request 5  → Server B
...
```

```csharp
public class RoundRobinBalancer
{
    private readonly List<string> _servers;
    private int _current = 0;

    public string GetNext()
    {
        // Interlocked for thread safety — multiple threads call this
        int idx = Interlocked.Increment(ref _current);
        return _servers[idx % _servers.Count];
    }
}
```

**Good:** Dead simple. Even distribution if all requests are equal weight.

**Bad:** Ignores server load. A slow 10-second request and a fast 10ms request are treated identically. One server might pile up while others are free.

---

### Algorithm 2: Weighted Round Robin

Some servers are more powerful. Give them more requests proportionally.

```
Server A: weight 3 (powerful)
Server B: weight 1 (weaker)

Order: A, A, A, B, A, A, A, B...
```

```csharp
public class WeightedRoundRobin
{
    // Expand servers by weight: [A, A, A, B]
    private readonly List<string> _pool;
    private int _current = 0;

    public WeightedRoundRobin(Dictionary<string, int> serverWeights)
    {
        _pool = new List<string>();
        foreach (var (server, weight) in serverWeights)
            for (int i = 0; i < weight; i++)
                _pool.Add(server);
    }

    public string GetNext()
    {
        int idx = Interlocked.Increment(ref _current);
        return _pool[idx % _pool.Count];
    }
}
```

**Good:** Accounts for unequal server capacity.

**Bad:** Still doesn't look at real-time load. A powerful server might be temporarily overwhelmed.

---

### Algorithm 3: Least Connections

Route to whichever server currently has the fewest active connections.

```
Server A: 45 active connections
Server B: 12 active connections  ← new request goes here
Server C: 38 active connections
```

```csharp
public class LeastConnectionsBalancer
{
    private readonly Dictionary<string, int> _activeConnections = new();
    private readonly object _lock = new();

    public string GetServer()
    {
        lock (_lock)
        {
            return _activeConnections.MinBy(kvp => kvp.Value).Key;
        }
    }

    public void OnRequestStart(string server)
    {
        lock (_lock) { _activeConnections[server]++; }
    }

    public void OnRequestEnd(string server)
    {
        lock (_lock) { _activeConnections[server]--; }
    }
}
```

**Good:** Adapts to real load. Long-running requests naturally push traffic elsewhere.

**Bad:** Requires tracking state (connection counts). Slightly more overhead.

**Best for:** Mixed workloads where some requests take milliseconds and others take seconds. Like AXS — a seat availability check (fast) vs. a payment processing request (slow) shouldn't be treated the same.

---

### Algorithm 4: IP Hash (Sticky Sessions)

Hash the user's IP address to always send them to the same server.

```
User IP 192.168.1.1 → hash → always Server B
User IP 10.0.0.5    → hash → always Server A
```

```csharp
public class IPHashBalancer
{
    private readonly List<string> _servers;

    public string GetServer(string clientIP)
    {
        int hash = Math.Abs(clientIP.GetHashCode());
        return _servers[hash % _servers.Count];
    }
}
```

**Good:** Same user always hits same server — useful if server has local state for that user (in-memory session, local cache).

**Bad:** Uneven distribution if many users share an IP (corporate NAT). If that server dies, all its "sticky" users lose their session.

---

### Algorithm 5: Least Response Time

Route to the server with the lowest average response time AND fewest connections.

```
Server A: avg 120ms, 20 connections
Server B: avg 45ms,  18 connections  ← winner
Server C: avg 200ms, 5 connections
```

**Good:** Most intelligent. Adapts to both load and performance.

**Bad:** Requires measuring response times continuously. More complex to implement.

---

### Algorithm 6: Random

Pick a server at random.

```csharp
public string GetServer()
{
    return _servers[Random.Shared.Next(_servers.Count)];
}
```

**Good:** Zero state. Extremely simple. Surprisingly effective at scale (law of large numbers evens it out).

**Bad:** No guarantee of even distribution for small request counts.

---

## Algorithm Comparison

| Algorithm           | State needed      | Handles unequal load | Handles unequal servers | Best for                         |
| ------------------- | ----------------- | -------------------- | ----------------------- | -------------------------------- |
| Round Robin         | Counter only      | ❌                    | ❌                       | Equal servers, equal requests    |
| Weighted RR         | Expanded pool     | ❌                    | ✅                       | Known capacity differences       |
| Least Connections   | Connection counts | ✅                    | Partially               | Mixed request durations          |
| IP Hash             | Hash function     | ❌                    | ❌                       | Sticky sessions needed           |
| Least Response Time | Metrics + counts  | ✅                    | ✅                       | Production, performance-critical |
| Random              | Nothing           | ❌                    | ❌                       | High scale, simplicity needed    |

## Health Checks — How LB Knows a Server is Dead

The load balancer constantly pings backend servers:

```
Every 5 seconds:
LB → "GET /health" → Server 1 → 200 OK ✅ (keep in pool)
LB → "GET /health" → Server 2 → timeout  ❌ (remove from pool)
LB → "GET /health" → Server 3 → 200 OK ✅ (keep in pool)
```

When Server 2 comes back:

```
LB → "GET /health" → Server 2 → 200 OK
→ wait 3 consecutive successes → add back to pool
```

This is **automatic failover**. No human needed. The system heals itself.

## Layer 4 vs Layer 7 Load Balancing

This trips people up. There are two fundamentally different types.

### Layer 4 (Transport Layer) — Fast but dumb

Operates at TCP level. Doesn't look inside the request. Just forwards packets.

```
Sees:  Source IP, Destination IP, Port
Doesn't see: URL, headers, cookies, request body
```

```
User → TCP connection to LB:443
LB → opens TCP connection to Server 2:443
LB → blindly forwards all packets
```

**Fast** — minimal processing. **Dumb** — can't make intelligent routing decisions based on content.

### Layer 7 (Application Layer) — Smart but heavier

Terminates the HTTP connection. Reads the actual request. Routes based on content.

```
Sees: URL path, headers, cookies, method, body
Can do:
  /api/bookings  → Booking Service cluster
  /api/payments  → Payment Service cluster
  /static/img    → CDN / Static file servers
  Cookie: userId=premium → Premium server pool
```

```
User → HTTP request "GET /api/events/123"
LB → reads URL → "/api/events" → routes to Event Service
LB → new HTTP request to Event Service
Event Service → response → LB → response → User
```

**Smart** — content-aware routing, SSL termination, request rewriting, A/B testing.
**Heavier** — more CPU, more latency (milliseconds).

### Which to use?

```
Layer 4: Raw performance critical, simple TCP routing, gaming servers
Layer 7: Web apps, microservices, API gateways, anything HTTP
```

In practice: **almost everything modern uses Layer 7**. AWS ALB is Layer 7. Nginx in proxy mode is Layer 7.


## Coding Questions From Load Balancing

Unlike sharding, load balancing has **direct coding questions** asked at top companies.

---

### Question 1: Implement Round Robin Load Balancer *(most commonly asked)*

> "Design a thread-safe round robin load balancer supporting add/remove servers."

```csharp
public class LoadBalancer
{
    private List<string> _servers = new();
    private int _counter = 0;
    private readonly ReaderWriterLockSlim _lock = new();

    public void AddServer(string server)
    {
        _lock.EnterWriteLock();
        try { _servers.Add(server); }
        finally { _lock.ExitWriteLock(); }
    }

    public void RemoveServer(string server)
    {
        _lock.EnterWriteLock();
        try { _servers.Remove(server); }
        finally { _lock.ExitWriteLock(); }
    }

    public string GetServer()
    {
        _lock.EnterReadLock();
        try
        {
            if (_servers.Count == 0)
                throw new InvalidOperationException("No servers available");

            int idx = Math.Abs(Interlocked.Increment(ref _counter));
            return _servers[idx % _servers.Count];
        }
        finally { _lock.ExitReadLock(); }
    }
}
```

**Key things to show:**
- Thread safety (`ReaderWriterLockSlim` — allows concurrent reads, exclusive writes)
- Handle empty server list
- Handle overflow of counter (`Math.Abs`)

---

### Question 2: Implement Least Connections *(medium difficulty)*

> "Implement a load balancer that always routes to the server with fewest active connections."

```csharp
public class LeastConnectionsBalancer
{
    // Min-heap by connection count
    private readonly PriorityQueue<string, int> _heap = new();
    private readonly Dictionary<string, int> _connections = new();
    private readonly object _lock = new();

    public void AddServer(string server)
    {
        lock (_lock)
        {
            _connections[server] = 0;
            _heap.Enqueue(server, 0);
        }
    }

    public string Acquire()
    {
        lock (_lock)
        {
            // Get min-connection server
            var server = _heap.Dequeue();
            _connections[server]++;
            // Re-enqueue with updated priority
            _heap.Enqueue(server, _connections[server]);
            return server;
        }
    }

    public void Release(string server)
    {
        lock (_lock)
        {
            _connections[server]--;
            // Rebuild heap entry — PriorityQueue doesn't support update
            // In production: use an indexed priority queue
        }
    }
}
```

**Interviewer follow-up:** "PriorityQueue doesn't support priority updates — how do you handle stale entries?"

**Answer:** Lazy deletion — when you dequeue, check if the stored count matches `_connections[server]`. If not, it's stale — discard and dequeue again.

### Question 3: Design a Rate-Limited Load Balancer *(hard)*

> "Extend your load balancer to reject requests if any server exceeds 1000 req/min."

```csharp
public class RateLimitedBalancer
{
    private readonly Dictionary<string, Queue<DateTime>> _requestTimes = new();
    private const int MaxRequestsPerMinute = 1000;

    public string GetServer(List<string> servers)
    {
        var now = DateTime.UtcNow;
        var cutoff = now.AddMinutes(-1);

        foreach (var server in servers)
        {
            if (!_requestTimes.ContainsKey(server))
                _requestTimes[server] = new Queue<DateTime>();

            var times = _requestTimes[server];

            // Evict requests older than 1 minute (sliding window)
            while (times.Count > 0 && times.Peek() < cutoff)
                times.Dequeue();

            if (times.Count < MaxRequestsPerMinute)
            {
                times.Enqueue(now);
                return server; // this server has capacity
            }
        }

        throw new Exception("All servers at capacity");
    }
}
```

This combines **load balancing + sliding window rate limiting** — two system design concepts in one coding question.

---

### Question 4: LeetCode 1195 — Task Scheduler *(disguised LB question)*

Not labeled as load balancing, but the core problem is identical:

> "Given tasks and a cooldown period n, find minimum time to finish all tasks."

The insight is **distributing tasks (requests) across time slots (servers) with constraints** — the same mental model as load balancing with rate limits.

---

## One-Line Summary for Interviews

> **A load balancer is the single entry point that distributes incoming requests across a fleet of servers using algorithms like round robin or least connections, enabling horizontal scaling, zero-downtime deploys, and automatic failover — with Layer 7 balancers adding content-aware routing that lets you split traffic by URL, headers, or user type across different backend services.**


# CDN — Deep Dive for System Design Interviews

## Start With a Real Feeling

You're in Mumbai. You want to watch a concert highlight video hosted on a server in New York.

Every time you press play, your request travels:

```
Mumbai → undersea fiber cable → New York → video data → back to Mumbai
```

That's **~14,000 km** each way. At the speed of light through fiber, that's ~140ms of pure travel time — before any processing. For a video that needs to stream 30 frames per second, this feels like the video is loading through wet cement.

Now imagine there's a copy of that video sitting in a data center **in Chennai, 350km away**. Your request travels 350km instead of 14,000km. It feels instant.

**That server in Chennai is a CDN edge node.** That's the entire idea.

---

## What a CDN Actually Is

CDN = **Content Delivery Network**

A globally distributed network of servers (called **edge nodes** or **Points of Presence / PoPs**) that store copies of your content close to users around the world.

```
Without CDN:
User in Tokyo    ──────────────────────────→ Origin Server (New York)
User in London   ──────────────────────────→ Origin Server (New York)
User in Mumbai   ──────────────────────────→ Origin Server (New York)
User in São Paulo ─────────────────────────→ Origin Server (New York)
                                              (all competing for same server)

With CDN:
User in Tokyo    → Edge Node (Tokyo)      ✅ ~5ms
User in London   → Edge Node (London)     ✅ ~8ms
User in Mumbai   → Edge Node (Mumbai)     ✅ ~12ms
User in São Paulo → Edge Node (São Paulo) ✅ ~10ms
                    (origin server is barely touched)
```

---

## The Cache Hit Flow — Step by Step

### First user ever requests a file (Cache MISS)

```
Step 1: User in Tokyo requests /images/event-banner.jpg
Step 2: Tokyo edge node checks its cache → NOT FOUND
Step 3: Edge node fetches from Origin Server (New York)
Step 4: Origin returns the file
Step 5: Edge node stores it in local cache
Step 6: Edge node serves it to the user
```

Slow for this first user. But only this once.

### Every subsequent user (Cache HIT)

```
Step 1: Another user in Tokyo requests /images/event-banner.jpg
Step 2: Tokyo edge node checks cache → FOUND ✅
Step 3: Serves directly from edge cache
Step 4: Origin server never contacted
```

Fast. Origin server protected. This is the CDN's superpower.

---

## What Content Goes on a CDN

### Perfect for CDN (static, cacheable)

```
Images          → event posters, seat map thumbnails, artist photos
Videos          → concert highlights, venue walkthroughs
CSS/JS files    → your React bundle, Bootstrap CSS
Fonts           → Google Fonts, custom typefaces
Audio files     → previews, ringtones
PDFs            → ticket PDFs, venue maps
HTML            → static pages, documentation
```

### Bad for CDN (dynamic, personalized)

```
"Show my bookings"          → different for every user, can't cache
"Is seat 5A available?"     → changes every second
Payment processing          → must go to origin, security-sensitive
User authentication         → session-specific, never cache
Real-time inventory         → stale cache = overselling tickets
```

The key question: **"Would two different users get the same response for this request?"**
If yes → CDN candidate. If no → goes to origin.

---

## How CDN Routing Works

How does a user's browser end up at the nearest edge node automatically? Two mechanisms:

### Mechanism 1: Anycast Routing

The same IP address is announced from multiple locations worldwide. The internet's routing protocol (BGP) automatically sends traffic to the **topologically nearest** server announcing that IP.

```
CDN IP: 104.16.85.20  (announced from 200 locations)

Tokyo user's packet  → internet routes to Tokyo PoP
London user's packet → internet routes to London PoP
Mumbai user's packet → internet routes to Mumbai PoP

Same IP. Different physical servers. Magic of BGP.
```

### Mechanism 2: DNS-Based Routing (GeoDNS)

When your browser looks up `cdn.axs.com`:

```
Browser: "What's the IP for cdn.axs.com?"
DNS Server: checks your IP's geographic location
           "You're in Tokyo → here's the Tokyo edge IP: 1.2.3.4"

Different user in London:
DNS Server: "You're in London → here's the London edge IP: 5.6.7.8"
```

Same domain. Different IPs per location. Browser goes to nearest node automatically.

---

## Cache Control — Who Decides What Gets Cached

Your origin server tells the CDN how to cache via HTTP headers:

```http
Cache-Control: public, max-age=86400
```
→ Cache this for 24 hours. Serve from edge. Don't ask origin.

```http
Cache-Control: no-store
```
→ Never cache this. Always go to origin. (Use for booking confirmations, payments.)

```http
Cache-Control: public, max-age=3600, stale-while-revalidate=60
```
→ Cache for 1 hour. After expiry, serve stale content for 60 more seconds while
  fetching fresh copy in background. User never waits. ← production trick

```http
Cache-Control: private
```
→ Browser can cache, but CDN cannot. (User-specific pages.)

### TTL Strategy for AXS

```
Event banner images       → max-age=604800  (7 days, rarely changes)
Artist photos             → max-age=86400   (1 day)
Venue seating map         → max-age=3600    (1 hour)
JS/CSS bundles (hashed)   → max-age=31536000 (1 year — filename changes on deploy)
Seat availability         → no-store        (changes every second)
Booking confirmation page → no-store        (personalized, sensitive)
```

## Cache Invalidation — The Hard Problem

You cached an event banner for 7 days. The event gets cancelled. Now users see stale content for 7 days.

This is the classic problem: **"There are two hard things in CS: cache invalidation and naming things."**

### Solution 1: URL Fingerprinting (best for static assets)

Include a hash of the file content in the filename:

```
event-banner.jpg          ← bad, cached forever
event-banner.a3f9c2.jpg   ← good, hash changes when file changes
```

When you update the image → new hash → new URL → CDN fetches fresh copy automatically → old URL naturally expires.

### Solution 2: CDN Purge API (for urgent invalidation)

Every CDN exposes an API to instantly invalidate cached content:

```csharp
// Cloudflare purge example
public async Task PurgeCDNCache(string[] urls)
{
    var request = new
    {
        files = urls
    };

    await httpClient.PostAsJsonAsync(
        $"https://api.cloudflare.com/client/v4/zones/{zoneId}/purge_cache",
        request
    );
}

// Usage: event cancelled, purge its banner immediately
await PurgeCDNCache(new[] {
    "https://cdn.axs.com/events/taylor-swift-banner.jpg",
    "https://cdn.axs.com/events/taylor-swift-poster.jpg"
});
```

Purge propagates across all edge nodes globally in seconds.

### Solution 3: Surrogate Keys / Cache Tags

Tag cached content with logical keys, then purge by tag:

```
Tag: "event-12345" → applied to all assets for this event
When event cancelled: purge tag "event-12345"
→ all related assets invalidated in one API call
```

## CDN for Dynamic Content — The Surprise

Most people think CDN = static files only. But modern CDNs handle dynamic content too.

### Edge Caching with Vary Headers

```http
Vary: Accept-Language
```
CDN caches different versions of the same URL for different languages:

```
/homepage + Accept-Language: en  → cached English version
/homepage + Accept-Language: ja  → cached Japanese version
```

### Edge Computing (Cloudflare Workers, Lambda@Edge)

Run actual code at the edge node — before the request even reaches origin:

```javascript
// Cloudflare Worker — runs in Tokyo for Tokyo users
addEventListener('fetch', event => {
  event.respondWith(handleRequest(event.request));
});

async function handleRequest(request) {
  const userId = getCookie(request, 'userId');
  
  // A/B test at the edge — no origin needed
  if (hash(userId) % 2 === 0)
    return fetch('https://origin.axs.com/homepage-variant-a');
  else
    return fetch('https://origin.axs.com/homepage-variant-b');
}
```

Use cases run at edge without hitting origin:
- A/B testing
- Bot detection and blocking
- Authentication token validation
- Geoblocking (region-restricted content)
- Request/response transformation

## CDN Failure Modes and Handling

### Failure 1: Edge node goes down

CDN provider handles this automatically — traffic reroutes to next nearest PoP. You don't intervene.

### Failure 2: CDN provider outage (rare but happens)

```
// Origin fallback in your app
<img
  src="https://cdn.axs.com/event-banner.jpg"
  onerror="this.src='https://origin.axs.com/event-banner.jpg'"
/>
```

Or at DNS level — if CDN health check fails, DNS fails over to origin IP.

### Failure 3: Cache poisoning attack

Attacker tricks CDN into caching a malicious response.

Prevention:
```http
Vary: Host          ← don't cache based on attacker-controlled headers
Cache-Control: private  ← for anything sensitive
```

Always validate that CDN cache keys don't include user-controlled input.

## Complete Architecture: AXS With CDN

```
User Browser
     │
     ▼
[Cloudflare / CloudFront]     ← CDN Layer
  Edge node (nearest city)
  ├── Static assets: images, JS, CSS → served from edge cache
  ├── Bot detection → block at edge, never reaches origin
  ├── DDoS absorption → edge absorbs volumetric attacks
  └── Cache miss → forward to origin
     │
     ▼
[AWS ALB - Load Balancer]     ← Origin Layer
     │
  ┌──┼──┐
  ▼  ▼  ▼
[App Servers]                 ← Only sees cache misses + dynamic requests
     │
  ┌──┼──┐
  ▼  ▼  ▼
[Redis Cache]  [PostgreSQL]   ← Only sees what app servers can't handle
```

During a Taylor Swift flash sale:
- 95% of traffic (seat map images, JS bundles, CSS) → served by CDN, never touches servers
- 5% of traffic (seat availability, booking API) → hits origin servers
- Origin servers handle 5% of load instead of 100% → survive the spike

## When to Use a CDN — Decision Framework

**Use a CDN when:**
```
✅ You have global users (latency matters)
✅ You serve static assets (images, JS, CSS, video)
✅ You need to survive traffic spikes (flash sales, viral moments)
✅ You need DDoS protection at scale
✅ Your origin bandwidth costs are high
✅ Video streaming is involved
```

**Don't bother when:**
```
❌ All users are in one city (local startup, internal tool)
❌ Everything is dynamic and personalized (no cache benefit)
❌ Compliance requires data not leaving a region
❌ Real-time data that can never be stale (trading systems)
```

## Coding Questions From CDN

CDN is the most architecture-heavy topic so far. Almost no direct "implement a CDN" coding questions. But it spawns 3 real coding problems:

### Question 1: LRU Cache *(most commonly asked, directly related)*

> "Implement an LRU Cache with get and put in O(1)."

This **is** what runs inside every CDN edge node. Each edge node has limited SSD space. When it fills up, it evicts the **Least Recently Used** cached item. That's LRU.

```csharp
public class LRUCache
{
    private readonly int _capacity;
    private readonly Dictionary<string, LinkedListNode<(string key, string value)>> _map;
    private readonly LinkedList<(string key, string value)> _list;

    public LRUCache(int capacity)
    {
        _capacity = capacity;
        _map = new Dictionary<string, LinkedListNode<(string, string)>>();
        _list = new LinkedList<(string, string)>();
    }

    // O(1) get — move to front (most recently used)
    public string Get(string key)
    {
        if (!_map.ContainsKey(key)) return null;

        var node = _map[key];
        _list.Remove(node);
        _list.AddFirst(node);
        return node.Value.value;
    }

    // O(1) put — add to front, evict from back if full
    public void Put(string key, string value)
    {
        if (_map.ContainsKey(key))
        {
            var existing = _map[key];
            _list.Remove(existing);
            _map.Remove(key);
        }

        if (_map.Count >= _capacity)
        {
            // Evict LRU — tail of the list
            var lru = _list.Last;
            _list.RemoveLast();
            _map.Remove(lru.Value.key);
        }

        var node = _list.AddFirst((key, value));
        _map[key] = node;
    }
}
```

This is **LeetCode 146** — one of the most asked questions at FAANG. The connection to CDN gives you the perfect "why does this matter" story in interviews.

### Question 2: Cache with TTL expiry

> "Extend your cache so entries expire after a TTL."

```csharp
public class TTLCache
{
    private readonly Dictionary<string, (string value, DateTime expiry)> _store = new();

    public void Put(string key, string value, TimeSpan ttl)
    {
        _store[key] = (value, DateTime.UtcNow.Add(ttl));
    }

    public string Get(string key)
    {
        if (!_store.TryGetValue(key, out var entry))
            return null;

        // Lazy expiry — check on read
        if (DateTime.UtcNow > entry.expiry)
        {
            _store.Remove(key);
            return null; // cache miss — TTL expired
        }

        return entry.value;
    }
}
```

**Interviewer follow-up:** "Lazy expiry means stale keys sit in memory until accessed. How do you proactively clean them?"

**Answer:** Background thread that scans for expired keys periodically — exactly how Redis does it (combination of lazy + active expiry).

### Question 3: Design a URL shortener with CDN *(system design + code)*

> "Design bit.ly. How does CDN play a role?"

```csharp
// Short URL redirect — perfect CDN candidate
// GET /abc123 → 301 redirect to original URL

public class UrlShortener
{
    public HttpResponse Redirect(string shortCode)
    {
        var originalUrl = _db.Lookup(shortCode);

        return new HttpResponse
        {
            StatusCode = 301, // permanent redirect
            Headers = new()
            {
                // Tell CDN to cache this redirect for 1 day
                // Same short code always goes to same original URL
                ["Cache-Control"] = "public, max-age=86400",
                ["Location"] = originalUrl
            }
        };
    }
}
```

CDN caches the redirect. Second time anyone hits `/abc123` from the same region → CDN responds instantly, origin never called.
