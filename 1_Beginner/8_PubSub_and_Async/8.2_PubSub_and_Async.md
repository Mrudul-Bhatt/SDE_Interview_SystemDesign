# Pub/Sub Pattern — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine a newspaper.

**The traditional message queue is home delivery.** You subscribe. The newspaper is delivered to YOUR door. Once you've read it, it's gone. Your neighbor doesn't get a copy unless they also subscribed separately. If you weren't home, the paper piles up at your door until you pick them all up.

**Pub/Sub is a broadcast radio station.** The station broadcasts. Anyone with a radio tuned to that frequency hears it simultaneously. The station doesn't know who's listening. Doesn't care. It just broadcasts. Ten listeners or ten million — the station does the same thing. If you turn off your radio, you miss what was said. When you turn it back on, you pick up from right now.

These aren't just analogies — they describe the fundamental difference between how message queues and pub/sub work, and therefore when to use each.

---

## What Pub/Sub Actually Is

Pub/Sub (Publish-Subscribe) is a **messaging pattern** where:

- **Publishers** send messages to named channels called **topics**
- **Subscribers** express interest in topics they care about
- Publishers and subscribers **never communicate directly** — they don't know each other exist
- A message broker sits in the middle, routing messages from publishers to subscribers

```
Without Pub/Sub (direct coupling):
  Booking Service → calls EmailService.Send()
  Booking Service → calls SMSService.Send()
  Booking Service → calls AnalyticsService.Record()
  Booking Service → calls FraudService.Check()
  Booking Service → calls InventoryService.Decrement()
  
  Booking Service knows about all 5 downstream services.
  Adding a 6th service = modify Booking Service.
  EmailService down = Booking Service needs error handling for it.
  All 5 must be available for booking to complete.

With Pub/Sub (decoupled):
  Booking Service → publishes "booking.created" to topic → done
  
  EmailService    → subscribed to "booking.created" → sends email
  SMSService      → subscribed to "booking.created" → sends SMS
  AnalyticsService → subscribed to "booking.created" → records event
  FraudService    → subscribed to "booking.created" → checks fraud
  InventoryService → subscribed to "booking.created" → decrements count
  
  Booking Service knows nothing about any of these.
  Adding a 6th service = zero changes to Booking Service.
  EmailService down = other subscribers unaffected, email retries independently.
  Booking completes as soon as message is published.
```

The booking service went from knowing about 5 services to knowing about zero. It just shouts into the void. Whoever cares about bookings is that service's problem to sort out.

## The Fundamental Difference — Queue vs Pub/Sub

This is the most important question in this topic. Get this right and the rest flows naturally.

### Message Queue — Work Distribution

```
One message → One consumer (competing consumers)

Queue: [booking #1][booking #2][booking #3][booking #4][booking #5]
         ↓              ↓              ↓
    Worker 1      Worker 2      Worker 3
    processes     processes     processes
    booking #1    booking #2    booking #3
    
    Each message processed by EXACTLY ONE worker.
    Workers compete for messages.
    Purpose: distribute work / load balance.
```

Think of a queue as a **to-do list** that multiple workers share. Each task gets done once.

### Pub/Sub — Event Broadcasting

```
One message → All subscribers (fanout)

Topic "booking.created" → message arrives
         ↓         ↓         ↓         ↓         ↓
    Email     SMS      Analytics  Fraud    Inventory
    Service   Service  Service    Service  Service
    
    EACH subscriber gets a COPY of the message.
    Purpose: broadcast events / notify multiple systems.
```

Think of pub/sub as a **broadcast channel** that multiple listeners tune into. Each listener gets the full broadcast.

### The Defining Test

Ask yourself: **"Should this event be processed by one thing, or by everything that cares about it?"**

```
"Process this payment" → ONE payment worker should process it
  (you don't want 3 workers all charging the same card)
  → MESSAGE QUEUE

"A booking was created" → EVERY interested system should know
  (email service, SMS service, analytics, fraud — all need to know)
  → PUB/SUB

"Resize this uploaded image" → ONE worker should resize it
  (you don't want 3 workers creating 3 copies of each thumbnail)
  → MESSAGE QUEUE

"A new user registered" → EVERY interested system should know
  (welcome email, onboarding flow, marketing attribution, analytics)
  → PUB/SUB
```

## Pub/Sub Delivery Models — A Critical Nuance

Not all pub/sub works the same way. There are two fundamentally different delivery models that interviewers distinguish between.

### Model 1: Fire-and-Forget (True Pub/Sub)

```
Publisher sends → Broker immediately delivers to connected subscribers
                  If subscriber disconnected → message LOST
                  
Redis PUBLISH/SUBSCRIBE works this way.

Timeline:
  T=0:  Subscriber A connected, Subscriber B disconnected
  T=1:  Publisher publishes "event X"
  T=1:  Broker → Subscriber A (receives it) ✅
  T=1:  Broker → Subscriber B is offline → message dropped ❌
  T=5:  Subscriber B reconnects
  T=5:  Subscriber B missed "event X" — no replay
```

Use when: Real-time notifications where missing messages is acceptable. Chat, live score updates, collaborative cursors.

### Model 2: Durable Pub/Sub (Subscription Queues)

```
Publisher sends → Broker stores message → Each subscriber has own queue
                  Subscriber reconnects → reads from their queue
                  No message loss

Google Pub/Sub, AWS SNS+SQS, Kafka consumer groups work this way.

Timeline:
  T=0:  Subscriber A connected, Subscriber B's queue exists (B is offline)
  T=1:  Publisher publishes "event X"
  T=1:  Broker → Subscriber A's queue (receives immediately) ✅
  T=1:  Broker → Subscriber B's queue (stored for later) 📦
  T=5:  Subscriber B reconnects → reads from queue → gets "event X" ✅
```

Use when: Processing pipelines where every subscriber must process every event. Payment notifications, audit logs, inventory updates.

## Kafka — The Production Pub/Sub Deep Dive

Kafka dominates at scale because it elegantly unifies queue and pub/sub semantics.

### How Kafka achieves both

```
Topic: "booking.created"
  Partition 0: [event1][event4][event7]
  Partition 1: [event2][event5][event8]
  Partition 2: [event3][event6][event9]

Consumer Group "email-service":
  Consumer A → reads Partition 0 exclusively
  Consumer B → reads Partition 1 exclusively
  Consumer C → reads Partition 2 exclusively
  
  → Each event processed by EXACTLY ONE consumer in the group
  → QUEUE behavior within the group

Consumer Group "analytics-service":
  Consumer D → reads ALL partitions
  
  → Gets ALL events independently from email-service
  → PUB/SUB behavior across groups
```

**Queue behavior:** Multiple consumers in one group split the work. No event processed twice within the group.

**Pub/Sub behavior:** Multiple consumer groups all read the same topic. Each group gets every event.

This is why Kafka is the right answer for complex systems — one topic can serve both patterns.

### Kafka offset management — the durability key

```
Consumer reads event at offset 47.
Consumer processes it successfully.
Consumer commits offset 48 (next to read).

Consumer crashes before committing:
  Restarts → reads offset 47 again → processes again
  → At-least-once delivery (need idempotency)

Consumer commits before processing:
  Commits 48 → crashes → event at 47 never processed
  → At-most-once delivery (can lose events)

Consumer processes + commits atomically (Kafka transactions):
  → Exactly-once delivery (most complex, highest overhead)
```

## Topic Design — Getting the Granularity Right

One of the most nuanced pub/sub design decisions is how to structure topics.

### Too coarse (one topic for everything):

```
Topic: "all_events"
  booking.created
  payment.processed
  user.registered
  seat.locked
  event.cancelled
  ...

Problems:
  Every subscriber receives everything
  Subscribers filter client-side (wasteful, complex)
  No ability to independently scale consumers per event type
  One noisy event type blocks others
```

### Too fine (one topic per entity action):

```
Topics:
  booking.created.userId.123
  booking.created.userId.124
  booking.created.userId.125
  ...millions of topics

Problems:
  Subscriber must subscribe to millions of topics
  Topic management overhead
  Can't subscribe to "all bookings created"
```

### Right granularity (domain events):

```
Topics by domain + major lifecycle event:
  bookings.created      → created bookings
  bookings.cancelled    → cancelled bookings
  payments.succeeded    → successful payments
  payments.failed       → failed payments
  users.registered      → new user registrations
  events.published      → new events going live
  events.cancelled      → cancelled events
  
Subscribers choose exactly which events they care about:
  EmailService:    bookings.created, bookings.cancelled, payments.succeeded
  FraudService:    payments.failed, users.registered
  InventoryService: bookings.created, bookings.cancelled
  Analytics:       ALL topics (subscribe to all)
```

## Fan-Out Patterns

### Pattern 1: Simple Fan-Out

```
One publisher → One topic → N subscribers
All subscribers receive every message.

booking.created topic
     │
     ├──► EmailService
     ├──► SMSService
     ├──► AnalyticsService
     └──► FraudService

Use: When all subscribers care about every event.
```

### Pattern 2: Fan-Out with Filtering

```
booking.created topic (with attributes/headers)
  {
    eventId: 123,
    tier: "vip",
    region: "us-west",
    amount: 500
  }

Subscribers filter:
  VIPService:       subscribes with filter: tier = "vip"
  USWestService:    subscribes with filter: region = "us-west"
  HighValueService: subscribes with filter: amount > 250
  
Only messages matching filter delivered to each subscriber.
```

### Pattern 3: SNS + SQS Fan-Out (AWS)

```
The canonical AWS durable pub/sub pattern:

Publisher → SNS Topic → SQS Queue A → Consumer A workers
                      → SQS Queue B → Consumer B workers
                      → SQS Queue C → Consumer C workers

SNS handles fan-out (pub/sub semantics).
SQS provides durability per consumer (queue semantics).
Each consumer group has own queue — messages persist even if consumer offline.
```

### Pattern 4: Topic Hierarchy (MQTT-style)

```
Topics form a hierarchy with wildcard subscriptions:

Published topic: events/us-east/concerts/taylor-swift/available

Subscriber A: events/# (all events)
Subscriber B: events/us-east/# (all US east events)
Subscriber C: events/+/concerts/# (all concerts anywhere)
Subscriber D: events/us-east/concerts/taylor-swift/# (specific artist)

Each receives only matching messages.
Used heavily in IoT systems.
```

## Implementation — In-Memory Pub/Sub

```csharp
public class InMemoryPubSub
{
    // topic → list of handlers
    private readonly ConcurrentDictionary<string, List<Func<object, Task>>> _handlers = new();
    private readonly ReaderWriterLockSlim _lock = new();

    public void Subscribe<T>(string topic, Func<T, Task> handler)
    {
        _lock.EnterWriteLock();
        try
        {
            var handlers = _handlers.GetOrAdd(topic, _ => new List<Func<object, Task>>());
            handlers.Add(async msg =>
            {
                if (msg is T typed)
                    await handler(typed);
            });
        }
        finally { _lock.ExitWriteLock(); }
    }

    public void Unsubscribe(string topic, Func<object, Task> handler)
    {
        _lock.EnterWriteLock();
        try
        {
            if (_handlers.TryGetValue(topic, out var handlers))
                handlers.Remove(handler);
        }
        finally { _lock.ExitWriteLock(); }
    }

    public async Task PublishAsync<T>(string topic, T message)
    {
        List<Func<object, Task>> handlers;

        _lock.EnterReadLock();
        try
        {
            if (!_handlers.TryGetValue(topic, out var existing))
                return;
            handlers = new List<Func<object, Task>>(existing);
        }
        finally { _lock.EnterReadLock(); }

        // Deliver to all subscribers in parallel
        var deliveryTasks = handlers.Select(async handler =>
        {
            try
            {
                await handler(message!);
            }
            catch (Exception ex)
            {
                // One subscriber failing doesn't affect others
                Console.Error.WriteLine(
                    $"Subscriber failed for topic '{topic}': {ex.Message}"
                );
            }
        });

        await Task.WhenAll(deliveryTasks);
    }

    public int GetSubscriberCount(string topic)
    {
        _lock.EnterReadLock();
        try
        {
            return _handlers.TryGetValue(topic, out var h) ? h.Count : 0;
        }
        finally { _lock.ExitReadLock(); }
    }
}

// Usage
var bus = new InMemoryPubSub();

// Wire up subscribers during startup
bus.Subscribe<BookingCreatedEvent>("bookings.created", async booking =>
{
    await emailService.SendConfirmationAsync(booking.UserId, booking.Id);
});

bus.Subscribe<BookingCreatedEvent>("bookings.created", async booking =>
{
    await analyticsService.RecordBookingAsync(booking.EventId, booking.Amount);
});

bus.Subscribe<BookingCreatedEvent>("bookings.created", async booking =>
{
    await inventoryService.DecrementSeatAsync(booking.SeatId);
});

// Publish — doesn't know about any subscribers
await bus.PublishAsync("bookings.created", new BookingCreatedEvent
{
    Id = "booking-456",
    UserId = 42,
    EventId = 123,
    SeatId = "A5",
    Amount = 250m
});
// All three subscribers notified in parallel
```

## Implementation — Redis Pub/Sub (Fire-and-Forget)

```csharp
public class RedisPubSubService
{
    private readonly IConnectionMultiplexer _redis;
    private readonly ISubscriber _subscriber;
    private readonly Dictionary<string, List<Action<string>>> _handlers = new();

    public RedisPubSubService(IConnectionMultiplexer redis)
    {
        _redis = redis;
        _subscriber = redis.GetSubscriber();
    }

    // Subscribe to a Redis channel
    public async Task SubscribeAsync(string channel, Action<string> handler)
    {
        if (!_handlers.ContainsKey(channel))
        {
            _handlers[channel] = new List<Action<string>>();

            // Only subscribe to Redis once per channel
            await _subscriber.SubscribeAsync(
                RedisChannel.Literal(channel),
                (_, message) =>
                {
                    // Deliver to all local handlers for this channel
                    foreach (var h in _handlers[channel])
                    {
                        try { h(message!); }
                        catch (Exception ex)
                        {
                            Console.Error.WriteLine($"Handler error: {ex.Message}");
                        }
                    }
                }
            );
        }

        _handlers[channel].Add(handler);
    }

    // Publish to a Redis channel
    public async Task<long> PublishAsync(string channel, string message)
    {
        // Returns number of subscribers that received the message
        return await _subscriber.PublishAsync(
            RedisChannel.Literal(channel),
            message
        );
    }
}

// Real-time seat update during flash sale
public class SeatAvailabilityService
{
    private readonly RedisPubSubService _pubSub;
    private readonly IDatabase _redis;

    // When a seat is locked, notify all interested clients
    public async Task LockSeatAsync(string eventId, string seatId, string userId)
    {
        // Update the lock in Redis
        await _redis.StringSetAsync(
            $"lock:seat:{eventId}:{seatId}",
            userId,
            TimeSpan.FromMinutes(10)
        );

        // Publish seat status change — all WebSocket servers receive this
        await _pubSub.PublishAsync(
            $"seat_updates:{eventId}",
            JsonSerializer.Serialize(new SeatUpdate
            {
                SeatId = seatId,
                Status = "locked",
                UpdatedAt = DateTime.UtcNow
            })
        );
    }

    // WebSocket server subscribes for its connected event rooms
    public async Task SubscribeToEventUpdatesAsync(
        string eventId,
        WebSocketConnectionManager wsManager)
    {
        await _pubSub.SubscribeAsync(
            $"seat_updates:{eventId}",
            async (message) =>
            {
                // Push to all WebSocket clients watching this event
                await wsManager.BroadcastToEventAsync(eventId, message);
            }
        );
    }
}
```

**Critical limitation:** Redis pub/sub is fire-and-forget. If a WebSocket server is down when a seat update is published, that update is lost. The WebSocket server must handle reconnection and state sync on reconnect.

## Implementation — Kafka Durable Pub/Sub

```csharp
public class KafkaPublisher
{
    private readonly IProducer<string, string> _producer;

    public KafkaPublisher(string bootstrapServers)
    {
        var config = new ProducerConfig
        {
            BootstrapServers = bootstrapServers,
            EnableIdempotence = true,      // exactly-once semantics
            Acks = Acks.All,               // wait for all replicas
            MessageSendMaxRetries = 3,
            RetryBackoffMs = 100
        };

        _producer = new ProducerBuilder<string, string>(config).Build();
    }

    public async Task PublishAsync<T>(
        string topic,
        string partitionKey, // ensures ordering for related events
        T message)
    {
        var json = JsonSerializer.Serialize(message);

        var result = await _producer.ProduceAsync(topic, new Message<string, string>
        {
            Key = partitionKey,  // same key → same partition → ordered
            Value = json,
            Headers = new Headers
            {
                { "event-type", Encoding.UTF8.GetBytes(typeof(T).Name) },
                { "published-at", Encoding.UTF8.GetBytes(DateTime.UtcNow.ToString("O")) },
                { "correlation-id", Encoding.UTF8.GetBytes(Activity.Current?.Id ?? Guid.NewGuid().ToString()) }
            }
        });

        Console.WriteLine(
            $"Published to {result.TopicPartitionOffset}"
        );
    }
}

public class KafkaConsumer<T>
{
    private readonly IConsumer<string, string> _consumer;
    private readonly Func<T, Task> _handler;
    private readonly string _topic;

    public KafkaConsumer(
        string bootstrapServers,
        string groupId,       // consumer group = pub/sub subscription
        string topic,
        Func<T, Task> handler)
    {
        _topic = topic;
        _handler = handler;

        var config = new ConsumerConfig
        {
            BootstrapServers = bootstrapServers,
            GroupId = groupId,
            AutoOffsetReset = AutoOffsetReset.Earliest, // start from beginning
            EnableAutoCommit = false  // manual commit for at-least-once
        };

        _consumer = new ConsumerBuilder<string, string>(config).Build();
    }

    public async Task ConsumeAsync(CancellationToken cancellationToken)
    {
        _consumer.Subscribe(_topic);

        while (!cancellationToken.IsCancellationRequested)
        {
            try
            {
                var consumeResult = _consumer.Consume(
                    TimeSpan.FromMilliseconds(100)
                );

                if (consumeResult?.Message == null) continue;

                var message = JsonSerializer.Deserialize<T>(consumeResult.Message.Value)
                    ?? throw new Exception("Null message");

                // Process the message
                await _handler(message);

                // Commit AFTER successful processing (at-least-once)
                _consumer.Commit(consumeResult);
            }
            catch (ConsumeException ex)
            {
                Console.Error.WriteLine($"Consume error: {ex.Error.Reason}");
            }
            catch (Exception ex)
            {
                Console.Error.WriteLine($"Processing error: {ex.Message}");
                // Don't commit — message will be redelivered (at-least-once)
            }
        }
    }
}

// Usage — independent consumer groups for different subscribers
// Email service consumer group
var emailConsumer = new KafkaConsumer<BookingCreatedEvent>(
    bootstrapServers: "kafka:9092",
    groupId: "email-service",        // unique group = independent subscription
    topic: "bookings.created",
    handler: async booking =>
    {
        await emailService.SendConfirmationAsync(booking.UserId, booking.Id);
    }
);

// Analytics consumer group — reads same topic independently
var analyticsConsumer = new KafkaConsumer<BookingCreatedEvent>(
    bootstrapServers: "kafka:9092",
    groupId: "analytics-service",    // different group = independent offset tracking
    topic: "bookings.created",
    handler: async booking =>
    {
        await analyticsService.RecordAsync(booking);
    }
);

// Both consumer groups get all messages
// email-service at its own pace
// analytics-service at its own pace
// Neither affects the other
await Task.WhenAll(
    emailConsumer.ConsumeAsync(cancellationToken),
    analyticsConsumer.ConsumeAsync(cancellationToken)
);
```

## AXS System Design — Pub/Sub Architecture

```
BOOKING FLOW with Pub/Sub:

[User clicks Buy] → [Booking Service]
                           │
                    (ACID transaction: DB write)
                           │
                    Publishes to Kafka:
                    Topic: "bookings.created"
                    Key: userId (ensures user's events ordered)
                    Payload: {bookingId, userId, eventId, seatId, amount}
                           │
                           ▼
                    Returns "Booking Confirmed" to user
                    (< 100ms — doesn't wait for subscribers)

SUBSCRIBERS (all independent, all async):

Consumer Group: "notification-email"
  → Sends booking confirmation email
  → Retry on failure (up to 3 times with backoff)
  → DLQ after 3 failures (manual review)

Consumer Group: "notification-sms"
  → Sends booking confirmation SMS
  → Independent of email (email failure doesn't block SMS)

Consumer Group: "fraud-detection"
  → Analyzes booking pattern for fraud signals
  → Publishes to "fraud.alerts" if suspicious
  → Independent — doesn't block booking flow

Consumer Group: "analytics"
  → Records booking in analytics warehouse
  → Batch processing, can be hours behind
  → Zero impact on booking latency

Consumer Group: "inventory"
  → Decrements seats_available in events service
  → Critical — processes at high priority
  → Idempotent: uses bookingId to prevent double-decrement

Consumer Group: "seat-map-cache"
  → Invalidates seat map cache for this event
  → Refreshes Redis cache with updated availability

REAL-TIME SEAT UPDATES (separate, Redis Pub/Sub):

[Seat status changes in DB]
  → Redis PUBLISH "seat_updates:{eventId}"
  
[WebSocket Server 1] subscribes → pushes to connected US users
[WebSocket Server 2] subscribes → pushes to connected EU users
[WebSocket Server 3] subscribes → pushes to connected AP users

All servers get the message via Redis pub/sub.
Each pushes to their locally connected WebSocket clients.
```

## Queue vs Pub/Sub — Complete Comparison

| Dimension | Message Queue | Pub/Sub |
|---|---|---|
| Delivery | One consumer gets the message | All subscribers get a copy |
| Purpose | Distribute work | Broadcast events |
| Producers know consumers? | Yes (implicitly) | No (decoupled) |
| Adding consumers | Adds parallel workers | Adds independent subscribers |
| Message persistence | Until consumed | Depends on model (ephemeral or durable) |
| Ordering | FIFO within queue | Per partition (Kafka) or none |
| Use case | Task queue, job processing | Event notification, fan-out |
| Examples | SQS, RabbitMQ queue | SNS, Redis pub/sub, Kafka topics |

### Question 1: Implement a pub/sub event bus with typed events

> "Build a type-safe in-memory event bus supporting multiple subscribers per event type."

```csharp
public class TypedEventBus
{
    private readonly Dictionary<Type, List<object>> _handlers = new();
    private readonly SemaphoreSlim _lock = new(1, 1);

    public async Task SubscribeAsync<TEvent>(Func<TEvent, Task> handler)
    {
        await _lock.WaitAsync();
        try
        {
            var eventType = typeof(TEvent);
            if (!_handlers.ContainsKey(eventType))
                _handlers[eventType] = new List<object>();
            _handlers[eventType].Add(handler);
        }
        finally { _lock.Release(); }
    }

    public async Task PublishAsync<TEvent>(TEvent @event)
    {
        List<object> handlers;

        await _lock.WaitAsync();
        try
        {
            if (!_handlers.TryGetValue(typeof(TEvent), out var h))
                return;
            handlers = new List<object>(h);
        }
        finally { _lock.Release(); }

        var failedHandlers = new List<(object handler, Exception error)>();

        await Task.WhenAll(handlers.Select(async handler =>
        {
            try
            {
                await ((Func<TEvent, Task>)handler)(@event);
            }
            catch (Exception ex)
            {
                lock (failedHandlers)
                    failedHandlers.Add((handler, ex));
            }
        }));

        if (failedHandlers.Any())
        {
            Console.Error.WriteLine(
                $"{failedHandlers.Count} handlers failed for {typeof(TEvent).Name}"
            );
        }
    }

    public async Task<int> GetSubscriberCountAsync<TEvent>()
    {
        await _lock.WaitAsync();
        try
        {
            return _handlers.TryGetValue(typeof(TEvent), out var h) ? h.Count : 0;
        }
        finally { _lock.Release(); }
    }
}
```

### Question 2: Implement a durable pub/sub with replay

> "Subscribers that reconnect should receive messages they missed."

```csharp
public class DurablePubSub
{
    // topic → ordered list of messages with IDs
    private readonly Dictionary<string, List<StoredMessage>> _topics = new();
    // subscriptionId → last consumed message ID
    private readonly Dictionary<string, long> _offsets = new();
    private long _messageCounter = 0;
    private readonly object _lock = new();

    public void CreateSubscription(string subscriptionId, string topic)
    {
        lock (_lock)
        {
            // New subscription starts from current end (doesn't get old messages)
            var messages = _topics.GetValueOrDefault(topic, new List<StoredMessage>());
            _offsets[subscriptionId] = messages.Count > 0
                ? messages.Last().Id
                : -1;
        }
    }

    public void Publish(string topic, string payload)
    {
        lock (_lock)
        {
            if (!_topics.ContainsKey(topic))
                _topics[topic] = new List<StoredMessage>();

            _topics[topic].Add(new StoredMessage
            {
                Id = Interlocked.Increment(ref _messageCounter),
                Payload = payload,
                PublishedAt = DateTime.UtcNow
            });
        }
    }

    // Pull messages since last consumed — enables replay on reconnect
    public List<StoredMessage> Pull(string subscriptionId, int maxMessages = 100)
    {
        lock (_lock)
        {
            if (!_offsets.TryGetValue(subscriptionId, out long lastOffset))
                throw new InvalidOperationException($"Unknown subscription: {subscriptionId}");

            // Find topic for this subscription
            string topic = GetTopicForSubscription(subscriptionId);
            var messages = _topics.GetValueOrDefault(topic, new List<StoredMessage>());

            return messages
                .Where(m => m.Id > lastOffset)
                .Take(maxMessages)
                .ToList();
        }
    }

    public void Acknowledge(string subscriptionId, long messageId)
    {
        lock (_lock)
        {
            if (_offsets.ContainsKey(subscriptionId))
                _offsets[subscriptionId] = Math.Max(_offsets[subscriptionId], messageId);
        }
    }

    private string GetTopicForSubscription(string subscriptionId)
    {
        // In real implementation, store subscription → topic mapping
        return subscriptionId.Split(':')[0];
    }
}
```

### Question 3: Design dead letter handling for failed subscribers

> "If a subscriber fails after 3 retries, move the message to a dead letter queue."

```csharp
public class ReliablePubSubConsumer<T>
{
    private readonly Func<T, Task> _handler;
    private readonly Queue<T> _deadLetterQueue = new();
    private readonly int _maxRetries;
    private readonly TimeSpan _baseRetryDelay;

    public ReliablePubSubConsumer(
        Func<T, Task> handler,
        int maxRetries = 3,
        TimeSpan? baseRetryDelay = null)
    {
        _handler = handler;
        _maxRetries = maxRetries;
        _baseRetryDelay = baseRetryDelay ?? TimeSpan.FromSeconds(1);
    }

    public async Task ProcessAsync(T message)
    {
        Exception? lastException = null;

        for (int attempt = 1; attempt <= _maxRetries; attempt++)
        {
            try
            {
                await _handler(message);
                return; // success
            }
            catch (Exception ex)
            {
                lastException = ex;
                Console.Error.WriteLine(
                    $"Attempt {attempt}/{_maxRetries} failed: {ex.Message}"
                );

                if (attempt < _maxRetries)
                {
                    // Exponential backoff: 1s, 2s, 4s
                    var delay = TimeSpan.FromMilliseconds(
                        _baseRetryDelay.TotalMilliseconds * Math.Pow(2, attempt - 1)
                        + Random.Shared.Next(0, 500) // jitter
                    );
                    await Task.Delay(delay);
                }
            }
        }

        // All retries exhausted → dead letter queue
        Console.Error.WriteLine(
            $"Moving to DLQ after {_maxRetries} failures: {lastException?.Message}"
        );

        _deadLetterQueue.Enqueue(message);
        await NotifyDeadLetterAsync(message, lastException!);
    }

    // Process dead letters manually or with reduced rate
    public async Task ReprocessDeadLettersAsync()
    {
        Console.WriteLine($"Reprocessing {_deadLetterQueue.Count} dead letters");

        while (_deadLetterQueue.TryDequeue(out var message))
        {
            try
            {
                await _handler(message);
                Console.WriteLine("Dead letter reprocessed successfully");
            }
            catch (Exception ex)
            {
                Console.Error.WriteLine($"Dead letter still failing: {ex.Message}");
                // Could send to permanent failure storage, alert on-call, etc.
            }

            await Task.Delay(TimeSpan.FromSeconds(1)); // throttle reprocessing
        }
    }

    private async Task NotifyDeadLetterAsync(T message, Exception ex)
    {
        // Alert on-call, write to monitoring, etc.
        Console.Error.WriteLine($"[ALERT] Dead letter: {JsonSerializer.Serialize(message)}");
        await Task.CompletedTask;
    }

    public int DeadLetterCount => _deadLetterQueue.Count;
}
```

## One-Line Summary for Interviews

> **Pub/Sub is a messaging pattern where publishers broadcast events to named topics without knowing who receives them, and subscribers declare interest in topics and receive every matching event — fundamentally different from message queues where one message goes to one worker for work distribution, versus pub/sub where one event goes to all subscribers for event broadcasting; implemented as fire-and-forget (Redis pub/sub — real-time, message lost if subscriber offline) or durable subscriptions (Kafka consumer groups, SNS+SQS — each subscriber has own offset/queue, no message loss on reconnect); Kafka elegantly unifies both patterns — consumer groups within a topic behave as queues (one consumer per group processes each message) while multiple consumer groups behave as pub/sub (each group gets every message independently) — and the right architecture for complex systems uses both: message queues for distributing parallel work and pub/sub for broadcasting domain events to all interested services without coupling them together.**

# Dead Letter Queue — Deep Dive for System Design Interviews

## Start With a Real Feeling

The post office delivers millions of letters every day. Most arrive fine. But some can't be delivered — wrong address, recipient moved, package too large for the mailbox, address in a foreign script the carrier can't read.

What does the post office do with undeliverable mail? They don't throw it away. They don't try delivering it forever. They put it in a special room: **"Dead Letter Mail."**

This room serves a crucial purpose:
- The undeliverable mail is preserved (not lost)
- The main delivery system keeps running (not blocked by problem mail)
- Postal workers can investigate: why couldn't this be delivered?
- Some mail can be fixed and redelivered. Some truly can't be — it stays in the room.

**A Dead Letter Queue (DLQ) is exactly this.** Messages that your system can't process — after retrying — go to a special queue where they wait. The main processing pipeline isn't blocked. The messages aren't lost. Engineers can investigate, fix the root cause, and replay the messages.

---

## What a Dead Letter Queue Is

A Dead Letter Queue is a **secondary queue** where messages are moved after they fail to be processed successfully after a maximum number of retry attempts.

```
Normal flow:
  Producer → [Main Queue] → Consumer → Processes successfully → Message removed

Failure flow without DLQ:
  Producer → [Main Queue] → Consumer → Fails
                               ↑ requeue ↓ fails again
                          Consumer → Fails
                               ↑ requeue ↓ fails again... forever
  The broken message blocks other messages or loops indefinitely

Failure flow with DLQ:
  Producer → [Main Queue] → Consumer → Fails
                               ↑ retry 1 → Fails
                               ↑ retry 2 → Fails
                               ↑ retry 3 → Fails (max retries reached)
                                        → Message moved to [DLQ]
                                        
  Main Queue: continues processing other messages normally
  DLQ: holds the problematic message for investigation
```

---

## Why Messages End Up in the DLQ

Understanding failure modes is critical for interviews. Messages end up in DLQs for different reasons that require different responses.

### Category 1: Transient Failures (Should Resolve with Retry)

```
Downstream service temporarily unavailable:
  Message: "send booking email to user 42"
  Failure: EmailService returns 503 (temporarily overloaded)
  
  Retry 1: 1 second later → still 503
  Retry 2: 2 seconds later → still 503
  Retry 3: 4 seconds later → 503 (service came back at second 5)
  Message → DLQ
  
  Root cause: transient service issue, now resolved
  Fix: replay from DLQ — will succeed now
  
Network timeout:
  Payment service took 32 seconds, consumer timeout is 30
  Service actually succeeded → consumer didn't know
  Message → DLQ
  
  Root cause: timeout too short for operation
  Fix: idempotency key ensures replay doesn't double-charge
       replay succeeds (payment already done, returns existing result)
```

### Category 2: Permanent Failures (Will Never Succeed Without Fix)

```
Malformed message:
  Message: "{ userId: abc, amount: -50 }"
  Failure: userId should be integer, amount can't be negative
  
  Retry 1: Same bad data → same failure
  Retry N: Same bad data → same failure
  Message → DLQ
  
  Root cause: producer sent invalid data
  Fix: fix producer, manually correct message, replay

Missing resource:
  Message: "create ticket for event_id=9999"
  Failure: event 9999 was deleted before message processed
  
  Retry N: event still deleted
  Message → DLQ
  
  Root cause: event deleted while message in flight
  Fix: decide — should we create the event back? skip this message?
       investigate and manually resolve

Schema version mismatch:
  Producer sends v2 schema, consumer only understands v1
  Consumer can't deserialize → throws exception
  
  Message → DLQ
  
  Root cause: consumer not updated with new schema
  Fix: update consumer, then replay DLQ
```

### Category 3: Business Logic Failures

```
Insufficient inventory:
  Message: "reserve seat A5 for user 42"
  Failure: seat A5 was booked by someone else between message sent and processed
  
  Message → DLQ
  
  Root cause: race condition, business rule violation
  Fix: manual decision — notify user, offer alternative seat
```

## DLQ Monitoring — What to Watch

A DLQ silently filling up is a silent disaster. Monitoring is not optional.

```
Critical metrics to alert on:

DLQ depth:
  Alert if: messages in DLQ > 0 (for critical queues)
  Alert if: messages in DLQ growing faster than investigation rate
  
  Why: DLQ growing = you're losing business. Messages in DLQ are
       bookings not confirmed, payments not processed, users not notified.

DLQ age of oldest message:
  Alert if: oldest message > 1 hour (investigate quickly)
  Alert if: oldest message > 24 hours (severe: customer impacted)

DLQ message rate:
  Alert if: sudden spike in DLQ arrivals
  Why: spike indicates a bug deployment or downstream failure

Main queue consumer errors:
  Track error rate before messages reach DLQ
  High error rate = DLQ incoming surge imminent
```

## The Replay Problem — Getting Messages Out of DLQ

Having a DLQ is half the battle. The other half is safely replaying messages after fixing the root cause.

### Replay Challenges

```
Challenge 1: Idempotency
  Message was partially processed before failing.
  Payment was charged but booking record wasn't created.
  Replay: charges again → double charge!
  
  Solution: idempotency keys throughout the pipeline.
  Replay is safe only if every operation is idempotent.

Challenge 2: Ordering
  Message B (booking update) depends on message A (booking creation).
  A failed, went to DLQ.
  B processed successfully (or also in DLQ).
  Replay A: processes booking creation.
  But booking update (B) must be replayed AFTER A.
  
  Solution: replay in timestamp order.
  Or: replay individual messages, handle dependency errors gracefully.

Challenge 3: Data staleness  
  Message: "set user email to old@email.com"
  User updated email to current@email.com while this was in DLQ.
  Replaying the message reverts to old email!
  
  Solution: include version/timestamp in message.
  Consumer checks: is this message still relevant given current state?

Challenge 4: Volume
  10,000 messages in DLQ.
  Replaying all at once floods the main queue.
  Same downstream system that caused the DLQ gets overwhelmed again.
  
  Solution: throttled replay — release N messages per minute.
  Monitor downstream health while replaying.
```

### Safe Replay Implementation

```csharp
public class DlqReplayer
{
    private readonly IAmazonSQS _sqs;
    private readonly string _dlqUrl;
    private readonly string _mainQueueUrl;

    public async Task ReplayAsync(
        ReplayOptions options,
        CancellationToken cancellationToken)
    {
        Console.WriteLine($"Starting DLQ replay with rate limit: {options.MessagesPerSecond}/sec");

        int replayed = 0;
        int skipped = 0;
        int errors = 0;

        var rateLimiter = new SemaphoreSlim(options.MessagesPerSecond);
        var refillTimer = new Timer(_ =>
        {
            try { rateLimiter.Release(options.MessagesPerSecond); }
            catch { /* semaphore full */ }
        }, null, TimeSpan.FromSeconds(1), TimeSpan.FromSeconds(1));

        while (!cancellationToken.IsCancellationRequested)
        {
            // Receive messages from DLQ
            var receiveResponse = await _sqs.ReceiveMessageAsync(new ReceiveMessageRequest
            {
                QueueUrl = _dlqUrl,
                MaxNumberOfMessages = 10,
                MessageAttributeNames = new List<string> { "All" },
                AttributeNames = new List<string> { "All" }
            });

            if (!receiveResponse.Messages.Any()) break; // DLQ empty

            foreach (var message in receiveResponse.Messages)
            {
                // Rate limiting
                await rateLimiter.WaitAsync(cancellationToken);

                // Optionally filter: only replay messages matching criteria
                if (options.Filter != null && !options.Filter(message))
                {
                    skipped++;
                    continue;
                }

                // Check if message is still relevant
                if (!await IsStillRelevantAsync(message))
                {
                    Console.WriteLine($"Skipping stale message: {message.MessageId}");
                    await _sqs.DeleteMessageAsync(_dlqUrl, message.ReceiptHandle);
                    skipped++;
                    continue;
                }

                try
                {
                    // Send back to main queue
                    await _sqs.SendMessageAsync(new SendMessageRequest
                    {
                        QueueUrl = _mainQueueUrl,
                        MessageBody = message.Body,
                        MessageAttributes = message.MessageAttributes,
                        // Add replay metadata
                        MessageSystemAttributes = new Dictionary<string, MessageSystemAttributeValue>
                        {
                            ["ReplayedAt"] = new() { StringValue = DateTime.UtcNow.ToString("O"), DataType = "String" },
                            ["OriginalMessageId"] = new() { StringValue = message.MessageId, DataType = "String" }
                        }
                    });

                    // Remove from DLQ after successful re-queue
                    await _sqs.DeleteMessageAsync(_dlqUrl, message.ReceiptHandle);
                    replayed++;

                    Console.WriteLine($"Replayed message {message.MessageId}");
                }
                catch (Exception ex)
                {
                    Console.Error.WriteLine($"Failed to replay {message.MessageId}: {ex.Message}");
                    errors++;
                }
            }
        }

        Console.WriteLine($"Replay complete: {replayed} replayed, {skipped} skipped, {errors} errors");
    }

    private async Task<bool> IsStillRelevantAsync(Message message)
    {
        // Check if the message is still relevant given current system state
        // Example: if this is a "send booking confirmation" message,
        // check that the booking still exists and confirmation wasn't already sent
        var payload = JsonSerializer.Deserialize<DlqMessagePayload>(message.Body);

        if (payload?.Type == "SendBookingConfirmation")
        {
            var booking = await _bookingService.GetAsync(payload.BookingId);
            if (booking == null) return false; // booking deleted, skip
            if (booking.ConfirmationSentAt.HasValue) return false; // already sent
        }

        return true;
    }
}

public class ReplayOptions
{
    public int MessagesPerSecond { get; set; } = 10; // throttle replay
    public Func<Message, bool>? Filter { get; set; }  // optional filter
    public int MaxMessages { get; set; } = int.MaxValue;
}
```

## DLQ Design Patterns

### Pattern 1: DLQ Per Consumer (Most Common)

```
booking-queue → booking-consumer → booking-dlq
email-queue   → email-consumer   → email-dlq
payment-queue → payment-consumer → payment-dlq

Each service has its own DLQ.
Failures isolated per service.
Each team owns their DLQ investigation.
```

### Pattern 2: Centralized DLQ with Metadata

```
All failed messages → central-dlq
  with metadata: {
    sourceQueue: "booking-queue",
    failureReason: "DatabaseException: connection timeout",
    failureCount: 3,
    originalTimestamp: ...,
    processorService: "booking-service"
  }

Central monitoring and investigation.
Replay routing: re-send to sourceQueue.
Single place to watch for system-wide health.
```

### Pattern 3: Tiered DLQ (Retry with Backoff)

```
Main Queue
  ↓ (failure)
Retry Queue 1 (delay: 1 minute)
  ↓ (failure)
Retry Queue 2 (delay: 10 minutes)
  ↓ (failure)
Retry Queue 3 (delay: 1 hour)
  ↓ (failure)
Dead Letter Queue (manual investigation)

Each tier uses SQS message delay or separate queue with delay.
Gives transient failures more time to resolve.
Only truly broken messages reach DLQ.
```

### Pattern 4: DLQ with Automated Triage

```
Messages arrive in DLQ
  ↓
Triage Lambda/Function:
  Pattern match error reason:
    "Connection timeout" → schedule retry in 5 min (likely transient)
    "InvalidSchema" → route to manual review (need code fix)
    "BookingNotFound" → mark as stale, discard
    "PaymentGatewayDown" → bulk replay when payment gateway recovers

Reduces manual investigation burden significantly.
```

### Question 1: Implement a message consumer with automatic DLQ routing

> "Build a consumer that retries failed messages and moves them to a DLQ after max retries."

```csharp
public class RetryingConsumer<T>
{
    private readonly IQueue<T> _mainQueue;
    private readonly IQueue<DeadLetterMessage<T>> _dlq;
    private readonly Func<T, Task> _processor;
    private readonly RetryPolicy _retryPolicy;
    private readonly ILogger _logger;

    public RetryingConsumer(
        IQueue<T> mainQueue,
        IQueue<DeadLetterMessage<T>> dlq,
        Func<T, Task> processor,
        RetryPolicy? retryPolicy = null,
        ILogger? logger = null)
    {
        _mainQueue = mainQueue;
        _dlq = dlq;
        _processor = processor;
        _retryPolicy = retryPolicy ?? new RetryPolicy();
        _logger = logger ?? NullLogger.Instance;
    }

    public async Task ConsumeAsync(CancellationToken cancellationToken)
    {
        await foreach (var envelope in _mainQueue.ConsumeAsync(cancellationToken))
        {
            await ProcessWithRetryAsync(envelope);
        }
    }

    private async Task ProcessWithRetryAsync(MessageEnvelope<T> envelope)
    {
        Exception? lastException = null;

        for (int attempt = 1; attempt <= _retryPolicy.MaxAttempts; attempt++)
        {
            try
            {
                _logger.LogDebug(
                    "Processing message {Id}, attempt {Attempt}/{Max}",
                    envelope.MessageId, attempt, _retryPolicy.MaxAttempts
                );

                await _processor(envelope.Message);

                // Success — acknowledge and we're done
                await _mainQueue.AcknowledgeAsync(envelope.ReceiptHandle);
                _logger.LogInformation(
                    "Message {Id} processed successfully on attempt {Attempt}",
                    envelope.MessageId, attempt
                );
                return;
            }
            catch (NonRetryableException ex)
            {
                // Don't retry — known unrecoverable error
                _logger.LogWarning(
                    "Message {Id} has non-retryable error: {Error}. Sending to DLQ immediately.",
                    envelope.MessageId, ex.Message
                );
                await SendToDlqAsync(envelope, ex, attempt);
                await _mainQueue.AcknowledgeAsync(envelope.ReceiptHandle);
                return;
            }
            catch (Exception ex)
            {
                lastException = ex;
                _logger.LogWarning(
                    "Message {Id} failed attempt {Attempt}: {Error}",
                    envelope.MessageId, attempt, ex.Message
                );

                if (attempt < _retryPolicy.MaxAttempts)
                {
                    // Exponential backoff with jitter
                    var delay = TimeSpan.FromMilliseconds(
                        _retryPolicy.BaseDelayMs * Math.Pow(2, attempt - 1)
                        + Random.Shared.Next(0, 1000)
                    );
                    _logger.LogDebug("Waiting {Delay}ms before retry", delay.TotalMilliseconds);
                    await Task.Delay(delay);
                }
            }
        }

        // All retries exhausted → DLQ
        _logger.LogError(
            "Message {Id} exceeded max retries ({Max}). Moving to DLQ. Last error: {Error}",
            envelope.MessageId, _retryPolicy.MaxAttempts, lastException?.Message
        );

        await SendToDlqAsync(envelope, lastException!, _retryPolicy.MaxAttempts);
        await _mainQueue.AcknowledgeAsync(envelope.ReceiptHandle);
    }

    private async Task SendToDlqAsync(
        MessageEnvelope<T> envelope,
        Exception exception,
        int attemptCount)
    {
        var dlqMessage = new DeadLetterMessage<T>
        {
            OriginalMessage = envelope.Message,
            OriginalMessageId = envelope.MessageId,
            FailureReason = exception.Message,
            FailureType = exception.GetType().Name,
            StackTrace = exception.StackTrace,
            AttemptCount = attemptCount,
            FirstFailedAt = envelope.FirstReceivedAt,
            LastFailedAt = DateTime.UtcNow,
            SourceQueue = _mainQueue.Name,
            ConsumerService = Environment.MachineName
        };

        await _dlq.SendAsync(dlqMessage);
    }
}

public class DeadLetterMessage<T>
{
    public T OriginalMessage { get; set; } = default!;
    public string OriginalMessageId { get; set; } = "";
    public string FailureReason { get; set; } = "";
    public string FailureType { get; set; } = "";
    public string? StackTrace { get; set; }
    public int AttemptCount { get; set; }
    public DateTime FirstFailedAt { get; set; }
    public DateTime LastFailedAt { get; set; }
    public string SourceQueue { get; set; } = "";
    public string ConsumerService { get; set; } = "";
}

public class RetryPolicy
{
    public int MaxAttempts { get; set; } = 3;
    public int BaseDelayMs { get; set; } = 1000;
    public int MaxDelayMs { get; set; } = 30000;
}

// Mark exception types that should never be retried
public class NonRetryableException : Exception
{
    public NonRetryableException(string message) : base(message) { }
}

// Usage
var consumer = new RetryingConsumer<BookingEvent>(
    mainQueue: bookingQueue,
    dlq: bookingDlq,
    processor: async booking =>
    {
        if (booking.UserId <= 0)
            throw new NonRetryableException($"Invalid userId: {booking.UserId}");

        await bookingService.ProcessAsync(booking);
    },
```

### Question 2: Implement DLQ inspection and replay tool

> "Build a tool to inspect messages in the DLQ and selectively replay them."

```csharp
public class DlqInspector<T>
{
    private readonly IQueue<DeadLetterMessage<T>> _dlq;
    private readonly IQueue<T> _mainQueue;

    public async Task<List<DlqMessageSummary<T>>> InspectAsync(
        int maxMessages = 100)
    {
        var messages = await _dlq.PeekAsync(maxMessages);

        return messages.Select(m => new DlqMessageSummary<T>
        {
            MessageId = m.OriginalMessageId,
            FailureReason = m.FailureReason,
            FailureType = m.FailureType,
            AttemptCount = m.AttemptCount,
            LastFailedAt = m.LastFailedAt,
            AgeHours = (DateTime.UtcNow - m.FirstFailedAt).TotalHours,
            Message = m.OriginalMessage,
            IsLikelyTransient = IsTransientFailure(m.FailureType, m.FailureReason),
            RecommendedAction = GetRecommendedAction(m)
        }).ToList();
    }

    public async Task<ReplayResult> ReplayAsync(
        string messageId,
        bool dryRun = false)
    {
        var message = await _dlq.FindAsync(messageId);
        if (message == null)
            return ReplayResult.NotFound(messageId);

        if (dryRun)
        {
            return ReplayResult.DryRun(messageId,
                $"Would replay: {JsonSerializer.Serialize(message.OriginalMessage)}");
        }

        try
        {
            await _mainQueue.SendAsync(message.OriginalMessage);
            await _dlq.DeleteAsync(messageId);

            return ReplayResult.Success(messageId);
        }
        catch (Exception ex)
        {
            return ReplayResult.Failed(messageId, ex.Message);
        }
    }

    public async Task<BulkReplayResult> ReplayByFilterAsync(
        Func<DeadLetterMessage<T>, bool> filter,
        int maxMessages = 1000,
        int messagesPerSecond = 10,
        bool dryRun = false)
    {
        var messages = await _dlq.PeekAsync(maxMessages);
        var matching = messages.Where(m => filter(m)).ToList();

        var result = new BulkReplayResult
        {
            TotalMatched = matching.Count,
            DryRun = dryRun
        };

        if (dryRun)
        {
            result.WouldReplay = matching.Count;
            return result;
        }

        var throttler = new SemaphoreSlim(messagesPerSecond);
        var refill = new Timer(_ =>
        {
            try { throttler.Release(messagesPerSecond); }
            catch { }
        }, null, 1000, 1000);

        foreach (var message in matching)
        {
            await throttler.WaitAsync();

            try
            {
                await _mainQueue.SendAsync(message.OriginalMessage);
                await _dlq.DeleteAsync(message.OriginalMessageId);
                result.Replayed++;
            }
            catch (Exception ex)
            {
                result.Failed++;
                result.Errors.Add($"{message.OriginalMessageId}: {ex.Message}");
            }
        }

        return result;
    }

    // Analyze DLQ to find patterns
    public async Task<DlqAnalysis> AnalyzeAsync()
    {
        var messages = await _dlq.PeekAsync(1000);

        var byErrorType = messages
            .GroupBy(m => m.FailureType)
            .Select(g => new { ErrorType = g.Key, Count = g.Count() })
            .OrderByDescending(g => g.Count)
            .ToList();

        var oldestMessage = messages.MinBy(m => m.FirstFailedAt);
        var avgAttempts = messages.Average(m => m.AttemptCount);

        var transientCount = messages.Count(m =>
            IsTransientFailure(m.FailureType, m.FailureReason));

        return new DlqAnalysis
        {
            TotalMessages = messages.Count,
            OldestMessageAge = oldestMessage != null
                ? DateTime.UtcNow - oldestMessage.FirstFailedAt
                : TimeSpan.Zero,
            AverageAttemptCount = avgAttempts,
            TransientFailureCount = transientCount,
            PermanentFailureCount = messages.Count - transientCount,
            ErrorBreakdown = byErrorType.ToDictionary(g => g.ErrorType, g => g.Count),
            RecommendedAction = GetBulkRecommendation(messages)
        };
    }

    private bool IsTransientFailure(string errorType, string reason)
    {
        var transientPatterns = new[]
        {
            "timeout", "connection", "unavailable", "503", "429",
            "throttl", "temporary", "transient"
        };

        var combined = $"{errorType} {reason}".ToLowerInvariant();
        return transientPatterns.Any(p => combined.Contains(p));
    }

    private string GetRecommendedAction(DeadLetterMessage<T> message)
    {
        if (IsTransientFailure(message.FailureType, message.FailureReason))
            return "REPLAY - likely transient failure";
        if (message.FailureType.Contains("Schema") || message.FailureType.Contains("Deserialization"))
            return "FIX CODE - schema mismatch, update consumer then replay";
        if (message.FailureType.Contains("NotFound"))
            return "INVESTIGATE - resource may have been deleted";
        return "MANUAL REVIEW";
    }

    private string GetBulkRecommendation(List<DeadLetterMessage<T>> messages)
    {
        int transient = messages.Count(m => IsTransientFailure(m.FailureType, m.FailureReason));
        double transientPct = (double)transient / messages.Count;

        if (transientPct > 0.8)
            return "Majority transient: safe to bulk replay after confirming root cause resolved";
        if (transientPct < 0.2)
            return "Majority permanent: requires code fix before replay";
        return "Mixed failures: review error breakdown, fix permanent failures first";
    }
}
```

### Question 3: Implement the outbox pattern to prevent DLQ overflow

> "Prevent messages from going to DLQ by ensuring they're only published when the business operation succeeds."

```csharp
// The problem this solves:
// If we publish to queue THEN write to DB, DB write can fail
// → booking exists in queue but not in DB
// → consumer tries to process → booking not found → DLQ

// If we write to DB THEN publish to queue, publish can fail
// → booking in DB but no downstream notification
// → silent failure

// Outbox pattern: atomic DB write + queue publish
public class BookingServiceWithOutbox
{
    private readonly DbContext _db;

    public async Task<string> CreateBookingAsync(BookingRequest request)
    {
        using var transaction = await _db.Database.BeginTransactionAsync();
        try
        {
            // Step 1: Create the booking
            var booking = new Booking
            {
                Id = Guid.NewGuid().ToString(),
                UserId = request.UserId,
                EventId = request.EventId,
                SeatId = request.SeatId,
                Amount = request.Amount,
                CreatedAt = DateTime.UtcNow
            };
            _db.Bookings.Add(booking);

            // Step 2: Write outbox record IN SAME TRANSACTION
            _db.OutboxMessages.Add(new OutboxMessage
            {
                Id = Guid.NewGuid().ToString(),
                Topic = "bookings.created",
                Payload = JsonSerializer.Serialize(new BookingCreatedEvent
                {
                    BookingId = booking.Id,
                    UserId = booking.UserId,
                    EventId = booking.EventId,
                    Amount = booking.Amount
                }),
                Status = "pending",
                CreatedAt = DateTime.UtcNow
            });

            // Both committed atomically
            // If either fails, neither persists
            await _db.SaveChangesAsync();
            await transaction.CommitAsync();

            return booking.Id;
        }
        catch
        {
            await transaction.RollbackAsync();
            throw;
        }
    }
}

// Background worker publishes outbox messages
public class OutboxPublisher : BackgroundService
{
    private readonly DbContext _db;
    private readonly IMessageQueue _queue;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            // Find unprocessed outbox messages
            var pending = await _db.OutboxMessages
                .Where(m => m.Status == "pending" && m.RetryCount < 5)
                .OrderBy(m => m.CreatedAt)
                .Take(50)
                .ToListAsync(stoppingToken);

            foreach (var message in pending)
            {
                try
                {
                    await _queue.PublishAsync(message.Topic, message.Payload);

                    message.Status = "published";
                    message.PublishedAt = DateTime.UtcNow;
                }
                catch (Exception ex)
                {
                    message.RetryCount++;
                    message.LastError = ex.Message;

                    if (message.RetryCount >= 5)
                    {
                        message.Status = "failed";
                        // This is now our "DLQ" — failed outbox messages
                        // Alert: message cannot be published after 5 tries
                        await AlertAsync(message);
                    }
                }

                await _db.SaveChangesAsync(stoppingToken);
            }

            await Task.Delay(TimeSpan.FromSeconds(1), stoppingToken);
        }
    }

    private async Task AlertAsync(OutboxMessage message)
    {
        Console.Error.WriteLine(
            $"[CRITICAL] Outbox message {message.Id} failed after 5 attempts. " +
            $"Topic: {message.Topic}. Last error: {message.LastError}"
        );
    }
}
```

**The key insight:** With the outbox pattern, messages only enter the queue when the business operation definitely succeeded. This dramatically reduces the number of messages that end up in DLQ, because consumers never encounter "booking not found" or similar inconsistencies.

## One-Line Summary for Interviews

> **A Dead Letter Queue is a secondary queue where messages are moved after failing to be processed successfully after a configurable number of retries — it serves three purposes: preventing the main processing pipeline from being blocked by poison messages, preserving failed messages so no work is permanently lost, and providing a place to investigate root causes and replay messages after fixing them; messages end up in DLQs due to transient failures (timeouts, temporary unavailability — safe to replay), permanent failures (schema mismatches, invalid data — require code fixes), or business logic violations (missing resources, race conditions — require manual decisions); the critical engineering practices are comprehensive DLQ monitoring with alerts (a silently filling DLQ is a business emergency), idempotent consumers so replay is always safe, and thorough runbooks for each failure category so engineers know exactly what to do when they get the alert at 2am.**
