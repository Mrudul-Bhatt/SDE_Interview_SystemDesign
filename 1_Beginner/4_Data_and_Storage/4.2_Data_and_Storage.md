# SQL vs NoSQL — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine you're organizing your life's paperwork.

**SQL is a filing cabinet with strict folders.** Every document goes into a predefined folder. Every folder has the same structure — name, date, amount, category. You can instantly answer "show me all receipts from March over $100" because everything is organized consistently. But if you suddenly need to store a receipt that has 47 line items while others have 2 — the cabinet struggles. You have to redesign the entire folder system.

**NoSQL is a giant flexible bin.** You throw in receipts, sticky notes, photos, audio recordings, whatever. Each item can look completely different. Adding a new type of document is trivial — just throw it in. But asking "show me all receipts from March over $100" means digging through everything, because there's no guaranteed structure to query against.

Neither is better. Your choice depends entirely on **what questions you need to answer and how your data is shaped**.

## What SQL Actually Gives You

SQL databases (PostgreSQL, MySQL, SQL Server) are **relational** — data is organized into tables with rows and columns, and tables relate to each other via foreign keys.

```
events table:
┌────┬──────────────────┬─────────────┬──────────┐
│ id │ name             │ venue_id    │ date     │
├────┼──────────────────┼─────────────┼──────────┤
│  1 │ Taylor Swift     │     12      │ 2025-08  │
│  2 │ Coldplay         │     15      │ 2025-09  │
└────┴──────────────────┴─────────────┴──────────┘

bookings table:
┌────┬──────────┬──────────┬──────────┬─────────┐
│ id │ user_id  │ event_id │ seat     │ price   │
├────┼──────────┼──────────┼──────────┼─────────┤
│  1 │   42     │    1     │   A5     │  250.00 │
│  2 │   87     │    1     │   A6     │  250.00 │
└────┴──────────┴──────────┴──────────┴─────────┘
```

The power: **JOIN**. Answer questions that span multiple tables in one query.

```sql
-- "Show me all bookings for events in August, 
--  with user email and venue name"
SELECT 
    u.email,
    e.name AS event_name,
    v.name AS venue_name,
    b.seat,
    b.price
FROM bookings b
JOIN users u    ON b.user_id  = u.id
JOIN events e   ON b.event_id = e.id
JOIN venues v   ON e.venue_id = v.id
WHERE e.date >= '2025-08-01' 
  AND e.date <  '2025-09-01';
```

One query. Four tables. Exact answer. This is SQL's superpower.

### ACID — The Core Guarantee

This is the most important concept in this topic for interviews.

**A — Atomicity**
A transaction either completes entirely or not at all. No half-states.

```sql
BEGIN TRANSACTION;
  UPDATE seats    SET status = 'booked'  WHERE id = 'A5';
  INSERT INTO bookings (user_id, seat_id, price) VALUES (42, 'A5', 250);
  UPDATE inventory SET available = available - 1 WHERE event_id = 1;
COMMIT;
-- If ANY of these fail, ALL of them roll back.
-- You never end up with a booked seat and no booking record.
```

**C — Consistency**
Data always moves from one valid state to another. Constraints are enforced.

```sql
ALTER TABLE bookings
    ADD CONSTRAINT fk_event
    FOREIGN KEY (event_id) REFERENCES events(id);
-- Can't create a booking for an event that doesn't exist.
-- Database enforces this. Always.
```

**I — Isolation**
Concurrent transactions don't see each other's intermediate states.

```sql
-- User A and User B both try to book seat A5 simultaneously.
-- Isolation guarantees exactly one succeeds.
-- The other gets a conflict error, not a corrupted half-booking.
```

**D — Durability**
Once committed, data survives crashes. Written to disk, not just memory.

```
Power cut the millisecond after COMMIT?
Your booking still exists when the server restarts.
```

**For AXS:** ACID is non-negotiable for bookings and payments. Losing a payment transaction or double-booking a seat is catastrophic. This alone justifies SQL for the transactional core.

## What NoSQL Actually Gives You

NoSQL is an umbrella term covering four fundamentally different data models. Treating "NoSQL" as one thing is a common interview mistake.

### Type 1: Document Stores (MongoDB, Firestore, DynamoDB)

Store JSON-like documents. Each document can have different fields.

```json
// One user document — everything about this user in one place
{
  "_id": "user-42",
  "name": "Mrudul",
  "email": "mrudul@email.com",
  "preferences": {
    "genres": ["rock", "pop"],
    "cities": ["Mumbai", "Delhi"],
    "notifyBeforeDays": 7
  },
  "paymentMethods": [
    { "type": "card", "last4": "4242", "default": true },
    { "type": "upi",  "id": "mrudul@upi" }
  ],
  "bookingHistory": ["booking-101", "booking-102", "booking-103"]
}
```

**No JOIN needed** — all related data embedded in one document. One read, complete picture.

**When it breaks:** You need to query across documents. "Find all users who prefer rock AND have booked in the last 30 days AND are in Mumbai." Now you're scanning millions of documents. SQL would answer this in milliseconds with indexes.

### Type 2: Key-Value Stores (Redis, DynamoDB, Memcached)

The simplest possible model. A key maps to a value. Nothing else.

```
"session:user-42"       → { userId: 42, role: "premium", expires: ... }
"lock:seat-A5:event-1"  → "user-42"  (with TTL of 10 minutes)
"rate:ip:192.168.1.1"   → "847"      (requests this minute)
"cache:event:123"       → { ...full event JSON... }
```

Blazing fast. O(1) lookup. No query language needed. No schema.

**When it breaks:** You need to query by value, not key. "Find all sessions expiring in the next hour." Key-value has no answer — you'd need to know every key, which you don't.

### Type 3: Wide-Column Stores (Cassandra, HBase, BigTable)

Like a table, but each row can have different columns. Optimized for **write-heavy, time-series, massive scale**.

```
Cassandra table — event_views by user and time:

partition key → user_id | timestamp        | event_id | duration_ms
                ─────────────────────────────────────────────────
                  42     | 2025-08-01 14:00 |   123    |   45000
                  42     | 2025-08-01 14:45 |   456    |   12000
                  42     | 2025-08-02 09:00 |   123    |   90000
                  87     | 2025-08-01 15:00 |   789    |   30000
```

All rows for user 42 live on the same node (partition key). Reading "all events user 42 watched in August" = single node read, extremely fast.

**When it breaks:** You need to query by a non-partition key column. "Find all views longer than 60 seconds across all users." Cassandra scans every partition = full table scan = terrible performance.

### Type 4: Graph Databases (Neo4j, Amazon Neptune)

Data is nodes and edges. Optimized for **relationship traversal**.

```
(Mrudul)-[:FRIENDS_WITH]->(Priya)
(Priya)-[:ATTENDED]->(Taylor Swift Concert)
(Mrudul)-[:MIGHT_LIKE]->(Taylor Swift Concert)

Query: "Find concerts attended by friends of Mrudul that Mrudul hasn't attended"
→ Trivial in graph DB (2 hops)
→ Nightmare in SQL (self-joins, recursive CTEs)
```

## The CAP Theorem — Why NoSQL Exists

To understand why NoSQL databases make the tradeoffs they do, you need CAP theorem. This is **always asked** when SQL vs NoSQL comes up.

In a distributed system, you can only guarantee **two of three**:

```
         Consistency
         (every read sees
          the latest write)
              /\
             /  \
            /    \
           /      \
          /________\
   Availability    Partition Tolerance
   (every request  (system works even if
    gets a response) nodes can't talk to each other)
```

**Partition Tolerance** is not optional in real distributed systems. Networks fail. Nodes get separated. You must handle it. So the real choice is:

**CP (Consistent + Partition Tolerant):**
When a network partition occurs, the system refuses to answer rather than risk returning stale data.
→ HBase, Zookeeper, traditional SQL clusters
→ "If I can't guarantee the answer is correct, I'd rather say nothing."

**AP (Available + Partition Tolerant):**
When a network partition occurs, the system still responds — but might return stale data.
→ Cassandra, DynamoDB, CouchDB
→ "I'd rather give you a possibly-stale answer than no answer."

```
AXS example:

Booking a ticket (CP behavior required):
  User pays $250. System must be consistent.
  If partition occurs → fail the transaction.
  Stale data here = double booking = fraud.

Seat availability display (AP behavior acceptable):
  User sees "143 seats available" when it's actually 141.
  A 2-seat discrepancy for 200ms is fine.
  System should stay available during partitions.
  Stale data here = minor visual glitch = acceptable.
```

**SQL databases are CA** (Consistent + Available) — they assume no partitions, which works for a single node or tightly coupled cluster. This is why they're hard to scale horizontally.

**Most NoSQL databases are AP** — they sacrifice consistency for availability and partition tolerance, which is why they scale so well horizontally.

## The Real Decision Framework

Stop thinking "SQL vs NoSQL" and start thinking about these five questions:

### Question 1: Is your data structured or variable?

```
Structured, uniform schema → SQL
  Users, events, bookings — all have the same fields.
  Schema enforced at DB level catches bugs early.

Variable, heterogeneous schema → Document NoSQL
  Product catalog — a t-shirt has size/color,
  a laptop has RAM/CPU, a book has ISBN/author.
  Forcing these into one table = many nullable columns.
```

### Question 2: Do you need transactions across multiple entities?

```
Yes, money or inventory involved → SQL
  "Charge card AND create booking AND decrement seat count"
  must all succeed or all fail. ACID required.

No, operations are self-contained → NoSQL fine
  "Log this user event" — if one log is lost, nobody cares.
  "Cache this response" — cache miss just hits DB again.
```

### Question 3: What queries do you need to run?

```
Complex, multi-entity queries → SQL
  "Revenue by venue by month for premium users"
  SQL answers this. Document DB requires multiple queries + app-layer join.

Simple lookup by known key → Key-Value / Document
  "Get user profile for user-42"
  One key, one document. NoSQL is faster here.

Time-series, append-only, massive write volume → Cassandra
  "Log every seat view for every event in real time"
  Cassandra handles 1M writes/second. SQL cannot.

Relationship traversal → Graph DB
  "Friends of friends who attended similar events"
  SQL requires 5 self-joins. Graph DB = 2 hops.
```

### Question 4: How much will this data grow?

```
Moderate, predictable growth → SQL scales fine
  Vertical scaling + read replicas handles most apps.
  PostgreSQL comfortably handles billions of rows with good indexing.

Massive, unpredictable growth → NoSQL
  Cassandra adds nodes with zero downtime, linear scaling.
  DynamoDB scales automatically, no capacity planning.
```

### Question 5: Does your team need flexibility to change the schema?

```
Stable, well-understood domain → SQL (schema is a feature, not a bug)
  Schema changes are controlled, migrations are trackable.

Rapidly changing requirements → Document NoSQL
  Add a new field to a document? Just add it.
  No ALTER TABLE, no migration scripts, no downtime.
```

## Common System Design Scenarios

### "Design Twitter / Social Feed"

```
Users, follows, tweets → different answers

Users table → SQL (structured, ACID for account ops)
Tweets → Document DB (variable media attachments, metadata)
Follow graph → Graph DB or adjacency list in SQL
Feed (pre-computed per user) → Redis sorted set (score = timestamp)
Tweet search → Elasticsearch (full-text search)
Analytics (impressions, clicks) → Cassandra (time-series, high write)
```

### "Design Netflix"

```
User accounts, billing → SQL (ACID required)
Video metadata (title, description, cast) → Document DB (schema varies by content type)
Watch history → Cassandra (time-series, massive write volume)
Recommendations → Graph DB or ML feature store
Search → Elasticsearch
Video files → Object storage (S3), not a DB
CDN cache → Redis (video metadata caching)
```

### "Design AXS Ticketing"

```
Events, venues, bookings, payments → SQL (PostgreSQL)
  Reason: ACID mandatory. Double-booking = fraud.
  Complex queries: revenue reports, seat analytics.
  
Seat locks during checkout → Redis (key-value, TTL)
  "seat:A5:event-1" → "user-42" with 10-minute TTL

User sessions → Redis (key-value)
  Fast lookup, automatic expiry

Event search → Elasticsearch
  Full-text search, faceted filtering (genre, city, date, price)

Analytics / reporting → Cassandra or data warehouse
  "Events viewed per user per day" — time-series, write-heavy

Notifications (preferences, history) → Document DB
  Schema varies: email prefs, push settings, SMS settings
```

## Coding Questions

SQL vs NoSQL is mostly architectural, but it generates direct coding questions — especially SQL query writing.

### Question 1: SQL schema design *(extremely common)*

> "Design the database schema for a ticket booking system."

```sql
CREATE TABLE venues (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(255) NOT NULL,
    city        VARCHAR(100) NOT NULL,
    capacity    INT NOT NULL
);

CREATE TABLE events (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(255) NOT NULL,
    venue_id    INT REFERENCES venues(id),
    event_date  TIMESTAMPTZ NOT NULL,
    total_seats INT NOT NULL,
    status      VARCHAR(20) DEFAULT 'active'
                CHECK (status IN ('active', 'cancelled', 'sold_out'))
);

CREATE TABLE seats (
    id          SERIAL PRIMARY KEY,
    event_id    INT REFERENCES events(id),
    row         CHAR(1) NOT NULL,
    number      INT NOT NULL,
    tier        VARCHAR(50) NOT NULL, -- 'VIP', 'General', 'Standing'
    price       DECIMAL(10,2) NOT NULL,
    status      VARCHAR(20) DEFAULT 'available'
                CHECK (status IN ('available', 'locked', 'booked')),
    UNIQUE (event_id, row, number)  -- no duplicate seats per event
);

CREATE TABLE users (
    id          SERIAL PRIMARY KEY,
    email       VARCHAR(255) UNIQUE NOT NULL,
    name        VARCHAR(255) NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE bookings (
    id              SERIAL PRIMARY KEY,
    user_id         INT REFERENCES users(id),
    event_id        INT REFERENCES events(id),
    seat_id         INT REFERENCES seats(id) UNIQUE, -- one booking per seat
    amount_paid     DECIMAL(10,2) NOT NULL,
    status          VARCHAR(20) DEFAULT 'confirmed'
                    CHECK (status IN ('confirmed', 'cancelled', 'refunded')),
    booked_at       TIMESTAMPTZ DEFAULT NOW()
);

-- Indexes for common queries
CREATE INDEX idx_events_date      ON events(event_date);
CREATE INDEX idx_seats_event      ON seats(event_id, status);
CREATE INDEX idx_bookings_user    ON bookings(user_id);
CREATE INDEX idx_bookings_event   ON bookings(event_id);
```

**What to talk about:**
- Foreign keys enforce referential integrity
- `UNIQUE` on seat prevents double booking at DB level
- `CHECK` constraints enforce valid states
- Indexes on columns you query/filter by

### Question 2: Write complex SQL queries *(asked at every company)*

> "Find the top 5 events by revenue this month, with venue name and seats remaining."

```sql
SELECT
    e.name                                          AS event_name,
    v.name                                          AS venue_name,
    COUNT(b.id)                                     AS tickets_sold,
    SUM(b.amount_paid)                              AS total_revenue,
    e.total_seats - COUNT(b.id)                     AS seats_remaining,
    ROUND(COUNT(b.id)::decimal / e.total_seats * 100, 1) AS occupancy_pct
FROM events e
JOIN venues v     ON e.venue_id = v.id
LEFT JOIN bookings b
    ON b.event_id = e.id
    AND b.status  = 'confirmed'
WHERE e.event_date >= DATE_TRUNC('month', NOW())
  AND e.event_date <  DATE_TRUNC('month', NOW()) + INTERVAL '1 month'
  AND e.status = 'active'
GROUP BY e.id, e.name, v.name, e.total_seats
ORDER BY total_revenue DESC
LIMIT 5;
```

### Question 3: Prevent double booking with SQL *(concurrency question)*

> "Two users try to book the same seat simultaneously. How does SQL prevent this?"

```sql
-- Optimistic locking approach
BEGIN TRANSACTION;

-- Step 1: Read current status
SELECT id, status
FROM seats
WHERE id = 101
FOR UPDATE;  -- ← acquires row-level lock, blocks other transactions

-- Step 2: Check availability (inside transaction)
-- If status != 'available', rollback and tell user

-- Step 3: Book it atomically
UPDATE seats
SET status = 'booked'
WHERE id = 101
  AND status = 'available';  -- ← double-check in the UPDATE itself

-- If 0 rows affected → another transaction got there first
GET DIAGNOSTICS rows_affected = ROW_COUNT;

INSERT INTO bookings (user_id, seat_id, amount_paid)
VALUES (42, 101, 250.00);

COMMIT;
```

**The `AND status = 'available'` in the UPDATE** is the critical line. Even if two transactions both read "available" before either writes, only one UPDATE will match the condition. The other gets 0 rows affected — clean, safe rejection.

### Question 5: Redis data structure design *(key-value schema design)*

> "Design Redis keys for a seat locking system during flash sale."

```csharp
public class SeatLockService
{
    private readonly IDatabase _redis;

    // Lock a seat for a user with 10-minute TTL
    public async Task<bool> TryLockSeat(int eventId, string seatId, int userId)
    {
        string key = $"lock:event:{eventId}:seat:{seatId}";
        string value = userId.ToString();

        // SetIfNotExists (NX) — atomic, only sets if key doesn't exist
        // Expiry (EX) — auto-releases after 10 minutes
        bool acquired = await _redis.StringSetAsync(
            key,
            value,
            TimeSpan.FromMinutes(10),
            When.NotExists  // ← this is the atomicity guarantee
        );

        return acquired; // false = seat already locked by someone else
    }

    // Check who holds a lock
    public async Task<int?> GetLockHolder(int eventId, string seatId)
    {
        string key = $"lock:event:{eventId}:seat:{seatId}";
        var value = await _redis.StringGetAsync(key);
        return value.HasValue ? int.Parse(value) : null;
    }

    // User releases lock (checkout abandoned)
    public async Task ReleaseLock(int eventId, string seatId, int userId)
    {
        string key = $"lock:event:{eventId}:seat:{seatId}";

        // Only release if this user holds the lock
        // Use Lua script for atomic check-and-delete
        const string lua = @"
            if redis.call('get', KEYS[1]) == ARGV[1] then
                return redis.call('del', KEYS[1])
            else
                return 0
            end";

        await _redis.ScriptEvaluateAsync(lua,
            new RedisKey[] { key },
            new RedisValue[] { userId.ToString() }
        );
    }

    // Track all locked seats for an event (for seat map display)
    public async Task<HashSet<string>> GetLockedSeats(int eventId)
    {
        // Use Redis SET to track which seats are locked
        string setKey = $"locked_seats:event:{eventId}";
        var members = await _redis.SetMembersAsync(setKey);
        return members.Select(m => m.ToString()).ToHashSet();
    }
}
```

**Key design insight:** `When.NotExists` makes the lock atomic — no race condition possible. Two users hitting this simultaneously: exactly one gets `true`, the other gets `false`. This is the Redis equivalent of SQL's `FOR UPDATE`.

## One-Line Summary for Interviews

> **Choose SQL when you need ACID transactions, complex multi-table queries, or a well-structured domain where schema enforcement catches bugs — it's non-negotiable for payments and bookings; choose NoSQL when you need horizontal scale beyond what one SQL server handles, flexible schema for heterogeneous data, or specialized access patterns like key-value caching in Redis, time-series in Cassandra, or relationship traversal in graph databases — and the most mature systems use both, with SQL as the transactional source of truth and NoSQL for caching, search, analytics, and anywhere eventual consistency is acceptable.**


# Database Indexing — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine a library with 1 million books, completely unsorted. You want "The Great Gatsby." 

Without an index — you walk every single aisle, check every spine. Could take hours.

With a card catalog (index) — you look up "Fitzgerald" alphabetically, get "Shelf 4B, Row 3." Walk directly there. Seconds.

**That card catalog is a database index.** A separate, sorted data structure that lets the database jump directly to your data instead of scanning every row.

But now imagine the librarian must **update the card catalog every time a book arrives, moves, or leaves**. With 5 catalogs organized by title, author, genre, year, and color — every book transaction requires updating all 5 catalogs. The librarian is now spending more time updating catalogs than helping people find books.

**That's over-indexing.** The trade-off at the heart of this topic.

---

## What a Table Scan Actually Costs

Before indexing makes sense, feel the pain it solves.

```sql
-- Table: bookings — 50 million rows
SELECT * FROM bookings WHERE user_id = 42;
```

Without an index, the database does a **full table scan**:

```
Page 1:   read 1000 rows, check each user_id... none match
Page 2:   read 1000 rows, check each user_id... none match
...
Page 50,000: read 1000 rows, check each user_id... found 3 matches!
```

50 million row reads. On spinning disk, each page read takes ~10ms. That's minutes. Even on SSD, it's seconds. For a web request, this is catastrophic.

**With an index on user_id:**
```
Index lookup: user_id = 42 → row pointers [row 1,204,301], [row 8,442,190], [row 31,004,772]
Fetch those 3 rows directly.
```

3 reads instead of 50 million. That's the value of an index.

---

## How a B-Tree Index Works — The Real Internals

Almost every SQL database uses a **B-Tree (Balanced Tree)** as the default index structure. Understanding this earns serious interview points.

### The structure

```
                    [Root Node]
                   [30 | 60 | 90]
                  /    |    |    \
         [10|20] [35|50] [70|80] [95|99]
         /  |  \   ...
      [8] [15] [25]  ← Leaf nodes (contain actual row pointers)
```

Every value is sorted. Every leaf node contains:
- The indexed value (e.g., `user_id = 42`)
- A pointer to the actual row on disk

### Finding `user_id = 42`

```
Start at root: [30 | 60 | 90]
  42 > 30, 42 < 60 → go to middle child

Middle child: [35 | 50]
  42 > 35, 42 < 50 → go to left child

Leaf: [36, 38, 42, 44]
  Found 42 → row pointer → fetch actual row
```

3 node reads to find any value among millions. This is O(log N) — the B-Tree guarantee.

### Why it's balanced

If one branch grew much deeper than others, some lookups would be much slower. B-Trees automatically rebalance when rows are inserted or deleted — maintaining consistent O(log N) lookup regardless of data distribution.

### Leaf node linking

```
[Leaf 1: 1–100] ←→ [Leaf 2: 101–200] ←→ [Leaf 3: 201–300]
```

Leaf nodes are **doubly linked**. This makes range queries efficient:

```sql
WHERE user_id BETWEEN 100 AND 500
```
Find the leaf containing 100 → follow links rightward → stop at 500.
No need to go back to root for each value. This is why B-Trees dominate.

## Types of Indexes

### 1. Single-Column Index

```sql
CREATE INDEX idx_bookings_user_id ON bookings(user_id);
```

Speeds up queries filtering or sorting by `user_id`. The most common index.

### 2. Composite Index (Multi-Column)

```sql
CREATE INDEX idx_bookings_user_event ON bookings(user_id, event_id);
```

The **column order matters enormously** — this is a top interview point.

```
Index stores rows sorted by: (user_id, then event_id within each user)

user_id=1, event_id=10
user_id=1, event_id=20
user_id=1, event_id=30
user_id=2, event_id=5
user_id=2, event_id=15
user_id=3, event_id=8
```

**Queries this index helps:**
```sql
WHERE user_id = 42                          ✅ (leftmost column)
WHERE user_id = 42 AND event_id = 100       ✅ (both columns)
WHERE user_id = 42 AND event_id > 50        ✅ (range on second column)
```

**Queries this index does NOT help:**
```sql
WHERE event_id = 100                        ❌ (skipped leftmost column)
WHERE event_id = 100 AND user_id = 42       ❌ (wrong order in index)
```

This is the **leftmost prefix rule** — the index is only usable if the query starts with the leftmost indexed column(s).

Think of it like a phone book sorted by last name, then first name. "Find all Smiths" → easy. "Find all Johns" → full scan, the book isn't sorted by first name.

### 3. Unique Index

```sql
CREATE UNIQUE INDEX idx_users_email ON users(email);
```

Enforces uniqueness at the database level AND provides fast lookup. Two guarantees in one structure. Primary keys automatically get a unique index.

### 4. Partial Index

Only indexes rows matching a condition. Smaller, faster.

```sql
-- Only index active bookings — cancelled ones rarely queried
CREATE INDEX idx_active_bookings 
ON bookings(user_id) 
WHERE status = 'confirmed';
```

If 80% of your bookings are cancelled, this index is 5x smaller than a full index. Faster to scan, faster to update, less disk space.

### 5. Covering Index

An index that contains all columns a query needs — so the database never touches the actual table rows.

```sql
-- Query only needs user_id, event_id, and status
SELECT user_id, event_id, status 
FROM bookings 
WHERE user_id = 42;

-- Create index covering all three columns
CREATE INDEX idx_covering 
ON bookings(user_id, event_id, status);
```

```
Query execution:
  1. Look up user_id=42 in index
  2. Index entry contains event_id and status already
  3. Return data directly from index
  4. Never touch the actual bookings table rows ← "index-only scan"
```

No table access at all. Extremely fast for read-heavy tables. The tradeoff: index is larger because it stores more data per entry.

### 6. Full-Text Index

```sql
-- Standard index won't help for substring search
WHERE description LIKE '%Taylor%'  ← full scan, index useless

-- Full-text index tokenizes and indexes individual words
CREATE INDEX idx_events_search ON events USING GIN(to_tsvector('english', description));

-- Now this is fast
WHERE to_tsvector('english', description) @@ to_tsquery('Taylor');
```

Used for search features. Elasticsearch is essentially a giant distributed full-text index.

### 7. Hash Index

```sql
CREATE INDEX idx_hash ON sessions USING HASH(session_token);
```

Stores hash(value) → row pointer. O(1) lookup — faster than B-Tree for exact equality lookups.

```
Hash index:
  hash("abc123token") = 8472 → row pointer
  Lookup: compute hash, jump directly to bucket

B-Tree:
  Traverse 3-4 nodes to find value
```

**BUT** — hash indexes are useless for:
```sql
WHERE session_token > 'abc'     ❌ (no ordering, can't do range)
WHERE session_token LIKE 'abc%' ❌ (no prefix matching)
ORDER BY session_token          ❌ (not sorted)
```

Use hash indexes only for pure equality lookups on high-cardinality columns.

## The Over-Indexing Problem — The Core Trade-off

Every index makes reads faster and **writes slower**. This is the fundamental tension.

### What happens on every INSERT

```sql
INSERT INTO bookings (user_id, event_id, seat_id, amount_paid)
VALUES (42, 123, 'A5', 250.00);
```

If bookings has 5 indexes:

```
Step 1: Write row to bookings table
Step 2: Update idx_bookings_user_id      ← B-Tree insertion, possible rebalance
Step 3: Update idx_bookings_event_id     ← B-Tree insertion, possible rebalance
Step 4: Update idx_bookings_created_at   ← B-Tree insertion, possible rebalance
Step 5: Update idx_bookings_status       ← B-Tree insertion, possible rebalance
Step 6: Update idx_bookings_composite    ← B-Tree insertion, possible rebalance
```

1 logical write became 6 physical operations. During a Taylor Swift flash sale with 50,000 bookings/second — this is 300,000 index update operations per second. Under this load, write latency degrades badly.

### What happens on every UPDATE

```sql
UPDATE bookings SET status = 'cancelled' WHERE id = 101;
```

For every index that includes `status`:
```
Old entry in index (status='confirmed') → DELETE from B-Tree
New entry in index (status='cancelled') → INSERT into B-Tree
```

An update is actually a delete + insert in every affected index. Two B-Tree operations per index per update.

### What happens on every DELETE

```sql
DELETE FROM bookings WHERE id = 101;
```

Every index entry pointing to this row must be removed. 5 indexes = 5 B-Tree deletions.

### Disk space cost

Each index is a full copy of the indexed data plus pointers. A table with 5 indexes might use 3x the disk space of the raw data. On large tables (hundreds of GB), this is significant infrastructure cost.

### The over-indexing smell

```sql
-- Someone added an index for every column "just in case"
CREATE INDEX idx1 ON bookings(user_id);
CREATE INDEX idx2 ON bookings(event_id);
CREATE INDEX idx3 ON bookings(seat_id);
CREATE INDEX idx4 ON bookings(status);
CREATE INDEX idx5 ON bookings(amount_paid);
CREATE INDEX idx6 ON bookings(created_at);
CREATE INDEX idx7 ON bookings(updated_at);
CREATE INDEX idx8 ON bookings(payment_method);
CREATE INDEX idx9 ON bookings(promo_code);
```

During a flash sale:
- Reads: marginally faster (most of these are never actually used)
- Writes: 10x the I/O per booking
- Result: write throughput collapses under load

## How to Find Unused or Bad Indexes

### PostgreSQL — find unused indexes

```sql
SELECT
    indexrelname AS index_name,
    relname      AS table_name,
    idx_scan     AS times_used,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
JOIN pg_statio_user_indexes USING (indexrelid)
WHERE idx_scan < 50  -- used fewer than 50 times since last reset
ORDER BY pg_relation_size(indexrelid) DESC;
```

An index that's 2GB and used 3 times since last stats reset is pure overhead. Drop it.

### PostgreSQL — EXPLAIN ANALYZE

The most important tool for index debugging:

```sql
EXPLAIN ANALYZE
SELECT * FROM bookings WHERE user_id = 42 AND status = 'confirmed';
```

```
Output:
Seq Scan on bookings (cost=0.00..89432.10 rows=3 width=64)
  (actual time=1823.442..1823.447 rows=3 loops=1)
  Filter: (user_id = 42 AND status = 'confirmed')
  Rows Removed by Filter: 4999997
Planning Time: 0.3 ms
Execution Time: 1823.5 ms  ← 1.8 SECONDS. Full table scan happening.
```

"Seq Scan" = sequential scan = no index used = problem.

```sql
-- After adding index:
EXPLAIN ANALYZE
SELECT * FROM bookings WHERE user_id = 42 AND status = 'confirmed';
```

```
Output:
Index Scan using idx_bookings_user_status on bookings
  (cost=0.56..12.63 rows=3 width=64)
  (actual time=0.043..0.051 rows=3 loops=1)
  Index Cond: (user_id = 42 AND status = 'confirmed')
Planning Time: 0.4 ms
Execution Time: 0.1 ms  ← 0.1 milliseconds. 18,000x faster.
```

"Index Scan" = index used = problem solved.

## Index Design Strategy — How to Think About It

### Step 1: Start with no indexes (except primary key)

Primary keys get a unique index automatically. Don't add anything else yet.

### Step 2: Add indexes for your most critical queries

```sql
-- Your most common query: "get all bookings for a user"
SELECT * FROM bookings WHERE user_id = ? ORDER BY created_at DESC;

-- Index needed:
CREATE INDEX idx_bookings_user_date ON bookings(user_id, created_at DESC);
-- Composite covers both the WHERE and the ORDER BY in one index.
```

### Step 3: Add indexes for foreign keys

```sql
-- Foreign keys without indexes cause slow JOINs
-- Every ON DELETE CASCADE on parent scans the child table
CREATE INDEX idx_bookings_event_id ON bookings(event_id);
CREATE INDEX idx_bookings_user_id  ON bookings(user_id);
```

### Step 4: Only add more indexes when you have evidence

Use slow query logs, EXPLAIN ANALYZE, and monitoring to find actual slow queries. Add indexes surgically, not speculatively.

### Step 5: Regularly audit and drop unused indexes

Especially after feature changes — old features may have left behind indexes that queries no longer use.

> *"What indexes would you design for the AXS bookings table?"*

```sql
CREATE TABLE bookings (
    id           SERIAL PRIMARY KEY,     -- ← automatic unique index
    user_id      INT NOT NULL,
    event_id     INT NOT NULL,
    seat_id      INT NOT NULL,
    status       VARCHAR(20) NOT NULL,
    amount_paid  DECIMAL(10,2),
    booked_at    TIMESTAMPTZ DEFAULT NOW()
);

-- Index 1: Most common query — "show my bookings"
-- Covers user lookup + sorted by date (most recent first)
CREATE INDEX idx_bookings_user_date
    ON bookings(user_id, booked_at DESC);

-- Index 2: Event management — "all bookings for this event"
-- Used by venue/organizer dashboard and revenue reports
CREATE INDEX idx_bookings_event
    ON bookings(event_id);

-- Index 3: Prevent double-booking at DB level
-- One confirmed booking per seat per event
CREATE UNIQUE INDEX idx_bookings_seat_unique
    ON bookings(seat_id)
    WHERE status = 'confirmed';   -- partial: only unique among confirmed

-- Index 4: Finance team queries — "all refunds this month"
-- Partial index: only index cancelled/refunded rows (minority of data)
CREATE INDEX idx_bookings_refunds
    ON bookings(booked_at, amount_paid)
    WHERE status IN ('cancelled', 'refunded');

-- What we deliberately DON'T index:
-- amount_paid alone (never queried in isolation)
-- booked_at alone (always combined with user_id or status)
-- status alone (low cardinality — only 4 values, index rarely helps)
```

**Key things to say:**
- "I'd use a partial unique index for seat uniqueness — only confirmed bookings need to be unique, a user can have multiple cancelled bookings for the same seat."
- "I deliberately don't index `status` alone — with only 4 possible values, the index is low-cardinality and the optimizer often ignores it and does a table scan anyway."
- "During flash sales, write throughput is critical — fewer indexes means faster INSERT for each booking."

### Question 1: Given a slow query, identify and fix it *(most common)*

> "This query takes 8 seconds on a 10M row table. Fix it."

```sql
-- Slow query

FROM users u
JOIN bookings b ON u.id = b.user_id
```

**Diagnosis process:**

```sql
EXPLAIN ANALYZE <query above>;
-- See: Seq Scan on bookings (cost=... actual time=7843ms)
-- Problem: no index on (status, booked_at) or (booked_at, status)
```

**Fix:**

```sql
-- Composite index covering both filter columns
-- Put equality condition first (status), range condition second (booked_at)
CREATE INDEX idx_bookings_status_date
ON bookings(status, booked_at)
WHERE status = 'confirmed';  -- partial: only index confirmed rows

CREATE INDEX idx_bookings_confirmed_covering

WHERE status = 'confirmed';
```

**After index — EXPLAIN shows:**
```

Hash Join with users   → 0.8ms
Total: ~1ms vs 8 seconds
```

### Question 2: Implement a B-Tree node *(data structures angle)*

> "Implement a simple B-Tree supporting insert and search."

```csharp
public class BTreeNode
{
    public List<int> Keys = new();
    public List<BTreeNode> Children = new();
    public bool IsLeaf;
    public int MinDegree; // minimum keys per node = MinDegree - 1

    public BTreeNode(int minDegree, bool isLeaf)
    {
        MinDegree = minDegree;
        IsLeaf = isLeaf;
    }
}

public class BTree
{
    private BTreeNode _root;
    private readonly int _t; // minimum degree

    public BTree(int minDegree)
    {
        _t = minDegree;
        _root = new BTreeNode(_t, true);
    }

    // Search — O(log N)
    public bool Search(BTreeNode node, int key)
    {
        int i = 0;

        // Find first key >= target
        while (i < node.Keys.Count && key > node.Keys[i])
            i++;

        // Found exact match
        if (i < node.Keys.Count && key == node.Keys[i])
            return true;

        // Reached leaf without finding key
        if (node.IsLeaf)
            return false;

        // Recurse into appropriate child
        return Search(node.Children[i], key);
    }

    public bool Search(int key) => Search(_root, key);
}
```

**What this demonstrates:** You understand that B-Tree search is O(log N) because at each node you halve (roughly) the search space, and you only recurse into one child — not all of them. This is exactly what the database does when using an index.

### Question 3: Design indexes for a given query workload *(system design coding)*

> "Given these 5 queries, design the minimum set of indexes."

```sql
-- Query 1 (runs 10,000x/day): 
SELECT * FROM bookings WHERE user_id = ? ORDER BY booked_at DESC LIMIT 20;

-- Query 2 (runs 1,000x/day):
SELECT * FROM bookings WHERE event_id = ? AND status = 'confirmed';

-- Query 3 (runs 100x/day — admin only):
SELECT event_id, SUM(amount_paid) FROM bookings 
WHERE booked_at BETWEEN ? AND ? GROUP BY event_id;

-- Query 4 (runs 50,000x/day — most critical):
SELECT * FROM bookings WHERE seat_id = ? AND status = 'confirmed';

-- Query 5 (runs 10x/day — reporting):
SELECT user_id, COUNT(*) FROM bookings WHERE status = ? GROUP BY user_id;
```

**Answer — think about frequency and criticality:**

```sql
-- For Query 1 (high frequency, user-facing):
CREATE INDEX idx_q1 ON bookings(user_id, booked_at DESC);
-- Covers both the WHERE and ORDER BY in one index.

-- For Query 2 (medium frequency):
CREATE INDEX idx_q2 ON bookings(event_id, status);
-- event_id first (equality), status second (equality).

-- For Query 3 (low frequency, admin):
-- Consider: is this worth an index? 100x/day is low.
-- If it's an analytical query, run it against a read replica.
-- OR: add a partial index only for recent bookings.
CREATE INDEX idx_q3 ON bookings(booked_at, event_id, amount_paid);
-- Covering index: includes amount_paid so no table fetch needed.

-- For Query 4 (HIGHEST frequency — seat lock check):
CREATE UNIQUE INDEX idx_q4 ON bookings(seat_id)
WHERE status = 'confirmed';
-- Partial unique index: fastest possible lookup + enforces no double-booking.

-- For Query 5 (very low frequency, reporting):
-- DON'T add index. 10x/day doesn't justify the write overhead.
-- Run against read replica. Accept 1-2 second query time.

-- Final decision: 4 indexes, not 5.
-- Query 5 is sacrificed because write cost > read benefit at that frequency.
```

**This answer demonstrates senior-level thinking:** you're not just adding indexes, you're weighing frequency, criticality, and write overhead to make a deliberate tradeoff.

### Question 4: Implement an LRU cache to simulate buffer pool *(indirect indexing question)*

Every database uses a **buffer pool** — an in-memory cache of recently accessed index pages. When the buffer pool is full, it evicts the least recently used page using LRU. This is why LRU cache (LeetCode 146) is connected to database internals.

```csharp
// The database's buffer pool IS an LRU cache of B-Tree nodes/pages
// When you access an index page, it goes to front of LRU
// When buffer fills up, the least-recently-used page is evicted to disk

public class BufferPool
{
    private readonly int _capacity;
    private readonly Dictionary<int, LinkedListNode<(int pageId, byte[] data)>> _map = new();
    private readonly LinkedList<(int pageId, byte[] data)> _lru = new();

    public BufferPool(int pageCapacity) => _capacity = pageCapacity;

    // Fetch a B-Tree page (index node) — O(1)
    public byte[] FetchPage(int pageId)
    {
        if (_map.TryGetValue(pageId, out var node))
        {
            // Cache hit — move to front (most recently used)
            _lru.Remove(node);
            _lru.AddFirst(node);
            return node.Value.data;
        }

        // Cache miss — read from disk
        var data = ReadFromDisk(pageId);

        if (_map.Count >= _capacity)
        {
            // Evict LRU page — back of list
            var lruPage = _lru.Last;
            _lru.RemoveLast();
            _map.Remove(lruPage.Value.pageId);
            WriteToDisk(lruPage.Value); // flush if dirty
        }

        var newNode = _lru.AddFirst((pageId, data));
        _map[pageId] = newNode;
        return data;
    }

    private byte[] ReadFromDisk(int pageId) => new byte[8192]; // 8KB page
    private void WriteToDisk((int pageId, byte[] data) page) { }
}
```

**Why this matters for interviews:** When an interviewer asks "why does adding more RAM help database performance?" — the answer is a larger buffer pool means more B-Tree index pages stay in memory, reducing disk reads. Understanding this connection shows depth.

## One-Line Summary for Interviews

> **A database index is a sorted B-Tree data structure that enables O(log N) lookups instead of O(N) full table scans — the leftmost prefix rule governs composite indexes, partial indexes reduce size by only indexing relevant rows, and covering indexes eliminate table access entirely — but every index slows writes because inserts, updates, and deletes must update every index, so the discipline of indexing is not "add indexes everywhere" but "add exactly the indexes your most critical queries need and ruthlessly drop the rest."**


# Object vs Block vs File Storage — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine you need to store things in your home.

**File Storage is a filing cabinet.** Everything goes into folders, inside drawers, with a clear path: `Cabinet 3 → Drawer B → Folder "Taxes" → "2024_return.pdf"`. Organized hierarchy. Everyone in the house knows where things are. But the cabinet is in one room — you can't easily split it across multiple buildings.

**Block Storage is a stack of blank LEGO bricks.** Raw, uniform chunks of space. No inherent organization. You (or your operating system) decide how to arrange them into something meaningful — a database, a filesystem, whatever you need. Maximum flexibility and performance, but you need to know exactly what you're doing.

**Object Storage is a massive warehouse with a package counter.** You walk up, hand over any package — a photo, a video, a 50GB backup file — and get a tracking number (a URL). The warehouse figures out where to put it internally. You never know or care which shelf it's on. You just use the tracking number to retrieve it later. The warehouse can have infinite shelves across infinite buildings.

Same data. Three completely different ways of thinking about storage.

---

## File Storage — The Familiar One

### What it is

File storage organizes data in a **hierarchical tree of directories and files**, exactly like the filesystem on your laptop. Data is accessed via paths.

```
/
├── events/
│   ├── 2025/
│   │   ├── taylor-swift/
│   │   │   ├── banner.jpg
│   │   │   ├── seating-map.png
│   │   │   └── promo-video.mp4
│   │   └── coldplay/
│   │       └── banner.jpg
├── users/
│   ├── avatars/
│   │   ├── user-42.jpg
│   │   └── user-87.jpg
└── reports/
    └── revenue-2025-q1.xlsx
```

### How it works

A **Network File System (NFS)** or **SMB/CIFS share** exposes this hierarchy over the network. Multiple servers mount the same filesystem. They all see the same files, same directory structure, same path conventions.

```
App Server 1 ──mounts──→ /mnt/shared/events/banner.jpg
App Server 2 ──mounts──→ /mnt/shared/events/banner.jpg
App Server 3 ──mounts──→ /mnt/shared/events/banner.jpg
                          ↑ all three see the same file
```

From the application's perspective — it's just reading and writing local files. The network is transparent.

### AWS equivalent: EFS (Elastic File System)

```csharp
// Application code — looks identical to local file access
string content = File.ReadAllText("/mnt/shared/events/config.json");
File.WriteAllBytes("/mnt/shared/reports/output.pdf", pdfBytes);
```

No special SDK. No special API. Just standard file I/O.

### What file storage is good at

```
✅ Shared access — multiple servers read/write same files simultaneously
✅ Familiar interface — standard filesystem operations
✅ Good for lift-and-shift — legacy apps that expect a filesystem work unchanged
✅ POSIX compliance — file locking, permissions, directory listings
✅ Media workflows — video editing pipelines where multiple tools access the same files
```

### What file storage struggles with

```
❌ Scale ceiling — filesystem hierarchy doesn't scale to billions of files
❌ Performance at scale — millions of files in one directory = slow directory listing
❌ Geographic distribution — hard to replicate across regions consistently
❌ Cost — network-attached file systems are expensive per GB
❌ Metadata — only basic attributes (size, modified date, permissions)
```

### When to use file storage

```
Shared configuration files across app servers
Content management systems expecting a filesystem
Legacy applications that can't be rewritten
Development environments and CI/CD shared workspaces
Video production pipelines where editors need shared access
Home directories for enterprise users
```

---

## Block Storage — The Raw Power One

### What it is

Block storage presents raw storage as a **virtual hard disk** — a sequence of fixed-size blocks (typically 512 bytes to 4KB each), with no inherent structure. The operating system formats it with a filesystem (ext4, NTFS, XFS), then uses it however it needs.

```
Physical reality:                  What OS sees:
[Block 0][Block 1][Block 2]...  →  Virtual disk drive
[Block N]...                    →  Format with ext4
                                →  Mount as /dev/sdb
                                →  Use as /var/lib/postgresql/data
```

The storage system doesn't know or care what's in the blocks. It just stores and retrieves them by address.

### AWS equivalent: EBS (Elastic Block Store)

```
EC2 Instance running PostgreSQL:
  /dev/sda1  ← root OS volume (EBS)
  /dev/sdb   ← PostgreSQL data directory (EBS, high IOPS)
  /dev/sdc   ← PostgreSQL WAL logs (EBS, separate volume)
```

Each EBS volume is **attached to exactly one EC2 instance** at a time. Exclusive access. This is a critical constraint.

### Why databases love block storage

Databases are the canonical use case. Here's why:

```
Database needs:
  1. Raw I/O control — database manages its own buffer pool,
     write-ahead log, page flushing. It doesn't want an OS
     filesystem making caching decisions for it.

  2. Predictable latency — SSDs with no filesystem overhead
     give consistent microsecond-level I/O.

  3. Atomic writes — databases write in page-aligned blocks
     to guarantee crash consistency.

  4. Direct I/O — bypass OS page cache, write directly to disk.
     PostgreSQL and MySQL both support O_DIRECT for this.

Block storage delivers all four. File storage and object storage cannot.
```

### Performance tiers

```
EBS Volume Types (AWS):
  gp3 (General Purpose SSD):
    → 3,000 IOPS baseline, up to 16,000 IOPS
    → 125 MB/s throughput
    → Use: most databases, boot volumes

  io2 Block Express (Provisioned IOPS):
    → Up to 256,000 IOPS
    → Sub-millisecond latency
    → Use: high-frequency trading, large OLTP databases

  st1 (Throughput Optimized HDD):
    → 500 MB/s throughput, low IOPS
    → Use: sequential reads of large files (data warehouses)

  sc1 (Cold HDD):
    → Cheapest option
    → Use: infrequently accessed large data
```

### What block storage is good at

```
✅ Maximum performance — lowest latency, highest IOPS available
✅ Database workloads — structured, random-access I/O patterns
✅ OS boot volumes — operating system needs a block device
✅ Full control — application decides filesystem, caching, I/O scheduling
✅ Crash consistency — atomic block writes for database durability
```

### What block storage struggles with

```
❌ One-to-one attachment — one volume, one server at a time
   (EBS multi-attach exists but is limited and complex)
❌ Not accessible over HTTP — need to SSH into the server, can't serve via URL
❌ No built-in redundancy — you manage replication
❌ Capacity planning — must provision size upfront, scaling requires work
❌ Not sharable — two app servers can't both write to the same EBS volume
```

### When to use block storage

```
Any relational database (PostgreSQL, MySQL, SQL Server)
Any NoSQL database (MongoDB, Cassandra, Elasticsearch)
Operating system boot volumes
High-performance applications needing raw disk access
Virtual machine disks
Message queue brokers (Kafka, RabbitMQ) needing local disk
```

---

## Object Storage — The Infinite Warehouse

### What it is

Object storage organizes data as **flat key-value pairs**. No hierarchy. No directories (the `/` in S3 key names is cosmetic — there are no actual folders). Just a key and a blob of bytes.

```
Key (URL)                                → Value (bytes)
─────────────────────────────────────────────────────────
events/123/banner.jpg                   → [JPEG bytes]
events/123/seating-map.png              → [PNG bytes]
users/42/avatar.jpg                     → [JPEG bytes]
backups/2025-08-01/db-snapshot.tar.gz   → [10GB of bytes]
```

Access is via **HTTP REST API**:
```
GET  https://bucket.s3.amazonaws.com/events/123/banner.jpg
PUT  https://bucket.s3.amazonaws.com/users/42/avatar.jpg
DELETE https://bucket.s3.amazonaws.com/old-file.txt
```

Any client, anywhere in the world, with the right credentials can access any object via its URL. No mounting. No SSH. No filesystem.

### What makes object storage architecturally special

```
Durability:
  AWS S3 stores each object across multiple AZs automatically.
  11 nines of durability (99.999999999%) — designed to lose
  1 object per 10,000 years if you store 10 million objects.

Scale:
  No practical limit. Exabytes of data in a single bucket.
  Millions of objects added per day without any capacity planning.

Availability:
  S3 is a distributed system — no single point of failure.
  Requests automatically routed around failed nodes.

Cost:
  $0.023 per GB/month (S3 Standard).
  EBS: $0.08–$0.125 per GB/month.
  Object storage is 3-5x cheaper per GB than block storage.
```

### Rich metadata — a hidden superpower

Every object can carry custom metadata:

```http
PUT /events/123/banner.jpg HTTP/1.1
Content-Type: image/jpeg
x-amz-meta-event-id: 123
x-amz-meta-uploaded-by: user-42
x-amz-meta-approved: true
x-amz-meta-expires: 2025-12-31
```

You can store business context right with the data. No separate database entry needed to track what this file is.

### Object Storage features that matter for system design

**Presigned URLs — serve private files securely**

```csharp
// Generate a URL that expires in 15 minutes
// User can download the file without needing AWS credentials
var request = new GetPreSignedUrlRequest
{
    BucketName = "axs-tickets",
    Key = $"tickets/user-{userId}/booking-{bookingId}.pdf",
    Expires = DateTime.UtcNow.AddMinutes(15),
    Verb = HttpVerb.GET
};

string presignedUrl = s3Client.GetPreSignedURL(request);
// → https://axs-tickets.s3.amazonaws.com/tickets/...?X-Amz-Expires=900&X-Amz-Signature=...

// Send this URL to the user's browser
// Browser downloads directly from S3 — your server not involved
```

### What object storage is good at

```
✅ Infinite scale — no practical limit on objects or total size
✅ HTTP access — serve directly to browsers, CDN, mobile apps
✅ Durability — 11 nines, multi-AZ redundancy built in
✅ Cost — cheapest per-GB storage available
✅ Global accessibility — any authorized client anywhere
✅ CDN integration — CloudFront sits in front of S3 trivially
✅ Metadata — rich custom attributes per object
✅ Lifecycle management — automatic tiering and deletion
✅ No capacity planning — scales automatically
```

### What object storage struggles with

```
❌ Latency — HTTP request overhead. First byte: 10-100ms.
   Block storage: microseconds. Object storage: milliseconds.

❌ No random access within objects — can't read bytes 1000-2000
   of a 1GB file without downloading the whole thing
   (HTTP range requests help but aren't the same as seek())

❌ No append — can't add to an existing object.
   Must download, modify, re-upload entire object.

❌ Eventual consistency — in some operations, a PUT followed
   immediately by a GET might return the old version.
   (S3 now offers strong consistency, but design for eventual.)

❌ Not mountable as filesystem — can't use as database storage.
   Tools like s3fs exist but are slow and unreliable for DBs.

❌ API overhead — every access is an HTTP call. Not suitable
   for thousands of small random reads per second.
```

```
Video and audio files
Database backups and snapshots
Application logs and analytics data
Machine learning training datasets
Software artifacts and releases
Email attachments
Ticket PDFs and receipts
Anything you'd serve via a URL
```

## The Fundamental Difference — Access Pattern

This is the key insight for interviews:

```
Block Storage:   Think like a hard drive.
                 Random reads/writes at byte-level granularity.
                 Microsecond latency.
                 One client. Local access.
                 Used by: databases, OS, anything needing raw disk.

File Storage:    Think like a shared network folder.
                 Read/write files via paths.
                 Millisecond latency.
                 Multiple clients simultaneously.
                 Used by: shared config, legacy apps, media workflows.

Object Storage:  Think like a REST API for bytes.
                 Read/write whole objects via HTTP.
                 Tens of milliseconds latency.
                 Infinite clients worldwide.
                 Used by: everything served over the internet.
```

## The Upload Flow — Putting It All Together

> *"User uploads a profile photo on AXS. Walk me through the storage decisions."*

```
Step 1: User selects photo on mobile app
Step 2: App requests a presigned upload URL from backend
        GET /api/upload-url?type=avatar&userId=42

Step 3: Backend generates presigned S3 PUT URL
        → URL valid for 5 minutes
        → Targets: s3://axs-media/users/42/avatar-{uuid}.jpg

Step 4: App uploads DIRECTLY to S3 using presigned URL
        PUT https://axs-media.s3.amazonaws.com/users/42/avatar-{uuid}.jpg
        → File never passes through your servers
        → S3 handles the bandwidth

Step 5: S3 triggers Lambda on upload complete
        → Lambda resizes to 3 sizes: 32px, 128px, 512px
        → Lambda writes thumbnails back to S3

Step 6: Backend stores S3 key in PostgreSQL (block storage)
        UPDATE users SET avatar_key = 'users/42/avatar-abc123.jpg'
        WHERE id = 42;

Step 7: Future requests serve via CloudFront CDN URL
        https://cdn.axs.com/users/42/avatar-abc123-128px.jpg
        → CloudFront caches at edge
        → S3 only hit on cache miss
```

This flow demonstrates:
- Object storage for the file itself
- Block storage (PostgreSQL on EBS) for the metadata/reference
- CDN in front of object storage for performance
- Presigned URLs for secure direct upload (server doesn't handle file bytes)

### Question 1: Implement a file storage service interface *(most common)*

> "Design a storage abstraction that works with local disk in development and S3 in production."

```csharp
// Abstract interface — storage type is an implementation detail
public interface IStorageService
{
    Task<string> UploadAsync(string key, Stream content, string contentType);
    Task<Stream> DownloadAsync(string key);
    Task DeleteAsync(string key);
    Task<string> GetPublicUrlAsync(string key, TimeSpan? expiry = null);
    Task<bool> ExistsAsync(string key);
}

// Local file system — for development/testing
public class LocalStorageService : IStorageService
{
    private readonly string _basePath;

    public LocalStorageService(string basePath)
    {
        _basePath = basePath;
        Directory.CreateDirectory(basePath);
    }

    public async Task<string> UploadAsync(string key, Stream content, string contentType)
    {
        var filePath = Path.Combine(_basePath, key.Replace('/', Path.DirectorySeparatorChar));
        Directory.CreateDirectory(Path.GetDirectoryName(filePath)!);

        using var file = File.Create(filePath);
        await content.CopyToAsync(file);

        return key;
    }

    public Task<Stream> DownloadAsync(string key)
    {
        var filePath = Path.Combine(_basePath, key.Replace('/', Path.DirectorySeparatorChar));
        Stream stream = File.OpenRead(filePath);
        return Task.FromResult(stream);
    }

    public Task DeleteAsync(string key)
    {
        var filePath = Path.Combine(_basePath, key.Replace('/', Path.DirectorySeparatorChar));
        if (File.Exists(filePath)) File.Delete(filePath);
        return Task.CompletedTask;
    }

    public Task<string> GetPublicUrlAsync(string key, TimeSpan? expiry = null)
        => Task.FromResult($"http://localhost:5000/files/{key}");

    public Task<bool> ExistsAsync(string key)
        => Task.FromResult(File.Exists(
            Path.Combine(_basePath, key.Replace('/', Path.DirectorySeparatorChar))
        ));
}

// S3 — for production
public class S3StorageService : IStorageService
{
    private readonly IAmazonS3 _s3;
    private readonly string _bucketName;

    public S3StorageService(IAmazonS3 s3, string bucketName)
    {
        _s3 = s3;
        _bucketName = bucketName;
    }

    public async Task<string> UploadAsync(string key, Stream content, string contentType)
    {
        await _s3.PutObjectAsync(new PutObjectRequest
        {
            BucketName = _bucketName,
            Key = key,
            InputStream = content,
            ContentType = contentType,
            ServerSideEncryptionMethod = ServerSideEncryptionMethod.AES256
        });
        return key;
    }

    public async Task<Stream> DownloadAsync(string key)
    {
        var response = await _s3.GetObjectAsync(_bucketName, key);
        return response.ResponseStream;
    }

    public async Task DeleteAsync(string key)
        => await _s3.DeleteObjectAsync(_bucketName, key);

    public async Task<string> GetPublicUrlAsync(string key, TimeSpan? expiry = null)
    {
        if (expiry == null)
            return $"https://{_bucketName}.s3.amazonaws.com/{key}";

        // Presigned URL for private objects
        return await Task.FromResult(_s3.GetPreSignedURL(new GetPreSignedUrlRequest
        {
            BucketName = _bucketName,
            Key = key,
            Expires = DateTime.UtcNow.Add(expiry.Value),
            Verb = HttpVerb.GET
        }));
    }

    public async Task<bool> ExistsAsync(string key)
    {
        try
        {
            await _s3.GetObjectMetadataAsync(_bucketName, key);
            return true;
        }
        catch (AmazonS3Exception e) when (e.StatusCode == HttpStatusCode.NotFound)
        {
            return false;
        }
    }
}
```

**What this demonstrates:**
- Storage type is abstracted — swap local ↔ S3 via DI configuration
- Production code handles encryption, presigned URLs
- Clean separation of concerns — callers don't know where files live

### Question 2: Implement multipart upload for large files

> "S3 has a 5GB limit on single PUT uploads. Implement multipart upload for large files."

```csharp
public class MultipartUploader
{
    private readonly IAmazonS3 _s3;
    private const int PartSizeBytes = 5 * 1024 * 1024; // 5MB minimum per part

    public MultipartUploader(IAmazonS3 s3) => _s3 = s3;

    public async Task<string> UploadLargeFileAsync(
        string bucket,
        string key,
        Stream fileStream,
        string contentType)
    {
        // Step 1: Initiate multipart upload — get upload ID
        var initResponse = await _s3.InitiateMultipartUploadAsync(
            new InitiateMultipartUploadRequest
            {
                BucketName = bucket,
                Key = key,
                ContentType = contentType
            }
        );
        string uploadId = initResponse.UploadId;

        var completedParts = new List<PartETag>();
        var buffer = new byte[PartSizeBytes];
        int partNumber = 1;

        try
        {
            int bytesRead;
            while ((bytesRead = await fileStream.ReadAsync(buffer)) > 0)
            {
                // Step 2: Upload each part
                using var partStream = new MemoryStream(buffer, 0, bytesRead);

                var uploadResponse = await _s3.UploadPartAsync(
                    new UploadPartRequest
                    {
                        BucketName = bucket,
                        Key = key,
                        UploadId = uploadId,
                        PartNumber = partNumber,
                        InputStream = partStream,
                        IsLastPart = bytesRead < PartSizeBytes
                    }
                );

                completedParts.Add(new PartETag
                {
                    PartNumber = partNumber,
                    ETag = uploadResponse.ETag
                });

                partNumber++;
                Console.WriteLine($"Uploaded part {partNumber - 1}, {bytesRead} bytes");
            }

            // Step 3: Complete — S3 assembles all parts into final object
            await _s3.CompleteMultipartUploadAsync(
                new CompleteMultipartUploadRequest
                {
                    BucketName = bucket,
                    Key = key,
                    UploadId = uploadId,
                    PartETags = completedParts
                }
            );

            return $"s3://{bucket}/{key}";
        }
        catch (Exception)
        {
            // If anything fails, abort to avoid storage charges for incomplete upload
            await _s3.AbortMultipartUploadAsync(
                new AbortMultipartUploadRequest
                {
                    BucketName = bucket,
                    Key = key,
                    UploadId = uploadId
                }
            );
            throw;
        }
    }
}
```

**Key interview points:**
- Parts can be uploaded in parallel — dramatically speeds up large file uploads
- Each part has an ETag — S3 uses these to verify integrity when assembling
- AbortMultipartUpload on failure — incomplete uploads still incur storage charges until aborted

### Question 3: Design a file deduplication system

> "Users upload the same event photo thousands of times. Don't store duplicates."

```csharp
public class DeduplicatingStorage
{
    private readonly IStorageService _storage;
    private readonly IDatabase _redis;

    // hash → S3 key mapping stored in Redis
    public DeduplicatingStorage(IStorageService storage, IDatabase redis)
    {
        _storage = storage;
        _redis = redis;
    }

    public async Task<string> UploadAsync(Stream content, string contentType)
    {
        // Read content to compute hash
        using var ms = new MemoryStream();
        await content.CopyToAsync(ms);
        var bytes = ms.ToArray();

        // Compute SHA-256 of file content
        using var sha = SHA256.Create();
        var hashBytes = sha.ComputeHash(bytes);
        var hash = Convert.ToHexString(hashBytes).ToLower();

        // Check if we already have this exact file
        var existingKey = await _redis.StringGetAsync($"dedup:{hash}");
        if (existingKey.HasValue)
        {
            Console.WriteLine($"Dedup hit — reusing {existingKey}");
            return existingKey.ToString(); // return existing S3 key
        }

        // New unique file — upload it
        string key = $"content/{hash[..2]}/{hash[2..4]}/{hash}";
        // ↑ shard by first 4 chars of hash — avoids S3 hot partitions

        using var uploadStream = new MemoryStream(bytes);
        await _storage.UploadAsync(key, uploadStream, contentType);

        // Cache the hash → key mapping
        await _redis.StringSetAsync(
            $"dedup:{hash}",
            key,
            TimeSpan.FromDays(90) // expire cache after 90 days
        );

        return key;
    }
}
```

**What this shows:**
- Content-addressable storage — key derived from content hash, not filename
- Exact same technique used by Git (objects stored by SHA of content)
- S3 key sharding by hash prefix — prevents S3 partition hotspot on sequential keys
- Redis as the deduplication index — fast O(1) lookup

> **Block storage is a raw virtual disk for databases and operating systems needing microsecond-latency random access; file storage is a shared network filesystem for multiple servers needing a familiar path-based interface; object storage is an infinite HTTP-accessible key-value store for everything served over the internet — and in practice, mature systems use all three simultaneously: block storage for databases, object storage for user-uploaded content and static assets, and file storage for shared configuration and legacy workloads that can't be re-architected.**


# ACID Properties — Deep Dive for System Design Interviews

## Start With a Real Feeling

You're buying a Taylor Swift concert ticket on AXS. You click "Buy Now." Your card gets charged $250. The system crashes mid-transaction.

What should happen?

**The worst possible outcomes:**
- Your card is charged but no ticket is created. Money gone, no ticket.
- Ticket is created but your card isn't charged. Free ticket, AXS loses revenue.
- Seat A5 is marked as booked, but two people both got booking confirmations for it.
- The seat count shows 100 available, but 150 people already have tickets.

**ACID properties exist to make all four of these impossible.**

ACID is the set of guarantees a database makes so that even when things go wrong — crashes, network failures, concurrent users, power cuts — your data never ends up in a nonsensical state.

---

## A — Atomicity: All or Nothing

### The guarantee

A transaction is **indivisible**. Every operation inside it either all succeeds together, or all fail together. There is no partial success.

### The real feeling

Think of a transaction as a light switch — not a dimmer. It's either fully ON or fully OFF. Never halfway.

### The booking transaction

```sql
BEGIN TRANSACTION;

  -- Step 1: Mark seat as booked
  UPDATE seats 
  SET status = 'booked' 
  WHERE id = 'A5' AND event_id = 123;

  -- Step 2: Create booking record
  INSERT INTO bookings (user_id, seat_id, event_id, amount_paid)
  VALUES (42, 'A5', 123, 250.00);

  -- Step 3: Charge the user
  UPDATE user_accounts 
  SET balance = balance - 250.00 
  WHERE user_id = 42;

  -- Step 4: Update available seat count
  UPDATE events 
  SET seats_available = seats_available - 1 
  WHERE id = 123;

COMMIT;
```

If the server crashes after Step 2 but before Step 4:

```
Without atomicity:
  ✅ Seat marked booked
  ✅ Booking record created
  ❌ User never charged
  ❌ Seat count not decremented
  → Corrupt state. User has a free ticket. Seat count is wrong.

With atomicity:
  Database rolls back ALL steps on crash.
  System restarts as if the transaction never happened.
  User sees a failed transaction. Tries again.
  → Clean state. No corruption.
```

### How databases implement atomicity

**Write-Ahead Log (WAL):**

Before making any change to actual data, the database writes the intended change to a log file on disk first.

```
WAL entry: "BEGIN TXN-8821"
WAL entry: "UPDATE seats SET status='booked' WHERE id='A5'"
WAL entry: "INSERT INTO bookings VALUES (...)"
WAL entry: "UPDATE user_accounts SET balance=balance-250"
WAL entry: "UPDATE events SET seats_available=seats_available-1"
WAL entry: "COMMIT TXN-8821"
            ↑ only here does the transaction become durable
```

On crash recovery:
- If `COMMIT` entry exists in WAL → replay all operations (redo)
- If `COMMIT` entry missing → discard all operations (rollback)

This is why PostgreSQL's data directory has a `pg_wal` folder — it's the heartbeat of atomicity.

### Savepoints — nested atomicity

```sql
BEGIN TRANSACTION;

  INSERT INTO bookings (...) VALUES (...);
  
  SAVEPOINT after_booking;  -- partial checkpoint
  
  UPDATE inventory SET count = count - 1;
  
  -- Something goes wrong with inventory update
  ROLLBACK TO SAVEPOINT after_booking;
  -- Only inventory update rolled back, booking preserved
  
  -- Try alternative inventory approach
  UPDATE inventory SET count = count - 1 WHERE count > 0;

COMMIT;
```

Savepoints give you surgical rollback within a transaction — useful in complex multi-step operations.

---

## C — Consistency: Valid State to Valid State

### The guarantee

A transaction brings the database from **one valid state to another valid state**. All rules, constraints, and invariants defined on the database are respected before and after every transaction.

### The real feeling

Consistency is the database enforcing your business rules automatically, at the storage level. Not in application code. Not in a separate validation service. In the database itself, enforced on every write.

### The rules that protect AXS data

```sql
-- Rule 1: Seat status can only be valid values
ALTER TABLE seats
ADD CONSTRAINT chk_seat_status 
CHECK (status IN ('available', 'locked', 'booked'));

-- Rule 2: Can't book a seat for a nonexistent event
ALTER TABLE bookings
ADD CONSTRAINT fk_event
FOREIGN KEY (event_id) REFERENCES events(id)
ON DELETE RESTRICT;  -- can't delete event if bookings exist

-- Rule 3: Can't book a seat for a nonexistent user
ALTER TABLE bookings
ADD CONSTRAINT fk_user
FOREIGN KEY (user_id) REFERENCES users(id);

-- Rule 4: Amount paid must be positive
ALTER TABLE bookings
ADD CONSTRAINT chk_amount
CHECK (amount_paid > 0);

-- Rule 5: Available seats cannot go negative
ALTER TABLE events
ADD CONSTRAINT chk_seats
CHECK (seats_available >= 0);

-- Rule 6: Each seat booked at most once
ALTER TABLE bookings
ADD UNIQUE (seat_id, event_id)
WHERE status = 'confirmed';
```

Now try to violate any of these:

```sql
-- Try to create a booking for event_id=9999 (doesn't exist)
INSERT INTO bookings (user_id, seat_id, event_id, amount_paid)
VALUES (42, 'A5', 9999, 250.00);
-- ERROR: insert or update on table "bookings" violates 
-- foreign key constraint "fk_event"
-- → Transaction automatically rolled back. Database stays consistent.

-- Try to set seats_available to -1
UPDATE events SET seats_available = -1 WHERE id = 123;
-- ERROR: new row for relation "events" violates 
-- check constraint "chk_seats"
-- → Rejected. You cannot oversell.
```

### Consistency vs. application logic

```
Application-level consistency (fragile):
  if (seat.Status == "available") {
    // Another thread might change it right here
    seat.Status = "booked";
    SaveChanges();
  }
  // Can fail silently, can be bypassed, can have race conditions

Database-level consistency (robust):
  CONSTRAINT unique_booked_seat UNIQUE (seat_id) WHERE status = 'confirmed'
  // Cannot be bypassed. Enforced on every write. Always.
  // Two concurrent transactions trying to book A5?
  // Exactly one succeeds. The other gets a constraint violation.
```

### Important nuance for interviews

The C in ACID is the **only property guaranteed by the application** rather than the database engine. The database guarantees AID. Consistency depends on whether you've defined the right constraints and written correct transactions. An incorrectly designed schema with no foreign keys or check constraints gives you no consistency guarantee even with a fully ACID database.

## I — Isolation: Concurrent Transactions Don't Interfere

### The guarantee

Concurrent transactions execute as if they were **sequential** — one after another. No transaction sees another's intermediate (incomplete) state.

### The real feeling

Imagine 10,000 users simultaneously trying to book seats at the Taylor Swift flash sale. Without isolation, they'd all read and write the same data simultaneously — seeing each other's half-finished transactions. Chaos.

With isolation, each transaction runs as if it's the only transaction happening. The database serializes them correctly.

### The four dirty reads isolation solves

**1. Dirty Read — reading uncommitted data**

```
Transaction A (booking):              Transaction B (query):
BEGIN;
  UPDATE seats 
  SET status = 'booked' 
  WHERE id = 'A5';
                                      SELECT status FROM seats 
                                      WHERE id = 'A5';
                                      → Returns 'booked'  ← DIRTY READ
                                         (A hasn't committed yet!)
ROLLBACK;  ← A fails, seat reverts to 'available'
                                      -- B read data that never existed
                                      -- B now has incorrect information
```

**2. Non-Repeatable Read — same query, different results in same transaction**

```
Transaction A (booking flow):         Transaction B (admin update):
BEGIN;
  SELECT price FROM events 
  WHERE id = 123;
  → Returns 250.00
                                      BEGIN;
                                        UPDATE events SET price = 300.00;
                                      COMMIT;
  -- ... processing payment at 250 ...

  SELECT price FROM events 
  WHERE id = 123;
  → Returns 300.00  ← DIFFERENT RESULT!
  -- Same query, same transaction, different answer
  -- User was quoted 250, charged 300
```

**3. Phantom Read — same query, different rows**

```
Transaction A (reporting):            Transaction B (new booking):
BEGIN;
  SELECT COUNT(*) FROM bookings
  WHERE event_id = 123;
  → Returns 500
                                      BEGIN;
                                        INSERT INTO bookings 
                                        VALUES (new booking);
                                      COMMIT;

  SELECT COUNT(*) FROM bookings
  WHERE event_id = 123;
  → Returns 501  ← PHANTOM ROW appeared!
  -- Same query, same transaction, ghost row appeared
```

**4. Lost Update — two transactions overwrite each other**

```
Transaction A:                        Transaction B:
BEGIN;                                BEGIN;
  SELECT seats_available 
  FROM events WHERE id = 123;
  → Returns 100
                                        SELECT seats_available
                                        FROM events WHERE id = 123;
                                        → Returns 100
  UPDATE events 
  SET seats_available = 99;
  COMMIT;
                                        UPDATE events
                                        SET seats_available = 99;
                                        -- WRONG! Should be 98!
                                        COMMIT;
-- Both transactions decremented from 100 to 99
-- One decrement was lost. Sold 2 tickets, count only went down 1.
```

### Isolation Levels — the spectrum

SQL standard defines four isolation levels. Each trades performance for protection:

```
SERIALIZABLE        → Maximum isolation. Maximum overhead.
REPEATABLE READ     → Prevents dirty reads + non-repeatable reads
READ COMMITTED      → Prevents dirty reads only (most common default)
READ UNCOMMITTED    → No protection. Fastest. Almost never used.
```

```sql
-- Set isolation level for a transaction
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN;
  -- Your transaction here
COMMIT;
```

### Isolation level decision for AXS

```
Booking transactions → SERIALIZABLE or REPEATABLE READ
  Reason: Must prevent lost updates on seat counts.
  Two users booking the same seat must be serialized.
  Cost: more locking, lower throughput — acceptable for bookings.

Read-only queries (seat map display) → READ COMMITTED
  Reason: Seeing a seat as available for 100ms when
  it was just booked is acceptable.
  Cost: minimal — no additional locking.

Reporting queries → READ COMMITTED with read replica
  Reason: Reports don't need perfect accuracy to the millisecond.
  Dirty reads are fine for a revenue dashboard.
  Run against replica to not impact write performance.
```

### How isolation is implemented

**Pessimistic Locking — lock first, work second**

```sql
BEGIN;

-- Lock the row before reading it
-- Other transactions block until this transaction commits
SELECT * FROM seats 
WHERE id = 'A5' 
FOR UPDATE;  ← exclusive row lock

-- Now safely update — nobody else can touch this row
UPDATE seats SET status = 'booked' WHERE id = 'A5';

COMMIT;  -- lock released here
```

**Optimistic Locking — work first, verify before commit**

```sql
-- Read with a version number
SELECT id, status, version FROM seats WHERE id = 'A5';
-- Returns: {id: 'A5', status: 'available', version: 7}

-- ... user spends time on payment page ...

-- Only update if version hasn't changed (nobody else modified it)
UPDATE seats 
SET status = 'booked', version = 8
WHERE id = 'A5' AND version = 7;  -- ← optimistic check

-- If 0 rows affected → version changed → someone else modified it
-- → Retry or show error to user
```

**MVCC (Multi-Version Concurrency Control) — PostgreSQL's approach**

Instead of blocking readers with locks, PostgreSQL keeps multiple versions of each row:

```
Row A5 history:
  version 1: status='available', committed at T=100
  version 2: status='locked',    committed at T=200
  version 3: status='booked',    committed at T=300  ← current

Transaction started at T=150 sees version 1 (status='available')
Transaction started at T=250 sees version 2 (status='locked')
Transaction started at T=350 sees version 3 (status='booked')

Readers never block writers. Writers never block readers.
Each transaction sees a consistent snapshot of the world
at the moment it began.
```

This is why PostgreSQL can handle high read concurrency — readers and writers don't block each other.

## D — Durability: Committed = Permanent

### The guarantee

Once a transaction commits, it **survives permanently** — even power cuts, server crashes, OS panics, or cosmic rays flipping bits.

### The real feeling

When a user sees "Booking Confirmed! Your tickets are ready," they trust that booking exists permanently. They screenshot it, tell their friends, travel to the venue. If the server crashes 1 second after that confirmation — the booking must still exist when the server restarts.

### How durability is implemented

**Write-Ahead Log (WAL) — the core mechanism**

```
Normal operation:
  COMMIT received
    → Write COMMIT record to WAL on disk  (fast sequential write)
    → Acknowledge success to application
    → Actual data pages updated later (async, in background)

On crash recovery:
  Database starts up
    → Reads WAL from last checkpoint
    → Re-applies all committed transactions (redo)
    → Removes all uncommitted transactions (undo)
    → Database in consistent state matching last COMMIT
```

The crucial insight: **the WAL is sequentially written** (fast) while actual data pages are randomly updated (slower). Durability is achieved through the sequential WAL, not through immediately flushing random data pages.

**fsync — the operating system guarantee**

```csharp
// Without fsync:
File.WriteAllBytes("wal.log", walEntry);
// → Data in OS page cache (RAM)
// → OS might not have written to disk yet
// → Power cut → data lost
// → NOT durable

// With fsync:
File.WriteAllBytes("wal.log", walEntry);
fsync(fileDescriptor);
// → Forces OS to flush page cache to physical disk
// → Only returns after disk confirms write
// → Power cut after this → data survives
// → DURABLE
```

PostgreSQL calls fsync after every WAL write before acknowledging COMMIT. This is why:
- `synchronous_commit = on` (default) → durable, slightly slower
- `synchronous_commit = off` → faster, but last ~few hundred ms of commits can be lost on crash

**Replication for durability beyond one machine**

```
Primary DB:
  COMMIT received
  → Write to WAL
  → Stream WAL to replica
  → Replica writes WAL
  → Replica acknowledges
  → Primary acknowledges to application ← only now is response sent

Now even if primary's entire disk fails, replica has the data.
```

Synchronous replication means even hardware failure doesn't lose committed data.

## How ACID Properties Interact

They're not independent — they work together as a system:

```
Atomicity ensures:
  Transactions don't leave half-applied changes.

Consistency uses atomicity:
  Constraints are checked as part of atomic commit.
  If any constraint fails → entire transaction rolls back.

Isolation uses atomicity:
  Other transactions see either ALL changes or NONE
  (never intermediate state) because of atomic commit.

Durability uses atomicity:
  WAL records atomic units — commit or rollback, never partial.
```

A useful mental model:

```
Atomicity  → protection from partial failures (crashes)
Consistency → protection from invalid data (bad writes)
Isolation  → protection from concurrent interference (race conditions)
Durability → protection from data loss (crashes after commit)

All four together → your data is trustworthy no matter what happens.
```

## ACID vs BASE — The NoSQL Trade-off

Interviews often ask you to contrast ACID with BASE (the NoSQL model):

```
BASE:
  Basically Available    → system stays up even during partial failures
  Soft state            → data may be in flux, not immediately consistent
  Eventually consistent → given no new writes, all nodes converge to same value
```

```
Booking payment → ACID required
  "Did money leave the account?" must have a definitive answer.
  Eventual consistency here means: sometimes yes, sometimes no, figure it out later.
  Unacceptable.

Seat view count → BASE acceptable
  "How many people are viewing event 123 right now?"
  Server A says 10,432. Server B says 10,419.
  Both are close enough. Exact answer not required.
  Eventual consistency fine.

Session cache → BASE acceptable
  User's session data replicated across Redis nodes.
  Slight lag between primary and replica is fine.
  Session read from slightly stale replica is acceptable.
```

**The interview answer:** Use ACID where correctness is non-negotiable (money, inventory, legal records). Use BASE where availability and scale matter more than perfect consistency (analytics, recommendations, caches).

## AXS Full Booking Transaction — All Four Properties

```sql
BEGIN TRANSACTION 
ISOLATION LEVEL SERIALIZABLE;  -- I: full isolation

  -- Atomicity: all steps or none
  
  -- Step 1: Lock the seat (pessimistic locking)
  SELECT id, status, price
  FROM seats
  WHERE id = 'A5' AND event_id = 123
  FOR UPDATE;  -- exclusive lock, nobody else can touch this row
  
  -- Step 2: Verify availability (consistency check in application)
  -- If status != 'available': ROLLBACK, return error to user
  
  -- Step 3: Mark seat as booked
  UPDATE seats 
  SET status = 'booked'
  WHERE id = 'A5' AND event_id = 123;
  -- Consistency: CHECK constraint ensures status is valid value
  
  -- Step 4: Create booking record
  INSERT INTO bookings (user_id, seat_id, event_id, amount_paid, status)
  VALUES (42, 'A5', 123, 250.00, 'confirmed');
  -- Consistency: FK constraint ensures user_id and event_id exist
  -- Consistency: UNIQUE constraint ensures seat not double-booked
  -- Consistency: CHECK ensures amount_paid > 0
  
  -- Step 5: Decrement available count
  UPDATE events
  SET seats_available = seats_available - 1
  WHERE id = 123;
  -- Consistency: CHECK constraint prevents seats_available < 0
  
  -- Step 6: Record payment
  INSERT INTO payments (booking_id, amount, method, status)
  VALUES (LASTVAL(), 250.00, 'card', 'completed');

COMMIT;
-- Atomicity: all 6 steps committed atomically
-- Durability: WAL flushed to disk, replica acknowledged
-- → User sees "Booking Confirmed"
-- → This booking exists permanently even if server dies right now
```

### Question 1: Implement optimistic locking to prevent lost updates

> "Implement a seat booking that uses optimistic locking — no database locks, but prevent double booking."

```csharp
public class OptimisticBookingService
{
    private readonly DbContext _db;
    private const int MaxRetries = 3;

    public async Task<BookingResult> BookSeatAsync(int userId, string seatId, int eventId)
    {
        for (int attempt = 0; attempt < MaxRetries; attempt++)
        {
            try
            {
                // Read current state including version
                var seat = await _db.Seats
                    .FirstOrDefaultAsync(s => s.Id == seatId && s.EventId == eventId);

                if (seat == null)
                    return BookingResult.Fail("Seat not found");

                if (seat.Status != "available")
                    return BookingResult.Fail("Seat not available");

                int originalVersion = seat.Version;

                // Simulate user spending time on payment page
                // During this time, another user might book this seat

                // Attempt update ONLY if version hasn't changed
                int rowsAffected = await _db.Database.ExecuteSqlRawAsync(
                    @"UPDATE seats 
                      SET status = 'booked', version = version + 1
                      WHERE id = {0} 
                        AND event_id = {1}
                        AND version = {2}  -- optimistic check
                        AND status = 'available'",
                    seatId, eventId, originalVersion
                );

                if (rowsAffected == 0)
                {
                    // Version changed — someone else modified this row
                    // Retry from the beginning
                    await Task.Delay(50 * (attempt + 1)); // backoff
                    continue;
                }

                // Version matched — we got the seat, create booking
                var booking = new Booking
                {
                    UserId = userId,
                    SeatId = seatId,
                    EventId = eventId,
                    AmountPaid = seat.Price,
                    Status = "confirmed"
                };

                _db.Bookings.Add(booking);
                await _db.SaveChangesAsync();

                return BookingResult.Success(booking.Id);
            }
            catch (DbUpdateConcurrencyException)
            {
                // EF Core throws this when optimistic concurrency fails
                if (attempt == MaxRetries - 1) throw;
                await Task.Delay(100 * (attempt + 1));
            }
        }

        return BookingResult.Fail("Could not complete booking after retries");
    }
}
```

### Question 2: Implement a transaction with savepoints

> "Implement a group booking — book multiple seats atomically, but if some fail, try to book whatever is available."

```csharp
public async Task<GroupBookingResult> BookMultipleSeatsAsync(
    int userId,
    List<string> requestedSeatIds,
    int eventId)
{
    using var transaction = await _db.Database.BeginTransactionAsync(
        IsolationLevel.RepeatableRead
    );

    var successfulBookings = new List<string>();
    var failedSeats = new List<string>();

    try
    {
        foreach (var seatId in requestedSeatIds)
        {
            // Create savepoint before each seat attempt
            await _db.Database.ExecuteSqlRawAsync(
                $"SAVEPOINT seat_{seatId.Replace("-", "_")}"
            );

            try
            {
                var seat = await _db.Seats
                    .Where(s => s.Id == seatId && s.Status == "available")
                    .FirstOrDefaultAsync();

                if (seat == null)
                {
                    // Seat unavailable — rollback just this seat's savepoint
                    await _db.Database.ExecuteSqlRawAsync(
                        $"ROLLBACK TO SAVEPOINT seat_{seatId.Replace("-", "_")}"
                    );
                    failedSeats.Add(seatId);
                    continue;
                }

                seat.Status = "booked";
                _db.Bookings.Add(new Booking
                {
                    UserId = userId,
                    SeatId = seatId,
                    EventId = eventId,
                    AmountPaid = seat.Price
                });

                await _db.SaveChangesAsync();
                successfulBookings.Add(seatId);
            }
            catch (Exception)
            {
                // Rollback to before this seat attempt
                await _db.Database.ExecuteSqlRawAsync(
                    $"ROLLBACK TO SAVEPOINT seat_{seatId.Replace("-", "_")}"
                );
                failedSeats.Add(seatId);
            }
        }

        // Commit whatever succeeded
        await transaction.CommitAsync();

        return new GroupBookingResult(successfulBookings, failedSeats);
    }
    catch (Exception)
    {
        await transaction.RollbackAsync(); // rollback everything
        throw;
    }
}
```

### Question 3: Implement a simple WAL for atomicity *(tests deep understanding)*

> "Implement a basic Write-Ahead Log to guarantee atomicity for a key-value store."

```csharp
public class WalKeyValueStore
{
    private readonly Dictionary<string, string> _data = new();
    private readonly string _walPath;
    private readonly object _lock = new();

    public WalKeyValueStore(string walPath)
    {
        _walPath = walPath;
        Recover(); // replay WAL on startup
    }

    public void ExecuteTransaction(Dictionary<string, string> writes)
    {
        lock (_lock)
        {
            string txnId = Guid.NewGuid().ToString();

            // Step 1: Write all intended changes to WAL BEFORE applying
            var walEntries = new StringBuilder();
            walEntries.AppendLine($"BEGIN:{txnId}");

            foreach (var (key, value) in writes)
                walEntries.AppendLine($"WRITE:{txnId}:{key}:{value}");

            walEntries.AppendLine($"COMMIT:{txnId}");

            // fsync equivalent — write to disk, don't buffer
            File.AppendAllText(_walPath, walEntries.ToString());

            // Step 2: NOW apply to in-memory data
            // If crash here, WAL has COMMIT → recovery will redo this
            foreach (var (key, value) in writes)
                _data[key] = value;
        }
    }

    private void Recover()
    {
        if (!File.Exists(_walPath)) return;

        var committedTxns = new HashSet<string>();
        var pendingWrites = new Dictionary<string, Dictionary<string, string>>();

        // Pass 1: find all committed transaction IDs
        foreach (var line in File.ReadLines(_walPath))
        {
            var parts = line.Split(':');
            if (parts[0] == "COMMIT")
                committedTxns.Add(parts[1]);
            else if (parts[0] == "BEGIN")
                pendingWrites[parts[1]] = new Dictionary<string, string>();
            else if (parts[0] == "WRITE")
                pendingWrites[parts[1]][parts[2]] = parts[3];
        }

        // Pass 2: replay only committed transactions (redo)
        foreach (var txnId in committedTxns)
        {
            if (pendingWrites.TryGetValue(txnId, out var writes))
                foreach (var (key, value) in writes)
                    _data[key] = value;
        }

        // Uncommitted transactions (no COMMIT entry) are simply ignored (undo)
        Console.WriteLine($"Recovery complete. {committedTxns.Count} transactions replayed.");
    }

    public string Get(string key)
        => _data.TryGetValue(key, out var value) ? value : null;
}
```

**What this demonstrates:** You understand that WAL is the foundation of atomicity — write intentions to disk first, then apply. On recovery, replay committed entries and discard uncommitted ones. This is exactly what PostgreSQL, MySQL, and SQLite do.

### Question 4: Deadlock detection and prevention

> "Two transactions are waiting for each other's locks. How do you detect and resolve this?"

```csharp
public class DeadlockAwareBookingService
{
    private readonly DbContext _db;

    public async Task<bool> SwapSeatsAsync(
        int userId1, string seat1,
        int userId2, string seat2,
        int eventId)
    {
        // Deadlock scenario without ordering:
        // Transaction A locks seat1, waits for seat2
        // Transaction B locks seat2, waits for seat1
        // → Deadlock. Both wait forever.

        // Prevention: always acquire locks in consistent order
        // Sort seat IDs lexicographically — always lock lower ID first
        var orderedSeats = new[] { seat1, seat2 }.OrderBy(s => s).ToArray();

        using var transaction = await _db.Database.BeginTransactionAsync(
            IsolationLevel.Serializable
        );

        try
        {
            // Lock in consistent order — prevents deadlock by design
            foreach (var seatId in orderedSeats)
            {
                await _db.Database.ExecuteSqlRawAsync(
                    "SELECT id FROM seats WHERE id = {0} FOR UPDATE",
                    seatId
                );
            }

            // Now safely swap the seat assignments
            await _db.Database.ExecuteSqlRawAsync(
                "UPDATE bookings SET seat_id = {0} WHERE user_id = {1} AND event_id = {2}",
                seat2, userId1, eventId
            );

            await _db.Database.ExecuteSqlRawAsync(
                "UPDATE bookings SET seat_id = {0} WHERE user_id = {1} AND event_id = {2}",
                seat1, userId2, eventId
            );

            await transaction.CommitAsync();
            return true;
        }
        catch (PostgresException ex) when (ex.SqlState == "40P01") // deadlock_detected
        {
            // Database detected deadlock and killed this transaction
            // The other transaction continues
            await transaction.RollbackAsync();

            // Retry after brief pause
            await Task.Delay(Random.Shared.Next(50, 200));
            return await SwapSeatsAsync(userId1, seat1, userId2, seat2, eventId);
        }
    }
}
```

**Key insight:** The canonical deadlock prevention technique is **consistent lock ordering** — if every transaction acquires locks in the same order (e.g., by ascending resource ID), circular wait is impossible. Deadlocks cannot form.

## One-Line Summary for Interviews

> **ACID is the contract a database makes with your application: Atomicity guarantees transactions are all-or-nothing via Write-Ahead Logging; Consistency guarantees constraints and rules are never violated; Isolation guarantees concurrent transactions don't corrupt each other's work — controlled by isolation levels from READ COMMITTED to SERIALIZABLE implemented through pessimistic locks or MVCC; and Durability guarantees committed data survives crashes by flushing WAL to disk before acknowledging success — together they make the database the one place in your system where you can trust the data is always correct regardless of failures, crashes, or concurrency.**


# Message Queues — Deep Dive for System Design Interviews

## Start With a Real Feeling

Taylor Swift tickets go on sale. 500,000 people click "Buy Now" simultaneously.

Without a message queue, every one of those clicks hits your booking service directly:

```
500,000 simultaneous requests
        ↓
  Booking Service
  (can handle 1,000/sec)
        ↓
  Database
  (melts at 5,000 writes/sec)
        💥 CRASH
```

Every user sees an error. AXS loses millions in revenue. Twitter explodes.

With a message queue:

```
500,000 simultaneous requests
        ↓
  Message Queue  ← absorbs everything instantly
  [■■■■■■■■■■■]  ← 500,000 messages waiting
        ↓
  Booking Service reads at its own pace (1,000/sec)
        ↓
  Database receives controlled load
        ✅ Everything processes. Nothing crashes.
```

Users see "You're in the queue — position 43,821." 8 minutes later, they get a booking confirmation or a polite "sold out" message. The system never crashed. Every request was handled.

**That's a message queue.** A buffer between producers of work and consumers of work, letting each operate at their own pace.

---

## What a Message Queue Is

A message queue is a **durable, ordered buffer** that decouples the components producing work from the components doing the work.

```
Producer                    Queue                    Consumer
(creates messages)          (stores messages)        (processes messages)

[Booking API] ──push──→ [■■■■■■■■■■■] ──pull──→ [Booking Service]
[Payment API] ──push──→ [■■■■■■■■■■■] ──pull──→ [Email Service]
[User API]    ──push──→ [■■■■■■■■■■■] ──pull──→ [Analytics Service]
```

Three things make it powerful:

**1. Durable** — messages are persisted to disk. If the consumer crashes, messages aren't lost. They wait until a consumer comes back.

**2. Ordered** — messages are (usually) processed in the order they were sent.

**3. Decoupled** — producer doesn't know or care who processes the message. Consumer doesn't know or care who sent it. They can be scaled, deployed, and failed independently.

---

## The Four Core Problems Message Queues Solve

### Problem 1: Load Leveling (The Spike Problem)

Traffic is never constant. It spikes unpredictably.

```
Without queue:
Traffic:  ─────▄█████▄──────▄███▄────  (spikes)
Servers:  ────────💥──────────💥──────  (crash on spikes)

With queue:
Traffic:  ─────▄█████▄──────▄███▄────  (spikes hit queue)
Queue:    ────▄████████▄──▄█████▄────  (absorbs spikes)
Servers:  ──────────────────────────  (constant, controlled load)
```

The queue acts as a **shock absorber**. Spikes fill the queue. Servers drain the queue at their own pace. No crashes.

### Problem 2: Service Decoupling (The Dependency Problem)

Without a queue, services are tightly coupled:

```
Booking Service calls synchronously:
  → Email Service     (what if it's down?)
  → SMS Service       (what if it's slow?)
  → Analytics Service (what if it crashes?)
  → Fraud Detection   (what if it times out?)

If ANY of these fail or are slow:
  → Booking request times out
  → User sees error
  → Booking might not even complete
```

With a queue:

```
Booking Service:
  → Creates booking in DB
  → Publishes "booking_created" event to queue
  → Returns "Booking Confirmed!" to user ← instant

Queue delivers to each service independently:
  → Email Service     (processes when ready, if it was down: retries)
  → SMS Service       (processes when ready)
  → Analytics Service (processes when ready)
  → Fraud Detection   (processes when ready)

Booking Service doesn't wait for any of them.
Each service fails independently without affecting booking.
```

### Problem 3: Reliability (The "At Least Once" Problem)

Synchronous calls can fail silently:

```
Booking Service → POST /email-service/send → timeout
                  (Did it send? Did it not? Unknown.)

Booking Service → pushes to queue
Queue → Email Service processes
Email Service → removes message from queue ONLY after success
              (If email service crashes mid-send → message stays in queue → retried)
```

Message queues give you **guaranteed delivery**. A message isn't removed until a consumer explicitly acknowledges it.

### Problem 4: Fan-Out (The "Tell Everyone" Problem)

One event needs to notify multiple systems:

```
"Booking Created" event → needs to reach:
  Email Service
  SMS Service
  Analytics Pipeline
  Fraud Detection
  Recommendation Engine (update user preferences)
  Inventory Service (decrement seat count)
  Push Notification Service

Without pub/sub:
  Booking Service must call all 7 synchronously.
  If any fail, partial notification.
  Tight coupling to 7 services.

With pub/sub message queue:
  Booking Service publishes ONE message.
  Each of the 7 services subscribes independently.
  Adding an 8th subscriber = zero changes to Booking Service.
```

---

## Queue vs Pub/Sub — The Key Distinction

Interviewers love this distinction. They're related but different.

### Queue (Point-to-Point)

```
Producer → [Queue] → Consumer 1
                  ← Consumer 2 (competing consumers)
                  ← Consumer 3

One message is delivered to EXACTLY ONE consumer.
Multiple consumers compete for messages.
Used for: work distribution, load balancing
```

```
Example: Booking processing queue
  500 booking messages in queue
  3 booking workers pull messages
  Each message processed by exactly one worker
  (you don't want 3 workers all processing the same booking)
```

### Pub/Sub (Publish-Subscribe)

```
Publisher → [Topic] → Subscription 1 → Email Service
                   → Subscription 2 → SMS Service
                   → Subscription 3 → Analytics

One message is delivered to ALL subscribers.
Each subscriber gets their own copy.
Used for: event broadcasting, fan-out
```

```
Example: booking_created topic
  1 booking event published
  Email Service gets a copy
  SMS Service gets a copy
  Analytics Service gets a copy
  (all three need to know about every booking)
```

### The tools

```
Queue model:    Amazon SQS, RabbitMQ (queues)
Pub/Sub model:  Google Pub/Sub, Amazon SNS, RabbitMQ (exchanges)
Both models:    Apache Kafka (consumer groups = queue, topics = pub/sub)
```

## Apache Kafka — The Deep Dive

Kafka is the dominant message queue in high-scale systems. Interviewers expect you to know it deeply.

### Why Kafka is different

Traditional queues delete messages after consumers read them. Kafka keeps messages in a **persistent, ordered log** — like a database of events.

```
Traditional Queue:
  Message → Consumer reads → Message deleted
  (message is gone after consumption)

Kafka Topic (Partition):
  Offset: 0    1    2    3    4    5    6
          [msg][msg][msg][msg][msg][msg][msg]  → append-only log
  
  Consumer A reads offset 0-3, moves its pointer to 4
  Consumer B (different group) reads offset 0-3 independently
  Both consumers read the SAME messages
  Messages stay in log for retention period (days/weeks)
```

### Kafka anatomy

```
Topic: "booking_events"
  
  Partition 0: [event1][event3][event5][event7]  ← events for user IDs 0-999
  Partition 1: [event2][event4][event6][event8]  ← events for user IDs 1000-1999
  Partition 2: [event9][event10]...               ← events for user IDs 2000+

  Each partition is an ordered, immutable log.
  Messages within a partition are strictly ordered.
  Messages across partitions have no ordering guarantee.
```

**Topic** — logical channel (like a category)
**Partition** — unit of parallelism (one partition = one ordered stream)
**Offset** — position of a message within a partition
**Consumer Group** — set of consumers that together consume a topic (queue behavior)
**Broker** — a Kafka server node

### Partitioning — the parallelism key

```
Booking events for event_id=123:
  producer uses: partition = hash(event_id) % num_partitions

All messages for event_id=123 → same partition → strict ordering guaranteed
Messages for event_id=456 → different partition → processed in parallel

Consumer Group with 3 workers:
  Worker 1 → reads Partition 0 exclusively
  Worker 2 → reads Partition 1 exclusively  
  Worker 3 → reads Partition 2 exclusively
  
  3 partitions → 3 workers in parallel → 3x throughput
  Want more throughput? Add partitions + workers.
```

**Critical rule:** You can have at most as many consumers in a group as partitions. Extra consumers sit idle.

### Consumer Groups — queue vs pub/sub in Kafka

```
Topic: "booking_created"
  Partition 0: [msg1][msg2][msg3]
  Partition 1: [msg4][msg5][msg6]

Consumer Group "email-service":
  Email Worker 1 → reads Partition 0
  Email Worker 2 → reads Partition 1
  → Each message processed by exactly ONE email worker (queue behavior)

Consumer Group "analytics-service":
  Analytics Worker → reads Partition 0 AND 1
  → Reads ALL messages independently (pub/sub behavior)

Both groups read the SAME messages from the SAME topic.
Email service and Analytics service both get every booking event.
But within each group, work is distributed (not duplicated).
```

This is why Kafka elegantly unifies queue and pub/sub in one system.

### Offset management — the reliability mechanism

```
Consumer reads messages → processes them → commits offset

"Committed offset" = "I've successfully processed everything up to here"

If consumer crashes at offset 47:
  Restarts → asks Kafka "what's my last committed offset?"
  Kafka → "47"
  Consumer → resumes from 48
  No messages lost. No messages skipped.

At-most-once: commit offset BEFORE processing (can lose messages)
At-least-once: commit offset AFTER processing (can process twice)
Exactly-once: Kafka transactions (complex, use when truly needed)
```

### Retention — Kafka as event store

```
Traditional queue:  Message deleted after consumption
Kafka:              Message retained for N days regardless of consumption

Benefits:
  New service deployed? → read from the beginning of the log
  Bug in processing?    → reset consumer offset, reprocess from offset 47
  Audit trail?          → last 30 days of all events, always available
  Event sourcing?       → topic IS the source of truth
```

## Delivery Guarantees — The Reliability Spectrum

Every queue has a delivery guarantee. Know these cold.

### At Most Once

Message delivered zero or one times. May be lost. Never duplicated.

```
Producer sends message
Queue delivers to consumer
Consumer crashes BEFORE processing
Queue marks message as delivered (it's gone)
Message lost.

Use when: Losing some messages is acceptable.
Examples: Logging, metrics, analytics (losing 0.01% of events is fine)
```

### At Least Once

Message delivered one or more times. Never lost. May be duplicated.

```
Producer sends message
Queue delivers to consumer
Consumer processes message successfully
Consumer crashes BEFORE acknowledging to queue
Queue times out → redelivers message
Consumer processes message AGAIN

Use when: Losing messages is unacceptable, but processing twice is handleable.
Examples: Most business events (handle duplicates with idempotency)
```

### Exactly Once

Message delivered exactly one time. Never lost, never duplicated.

```
Requires:
  Idempotent producers (Kafka assigns sequence numbers, detects duplicates)
  Transactional consumers (consume + process + produce in one transaction)
  Significant overhead

Use when: Correctness is critical AND idempotency is hard to implement.
Examples: Financial transactions, inventory decrement
```

### Idempotency — the practical solution to at-least-once

Since exactly-once is complex and expensive, most systems use at-least-once + idempotency:

```csharp
public async Task ProcessBookingMessage(BookingMessage message)
{
    // Check if we've already processed this message
    // Use message ID as idempotency key
    bool alreadyProcessed = await _redis.SetAsync(
        $"processed:booking:{message.MessageId}",
        "1",
        TimeSpan.FromDays(7),  // keep for 7 days (retry window)
        When.NotExists         // only set if not exists
    );

    if (!alreadyProcessed)
    {
        // Already processed — this is a duplicate delivery
        // Acknowledge without reprocessing
        return;
    }

    // First time seeing this message — process it
    await CreateBookingInDatabase(message);
    await SendConfirmationEmail(message);
}
```

The message might be delivered twice. The idempotency check ensures it's only processed once.

## Dead Letter Queues — Handling Poison Messages

Some messages fail repeatedly — malformed data, business rule violations, downstream service permanently down. Without a DLQ, these messages block the queue or get retried infinitely.

```
Normal flow:
  Queue → Consumer → Success → Acknowledge → Message gone

Failure flow:
  Queue → Consumer → Failure → Message requeued
  Queue → Consumer → Failure → Message requeued  (retry 1)
  Queue → Consumer → Failure → Message requeued  (retry 2)
  Queue → Consumer → Failure → Message requeued  (retry 3)
  Max retries exceeded → Message moved to DLQ

Dead Letter Queue:
  [poison messages sit here]
  → Engineers investigate
  → Fix the bug or data issue
  → Replay messages from DLQ back to main queue
```

```csharp
// SQS DLQ configuration
var queueAttributes = new Dictionary<string, string>
{
    ["RedrivePolicy"] = JsonSerializer.Serialize(new
    {
        maxReceiveCount = "3",    // retry 3 times
        deadLetterTargetArn = dlqArn  // then send to DLQ
    })
};
```

DLQs are essential for production systems. Forgetting to mention them is a gap interviewers notice.

```
        ↓

    2. Create booking in PostgreSQL (ACID transaction)

    4. Acknowledge SQS message (remove from queue)

─────────────────────────────────────────────────────────
  Kafka topic: "booking_created"

─────────────────────────────────────────────────────────
```

### Question 1: Implement a basic in-memory message queue *(most common)*

> "Implement a thread-safe message queue with enqueue, dequeue, and acknowledgment."

```csharp
public class MessageQueue<T>
{
    private readonly Queue<QueueMessage<T>> _queue = new();
    private readonly Dictionary<string, QueueMessage<T>> _inFlight = new();
    private readonly TimeSpan _visibilityTimeout;
    private readonly object _lock = new();

    public MessageQueue(TimeSpan visibilityTimeout)
    {
        _visibilityTimeout = visibilityTimeout;
    }

    // Producer sends message
    public string Enqueue(T body)
    {
        lock (_lock)
        {
            var message = new QueueMessage<T>
            {
                Id = Guid.NewGuid().ToString(),
                Body = body,
                EnqueuedAt = DateTime.UtcNow,
                ReceiveCount = 0
            };

            _queue.Enqueue(message);
            return message.Id;
        }
    }

    // Consumer receives message (makes it invisible to other consumers)
    public QueueMessage<T>? Receive()
    {
        lock (_lock)
        {
            // Also check for visibility timeout expiry (requeue timed-out messages)
            RequeueTimedOutMessages();

            if (_queue.Count == 0) return null;

            var message = _queue.Dequeue();
            message.ReceiveCount++;
            message.VisibilityDeadline = DateTime.UtcNow.Add(_visibilityTimeout);

            // Move to in-flight tracking
            _inFlight[message.Id] = message;

            return message;
        }
    }

    // Consumer acknowledges successful processing
    public bool Acknowledge(string messageId)
    {
        lock (_lock)
        {
            return _inFlight.Remove(messageId);
            // Message is now gone — never redelivered
        }
    }

    // Consumer signals processing failure — make immediately available
    public bool Reject(string messageId)
    {
        lock (_lock)
        {
            if (!_inFlight.TryGetValue(messageId, out var message))
                return false;

            _inFlight.Remove(messageId);

            if (message.ReceiveCount >= 3)
            {
                // Send to DLQ instead of requeuing
                OnDeadLetter?.Invoke(message);
                return true;
            }

            _queue.Enqueue(message); // requeue for retry
            return true;
        }
    }

    // Called periodically — visibility timeout expired messages go back to queue
    private void RequeueTimedOutMessages()
    {
        var timedOut = _inFlight.Values
            .Where(m => DateTime.UtcNow > m.VisibilityDeadline)
            .ToList();

        foreach (var message in timedOut)
        {
            _inFlight.Remove(message.Id);
            _queue.Enqueue(message); // consumer crashed — redeliver
        }
    }

    public int Count => _queue.Count;
    public int InFlightCount => _inFlight.Count;
    public Action<QueueMessage<T>>? OnDeadLetter { get; set; }
}

public class QueueMessage<T>
{
    public string Id { get; set; } = "";
    public T Body { get; set; } = default!;
    public DateTime EnqueuedAt { get; set; }
    public DateTime VisibilityDeadline { get; set; }
    public int ReceiveCount { get; set; }
}
```

**Key design decisions to explain:**
- Visibility timeout means "in-flight" messages are hidden but not deleted — crash safety
- Acknowledge removes permanently — only called on success
- Reject requeues — consumer failed, another worker retries
- ReceiveCount tracks retries → DLQ after max retries

### Question 2: Implement a pub/sub event bus

> "Multiple services need to react to booking events. Implement a pub/sub system."

```csharp
public class EventBus
{
    // topic → list of subscriber handlers
    private readonly Dictionary<string, List<Func<object, Task>>> _subscribers = new();
    private readonly object _lock = new();

    // Service subscribes to a topic
    public void Subscribe<T>(string topic, Func<T, Task> handler)
    {
        lock (_lock)
        {
            if (!_subscribers.ContainsKey(topic))
                _subscribers[topic] = new List<Func<object, Task>>();

            // Wrap typed handler in untyped func
            _subscribers[topic].Add(async (msg) =>
            {
                if (msg is T typedMsg)
                    await handler(typedMsg);
            });
        }
    }

    // Publisher sends event to all subscribers
    public async Task PublishAsync<T>(string topic, T message)
    {
        List<Func<object, Task>> handlers;

        lock (_lock)
        {
            if (!_subscribers.TryGetValue(topic, out var subs))
                return; // no subscribers — fire and forget

            handlers = new List<Func<object, Task>>(subs);
        }

        // Deliver to all subscribers in parallel
        var tasks = handlers.Select(async handler =>
        {
            try
            {
                await handler(message!);
            }
            catch (Exception ex)
            {
                // One subscriber failing doesn't affect others
                Console.Error.WriteLine($"Subscriber failed for topic {topic}: {ex.Message}");
            }
        });

        await Task.WhenAll(tasks);
    }
}

// Usage — wiring up AXS services
var bus = new EventBus();

// Email service subscribes
bus.Subscribe<BookingCreatedEvent>("booking.created", async (evt) =>
{
    await emailService.SendConfirmationAsync(evt.UserId, evt.BookingId);
});

// Analytics service subscribes
bus.Subscribe<BookingCreatedEvent>("booking.created", async (evt) =>
{
    await analytics.RecordBookingAsync(evt.EventId, evt.Amount);
});

// Fraud detection subscribes
bus.Subscribe<BookingCreatedEvent>("booking.created", async (evt) =>
{
    await fraudDetection.CheckBookingAsync(evt.UserId, evt.Amount);
});

// Booking service publishes — doesn't know about subscribers
await bus.PublishAsync("booking.created", new BookingCreatedEvent
{
    BookingId = 8821,
    UserId = 42,
    EventId = 123,
    Amount = 250.00m
});
// All three subscribers notified in parallel. Booking service moves on.
```

### Question 3: Implement rate-limited queue consumer *(producer-consumer pattern)*

> "Process booking messages but never exceed 1,000/second to protect the database."

```csharp
public class RateLimitedConsumer<T>
{
    private readonly MessageQueue<T> _queue;
    private readonly Func<T, Task> _processor;
    private readonly int _maxPerSecond;
    private readonly SemaphoreSlim _rateLimiter;

    public RateLimitedConsumer(
        MessageQueue<T> queue,
        Func<T, Task> processor,
        int maxPerSecond)
    {
        _queue = queue;
        _processor = processor;
        _maxPerSecond = maxPerSecond;
        _rateLimiter = new SemaphoreSlim(maxPerSecond, maxPerSecond);
    }

    public async Task StartAsync(CancellationToken cancellationToken)
    {
        // Restore rate limiter tokens every second
        _ = Task.Run(async () =>
        {
            while (!cancellationToken.IsCancellationRequested)
            {
                await Task.Delay(1000, cancellationToken);

                // Refill to max — release however many were consumed
                int toRelease = _maxPerSecond - _rateLimiter.CurrentCount;
                if (toRelease > 0)
                    _rateLimiter.Release(toRelease);
            }
        }, cancellationToken);

        // Consume messages respecting rate limit
        while (!cancellationToken.IsCancellationRequested)
        {
            var message = _queue.Receive();

            if (message == null)
            {
                await Task.Delay(100, cancellationToken); // queue empty, wait
                continue;
            }

            // Acquire rate limit token (blocks if at limit)
            await _rateLimiter.WaitAsync(cancellationToken);

            // Process without awaiting — move to next message immediately
            _ = Task.Run(async () =>
            {
                try
                {
                    await _processor(message.Body);
                    _queue.Acknowledge(message.Id);
                }
                catch (Exception ex)
                {
                    Console.Error.WriteLine($"Processing failed: {ex.Message}");
                    _queue.Reject(message.Id);
                }
            }, cancellationToken);
        }
    }
}

// Usage
var consumer = new RateLimitedConsumer<BookingMessage>(
    queue: bookingQueue,
    processor: async (booking) =>
    {
        await bookingService.ProcessAsync(booking);
    },
    maxPerSecond: 1000  // protect database
);

await consumer.StartAsync(stoppingToken);
```

### Question 4: Implement the outbox pattern *(advanced, impresses interviewers)*

> "Guarantee that every database write also publishes a message — no dual-write inconsistency."

The dual-write problem:

```
// Naive approach — NOT safe
await _db.SaveBookingAsync(booking);           // Step 1: DB write
await _queue.PublishAsync("booking_created");  // Step 2: Queue publish

// If server crashes between step 1 and 2:
// → Booking exists in DB
// → Message never published
// → Email never sent, analytics never updated
// → Inconsistent state
```

The outbox pattern solution:

```csharp
public class BookingServiceWithOutbox
{
    private readonly DbContext _db;

    public async Task CreateBookingAsync(BookingRequest request)
    {
        using var transaction = await _db.Database.BeginTransactionAsync();

        try
        {
            // Step 1: Create the booking
            var booking = new Booking
            {
                UserId = request.UserId,
                SeatId = request.SeatId,
                AmountPaid = request.Amount,
                Status = "confirmed"
            };
            _db.Bookings.Add(booking);

            // Step 2: Write outbox message IN THE SAME TRANSACTION
            var outboxMessage = new OutboxMessage
            {
                Id = Guid.NewGuid(),
                Topic = "booking.created",
                Payload = JsonSerializer.Serialize(new BookingCreatedEvent
                {
                    BookingId = booking.Id,
                    UserId = booking.UserId,
                    Amount = booking.AmountPaid
                }),
                CreatedAt = DateTime.UtcNow,
                Published = false
            };
            _db.OutboxMessages.Add(outboxMessage);

            // Both booking AND outbox message committed atomically
            // If this fails → neither exists → consistent state
            // If this succeeds → both exist → consistent state
            await _db.SaveChangesAsync();
            await transaction.CommitAsync();
        }
        catch
        {
            await transaction.RollbackAsync();
            throw;
        }
    }
}

// Separate background worker publishes outbox messages to queue
public class OutboxPublisher : BackgroundService
{
    private readonly DbContext _db;
    private readonly IMessageQueue _queue;

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            // Find unpublished outbox messages
            var pending = await _db.OutboxMessages
                .Where(m => !m.Published)
                .OrderBy(m => m.CreatedAt)
                .Take(100)
                .ToListAsync(stoppingToken);

            foreach (var message in pending)
            {
                try
                {
                    await _queue.PublishAsync(message.Topic, message.Payload);

                    message.Published = true;
                    message.PublishedAt = DateTime.UtcNow;
                    await _db.SaveChangesAsync(stoppingToken);
                }
                catch (Exception ex)
                {
                    // Queue unavailable — will retry on next poll
                    Console.Error.WriteLine($"Outbox publish failed: {ex.Message}");
                }
            }

            await Task.Delay(1000, stoppingToken); // poll every second
        }
    }
}
```

**Why this works:** The booking and its outbox message are written in a single ACID transaction. Either both exist or neither does. The outbox publisher can fail and retry safely — publishing is idempotent (consumers handle duplicate messages with idempotency keys).

This is a production-grade pattern used at scale. Mentioning it without being prompted signals senior-level thinking.

## One-Line Summary for Interviews

> **A message queue is a durable, ordered buffer between producers and consumers that solves four problems simultaneously: load leveling by absorbing traffic spikes, service decoupling by letting services fail independently, reliability via at-least-once delivery with acknowledgment-based removal, and fan-out via pub/sub where one event reaches many subscribers — Kafka extends this with a persistent log enabling consumer groups for competing workers, independent subscriber groups for fan-out, offset management for crash recovery, and message replay for new services or reprocessing bugs — and the outbox pattern is the production solution for guaranteeing that database writes and message publishing stay consistent without dual-write risk.**
