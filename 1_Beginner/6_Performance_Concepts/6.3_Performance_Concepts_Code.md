## Q1. Implement a Circuit Breaker

> Wrap a remote service call with a circuit breaker. Track failures and open the circuit when a threshold is exceeded. After a timeout, probe once (half-open). If the probe succeeds, close the circuit. If it fails, stay open.

### The Three States
```
CLOSED   → requests flow through, failures counted
OPEN     → requests fail immediately (no call made), service gets recovery time
HALF-OPEN → one probe request allowed through to test if service recovered

Transition:
  CLOSED  → failures >= threshold           → OPEN
  OPEN    → timeout elapsed                 → HALF-OPEN
  HALF-OPEN → probe succeeds               → CLOSED
  HALF-OPEN → probe fails                  → OPEN (reset timer)
```

```csharp
using System;

public class CircuitBreaker
{
    public enum State { Closed, Open, HalfOpen }

    private State _state = State.Closed;
    private int _failureCount = 0;
    private DateTime _openedAt;

    private readonly int _failureThreshold;
    private readonly TimeSpan _recoveryTimeout;

    public State CurrentState => _state;

    public CircuitBreaker(int failureThreshold = 3, int recoveryTimeoutSeconds = 10)
    {
        _failureThreshold = failureThreshold;
        _recoveryTimeout  = TimeSpan.FromSeconds(recoveryTimeoutSeconds);
    }

    public T Execute<T>(Func<T> action, Func<T>? fallback = null)
    {
        switch (_state)
        {
            case State.Open:
                if (DateTime.UtcNow - _openedAt >= _recoveryTimeout)
                {
                    TransitionTo(State.HalfOpen);
                    return TryExecute(action, fallback, isProbe: true);
                }
                Console.WriteLine($"  [CB] OPEN — failing fast (no call made)");
                return UseFallback(fallback);

            case State.HalfOpen:
                return TryExecute(action, fallback, isProbe: true);

            default: // Closed
                return TryExecute(action, fallback, isProbe: false);
        }
    }

    private T TryExecute<T>(Func<T> action, Func<T>? fallback, bool isProbe)
    {
        try
        {
            T result = action();
            OnSuccess(isProbe);
            return result;
        }
        catch (Exception ex)
        {
            OnFailure(isProbe, ex.Message);
            return UseFallback(fallback);
        }
    }

    private void OnSuccess(bool isProbe)
    {
        _failureCount = 0;
        if (_state != State.Closed)
        {
            TransitionTo(State.Closed);
        }
    }

    private void OnFailure(bool isProbe, string reason)
    {
        _failureCount++;
        Console.WriteLine($"  [CB] Failure #{_failureCount}: {reason}");

        if (_state == State.HalfOpen || _failureCount >= _failureThreshold)
        {
            _openedAt = DateTime.UtcNow;
            TransitionTo(State.Open);
        }
    }

    private void TransitionTo(State newState)
    {
        Console.WriteLine($"  [CB] {_state} → {newState}");
        _state = newState;
    }

    private static T UseFallback<T>(Func<T>? fallback)
    {
        if (fallback != null) return fallback();
        throw new InvalidOperationException("Circuit open — service unavailable, no fallback provided");
    }
}

// Usage — payment service that starts failing
var cb = new CircuitBreaker(failureThreshold: 3, recoveryTimeoutSeconds: 2);

int callCount = 0;
string CallPaymentService()
{
    callCount++;
    if (callCount <= 5) throw new Exception("Connection timeout");
    return "Payment processed";
}

string Fallback() => "Payment queued for retry";

Console.WriteLine("=== Calls 1-3: failures accumulate ===");
for (int i = 0; i < 3; i++)
    Console.WriteLine($"Result: {cb.Execute(CallPaymentService, Fallback)}");

Console.WriteLine($"\n=== Calls 4-5: circuit OPEN, failing fast ===");
for (int i = 0; i < 2; i++)
    Console.WriteLine($"Result: {cb.Execute(CallPaymentService, Fallback)}");

Console.WriteLine($"\n=== Wait for recovery timeout ===");
System.Threading.Thread.Sleep(2100);

Console.WriteLine($"\n=== Call 6: HALF-OPEN probe (still failing) ===");
Console.WriteLine($"Result: {cb.Execute(CallPaymentService, Fallback)}");

Console.WriteLine($"\n=== Call 7: HALF-OPEN probe (service recovered) ===");
Console.WriteLine($"Result: {cb.Execute(CallPaymentService, Fallback)}"); // callCount=7 > 5 → succeeds

Console.WriteLine($"\n=== Call 8: back to CLOSED ===");
Console.WriteLine($"Result: {cb.Execute(CallPaymentService, Fallback)}");

// Output:
// === Calls 1-3: failures accumulate ===
//   [CB] Failure #1: Connection timeout
//   [CB] Failure #2: Connection timeout
//   [CB] Failure #3: Connection timeout
//   [CB] Closed → Open
// Result: Payment queued for retry  (×3)
//
// === Calls 4-5: circuit OPEN, failing fast ===
//   [CB] OPEN — failing fast (no call made)
// Result: Payment queued for retry  (×2)
//
// === Call 6: HALF-OPEN probe (still failing) ===
//   [CB] Open → HalfOpen
//   [CB] Failure #4: Connection timeout
//   [CB] HalfOpen → Open
// Result: Payment queued for retry
//
// === Call 7: HALF-OPEN probe (service recovered) ===
//   [CB] Open → HalfOpen
//   [CB] HalfOpen → Closed
// Result: Payment processed
```

### Why Circuit Breaker Beats Unlimited Retries
```
Without circuit breaker:
  Service B is down for 30s
  Service A retries every 500ms → 60 retry storms hit Service B
  Service B's thread pool exhausted → Service C calls A → A fails → cascades

With circuit breaker:
  After 3 failures → OPEN → all calls fail fast (no network calls made)
  Service B gets 30s of zero traffic to recover
  After timeout → HALF-OPEN → one probe → recovers cleanly
```

**Complexity:** Execute O(1) — all state transitions are O(1)

## Q2. Implement an Idempotency Key Store

> Clients send a unique key with each request. If the server has already processed that key, return the cached result instead of processing again. Keys expire after 24 hours. This prevents duplicate charges on retries.

```csharp
public class IdempotencyStore<TResult>
{
    private class Entry
    {
        public TResult Result      { get; init; } = default!;
        public DateTime ExpiresAt  { get; init; }
        public bool IsExpired      => DateTime.UtcNow >= ExpiresAt;
    }

    private readonly Dictionary<string, Entry> _store = new();
    private readonly TimeSpan _ttl;
    private readonly object _lock = new();

    public IdempotencyStore(TimeSpan? ttl = null)
    {
        _ttl = ttl ?? TimeSpan.FromHours(24);
    }

    // Execute the action only if the key hasn't been seen before.
    // Returns (result, wasDuplicate)
    public (TResult Result, bool WasDuplicate) Execute(string key, Func<TResult> action)
    {
        lock (_lock)
        {
            // Check for existing result
            if (_store.TryGetValue(key, out var existing))
            {
                if (!existing.IsExpired)
                {
                    Console.WriteLine($"  [IDEM] Key '{key}' already processed — returning cached result");
                    return (existing.Result, true);
                }
                // Expired — treat as new request
                _store.Remove(key);
            }

            // Process and store result
            Console.WriteLine($"  [IDEM] Key '{key}' is new — processing...");
            TResult result = action();

            _store[key] = new Entry
            {
                Result    = result,
                ExpiresAt = DateTime.UtcNow + _ttl
            };

            return (result, false);
        }
    }

    public int ActiveKeys => _store.Count(kv => !kv.Value.IsExpired);
}

// Usage — payment processor with idempotency
var store = new IdempotencyStore<string>(ttl: TimeSpan.FromSeconds(10));

int chargeCount = 0;
string ChargeCard(string amount)
{
    chargeCount++;
    return $"txn_{chargeCount:000} — charged {amount}";
}

Console.WriteLine("=== First request (client generates key) ===");
string key = "client-req-550e8400";
var (result1, dup1) = store.Execute(key, () => ChargeCard("$99.99"));
Console.WriteLine($"Result: {result1}  (duplicate: {dup1})");

Console.WriteLine("\n=== Network failure — client retries with SAME key ===");
var (result2, dup2) = store.Execute(key, () => ChargeCard("$99.99"));
Console.WriteLine($"Result: {result2}  (duplicate: {dup2})");
Console.WriteLine($"Charge count: {chargeCount}  ← still 1, no double charge!");

Console.WriteLine("\n=== Different key = different request ===");
var (result3, dup3) = store.Execute("client-req-661f9511", () => ChargeCard("$49.99"));
Console.WriteLine($"Result: {result3}  (duplicate: {dup3})");
Console.WriteLine($"Charge count: {chargeCount}  ← now 2, correct");

Console.WriteLine("\n=== Same key retried many times (safe) ===");
for (int i = 0; i < 5; i++)
    store.Execute(key, () => ChargeCard("$99.99"));
Console.WriteLine($"Charge count after 5 retries: {chargeCount}  ← still 2!");

// Output:
// === First request ===
//   [IDEM] Key 'client-req-550e8400' is new — processing...
// Result: txn_001 — charged $99.99  (duplicate: False)
//
// === Network failure — client retries with SAME key ===
//   [IDEM] Key 'client-req-550e8400' already processed — returning cached result
// Result: txn_001 — charged $99.99  (duplicate: True)
// Charge count: 1  ← still 1, no double charge!
//
// === Different key = different request ===
//   [IDEM] Key 'client-req-661f9511' is new — processing...
// Result: txn_002 — charged $49.99  (duplicate: False)
// Charge count: 2  ← now 2, correct
```

### Key Design Points
```
1. Key generation — client's responsibility:
   → Use UUID v4: Guid.NewGuid().ToString()
   → Or hash of (user_id + amount + timestamp_minute)
   → Must be unique per logical request, stable across retries

2. TTL — how long to keep keys:
   → Stripe: 24 hours
   → AWS SQS: 5 minutes (MessageDeduplicationId)
   → Match your retry window — if clients retry for up to 1 hour, TTL >= 1 hour

3. Storage — where to store keys:
   → Redis with SETNX (set if not exists) — atomic, built-in TTL
   → Database with unique constraint on idempotency_key column
   → In-memory (this impl) — lost on restart, only for demo

4. Database pattern (production):
   INSERT INTO idempotency_keys (key, result, expires_at)
   VALUES (@key, @result, @expires)
   ON CONFLICT (key) DO NOTHING;
   -- Then SELECT result WHERE key = @key
```

**Complexity:** Execute O(1) with hash map, O(1) with Redis SETNX

## Q3. Prevent Cache Stampede with a Mutex Lock

> When a popular cache key expires, thousands of requests simultaneously miss the cache and hammer the database. Implement a mutex that lets only the first request fetch from DB while others wait for the result.

### The Problem
```
t=0: Cache key "leaderboard" expires (TTL hit)
t=0: 10,000 concurrent requests all get a cache miss
t=0: All 10,000 hit the database simultaneously
     → DB overwhelmed → latency spikes → system degrades

This is called "Cache Stampede" or "Thundering Herd"
```

```csharp
using System.Collections.Concurrent;

public class StampedeProtectedCache<TKey, TValue> where TKey : notnull
{
    private class CacheEntry
    {
        public TValue Value      { get; init; } = default!;
        public DateTime ExpiresAt { get; init; }
        public bool IsExpired    => DateTime.UtcNow >= ExpiresAt;
    }

    private readonly ConcurrentDictionary<TKey, CacheEntry> _cache = new();
    // One lock object per key — only one fetcher per key, not one lock for all keys
    private readonly ConcurrentDictionary<TKey, SemaphoreSlim> _locks = new();

    public async Task<TValue> GetOrFetchAsync(TKey key, Func<Task<TValue>> fetch, TimeSpan ttl)
    {
        // Fast path — cache hit (no lock needed)
        if (_cache.TryGetValue(key, out var entry) && !entry.IsExpired)
        {
            Console.WriteLine($"  [CACHE] HIT  key={key}");
            return entry.Value;
        }

        // Slow path — acquire per-key lock
        var keyLock = _locks.GetOrAdd(key, _ => new SemaphoreSlim(1, 1));

        await keyLock.WaitAsync();
        try
        {
            // Double-check: another thread may have populated cache while we waited
            if (_cache.TryGetValue(key, out entry) && !entry.IsExpired)
            {
                Console.WriteLine($"  [CACHE] HIT  key={key} (populated while waiting for lock)");
                return entry.Value;
            }

            // We are the one fetcher — go to DB
            Console.WriteLine($"  [DB]    FETCH key={key} (cache miss — fetching from source)");
            TValue value = await fetch();

            _cache[key] = new CacheEntry { Value = value, ExpiresAt = DateTime.UtcNow + ttl };
            return value;
        }
        finally
        {
            keyLock.Release();
        }
    }

    public void Invalidate(TKey key) => _cache.TryRemove(key, out _);
}

// Usage — simulate 10 concurrent requests for an expired leaderboard
var cache = new StampedeProtectedCache<string, string>();

int dbCallCount = 0;
async Task<string> FetchLeaderboardFromDB()
{
    dbCallCount++;
    await Task.Delay(100); // simulate DB query time
    return "Alice:100, Bob:80, Charlie:60";
}

Console.WriteLine("=== 10 concurrent requests, cold cache ===");
var tasks = Enumerable.Range(1, 10).Select(async i =>
{
    string result = await cache.GetOrFetchAsync(
        "leaderboard",
        FetchLeaderboardFromDB,
        ttl: TimeSpan.FromSeconds(30)
    );
    Console.WriteLine($"  Request {i:00}: {result}");
});
await Task.WhenAll(tasks);

Console.WriteLine($"\nDB calls made: {dbCallCount}  ← should be 1, not 10!");

Console.WriteLine("\n=== Next 10 requests hit cache ===");
dbCallCount = 0;
await Task.WhenAll(Enumerable.Range(1, 10).Select(async i =>
    await cache.GetOrFetchAsync("leaderboard", FetchLeaderboardFromDB, TimeSpan.FromSeconds(30))
));
Console.WriteLine($"DB calls made: {dbCallCount}  ← 0, all served from cache");

// Output:
// === 10 concurrent requests, cold cache ===
//   [DB]    FETCH key=leaderboard (cache miss — fetching from source)
//   [CACHE] HIT  key=leaderboard (populated while waiting for lock)  ×9
//   Request 01: Alice:100, Bob:80, Charlie:60  ×10
//
// DB calls made: 1  ← should be 1, not 10!
```

### Three Stampede Prevention Strategies
```
1. Mutex lock (this implementation):
   → Only 1 thread fetches; others wait then read from cache
   → Pro: simple, exact
   → Con: waiting threads add latency

2. Probabilistic early expiration (PER):
   → Before key actually expires, some requests start refreshing it early
   → Formula: refresh if (ttl - elapsed) < -log(random()) × beta
   → Pro: no locking, zero wait
   → Con: slightly more DB calls (staggered instead of stampede)

3. Background refresh:
   → Serve stale data while a background job refreshes the cache
   → "stale-while-revalidate" HTTP header does this for CDNs
   → Pro: zero latency impact, no lock needed
   → Con: clients briefly see stale data
```

**Complexity:** GetOrFetchAsync O(1) cache hit, O(fetch time) on miss — only 1 fetcher per key at a time

## Q4. Compute Latency Percentiles (p50, p95, p99)

> Given a list of request latency samples, compute p50, p95, and p99. This is how engineers measure "tail latency" — the worst-case experience that SLA guarantees are built around.

```csharp
public static class LatencyAnalyzer
{
    public static double Percentile(List<double> samples, double percentile)
    {
        if (samples.Count == 0) throw new ArgumentException("No samples");
        if (percentile < 0 || percentile > 100) throw new ArgumentOutOfRangeException(nameof(percentile));

        var sorted = new List<double>(samples);
        sorted.Sort();

        // Nearest-rank method
        double rank = (percentile / 100.0) * sorted.Count;
        int index   = (int)Math.Ceiling(rank) - 1;
        index       = Math.Max(0, Math.Min(index, sorted.Count - 1));

        return sorted[index];
    }

    public static void PrintReport(List<double> samples, string label = "Latency Report")
    {
        var sorted = new List<double>(samples);
        sorted.Sort();

        double avg  = samples.Average();
        double min  = sorted[0];
        double max  = sorted[^1];
        double p50  = Percentile(samples, 50);
        double p95  = Percentile(samples, 95);
        double p99  = Percentile(samples, 99);
        double p999 = Percentile(samples, 99.9);

        Console.WriteLine($"\n=== {label} ({samples.Count} samples) ===");
        Console.WriteLine($"  Min:   {min,8:F1}ms");
        Console.WriteLine($"  Avg:   {avg,8:F1}ms   ← misleading if distribution is skewed");
        Console.WriteLine($"  p50:   {p50,8:F1}ms   ← median: half of requests are faster");
        Console.WriteLine($"  p95:   {p95,8:F1}ms   ← 95% of requests complete within this");
        Console.WriteLine($"  p99:   {p99,8:F1}ms   ← tail latency: SLA is usually set here");
        Console.WriteLine($"  p99.9: {p999,8:F1}ms   ← 1 in 1000 requests is this slow");
        Console.WriteLine($"  Max:   {max,8:F1}ms");
    }

    // Simulate a realistic latency distribution: fast most of the time, occasional spikes
    public static List<double> SimulateLatencies(int count, Random? rng = null)
    {
        rng ??= new Random(42);
        var samples = new List<double>(count);

        for (int i = 0; i < count; i++)
        {
            double u = rng.NextDouble();
            double latency = u switch
            {
                < 0.70 => 20 + rng.NextDouble() * 30,    // 70%: fast  20–50ms
                < 0.95 => 50 + rng.NextDouble() * 150,   // 25%: medium 50–200ms
                < 0.99 => 200 + rng.NextDouble() * 600,  //  4%: slow  200–800ms
                _      => 800 + rng.NextDouble() * 4200   //  1%: spike 800–5000ms
            };
            samples.Add(latency);
        }
        return samples;
    }
}

// Usage
var samples = LatencyAnalyzer.SimulateLatencies(10_000);
LatencyAnalyzer.PrintReport(samples, "API Gateway Latency");

// Demonstrate why average is misleading
Console.WriteLine("\n=== Why average lies ===");
var skewed = new List<double> { 10, 10, 10, 10, 10, 10, 10, 10, 10, 5000 };
LatencyAnalyzer.PrintReport(skewed, "10 requests (1 spike)");

// Output:
// === API Gateway Latency (10000 samples) ===
//   Min:       20.0ms
//   Avg:       96.4ms   ← misleading if distribution is skewed
//   p50:       38.2ms   ← median: half of requests are faster
//   p95:      187.3ms   ← 95% of requests complete within this
//   p99:      673.1ms   ← tail latency: SLA is usually set here
//   p99.9:   3241.8ms   ← 1 in 1000 requests is this slow
//   Max:     4987.2ms
//
// === 10 requests (1 spike) ===
//   Avg:      501.0ms   ← 1 spike inflates average 50x
//   p50:       10.0ms   ← half the requests are lightning fast
//   p99:     5000.0ms   ← the spike shows up here, not in average
```

### Why p99 Matters More Than Average
```
Scenario: microservice fans out to 10 backends

Each backend: p99 = 500ms
Probability that at least one backend takes >500ms:
  = 1 - (0.99)^10
  = 1 - 0.904
  = 9.6% of all user requests hit >500ms latency

At 1000 req/s → 96 users/second experience a slow response

Rule of thumb: your system's p99 ≈ your worst backend's p999
→ Always optimize tail latency, not average latency
→ SLA: "p99 < 200ms" is much more meaningful than "avg < 50ms"
```

**Complexity:** Percentile O(n log n) — dominated by sorting; O(1) after sort for each percentile query

## Complexity Summary

| Algorithm | Key Operation | Time | What It Prevents |
|-----------|--------------|------|-----------------|
| Circuit Breaker | Execute | O(1) | Cascading failures |
| Idempotency Store | Execute | O(1) | Duplicate processing |
| Stampede Protection | GetOrFetchAsync | O(1) hit, O(fetch) miss | Thundering herd on DB |
| Percentile Calculator | Percentile | O(n log n) | Misleading average metrics |

## What Interviewers Test

```
1. Circuit Breaker — do you implement the double-check in HALF-OPEN?
   → HALF-OPEN must send exactly one probe, then decide
   → Common bug: forgetting to reset failure count on success

2. Circuit Breaker — what is the fallback strategy?
   → Return stale cached value, return default, queue for retry
   → "Fail fast with graceful degradation" is the key phrase

3. Idempotency — who generates the key, client or server?
   → Client generates — it must be stable across retries
   → Server-generated keys can't help if the first response is lost

4. Idempotency — what if two concurrent requests arrive with the same key?
   → Lock (mutex) or database unique constraint ensures only one processes
   → This implementation uses lock(_lock) for thread safety

5. Cache stampede — what is the double-check pattern?
   → After acquiring lock, check cache AGAIN before fetching from DB
   → Without it, 2nd thread to acquire lock fetches DB again unnecessarily

6. Percentiles — why p99 over average for SLAs?
   → Average is pulled down by fast requests, hides spikes
   → p99 = worst 1% — the experience your slowest users get
   → Fan-out: 10 services at p99=100ms → ~10% of requests take >100ms
```
