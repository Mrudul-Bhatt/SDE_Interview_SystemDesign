# Circuit Breaker

---

## What Problem Does It Solve?

In a distributed system, services call other services over the network. Networks fail. Services crash. Databases become slow.

Without any protection, a slow downstream service causes a **cascading failure**:

```
[Order Service] → calls → [Payment Service]  ← Payment Service is slow (DB overloaded)

What happens:
1. Order Service waits... (default timeout: 30 seconds)
2. New requests keep arriving — each one also waits 30 seconds
3. Order Service's thread pool fills up waiting for Payment Service
4. Order Service itself starts timing out for its callers
5. API Gateway waits for Order Service...
6. Entire system hangs

         [User] → [API Gateway] → [Order Service] → [Payment Service]
                       ↑                ↑                   ↑
                   Now slow          Now slow           Slow/down
                   (cascade)         (cascade)          (root cause)
```

The root cause is one service. The result is total system failure.

**The circuit breaker stops the cascade.** When Payment Service starts failing, Order Service immediately returns an error instead of waiting — protecting its own threads, and preventing the failure from spreading upward.

## The Electrical Analogy

The name comes from an electrical circuit breaker — the device in your fuse box that trips when there's an overload.

```
Electrical circuit breaker:
  Normal:   current flows freely         → CLOSED (circuit is connected)
  Overload: too much current             → OPEN   (circuit is disconnected, stops the surge)
  Reset:    test if issue is resolved    → try again carefully

Software circuit breaker:
  Normal:   requests flow to service     → CLOSED
  Failures: too many errors/timeouts     → OPEN   (fail immediately, don't call service)
  Recovery: test if service is healthy   → HALF-OPEN (send a probe request)
```

## The Three States

### State 1: CLOSED (normal operation)

```
[Caller] ──────────────────────────────→ [Downstream Service]
           requests flow through freely
           failures are counted
```

- All requests pass through to the downstream service
- A **failure counter** tracks errors and timeouts
- If failures exceed the **threshold** within a time window → trip to OPEN
- On success, failure counter resets (or decays)

```
CLOSED state — failure counter:
  Request 1: SUCCESS → counter = 0
  Request 2: SUCCESS → counter = 0
  Request 3: TIMEOUT → counter = 1
  Request 4: ERROR   → counter = 2
  Request 5: TIMEOUT → counter = 3  ← threshold hit → TRIP TO OPEN
```

---

### State 2: OPEN (failing fast)

```
[Caller] ──→ [Circuit Breaker] ──✗──→ [Downstream Service]
               returns error               NOT called
               immediately
```

- **All requests are immediately rejected** without calling the downstream service
- The caller gets a fast error (e.g. `CircuitOpenException`) instead of waiting for a timeout
- A **reset timer** starts (e.g. 30 seconds)
- When the timer expires → move to HALF-OPEN

**Why fail fast instead of waiting?**

```
Without circuit breaker (slow fail):
  Each request waits 30s for timeout
  100 concurrent users × 30s = 3,000 thread-seconds wasted
  Thread pool exhausted → system unresponsive

With circuit breaker (fast fail):
  Each request fails in < 1ms
  100 concurrent users × 1ms = 0.1 thread-seconds
  Thread pool free for healthy services
```

---

### State 3: HALF-OPEN (probing for recovery)

```
[Caller] ──→ [Circuit Breaker] ──→ [Downstream Service]
               allows 1 probe request through
               watching the result
```

- After the reset timer expires, the circuit moves to HALF-OPEN
- A **single probe request** (or a small number) is allowed through
- If the probe **succeeds** → reset failure counter → return to CLOSED
- If the probe **fails** → immediately return to OPEN (restart the timer)

```
HALF-OPEN probe result:
  Probe request → SUCCESS → back to CLOSED (service recovered)
  Probe request → FAIL    → back to OPEN   (service still down)
```

## State Transition Diagram

```
                    failure threshold exceeded
         ┌─────────────────────────────────────────┐
         │                                         ▼
      ┌──────┐                                 ┌────────┐
      │      │     reset timer expires         │        │
      │CLOSED│◄────────────────────────────────│  OPEN  │
      │      │     (via HALF-OPEN probe)        │        │
      └──────┘                                 └────────┘
         ▲                                         │
         │         reset timer expires             │
         │    ┌──────────────────────────────────  │
         │    ▼                                    │
         │ ┌──────────┐     probe fails            │
         │ │          │──────────────────────────→ │
         └─│HALF-OPEN │                            │
           │          │
           └──────────┘
           probe succeeds → CLOSED
```

| From | Event | To |
|------|-------|----|
| CLOSED | Failures ≥ threshold | OPEN |
| OPEN | Reset timer expires | HALF-OPEN |
| HALF-OPEN | Probe succeeds | CLOSED |
| HALF-OPEN | Probe fails | OPEN |

## Configuration Parameters

Every circuit breaker has knobs to tune for your specific service:

```
CircuitBreaker(
    failureThreshold  = 5,       // how many failures before tripping
    failureWindow     = 60s,     // count failures within this window
    resetTimeout      = 30s,     // how long to stay OPEN before probing
    successThreshold  = 2,       // probes needed in HALF-OPEN before CLOSED
    timeout           = 2s       // how long to wait before counting as failure
)
```

**Tuning guidance:**

| Parameter | Too Low | Too High |
|-----------|---------|----------|
| `failureThreshold` | Trips on transient blips (false positive) | Slow to protect (too tolerant) |
| `failureWindow` | Stale failures trigger trips | Only recent failures matter |
| `resetTimeout` | Retries too soon, downstream still recovering | Slow recovery, unnecessary downtime |
| `successThreshold` | One lucky probe reopens circuit prematurely | Takes too long to recover |
| `timeout` | Slow responses not counted as failures | Waits too long before failing |

## Failure Rate vs Count-Based Tripping

**Count-based:** trip after N failures in a window.
```
threshold = 5 failures in 60 seconds
→ works well for low-volume services
→ problem: if 5 out of 5 requests fail (100% failure rate), trip
           if 5 out of 10,000 fail (0.05% failure rate), also trips — too sensitive
```

**Rate-based (more sophisticated):** trip after X% failure rate with minimum call volume.
```
threshold = 50% failure rate, minimum 10 calls
→ 5/5 requests fail but only 5 calls → not enough data, don't trip yet
→ 5,100/10,000 requests fail (51%) → trip
→ used by Netflix Hystrix, Resilience4j
```

## What Counts as a Failure?

Not all errors should count the same way:

```
SHOULD count as circuit breaker failure:
  ✗ HTTP 500, 502, 503, 504   (server errors)
  ✗ Connection timeout
  ✗ Read timeout
  ✗ Connection refused (service is down)

SHOULD NOT count as circuit breaker failure:
  ✓ HTTP 400 (bad request — caller's fault, not service's fault)
  ✓ HTTP 401, 403 (auth errors — not a service health issue)
  ✓ HTTP 404 (not found — resource doesn't exist, service is healthy)
```

Counting 4xx errors as failures would trip the circuit when the caller sends bad data — the downstream service is healthy, the caller is buggy.

## Fallback Strategies

When the circuit is OPEN, you need a fallback. Options:

**1. Return a cached response**
```
circuit open → return last known good result from cache
→ user sees slightly stale data rather than an error
→ best for: product listings, recommendations, user profiles
```

**2. Return a default/degraded response**
```
circuit open → return empty list, zero balance, "service unavailable"
→ partial functionality instead of total failure
→ best for: non-critical features (recommendations, social feed)
```

**3. Return an error immediately (fast fail)**
```
circuit open → throw CircuitOpenException → HTTP 503
→ caller knows to not retry and show an error page
→ best for: critical flows where stale data is dangerous (payments)
```

**4. Try a secondary service**
```
circuit open → route to backup service or degraded endpoint
→ best for: read traffic that can tolerate a replica
```

## Circuit Breaker vs Retry

These two patterns are often confused but serve different purposes:

```
Retry:           assumes the failure is transient — try again immediately
Circuit Breaker: assumes the service is degraded — stop calling it for a while

When to use Retry:
→ Transient network hiccup (packet loss, brief DNS failure)
→ The downstream service is healthy, just momentarily busy
→ Idempotent requests (safe to repeat)

When to use Circuit Breaker:
→ Downstream service is down or severely degraded
→ Every retry just adds load to an already-struggling service
→ You need to protect your own thread pool

Danger of Retry without Circuit Breaker:
→ Service is down for 5 minutes
→ Each caller retries 3× with exponential backoff
→ 1,000 users × 3 retries = 3,000 calls hammering a sick service
→ Prevents recovery (thundering herd on restart)

Best practice: use both together
→ Circuit breaker wraps retry
→ Retry handles transient (1-2 attempts)
→ Circuit breaker catches sustained failure (opens after N failures)
```

## Bulkhead Pattern (Companion to Circuit Breaker)

The circuit breaker stops calls to a failing service. The **bulkhead** limits how many concurrent calls can be in-flight to a service — isolating failures by resource.

```
Without bulkhead:
[Order Service] has 100 threads total
→ Payment Service is slow → 100 threads all waiting on Payment
→ Order Service can't serve ANY request, including those not touching Payment

With bulkhead:
[Order Service]
  → Payment Service thread pool: 10 threads max
  → Inventory Service thread pool: 10 threads max
  → Remaining: 80 threads for other work

→ Payment slowness consumes only 10 threads
→ Other 90 threads keep serving requests
```

Named after ship bulkheads — watertight compartments that prevent a single hull breach from sinking the entire ship.

|---------|----------|-------|
| **Resilience4j** | Java | Modern, Netflix Hystrix replacement, annotation-based |
| **Hystrix** | Java | Netflix original, now in maintenance mode |
| **Polly** | C# / .NET | De-facto standard in .NET, fluent API |

| **go-resilience** | Go | — |
| **AWS App Mesh** | Infrastructure | Service mesh with built-in circuit breaking |
| **Istio** | Infrastructure | Kubernetes service mesh, circuit breaking at the network layer |

Infrastructure-level circuit breaking (Istio, App Mesh) has the advantage that it's language-agnostic — no library needed in each service.

## Where Circuit Breakers Live in a System

```
[Client App]
      ↓
[API Gateway]          ← circuit breaker at gateway level (protects all downstream)
      ↓
[Order Service]
  ├─ CB → [Payment Service]    ← per-dependency circuit breaker
  ├─ CB → [Inventory Service]
  └─ CB → [Notification Service]
      ↓
[Payment Service]
  └─ CB → [Bank API]           ← even third-party calls should be wrapped
```

Rule of thumb: **every network call** is a candidate for a circuit breaker.

## Side-by-Side Comparison

| Dimension | Without Circuit Breaker | With Circuit Breaker |
|-----------|------------------------|---------------------|
| Downstream failure effect | Cascades up the call chain | Contained at the breaker |
| Thread pool during failure | Exhausted (all threads waiting) | Free (fast fail < 1ms) |
| Recovery | Manual restart often needed | Automatic via HALF-OPEN probe |
| User experience | Total hang, then timeout error | Immediate error or degraded response |
| Downstream load during outage | Constant retry storm | Zero calls (OPEN state) |
| Latency on failure | Full timeout (e.g. 30s) | Near-zero (< 1ms) |

## When to Use Circuit Breakers

```
Always use when:
✓ Making synchronous HTTP/gRPC calls to another service
✓ Calling a database or cache that could be slow/down
✓ Calling any third-party API (payment gateway, SMS provider)
✓ Any call where a downstream failure could consume threads

May not need when:
→ Pure async event-driven (producing to Kafka — Kafka handles backpressure)
→ Fire-and-forget calls with no thread waiting for response
→ Calls to in-process libraries (no network, no timeout risk)
```

## Interview Answer Framework

When asked "How does a circuit breaker work?" in a design round:

1. **State the problem:** cascading failures in distributed systems
2. **Name the three states:** CLOSED (normal), OPEN (fail fast), HALF-OPEN (probing)
3. **Explain the transition:** failure threshold → OPEN → timer → HALF-OPEN → probe → CLOSED or OPEN
4. **Mention fail-fast benefit:** protects thread pools, prevents cascade
5. **Add fallback strategy:** cache, default response, or fast 503
6. **Contrast with retry:** circuit breaker is for sustained failure, retry is for transient failure
7. **Name a tool:** Polly (.NET), Resilience4j (Java), or Istio (infra-level)
