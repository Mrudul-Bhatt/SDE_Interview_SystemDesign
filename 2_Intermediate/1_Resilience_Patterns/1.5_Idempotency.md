# Idempotency

---

## What Problem Does It Solve?

In distributed systems, networks are unreliable. A client sends a request, the server processes it, but the response is lost in transit. The client retries. Now the server processes it **twice**.

```
Without idempotency:
  Client                     Server                  Bank
    │──── POST /pay $100 ──────→│──── debit $100 ──→│
    │                           │     (success)      │
    │         (response lost)   │                    │
    │                           │                    │
    │──── POST /pay $100 ──────→│──── debit $100 ──→│  ← charged twice!
    │         (timeout retry)   │     (oops)         │

With idempotency key:
  Client                     Server                  Bank
    │── POST /pay $100          │                    │
    │   Idempotency-Key: abc123 │──── debit $100 ──→│
    │                           │  store result(abc123)
    │         (response lost)   │                    │
    │                           │                    │
    │── POST /pay $100 (retry)  │                    │
    │   Idempotency-Key: abc123 │  key found! return │
    │←────────── cached result ─│  stored result     │  ← no second charge
```

An operation is **idempotent** if executing it multiple times produces the same result as executing it once.

## Real-World Scenarios

```
1. Payment processing
   → User double-clicks "Pay" button
   → Network timeout → client retries
   → Server must not charge twice

2. Order placement
   → Mobile app loses connection after submit
   → App retries on reconnect
   → Warehouse must receive exactly one pick instruction

3. Email / notification sending
   → Notification service crashes mid-send
   → Retry job sends the same batch again
   → User must not receive duplicate emails

4. Inventory decrement
   → "Reserve 2 units" replayed twice = -4 units
   → Deduplication required even if the reservation was already done

5. Webhook processing
   → Stripe, GitHub, Twilio guarantee at-least-once delivery
   → Your handler MUST be idempotent — duplicates will arrive
```

## HTTP Methods and Idempotency

```
Method      Idempotent?   Safe?    Meaning
──────────────────────────────────────────────────────
GET         ✓ Yes         ✓ Yes    Read — no state change
HEAD        ✓ Yes         ✓ Yes    Read headers only
PUT         ✓ Yes         ✗ No     Replace entire resource
DELETE      ✓ Yes         ✗ No     Remove resource (2nd DELETE = already gone = same outcome)
OPTIONS     ✓ Yes         ✓ Yes    Capability check
POST        ✗ No          ✗ No     Create / action — new resource each time
PATCH       ✗ No*         ✗ No     Partial update — depends on implementation

* PATCH can be made idempotent if the operation is "set field X to Y"
  rather than "increment field X by 1"
```

```
PUT vs POST for idempotency:

POST /orders          → creates a new order each time (not idempotent)
PUT  /orders/order-99 → creates or replaces order-99 each time (idempotent)

The key insight: PUT specifies the target; POST lets the server decide.
```

## The Idempotency Key Pattern

The standard approach for making non-idempotent operations safe to retry:

```
1. Client generates a unique key before the request (UUID or hash of payload)
2. Client sends the key in every attempt: Idempotency-Key: <uuid>
3. Server stores (key → result) after first execution
4. On retry, server finds the key, returns the stored result without re-executing
5. Key expires after N hours/days (TTL)

Flow:
┌────────┐              ┌───────────┐          ┌──────────┐
│ Client │              │  Server   │          │   Redis  │
└────────┘              └───────────┘          └──────────┘
    │ POST /charge           │                      │
    │ Idempotency-Key: abc   │                      │
    │───────────────────────→│                      │
    │                        │── GET abc ──────────→│
    │                        │← null (not seen) ────│
    │                        │── process payment    │
    │                        │── SET abc result ───→│
    │                        │← stored ─────────────│
    │←── 200 OK result ──────│                      │
    │                        │                      │
    │ (network glitch, retry)│                      │
    │                        │                      │
    │ POST /charge           │                      │
    │ Idempotency-Key: abc   │                      │
    │───────────────────────→│                      │
    │                        │── GET abc ──────────→│
    │                        │← result (found!) ────│
    │←── 200 OK (same result)│                      │  ← no re-execution
```

### What to use as the key?

```
Option 1: Client-generated UUID (preferred)
  → Client creates key before sending
  → Key is opaque to the server
  → Client stores it in local state: { paymentId → idempotencyKey }
  → Stripe, Braintree, Adyen all use this approach

Option 2: Deterministic hash of the payload
  → SHA256(userId + amount + currency + timestamp_bucket)
  → Timestamp bucket: round to nearest 5 minutes
  → Pros: stateless client, same inputs = same key
  → Cons: legitimate duplicates (pay again) hash the same → rejected

Option 3: Business-level key (order ID, request ID)
  → Use the order ID itself as the idempotency key for order creation
  → "Create order-99" is idempotent if order-99 means the same thing
```

## Database Upsert Patterns

Idempotency at the persistence layer — no application-level key cache needed.

### INSERT ... ON CONFLICT DO NOTHING (PostgreSQL)

```sql
-- Idempotent insert: ignore if already exists
INSERT INTO payments (payment_id, user_id, amount, status)
VALUES ('pay-uuid-123', 42, 100.00, 'completed')
ON CONFLICT (payment_id) DO NOTHING;

-- Returns 0 rows affected on duplicate — not an error.
-- The original row is untouched.
```

### INSERT ... ON CONFLICT DO UPDATE (UPSERT)

```sql
-- Idempotent upsert: create or overwrite with same data
INSERT INTO idempotency_keys (key, response_body, created_at)
VALUES ('abc-123', '{"status":"ok"}', NOW())
ON CONFLICT (key) DO UPDATE
  SET response_body = EXCLUDED.response_body;
  -- Only updates if re-inserting with new data
```

### Conditional UPDATE (optimistic check)

```sql
-- Only mark as refunded if still in 'completed' state
-- Running twice: second UPDATE affects 0 rows → idempotent
UPDATE payments
SET status = 'refunded'
WHERE payment_id = 'pay-uuid-123'
  AND status = 'completed';
```

## Redis-Based Deduplication

Fast O(1) deduplication without hitting the database for every request.

```
Pattern: SET key result NX EX ttl

On first request:
  SET idempotency:abc-123 "processing" NX EX 3600
  → returns OK (key was not set)
  → process the operation
  → SET idempotency:abc-123 <serialized_result> EX 3600

On retry:
  SET idempotency:abc-123 "processing" NX EX 3600
  → returns nil (key already exists)
  → GET idempotency:abc-123 → return cached result

Edge case — in-flight request:
  First request: key = "processing" (not yet done)
  Retry arrives while first is still running
  → Returns 409 Conflict: "request in progress, try again shortly"
  → Client retries after delay → eventually gets final result
```

## Delivery Guarantees

```
At-most-once  → fire and forget; message may be lost; no duplicates
               Example: UDP, fire-and-forget logging

At-least-once → retried until acknowledged; duplicates possible
               Example: Kafka (default), SQS, most HTTP retries
               → Consumer MUST be idempotent

Exactly-once  → no loss, no duplicates; hardest to achieve
               Example: Kafka transactions + idempotent producer
               → Requires coordination between producer and consumer
               → Often approximated with at-least-once + idempotent consumer

In practice:
  Build for at-least-once delivery → write idempotent consumers.
  Exactly-once is expensive (2PC or Kafka transactions) — avoid unless necessary.
```

```
At-least-once delivery with idempotent consumer:
  Producer          Kafka         Consumer          DB
     │── publish ──→│             │                 │
     │              │── deliver ─→│── upsert ──────→│ row created
     │              │  (ack lost) │                 │
     │              │── redeliver→│── upsert ──────→│ same row, no-op
     │              │             │                 │  ← effectively exactly-once
```

## Idempotency vs Exactly-Once vs Deduplication

```
Term              Meaning
──────────────────────────────────────────────────────────────────
Idempotent        Operation can be repeated without changing outcome
                  Focus: the OPERATION itself (HTTP, business logic)

Exactly-once      Message is processed exactly once end-to-end
                  Focus: the DELIVERY PIPELINE (Kafka, SQS, MQ)

Deduplication     Detect and discard duplicate messages
                  Focus: the DETECTION MECHANISM (key store, bloom filter)

Relationship:
  Deduplication implements exactly-once semantics for the consumer.
  An idempotent operation doesn't need deduplication but is safe with it.
  Exactly-once = at-least-once delivery + idempotent consumer.
```

## Common Pitfalls

```
1. Using wall-clock time as the idempotency key
   → Two requests at "the same second" get the same key → false dedup
   → Or two retries 61 seconds apart get different keys → double execution
   → Always use client-generated UUID, not timestamps

2. Key expires before client retries
   → TTL too short → client retries after expiry → duplicate execution
   → Stripe uses 24-hour TTL; 1–24 hours is typical

3. Idempotent reads, non-idempotent side effects
   → GET /order/99 is safe, but it sends a "viewed" analytics event
   → The read is idempotent; the side effect may not be
   → Separate primary effect from side effects; make side effects idempotent too

4. Partial execution not handled
   → Server charges card → crashes → key never stored
   → Retry re-executes: card charged again
   → Fix: store key BEFORE executing, update to final result after
   → Or: use a distributed transaction / saga with compensation

5. Different clients, same key collision
   → Two different users happen to generate the same UUID (astronomically rare)
   → Scope the key: store as {userId}:{idempotencyKey}, not just {idempotencyKey}

6. Returning 200 vs 201 on replay
   → First request: 201 Created
   → Replay: return 200 OK (or 201 again — depends on API contract)
   → Always return the same response body (the original result)
```

## Idempotency Key Storage: Redis vs Database

```
             Redis               Database (PostgreSQL)
───────────────────────────────────────────────────────────────
Speed        O(1), sub-ms        O(log n) index lookup
Durability   In-memory (AOF/RDB) ACID, always durable
TTL          Native, automatic   Requires scheduled cleanup
Scale        Horizontal easily   Single master for writes
Complexity   Simple              Simple (one table)

When to use Redis:
  → High-throughput APIs (>1k req/s)
  → TTL-based expiry is important
  → Already have Redis in stack

When to use DB:
  → Consistency with the main transaction is critical
  → "Check and process" must be atomic (same DB transaction)
  → Simpler stack, fewer moving parts

Best pattern: same DB transaction
  BEGIN;
  INSERT INTO idempotency_keys (key, status) VALUES (?, 'processing')
    ON CONFLICT DO NOTHING;
  -- if 0 rows inserted: key existed → return cached result
  -- if 1 row inserted: proceed with business logic
  UPDATE payments SET ...;
  UPDATE idempotency_keys SET status='done', result=? WHERE key=?;
  COMMIT;
  → Atomic: either the whole thing happens or nothing does
```

## Idempotency in Message Queues

```
Kafka consumer idempotency:

Message: { orderId: 99, action: "ship" }

Pattern 1: Natural key upsert
  → INSERT INTO shipments (order_id, status) VALUES (99, 'shipped')
    ON CONFLICT (order_id) DO NOTHING;
  → Duplicate message: conflict, no-op ✓

Pattern 2: Message offset as idempotency key
  → Store Kafka offset alongside the processed record
  → Before processing: check if offset already seen
  → If seen: skip; if not: process + store offset atomically

Pattern 3: Outbox pattern (guaranteed exactly-once effect)
  → Write to DB + write to outbox table in same transaction
  → Separate relay process reads outbox and publishes to Kafka
  → On consumer side: idempotent upsert using the message ID
```

## Interview Answer Framework

When asked "How do you make a payment API idempotent?" in a design round:

1. **State the problem:** network failures cause retries → duplicate charges
2. **Introduce idempotency key:** client generates UUID before sending; included in every retry
3. **Server-side storage:** Redis `SET NX` or DB `INSERT ON CONFLICT DO NOTHING`
4. **Three outcomes:** key not seen → execute; key seen with result → return cached; key seen as in-flight → 409 retry later
5. **Scope the key:** namespace by user/tenant to prevent cross-user collisions
6. **TTL:** 24 hours is standard for payment APIs
7. **Atomic store:** store key BEFORE executing to handle crash between execution and storage
8. **Delivery guarantee:** most queues are at-least-once → idempotent consumers are mandatory
9. **Real-world example:** Stripe uses `Idempotency-Key` header; replays return 200 with original response body
