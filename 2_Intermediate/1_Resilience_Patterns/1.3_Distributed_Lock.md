# Distributed Lock

---

## What Problem Does It Solve?

A regular mutex (`lock` in C#, `synchronized` in Java) protects shared state within a **single process**. In a distributed system, the same resource is accessed by multiple processes on multiple servers simultaneously.

```
Single server — regular mutex works:
┌─────────────────────────────────┐
│  Server 1                       │
│  Thread A ──┐                   │
│             ├─→ lock → DB row   │
│  Thread B ──┘  (one at a time)  │
└─────────────────────────────────┘

Multiple servers — regular mutex is useless:
┌────────────┐        ┌────────────┐
│  Server 1  │        │  Server 2  │
│  Thread A  │        │  Thread B  │
│  lock() ✓  │        │  lock() ✓  │  ← both acquired their OWN local lock
│            │        │            │
│  read row  │        │  read row  │  ← both read "seats available: 1"
│  book seat │        │  book seat │  ← BOTH book the same seat
└────────────┘        └────────────┘
         Double-booking — data corruption
```

A **distributed lock** is a lock that all servers agree on — only one holder at a time, across the entire cluster.

---

## Real-World Scenarios

```
1. Seat / ticket booking
   → Only one user should claim the last seat on a flight
   → 10 servers handling requests simultaneously

2. Inventory decrement
   → "1 item left in stock" — two orders arrive at the same time
   → Both read stock=1, both decrement, stock goes to -1

3. Cron job deduplication
   → A batch job runs on 5 servers; should only execute on one
   → Without a lock, all 5 fire the job simultaneously

4. Leader election
   → Among 5 service instances, one must be the "leader"
   → Leader sends notifications, others stand by
   → First to acquire the lock becomes leader

5. Rate limiting (distributed)
   → Shared counter in Redis: all servers must increment atomically
```

---

## Properties a Distributed Lock Must Have

```
1. Mutual exclusion (Safety)
   → Only ONE client holds the lock at any moment
   → This is the whole point — violating this is a bug

2. Deadlock-free (Liveness)
   → Lock must eventually be released, even if the holder crashes
   → Achieved via TTL (time-to-live): lock auto-expires after N seconds

3. Fault-tolerant
   → Lock must survive the lock server losing some nodes
   → A lock held by a crashed client must not block forever

4. No self-deadlock
   → The lock holder can re-acquire its own lock (re-entrant)
   → Or: detecting re-entrancy and throwing instead of deadlocking
```

## Approach 1: Redis SETNX (Simple, Single Node)

Redis is the most common choice for distributed locks in practice. The key command:

```
SET lock_key unique_value NX EX 30
```

- `NX` — Set only if Not eXists (atomic check-and-set)
- `EX 30` — Expire in 30 seconds (prevents deadlock if holder crashes)
- `unique_value` — random UUID so only the holder can release it

```
Server 1:  SET booking:seat-A5 "uuid-abc" NX EX 30  → OK   (acquired)
Server 2:  SET booking:seat-A5 "uuid-xyz" NX EX 30  → nil  (already held)

Server 1 finishes, releases:
  GET booking:seat-A5 → "uuid-abc"  (confirm it's ours)
  DEL booking:seat-A5               (release)
```

### Why check the value before releasing?

```
Problem without value check:
  t=0:   Server 1 acquires lock (TTL=30s)
  t=25:  Server 1 gets GC paused (30+ seconds)
  t=30:  Lock EXPIRES automatically
  t=31:  Server 2 acquires the lock
  t=32:  Server 1 resumes, calls DEL → deletes Server 2's lock!

With value check:
  t=32:  Server 1: GET → "uuid-xyz" (not mine!) → does NOT delete
  Result: Server 2's lock is safe
```

### Atomic release with Lua script

The check-then-delete must be atomic — two separate commands can be interrupted:

```lua
-- Lua script runs atomically in Redis
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

A non-atomic approach (GET then DEL) has a race: another client could acquire between GET and DEL.

## Approach 2: Redlock (Redis Multi-Node)

Single-node Redis is a single point of failure. If Redis goes down, the lock is gone. **Redlock** acquires the lock on a majority of N independent Redis nodes.

```
Setup: 5 independent Redis nodes (no replication between them)

Acquire:
  t1 = current time
  Try SET NX EX on all 5 nodes
  Count how many succeeded

  If successes >= 3 (majority) AND (current_time - t1) < TTL:
      Lock acquired ✓
  Else:
      Release on all nodes, try again later

Release:
  Run the Lua script on ALL 5 nodes
```

### Why majority (quorum)?

```
5 nodes, acquire on 3:
  Node 1: acquired ✓
  Node 2: acquired ✓
  Node 3: acquired ✓
  Node 4: failed (network partition)
  Node 5: failed (node down)

Client A holds the lock (3/5 = majority).

Client B tries:
  Node 1: already held → fail
  Node 2: already held → fail
  Node 3: already held → fail
  Node 4: acquired ✓ (but only 1 success so far)
  Node 5: down        → fail

Client B gets 2/5 — NOT a majority → lock NOT acquired.
Mutual exclusion preserved even with 2 nodes down.
```

### Redlock criticism (Martin Kleppmann)

Redlock has a known vulnerability: clock skew and process pauses can cause two clients to both believe they hold the lock simultaneously.

```
t=0:   Client A acquires Redlock (TTL=30s)
t=25:  Client A gets GC paused
t=30:  Lock expires on all nodes
t=31:  Client B acquires Redlock
t=32:  Client A resumes — believes it still holds lock
       Both A and B now act as lock holders
```

This is why **fencing tokens** exist.

## Fencing Tokens (Making Locks Safe Under Pauses)

A fencing token is a **monotonically increasing number** returned when a lock is acquired. The protected resource rejects writes with a stale (lower) token.

```
t=0:   Client A acquires lock → gets token=33
t=1:   Client A sends write to Storage: {data, token=33} → accepted
t=25:  Client A paused (GC, network)
t=30:  Lock expires
t=31:  Client B acquires lock → gets token=34
t=32:  Client B sends write to Storage: {data, token=34} → accepted
t=35:  Client A resumes, sends write: {data, token=33}
       Storage: token 33 < current 34 → REJECTED ✓
```

```
[Lock Service] issues token=33 → Client A
[Lock Service] issues token=34 → Client B (A's lock expired)

[Storage] tracks highest seen token
  → Accepts writes only if token >= last_seen_token
  → Stale writes from Client A are safely rejected
```

Fencing tokens require the storage system's cooperation — the resource itself must check the token.

## Approach 3: Database-Based Locking

When you don't have Redis, a database row can act as a lock.

### Pessimistic locking (SELECT FOR UPDATE)

```sql
BEGIN;
SELECT * FROM seats WHERE seat_id = 'A5' FOR UPDATE;
-- Row is now locked for this transaction
-- Other transactions trying to SELECT FOR UPDATE will BLOCK
UPDATE seats SET booked_by = 'user_1' WHERE seat_id = 'A5';
COMMIT;
-- Lock released
```

- Lock held for the duration of the transaction
- Other writers block (not fail fast)
- Works within a single database — no help for distributed locks

### Optimistic locking (version number)

```sql
-- Read with version
SELECT balance, version FROM accounts WHERE id = 1;
-- balance=100, version=5

-- Write with version check
UPDATE accounts
SET balance = 90, version = 6
WHERE id = 1 AND version = 5;
-- If 0 rows updated → someone else changed it → retry
```

- No blocking — reads are free
- Write detects conflict and retries
- Best when conflicts are rare (read-heavy workloads)
- Terrible under high contention — many retries waste work

## Approach 4: etcd / ZooKeeper Leases

etcd and ZooKeeper are distributed coordination systems built for exactly this use case. They use the Raft consensus algorithm, which is stronger than Redlock.

```
etcd:
  PUT /locks/booking-seat-A5 value="server1" lease=30s
  → If server1 crashes, etcd removes the key automatically when lease expires
  → Lease is kept alive by the holder sending periodic KeepAlive heartbeats

  If server1 becomes partitioned and can't send KeepAlive:
  → Lease expires in 30s
  → Key deleted
  → Another server acquires the lock
```

etcd/ZooKeeper are more operationally complex than Redis but provide stronger consistency guarantees (linearizability via Raft vs Redis's eventual consistency).

## Lock Expiry vs Watchdog

### The TTL dilemma

```
TTL too short:
  → Lock expires while holder is still working
  → Two holders at once (safety violation)

TTL too long:
  → Holder crashes, lock not released for 5 minutes
  → System stalled waiting for expiry

Solution: Watchdog thread
  → Lock acquired with short TTL (e.g. 10s)
  → Holder runs a background thread that extends TTL every 5s
  → If holder crashes, watchdog dies too → TTL expires naturally
  → Redisson (Java) calls this "lock watchdog"
```

```
t=0:   Acquire lock (TTL=10s)
t=5:   Watchdog extends TTL to t=15
t=10:  Watchdog extends TTL to t=20
t=12:  Process crashes — watchdog dies too
t=20:  TTL expires → lock automatically released
t=21:  Another client acquires lock ✓
```

## Distributed Lock vs Optimistic Locking

| Dimension | Distributed Lock | Optimistic Locking |
|-----------|-----------------|-------------------|
| Mechanism | Acquire lock before read+write | Read freely, check version on write |
| Blocking | Other writers blocked or fail-fast | No blocking — detect conflict on write |
| Contention | Serializes writes (one at a time) | Multiple writers proceed, losers retry |
| Best for | High contention, writes must not be lost | Low contention, conflicts are rare |
| Failure mode | Lock not released (holder crash) | Retry storm under high contention |
| Example | Seat booking (last seat) | User profile update (rare conflicts) |

## Common Pitfalls

```
1. Not setting a TTL
   → Holder crashes → lock held forever → deadlock
   → Always set TTL

2. Using the lock server's clock for TTL validation
   → Clock skew between nodes → inconsistent expiry times
   → Use monotonic clocks or consensus protocols

3. Checking then acting (non-atomic)
   → Check if lock is free → lock is acquired between check and acquire
   → Always use atomic SET NX EX

4. Releasing another client's lock
   → Your lock expired, another client acquired it, you DEL their lock
   → Always check the value (UUID) before releasing

5. Long-running operations inside the lock
   → GC pause, slow DB query → lock expires mid-work
   → Keep critical section short, use watchdog for long operations

6. Using a lock when you don't need one
   → Redis INCR / GETSET / COMPARE-AND-SWAP are atomic natively
   → Use atomic operations instead of locking when possible
```

## Interview Answer Framework

When asked "How does a distributed lock work?" in a design round:

1. **State the problem:** multiple servers accessing shared state → race conditions
2. **Name the simplest approach:** Redis `SET NX EX` with a UUID value
3. **Explain TTL:** prevents deadlock when the holder crashes
4. **Explain UUID check before release:** prevents releasing another client's lock
5. **Mention atomic release:** Lua script for check-then-delete atomicity
6. **Acknowledge the flaw:** GC pause / clock skew → two holders possible
7. **Mention fencing tokens:** storage rejects stale token writes
8. **Contrast with optimistic locking:** lock = high contention, optimistic = low contention
9. **Name a production tool:** Redisson (Java) or DistributedLock (.NET)
