# Logging & Monitoring

---

## What Problem Does It Solve?

A system that cannot be observed cannot be debugged, cannot be scaled confidently, and cannot meet SLAs. When something goes wrong in production, logs, metrics, and traces are the only way to understand what happened.

```
Without observability:
  User reports: "checkout is slow"
  Engineer: looks at 5 different services, no shared context, no timeline
  Resolution time: hours вҶ’ revenue loss

With observability:
  Alert fires: p99 latency > 2s on /checkout
  Trace: shows payment-service taking 1.8s
  Logs (filtered by traceId): DB query missing index, full table scan
  Resolution time: minutes вҶ’ pinpoint the exact line
```

Observability is built on three pillars:

```
Logs    вҶ’ What happened (discrete events with context)
Metrics вҶ’ How much / how often (aggregated numbers over time)
Traces  вҶ’ Where time was spent (end-to-end request flow across services)

Each pillar answers a different question:
  "What errors occurred at 14:32?"          вҶ’ Logs
  "Is error rate above normal right now?"   вҶ’ Metrics
  "Which service caused this request to be slow?" вҶ’ Traces
```

## Pillar 1: Logging

### Structured vs Unstructured Logging

```
Unstructured (plain text) вҖ” hard to query at scale:
  2024-03-15 14:32:01 ERROR Payment failed for user 42 order 99 amount 149.99

Structured (JSON) вҖ” machine-readable, filterable, indexable:
  {
    "timestamp": "2024-03-15T14:32:01Z",
    "level":     "ERROR",
    "service":   "payment-service",
    "traceId":   "abc123",
    "spanId":    "def456",
    "userId":    42,
    "orderId":   99,
    "amount":    149.99,
    "error":     "card_declined",
    "message":   "Payment failed"
  }

Benefits of structured:
  вҶ’ Filter: level=ERROR AND service=payment-service
  вҶ’ Aggregate: count errors by error_code per hour
  вҶ’ Join: correlate logs from 5 services by traceId
  вҶ’ Alert: alert when error_count > 10/min
```

### Log Levels

```
Level     When to use                               Example
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
TRACE     Extremely verbose; method entry/exit      "Entering validateCard()"
DEBUG     Developer context; variable values        "Cart total computed: 149.99"
INFO      Normal operations worth recording         "Order 99 placed by user 42"
WARN      Unexpected but recoverable               "Retrying payment (attempt 2/3)"
ERROR     Operation failed; needs attention         "Payment failed: card declined"
FATAL     System cannot continue; immediate action  "DB connection pool exhausted"

Production default level: INFO
  вҶ’ DEBUG/TRACE suppressed in production (too noisy, expensive to store)
  вҶ’ Dynamic log level change: raise to DEBUG for 5 minutes while debugging,
    then drop back to INFO вҖ” without restart (hot reload via config)
```

### What to Log (and What Not To)

```
вң“ Log:
  Request: method, path, status code, latency
  Business events: order placed, payment failed, user registered
  Errors: full stack trace, relevant context (userId, orderId)
  System events: service start/stop, config reload, circuit breaker state change

вң— Do NOT log:
  Passwords, API keys, card numbers (PCI DSS violation)
  PII that does not need to be in logs (full name, SSN, DOB)
  Excessive debug in production (storage cost, performance overhead)
  Binary blobs, large payloads (truncate or log metadata only)
```

## Pillar 2: Metrics

### Metric Types

```
Counter  вҶ’ monotonically increasing number; only resets on restart
  Examples: total_requests, total_errors, total_bytes_sent
  Query:  rate(total_requests[5m])  вҶ’ requests per second over 5 min
  Never decrements (use Gauge for values that go up and down)

Gauge    вҶ’ current value; can go up or down
  Examples: active_connections, queue_depth, memory_usage_bytes, cpu_percent
  Query:  active_connections > 1000  вҶ’ alert threshold

Histogram вҶ’ distribution of values across configurable buckets
  Examples: request_duration_ms, response_size_bytes, db_query_ms
  Buckets:  [10ms, 50ms, 100ms, 250ms, 500ms, 1s, 2s, +Inf]
  Derived:  p50, p95, p99, p999 latency (percentiles)
  WHY:  averages are misleading вҖ” a p99=2s hidden in an average of 50ms
        means 1% of users have terrible experience

Summary  вҶ’ like histogram but computes quantiles client-side
  Less common; used when quantiles must be accurate (vs approximate)
```

### The RED Method (for services)

```
Rate     вҶ’ requests per second
Errors   вҶ’ error rate (errors / total)
Duration вҶ’ latency distribution (p50/p95/p99)

Every service should expose these three metrics.
They are the minimum needed to detect and diagnose a service degradation.

Alert:
  Rate   drops to 0        вҶ’ service not receiving traffic (upstream broken)
  Errors > 1%              вҶ’ something is failing
  p99 Duration > SLO (2s)  вҶ’ users experiencing slow responses
```

### The USE Method (for resources/infrastructure)

```
Utilization  вҶ’ % of resource being used (CPU 70%, disk 80%)
Saturation   вҶ’ how much work is queued/waiting (run queue depth, disk I/O wait)
Errors       вҶ’ error events from the resource (disk errors, NIC drops)

Every infrastructure component (CPU, memory, disk, network) should
be monitored with these three dimensions.

Alert:
  CPU utilization > 85% sustained for 5 min   вҶ’ scale out
  Memory saturation (swap in use)              вҶ’ OOM risk
  Disk utilization > 90%                       вҶ’ fill risk
```

## Pillar 3: Distributed Tracing

A single user request often touches 5вҖ“10 services. Tracing follows the request across all of them.

```
Concepts:
  Trace   вҶ’ the entire journey of one request end-to-end (has a traceId)
  Span    вҶ’ one unit of work within a trace (service call, DB query, cache lookup)
  Parent  вҶ’ span that created this span (forms a tree)

Trace tree for a checkout request:
  [Trace: abc123]
  в”ңв”Җв”Җ Span: API Gateway          10ms
  в”Ӯ   в”ңв”Җв”Җ Span: Auth Service     15ms
  в”Ӯ   в”ңв”Җв”Җ Span: Order Service    180ms
  в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Span: DB query     120ms   вҶҗ bottleneck
  в”Ӯ   в”Ӯ   в””в”Җв”Җ Span: Cache lookup   5ms
  в”Ӯ   в””в”Җв”Җ Span: Payment Service   50ms
  в””в”Җв”Җ Total: 260ms

Without tracing: "checkout is slow" вҖ” no idea which service.
With tracing:    "Order Service DB query takes 120ms of 260ms" вҶ’ check index.
```

### Propagating Trace Context

```
HTTP headers carry the trace context between services:
  traceparent: 00-abc123-def456-01
    version=00, traceId=abc123, spanId=def456, flags=01 (sampled)

  Service A вҶ’ Service B: includes traceparent header
  Service B creates a child span with parentSpanId = A's spanId
  Both log traceId=abc123 вҶ’ logs from all services correlatable

W3C TraceContext standard (traceparent/tracestate headers) is now universal.
B3 propagation (Zipkin) is older but still widely used.
```

### Sampling

```
Problem: tracing 100% of requests is expensive (storage, CPU).
Solution: sample a fraction вҖ” record only some traces.

Head-based sampling (decided at trace start):
  Sample 1% of all requests randomly.
  Simple; misses rare slow requests.

Tail-based sampling (decided at trace end):
  Always record errors and slow requests (>500ms).
  Sample 0.1% of fast healthy requests.
  Best signal-to-noise; requires buffering spans until the trace ends.

Adaptive sampling:
  Auto-adjust sample rate based on traffic volume:
    Low traffic  вҶ’ 100% (every request is interesting)
    High traffic вҶ’ 0.1% (too many to store)
```

## Alerting

```
Alert types:
  Threshold    вҶ’ metric crosses a fixed value
                 e.g. error_rate > 5% for 2 minutes
  Anomaly      вҶ’ metric deviates from historical baseline
                 e.g. request_rate drops 40% vs same time last week
  Absence      вҶ’ expected metric stops arriving
                 e.g. no heartbeat from service in 30s
  Composite    вҶ’ combination of conditions
                 e.g. p99 > 2s AND error_rate > 1%

Alert fatigue вҖ” the #1 problem with alerting:
  Too many alerts вҶ’ engineers ignore them вҶ’ real incidents missed
  Fix:
    Only alert on conditions that require human action
    Every alert must have a runbook (what to do)
    False positive rate > 5% вҶ’ fix the alert, not the on-call process
    Alert on symptoms (user impact), not causes (CPU high)

SLO-based alerting (Google SRE):
  Define: 99.9% of requests succeed in <500ms over 30 days
  Alert:  burn rate > 5Г— normal   вҶ’ will exhaust error budget in 6 hours
  This focuses alerts on user impact, not arbitrary thresholds
```

## Log Aggregation Pipeline

```
Service A в”Җв”ҖвҶ’в”җ
Service B в”Җв”ҖвҶ’в”ңв”Җв”ҖвҶ’ Log Shipper в”Җв”ҖвҶ’ Message Queue в”Җв”ҖвҶ’ Indexer в”Җв”ҖвҶ’ Search/Alert
Service C в”Җв”ҖвҶ’в”ҳ    (Filebeat,       (Kafka, SQS)     (Logstash,   (Elasticsearch,
                   Fluentd)                          Fluentd)      Splunk)

Steps:
  1. Services write structured JSON logs to stdout (12-factor app principle)
  2. Log shipper collects from stdout/files, adds hostname/pod metadata
  3. Queue buffers spikes вҖ” shipper can lag without losing logs
  4. Indexer parses, enriches, stores in search index
  5. Dashboard / alert engine queries the index

Retention policy:
  Hot  (last 7 days):  fast SSD, full-text indexed вҖ” for active debugging
  Warm (7вҖ“30 days):    slower storage, indexed вҖ” for incident review
  Cold (30dвҖ“1 year):   object storage (S3), compressed вҖ” for compliance
  Archive (1y+):       compressed glacier storage вҖ” rarely accessed
```

## Key Metrics Every Service Should Expose

```
HTTP services:
  http_requests_total{method, path, status}     вҶ’ Counter
  http_request_duration_seconds{method, path}   вҶ’ Histogram (p50/p95/p99)
  http_requests_in_flight                       вҶ’ Gauge

Database:
  db_query_duration_seconds{query_type}         вҶ’ Histogram
  db_connections_active                         вҶ’ Gauge
  db_connections_pool_size                      вҶ’ Gauge

Queue consumer:
  queue_messages_consumed_total                 вҶ’ Counter
  queue_consumer_lag                            вҶ’ Gauge  вҶҗ critical: are we keeping up?
  queue_message_processing_duration_seconds     вҶ’ Histogram

JVM / .NET runtime:
  process_memory_bytes                          вҶ’ Gauge
  gc_pause_duration_seconds                     вҶ’ Histogram
  thread_pool_active_threads                    вҶ’ Gauge
```

## Common Pitfalls

```
1. Logging at wrong level
   вҶ’ Every request logged at ERROR вҶ’ alert noise вҶ’ fatigue вҶ’ real errors missed
   вҶ’ Business-as-usual events at INFO; only actual failures at ERROR

2. No correlation ID
   вҶ’ 5 services log the same request with no shared ID
   вҶ’ Impossible to reconstruct the flow after the fact
   вҶ’ Inject traceId at the entry point; propagate to every downstream call and log

3. Logging PII / secrets
   вҶ’ Credit card numbers, passwords, API keys in logs
   вҶ’ PCI DSS / GDPR violation; security breach via log access
   вҶ’ Use a log sanitizer / redaction middleware before writing

4. Not alerting on absence
   вҶ’ Service crashes silently вҖ” no errors, just stops sending metrics
   вҶ’ Heartbeat metric: service emits alive=1 every 30s
   вҶ’ Alert if no heartbeat for 60s

5. Alert on causes, not symptoms
   вҶ’ "CPU > 90%" does not mean users are affected
   вҶ’ "p99 latency > SLO" does mean users are affected
   вҶ’ Symptom-based alerts reduce noise; cause-based metrics inform root cause

6. Metrics with high cardinality labels
   вҶ’ http_requests_total{userId=...} creates millions of time series вҶ’ OOM in Prometheus
   вҶ’ Only low-cardinality labels: method, path (bucketed), status_code

7. Unbounded log growth
   вҶ’ Service logs 1GB/hour with no rotation or retention policy
   вҶ’ Disk fills вҶ’ service crashes (no disk to write to)
   вҶ’ Log rotation (logrotate), retention policy, remote shipping to object storage
```

## Interview Answer Framework

When asked "How would you design logging and monitoring for a microservices system?" in a design round:

1. **Three pillars:** logs (what happened), metrics (how much/often), traces (where time went) вҖ” each answers a different question
2. **Structured logging:** JSON logs with traceId, spanId, service name, userId вҖ” machine-queryable, correlatable across services
3. **Log levels:** INFO in prod by default; ERROR for actionable failures; dynamic level change without restart
4. **Metrics RED:** every service exposes Rate, Error rate, Duration (p99) вҖ” minimum to detect degradation
5. **Metrics USE:** every resource exposes Utilization, Saturation, Errors вҖ” CPU/memory/disk/network
6. **Distributed tracing:** inject traceId at API gateway, propagate via `traceparent` header, all services log the traceId вҖ” reconstruct the full call tree
7. **Alerting on symptoms:** alert on p99 latency > SLO and error rate > threshold вҖ” not on CPU % (no user impact guarantee)
8. **Log pipeline:** stdout вҶ’ Fluentd/Filebeat вҶ’ Kafka вҶ’ Elasticsearch; retention: 7 days hot, 30 days warm, 1 year cold
9. **Avoid cardinality explosion:** no userId/orderId in metric labels вҖ” only method, path-template, status code
10. **Name tools:** Prometheus + Grafana (metrics), ELK/Loki (logs), Jaeger/Tempo (traces), OpenTelemetry SDK (instrumentation)
