# Reverse Proxy

---

## What Problem Does It Solve?

A reverse proxy sits in front of backend servers and handles requests on their behalf. Clients never talk to backend servers directly — they only see the proxy.

```
Without reverse proxy — clients talk directly to backends:
  Client ──→ Web Server 1  :80
  Client ──→ Web Server 2  :80   ← client must know all IPs
  Client ──→ Web Server 3  :80

  Problems:
    → Clients know backend IPs — exposing internal topology
    → TLS certificate on every backend — expensive, hard to rotate
    → No single place to absorb traffic spikes or add caching
    → Deploying a new backend version requires client reconfiguration

With reverse proxy — single façade:
  Client ──→ Reverse Proxy ──→ Web Server 1
                           ──→ Web Server 2   ← proxy chooses
                           ──→ Web Server 3

  Benefits:
    → Backend topology hidden from clients
    → TLS terminated once at the proxy — backends use plain HTTP internally
    → Load balancing, caching, compression — one place
    → Zero-downtime deploys: drain one backend, upgrade, re-add
```

---

## Forward Proxy vs Reverse Proxy

A common point of confusion:

```
Forward Proxy — sits in front of CLIENTS:
  Client A ──→ Forward Proxy ──→ Internet (any server)
  Client B ──→

  Purpose: anonymise clients, enforce corporate policies, cache for many users
  Who configures it: the client (browser proxy settings)
  Who knows about it: the client; the server sees the proxy's IP

Reverse Proxy — sits in front of SERVERS:
  Internet (any client) ──→ Reverse Proxy ──→ Server A
                                           ──→ Server B

  Purpose: protect servers, load balance, terminate TLS, cache responses
  Who configures it: the server operator
  Who knows about it: the server; the client sees the proxy's IP

Mnemonic:
  Forward proxy  = proxy for clients  (hides client identity)
  Reverse proxy  = proxy for servers  (hides server identity)
```

## Core Responsibilities

### 1. SSL/TLS Termination

```
Client ──HTTPS──→ Reverse Proxy ──HTTP──→ Backend

Benefits:
  → Single TLS certificate at the proxy (one place to rotate/renew)
  → Backend servers need no TLS config — simpler, faster (no crypto overhead)
  → HTTP/2 or HTTP/3 to client; HTTP/1.1 to backend if needed
  → Internal network is trusted (private VPC/LAN) — plain HTTP is acceptable

Passthrough mode (SNI proxy):
  → Proxy forwards TLS without decrypting (L4 routing by SNI hostname)
  → Backend handles its own TLS — useful for end-to-end encryption
  → Proxy cannot inspect or modify request body in this mode
```

### 2. Load Balancing

```
Round-Robin (default):
  Request 1 → Backend A
  Request 2 → Backend B
  Request 3 → Backend C
  Request 4 → Backend A  ← cycle repeats
  Pros: simple, even distribution for uniform requests.
  Cons: ignores backend load — a slow request on A doesn't reduce its share.

Least Connections:
  Always send to the backend with fewest active connections.
  Request arrives → check active counts → pick the minimum.
  Best when request processing times vary significantly (some fast, some slow).

IP Hash (sticky sessions):
  hash(client_ip) % backend_count → always same client → same backend
  Required when backend is stateful (session data not shared).
  Problem: one client can overload one backend if traffic is uneven.

Weighted Round-Robin:
  Backend A: weight=3 (more powerful) → 3 out of every 5 requests
  Backend B: weight=2                  → 2 out of every 5 requests
  Use when backends have different capacities.

Random with two choices (power of two):
  Pick two backends at random → send to the one with fewer connections.
  Near-optimal distribution with O(1) cost — used by Nginx and Envoy.
```

### 3. Health Checks

```
Active health check (proxy polls backends):
  Every N seconds: proxy sends HEAD / or GET /health to each backend.
  If backend returns 5xx or times out → mark DOWN, remove from pool.
  When it recovers → mark UP, re-add to pool.

  Proxy ──→ Backend A: GET /health → 200 ✓ (healthy)
  Proxy ──→ Backend B: GET /health → 503 ✗ (remove from rotation)
  Proxy ──→ Backend C: GET /health → 200 ✓ (healthy)
  Traffic now distributed only between A and C.

Passive health check (circuit breaker on real traffic):
  Proxy observes real request outcomes.
  If backend returns 5xx on N consecutive requests → mark DOWN.
  After cooldown period → probe with one request → if success, mark UP.
  Less overhead than active polling; slower to detect silent failures.

Graceful drain:
  Before removing a backend (deploy/shutdown):
    → Stop sending NEW requests to the backend
    → Wait for in-flight requests to complete (drain timeout: e.g. 30s)
    → Then shut down the backend
  Clients never see an error; zero-downtime deploys.
```

### 4. Header Manipulation

```
X-Forwarded-For: chain of client IPs through proxies
  Client (1.2.3.4) → Proxy 1 (10.0.0.1) → Proxy 2 → Backend

  Proxy 1 adds: X-Forwarded-For: 1.2.3.4
  Proxy 2 adds: X-Forwarded-For: 1.2.3.4, 10.0.0.1
  Backend sees the full chain; takes leftmost IP as real client IP.

X-Real-IP: just the original client IP (simpler, single value)
  Proxy sets: X-Real-IP: 1.2.3.4

X-Forwarded-Proto: original protocol (http or https)
  Proxy sets: X-Forwarded-Proto: https
  Backend knows the original client used HTTPS even though it received HTTP.

Host rewriting:
  Client sends:  Host: api.example.com
  Proxy rewrites: Host: internal-backend-cluster
  Backend receives the internal hostname — useful for virtual hosting.
```

### 5. Caching

```
Proxy cache serves responses without hitting the backend:
  Cache-Control: max-age=60 → cache for 60s
  Cache-Control: no-cache   → validate with backend (conditional GET)
  Cache-Control: private    → do NOT cache at proxy (user-specific data)

Cache key: URL + Vary header values (e.g. Accept-Encoding, Accept-Language)

Benefits at proxy vs CDN:
  → Proxy cache: low latency (same data centre), fine-grained control
  → CDN: global distribution, much closer to end user

Stale-while-revalidate:
  Serve stale cached content immediately while revalidating in background.
  User gets fast response; next request gets fresh content.
  Cache-Control: max-age=60, stale-while-revalidate=30
```

### 6. Compression

```
Client sends: Accept-Encoding: gzip, br
Proxy compresses response before sending (if not already compressed).
Backend sends uncompressed response — proxy compresses it.

Reduces bandwidth:
  JSON response 100KB → gzip → ~15KB  (85% reduction)
  HTML page  50KB → gzip → ~10KB

Content-Encoding header tells client how to decompress:
  Content-Encoding: gzip
```

### 7. Connection Pooling

```
Without connection pooling:
  Each request → new TCP connection to backend → TLS handshake → request → close
  Cost: ~100ms per new TCP+TLS connection

With connection pooling (keep-alive):
  Proxy maintains a pool of open connections to each backend.
  Incoming request → reuse existing connection → immediate request send
  Cost: ~1ms (no handshake)

  Pool parameters:
    max_connections_per_backend: 100
    keepalive_timeout: 60s
    idle_connection_eviction: every 30s
```

## Reverse Proxy vs API Gateway vs Load Balancer

```
                Load Balancer       Reverse Proxy        API Gateway
─────────────────────────────────────────────────────────────────────────────
Primary job     Distribute traffic  Forward + enhance     Application routing
OSI Layer       L4 (TCP) or L7      L7 (HTTP)             L7 (HTTP)
SSL termination Sometimes           Yes                   Yes
Load balancing  Yes                 Yes                   Yes
Caching         No                  Yes                   Yes
Compression     No                  Yes                   Sometimes
Auth            No                  No                    Yes
Rate limiting   No                  No                    Yes
Transformation  No                  Headers only          Full body + protocol
Observability   Basic               Access logs           Full metrics/tracing
API versioning  No                  No                    Yes

A Reverse Proxy is a foundation that API Gateways build on top of.
Nginx can act as a load balancer, reverse proxy, or (with Lua) an API gateway.
```

## Connection Draining and Zero-Downtime Deploys

```
Problem: backend receives new request while shutting down mid-flight.

Solution — graceful drain:
  1. Remove backend from load balancer pool (stop new requests)
  2. Wait for in-flight requests to finish (drain timeout)
  3. Send SIGTERM to backend process
  4. Backend completes current work, closes connections, exits

Kubernetes readiness probe:
  Backend sets readinessProbe: fail before shutdown.
  K8s removes pod from Service endpoints.
  Proxy stops routing to it. In-flight requests drain.
  Pod terminates cleanly after terminationGracePeriodSeconds.

Blue-Green with proxy:
  Current traffic → Blue backends
  Deploy Green backends → health checks pass
  Shift traffic: proxy repoints upstream to Green
  Drain Blue → remove
  Instant, zero-downtime, instant rollback (repoint to Blue)
```

## Common Pitfalls

```
1. Not stripping internal headers from client requests
   → Client sends X-User-Id: 999 → backend trusts it → privilege escalation
   → Proxy must strip X-User-Id, X-Real-IP, X-Forwarded-For on ingress
      before adding its own trustworthy values

2. Forgetting X-Forwarded-Proto
   → App generates HTTP links even though client used HTTPS
   → Fix: set X-Forwarded-Proto: https; app reads it for redirect generation

3. Proxy timeouts shorter than backend processing time
   → Proxy times out and returns 504 while backend is still processing
   → Client retries → backend processes the request twice (idempotency issue)
   → Fix: proxy timeout > backend's worst-case processing time

4. SSL certificate not auto-renewed
   → Certificate expires → clients get TLS error → full outage
   → Fix: Let's Encrypt + certbot / cert-manager (K8s) with auto-renewal

5. Caching responses that should not be cached
   → POST /search cached by mistake → wrong results returned to other users
   → Fix: only cache safe methods; honour Cache-Control: private / no-store

6. Health check endpoint is too shallow
   → GET /health returns 200 but DB is down → proxy keeps sending traffic
   → Fix: health check probes actual dependencies (DB ping, Redis ping)

7. Single proxy instance = SPOF
   → Proxy crashes → entire system unreachable
   → Fix: run proxy in active-active HA pair behind a hardware/cloud load balancer
```

## Interview Answer Framework

When asked "How does a reverse proxy work and why use one?" in a design round:

1. **Define it:** sits in front of backend servers; clients only see the proxy — backend topology hidden
2. **SSL termination:** single TLS cert at the proxy; backends use plain HTTP internally; one place to renew
3. **Load balancing:** round-robin for uniform requests; least-connections when request times vary; IP hash for sticky sessions
4. **Health checks:** active polling (`GET /health` every N seconds); passive detection (consecutive 5xx → mark down); graceful drain before deploys
5. **Header injection:** `X-Forwarded-For` (real client IP), `X-Forwarded-Proto` (original scheme), `X-Real-IP` — strip these from client on ingress
6. **Caching:** proxy caches `Cache-Control: max-age` responses; reduces backend load; key on URL + Vary headers
7. **Compression:** proxy gzips responses; backend sends raw; 85% bandwidth reduction for JSON/HTML
8. **Connection pooling:** proxy reuses keep-alive connections to backends; amortises TCP handshake cost
9. **HA:** run two proxy instances active-active behind a cloud load balancer (anycast IP)
10. **Name a tool:** Nginx (most common), HAProxy (high-throughput TCP), Traefik (K8s auto-discovery), YARP (.NET)
