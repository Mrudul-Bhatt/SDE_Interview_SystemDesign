# Notification System

---

## What Problem Does It Solve?

Applications need to communicate events to users across multiple channels (email, SMS, push, in-app) in a reliable, scalable way — without every service implementing its own delivery logic.

```
Without a centralized notification service:
  Order service sends email directly via SMTP
  Payment service sends SMS directly via Twilio
  Shipping service sends push directly via FCM
  → Duplicated logic, inconsistent retry, no opt-out management,
    no deduplication, no rate limiting

With a centralized notification service:
  All services publish events → Notification Service handles:
    ✓ Channel routing (email / SMS / push / in-app / webhook)
    ✓ User preferences and opt-out
    ✓ Deduplication (same notification not sent twice)
    ✓ Rate limiting (no notification spam per user)
    ✓ Retry with backoff
    ✓ Delivery tracking and audit log
```

---

## Core Architecture

```
Producers (Order, Payment, Shipping services)
    │
    ▼
[Event Bus / API]          ← producers publish NotificationRequest
    │
    ▼
[Notification Service]
    ├── Preference Check   ← does user want this channel?
    ├── Deduplication      ← idempotency key: already sent?
    ├── Rate Limiter       ← per-user per-channel throttle
    ├── Template Engine    ← render subject/body from template + variables
    ├── Priority Queue     ← CRITICAL > HIGH > NORMAL > MARKETING
    │
    ▼
[Channel Router]
    ├── EmailAdapter      → SendGrid / SES / Postmark
    ├── SmsAdapter        → Twilio / SNS
    ├── PushAdapter       → FCM (Android) / APNs (iOS)
    ├── InAppAdapter      → WebSocket push / long-poll / SSE
    └── WebhookAdapter    → HTTP POST to customer endpoint
    │
    ▼
[Delivery Status Store]   ← QUEUED → SENT → DELIVERED → FAILED
    │
    ▼
[Dead Letter Queue]       ← permanently failed after N retries
```

## Priority Queues

```
Priority    Examples                          SLA
──────────────────────────────────────────────────────────
CRITICAL    OTP, fraud alert, PW reset        Delivered in < 5s
HIGH        Order confirmed, payment failed   Delivered in < 30s
NORMAL      Shipping update, receipt          Delivered in < 2min
MARKETING   Promo, weekly digest              Delivered in < 1h (batched)

Implementation:
  Separate queues per priority (not one queue with priority field)
  → CRITICAL queue processed by dedicated high-throughput workers
  → MARKETING queue rate-limited and batched (avoid sending at 3am)
  → Marketing vs transactional: never compete for same throughput
```

## Template Engine

```
Templates stored in DB, rendered at send time:

Template "order_confirmed":
  Subject: "Your order {{orderId}} is confirmed!"
  Body:    "Hi {{firstName}}, your order of {{total}} ships {{estimatedDate}}."

Variables injected by producer:
  { orderId: "ORD-99", firstName: "Alice", total: "$149.99", estimatedDate: "Mar 20" }

Rendered:
  Subject: "Your order ORD-99 is confirmed!"
  Body:    "Hi Alice, your order of $149.99 ships Mar 20."

Benefits:
  → Content managed by non-engineers (product / marketing)
  → A/B test templates without code deploys
  → Per-locale templates (i18n)
  → HTML + plain-text variants in one template record
```

## User Preferences and Opt-Out

```
Per-user, per-channel, per-category preferences:

user_id  channel  category      opted_in
───────────────────────────────────────────
42       EMAIL    TRANSACTIONAL  true     ← can't opt out (legal requirement)
42       EMAIL    MARKETING      false    ← opted out
42       SMS      ALL            false    ← no SMS
42       PUSH     ALL            true

Rules:
  → TRANSACTIONAL (OTP, receipts) often cannot be opted out of
  → MARKETING always requires explicit opt-in (CAN-SPAM, GDPR)
  → Check preferences BEFORE rendering template (save CPU)
  → Store opt-out event with timestamp + channel for audit
```

## Deduplication

```
Problem: producer retries, network duplicates → user gets same notification 2×

Solution: idempotency key on every notification request
  key = hash(userId + templateId + contextId + channel)
  Example: SHA256("42:order_confirmed:ORD-99:EMAIL")

Before sending:
  IF key already in dedup store (TTL = 24h) → SKIP
  ELSE → mark key, proceed

Why TTL?
  → After 24h, same event is likely a new occurrence
  → Infinite TTL wastes storage

Dedup store: Redis with TTL (SETNX EX)
```

## Rate Limiting

```
Dimensions to throttle:
  Per-user per-channel: max 10 SMS/hour, max 50 emails/day
  Per-channel global:   max 1000 SMS/second (provider limit)
  Per-template:         marketing emails max 1M/hour (sender reputation)

Algorithm: Token bucket per (userId, channel)
  → 10 tokens/hour; each send consumes 1 token
  → Burst allowed (use tokens immediately, refill over time)

Exceeding limit:
  CRITICAL: bypass rate limit (fraud alerts always go through)
  HIGH:     queue and send on next available slot
  MARKETING: drop and log (not worth re-queuing promotional messages)
```

## Retry Strategy

```
Transient failures (provider timeout, 5xx) → retry with backoff:
  Attempt 1: immediate
  Attempt 2: 30s
  Attempt 3: 5min
  Attempt 4: 30min
  Attempt 5: 2h
  After 5 failures → Dead Letter Queue (DLQ)

Permanent failures (invalid address, unsubscribed) → no retry:
  HTTP 400 / "user unknown" / "address bounced"
  → Mark user's email as bounced in preference store
  → Alert ops (high bounce rate hurts sender reputation)

DLQ handling:
  → Ops reviews DLQ items manually or via dashboard
  → Re-drive after fixing root cause (bad template variable, wrong phone format)
```

## Fan-Out Pattern

```
One event → multiple channels simultaneously:
  Order confirmed:
    ├── Email (receipt)
    ├── SMS  ("Your order ORD-99 is confirmed")
    └── Push ("Order placed! Tap to track")

Implementation:
  NotificationRequest → NotificationService
  → Spawns one task per enabled channel (parallel fan-out)
  → Each channel independently retries
  → One channel failure does not block others
  → Aggregate result: { email: SENT, sms: FAILED, push: SENT }
```

## Delivery Status Tracking

```
States:
  QUEUED       → request accepted, waiting for worker
  PROCESSING   → worker picked it up, calling provider
  SENT         → provider accepted the message
  DELIVERED    → provider confirmed delivery (SMS delivery receipt, push ACK)
  READ         → user opened / clicked (email pixel, push tap event)
  FAILED       → all retries exhausted
  SKIPPED      → dedup hit or user opted out

Storage:
  notification_log(id, userId, channel, templateId, status, createdAt, updatedAt)
  → queryable by user, by status, by time range
  → powers "notification history" in user settings page
  → enables support team to verify "did user get the OTP?"
```

## Common Pitfalls

```
1. Not respecting opt-out → GDPR/CAN-SPAM violation → fines
   → Check preferences before rendering; never hardcode "always send"

2. No deduplication → user gets 3 identical "order confirmed" emails
   → Producer retries + network retries both cause duplicates without idempotency key

3. Marketing mixed with transactional in same queue
   → Burst of marketing emails delays OTPs → user can't log in
   → Separate queues; CRITICAL workers never touch marketing queue

4. Cardinality in metrics (userId as label)
   → notification_sent_total{userId=...} → millions of time series → OOM
   → Labels: channel, template_id, status — never userId

5. No bounce handling → sending to invalid addresses
   → Hurts sender IP reputation → all emails go to spam
   → Listen to provider bounce webhooks; auto-disable bounced addresses

6. Sending at wrong time → marketing push at 3am local time
   → User uninstalls app
   → Respect user's local timezone; queue marketing for 10am–8pm window

7. Template missing variable → "Hi {{firstName}}" in subject
   → Validate all required variables present before rendering
   → Use default values for optional variables
```

## Real-World Tools

```
Providers:
  Email:  SendGrid, Amazon SES, Postmark, Mailgun
  SMS:    Twilio, Amazon SNS, Vonage
  Push:   Firebase Cloud Messaging (FCM), Apple Push Notification Service (APNs)

Orchestration platforms (manage all channels in one):
  Amazon SNS      Managed fan-out; native AWS integration
  Courier         Multi-channel notification routing SaaS
  OneSignal       Push-first; also email/SMS; generous free tier
  Knock           Developer-first; preference management built-in
  Novu            Open-source; self-hostable; all channels

Self-built stack:
  Queue:    Kafka or SQS (per-priority queues)
  Store:    Redis (dedup + rate limit), PostgreSQL (delivery log + preferences)
  Workers:  Kubernetes Deployment per channel adapter
  Template: Handlebars / Mustache stored in DB or S3
```

## Interview Answer Framework

When asked "Design a notification service" in a system design round:

1. **Channels:** email / SMS / push / in-app — each is a separate adapter with its own provider, failure modes, and cost
2. **Architecture:** producers publish events → notification service → priority queue → channel workers → providers
3. **Priority queues:** CRITICAL (OTP, fraud) isolated from MARKETING (promos) — different workers, different SLAs
4. **Template engine:** templates in DB, variables injected at render time — content managed without code deploys
5. **User preferences:** per-channel opt-out checked before any work; transactional exempted; GDPR-compliant audit log
6. **Deduplication:** idempotency key = hash(userId+templateId+contextId+channel); Redis SETNX with 24h TTL
7. **Rate limiting:** token bucket per (userId, channel); CRITICAL bypasses; MARKETING dropped on limit
8. **Retry:** exponential backoff up to 5 attempts; permanent failures (bounce) trigger address suppression; DLQ for review
9. **Fan-out:** parallel sends per channel; one failure doesn't block others; aggregate delivery status per notification
10. **Delivery tracking:** QUEUED→SENT→DELIVERED→READ→FAILED; queryable by user for support/audit
