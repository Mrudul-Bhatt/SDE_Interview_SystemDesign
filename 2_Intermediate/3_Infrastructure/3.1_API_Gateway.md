# API Gateway

---

## What Problem Does It Solve?

In a microservices architecture, clients would need to know about dozens of backend services, each with its own address, auth scheme, and protocol. Cross-cutting concerns (auth, rate limiting, logging, SSL) would be duplicated in every service.

```
Without API Gateway — client talks to every service directly:
  Mobile App ──→ Order Service      :8001
  Mobile App ──→ Payment Service    :8002
  Mobile App ──→ Inventory Service  :8003
  Mobile App ──→ User Service       :8004

  Problems:
    → Client must know all service addresses
    → Auth logic duplicated in every service
    → Rate limiting duplicated in every service
    → CORS, SSL termination duplicated everywhere
    → Mobile app breaks when a service moves or is split
    → No single place to add observability

With API Gateway — single entry point:
  Mobile App ──→ API Gateway ──→ Order Service
                             ──→ Payment Service
                             ──→ Inventory Service
                             ──→ User Service

  Benefits:
    → Client talks to one address
    → Auth, rate limiting, logging at the gateway — once
    → Backend services can be refactored freely
    → Canary routing, A/B testing at one point
```

## Core Responsibilities

```
1. Request Routing
   → Map incoming path/method to the correct backend service
   → /api/orders/**  → Order Service
   → /api/payments/** → Payment Service
   → /api/users/**   → User Service

2. Authentication & Authorization
   → Validate JWT / API key before the request reaches any service
   → Reject unauthenticated requests at the edge — services never see them
   → Optionally: forward decoded claims as headers (X-User-Id, X-User-Role)

3. Rate Limiting
   → Enforce per-client/per-IP/per-API-key quotas at the edge
   → 1000 req/min per API key; burst of 50 req/sec
   → Returns HTTP 429 before any backend is touched

4. SSL/TLS Termination
   → Decrypt HTTPS at the gateway
   → Forward HTTP internally (faster, simpler cert management)
   → Backends need no TLS config; single cert at the gateway

5. Request / Response Transformation
   → Add/remove headers (X-Request-Id, X-Forwarded-For)
   → Translate protocol: REST → gRPC to backend
   → Response shaping: strip internal fields before returning to client

6. Load Balancing
   → Distribute traffic across multiple instances of a service
   → Health checks: remove unhealthy instances automatically

7. Caching
   → Cache GET responses at the gateway (Redis or in-memory)
   → Serve cached responses without hitting backend
   → Keyed on URL + Vary headers

8. Observability
   → Centralized access logs, latency metrics, error rates
   → Correlation ID injected on every request — traces the full journey
   → No per-service logging setup needed

9. Circuit Breaking
   → Stop forwarding to a failing backend — return cached or error response
   → Prevents cascading failure (covered in Resilience Patterns)

10. API Versioning
    → Route /v1/orders → OrderServiceV1
    → Route /v2/orders → OrderServiceV2
    → Version sunset: redirect v1 → v2 with deprecation header
```

## Request Routing Patterns

```
Path-based routing (most common):
  /api/orders/**   → orders-service:8001
  /api/users/**    → user-service:8002
  /api/payments/** → payment-service:8003

Method-based routing:
  GET  /api/products/** → product-read-service   (CQRS read path — fast)
  POST /api/products/** → product-write-service  (CQRS write path — validated)

Header-based routing:
  X-Api-Version: 2  → order-service-v2
  X-Api-Version: 1  → order-service-v1

Query param routing:
  ?region=eu → eu-order-service
  ?region=us → us-order-service

Canary routing (weighted):
  95% of traffic → stable service
   5% of traffic → canary service (new version)
  Gradually shift weight as confidence grows.

Host-based routing (multi-tenant):
  acme.api.example.com  → acme tenant cluster
  beta.api.example.com  → beta-tester cluster
```

## Authentication at the Gateway

The gateway is the single enforcement point. Backend services trust that the gateway has already validated the caller.

```
Flow with JWT:
  Client                 Gateway                   Backend
    │ POST /api/orders     │                           │
    │ Authorization: Bearer│                           │
    │ eyJhbGci...          │                           │
    │─────────────────────→│                           │
    │                      │ 1. Decode JWT             │
    │                      │ 2. Verify signature       │
    │                      │ 3. Check expiry           │
    │                      │ 4. Extract claims         │
    │                      │    (userId, role, scope)  │
    │                      │─────────────────────────→ │
    │                      │  X-User-Id: 42            │
    │                      │  X-User-Role: customer    │
    │                      │  (original JWT stripped)  │

Backend services:
  → Never see raw JWTs
  → Trust X-User-Id header (only gateway can set it — internal network)
  → Implement authorization (can this role do this action?) not authentication

For API keys:
  Gateway: look up API key in Redis/DB → get associated userId + quotas
  Inject X-User-Id + rate limit context
  Log key usage for billing
```

## Rate Limiting at the Gateway

Covered in depth in 7.4/7.5 (Token Bucket, Leaky Bucket). At the gateway level:

```
Dimensions to rate limit:
  Per IP address    → prevent abuse from single source
  Per API key       → per-customer quota (billing tiers)
  Per user          → prevent single user from flooding
  Per endpoint      → protect expensive endpoints (search, export)
  Global            → protect the entire system from overload

Response on limit exceeded:
  HTTP 429 Too Many Requests
  Headers:
    Retry-After: 30               ← seconds until quota resets
    X-RateLimit-Limit: 1000       ← quota for the window
    X-RateLimit-Remaining: 0      ← requests left
    X-RateLimit-Reset: 1714000000 ← Unix timestamp of reset

Distributed rate limiting:
  → Multiple gateway instances need shared counters
  → Use Redis INCR with TTL: atomic, fast, works across instances
  → Lua script for atomic check-and-increment
```

## Request / Response Transformation

```
Common request transformations:
  → Add Correlation-Id header if absent:  X-Request-Id: <uuid>
  → Add timestamp header:                 X-Request-Time: <iso8601>
  → Strip Authorization header before forwarding (replaced by X-User-Id)
  → Rewrite path: /api/v1/users → /users  (version stripping)
  → Inject tenant ID from JWT into header: X-Tenant-Id: acme

Common response transformations:
  → Add CORS headers: Access-Control-Allow-Origin: *
  → Remove internal headers: X-Internal-Service, X-Trace-Id
  → Wrap response body: { "data": <original>, "meta": { "requestId": "..." } }
  → Compress response: gzip/br if client supports it
  → Translate gRPC error codes → HTTP status codes

Protocol translation:
  External: REST (HTTP/JSON)
  Internal: gRPC (Protobuf) — faster, binary, strongly typed
  Gateway translates: JSON → Protobuf on request, Protobuf → JSON on response
  Backend services need not expose REST at all
```

## API Versioning Strategies

```
URL versioning (most common, most explicit):
  /v1/orders → OrderServiceV1
  /v2/orders → OrderServiceV2
  Pros: explicit, cacheable, bookmarkable.
  Cons: pollutes URL space.

Header versioning:
  Accept: application/vnd.myapi.v2+json
  Pros: clean URLs.
  Cons: not browser-friendly, hard to test in browser.

Query param versioning:
  /orders?version=2
  Pros: easy to test.
  Cons: version in query is ignored by caches (cache misses).

Gateway-level sunset:
  1. Route /v1/** → v1 service (with Deprecation header warning)
  2. Route /v2/** → v2 service
  3. After sunset date: /v1/** → 301 redirect to /v2/**
  4. Eventually: /v1/** → 410 Gone
```

## Caching at the Gateway

```
Cache key: method + URL + Vary headers (e.g. Accept-Language)

Cache-Control semantics:
  Cache-Control: max-age=60      → cache for 60 seconds
  Cache-Control: no-cache        → validate with backend before serving
  Cache-Control: no-store        → never cache
  Cache-Control: private         → client cache only, not gateway cache
  Cache-Control: public          → gateway may cache

Gateway cache hit flow:
  Client → Gateway → Redis.GET(key) → found → return cached response
                                     → miss  → forward to backend
                                              → cache response
                                              → return to client

What to cache at the gateway:
  ✓ Product listings, catalog data (changes infrequently)
  ✓ Configuration responses
  ✗ User-specific data (unless keyed per user)
  ✗ Write endpoints (POST/PUT/DELETE)
  ✗ Real-time data (stock prices, live scores)
```

## Gateway vs Reverse Proxy vs Load Balancer

```
Component          Primary Job              Typical Features
──────────────────────────────────────────────────────────────────────────────
Load Balancer    Distribute traffic         L4/L7 balancing, health checks
                 across instances           Minimal request awareness

Reverse Proxy    Forward requests,          SSL termination, caching,
                 hide backend topology      compression, static files

API Gateway      Application-layer          All of above + auth, rate limiting,
                 traffic management         routing, transformation, versioning,
                                            observability, circuit breaking

Relationship:
  Load Balancer ⊂ Reverse Proxy ⊂ API Gateway
  Each builds on the capabilities of the one before.
  A full API Gateway does everything a reverse proxy does, and more.
```

## Service Mesh vs API Gateway

```
API Gateway:
  → North-South traffic (client → backend)
  → Single entry point at the edge
  → Handles external-facing concerns: auth, rate limiting, versioning

Service Mesh (Istio, Linkerd):
  → East-West traffic (service → service)
  → Sidecar proxy injected alongside every service
  → Handles internal concerns: mTLS, retries, circuit breaking, telemetry

They complement each other:
  Client → API Gateway (auth, rate limit) → Service A
                                              ↕ (service mesh: mTLS, retry)
                                           Service B
```

## Common Pitfalls

```
1. Making the gateway too smart (God Gateway)
   → Business logic in the gateway (discount rules, inventory checks)
   → Gateway becomes a bottleneck and a deployment risk
   → Fix: gateway handles cross-cutting concerns only; business logic stays in services

2. Single point of failure
   → One gateway instance → it goes down → entire system down
   → Fix: run multiple gateway instances behind a load balancer

3. Latency added by the gateway
   → Every request goes through an extra hop
   → Fix: keep gateway logic O(1) — Redis lookups, not DB joins

4. Bypassing the gateway
   → Services accessible directly (internal DNS exposed externally)
   → Fix: network policy / VPC rules ensure only gateway can reach services

5. Cache stampede on cold start
   → Gateway restarts, all cached entries gone → flood of backend requests
   → Fix: staggered TTLs, warming cache from a snapshot on startup

6. Auth token forwarding instead of claims forwarding
   → Gateway forwards raw JWT to backend → backend re-validates → wasted CPU
   → Fix: gateway validates once, forwards decoded claims as trusted headers

7. Rate limit counters not shared across gateway instances
   → Each gateway instance has its own counter → N instances = N× the quota
   → Fix: use shared Redis counter, not in-memory
```

## Interview Answer Framework

When asked "How would you design an API Gateway?" in a design round:

1. **State the problem:** clients shouldn't talk directly to dozens of services; cross-cutting concerns shouldn't be duplicated
2. **Single entry point:** all external traffic enters through the gateway — one address for clients
3. **Core responsibilities:** routing (path/method → service), auth (JWT validation → inject claims), rate limiting (Redis counters per key), SSL termination
4. **Request flow:** client → TLS termination → auth → rate limit check → routing → transform request → backend → transform response → return
5. **High availability:** multiple gateway instances behind a load balancer; Redis for shared state (rate limit counters, cache)
6. **Caching:** cache GET responses with Cache-Control semantics; key on URL + Vary headers
7. **Observability:** inject Correlation-Id, emit structured logs and latency metrics per route
8. **Avoid business logic in the gateway:** routing and cross-cutting concerns only — keep it thin and fast
9. **Name a tool:** Kong (open source, plugin model), AWS API Gateway (managed, serverless), Ocelot/YARP (.NET)
