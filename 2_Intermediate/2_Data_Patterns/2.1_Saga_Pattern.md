# Saga Pattern

---

## What Problem Does It Solve?

In a monolith, a multi-step operation is wrapped in a single database transaction — either everything commits or everything rolls back (ACID). In microservices, each service owns its own database. There is no global transaction manager that can atomically span multiple services.

```
Monolith (single DB — easy):
  BEGIN TRANSACTION
    debit  user wallet       → wallet_db
    debit  inventory         → inventory_db
    create order record      → orders_db
    send   confirmation      → notification_db
  COMMIT  ← all succeed, or all roll back together

Microservices (separate DBs — the problem):
  ┌──────────────┐   ┌─────────────────┐   ┌──────────────┐
  │ Order Service│   │ Inventory Service│   │Payment Service│
  │  orders_db   │   │  inventory_db   │   │  payment_db  │
  └──────────────┘   └─────────────────┘   └──────────────┘
         ↑                   ↑                     ↑
  No shared transaction manager — you cannot BEGIN across all three.

Failure scenario without saga:
  Step 1: Payment charged ✓
  Step 2: Inventory reserved ✓
  Step 3: Shipping label created — FAILS ✗
  Result: money taken, item reserved, no shipment → data inconsistency
```

A **Saga** is a sequence of local transactions. Each step publishes an event or sends a command. If any step fails, compensating transactions undo the already-completed steps.

---

## Core Concepts

```
Local transaction   → a single-service DB operation; ACID within one service
Compensating tx     → the semantic undo of a completed local transaction
                      (not a rollback — the original tx already committed)

Saga execution:
  T1 → T2 → T3 → ... → Tn       (happy path — all succeed)

Failure at step Ti:
  T1 → T2 → Ti FAILS
         ↓
  C(i-1) → C(i-2) → C1          (compensating transactions in reverse order)

Where C(k) = compensating transaction for step k
```

```
Order placement saga — happy path:
  1. Create order (PENDING)
  2. Reserve inventory
  3. Charge payment
  4. Schedule shipping
  5. Mark order CONFIRMED

Failure at step 3 (payment declined):
  5. (not reached)
  4. (not reached)
  3. Payment FAILED
  C2. Release inventory reservation     ← compensating tx for step 2
  C1. Cancel order (set status FAILED)  ← compensating tx for step 1
```

## Real-World Scenarios

```
1. E-commerce order placement
   → Reserve stock → Charge card → Schedule delivery
   → If card fails: release stock → cancel order

2. Hotel + flight booking
   → Book hotel → Book flight → Charge card
   → If flight unavailable: cancel hotel → refund card

3. Bank money transfer
   → Debit source account → Credit destination account
   → If credit fails: refund source account (compensating tx)

4. Food delivery
   → Accept order → Assign driver → Charge customer → Notify restaurant
   → If no driver: cancel order → refund customer

5. SaaS subscription signup
   → Create account → Provision resources → Charge card → Send welcome email
   → If provisioning fails: delete account → refund charge
```

## Approach 1: Choreography-Based Saga

Each service listens for events and reacts. No central coordinator. Services are decoupled — they only know about events, not each other.

```
Order      Inventory     Payment      Shipping      Event Bus
  │                                                     │
  │── OrderCreated ─────────────────────────────────→  │
  │                  │                                  │
  │         InventoryReserved ───────────────────────→  │
  │                              │                      │
  │                     PaymentCharged ──────────────→  │
  │                                          │          │
  │                             ShipmentScheduled ───→  │
  │                                                     │
  │←── ShipmentScheduled ───────────────────────────── │
  │    (Order marks itself CONFIRMED)                   │

Failure path — payment fails:
  │                              │                      │
  │                     PaymentFailed ───────────────→  │
  │         InventoryReleased ←──────────────────────── │
  │    OrderCancelled ←─────────────────────────────── │
```

### Choreography: Pros and Cons

```
Pros:
  + No single point of failure (no orchestrator to crash)
  + Services are loosely coupled — adding a new service = subscribe to events
  + Simple for short sagas (2–3 steps)

Cons:
  - Difficult to trace: saga state is distributed across event logs
  - Cyclic dependencies creep in as services start reacting to each other's events
  - Hard to test end-to-end: no single place to see the full saga state
  - Compensating logic is scattered: each service handles its own undo
```

## Approach 2: Orchestration-Based Saga

A central **Saga Orchestrator** drives the saga by sending commands to each participant and waiting for replies. The orchestrator holds the entire saga state.

```
                   ┌─────────────────┐
                   │  Saga           │
                   │  Orchestrator   │
                   └────────┬────────┘
                            │
          ┌─────────────────┼─────────────────┐
          ↓                 ↓                 ↓
   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
   │   Order      │  │  Inventory   │  │   Payment    │
   │   Service    │  │   Service    │  │   Service    │
   └──────────────┘  └──────────────┘  └──────────────┘

Execution:
  Orchestrator → ReserveInventory → Inventory Service
  Inventory Service → InventoryReserved → Orchestrator
  Orchestrator → ChargePayment → Payment Service
  Payment Service → PaymentFailed → Orchestrator
  Orchestrator → ReleaseInventory → Inventory Service  ← compensation
  Orchestrator → CancelOrder → Order Service           ← compensation
  Orchestrator → Saga FAILED (terminal state)
```

### Orchestration: Pros and Cons

```
Pros:
  + Full saga state visible in one place (easy to monitor/debug)
  + Clear control flow: orchestrator is the source of truth
  + Easy to add retries, timeouts, alerts at the orchestrator level
  + Better for complex sagas (5+ steps, conditional branches)

Cons:
  - Orchestrator is a new service to build and operate
  - Can become a "God service" that knows too much about each participant
  - Single point of failure unless the orchestrator itself is made durable
```

## Compensating Transactions

A compensating transaction is the **semantic undo** — not a database rollback.

```
Original tx:      INSERT INTO reservations (item_id, user_id) VALUES (42, 7)
Compensating tx:  DELETE FROM reservations WHERE item_id=42 AND user_id=7

Original tx:      UPDATE accounts SET balance = balance - 100 WHERE id = 1
Compensating tx:  UPDATE accounts SET balance = balance + 100 WHERE id = 1

Key difference from DB rollback:
  Rollback     → as if the original tx never happened (MVCC undo)
  Compensating → a new forward tx that logically reverses the effect
                 (the original tx DID commit and may have been visible to others)
```

### Not all operations can be compensated

```
Compensatable:   Reserve seat       → Release seat
                 Charge card        → Refund card
                 Send email         → Send "sorry, cancelled" email  ← best-effort
                 Debit account      → Credit account

Cannot undo:     Email already opened by user
                 Webhook notification already processed by 3rd party
                 Physical shipment already dispatched
                 → These are "pivot transactions" — once done, saga cannot roll back past them
```

## Saga State Machine

Each saga has a state machine. The orchestrator (or event log) drives transitions.

```
                    ┌─────────────┐
              ───→  │   STARTED   │
                    └──────┬──────┘
                           │ inventory reserved
                    ┌──────▼──────┐
                    │  INVENTORY  │
                    │  RESERVED   │
                    └──────┬──────┘
                 ┌─────────┴──────────┐
          payment ok            payment failed
                 │                    │
          ┌──────▼──────┐    ┌────────▼────────┐
          │   PAYMENT   │    │   COMPENSATING  │
          │  COMPLETED  │    │   (releasing)   │
          └──────┬──────┘    └────────┬────────┘
                 │                    │ inventory released
          ┌──────▼──────┐    ┌────────▼────────┐
          │  CONFIRMED  │    │    CANCELLED    │
          └─────────────┘    └─────────────────┘
              (terminal)           (terminal)
```

## Saga vs Two-Phase Commit (2PC)

```
                    Saga                    Two-Phase Commit (2PC)
──────────────────────────────────────────────────────────────────────────
Isolation       None (reads may see        Full ACID isolation
                in-progress state)
Consistency     Eventual                   Strong (synchronous)
Availability    High (no blocking locks)   Low (coordinator holds locks)
Coupling        Loose (events/commands)    Tight (all participants must respond)
Failure handling Compensate (undo)         Coordinator crash → indefinite lock
Latency         Async, no waiting          Sync, slow (2 round trips)
Scalability     High                       Poor (blocking across services)
Best for        Microservices, long txns   Legacy databases, short txns

Why 2PC fails in microservices:
  1. Coordinator crash during phase 2 → participants hold locks forever
  2. Network partition between prepare and commit → uncertain state
  3. Every participant must be available simultaneously
  4. Doesn't work across different DB vendors / cloud services
```

## Dealing with Failures

### Retries before compensation

```
Not every failure warrants compensation. Transient errors should be retried.

Retry:
  → Network timeout calling Inventory Service
  → Service temporarily unavailable (HTTP 503)
  → Optimistic lock conflict (retry the local tx)

Compensate:
  → Business failure: item out of stock (no retry will fix this)
  → Payment declined (card permanently rejected)
  → Validation error (user data invalid)
```

### Idempotency in saga steps

```
Each step must be idempotent — the orchestrator may retry a step
after a crash before the reply was received.

Pattern: use the saga ID + step name as the idempotency key.
  "saga-order-99:reserve-inventory" → idempotency key
  If the reservation already exists → return existing result, do not re-reserve.
```

### Saga timeouts

```
If a participant never responds (crashed, network partition):
  → Orchestrator has a per-step timeout
  → On timeout: treat as failure → begin compensation
  → Participant may recover and complete after compensation started
    → Must handle "late success" gracefully (compensate anyway or check saga state)
```

## Outbox Pattern (Making Saga Steps Durable)

A common problem: service updates its DB but then crashes before publishing the event.

```
Problem:
  UPDATE orders SET status='PENDING' WHERE id=99   ← committed to DB
  Publish OrderCreated to event bus                ← CRASH — never sent
  → Saga never progresses past step 1

Solution: Outbox table (transactional outbox)
  BEGIN TRANSACTION
    UPDATE orders SET status='PENDING' WHERE id=99
    INSERT INTO outbox (event_type, payload) VALUES ('OrderCreated', {...})
  COMMIT
  ← both in same DB transaction — atomic
  A separate relay process reads outbox → publishes to event bus → deletes row

Result: event is guaranteed to be published at-least-once.
Consumers must be idempotent (they may receive the event more than once).
```

## Choreography vs Orchestration — When to Use

```
Use Choreography when:
  → Saga has 2–3 steps
  → Services are truly independent with no shared context
  → You want maximum decoupling
  → Teams own their services end-to-end

Use Orchestration when:
  → Saga has 4+ steps or conditional branches
  → You need a single place to monitor and debug
  → You need explicit retry/timeout policies per step
  → Compensations are complex or conditional
  → Using a workflow engine (Temporal, Conductor, AWS Step Functions)
```

## Common Pitfalls

```
1. Forgetting that compensating transactions are not rollbacks
   → Compensation executes after the original tx committed and may have been seen
   → UI may have shown "payment received" before compensation → communicate clearly

2. Not making saga steps idempotent
   → Orchestrator retries a step after crash → double reservation, double charge
   → Every step must be safe to call multiple times with the same saga+step ID

3. Missing compensation for every step
   → If step 5 of 8 fails, you need compensations for steps 1–4
   → Common to forget compensation for the very first step

4. Blocking on pivot transactions
   → Email sent, webhook fired, physical item shipped — cannot be recalled
   → Design the saga so pivot transactions come last, after all reversible steps

5. Saga state stored only in memory
   → Orchestrator crashes → saga state lost → stuck in partial state
   → Always persist saga state to a durable store before each step

6. Infinite compensation loop
   → Compensation itself fails → retry compensation → fails again
   → Need a dead-letter queue / manual intervention for stuck compensations
```

## Interview Answer Framework

When asked "How do you handle a distributed transaction across microservices?" in a design round:

1. **State why 2PC fails:** blocking locks, coordinator SPOF, doesn't scale across services
2. **Introduce saga:** sequence of local transactions with compensating transactions on failure
3. **Name the two approaches:** choreography (event-driven, no coordinator) vs orchestration (central driver)
4. **Give a concrete example:** order → reserve inventory → charge payment; if payment fails → release inventory → cancel order
5. **Explain compensation:** not a DB rollback — a new forward transaction that semantically undoes the effect
6. **Mention idempotency:** each saga step must be safe to retry (use saga ID + step as idempotency key)
7. **Mention the outbox pattern:** guarantee event publication by writing to an outbox table in the same DB transaction
8. **Acknowledge the consistency tradeoff:** saga provides eventual consistency, not strong isolation — dirty reads are possible during saga execution
9. **Name a production tool:** Temporal for complex workflows, MassTransit/NServiceBus for .NET
