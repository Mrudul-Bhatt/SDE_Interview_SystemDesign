# CQRS (Command Query Responsibility Segregation)

---

## What Problem Does It Solve?

In a traditional CRUD system, the same model handles both reads and writes against the same database. This causes friction as the system scales:

```
Traditional CRUD — one model, one DB:
  ┌────────────┐     read/write      ┌──────────┐
  │  API Layer │ ─────────────────→  │    DB    │
  └────────────┘                     └──────────┘

Problems at scale:
  Writes:
    → Need strong consistency, validation, business rules
    → Often touch many tables (joins, cascades)
    → Throughput limited by write lock contention

  Reads:
    → Need fast, denormalized views for UIs (dashboards, feeds)
    → Complex joins slow down under load
    → Read pattern often very different from write schema
    → 95% of traffic is reads — slowed by write-optimized schema

  The same data model cannot be optimized for both at once.
  Normalization helps writes (no duplication), hurts reads (many joins).
  Denormalization helps reads (precomputed), hurts writes (must update many places).
```

**CQRS** separates the system into two distinct sides:
- **Command side** — handles writes (create, update, delete); enforces business rules
- **Query side** — handles reads; serves precomputed, denormalized read models

---

## Core Concepts

```
Command   → an intent to change state ("PlaceOrder", "TransferFunds", "CancelBooking")
            Returns: nothing (or just an acknowledgement ID)
            Properties: validated, enforces business rules, may be rejected

Query     → a request for data ("GetOrderSummary", "GetUserDashboard")
            Returns: a read model (DTO/view)
            Properties: no side effects, always succeeds, can be cached

Separation:
  ┌──────────────────────────────────────────────────────────────────┐
  │                       API / Application                          │
  └─────────────────┬────────────────────────┬────────────────────── ┘
                    │ Commands                │ Queries
                    ▼                         ▼
          ┌──────────────────┐     ┌──────────────────────┐
          │  Command Handler │     │    Query Handler     │
          │  (write model)   │     │    (read model)      │
          └────────┬─────────┘     └──────────┬───────────┘
                   │                           │
                   ▼                           ▼
          ┌──────────────────┐     ┌──────────────────────┐
          │   Write DB       │     │     Read DB          │
          │  (normalized,    │     │  (denormalized,      │
          │   consistent)    │     │   pre-aggregated)    │
          └──────────────────┘     └──────────────────────┘
                   │                           ▲
                   └── event/sync ─────────────┘
                        (projections update
                         read models after write)
```

## Real-World Scenarios

```
1. E-commerce order feed
   → Write: complex order entity with validation, inventory checks, payment
   → Read: order history page needs denormalized view: item names, total, status
   → Read model is a flat table optimized for "give me user-42's last 20 orders"

2. Banking dashboard
   → Write: strict ACID transaction updating ledger entries
   → Read: account summary shows running balance, last 5 transactions
   → Read model precomputes the summary; never recalculates from raw ledger on each read

3. Social media feed
   → Write: "user posted" event processed once
   → Read: fan-out creates pre-built feed rows for each follower
   → Read is O(1) lookup; without CQRS it would be an O(followers) join on every page load

4. Analytics dashboard
   → Write: raw events streamed in (clicks, purchases, page views)
   → Read: aggregated counters per day/product/region, updated by projections
   → Dashboard reads pre-aggregated rows; never scans the raw event log

5. Inventory system
   → Write: individual stock adjustments with strict locking
   → Read: stock level page can serve slightly stale data from a read replica
   → Read does not need the same strong consistency as the write path
```

## Projections: Keeping Read Models in Sync

After a command mutates the write model, the read model must be updated. This is done through **projections** (also called read model updaters or event handlers).

```
Flow:
  Command arrives → Command Handler validates → Write DB updated
                                                     │
                                              Emit event / trigger
                                                     │
                                                     ▼
                                            Projection Handler
                                                     │
                                       ┌─────────────┴──────────────┐
                                       ▼                             ▼
                              Update read table A          Update read table B
                              (order_summaries)            (user_dashboard)

Example:
  Write: INSERT INTO orders (id, user_id, items, total) → normalized
  Projection: UPDATE order_summaries
              SET last_order_total = 129.00,
                  last_order_date  = NOW()
              WHERE user_id = 42
  Read: SELECT * FROM order_summaries WHERE user_id = 42 → instant, no joins
```

### Synchronous vs Asynchronous projections

```
Synchronous (same transaction):
  Write DB update + read model update in one transaction.
  Pros: read model immediately consistent.
  Cons: slows writes; tightly couples write and read paths.

Asynchronous (event-driven):
  Write DB commits → event published to message queue → projection handler updates read model.
  Pros: write path fast and decoupled; read model can be rebuilt from events.
  Cons: read model is eventually consistent (stale window between write and projection).

Which to use:
  → Asynchronous is standard for CQRS at scale.
  → Accept eventual consistency on the read side.
  → If the UI shows stale data for <1s, that is typically acceptable.
```

## Eventual Consistency in CQRS

```
t=0:  User submits "Place Order" command
t=1:  Write DB commits (order exists in write store)
t=2:  Event published to queue
t=3:  Projection handler reads event, updates order_summaries table
t=4:  User hits "My Orders" page — reads from order_summaries

If user hits the page at t=1.5 (before projection finishes):
  → Order not visible in the read model yet
  → User may see stale data

Strategies to handle the stale window:
  1. Optimistic UI update: client shows the new order immediately without waiting
     for the read model (the command already returned success).
  2. Read-your-own-writes: after a write, route the user's next read to the
     write DB directly (bypassing the eventually-consistent read model) for ~1s.
  3. Version token: command returns a version number; query waits until read model
     reaches that version before returning.
```

## CQRS + Event Sourcing

CQRS and Event Sourcing are separate patterns but are often combined.

```
Without Event Sourcing (state-based CQRS):
  Write DB holds current state (latest snapshot).
  Projection reads from write DB or listens to change events (CDC).
  Simple to understand; current state is directly queryable.

With Event Sourcing:
  Write DB holds an append-only log of events (never updates/deletes).
  Current state = replay of all events from the beginning.
  Projections subscribe to the event log and build read models.

  Event log:   OrderCreated → ItemAdded → PaymentCharged → OrderShipped
  Read model:  { orderId, status: "Shipped", total: 99.99, itemCount: 2 }

Benefits of Event Sourcing addition:
  → Full audit trail (every state change is recorded forever)
  → Time travel (replay events to any point in time)
  → New read models can be built by replaying the event log from scratch

Downsides:
  → More complex (event schema evolution, snapshotting for long-lived aggregates)
  → Querying current state requires projection — not direct SQL
```

## Simple CQRS vs Full CQRS

```
Simple CQRS (same DB, separate models in code):
  → Commands go through a command handler with validation and business logic
  → Queries go through a separate query handler that reads directly from DB
  → Same physical DB — just logical separation in the application layer
  → Low overhead; good starting point for most systems

  ┌──────────┐       ┌────────────────┐
  │ Command  │──────→│ Command Handler│──→ DB (write)
  └──────────┘       └────────────────┘     │
                                             │ same DB
  ┌──────────┐       ┌────────────────┐     │
  │  Query   │──────→│ Query Handler  │──→ DB (read, different table/view)
  └──────────┘       └────────────────┘

Full CQRS (separate DBs):
  → Write DB: relational, normalized, strong consistency (Postgres, MySQL)
  → Read DB: optimized for queries (Redis, Elasticsearch, Cassandra, read replica)
  → Projections keep read DB in sync via events
  → Higher complexity; justified when read and write scaling needs diverge significantly
```

## CQRS vs Traditional CRUD

```
Dimension          Traditional CRUD          CQRS
─────────────────────────────────────────────────────────────────────
Model              One model for all ops     Separate command + query models
DB                 One shared DB             Optionally separate write/read DBs
Consistency        Strong (immediate)        Eventual (async projections)
Read performance   Joins on every request    Pre-aggregated, denormalized reads
Write performance  Competes with reads       Isolated write path, no read pressure
Complexity         Low                       Higher (projections, sync, two models)
Schema flexibility Low (one schema fits all) High (read model shaped for each query)
Best for           Simple CRUD apps          High read traffic, complex queries, audit needs
```

## When to Use CQRS

```
Use CQRS when:
  → Read traffic >> write traffic (10:1 or more)
  → Read queries are complex (many joins, aggregations, fan-out)
  → Read and write scaling requirements differ (scale read replicas independently)
  → You need multiple read representations of the same data
  → Audit trail or event sourcing is a requirement
  → Domain logic is complex and benefits from a rich command model

Do NOT use CQRS when:
  → Simple CRUD application (todo lists, admin panels)
  → Small team — the extra complexity adds overhead without payoff
  → Strong consistency is required on every read
  → Domain is simple — the overhead of two models outweighs the benefit
  → "We might need it later" — start with CRUD, add CQRS when the pain is real
```

## Common Pitfalls

```
1. Applying CQRS everywhere
   → Most services don't need it; CRUD is simpler and sufficient
   → Only apply at the aggregate/bounded context level where scaling pain exists

2. Making commands return data
   → Commands should return acknowledgement only (ID or success/failure)
   → If you need the created object, issue a query after the command
   → Mixing command and query in one operation defeats the purpose

3. Read model gets too large / catches too many events
   → Each read model should serve one specific query pattern
   → If you're building a "universal" read model, you're rebuilding the write model

4. Forgetting to handle projection failure
   → If the projection handler crashes, read models go stale
   → Use idempotent projections + a dead-letter queue + replay from event log

5. Projection rebuilds are too slow
   → For Event Sourcing, replaying millions of events on startup is slow
   → Use snapshots: persist state at checkpoint intervals (every N events)

6. Not versioning events/commands
   → Event schema changes break old projections
   → Always version your events (OrderCreatedV1, OrderCreatedV2)
```

## Interview Answer Framework

When asked "How would you design a system with separate read and write paths?" in a design round:

1. **Name the pattern:** CQRS — Command Query Responsibility Segregation
2. **State the problem:** one model cannot be simultaneously optimized for complex writes (normalization, validation) and complex reads (denormalized, pre-aggregated)
3. **Split the model:** commands go to command handlers (enforce business rules, write to write DB); queries go to query handlers (serve precomputed read models)
4. **Explain projections:** after a write commits, an event triggers a projection that updates the read model asynchronously
5. **Acknowledge eventual consistency:** read model may be milliseconds stale; handle with optimistic UI updates or read-your-own-writes routing
6. **Choose the read store:** Postgres read replica for relational queries; Redis for hot paths; Elasticsearch for full-text; Cassandra for time-series
7. **Scale independently:** write DB scaled for write throughput; read DB scaled horizontally for read traffic
8. **Mention Event Sourcing as optional extension:** append-only event log enables full audit trail and read model rebuild from scratch
9. **State when NOT to use it:** simple CRUD, small team, strong consistency required — CQRS adds complexity that must be justified by scale
