# Schema Migrations in Large Systems

---

## What Problem Does It Solve?

A schema migration is any change to the database structure: adding a column, renaming a table, dropping an index, changing a data type. In a small system you can take downtime, run `ALTER TABLE`, and bring the service back up. In a large system that is not an option.

```
Small system — downtime migration (acceptable at 3 AM):
  Stop service → ALTER TABLE → Start service
  Downtime: seconds to minutes. Acceptable.

Large system — why this fails:
  Availability SLA: 99.99% → 52 minutes downtime/year budget
  Table size: 500M rows → ALTER TABLE takes hours (full table rewrite)
  Rolling deployment: 10 pods, upgraded one at a time
    → Old code + new schema running simultaneously for minutes
    → New code + old schema running simultaneously during rollback
  Multi-region: no global pause possible
```

The core problem in production: **during a deployment, old and new code versions run simultaneously against the same database**. The schema must be valid for both at the same time.

---

## Two Classes of Migrations

```
Additive (backward-compatible) — safe to apply without coordination:
  + Add a nullable column
  + Add a new table
  + Add an index (ideally CONCURRENTLY)
  + Add a new enum value
  → Old code ignores new columns/tables it doesn't know about.
  → New code reads/writes them. Both versions work simultaneously.

Destructive (breaking) — require multi-phase approach:
  - Rename a column
  - Drop a column or table
  - Change a column's type
  - Add a NOT NULL column without a default
  - Remove an enum value
  → Old code breaks if the column/table it expects is gone or renamed.
  → Must be done in multiple steps across multiple deployments.
```

## The Expand-Contract Pattern (Blue-Green Safe)

The industry-standard technique for zero-downtime breaking changes. Three phases across three separate deployments.

```
Goal: rename column  email  →  email_address

────────────────────────────────────────────────────────
Phase 1: EXPAND (add the new, keep the old)
────────────────────────────────────────────────────────
Migration:
  ALTER TABLE users ADD COLUMN email_address VARCHAR(255);

Application (v2 — deployed after migration):
  READ:  check email_address first, fall back to email
  WRITE: write to BOTH email and email_address

State: both columns exist, both being written
Old code (v1) still in service: reads email ✓
New code (v2) in service: reads email_address ✓

────────────────────────────────────────────────────────
Phase 2: MIGRATE DATA (backfill the old into the new)
────────────────────────────────────────────────────────
Migration (run in batches — never one massive UPDATE):
  UPDATE users
  SET email_address = email
  WHERE email_address IS NULL
  LIMIT 10000;   ← repeat until done

All rows now have email_address populated.
Application still writes to both.

────────────────────────────────────────────────────────
Phase 3: CONTRACT (remove the old)
────────────────────────────────────────────────────────
Application (v3 — deployed before migration):
  READ:  email_address only (stop reading email)
  WRITE: email_address only (stop writing email)

Migration (after v3 fully deployed — no code reads email):
  ALTER TABLE users DROP COLUMN email;

State: only email_address exists ✓
```

```
Timeline:

Deploy   │ Migrate │ Deploy │  Backfill  │ Deploy │ Migrate
  v2     │ Phase 1 │  v2    │            │  v3    │ Phase 3
─────────┼─────────┼────────┼────────────┼────────┼─────────
         +new col  │        │ copy data  │        │ drop old col
         keep old  │        │ in batches │        │
```

## Zero-Downtime ALTER TABLE Strategies

A naive `ALTER TABLE ADD COLUMN NOT NULL` rewrites the entire table and holds a full table lock. On a 500M-row table this means hours of downtime.

### Strategy 1: Add as nullable first, backfill, add constraint

```sql
-- Step 1: Add nullable — instant, no table rewrite (Postgres 11+)
ALTER TABLE orders ADD COLUMN shipped_at TIMESTAMPTZ;

-- Step 2: Backfill in batches (offline, no lock)
UPDATE orders SET shipped_at = created_at
WHERE shipped_at IS NULL AND status = 'shipped'
LIMIT 10000;  -- repeat

-- Step 3: Add NOT NULL constraint — Postgres validates without full rewrite
ALTER TABLE orders ALTER COLUMN shipped_at SET NOT NULL;
-- Or safer: ADD CONSTRAINT ... NOT NULL NOT VALID; then VALIDATE CONSTRAINT;
```

### Strategy 2: CONCURRENTLY index creation

```sql
-- Blocks ALL reads and writes — dangerous on large tables:
CREATE INDEX idx_orders_user ON orders(user_id);

-- Non-blocking — builds index while table is live:
CREATE INDEX CONCURRENTLY idx_orders_user ON orders(user_id);
-- Takes longer but does not block.
-- Cannot run inside a transaction block.
```

### Strategy 3: Table rename via ghost table (pt-online-schema-change)

```
For changes that require a full table rewrite (e.g. change column type):
  1. Create new_orders table with the new schema
  2. Copy rows from orders → new_orders in batches
  3. Set up triggers: writes to orders are mirrored to new_orders
  4. When caught up: rename orders → orders_old, new_orders → orders (instant)
  5. Drop orders_old

Tools: pt-online-schema-change (MySQL), pg_repack (Postgres), gh-ost (GitHub)
```

## Migration Versioning

Every migration is a versioned, numbered file applied exactly once. Never edit a previously applied migration.

```
migrations/
  V001__create_users_table.sql
  V002__add_email_index.sql
  V003__add_orders_table.sql
  V004__rename_email_to_email_address_phase1.sql
  V005__backfill_email_address.sql
  V006__drop_old_email_column.sql

Migration tracking table (managed by the tool):
  schema_migrations
  ┌────────────┬──────────────────────────────────┬──────────────────────┐
  │ version    │ description                      │ applied_at           │
  ├────────────┼──────────────────────────────────┼──────────────────────┤
  │ V001       │ create_users_table               │ 2024-01-10 09:00:00  │
  │ V002       │ add_email_index                  │ 2024-01-11 14:30:00  │
  │ V003       │ add_orders_table                 │ 2024-02-01 11:00:00  │
  └────────────┴──────────────────────────────────┴──────────────────────┘

Rules:
  → Applied migrations are IMMUTABLE — never modify them
  → New changes = new migration file
  → Checksums stored to detect accidental edits
  → Applied once and exactly once (idempotency via tracking table)
```

## Rollback Strategies

Schema rollbacks are harder than code rollbacks — you cannot un-drop a column with data.

```
Option 1: Write undo scripts (forward-only with manual undo)
  V004__add_column.sql      ← applied forward
  V004__add_column.undo.sql ← written alongside, run manually on rollback
  Pros: explicit control.
  Cons: undo scripts often not tested, can cause data loss.

Option 2: Forward-only migrations (preferred for production)
  Never roll back a migration. Instead, write a new migration to undo.
  V004__add_column.sql
  V005__remove_column.sql   ← "rollback" is just a new forward migration
  Pros: always moving forward; tested path; no data loss.
  Cons: schema churn accumulates over time.

Option 3: Blue-Green deployment
  Keep the old database instance alive.
  If the new deployment fails: switch traffic back to old DB + old code.
  Requires: the new migration must be backward-compatible with old code (Expand phase only).
  Works for additive migrations — not for destructive ones.

Option 4: Point-in-time restore
  If a migration caused data corruption: restore DB to snapshot before migration.
  Very destructive — loses all writes since the snapshot.
  Last resort for catastrophic errors.
```

## Large Table Backfills

When adding a column that requires populating existing rows, a single `UPDATE` is catastrophic:

```
Bad — one massive UPDATE:
  UPDATE orders SET region = 'us-east' WHERE region IS NULL;
  → Locks millions of rows → blocks all writes → minutes of downtime

Good — batched updates with sleep:
  WHILE rows_updated > 0:
    UPDATE orders
    SET region = derive_region(shipping_zip)
    WHERE region IS NULL
    ORDER BY id
    LIMIT 5000;                  ← small batch: short lock window
    SLEEP 10ms;                  ← yield to application writes between batches
    SET rows_updated = affected_rows;

Why sleep between batches?
  → Prevents replication lag (replica falls behind on a write flood)
  → Prevents lock contention with live application writes
  → Allows the query planner to keep using the index on each batch

Typical batch sizes: 1,000–10,000 rows per batch
Typical sleep: 10–100ms between batches
```

## Deployment Ordering: Migrations vs Code

The order in which you deploy migrations and code matters critically.

```
Rule: Always deploy migrations BEFORE code that requires them.
      Always remove old code BEFORE migrations that break it.

For additive migrations (new column, new table):
  1. Deploy migration  ← new column exists, old code ignores it
  2. Deploy new code   ← new code reads/writes new column
  Rollback: safe — old code still works with new column (it ignores it)

For destructive migrations (drop column):
  1. Deploy new code   ← code no longer references old column
  2. Deploy migration  ← drop old column safely (nothing reads it)
  Rollback: new code deployed but old migration not run — old code rollback
            would fail because code expects column gone but it's still there
            → NEVER combine code rollback with schema rollback

Summary:
  Additive: migrate first, code second
  Destructive: code first (stop using it), migrate second (remove it)
```

## Multi-Tenant Schema Migrations

Large SaaS systems sometimes have per-tenant schemas. Migrations must run across thousands of schemas.

```
Approach 1: Shared schema (all tenants in one table with tenant_id column)
  → Single migration runs once ✓
  → Simple but less isolation

Approach 2: Per-tenant schema (schema per tenant in Postgres)
  → Migration must run N times (one per tenant)
  → Must be parallelized: sequential = hours for 10,000 tenants
  → Migration runner iterates schemas, applies in parallel batches
  → Failed tenants tracked and retried separately

Approach 3: Per-tenant database
  → Maximum isolation; most complex
  → Migration runs against each DB; tracked in a control plane DB
  → Blue-green per tenant; tenant can be pinned to old schema during rollout
```

## Common Pitfalls

```
1. Editing an already-applied migration
   → Other environments have the old version; checksums diverge
   → Flyway/Liquibase fail with checksum mismatch error
   → Always create a new migration instead

2. Long-running migration in a transaction
   → BEGIN; ALTER TABLE ... (30 minutes); COMMIT;
   → Holds a lock for the entire duration
   → Break large migrations into small transactional steps

3. Deploying destructive migration before removing code references
   → Code tries to read a column that no longer exists → 500 errors
   → Always use expand-contract: code stops using the column before you drop it

4. Not testing rollback path
   → Forward-only policy helps, but undo scripts that are never tested
      will fail when you need them most
   → Run migration + rollback in CI on every PR

5. Backfilling in one UPDATE
   → Blocks the table, replication lag, OOM on large rows
   → Always batch: LIMIT + loop + sleep

6. Adding NOT NULL column with no default to a large table
   → Full table rewrite in older Postgres (pre-11), huge lock
   → Add nullable first, backfill, then add constraint

7. Schema drift between environments
   → Dev DB manually altered, migration never written
   → CI always runs migrations from scratch against a clean DB
```

## Interview Answer Framework

When asked "How do you handle database schema migrations in a large system?" in a design round:

1. **State the core problem:** old and new code run simultaneously during rolling deploys — schema must be valid for both
2. **Classify the change:** additive (safe, deploy migration first) vs destructive (breaking, requires multi-phase)
3. **Introduce expand-contract:** Phase 1 add new column; Phase 2 backfill; Phase 3 remove old — three separate deployments
4. **Explain deployment ordering:** additive → migrate then deploy code; destructive → deploy code (stop using) then migrate (remove)
5. **Address large table concerns:** `ALTER TABLE` holds a lock — add nullable first, backfill in small batches with sleep, then add constraint
6. **Mention CONCURRENTLY for indexes:** `CREATE INDEX CONCURRENTLY` — non-blocking
7. **Versioned migration files:** V001, V002... applied exactly once, tracked in schema_migrations table, never edited
8. **Rollback strategy:** prefer forward-only migrations (new file undoes old) over undo scripts; blue-green for additive-only rollouts
9. **Name a tool:** Flyway (Java/SQL), EF Core Migrations (.NET), Alembic (Python)
