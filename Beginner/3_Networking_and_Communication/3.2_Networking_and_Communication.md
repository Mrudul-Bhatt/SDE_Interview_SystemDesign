# REST vs GraphQL vs gRPC — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine you're at a restaurant.

**REST** is a fixed menu. You order "Meal #3" and get exactly what's on the menu — a burger, fries, and a drink. Want just the burger? Too bad, you get the whole meal. Want a burger AND a salad from Meal #7? That's two separate orders.

**GraphQL** is a custom order system. You tell the chef exactly what you want — "burger with no pickles, small fries, and a water." One trip to the kitchen. You get precisely what you asked for. Nothing extra.

**gRPC** is a walkie-talkie between two chefs in the same kitchen. Extremely fast, very precise, uses a shared codebook (proto file) so they don't waste words. Not designed for customers — designed for internal kitchen-to-kitchen communication.

---

## REST — The Default Everyone Knows

### What it is

REST (Representational State Transfer) is an architectural style where you interact with **resources** via standard HTTP methods.

```
Resource = a noun (user, event, booking, seat)
Method   = a verb (GET, POST, PUT, DELETE, PATCH)

GET    /events/123          → fetch event 123
POST   /bookings            → create a booking
PUT    /bookings/456        → replace booking 456
PATCH  /bookings/456        → partially update booking 456
DELETE /bookings/456        → cancel booking 456
```

### The mental model

Every URL is a **resource**. Every HTTP method is an **action on that resource**. The combination is self-describing.

### What a REST response looks like

```json
GET /events/123

{
  "id": 123,
  "name": "Taylor Swift - Eras Tour",
  "venue": "SoFi Stadium",
  "date": "2025-08-15",
  "capacity": 70000,
  "ticketsRemaining": 1243,
  "artist": {
    "id": 55,
    "name": "Taylor Swift",
    "genre": "Pop",
    "monthlyListeners": 95000000
  },
  "organizer": {
    "id": 12,
    "name": "Live Nation",
    "founded": 1996,
    "country": "USA"
  }
}
```

### The two core problems with REST

**Problem 1: Over-fetching**

You're building a mobile app that shows a list of events. You only need `id`, `name`, and `date`. But REST gives you the entire object — venue, capacity, artist details, organizer history. You're downloading 10x more data than you need. On a 3G mobile connection, this is real pain.

**Problem 2: Under-fetching (N+1 problem)**

You want to show a user's dashboard: their profile + last 5 bookings + each booking's event name.

```
Request 1: GET /users/42              → get user profile
Request 2: GET /users/42/bookings     → get booking IDs [101, 102, 103, 104, 105]
Request 3: GET /events/201            → event for booking 101
Request 4: GET /events/202            → event for booking 102
Request 5: GET /events/203            → event for booking 103
Request 6: GET /events/204            → event for booking 104
Request 7: GET /events/205            → event for booking 105
```

7 round trips for one screen. Each round trip adds latency. On mobile this is awful.

You could create a custom endpoint `/users/42/dashboard` that returns everything in one call — but now you're designing endpoints around UI screens, not resources. REST is breaking down.

## GraphQL — Ask For Exactly What You Need

### What it is

GraphQL is a **query language for your API**. Instead of the server deciding what data comes back, **the client specifies exactly what it wants** in a single request.

One endpoint. One request. Precisely the data you need.

```
Endpoint: POST /graphql
```

Always the same URL. The query in the body determines what comes back.

### The same dashboard — one GraphQL query

```graphql
query UserDashboard {
  user(id: 42) {
    name
    email
    bookings(last: 5) {
      id
      date
      event {
        name
        venue
      }
    }
  }
}
```

**One request. Exactly this response:**

```json
{
  "data": {
    "user": {
      "name": "Mrudul",
      "email": "mrudul@email.com",
      "bookings": [
        {
          "id": 101,
          "date": "2025-03-15",
          "event": { "name": "Taylor Swift", "venue": "SoFi Stadium" }
        },
        {
          "id": 102,
          "date": "2025-04-20",
          "event": { "name": "Coldplay", "venue": "Rose Bowl" }
        }
      ]
    }
  }
}
```

No over-fetching. No 7 round trips. No custom endpoints per screen.

### How GraphQL works internally

```
Client sends query
       ↓
GraphQL server parses query
       ↓
Resolvers execute for each field
  user(id:42)    → DB query for user
  bookings(last:5) → DB query for bookings
  event { name } → DB query for events
       ↓
Results assembled into exact shape client requested
       ↓
Single response returned
```

Each field has a **resolver** — a function that fetches that piece of data. GraphQL orchestrates them.

### The N+1 problem inside GraphQL

Wait — doesn't GraphQL have the same N+1 problem internally?

```
user → 5 bookings → for each booking, fetch event
                    → 5 separate DB queries ← N+1 inside resolver
```

Yes. This is GraphQL's most famous internal problem.

**Solution: DataLoader (batching)**

```javascript
// Without DataLoader: 5 separate queries
booking 101 → SELECT * FROM events WHERE id = 201
booking 102 → SELECT * FROM events WHERE id = 202
...

// With DataLoader: 1 batched query
[201, 202, 203, 204, 205] → SELECT * FROM events WHERE id IN (201,202,203,204,205)
```

DataLoader collects all event IDs requested within one tick of the event loop, then fetches them in one batch. This is a key GraphQL internals question.

### GraphQL weaknesses

```
Complexity:    Schema design, resolvers, DataLoader — more upfront work
Caching:       REST caches at URL level (trivial). GraphQL POST requests
               don't cache at HTTP level — need custom cache logic.
Rate limiting: REST limits per endpoint. GraphQL — one endpoint,
               complex queries can be arbitrarily expensive.
Query depth:   Malicious query can ask user→bookings→event→venue→events→...
               infinitely deep. Need depth limiting.
```

## gRPC — Speed Between Services

### What it is

gRPC is a **Remote Procedure Call framework** built by Google. Instead of thinking in HTTP resources, you think in **function calls** between services.

```
REST:  "I want to GET the resource at /bookings/123"
gRPC:  "I want to call the GetBooking function with argument id=123"
```

It uses **Protocol Buffers (protobuf)** — a binary serialization format — instead of JSON. And it runs on **HTTP/2** instead of HTTP/1.1.

### The proto file — the contract

Before writing any code, you define your service in a `.proto` file:

```protobuf
syntax = "proto3";

service BookingService {
  rpc GetBooking (BookingRequest) returns (BookingResponse);
  rpc CreateBooking (CreateBookingRequest) returns (BookingResponse);
  rpc StreamSeatUpdates (EventRequest) returns (stream SeatUpdate);
}

message BookingRequest {
  int32 id = 1;
}

message BookingResponse {
  int32 id = 1;
  string event_name = 2;
  string seat = 3;
  float price = 4;
}

message SeatUpdate {
  string seat_id = 1;
  bool is_available = 2;
}
```

From this file, gRPC **auto-generates** client and server code in any language — C#, Go, Python, Java, etc. The generated code handles serialization, networking, and error handling automatically.

### Why gRPC is fast

**Reason 1: Binary encoding (protobuf vs JSON)**

```
JSON (text):
{"id":123,"event_name":"Taylor Swift","seat":"A5","price":250.00}
→ 63 bytes, must be parsed as text

Protobuf (binary):
→ ~20 bytes, parsed directly into memory structures
→ 3x smaller, 5-10x faster to serialize/deserialize
```

**Reason 2: HTTP/2 multiplexing**

```
HTTP/1.1:                    HTTP/2:
Request 1  ──────────→       Request 1 ──┐
(wait for response)          Request 2 ──┤→ all on one connection
Response 1 ←──────────       Request 3 ──┘   simultaneously
Request 2  ──────────→       Response 1 ←─┐
...                          Response 3 ←─┤ out of order fine
                             Response 2 ←─┘
```

One connection. Multiple requests in flight simultaneously. No head-of-line blocking.

**Reason 3: Streaming**

gRPC has 4 communication patterns REST simply doesn't have natively:

```
1. Unary (like REST):
   Client sends one request → Server sends one response

2. Server streaming:
   Client sends one request → Server streams many responses
   Use case: "Stream me live seat availability updates for Event 123"

3. Client streaming:
   Client streams many requests → Server sends one response
   Use case: Client streams sensor data, server aggregates and responds once

4. Bidirectional streaming:
   Client and server both stream simultaneously
   Use case: Live chat, real-time collaborative editing
```

```csharp
// Server streaming in C# — AXS seat availability
public override async Task StreamSeatUpdates(
    EventRequest request,
    IServerStreamWriter<SeatUpdate> responseStream,
    ServerCallContext context)
{
    while (!context.CancellationToken.IsCancellationRequested)
    {
        var updates = await _seatService.GetUpdates(request.EventId);

        foreach (var update in updates)
            await responseStream.WriteAsync(update);

        await Task.Delay(500); // push updates every 500ms
    }
}
```

### gRPC weaknesses

```
Browser support:  Browsers can't call gRPC directly (HTTP/2 trailers
                  not supported). Need gRPC-Web or a gateway layer.
Human readable:   Binary protobuf is not human readable.
                  REST you can curl. gRPC you need tooling.
Flexibility:      Schema changes require recompiling proto files
                  and updating all clients. Tight coupling.
Debugging:        Wireshark shows binary blobs. REST shows JSON.
```

| Dimension       | REST              | GraphQL                | gRPC                   |
| --------------- | ----------------- | ---------------------- | ---------------------- |
| Protocol        | HTTP/1.1          | HTTP/1.1               | HTTP/2                 |
| Format          | JSON/XML          | JSON                   | Protobuf (binary)      |
| Contract        | OpenAPI/Swagger   | Schema (SDL)           | Proto file             |
| Flexibility     | Fixed endpoints   | Client-defined queries | Fixed methods          |
| Over-fetching   | Common problem    | Solved                 | Not an issue           |
| Caching         | Easy (HTTP cache) | Hard (custom needed)   | Hard (custom needed)   |
| Streaming       | Partial (SSE, WS) | Subscriptions          | Native, 4 patterns     |
| Browser support | ✅ Native          | ✅ Native               | ⚠️ Needs gRPC-Web       |
| Speed           | Moderate          | Moderate               | Fastest                |
| Learning curve  | Low               | Medium                 | High                   |
| Best for        | Public APIs       | Complex client needs   | Internal microservices |

## When to Use What — The Decision Framework

```
Is this a public API consumed by external developers?
  → REST. Universally understood. Easy to document.

Do you have multiple clients (mobile, web, TV) with different data needs?
  → GraphQL. Each client asks for exactly what it needs.

Is this communication between two internal backend services?
  → gRPC. Fastest. Auto-generated clients. Type-safe contract.

Do you need real-time streaming (live scores, seat updates)?
  → gRPC streaming. OR WebSockets over REST.

Is your team small and moving fast?
  → REST. Least overhead to set up and debug.

Are you building a data-rich dashboard with many entity relationships?
  → GraphQL. Avoid N+1 round trips, avoid over-fetching.
```

### Question 2: Implement rate limiting for a REST API

> "Implement a fixed window rate limiter — max 100 requests per minute per user."

```csharp
public class RateLimiter
{
    // userId → (request count, window start time)
    private readonly Dictionary<string, (int count, DateTime windowStart)> _store = new();
    private const int MaxRequests = 100;
    private readonly TimeSpan _window = TimeSpan.FromMinutes(1);
    private readonly object _lock = new();

    public bool IsAllowed(string userId)
    {
        lock (_lock)
        {
            var now = DateTime.UtcNow;

            if (!_store.TryGetValue(userId, out var state) ||
                now - state.windowStart >= _window)
            {
                // New window
                _store[userId] = (1, now);
                return true;
            }

            if (state.count >= MaxRequests)
                return false; // 429 Too Many Requests

            _store[userId] = (state.count + 1, state.windowStart);
            return true;
        }
    }
}
```

### Question 3: Serialize/deserialize without a library *(tests protobuf understanding)*

> "Implement a simple binary serializer for a message with fields: id (int), name (string)."

```csharp
public class SimpleSerializer
{
    // Format: [4 bytes: id][4 bytes: name length][N bytes: name]
    public byte[] Serialize(int id, string name)
    {
        var nameBytes = Encoding.UTF8.GetBytes(name);
        var result = new byte[4 + 4 + nameBytes.Length];

        BitConverter.GetBytes(id).CopyTo(result, 0);
        BitConverter.GetBytes(nameBytes.Length).CopyTo(result, 4);
        nameBytes.CopyTo(result, 8);

        return result;
    }

    public (int id, string name) Deserialize(byte[] data)
    {
        int id = BitConverter.ToInt32(data, 0);
        int nameLength = BitConverter.ToInt32(data, 4);
        string name = Encoding.UTF8.GetString(data, 8, nameLength);
        return (id, name);
    }
}
```

This shows you understand **why protobuf is fast** — binary beats text every time.

### Question 4: Implement a simple GraphQL-like field selector

> "Given a JSON object and a list of requested fields, return only those fields."

```csharp
public Dictionary<string, object> SelectFields(
    Dictionary<string, object> data,
    HashSet<string> requestedFields)
{
    return data
        .Where(kvp => requestedFields.Contains(kvp.Key))
        .ToDictionary(kvp => kvp.Key, kvp => kvp.Value);
}

// Usage
var event = new Dictionary<string, object> {
    ["id"] = 123,
    ["name"] = "Taylor Swift",
    ["capacity"] = 70000,
    ["organizer"] = "Live Nation"
};

var requested = new HashSet<string> { "id", "name" };
var result = SelectFields(event, requested);
// → { "id": 123, "name": "Taylor Swift" }
// Capacity and organizer not included — no over-fetching
```

Simple but shows you understand the core mechanic GraphQL solves.

## One-Line Summary for Interviews

> **REST models resources over HTTP for universal interoperability; GraphQL lets clients query exactly the data shape they need in one request, eliminating over-fetching and round trips; gRPC uses binary protobuf over HTTP/2 for maximum speed and native streaming between internal services — and the right answer in any system design is almost always REST for public APIs, GraphQL for complex multi-client frontends, and gRPC for internal microservice communication.**


# TCP vs UDP — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine you're sending something important across the country.

**TCP is a courier service with tracking.** You hand the package to FedEx. They give you a tracking number. They confirm delivery. If the package gets lost, they reship it automatically. You're guaranteed it arrives, in the right order, intact. But all that tracking and confirmation takes time and overhead.

**UDP is dropping a letter in a mailbox.** You write the address, drop it in, walk away. No confirmation. No tracking. No guarantee it arrives. But it's instant — no waiting for receipts, no back-and-forth handshakes. If you're sending a million flyers and don't care if a few get lost, this is far more efficient.

The question isn't "which is better." It's **"what does your application actually need?"**

---

## What TCP Does — And Why It Costs

TCP = **Transmission Control Protocol**

It guarantees three things:
- **Delivery** — every byte arrives
- **Order** — bytes arrive in the sequence they were sent
- **Integrity** — corrupted data is detected and retransmitted

### The TCP Handshake (cost you pay upfront)

Before a single byte of real data flows, TCP does this:

```
Client                    Server
  │                          │
  │──── SYN ────────────────→│   "I want to connect"
  │                          │
  │←─── SYN-ACK ────────────│   "OK, I'm ready"
  │                          │
  │──── ACK ────────────────→│   "Great, let's go"
  │                          │
  │   [data flows now]       │
```

**3 messages before data.** This is the **3-way handshake**. On a 100ms latency connection, you've spent 300ms before sending anything.

For HTTPS (TLS over TCP), add a TLS handshake on top — easily 5–7 round trips total before real data flows.

### How TCP guarantees delivery

Every packet gets a sequence number. The receiver sends an ACK (acknowledgment) for every packet received.

```
Sender                        Receiver
  │──── Packet 1 (seq=1) ────→│
  │──── Packet 2 (seq=2) ────→│
  │──── Packet 3 (seq=3) ──✗  │  (lost in network)
  │←─── ACK 1 ───────────────│
  │←─── ACK 2 ───────────────│
  │     (no ACK 3 arrives)    │
  │     (timeout...)          │
  │──── Packet 3 (resend) ───→│  retransmitted automatically
  │←─── ACK 3 ───────────────│
```

If packet 3 is lost, **everything waits**. Even if packets 4, 5, 6 arrived fine, the application doesn't see them until 3 is retransmitted and slotted in order. This is called **Head-of-Line Blocking** — a critical concept for interviews.

### TCP flow control and congestion control

TCP also adjusts its sending rate based on:
- **Receiver's buffer** (don't overwhelm the receiver)
- **Network congestion** (if packets drop, slow down)

This makes TCP a good citizen of the network — but adds latency overhead in high-throughput scenarios.

---

## What UDP Does — And Why It's Fast

UDP = **User Datagram Protocol**

It guarantees exactly **nothing**. No handshake. No acknowledgment. No ordering. No retransmission.

```
Sender                        Receiver
  │──── Packet ──────────────→│  (maybe arrives)
  │──── Packet ──────────────→│  (maybe arrives, maybe out of order)
  │──── Packet ──────────✗    │  (lost — sender never knows)
  │──── Packet ──────────────→│  (arrives fine)
```

### Why this is sometimes exactly what you want

**Scenario: You're streaming live video of a concert.**

A video frame from 500ms ago is worthless. If TCP retransmits it, you're receiving old data while new frames pile up — the stream freezes and stutters. You'd rather just **skip the lost frame** and show the next one. A tiny visual glitch beats a frozen screen.

**Scenario: You're playing a multiplayer game.**

Your character's position updates 60 times per second. If one position update is lost, you don't want TCP to pause everything and wait for a retransmit. By the time it arrives, your character has moved 5 more positions. The old position update is **irrelevant**. Just use the newest one.

**Scenario: DNS lookup.**

You send one small packet asking "what's the IP of axs.com?" The response is one small packet back. If it's lost, you just send the question again. Building a full TCP connection for this one-shot exchange would be enormous overhead.

UDP's speed comes from one thing: **no waiting**. No handshake, no ACKs, no retransmits, no ordering buffer. Packets fly and whatever arrives, arrives.

---

## The Core Difference, Visualized

```
TCP — reliable, ordered, slow:

Sent:     [1][2][3][4][5]
Network:  [1][2]  ✗  [4][5]    (3 lost)
Received: [1][2][WAIT...][3][4][5]   ← waits for 3, then delivers in order

UDP — unreliable, unordered, fast:

Sent:     [1][2][3][4][5]
Network:  [1][2]  ✗  [4][5]    (3 lost)
Received: [1][2][4][5]          ← 3 just gone, no waiting, app handles it
```

## TCP vs UDP — Full Comparison

| Dimension             | TCP                        | UDP                       |
| --------------------- | -------------------------- | ------------------------- |
| Connection            | Stateful (3-way handshake) | Stateless (no setup)      |
| Delivery guarantee    | ✅ Every packet             | ❌ Best effort             |
| Ordering guarantee    | ✅ In sequence              | ❌ May arrive out of order |
| Error recovery        | ✅ Retransmits lost packets | ❌ Lost = gone             |
| Speed                 | Slower (overhead)          | Faster (no overhead)      |
| Head-of-line blocking | ✅ Yes (a problem)          | ❌ No                      |
| Congestion control    | ✅ Built in                 | ❌ Must implement yourself |
| Header size           | 20–60 bytes                | 8 bytes                   |
| Best for              | Accuracy-critical data     | Latency-critical data     |

---

## When to Choose TCP

Choose TCP when **losing or reordering data is unacceptable:**

```
Web browsing (HTTP/HTTPS)
  → HTML must arrive complete and in order
  → Missing a CSS byte = broken page layout

File transfers (FTP, SFTP)
  → A corrupted ZIP file is useless
  → Every byte must arrive perfectly

Email (SMTP, IMAP)
  → Missing half an email is worse than a delayed email

Database connections
  → A missing byte in a SQL query = wrong query executed
  → Catastrophic for financial data

Payment processing
  → $250.00 must not become $25.00 due to a dropped byte

SSH / remote terminal
  → Commands must arrive in exact order
  → "rm -rf /" with bits dropped is dangerous

Chat messages (WhatsApp, Slack)
  → Messages must arrive and in order
  → Delivery receipts matter
```

---

## When to Choose UDP

Choose UDP when **latency matters more than perfection:**

```
Live video streaming (Zoom, YouTube Live, Twitch)
  → Stale frame = visual glitch (acceptable)
  → TCP retransmit = freeze (unacceptable)

Online multiplayer games (Valorant, CS2, FIFA)
  → Position update from 100ms ago is irrelevant
  → Low latency beats perfect delivery

Voice calls (VoIP — WhatsApp calls, Zoom audio)
  → Dropped audio sample = tiny click (acceptable)
  → Delayed retransmit = robotic voice (unacceptable)

DNS lookups
  → One packet out, one packet back
  → TCP handshake would cost more than the query itself

IoT sensor data
  → Sending temperature every 100ms
  → Missing one reading is fine
  → Connection overhead per reading is wasteful

Live sports scores / stock tickers
  → Old score is irrelevant once new one arrives
  → Speed beats completeness
```

## The Middle Ground — When You Build Reliability on UDP

This is the most interesting part for interviews. Sometimes you need **some** of TCP's guarantees but not all — and you need UDP's speed.

The solution: **implement selective reliability on top of UDP yourself.**

This is exactly what several modern protocols do:

### QUIC (HTTP/3) — UDP with reliability built in

Google built QUIC (now standardized as HTTP/3) because TCP has a fundamental flaw for modern web: **head-of-line blocking at the transport layer.**

```
HTTP/2 over TCP:
Stream 1: [A1][A2][A3]    (image download)
Stream 2: [B1][B2][B3]    (CSS file)
Stream 3: [C1][C2][C3]    (JS file)

If B2 is lost:
→ TCP blocks ALL streams waiting for B2 retransmit
→ Stream 1 and 3 are stuck even though their packets arrived fine
```

```
HTTP/3 over QUIC (UDP-based):
If B2 is lost:
→ Only Stream 2 pauses
→ Streams 1 and 3 continue unaffected
→ No cross-stream head-of-line blocking
```

QUIC runs on UDP but implements its own:
- Packet acknowledgment
- Loss detection and retransmission
- Congestion control
- Encryption (TLS 1.3 built in)
- Connection migration (your connection survives switching from WiFi to 4G)

### WebRTC — Real-time browser communication

Used for video calls in browsers. Runs on UDP internally, but adds:
- SRTP (Secure RTP) for encrypted audio/video
- RTCP for feedback and loss statistics
- Application-level forward error correction (FEC)

### Game networking protocols

Games like Valorant use custom UDP-based protocols that:
- Assign sequence numbers to packets
- Track which were received
- Only retransmit **critical** packets (player actions)
- Drop **non-critical** ones (position interpolation can fill gaps)

This lets games choose per-message what matters:
```
"Player fired gun"       → retransmit if lost (critical)
"Player position update" → drop if lost, next update replaces it (not critical)
```

## Coding Questions From TCP/UDP

Like CDN and load balancers, you won't be asked to "implement TCP." But the concepts directly generate coding questions:

---

### Question 1: Implement a reliable message delivery system over UDP

> "Implement a simple reliable UDP sender — messages must be delivered exactly once, in order."

This is the core of what QUIC does. Tests if you understand what TCP actually provides.

```csharp
public class ReliableUdpSender
{
    private readonly UdpClient _client;
    private int _sequenceNumber = 0;
    // seq → (message, timestamp, ack received)
    private readonly Dictionary<int, (byte[] data, DateTime sent, bool acked)> _unacked = new();

    public async Task SendAsync(byte[] message, IPEndPoint endpoint)
    {
        int seq = Interlocked.Increment(ref _sequenceNumber);

        // Prepend sequence number to message
        var packet = new byte[4 + message.Length];
        BitConverter.GetBytes(seq).CopyTo(packet, 0);
        message.CopyTo(packet, 4);

        _unacked[seq] = (packet, DateTime.UtcNow, false);

        await _client.SendAsync(packet, packet.Length, endpoint);

        // Retry if no ACK received within timeout
        _ = RetryIfUnackedAsync(seq, endpoint);
    }

    private async Task RetryIfUnackedAsync(int seq, IPEndPoint endpoint)
    {
        await Task.Delay(200); // wait 200ms for ACK

        if (_unacked.TryGetValue(seq, out var state) && !state.acked)
        {
            // Retransmit
            await _client.SendAsync(state.data, state.data.Length, endpoint);
            _ = RetryIfUnackedAsync(seq, endpoint); // retry again
        }
    }

    public void OnAckReceived(int seq)
    {
        if (_unacked.TryGetValue(seq, out var state))
            _unacked[seq] = (state.data, state.sent, true);
    }
}
```

**What this shows:** You understand that TCP's reliability isn't magic — it's sequence numbers + acknowledgments + retransmission timers. You can implement a version of it.

### Question 2: Sliding window flow control *(medium-hard)*

> "Implement a sliding window protocol — sender can have at most W unacknowledged packets in flight."

This is TCP's flow control mechanism. Classic interview question at networking-aware companies.

```csharp
public class SlidingWindowSender
{
    private readonly int _windowSize;
    private int _base = 0;       // oldest unacked packet
    private int _nextSeq = 0;    // next packet to send
    private readonly Dictionary<int, byte[]> _buffer = new();
    private readonly SemaphoreSlim _windowSlot;

    public SlidingWindowSender(int windowSize)
    {
        _windowSize = windowSize;
        _windowSlot = new SemaphoreSlim(windowSize, windowSize);
    }

    public async Task SendAsync(byte[] packet)
    {
        // Block if window is full (W unacked packets in flight)
        await _windowSlot.WaitAsync();

        int seq = _nextSeq++;
        _buffer[seq] = packet;

        await TransmitAsync(seq, packet);
    }

    public void OnAck(int ackSeq)
    {
        // Slide window forward — free up slots for new packets
        for (int seq = _base; seq <= ackSeq; seq++)
        {
            if (_buffer.ContainsKey(seq))
            {
                _buffer.Remove(seq);
                _windowSlot.Release(); // window slides → allow more sends
            }
        }
        _base = ackSeq + 1;
    }

    private async Task TransmitAsync(int seq, byte[] packet)
    {
        // Actual UDP send + start retransmit timer
        Console.WriteLine($"Sending packet {seq}");
        await Task.CompletedTask;
    }
}
```

**Why this matters:** TCP's throughput is determined by window size ÷ RTT. Understanding this lets you reason about why a high-latency link (satellite: 600ms RTT) has terrible TCP throughput even with plenty of bandwidth.

### Question 4: Design a DNS resolver *(classic — uses UDP)*

> "Explain how DNS works and why it uses UDP. Then implement a simple DNS cache."

```csharp
public class DnsCache
{
    private readonly Dictionary<string, (string ip, DateTime expiry)> _cache = new();

    public string Resolve(string domain)
    {
        // Check cache first
        if (_cache.TryGetValue(domain, out var entry)
            && DateTime.UtcNow < entry.expiry)
        {
            return entry.ip; // cache hit — no UDP query needed
        }

        // Cache miss — send UDP query to DNS server
        var ip = QueryDnsServer(domain);

        // Cache the result with TTL
        _cache[domain] = (ip, DateTime.UtcNow.AddMinutes(5));

        return ip;
    }

        // Real implementation uses UDP socket

        return "93.184.216.34"; // example.com
    }
```

**Why DNS uses UDP:** A DNS query and response fit in one packet each. TCP handshake costs 3 packets before you even send the query. For billions of DNS lookups per day, this overhead is enormous. If a UDP DNS response is lost, the client simply retries — no complex state machine needed.

# Reverse Proxy — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine a big corporation's office building.

There's a **receptionist at the front desk**. Every visitor — client, delivery person, job applicant — talks to the receptionist first. The receptionist decides:

- "You're here for a meeting? I'll connect you to the Sales team on floor 3."
- "You have a delivery? I'll sign for it and bring it to the right department."
- "You're a journalist? Sorry, you can't go past this desk."
- "You look suspicious — let me verify your ID before you enter."

The receptionist is a **reverse proxy**. Nobody inside the building deals directly with the outside world. Everything flows through that one controlled point.

Now notice — the receptionist also decides **which of the 3 sales people** on floor 3 takes your meeting, based on who's least busy. That specific job of distributing visitors among multiple identical people? That's **load balancing** — one thing the receptionist can do, but not the only thing.

**A load balancer is a type of reverse proxy. A reverse proxy is not always a load balancer.**

---

## What a Reverse Proxy Actually Is

A reverse proxy sits **between clients and your backend servers**. Clients talk to the proxy. The proxy talks to your servers. Clients never directly reach your servers.

```
Without reverse proxy:
Client → directly hits Server 1's IP (192.168.1.10)
         (server IP is public, exposed to internet)
         (client knows your internal architecture)

With reverse proxy:
Client → Proxy (only public IP: 104.21.30.5)
              → Server 1 (192.168.1.10, private)
              → Server 2 (192.168.1.11, private)
              → Server 3 (192.168.1.12, private)
         (servers are completely hidden)
         (client only knows the proxy exists)
```

The word **"reverse"** distinguishes it from a **forward proxy**.

- **Forward proxy:** sits in front of clients, represents clients to the internet. (Corporate firewall, VPN, anonymous browsing)
- **Reverse proxy:** sits in front of servers, represents servers to clients. (Nginx, Cloudflare, API Gateway)

```
Forward Proxy:                    Reverse Proxy:
Client → [Proxy] → Internet       Internet → [Proxy] → Servers
         ↑                                    ↑
   hides the client                    hides the servers
```

---

## Everything a Reverse Proxy Does

This is what separates a reverse proxy from "just" a load balancer. A reverse proxy is a **Swiss Army knife** of infrastructure.

### 1. SSL/TLS Termination

Your backend servers speak plain HTTP internally. The reverse proxy handles all the HTTPS encryption/decryption.

```
Client                  Reverse Proxy              Backend Server
  │                          │                           │
  │── HTTPS (encrypted) ────→│                           │
  │                          │── HTTP (plain) ──────────→│
  │                          │←─ HTTP (plain) ───────────│
  │←─ HTTPS (encrypted) ─────│                           │
```

**Why this matters:**
- SSL handshake is computationally expensive. Do it once at the proxy, not on every server.
- Backend servers don't need SSL certificates. Simpler configuration.
- SSL certificates managed in one place — renew once, applies everywhere.
- Backend traffic stays on a private network — plain HTTP is fine there.

```nginx
# Nginx SSL termination config
server {
    listen 443 ssl;
    ssl_certificate /etc/nginx/ssl/axs.com.crt;
    ssl_certificate_key /etc/nginx/ssl/axs.com.key;

    location / {
        proxy_pass http://backend_servers;  # plain HTTP to backends
    }
}
```

---

### 2. Request Routing

Direct different requests to different backend services based on URL, headers, or any other attribute.

```
/api/events/*     → Event Service cluster
/api/bookings/*   → Booking Service cluster
/api/payments/*   → Payment Service cluster (PCI zone)
/static/*         → CDN / object storage
/admin/*          → Admin panel (internal only)
```

```nginx
location /api/events/ {
    proxy_pass http://event_service;
}

location /api/payments/ {
    proxy_pass http://payment_service;
    # Extra security headers for PCI compliance
    proxy_set_header X-Forwarded-Proto https;
}

location /static/ {
    proxy_pass http://s3_bucket;
}
```

This is **one machine** doing the work of a sophisticated API router — no application code needed.

---

### 3. Caching

The reverse proxy caches responses and serves them without touching your backend.

```
Request 1: GET /events/123
  → Proxy checks cache → MISS
  → Forwards to Event Service
  → Event Service returns response
  → Proxy caches response (TTL: 60s)
  → Returns to client

Request 2–1000: GET /events/123 (within 60 seconds)
  → Proxy checks cache → HIT
  → Returns directly from proxy cache
  → Event Service never contacted ← protected from spike
```

During a Taylor Swift flash sale — the event details page gets 100K hits/second. The reverse proxy absorbs 99% of them from cache. Your backend sees a trickle.

---

### 4. Compression

The proxy compresses responses before sending to clients. Backends don't need to implement this.

```
Backend → [10KB JSON response] → Proxy → [2KB gzipped] → Client

Content-Encoding: gzip
```

Saves bandwidth. Pages load faster. Backend developers don't think about it.

---

### 5. Security and Authentication

The proxy is your first line of defense — problems are caught before they reach application code.

```
Incoming request
       ↓
[Reverse Proxy]
  ├── Rate limiting: > 1000 req/min from this IP → 429, blocked
  ├── Bot detection: suspicious user agent → block
  ├── DDoS absorption: volumetric attack → absorb or drop
  ├── IP allowlist: /admin only from office IPs → 403 if not
  ├── JWT validation: invalid token → 401 before hitting app
  ├── SQL injection patterns: malicious payload → 400, logged
  └── Request passes → forward to backend
```

Your backend servers only receive clean, validated, authorized requests.

---

### 6. Observability — Logging and Monitoring

Every request flows through the proxy. One place to log everything.

```
192.168.1.1 [15/Apr/2025:14:32:01] "GET /events/123 HTTP/2" 200 1234ms
192.168.1.2 [15/Apr/2025:14:32:01] "POST /bookings HTTP/2" 201 340ms
10.0.0.5    [15/Apr/2025:14:32:02] "GET /admin HTTP/2" 403 12ms
```

Without a reverse proxy, you'd need to aggregate logs from every backend server separately. With it — everything is in one stream.

---

### 7. A/B Testing and Canary Deployments

Route a percentage of traffic to a new version without touching your servers.

```nginx
# Send 5% of traffic to new version, 95% to stable
split_clients "${request_id}" $upstream {
    5%    new_version_servers;
    *     stable_servers;
}
```

```
100 users hit your site:
 95 → Server fleet running v2.3 (stable)
  5 → Server fleet running v2.4 (canary)

Monitor error rates on canary.
If good → gradually increase %.
If bad  → set canary to 0%, no users affected.
```

Zero downtime. Zero risk. No code changes needed.

---

### 8. Load Balancing ← One of Many Jobs

Yes, the reverse proxy also load balances. But now you see it's just one feature among many.

```
Incoming requests
       ↓
[Reverse Proxy]
  ├── SSL termination ✓
  ├── Caching ✓
  ├── Security ✓
  ├── Routing ✓
  └── Load balancing ← distributes across backend servers
         ├── Server 1
         ├── Server 2
         └── Server 3
```

## Reverse Proxy vs Load Balancer — The Clear Distinction

This is the interview question. Here's how to think about it:

```
Load Balancer answers ONE question:
  "Which server should handle this request?"

Reverse Proxy answers MANY questions:
  "Should this request be allowed?"
  "Is this response already cached?"
  "Which service should handle this?"
  "Should I compress this response?"
  "Which server should handle this?" ← load balancing is one of these
```

| Capability                        | Load Balancer  | Reverse Proxy |

| Compression                       | ❌ No           | ✅ Yes         |

| Logging / observability           | ⚠️ Basic        | ✅ Rich        |

**One-line version for interviews:**

> "All load balancers are reverse proxies, but not all reverse proxies are load balancers. A load balancer's only job is traffic distribution. A reverse proxy is a full intermediary layer that also happens to load balance."

## Service Mesh — Reverse Proxy Taken to Microservices

This is advanced territory that impresses interviewers.

In a microservices architecture, every service needs to talk to every other service. You have the same problems at the internal level — security, retries, observability, load balancing — but now between 50 services instead of between client and server.

**Solution: sidecar proxy pattern.**

Every service gets a tiny reverse proxy (Envoy) deployed alongside it:

```
┌─────────────────┐     ┌─────────────────┐
│  Booking Service│     │ Payment Service  │
│  ┌───────────┐  │     │  ┌───────────┐  │
│  │   App     │  │     │  │   App     │  │
│  └─────┬─────┘  │     │  └─────┬─────┘  │
│  ┌─────▼─────┐  │     │  ┌─────▼─────┐  │
│  │  Envoy    │◄─┼─────┼─►│  Envoy    │  │
│  │  Sidecar  │  │     │  │  Sidecar  │  │
│  └───────────┘  │     │  └───────────┘  │
└─────────────────┘     └─────────────────┘
```

The app never talks directly to other services. It talks to its local Envoy sidecar. Envoy handles:
- Mutual TLS between services (encryption)
- Retries and circuit breaking
- Distributed tracing
- Load balancing across instances
- Traffic shaping between versions

The application code is completely unaware of all this. It thinks it's talking to localhost.

This is **Istio** — and it's built entirely on the reverse proxy concept, taken inside your cluster.

### Question 1: Implement a basic reverse proxy *(most direct)*

> "Build a simple HTTP reverse proxy that forwards requests to backend servers."

```csharp
public class SimpleReverseProxy
{
    private readonly HttpClient _httpClient = new();
    private readonly List<string> _backends;
    private int _counter = 0;

    public SimpleReverseProxy(List<string> backends)
    {
        _backends = backends;
    }

    // ASP.NET Core middleware
    public async Task HandleRequest(HttpContext context)
    {
        // Pick backend (round robin)
        int idx = Math.Abs(Interlocked.Increment(ref _counter));
        string backend = _backends[idx % _backends.Count];

        // Build forwarded request
        string targetUrl = $"{backend}{context.Request.Path}{context.Request.QueryString}";
        var forwardRequest = new HttpRequestMessage
        {
            Method = new HttpMethod(context.Request.Method),
            RequestUri = new Uri(targetUrl)
        };

        // Copy headers (preserve original client info)
        forwardRequest.Headers.TryAddWithoutValidation(
            "X-Forwarded-For",
            context.Connection.RemoteIpAddress?.ToString()
        );
        forwardRequest.Headers.TryAddWithoutValidation(
            "X-Forwarded-Host",
            context.Request.Host.ToString()
        );

        // Copy request body for POST/PUT
        if (context.Request.ContentLength > 0)
            forwardRequest.Content = new StreamContent(context.Request.Body);

        // Forward to backend
        var backendResponse = await _httpClient.SendAsync(forwardRequest);

        // Copy response back to client
        context.Response.StatusCode = (int)backendResponse.StatusCode;
        foreach (var header in backendResponse.Headers)
            context.Response.Headers[header.Key] = header.Value.ToArray();

        await backendResponse.Content.CopyToAsync(context.Response.Body);
    }
}
```

**Key things to show:**
- Preserve `X-Forwarded-For` so backend knows real client IP
- Forward all headers and body
- Copy response back including status code and headers
- Thread-safe round robin counter

### Question 2: Add caching to your reverse proxy

> "Extend your proxy to cache GET responses with a TTL."

```csharp
public class CachingReverseProxy
{
    private readonly Dictionary<string, (byte[] body, DateTime expiry, int statusCode)> _cache = new();
    private readonly TimeSpan _defaultTtl = TimeSpan.FromSeconds(60);
    private readonly HttpClient _httpClient = new();

    public async Task HandleRequest(HttpContext context)
    {
        // Only cache GET requests
        if (context.Request.Method != "GET")
        {
            await ForwardDirectly(context);
            return;
        }

        string cacheKey = context.Request.Path + context.Request.QueryString;

        // Check cache
        if (_cache.TryGetValue(cacheKey, out var cached)
            && DateTime.UtcNow < cached.expiry)
        {
            // Cache HIT — serve without hitting backend
            context.Response.StatusCode = cached.statusCode;
            context.Response.Headers["X-Cache"] = "HIT";
            await context.Response.Body.WriteAsync(cached.body);
            return;
        }

        // Cache MISS — forward to backend
        context.Response.Headers["X-Cache"] = "MISS";
        var response = await _httpClient.GetAsync(
            $"http://backend{context.Request.Path}"
        );

        var body = await response.Content.ReadAsByteArrayAsync();

        // Store in cache
        _cache[cacheKey] = (body, DateTime.UtcNow.Add(_defaultTtl), (int)response.StatusCode);

        context.Response.StatusCode = (int)response.StatusCode;
        await context.Response.Body.WriteAsync(body);
    }

    private async Task ForwardDirectly(HttpContext context) { /* forward without caching */ }
}
```

**Interviewer follow-up:** "What happens when two simultaneous requests both get a cache miss for the same key?"

**Answer:** Cache stampede — both hit the backend. Fix with a per-key lock or probabilistic early expiration.

```csharp
// Fix: per-key locking to prevent stampede
private readonly Dictionary<string, SemaphoreSlim> _keyLocks = new();

private SemaphoreSlim GetLock(string key)
{
    lock (_keyLocks)
    {
        if (!_keyLocks.ContainsKey(key))
            _keyLocks[key] = new SemaphoreSlim(1, 1);
        return _keyLocks[key];
    }
}

// Before backend call:
var keyLock = GetLock(cacheKey);
await keyLock.WaitAsync();
try
{
    // Double-check cache after acquiring lock
    if (_cache.TryGetValue(cacheKey, out cached) && DateTime.UtcNow < cached.expiry)
        return; // another thread populated it while we waited

    // Fetch from backend and populate cache...
}
finally { keyLock.Release(); }
```

### Question 3: Rate limiting middleware *(extremely common)*

> "Implement a reverse proxy that rate limits requests — max N requests per minute per IP."

```csharp
public class RateLimitingProxy
{
    // IP → sliding window of request timestamps
    private readonly Dictionary<string, Queue<DateTime>> _windows = new();
    private const int MaxRequests = 100;
    private readonly TimeSpan _window = TimeSpan.FromMinutes(1);
    private readonly object _lock = new();

    public async Task HandleRequest(HttpContext context, RequestDelegate next)
    {
        string clientIp = context.Connection.RemoteIpAddress?.ToString() ?? "unknown";

        if (!IsAllowed(clientIp))
        {
            context.Response.StatusCode = 429; // Too Many Requests
            context.Response.Headers["Retry-After"] = "60";
            await context.Response.WriteAsync("Rate limit exceeded");
            return;
        }

        await next(context); // forward to backend
    }

    private bool IsAllowed(string ip)
    {
        lock (_lock)
        {
            var now = DateTime.UtcNow;
            var cutoff = now.Subtract(_window);

            if (!_windows.ContainsKey(ip))
                _windows[ip] = new Queue<DateTime>();

            var window = _windows[ip];

            // Evict old timestamps outside window
            while (window.Count > 0 && window.Peek() < cutoff)
                window.Dequeue();

            if (window.Count >= MaxRequests)
                return false;

            window.Enqueue(now);
            return true;
        }
    }
}
```

### Question 4: URL-based request router

> "Implement a reverse proxy that routes /api/events/* to one cluster and /api/bookings/* to another."

```csharp
public class RoutingProxy
{
    private readonly Dictionary<string, string> _routes = new()
    {
        ["/api/events"] = "http://event-service:8080",
        ["/api/bookings"] = "http://booking-service:8081",
        ["/api/payments"] = "http://payment-service:8082",
        ["/static"] = "http://cdn-origin:9000"
    };

    private readonly HttpClient _client = new();

    public async Task Route(HttpContext context)
    {
        string path = context.Request.Path.Value ?? "/";

        // Find longest matching prefix
        string? matchedPrefix = _routes.Keys
            .Where(prefix => path.StartsWith(prefix))
            .OrderByDescending(prefix => prefix.Length)
            .FirstOrDefault();

        if (matchedPrefix == null)
        {
            context.Response.StatusCode = 404;
            return;
        }

        string backend = _routes[matchedPrefix];
        string targetPath = path[matchedPrefix.Length..]; // strip prefix
        string targetUrl = $"{backend}{targetPath}{context.Request.QueryString}";

        var response = await _client.GetAsync(targetUrl);
        context.Response.StatusCode = (int)response.StatusCode;
        await response.Content.CopyToAsync(context.Response.Body);
    }
}
```

**Why longest prefix matching:** `/api/events/popular` should match `/api/events` not `/api`. Same algorithm used in IP routing tables — a networking concept bleeding into application-layer proxying.

# Long Polling vs WebSockets vs SSE — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine you ordered a pizza and want to know when it's ready.

**Short Polling** is calling the restaurant every 2 minutes: "Is my pizza ready?" "No." "Is it ready now?" "No." "Now?" This is wasteful — most calls get "no."

**Long Polling** is calling once and staying on hold. The restaurant keeps you on hold until the pizza IS ready, then tells you immediately. You hang up, then call again to wait for the next update.

**Server-Sent Events** is the restaurant texting you updates automatically — "Your order is received," "Pizza is in the oven," "Pizza is ready." One-way. You never had to ask. But you can't text them back through this channel.

**WebSockets** is a walkie-talkie with the restaurant. Both sides talk whenever they want. You can say "add extra cheese" mid-cook. They can say "we're out of pepperoni, want mushrooms?" Full two-way conversation, always open.

Each solves the same core problem — **how does the server push information to the client in real time** — but with different tradeoffs.

## The Root Problem — HTTP Is Pull, Not Push

Standard HTTP is designed for **request-response**. Client asks, server answers, connection closes.

```
Standard HTTP:
Client ──── GET /events ────→ Server
Client ←─── 200 OK ──────── Server
            [connection closes]

Client ──── GET /events ────→ Server   (1 minute later)
Client ←─── 200 OK ──────── Server
            [connection closes]
```

This works for fetching a webpage. It breaks down when the server needs to **proactively notify** the client — new message arrived, seat just became available, score just changed.

The naive fix is **short polling** — client repeatedly asks "anything new?"

```
Every 2 seconds:
Client → "anything new?" → Server → "no"  → connection closes
Client → "anything new?" → Server → "no"  → connection closes
Client → "anything new?" → Server → "no"  → connection closes
Client → "anything new?" → Server → "YES" → connection closes
```

Wasteful. Most requests return nothing. Server handles thousands of empty requests per second. This is what all three techniques replace.

## Long Polling — Hold The Line

### How it works

Client makes a request. Server **holds the connection open** instead of responding immediately. When data is available, server responds. Client immediately makes another request and waits again.

```
Client                              Server
  │                                    │
  │──── GET /updates ─────────────────→│
  │                                    │  (server holds connection open)
  │                                    │  (waiting for something to happen...)
  │                                    │  (30 seconds pass...)
  │                                    │  (event occurs!)
  │←─── 200 OK {data: "seat freed"} ───│
  │                                    │
  │──── GET /updates ─────────────────→│  (immediately reconnects)
  │                                    │  (waiting again...)
```

### The mechanics

```csharp
// Server side — ASP.NET Core
[HttpGet("updates")]
public async Task<IActionResult> LongPoll(
    string userId,
    CancellationToken cancellationToken)
{
    var timeout = TimeSpan.FromSeconds(30); // max hold time
    var deadline = DateTime.UtcNow.Add(timeout);

    while (DateTime.UtcNow < deadline)
    {
        // Check if there's anything new for this user
        var update = await _updateQueue.DequeueAsync(userId);

        if (update != null)
            return Ok(update); // respond immediately with data

        // Nothing yet — wait a bit and check again
        await Task.Delay(500, cancellationToken);
    }

    // Timeout — return empty, client will reconnect
    return NoContent(); // 204 — "nothing new, try again"
}

// Client side — JavaScript
async function longPoll() {
    try {
        const response = await fetch('/updates?userId=42', {
            signal: AbortSignal.timeout(35000) // slightly > server timeout
        });

        if (response.status === 200) {
            const data = await response.json();
            handleUpdate(data); // process the update
        }
        // 204 = timeout, just reconnect
    } catch (e) {
        await sleep(1000); // brief pause on error before retry
    }

    longPoll(); // always reconnect
}

longPoll(); // start the loop
```

### What long polling is good at

```
✅ Works everywhere — plain HTTP, no special protocol
✅ Passes through all firewalls and proxies (looks like normal HTTP)
✅ Works with HTTP/1.1 — no upgrade needed
✅ Simple to implement — existing REST infrastructure works
✅ Good for low-frequency updates (every few seconds is fine)
```

### What long polling struggles with

```
❌ Latency gap — brief reconnection delay between messages
❌ Server connection cost — thousands of held connections eat memory
❌ Message ordering — if two events happen between reconnects, ordering is tricky
❌ Overhead — HTTP headers resent on every reconnect (hundreds of bytes)
❌ Server must track which updates each client has seen
```

### When to use long polling

```
Notification systems (email, Slack notifications)
  → Low frequency, occasional updates, must work everywhere

Legacy systems that can't support WebSockets
  → Old browsers, restrictive corporate firewalls

Simple status polling (job completion, payment status)
  → "Has my background job finished?" — check every few seconds

When you need HTTP semantics (auth headers, cookies, caching)
  → Each reconnect carries full HTTP headers automatically
```

## Server-Sent Events (SSE) — One-Way Stream

### How it works

Client makes **one HTTP request**. Server keeps the connection open **indefinitely** and pushes events down the wire whenever it has something to say. Connection stays open. Client never needs to reconnect for each update.

```
Client                              Server
  │                                    │
  │──── GET /stream ──────────────────→│
  │                                    │  [connection stays open]
  │←─── event: seat_freed ────────────│  (30s later)
  │←─── event: price_changed ─────────│  (45s later)
  │←─── event: show_sold_out ─────────│  (2min later)
  │                                    │  [connection stays open...]
```

### The SSE wire format

SSE uses a beautifully simple text protocol:

```
HTTP/1.1 200 OK
Content-Type: text/event-stream    ← magic header
Cache-Control: no-cache
Connection: keep-alive

data: {"seat": "A5", "status": "freed"}\n\n

event: price_update
data: {"eventId": 123, "newPrice": 199.99}\n\n

id: 42
data: {"message": "10 seats remaining"}\n\n

: this is a comment, ignored by client\n\n
```

Each event is separated by a blank line (`\n\n`). Fields:
- `data:` — the payload
- `event:` — event type (client can listen for specific types)
- `id:` — sequence ID (client sends this on reconnect so server knows what was missed)
- `retry:` — milliseconds before client retries after disconnect

### Implementation

```csharp
// Server side — ASP.NET Core SSE endpoint
[HttpGet("stream")]
public async Task Stream(string eventId, CancellationToken cancellationToken)
{
    Response.Headers["Content-Type"] = "text/event-stream";
    Response.Headers["Cache-Control"] = "no-cache";
    Response.Headers["X-Accel-Buffering"] = "no"; // disable Nginx buffering

    var lastEventId = Request.Headers["Last-Event-ID"].ToString();
    // If reconnecting, send missed events since lastEventId

    await foreach (var update in _seatUpdateService
        .GetUpdatesAsync(eventId, cancellationToken))
    {
        // Write SSE formatted event
        var sseMessage =
            $"id: {update.Id}\n" +
            $"event: seat_update\n" +
            $"data: {JsonSerializer.Serialize(update)}\n\n";

        await Response.WriteAsync(sseMessage, cancellationToken);
        await Response.Body.FlushAsync(cancellationToken); // push immediately
    }
}

// Client side — JavaScript (browser has built-in EventSource API)
const source = new EventSource('/stream?eventId=123');

// Listen to all events
source.onmessage = (event) => {
    const data = JSON.parse(event.data);
    updateSeatMap(data);
};

// Listen to specific event types
source.addEventListener('seat_update', (event) => {
    const seat = JSON.parse(event.data);
    markSeat(seat.id, seat.status);
});

source.addEventListener('price_update', (event) => {
    const price = JSON.parse(event.data);
    updatePriceDisplay(price.newPrice);
});

// Browser automatically reconnects if connection drops
source.onerror = () => console.log('Reconnecting...');
```

### SSE's automatic reconnection with Last-Event-ID

This is SSE's killer feature for reliability:

```
Client connects → receives events id=1, id=2, id=3
Connection drops (network blip)
Client automatically reconnects
Client sends header: Last-Event-ID: 3
Server sees this → replays events id=4, id=5 that were missed
Client is fully caught up

No message loss. No manual reconnection logic. Built into the protocol.
```

### What SSE is good at

```
✅ Efficient — one connection, server pushes continuously
✅ Simple — built-in browser EventSource API, no libraries needed
✅ Automatic reconnection with missed event replay (Last-Event-ID)
✅ Works with HTTP/1.1 and HTTP/2
✅ HTTP/2 multiplexing — thousands of SSE streams on one TCP connection
✅ Passes through most proxies (it's just HTTP)
✅ Named event types — client can filter what it listens to
```

### What SSE struggles with

```
❌ One-way only — server → client. Client cannot send data back via SSE.
❌ HTTP/1.1 browser limit — browsers allow only 6 connections per domain.
   With HTTP/1.1, 6 SSE tabs = all connections used up.
   HTTP/2 solves this (multiplexed), but server must support HTTP/2.
❌ Not supported in IE (but IE is dead, so...)
```

### When to use SSE

```
Live dashboards (seat availability map, metrics, scoreboards)
  → Server pushes updates, client only watches

News feeds, social media timelines
  → Posts appear in real time, one direction

Progress tracking (file upload progress, job status)
  → Server reports progress, client watches

Financial tickers (stock prices, crypto rates)
  → Prices stream in, client reads them

Notifications (email alerts, system alerts)
  → Server pushes when events happen
```

## WebSockets — Full Duplex Communication

### How it works

Client and server perform an **HTTP upgrade handshake**, then the connection transforms into a persistent, **bidirectional** channel. Either side can send messages at any time, independently, simultaneously.

```
Client                              Server
  │                                    │
  │── GET /ws (Upgrade: websocket) ───→│   HTTP upgrade handshake
  │←─ 101 Switching Protocols ─────────│
  │                                    │   [WebSocket connection open]
  │──── "join room: event-123" ───────→│   client sends
  │←─── "user count: 1,243" ───────────│   server sends
  │←─── "seat A5 freed" ───────────────│   server sends
  │──── "lock seat A5" ───────────────→│   client sends
  │←─── "A5 locked for 10min" ─────────│   server sends
  │──── "confirm booking" ────────────→│   client sends
  │←─── "booking confirmed #8821" ─────│   server sends
  │                                    │   [connection stays open]
```

### The upgrade handshake

```http
Client request:
GET /ws HTTP/1.1
Host: axs.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

Server response:
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

After this, the connection is **no longer HTTP**. It's a raw TCP connection with WebSocket framing. Both sides can send frames at any time.

### Implementation

```csharp
// Server side — ASP.NET Core WebSocket handler
app.UseWebSockets();

app.Map("/ws", async (HttpContext context) =>
{
    if (!context.WebSockets.IsWebSocketRequest)
    {
        context.Response.StatusCode = 400;
        return;
    }

    using var ws = await context.WebSockets.AcceptWebSocketAsync();
    var userId = context.Request.Query["userId"].ToString();

    // Register this connection
    _connectionManager.Add(userId, ws);

    var buffer = new byte[4096];

    try
    {
        while (ws.State == WebSocketState.Open)
        {
            // Receive message from client
            var result = await ws.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
                break;

            var message = JsonSerializer.Deserialize<ClientMessage>(
                buffer[..result.Count]
            );

            // Handle different client message types
            await HandleClientMessage(userId, message, ws);
        }
    }
    finally
    {
        _connectionManager.Remove(userId);
        await ws.CloseAsync(WebSocketCloseStatus.NormalClosure,
            "Closing", CancellationToken.None);
    }
});

private async Task HandleClientMessage(
    string userId,
    ClientMessage message,
    WebSocket ws)
{
    switch (message.Type)
    {
        case "lock_seat":
            var lockResult = await _seatService.TryLock(message.SeatId, userId);
            await SendAsync(ws, new {
                type = "seat_locked",
                seatId = message.SeatId,
                success = lockResult,
                ttlSeconds = 600
            });
            break;

        case "join_event":
            await _roomManager.Join(message.EventId, userId);
            // Broadcast to everyone in room
            await BroadcastToRoom(message.EventId, new {
                type = "user_joined",
                count = await _roomManager.GetCount(message.EventId)
            });
            break;
    }
}

// Send helper
private async Task SendAsync(WebSocket ws, object data)
{
    var json = JsonSerializer.SerializeToUtf8Bytes(data);
    await ws.SendAsync(json, WebSocketMessageType.Text, true, CancellationToken.None);
}

// Server-initiated push (e.g., from background service)
public async Task BroadcastSeatUpdate(string eventId, SeatUpdate update)
{
    var connections = _connectionManager.GetByEvent(eventId);
    var message = JsonSerializer.SerializeToUtf8Bytes(update);

    foreach (var ws in connections)
    {
        if (ws.State == WebSocketState.Open)
            await ws.SendAsync(message, WebSocketMessageType.Text, true, CancellationToken.None);
    }
}

// Client side — JavaScript
const ws = new WebSocket('wss://axs.com/ws?userId=42');

ws.onopen = () => {
    // Client can send immediately
    ws.send(JSON.stringify({ type: 'join_event', eventId: '123' }));
};

ws.onmessage = (event) => {
    const data = JSON.parse(event.data);

    switch(data.type) {
        case 'seat_update':   updateSeatMap(data); break;
        case 'seat_locked':   showLockConfirmation(data); break;
        case 'user_joined':   updateViewerCount(data.count); break;
    }
};

// Client sends whenever it wants
document.getElementById('lockBtn').onclick = () => {
    ws.send(JSON.stringify({ type: 'lock_seat', seatId: 'A5' }));
};

ws.onclose = () => reconnect(); // handle disconnection
ws.onerror = (err) => console.error(err);
```

### Scaling WebSockets — the hard problem

This is where interviews go deep. WebSockets are **stateful** — a user's connection is pinned to one server. This breaks horizontal scaling.

```
Problem:
User A connected to Server 1.
User B connected to Server 2.
User A and B are in the same event room.
Server 1 gets a seat update — how does it tell Server 2 to notify User B?

Server 1 doesn't know about User B's connection on Server 2.
```

**Solution: Pub/Sub with Redis**

```
Server 1                Redis               Server 2
  │                      │                     │
  │  seat freed in       │                     │
  │  event-123           │                     │
  │──PUBLISH event-123──→│                     │
  │  {seat: "A5"}        │──broadcast to ─────→│
  │                      │  all subscribers    │
  │                      │                     │──push to User B's
  │                      │                     │  WebSocket connection
```

```csharp
// Server 1: publishes event when seat is freed
await _redis.PublishAsync("event:123", JsonSerializer.Serialize(new {
    type = "seat_freed",
    seat = "A5"
}));

// All servers: subscribe to events for rooms they have connections in
await _redis.SubscribeAsync("event:123", async (channel, message) => {
    // Find all WebSocket connections on THIS server for event 123
    var localConnections = _connectionManager.GetByEvent("123");

    foreach (var ws in localConnections)
        await ws.SendAsync(message);
});
```

Every server only manages its own connections. Redis is the message bus connecting them. **This is the canonical WebSocket scaling pattern** — expect to explain it.

### What WebSockets are good at

```
✅ Full duplex — both sides send independently and simultaneously
✅ Low latency — no reconnection overhead, no HTTP headers per message
✅ Binary support — send raw bytes for efficiency (protobuf, MessagePack)
✅ Real-time two-way interaction — the only protocol designed for this
```

### What WebSockets struggle with

```
❌ Stateful — each user pinned to one server, hard to scale horizontally
❌ Proxies and firewalls — some corporate proxies block WebSocket upgrades
❌ No built-in reconnection — you implement this yourself
❌ No built-in message history — missed messages while disconnected are gone
❌ More complex server infrastructure — need sticky sessions or pub/sub
❌ HTTP/2 doesn't help — WebSockets don't benefit from HTTP/2 multiplexing
```

### When to use WebSockets

```
Live collaborative editing (Google Docs, Figma)
  → Both users type simultaneously, changes flow both ways

Multiplayer games
  → Player actions and game state flow bidirectionally at high frequency

Live chat (WhatsApp Web, Slack, Discord)
  → Messages go both ways, presence indicators, typing notifications

Interactive seat selection (AXS flash sale)
  → User locks a seat (client→server) + seat availability updates (server→client)
  → Both directions matter

Trading platforms
  → User places orders (client→server) + price updates (server→client)
  → Ultra-low latency required
```

## AXS Interview Angle

> *"How do you show real-time seat availability during a Taylor Swift flash sale to 500,000 concurrent users?"*

```
Seat availability map updates (server → client only):
  → SSE
  Reason: Server pushes seat status changes to all viewers.
  Client only reads — never writes seat availability back.
  SSE handles 500K connections with HTTP/2 multiplexing.
  Last-Event-ID handles reconnections without missing updates.
  Simpler than WebSockets for this one-way use case.

Seat locking interaction (bidirectional):
  → WebSockets
  Reason: User clicks seat → client sends "lock request" to server.
  Server sends "lock confirmed" or "seat taken" back.
  Both directions needed for the interactive selection flow.
  Redis pub/sub scales WebSocket servers horizontally.

Mobile app on poor connection (fallback):
  → Long Polling
  Reason: WebSockets sometimes fail on mobile networks switching
  between towers. Long polling works everywhere as fallback.
  Detect WebSocket failure → gracefully degrade to long polling.

Architecture:
  User opens seat map → SSE connection for live availability
  User clicks a seat  → WebSocket message "lock seat A5"
  Server responds     → WebSocket message "A5 locked, 10 min TTL"
  Map updates         → SSE pushes A5 as "locked" to all viewers
  User completes booking → REST POST /bookings (standard HTTP)
```

This answer shows you understand that **different interactions on the same page use different protocols** — a mark of senior-level thinking.

### Question 1: Implement a chat server with WebSockets *(most common)*

> "Build a real-time group chat where messages from one user appear instantly for all others."

```csharp
public class ChatRoom
{
    // roomId → list of connected WebSockets
    private readonly ConcurrentDictionary<string, ConcurrentBag<WebSocket>>
        _rooms = new();

    public void Join(string roomId, WebSocket ws)
    {
        _rooms.GetOrAdd(roomId, _ => new ConcurrentBag<WebSocket>()).Add(ws);
    }

    public async Task BroadcastAsync(string roomId, string senderId, string message)
    {
        if (!_rooms.TryGetValue(roomId, out var connections))
            return;

        var payload = JsonSerializer.SerializeToUtf8Bytes(new {
            senderId,
            message,
            timestamp = DateTime.UtcNow
        });

        var tasks = connections
            .Where(ws => ws.State == WebSocketState.Open)
            .Select(ws => ws.SendAsync(
                payload,
                WebSocketMessageType.Text,
                true,
                CancellationToken.None
            ));

        await Task.WhenAll(tasks); // send to all concurrently
    }

    public async Task HandleConnection(string roomId, string userId, WebSocket ws)
    {
        Join(roomId, ws);
        var buffer = new byte[4096];

        while (ws.State == WebSocketState.Open)
        {
            var result = await ws.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
                break;

            var message = Encoding.UTF8.GetString(buffer, 0, result.Count);

            // Broadcast to everyone in the room
            await BroadcastAsync(roomId, userId, message);
        }
    }
}
```

**Follow-up questions they ask:**
- "What if a user is connected to a different server?" → Redis pub/sub
- "How do you send message history to a new joiner?" → Store messages in DB, send on join
- "How do you handle 10,000 users in one room?" → Fan-out problem, batched sends

### Question 2: Implement SSE endpoint with missed message replay

> "Build an SSE endpoint that replays missed events when a client reconnects."

```csharp
public class SseController : ControllerBase
{
    private readonly IEventStore _eventStore;

    [HttpGet("stream/{eventId}")]
    public async Task Stream(string eventId, CancellationToken cancellationToken)
    {
        Response.Headers["Content-Type"] = "text/event-stream";
        Response.Headers["Cache-Control"] = "no-cache";
        Response.Headers["X-Accel-Buffering"] = "no";

        // Get last seen event ID from reconnecting client
        var lastEventId = Request.Headers["Last-Event-ID"].ToString();
        long lastSeenId = long.TryParse(lastEventId, out var id) ? id : 0;

        // Replay missed events first
        var missedEvents = await _eventStore.GetEventsSince(eventId, lastSeenId);
        foreach (var missed in missedEvents)
        {
            await WriteEvent(missed.Id, missed.Type, missed.Data);
        }

        // Then stream live events
        await foreach (var update in _eventStore
            .SubscribeAsync(eventId, cancellationToken))
        {
            await WriteEvent(update.Id, update.Type, update.Data);
        }
    }

    private async Task WriteEvent(long id, string type, object data)
    {
        var json = JsonSerializer.Serialize(data);
        var sse = $"id: {id}\nevent: {type}\ndata: {json}\n\n";

        await Response.WriteAsync(sse);
        await Response.Body.FlushAsync(); // critical — push immediately
    }
}
```

**Key insight:** `FlushAsync()` after every event is critical. Without it, the web server buffers responses and the client sees nothing until the buffer fills — defeating the purpose of SSE.

### Question 3: Implement exponential backoff reconnection *(WebSocket / Long Polling)*

> "Your WebSocket client disconnects. Implement reconnection with exponential backoff."

```csharp
public class ReconnectingWebSocketClient
{
    private readonly string _url;
    private WebSocket _ws;
    private int _retryCount = 0;
    private const int MaxRetries = 10;
    private const int BaseDelayMs = 1000; // 1 second
    private const int MaxDelayMs = 30000; // 30 seconds cap

    public async Task ConnectAsync()
    {
        while (_retryCount < MaxRetries)
        {
            try
            {
                _ws = new ClientWebSocket();
                await _ws.ConnectAsync(new Uri(_url), CancellationToken.None);

                _retryCount = 0; // reset on successful connection
                await ListenAsync();
            }
            catch (Exception ex)
            {
                _retryCount++;

                if (_retryCount >= MaxRetries)
                    throw new Exception("Max retries exceeded", ex);

                // Exponential backoff with jitter
                int delay = Math.Min(
                    BaseDelayMs * (int)Math.Pow(2, _retryCount),
                    MaxDelayMs
                );

                // Add jitter — prevent thundering herd
                // (all clients reconnecting at exact same time)
                delay += Random.Shared.Next(0, 1000);

                Console.WriteLine($"Retry {_retryCount} in {delay}ms...");
                await Task.Delay(delay);
            }
        }
    }

    private async Task ListenAsync()
    {
        var buffer = new byte[4096];
        while (_ws.State == WebSocketState.Open)
        {
            var result = await _ws.ReceiveAsync(buffer, CancellationToken.None);
            if (result.MessageType == WebSocketMessageType.Close)
                break;
            // handle message...
        }
    }
}
```

**Why jitter matters:** If 10,000 clients all disconnect at once (server restart) and all retry after exactly 2 seconds — 10,000 simultaneous reconnections hit the server. Jitter spreads them out. This is the **thundering herd problem** — expect a follow-up question on it.

### Question 4: Design a live leaderboard *(combines SSE + data structure)*

> "Design a system where scores update in real time and top 10 players are always visible."

```csharp
public class LiveLeaderboard
{
    // Sorted set: score → userId (Redis does this natively)
    private readonly SortedDictionary<int, string> _scores = new(
        Comparer<int>.Create((a, b) => b.CompareTo(a)) // descending
    );

    private readonly List<StreamWriter> _subscribers = new();
    private readonly object _lock = new();

    public void UpdateScore(string userId, int newScore)
    {
        List<(string userId, int score)> top10;

        lock (_lock)
        {
            // Update score
            _scores[newScore] = userId;

            // Get top 10
            top10 = _scores.Take(10)
                .Select(kvp => (kvp.Value, kvp.Key))
                .ToList();
        }

        // Push to all SSE subscribers
        BroadcastLeaderboard(top10);
    }

    public async Task Subscribe(HttpResponse response, CancellationToken ct)
    {
        response.Headers["Content-Type"] = "text/event-stream";
        var writer = new StreamWriter(response.Body);

        lock (_lock) { _subscribers.Add(writer); }

        // Send current state immediately on connect
        var current = _scores.Take(10)
            .Select(kvp => (kvp.Value, kvp.Key)).ToList();
        await WriteLeaderboard(writer, current);

        // Keep connection alive until client disconnects
        try { await Task.Delay(Timeout.Infinite, ct); }
        catch (TaskCanceledException) { }
        finally { lock (_lock) { _subscribers.Remove(writer); } }
    }

    private void BroadcastLeaderboard(List<(string userId, int score)> top10)
    {
        var json = JsonSerializer.Serialize(top10);
        var sse = $"event: leaderboard\ndata: {json}\n\n";

        lock (_lock)
        {
            foreach (var writer in _subscribers)
            {
                try
                {
                    writer.Write(sse);
                    writer.Flush();
                }
                catch { /* client disconnected */ }
            }
        }
    }

    private async Task WriteLeaderboard(
        StreamWriter writer,
        List<(string userId, int score)> top10)
    {
        var json = JsonSerializer.Serialize(top10);
        await writer.WriteAsync($"event: leaderboard\ndata: {json}\n\n");
        await writer.FlushAsync();
    }
}
```

## One-Line Summary for Interviews

> **Long polling repeatedly reconnects to simulate push — simple and firewall-friendly but inefficient for high-frequency updates; SSE keeps one HTTP connection open for the server to stream events to the client — perfect for dashboards and feeds where only the server pushes; WebSockets upgrade to a persistent bidirectional TCP channel — the only real choice when both client and server need to send independently and simultaneously, like seat locking, live chat, or multiplayer games — and the hardest engineering challenge with WebSockets is horizontal scaling, solved by decoupling connection state from message routing via Redis pub/sub.**
