# Monolithic vs Microservices Architecture — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine building a restaurant.

**A monolith is one big restaurant** — kitchen, dining room, bar, bakery, and storage all under one roof. One manager runs everything. When the health inspector comes, they inspect the whole place. When the kitchen is on fire, the whole restaurant closes. But on day one, it was incredibly fast to set up — just one lease, one manager, one set of equipment.

**Microservices is a food court.** Each stall is independent — pizza place, sushi bar, burger joint, dessert shop. Each has its own staff, its own kitchen, its own operating hours. The sushi bar closing for renovation doesn't affect the pizza place. Each stall can hire specialized chefs. But coordinating a food court opening requires managing 15 different vendors, health inspectors visit each stall separately, and if a customer orders from three stalls, they have to carry their own tray between them.

Neither is better. A food court is the wrong choice for your first restaurant. A single kitchen is the wrong choice when you're serving 50 different cuisines to 10,000 customers simultaneously.

## What a Monolith Actually Is

A monolith is a **single deployable unit** where all application functionality lives in one codebase, runs in one process, and is deployed together.

```
Monolithic AXS Application:
┌─────────────────────────────────────────────────────────────┐
│                    axs-app.dll / axs-app.jar                │
│                                                             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
│  │   Event      │  │   Booking    │  │    Payment       │  │
│  │   Module     │  │   Module     │  │    Module        │  │
│  └──────────────┘  └──────────────┘  └──────────────────┘  │
│                                                             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
│  │   User       │  │  Notification│  │    Analytics     │  │
│  │   Module     │  │   Module     │  │    Module        │  │
│  └──────────────┘  └──────────────┘  └──────────────────┘  │
│                                                             │
│  All modules share:                                         │
│  - One database                                             │
│  - One deployment pipeline                                  │
│  - One runtime process                                      │
│  - One technology stack                                     │
└─────────────────────────────────────────────────────────────┘
         │
         ▼
   Single PostgreSQL
   database
```

### Types of monoliths

**Single-process monolith:** One process on one machine. Simplest form.

**Modular monolith:** Single deployment unit, but internally organized into well-defined modules with clear boundaries. Each module owns its data, exposes interfaces, communicates through method calls. The sweet spot many experienced engineers recommend as a starting point.

**Distributed monolith:** Multiple processes but tightly coupled — deployed together, shared database, can't function independently. Worst of both worlds. What happens when teams add network calls without proper service boundaries.

## What Microservices Actually Are

Microservices is an architectural style where an application is built as a collection of **small, independently deployable services**, each responsible for a specific business capability, communicating over a network.

```
Microservices AXS Architecture:

                        API Gateway
                            │
         ┌──────────────────┼──────────────────┐
         │                  │                  │
         ▼                  ▼                  ▼
  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
  │   Event     │   │   Booking   │   │   Payment   │
  │   Service   │   │   Service   │   │   Service   │
  │             │   │             │   │             │
  │  Events DB  │   │ Bookings DB │   │ Payments DB │
  └─────────────┘   └─────────────┘   └─────────────┘
         │                  │                  │
         ▼                  ▼                  ▼
  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
  │    User     │   │Notification │   │  Analytics  │
  │   Service   │   │   Service   │   │   Service   │
  │             │   │             │   │             │
  │   Users DB  │   │  (stateless)│   │ Analytics DB│
  └─────────────┘   └─────────────┘   └─────────────┘

Each service:
  - Owns its own database (database per service)
  - Deploys independently
  - Scales independently
  - Can use different technology
  - Communicates via network (REST, gRPC, or message queue)
```

### The key principles

**Single responsibility:** Each service does one thing and does it well. Booking Service handles bookings. It does not send emails. It does not process payments.

**Own your data:** Services do not share databases. Event Service has its own database. Booking Service has its own. No direct SQL joins across service boundaries.

**Deploy independently:** Deploying a new version of Payment Service requires zero changes to Event Service. No coordinated deployments.

**Failure isolation:** Payment Service crashing does not crash Event Service. The blast radius of any failure is bounded.

## Monolith — The Real Strengths

Before dismissing monoliths as "old" or "wrong," understand why experienced engineers often prefer them early.

### Strength 1: Simplicity of development

```
Developer experience in monolith:
  git clone → one repo
  dotnet run → everything starts
  Set breakpoint anywhere → full stack debugging
  Change module A, see effect in module B → instant feedback
  No need to understand network protocols, service discovery,
  distributed tracing, or API contracts between your own code
```

In microservices, adding a feature that spans three services requires:
- Updating API contracts for all three
- Coordinating deployments across all three
- Debugging across multiple log streams
- Handling partial failures between services

In a monolith, it's a method call and a single deployment.

### Strength 2: Transactions are easy

```csharp
// Monolith: booking + payment + seat lock in one ACID transaction
using var transaction = db.BeginTransaction();

var booking = new Booking { UserId = 42, SeatId = "A5", EventId = 123 };
db.Bookings.Add(booking);

var payment = new Payment { BookingId = booking.Id, Amount = 250 };
db.Payments.Add(payment);

db.Seats.Find("A5").Status = "booked";

await db.SaveChangesAsync();
await transaction.CommitAsync();
// Either all three happen, or none. ACID guaranteed.
```

In microservices, "booking + payment + seat lock" spans three services and three databases. ACID is impossible. You need sagas, two-phase commit, or eventual consistency — all complex.

### Strength 3: Performance

```
Monolith: BookingModule calls PaymentModule
  → In-process method call
  → ~nanoseconds
  → No network, no serialization, no retry logic

Microservices: Booking Service calls Payment Service
  → HTTP or gRPC call
  → ~1-10 milliseconds
  → Network, serialization, possible failure, need retry/circuit breaker
```

At high call frequency, this difference adds up significantly.

### Strength 4: Operational simplicity

```
Monolith operations:
  Deploy: push one artifact
  Monitor: one log stream, one metrics dashboard
  Debug: one process to attach debugger to
  Scale: put behind load balancer, scale horizontally
  
Microservices operations:
  Deploy: 10-50 separate pipelines
  Monitor: distributed tracing, correlation IDs across services
  Debug: reconstruct request across multiple log streams
  Scale: Kubernetes, service mesh, container orchestration
  On-call: an error could be in any of 50 services
```

Netflix runs ~1,000 microservices. Their on-call rotation and tooling investment is enormous. A 5-person startup copying this architecture is misallocating engineering effort.

## Monolith — The Real Weaknesses

### Weakness 1: Scaling is all-or-nothing

```
Bottleneck: Event Service is CPU-bound during Taylor Swift sale

Monolith:
  Must scale the entire application
  Event module, Booking module, Payment module — all scale up
  Payment module has no load but gets extra resources anyway
  Wasteful and expensive

Microservices:
  Only Event Service scaled up
  Payment Service remains at minimum instances
  Cost-efficient, targeted
```

### Weakness 2: Technology lock-in

```
Monolith built in .NET 6:
  All modules must use .NET 6
  Want to use Python for ML recommendation engine?
  Can't — everything must compile and run together
  
  Upgrade to .NET 8?
  Must upgrade everything simultaneously
  High-risk coordinated effort
```

### Weakness 3: Deployment coupling

```
Team A owns Booking module — ready to deploy
Team B owns Event module — has a bug, not ready

In monolith:
  One codebase, one deployment artifact
  Team B's bug blocks Team A's deployment
  Teams are coupled through deployment pipeline
  As teams grow, deployment coordination becomes a bottleneck
```

### Weakness 4: Codebase complexity at scale

```
50 engineers working in one codebase:
  Merge conflicts constantly
  One engineer's change can break another module
  Understanding the full codebase becomes impossible
  Module boundaries erode over time (spaghetti code)
  Testing requires spinning up entire application
```

## Microservices — The Real Strengths

### Strength 1: Independent scaling

```
AXS during Taylor Swift sale:
  Event Service: 50x normal traffic → scale to 100 instances
  Booking Service: 30x normal traffic → scale to 60 instances
  Payment Service: 10x normal traffic → scale to 20 instances
  Email Service: 2x normal traffic → keep at 4 instances

Only scale what needs scaling.
Cost proportional to actual load per service.
```

### Strength 2: Independent deployment

```
25 engineering teams, each owns one service:
  Search team deploys their improvements 10 times per day
  Payment team deploys once per week (regulatory review required)
  Recommendation team deploys ML model updates hourly
  
Each team ships at their own pace.
No deployment coordination meetings.
Team B's bug doesn't block Team A's release.
```

### Strength 3: Technology heterogeneity

```
Event Service:      .NET 8 (your team's expertise)
Recommendation:     Python + FastAPI (ML models)
Real-time updates:  Node.js (event-driven, WebSocket-heavy)
Analytics pipeline: Scala + Spark (data processing)
Search:             Elasticsearch + custom Java service

Each team picks the right tool for their problem.
No global technology decision blocks innovation.
```

### Strength 4: Fault isolation

```
Payment gateway has an outage:
  Payment Service degrades
  Circuit breaker opens in Booking Service
  Booking Service returns "payment temporarily unavailable"
  Event Service, Search Service, User Service: completely unaffected
  Users can still browse events, manage accounts
  
vs. Monolith:
  Payment module throws exceptions
  Unhandled exceptions in one thread can crash the process
  Or if handled: entire booking flow degrades, but browsing also slows
  (shared thread pool, shared connection pool, shared memory)
```

## Microservices — The Real Costs

### Cost 1: Distributed systems complexity

```
Simple monolith: function call from BookingModule to PaymentModule
  → Can't fail (it's a method call)
  → No latency (nanoseconds)
  → No serialization needed
  → Debug with one breakpoint

Simple microservice: HTTP from Booking Service to Payment Service
  → Can fail (network failure, service down, timeout)
  → Has latency (1-10ms per call)
  → Must serialize/deserialize (CPU cost, bandwidth)
  → Debug requires distributed tracing (Jaeger, Zipkin)
  → Need retry logic
  → Need circuit breaker
  → Need timeout configuration
  → Need health checks
  → Need service discovery
  
Every network call = distributed systems problem
```

### Cost 2: Data consistency challenges

```
Booking Service creates booking (in Bookings DB)
Must also decrement seat count (in Events DB — different service!)

In monolith: one ACID transaction. Done.

In microservices: distributed transaction needed.
Options:
  1. Two-phase commit: complex, slow, not supported by most services
  2. Saga pattern: sequence of local transactions + compensating transactions
     - Complex to implement
     - Eventual consistency (brief inconsistency windows)
     - Rollback logic for every possible failure scenario
  3. Eventual consistency: accept that briefly, seat count is wrong
     - Need to handle overselling
     - Need reconciliation jobs
```

### Cost 3: Operational overhead

```
10 microservices require:
  Container orchestration: Kubernetes cluster
  Service discovery: Consul or Kubernetes DNS
  API Gateway: Kong, AWS API Gateway
  Distributed tracing: Jaeger or Zipkin
  Centralized logging: ELK stack or Datadog
  Circuit breakers: Polly, Hystrix, Istio
  Service mesh: Istio or Linkerd (optional but recommended)
  Multiple CI/CD pipelines: one per service
  Multiple monitoring dashboards: one per service
  
Running this infrastructure requires a dedicated platform team.
For a 5-person startup: this is all overhead, zero features shipped.
```

### Cost 4: Testing complexity

```
Monolith integration test:
  Spin up one process
  Run tests against it
  Tests run in 5 minutes

Microservices integration test:
  Spin up 10 services
  Manage 10 databases
  Mock services that aren't ready
  Test inter-service contracts (consumer-driven contract tests)
  Tests run in 30 minutes
  Flaky tests due to network timing
  Debugging test failures requires distributed tracing
```

## The Migration Path — Strangler Fig Pattern

Most real systems start as monoliths and migrate to microservices. The Strangler Fig pattern is the standard approach.

Named after the strangler fig tree that grows around a host tree, slowly replacing it while the host still functions.

```
Phase 1: Monolith running, identify first service to extract
  ┌────────────────────────────────┐
  │         Monolith               │
  │  Events | Booking | Payment    │
  └────────────────────────────────┘

Phase 2: Extract Notification Service (least coupled, most isolated)
  ┌─────────────────────────┐    ┌──────────────────┐
  │    Monolith              │    │  Notification    │
  │  Events | Booking        │───→│  Service         │
  │  Payment | (no email)    │    │                  │
  └─────────────────────────┘    └──────────────────┘

Phase 3: Extract Search Service
  ┌─────────────────┐    ┌──────────────┐    ┌──────────┐
  │    Monolith     │    │ Notification │    │  Search  │
  │  Booking        │───→│  Service     │    │  Service │
  │  Payment        │    └──────────────┘    └──────────┘
  └─────────────────┘

Phase 4: Extract Payment Service
... continue until monolith is fully decomposed
    or until the remaining monolith is a comfortable size
```

```csharp
// API Gateway / Router handles the split
// Routes some requests to microservices, rest to monolith

public class StranglerFigRouter
{
    public async Task<IActionResult> RouteAsync(HttpContext context)
    {
        string path = context.Request.Path;

        // Extracted services get routed there
        if (path.StartsWith("/api/notifications"))
            return await ForwardToService("http://notification-service", context);

        if (path.StartsWith("/api/search"))
            return await ForwardToService("http://search-service", context);

        // Everything else still goes to monolith
        return await ForwardToService("http://monolith", context);
    }
}
```

## When to Choose What — The Decision Framework

```
Choose Monolith when:
  ✅ Early-stage product (finding product-market fit)
  ✅ Team < 10 engineers
  ✅ Domain not yet well understood (boundaries not clear)
  ✅ Need to iterate quickly
  ✅ Can't afford platform team
  ✅ Single geographic market
  ✅ Low scale requirements (< 100K users)

Choose Microservices when:
  ✅ Multiple teams (> 2 teams working in the same codebase becomes painful)
  ✅ Different scaling requirements per component
  ✅ Different technology requirements per component
  ✅ High scale (millions of users, global)
  ✅ Different deployment frequencies per component
  ✅ Regulatory isolation required (PCI-DSS for payments)
  ✅ Clear, stable domain boundaries understood by team
```

The most important principle: **microservices are an optimization, not a starting point.** You can't extract the right services until you understand the domain deeply. Amazon, Netflix, and Uber all started as monoliths.

### Question 1: Implement an API Gateway *(most direct)*

> "Build a simple API gateway that routes requests to different services and handles authentication."

```csharp
public class ApiGateway
{
    private readonly HttpClient _httpClient;
    private readonly IAuthService _authService;

    // Route table: path prefix → service URL
    private readonly Dictionary<string, string> _routes = new()
    {
        ["/api/events"]        = "http://event-service:8080",
        ["/api/bookings"]      = "http://booking-service:8081",
        ["/api/payments"]      = "http://payment-service:8082",
        ["/api/users"]         = "http://user-service:8083",
        ["/api/search"]        = "http://search-service:8084"
    };

    public async Task HandleRequestAsync(HttpContext context)
    {
        // Step 1: Authentication
        string? token = context.Request.Headers["Authorization"]
            .FirstOrDefault()?.Replace("Bearer ", "");

        if (!string.IsNullOrEmpty(token))
        {
            var userId = await _authService.ValidateTokenAsync(token);
            if (userId != null)
                // Inject user ID as header for downstream services
                context.Request.Headers["X-User-Id"] = userId;
        }

        // Step 2: Rate limiting check
        string clientIp = context.Connection.RemoteIpAddress?.ToString() ?? "unknown";
        if (!await _rateLimiter.AllowAsync(clientIp))
        {
            context.Response.StatusCode = 429;
            await context.Response.WriteAsync("Rate limit exceeded");
            return;
        }

        // Step 3: Route to correct service
        string path = context.Request.Path.Value ?? "/";
        string? targetService = _routes
            .Where(r => path.StartsWith(r.Key))
            .OrderByDescending(r => r.Key.Length) // longest prefix wins
            .Select(r => r.Value)
            .FirstOrDefault();

        if (targetService == null)
        {
            context.Response.StatusCode = 404;
            return;
        }

        // Step 4: Forward request
        await ForwardRequestAsync(context, targetService);
    }

    private async Task ForwardRequestAsync(HttpContext context, string targetService)
    {
        string targetUrl = $"{targetService}{context.Request.Path}{context.Request.QueryString}";

        var forwardRequest = new HttpRequestMessage
        {
            Method = new HttpMethod(context.Request.Method),
            RequestUri = new Uri(targetUrl)
        };

        // Copy headers
        foreach (var header in context.Request.Headers)
            forwardRequest.Headers.TryAddWithoutValidation(header.Key, header.Value.ToArray());

        // Add correlation ID for distributed tracing
        string correlationId = context.Request.Headers["X-Correlation-Id"]
            .FirstOrDefault() ?? Guid.NewGuid().ToString();
        forwardRequest.Headers.TryAddWithoutValidation("X-Correlation-Id", correlationId);
        context.Response.Headers["X-Correlation-Id"] = correlationId;

        // Copy body for POST/PUT
        if (context.Request.ContentLength > 0)
        {
            context.Request.EnableBuffering();
            forwardRequest.Content = new StreamContent(context.Request.Body);
            forwardRequest.Content.Headers.ContentType =
                System.Net.Http.Headers.MediaTypeHeaderValue.Parse(
                    context.Request.ContentType ?? "application/json"
                );
        }

        try
        {
            var response = await _httpClient.SendAsync(forwardRequest);

            context.Response.StatusCode = (int)response.StatusCode;
            foreach (var header in response.Headers)
                context.Response.Headers[header.Key] = header.Value.ToArray();

            await response.Content.CopyToAsync(context.Response.Body);
        }
        catch (HttpRequestException)
        {
            context.Response.StatusCode = 502; // Bad Gateway
            await context.Response.WriteAsync("Service temporarily unavailable");
        }
    }
}
```

### Question 2: Implement the Saga pattern for distributed transactions

> "Booking requires payment + seat decrement across two services. Implement rollback if either fails."

```csharp
// Saga: sequence of local transactions with compensating transactions for rollback
public class BookingSaga
{
    private readonly IBookingService _bookingService;
    private readonly IPaymentService _paymentService;
    private readonly ISeatService _seatService;

    public async Task<SagaResult> ExecuteAsync(BookingRequest request)
    {
        string sagaId = Guid.NewGuid().ToString();
        var executedSteps = new Stack<SagaStep>();

        try
        {
            // Step 1: Lock the seat
            await _seatService.LockSeatAsync(request.SeatId, request.UserId);
            executedSteps.Push(new SagaStep(
                name: "SeatLock",
                compensate: async () =>
                    await _seatService.UnlockSeatAsync(request.SeatId, request.UserId)
            ));

            // Step 2: Create booking record (pending status)
            var booking = await _bookingService.CreatePendingAsync(request);
            executedSteps.Push(new SagaStep(
                name: "BookingCreate",
                compensate: async () =>
                    await _bookingService.CancelAsync(booking.Id, reason: "saga_rollback")
            ));

            // Step 3: Process payment
            var payment = await _paymentService.ChargeAsync(
                bookingId: booking.Id,
                userId: request.UserId,
                amount: request.Amount,
                idempotencyKey: $"saga:{sagaId}:payment"
            );
            executedSteps.Push(new SagaStep(
                name: "Payment",
                compensate: async () =>
                    await _paymentService.RefundAsync(payment.Id, reason: "saga_rollback")
            ));

            // Step 4: Confirm booking (mark as confirmed)
            await _bookingService.ConfirmAsync(booking.Id, payment.Id);
            executedSteps.Push(new SagaStep(
                name: "BookingConfirm",
                compensate: async () =>
                    await _bookingService.CancelAsync(booking.Id, reason: "saga_rollback")
            ));

            // Step 5: Decrement seat count
            await _seatService.DecrementAvailableAsync(request.EventId);
            // No compensation for this — can be reconciled asynchronously

            return SagaResult.Success(booking.Id);
        }
        catch (Exception ex)
        {
            Console.Error.WriteLine($"Saga {sagaId} failed at step: {ex.Message}");

            // Execute compensating transactions in reverse order
            var compensationErrors = new List<string>();
            while (executedSteps.Count > 0)
            {
                var step = executedSteps.Pop();
                try
                {
                    await step.Compensate();
                    Console.WriteLine($"Compensated step: {step.Name}");
                }
                catch (Exception compensationEx)
                {
                    // Compensation failed — need manual intervention
                    compensationErrors.Add(
                        $"Failed to compensate {step.Name}: {compensationEx.Message}"
                    );
                    Console.Error.WriteLine(compensationErrors.Last());
                    // In production: alert on-call, add to manual review queue
                }
            }

            return SagaResult.Failed(ex.Message, compensationErrors);
        }
    }
}

public class SagaStep
{
    public string Name { get; }
    public Func<Task> Compensate { get; }

    public SagaStep(string name, Func<Task> compensate)
    {
        Name = name;
        Compensate = compensate;
    }
}

public class SagaResult
{
    public bool Succeeded { get; private set; }
    public string? BookingId { get; private set; }
    public string? ErrorMessage { get; private set; }
    public List<string> CompensationErrors { get; private set; } = new();

    public static SagaResult Success(string bookingId) =>
        new() { Succeeded = true, BookingId = bookingId };

    public static SagaResult Failed(string error, List<string> compensationErrors) =>
        new() { Succeeded = false, ErrorMessage = error, CompensationErrors = compensationErrors };
}
```

### Question 3: Implement service discovery

> "Services need to find each other by name, not hardcoded IP. Implement a simple service registry."

```csharp
public class ServiceRegistry
{
    // serviceName → list of healthy instances
    private readonly ConcurrentDictionary<string, List<ServiceInstance>> _registry = new();
    private readonly object _lock = new();

    // Service registers itself on startup
    public void Register(string serviceName, string instanceId, string address, int port)
    {
        lock (_lock)
        {
            if (!_registry.ContainsKey(serviceName))
                _registry[serviceName] = new List<ServiceInstance>();

            _registry[serviceName].Add(new ServiceInstance
            {
                InstanceId = instanceId,
                Address = address,
                Port = port,
                LastHeartbeat = DateTime.UtcNow,
                IsHealthy = true
            });

            Console.WriteLine($"Registered: {serviceName}/{instanceId} at {address}:{port}");
        }
    }

    // Service deregisters on shutdown
    public void Deregister(string serviceName, string instanceId)
    {
        lock (_lock)
        {
            if (_registry.TryGetValue(serviceName, out var instances))
            {
                instances.RemoveAll(i => i.InstanceId == instanceId);
                Console.WriteLine($"Deregistered: {serviceName}/{instanceId}");
            }
        }
    }

    // Client discovers a healthy instance (round-robin load balancing)
    private readonly ConcurrentDictionary<string, int> _roundRobinCounters = new();

    public ServiceInstance? Discover(string serviceName)
    {
        lock (_lock)
        {
            if (!_registry.TryGetValue(serviceName, out var instances))
                return null;

            var healthy = instances.Where(i => i.IsHealthy).ToList();
            if (healthy.Count == 0) return null;

            // Round-robin selection
            int counter = _roundRobinCounters.AddOrUpdate(
                serviceName, 0, (_, c) => c + 1
            );
            return healthy[counter % healthy.Count];
        }
    }

    // Health check: mark instances as unhealthy if heartbeat missed
    public void Heartbeat(string serviceName, string instanceId)
    {
        lock (_lock)
        {
            if (_registry.TryGetValue(serviceName, out var instances))
            {
                var instance = instances.FirstOrDefault(i => i.InstanceId == instanceId);
                if (instance != null)
                    instance.LastHeartbeat = DateTime.UtcNow;
            }
        }
    }

    // Background job: mark instances unhealthy after missing heartbeats
    public void CheckHealth(TimeSpan heartbeatTimeout)
    {
        lock (_lock)
        {
            var cutoff = DateTime.UtcNow - heartbeatTimeout;
            foreach (var instances in _registry.Values)
                foreach (var instance in instances)
                    instance.IsHealthy = instance.LastHeartbeat >= cutoff;
        }
    }
```

### Question 4: Implement correlation ID for distributed tracing

> "When a request spans multiple services, how do you trace it end-to-end?"

```csharp
// Middleware: generates or propagates correlation ID
public class CorrelationIdMiddleware
{
    private readonly RequestDelegate _next;
    private const string CorrelationIdHeader = "X-Correlation-Id";

    public CorrelationIdMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext context)
    {
        // Use incoming correlation ID or generate new one
        string correlationId = context.Request.Headers[CorrelationIdHeader]
            .FirstOrDefault() ?? Guid.NewGuid().ToString();

        // Make available throughout request
        context.Items["CorrelationId"] = correlationId;
        context.Response.Headers[CorrelationIdHeader] = correlationId;

        // Add to logging scope so all logs include it
        using var scope = LogContext.PushProperty("CorrelationId", correlationId);

        await _next(context);
    }
}

// HTTP client that propagates correlation ID to downstream services
public class TracingHttpClientHandler : DelegatingHandler
{
    private readonly IHttpContextAccessor _contextAccessor;

    public TracingHttpClientHandler(IHttpContextAccessor contextAccessor)
    {
        _contextAccessor = contextAccessor;
    }

    protected override async Task<HttpResponseMessage> SendAsync(
        HttpRequestMessage request,
        CancellationToken cancellationToken)
    {
        // Propagate correlation ID to downstream service
        if (_contextAccessor.HttpContext?.Items["CorrelationId"] is string correlationId)
        {
            request.Headers.TryAddWithoutValidation(
                "X-Correlation-Id", correlationId
            );
        }

        return await base.SendAsync(request, cancellationToken);
    }
}
```

## One-Line Summary for Interviews

> **A monolith is a single deployable unit where simplicity of development, ACID transactions, and operational clarity make it the correct choice for early-stage systems and small teams — while microservices decompose the system into independently deployable services each owning their data, enabling independent scaling, independent deployment, and fault isolation at the cost of distributed systems complexity, eventual consistency challenges, and significant operational overhead requiring service discovery, distributed tracing, and container orchestration — the practical senior-level answer is to start with a modular monolith that enforces clear boundaries internally, extract services only when you have a specific scaling or team autonomy problem that justifies the complexity, and use the strangler fig pattern to migrate incrementally rather than attempting a big-bang rewrite.**

# Single Point of Failure — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine a bridge over a river. It's the only way to cross. Every car, every bus, every pedestrian uses it.

One day the bridge develops a crack. The entire city's transportation stops. Not because most of the city is broken — just because **one critical component** failed.

Now imagine the city built three bridges side by side. One develops a crack. Traffic flows over the other two. The city keeps moving.

**A Single Point of Failure (SPOF) is that first bridge.** Any component in your system whose failure causes complete system failure, regardless of how healthy everything else is.

```
System with SPOF:
  Users → Load Balancer → [App Server 1]  ✅
                        → [App Server 2]  ✅
                        → [App Server 3]  ✅
                        → Single Database ← SPOF
                          (if this dies, 
                           all three app servers 
                           are useless)

Users see 500 errors even though 3 app servers are healthy.
```

The database is healthy 99.9% of the time. But that 0.1% = 8.7 hours of downtime per year. For AXS during a Taylor Swift sale — that 0.1% might be the exact moment of peak traffic.

---

## Why SPOFs Are Inevitable — And Why You Must Plan For Them

Everything fails eventually. Disks fail. Network cards fail. Data centers lose power. Cloud providers have outages. The question is never "will this fail?" It's "when this fails, what happens to the system?"

```
Failure statistics for common components:
  Hard disk:          MTBF ~1 million hours (~114 years)
                      But you have 1,000 disks → failure every 40 days
  
  Server:             MTBF ~100,000 hours (~11 years)
                      But you have 100 servers → failure every 5 weeks
  
  Network switch:     MTBF ~200,000 hours (~23 years)
                      But data center has 50 switches → failure every 6 months
  
  Software process:   Crashes due to memory leaks, bugs, OOM → weekly
  
  Cloud provider AZ:  Major outage ~2-3 times per year
```

At scale, something is always failing somewhere. System design is about ensuring individual failures don't cascade into system failures.

---

## How to Identify SPOFs — The Systematic Approach

Draw your system. For every component, ask: **"If this component disappeared right now, does anything still work?"**

```
AXS System Audit:

DNS → Users can't resolve your domain
  SPOF? Yes, if only one DNS provider
  Fix: Use two DNS providers (Route 53 + Cloudflare)

Load Balancer → No traffic gets to app servers
  SPOF? Yes, if single load balancer
  Fix: Active-passive LB pair, or multi-AZ ALB

App Servers → Requests can't be processed
  SPOF? No — already have multiple
  Current state: ✅ redundant

Redis (session cache) → Sessions lost, users logged out
  SPOF? Yes, if single Redis node
  Fix: Redis Sentinel or Redis Cluster

Primary Database → No reads or writes
  SPOF? Yes, if single PostgreSQL node
  Fix: Primary + synchronous replica + automatic failover

Message Queue → Bookings can't be processed async
  SPOF? Partially — depends on use
  Fix: Kafka with replication factor ≥ 3

Payment Gateway → Can't complete purchases
  SPOF? Yes, if only one gateway
  Fix: Primary (Stripe) + fallback (Razorpay/PayU)

Data Center → Everything in one DC fails together
  SPOF? Yes, if single AZ deployment
  Fix: Multi-AZ deployment (active-active or active-passive)
```

## The Core Elimination Strategies

### Strategy 1: Redundancy — The Universal Fix

Add multiple instances of the same component. If one fails, others absorb the load.

```
Types of redundancy:

Active-Active:
  Both instances serve traffic simultaneously
  Load is distributed between them
  Either can fail with zero downtime
  Capacity: N instances → handles N × capacity
  
  LB → [App Server 1] ← both serving
        [App Server 2] ← both serving
  
  If Server 1 fails: Server 2 handles 100% of traffic
  Zero downtime. Full capacity utilized normally.

Active-Passive (Hot Standby):
  One instance serves traffic
  Standby monitors the active and takes over on failure
  Failover time: seconds (hot) to minutes (warm) to longer (cold)
  
  LB → [Primary DB] ← serving all traffic
        [Replica DB]  ← monitoring, ready to take over
  
  If Primary fails: Replica promoted, becomes new primary
  Downtime: ~30 seconds for automatic failover
  
  Use when: Active-active is too complex (e.g., write conflicts in DB)
```

### Strategy 2: Health Checks and Automatic Failover

Redundancy only helps if failures are detected and traffic rerouted automatically.

```csharp
public class HealthCheckMonitor
{
    private readonly Dictionary<string, ServiceHealth> _services = new();
    private readonly ILoadBalancer _loadBalancer;

    public async Task MonitorAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            foreach (var (serviceName, health) in _services)
            {
                bool isHealthy = await CheckHealthAsync(health.HealthEndpoint);

                if (isHealthy && !health.IsHealthy)
                {
                    // Service recovered
                    await _loadBalancer.AddInstanceAsync(serviceName, health.Address);
                    Console.WriteLine($"{serviceName} recovered — added back to pool");
                    health.IsHealthy = true;
                    health.ConsecutiveFailures = 0;
                }
                else if (!isHealthy)
                {
                    health.ConsecutiveFailures++;

                    if (health.ConsecutiveFailures >= 3 && health.IsHealthy)
                    {
                        // 3 consecutive failures → remove from pool
                        await _loadBalancer.RemoveInstanceAsync(serviceName, health.Address);
                        Console.WriteLine($"{serviceName} unhealthy — removed from pool");
                        health.IsHealthy = false;
                        await AlertAsync($"Instance down: {serviceName}");
                    }
                }
            }

            await Task.Delay(TimeSpan.FromSeconds(10), ct);
        }
    }

    private async Task<bool> CheckHealthAsync(string endpoint)
    {
        try
        {
            using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(3));
            var response = await _httpClient.GetAsync(endpoint, cts.Token);
            return response.IsSuccessStatusCode;
        }
        catch
        {
            return false;
        }
    }
}
```

### Strategy 3: Geographic Distribution

Even a fully redundant single data center is a SPOF for regional outages.

```
Single Region (SPOF at region level):
  All of AXS runs in AWS us-east-1
  AWS us-east-1 has major outage (happened December 2021)
  AXS entirely down

Multi-Region Active-Active:
  us-east-1: primary for US users
  eu-west-1: primary for EU users
  ap-south-1: primary for India users

  Each region handles its local traffic.
  DNS routes users to nearest region.
  If us-east-1 fails: DNS routes US users to eu-west-1
  Degraded latency but not outage.

Multi-AZ (within one region):
  us-east-1a: App servers, DB primary
  us-east-1b: App servers, DB replica
  us-east-1c: App servers, DB replica
  
  AZ failure: traffic routes to other AZs
  Most AWS outages are AZ-level, not region-level
  Cheaper and faster to implement than multi-region
```

### Strategy 4: No Single Human

SPOFs aren't always technical. A process that requires one specific person is a SPOF.

```
Manual deployment process:
  Only Alice knows how to deploy
  Alice gets sick on Taylor Swift sale day
  → Can't deploy hotfix → SPOF

Fix: Automated CI/CD pipeline
  → Any engineer can trigger deployment
  → Deployment is documented, reproducible, automated

Single on-call engineer with no runbook:
  Bob is on-call but the alert is for a system he's never seen
  → 2 hours of debugging while system is down

Fix: Runbooks for every alert, blameless on-call rotation,
     alerts that include context and standard remediation steps
```

## Every Layer Has SPOFs — The Full Stack

### Layer 1: DNS — Often Forgotten

```
SPOF scenario:
  Your entire DNS is managed by one provider (e.g., only Route 53)
  Provider has outage (it happens)
  Users can't resolve axs.com
  All your redundant servers are unreachable — nobody can get to them

Fix: DNS failover with multiple providers
  Primary: AWS Route 53
  Secondary: Cloudflare

  Route 53 goes down → Cloudflare still resolves your domain
  
  Implementation: Set TTL low (60 seconds) so failover propagates quickly
  
  Route 53 health check → if origin unhealthy, Route 53 itself switches
  to backup IP automatically:
  
  resource "aws_route53_health_check" "primary" {
    fqdn              = "axs.com"
    port              = 443
    type              = "HTTPS"
    failure_threshold = 3
    request_interval  = 30
  }
```

### Layer 2: Load Balancer

```
SPOF scenario:
  Single Nginx load balancer
  Nginx process crashes (OOM, bug, certificate expiry)
  All traffic has nowhere to go

Fix 1: Cloud managed load balancer (AWS ALB)
  AWS manages the LB infrastructure
  Multiple nodes behind the scenes automatically
  "Managed" means AWS's SPOF problem, not yours

Fix 2: Active-passive Nginx with keepalived
  Master Nginx: serves traffic, owns Virtual IP (VIP)
  Backup Nginx: monitors master via heartbeat
  
  If master dies:
    keepalived detects missed heartbeat
    Backup claims the VIP
    Traffic flows to backup
    Failover: < 1 second
  
  Master Nginx:  10.0.0.1 → owns 10.0.0.100 (VIP)
  Backup Nginx:  10.0.0.2 → monitors 10.0.0.1
  Clients connect to: 10.0.0.100 (always)
```

### Layer 3: Application Servers

```
Already covered by standard horizontal scaling.
Multiple app servers behind load balancer.
Health checks remove failing instances.

But watch for:
  Session state stored locally:
    User's cart is in memory of Server 1
    Server 1 crashes → cart lost → user logs in again, cart empty
    
    Fix: Externalize all state to Redis
    Sessions in Redis → any app server handles any request
    
  Configuration drift:
    Server 3 has different config than Servers 1 and 2
    Requests hitting Server 3 behave differently
    
    Fix: Immutable infrastructure, config from environment variables
    All servers built from same Docker image, same config source
```

### Layer 4: Database — Most Critical SPOF

```
The single most important SPOF to eliminate.
Database holds the source of truth. If it's unavailable, nothing works.

Failure modes:
  Hardware failure: disk, RAM, motherboard
  Software failure: PostgreSQL crash, OOM kill
  Network partition: DB unreachable but still running
  Corruption: data corrupted by software bug or hardware failure
  Human error: accidental DELETE or DROP TABLE

Solutions at each level:

Level 1 — Primary + Sync Replica (minimum viable)
  Primary: accepts all reads and writes
  Sync Replica: receives every write before primary acknowledges it
  
  Primary fails:
    Replica promoted (Patroni, pgBouncer, or cloud managed)
    No data loss (synchronous replication)
    Downtime: ~30 seconds automatic failover
  
  Still a regional SPOF if both are in same AZ

Level 2 — Multi-AZ with automatic failover
  Primary: us-east-1a
  Sync Replica: us-east-1b (different AZ)
  Async Replica(s): us-east-1c (for read scaling)
  
  AZ-level failure: primary fails, replica in different AZ promotes
  Survives most AWS outages

Level 3 — Multi-Region (for global availability)
  Primary: us-east-1 (US traffic writes here)
  Replica: eu-west-1 (EU traffic reads here, async)
  
  Cross-region writes: routed to primary (higher latency)
  Or: multi-master with conflict resolution (complex)
  
  Survives full region outage

Level 4 — Regular backups + point-in-time recovery
  Even with all the above: protect against corruption
  Daily full backup to S3
  Continuous WAL archival (point-in-time recovery to any second)
  Regular restore tests (tested backup is a backup; untested is hope)
```

### Layer 5: Cache (Redis)

```
Redis as SPOF:
  Many systems make Redis critical path — sessions, locks, data
  Single Redis node fails → cascading failures

  Scenario: AXS uses Redis for seat soft locks
  Redis node fails during Taylor Swift sale
  All soft locks lost → multiple users booking same seat → double bookings

Fix 1: Redis Sentinel (high availability)
  1 Primary Redis
  2 Replica Redises
  3 Sentinel processes (monitors and orchestrates failover)
  
  Primary fails:
    Sentinels detect (requires quorum — 2/3 must agree)
    Replica promoted to primary
    Clients reconnect to new primary
    Failover: 10-30 seconds

Fix 2: Redis Cluster (high availability + horizontal scale)
  Data sharded across multiple primary nodes
  Each primary has 1-2 replicas
  
  One primary fails:
    Its replica promotes
    Other shards unaffected
    Cluster continues serving
  
  Handles more data and traffic than single primary setup

Fix 3: Design for Redis unavailability
  Cache-aside pattern: if Redis unavailable, fallback to DB
  Not all caches need to be critical path
  Make Redis failure graceful, not catastrophic
  
  var cachedSeat = await redis.GetAsync(key);
  if (cachedSeat == null)
      // Redis down or cache miss — go to DB
      cachedSeat = await db.GetSeatAsync(seatId);
```

### Layer 6: Message Queues

```
Kafka as SPOF:
  Single broker → all async processing stops

Fix: Kafka replication
  Minimum 3 brokers in production
  Replication factor 3 for critical topics
  min.insync.replicas = 2 (write must reach 2 brokers)
  
  One broker fails:
    Partition leadership transfers to another broker
    No message loss (replication factor 3)
    Producers and consumers reconnect automatically
    Downtime: seconds during leader election

  Topic configuration:
  bin/kafka-topics.sh --create \
    --topic booking-events \
    --replication-factor 3 \    ← 3 copies of every message
    --partitions 12 \
    --config min.insync.replicas=2  ← must write to 2 before ack
```

### Layer 7: External Dependencies

```
Third-party services are SPOFs you don't control:

Payment gateway (Stripe):
  Stripe has outage → nobody can buy tickets
  
  Fix: Multiple payment providers
    Primary: Stripe (preferred, better UX)
    Fallback: Razorpay / PayU
    
    Circuit breaker monitors Stripe failure rate
    On circuit open: route to Razorpay automatically
    
    async Task<PaymentResult> ProcessPaymentAsync(PaymentRequest req)
    {
        try
        {
            return await _stripeCircuitBreaker.ExecuteAsync(
                () => _stripe.ChargeAsync(req)
            );
        }
        catch (CircuitBreakerOpenException)
        {
            // Stripe unavailable — try Razorpay
            return await _razorpay.ChargeAsync(req);
        }
    }

Email provider (SendGrid):
  Outage → confirmation emails not sent
  
  Fix: Queue emails to SQS
  Retry with exponential backoff
  If SendGrid down for > 1 hour: switch to AWS SES
  User doesn't need email instantly — async is fine

CDN (CloudFront):
  CDN outage → images don't load
  
  Fix: Origin fallback in HTML
  <img src="cdn.axs.com/banner.jpg"
       onerror="this.src='origin.axs.com/banner.jpg'">
```

## The Availability Math — Why This Matters

System availability = product of all component availabilities.

```
Component availabilities:
  DNS:           99.99%
  Load Balancer: 99.99%
  App Server:    99.9% (but you have 3 → 99.9999%)
  Database:      99.9%
  Redis:         99.9%
  Message Queue: 99.9%
  Stripe:        99.99%

Without redundancy (all components in series):
  System availability = 0.9999 × 0.9999 × 0.999 × 0.999 × 0.999 × 0.9999
                      = 0.9951 = 99.51%
  
  Downtime: 0.49% × 8760 hours = 43 hours per year

With redundancy (replicate each SPOF):
  Database:  99.9% → replicated → 99.9999%
  Redis:     99.9% → replicated → 99.999%
  App:       99.9% → 3 instances → 99.9999%
  
  System availability: > 99.99%
  Downtime: < 1 hour per year
```

This is why SLAs say "four nines" (99.99%) or "five nines" (99.999%). Each additional nine requires eliminating more SPOFs and adding more redundancy.

## Chaos Engineering — Proactively Finding SPOFs

The best way to know if you've eliminated SPOFs is to deliberately cause failures in production (in a controlled way).

Netflix's Chaos Monkey randomly terminates instances in production. This forces engineers to build systems that survive failures — because you know the Chaos Monkey will come.

```csharp
// Simple chaos engineering for testing SPOF elimination
public class ChaosMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ChaosSettings _settings;

    public ChaosMiddleware(RequestDelegate next, ChaosSettings settings)
    {
        _next = next;
        _settings = settings;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // Only active in non-production environments
        if (!_settings.Enabled)
        {
            await _next(context);
            return;
        }

        // Randomly inject latency
        if (Random.Shared.NextDouble() < _settings.LatencyProbability)
        {
            var delay = TimeSpan.FromMilliseconds(
                Random.Shared.Next(100, (int)_settings.MaxLatency.TotalMilliseconds)
            );
            Console.WriteLine($"[CHAOS] Injecting {delay.TotalMs}ms latency");
            await Task.Delay(delay);
        }

        // Randomly return errors
        if (Random.Shared.NextDouble() < _settings.ErrorProbability)
        {
            Console.WriteLine("[CHAOS] Injecting 503 error");
            context.Response.StatusCode = 503;
            await context.Response.WriteAsync("Chaos: Service unavailable");
            return;
        }

        // Randomly kill the connection
        if (Random.Shared.NextDouble() < _settings.AbortProbability)
        {
            Console.WriteLine("[CHAOS] Aborting connection");
            context.Abort();
            return;
        }

        await _next(context);
    }
}
```

### Question 1: Implement a primary-backup failover system

> "Build a system where a backup automatically takes over when the primary fails."

```csharp
public class PrimaryBackupStore
{
    private readonly List<IDataStore> _nodes;
    private int _primaryIndex = 0;
    private readonly object _lock = new();
    private readonly TimeSpan _healthCheckInterval;
    private readonly TimeSpan _requestTimeout;

    public PrimaryBackupStore(
        List<IDataStore> nodes,
        TimeSpan? healthCheckInterval = null,
        TimeSpan? requestTimeout = null)
    {
        _nodes = nodes;
        _healthCheckInterval = healthCheckInterval ?? TimeSpan.FromSeconds(5);
        _requestTimeout = requestTimeout ?? TimeSpan.FromSeconds(2);

        // Start health monitoring in background
        _ = Task.Run(MonitorPrimaryAsync);
    }

    // All writes go to primary
    public async Task<bool> WriteAsync(string key, string value)
    {
        var primary = GetCurrentPrimary();
        try
        {
            using var cts = new CancellationTokenSource(_requestTimeout);
            return await primary.WriteAsync(key, value, cts.Token);
        }
        catch (Exception ex)
        {
            Console.Error.WriteLine($"Primary write failed: {ex.Message}");
            await FailoverAsync();
            // Retry on new primary
            return await WriteAsync(key, value);
        }
    }

    // Reads can go to any healthy node (for read scaling)
    public async Task<string?> ReadAsync(string key, bool requirePrimary = false)
    {
        var node = requirePrimary
            ? GetCurrentPrimary()
            : GetAnyHealthyNode();

        try
        {
            using var cts = new CancellationTokenSource(_requestTimeout);
            return await node.ReadAsync(key, cts.Token);
        }
        catch
        {
            // Try primary as fallback
            return await GetCurrentPrimary().ReadAsync(key);
        }
    }

    private IDataStore GetCurrentPrimary()
    {
        lock (_lock) { return _nodes[_primaryIndex]; }
    }

    private IDataStore GetAnyHealthyNode()
    {
        lock (_lock)
        {
            // Round-robin across healthy nodes
            var healthy = _nodes.Where(n => n.IsHealthy).ToList();
            if (healthy.Count == 0) return _nodes[_primaryIndex];
            return healthy[Random.Shared.Next(healthy.Count)];
        }
    }

    private async Task FailoverAsync()
    {
        lock (_lock)
        {
            // Find next healthy node
            for (int i = 1; i < _nodes.Count; i++)
            {
                int candidateIndex = (_primaryIndex + i) % _nodes.Count;
                if (_nodes[candidateIndex].IsHealthy)
                {
                    Console.WriteLine(
                        $"Failing over from node {_primaryIndex} to node {candidateIndex}"
                    );
                    _primaryIndex = candidateIndex;
                    return;
                }
            }
            throw new NoHealthyNodeException("All nodes are unhealthy");
        }
    }

    private async Task MonitorPrimaryAsync()
    {
        while (true)
        {
            await Task.Delay(_healthCheckInterval);

            var primary = GetCurrentPrimary();
            try
            {
                using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(2));
                bool healthy = await primary.PingAsync(cts.Token);

                if (!healthy)
                {
                    Console.WriteLine("Primary health check failed — initiating failover");
                    primary.IsHealthy = false;
                    await FailoverAsync();
                }
            }
            catch
            {
                primary.IsHealthy = false;
                await FailoverAsync();
            }
        }
    }
}
```

### Question 2: Implement a multi-region active-active router

> "Route users to the nearest healthy region. Fall back to another region on failure."

```csharp
public class MultiRegionRouter
{
    private readonly Dictionary<string, RegionConfig> _regions;

    public MultiRegionRouter(Dictionary<string, RegionConfig> regions)
    {
        _regions = regions;
        // Start health monitoring for all regions
        foreach (var region in _regions.Values)
            _ = Task.Run(() => MonitorRegionAsync(region));
    }

    // Route user to nearest healthy region
    public string RouteRequest(string userRegion, string path)
    {
        // Try nearest region first
        if (_regions.TryGetValue(userRegion, out var nearest) && nearest.IsHealthy)
            return $"{nearest.BaseUrl}{path}";

        // Nearest unhealthy — find closest healthy alternative
        var healthyRegion = _regions.Values
            .Where(r => r.IsHealthy && r.Name != userRegion)
            .OrderBy(r => GetLatency(userRegion, r.Name))
            .FirstOrDefault();

        if (healthyRegion == null)
            throw new AllRegionsDownException("No healthy regions available");

        Console.WriteLine(
            $"Routing {userRegion} user to {healthyRegion.Name} " +
            $"(primary region unhealthy)"
        );

        return $"{healthyRegion.BaseUrl}{path}";
    }

    private int GetLatency(string fromRegion, string toRegion)
    {
        // Approximate latency matrix (ms)
        var latencies = new Dictionary<(string, string), int>
        {
            [("us-east", "us-west")] = 70,
            [("us-east", "eu-west")] = 90,
            [("us-east", "ap-south")] = 200,
            [("eu-west", "ap-south")] = 150,
            [("us-west", "ap-south")] = 180,
        };

        var key = fromRegion.CompareTo(toRegion) < 0
            ? (fromRegion, toRegion)
            : (toRegion, fromRegion);

        return latencies.GetValueOrDefault(key, 999);
    }

    private async Task MonitorRegionAsync(RegionConfig region)
    {
        int consecutiveFailures = 0;

        while (true)
        {
            await Task.Delay(TimeSpan.FromSeconds(10));

            try
            {
                using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5));
                var response = await _httpClient.GetAsync(
                    $"{region.BaseUrl}/health", cts.Token
                );

                if (response.IsSuccessStatusCode)
                {
                    consecutiveFailures = 0;
                    if (!region.IsHealthy)
                    {
                        region.IsHealthy = true;
                        Console.WriteLine($"Region {region.Name} recovered");
                    }
                }
                else
                {
                    consecutiveFailures++;
                }
            }
            catch
            {
                consecutiveFailures++;
            }

            if (consecutiveFailures >= 3 && region.IsHealthy)
            {
                region.IsHealthy = false;
                Console.WriteLine($"Region {region.Name} marked unhealthy");
                await AlertAsync($"Region {region.Name} is down!");
            }
        }
    }
}
```

### Question 4: Implement a distributed lock with automatic expiry (eliminate lock SPOF)

> "A distributed lock held by a crashed process would block the system forever — a SPOF. Fix it."

```csharp
public class SafeDistributedLock
{
    private readonly IDatabase _redis;
    private readonly TimeSpan _lockTtl;

    public SafeDistributedLock(IDatabase redis, TimeSpan lockTtl)
    {
        _redis = redis;
        _lockTtl = lockTtl;
    }

    public async Task<IAsyncDisposable?> AcquireAsync(
        string resourceId,
        string holderId,
        CancellationToken ct = default)
    {
        string lockKey = $"lock:{resourceId}";
        string lockValue = holderId; // who holds the lock

        // Atomic acquire: SET NX EX
        bool acquired = await _redis.SetAsync(
            lockKey,
            lockValue,
            _lockTtl,
            When.NotExists
        );

        if (!acquired) return null; // lock held by someone else

        // Start background renewal to prevent expiry while we hold it
        var renewalCts = new CancellationTokenSource();
        _ = Task.Run(async () =>
        {
            while (!renewalCts.Token.IsCancellationRequested)
            {
                await Task.Delay(_lockTtl / 2, renewalCts.Token);

                // Only renew if WE still hold the lock
                const string renewScript = @"
                    if redis.call('get', KEYS[1]) == ARGV[1] then
                        return redis.call('expire', KEYS[1], ARGV[2])
                    else
                        return 0
                    end";

                var renewed = (long)await _redis.ScriptEvaluateAsync(
                    renewScript,
                    new RedisKey[] { lockKey },
                    new RedisValue[] { lockValue, (long)_lockTtl.TotalSeconds }
                );

                if (renewed == 0)
                {
                    // Lock was taken by someone else (shouldn't happen normally)
                    Console.Error.WriteLine($"Lost lock on {resourceId}!");
                    break;
                }
            }
        }, renewalCts.Token);

        // Return disposable that releases the lock
        return new LockHandle(async () =>
        {
            await renewalCts.CancelAsync(); // stop renewal

            // Only release if we still hold it (atomic check-and-delete)
            const string releaseScript = @"
                if redis.call('get', KEYS[1]) == ARGV[1] then
                    return redis.call('del', KEYS[1])
                else
                    return 0
                end";

            await _redis.ScriptEvaluateAsync(
                releaseScript,
                new RedisKey[] { lockKey },
                new RedisValue[] { lockValue }
            );
        });
    }
}

public class LockHandle : IAsyncDisposable
{
    private readonly Func<Task> _release;
    public LockHandle(Func<Task> release) => _release = release;
    public async ValueTask DisposeAsync() => await _release();
}

// Usage
var lockService = new SafeDistributedLock(redis, TimeSpan.FromMinutes(10));

await using var lockHandle = await lockService.AcquireAsync("seat:A5:event:123", processId);
if (lockHandle == null)
{
    return BookingResult.Fail("Seat is being processed by another request");
}

// Do booking work here — lock auto-expires even if process crashes
// Auto-renewed while we hold it — won't expire during long operations
await ProcessBookingAsync();
// Lock released when lockHandle is disposed
```

## One-Line Summary for Interviews

> **A Single Point of Failure is any component whose failure causes complete system failure regardless of how healthy everything else is — identified by asking "if this component disappears right now, does anything still work?" — eliminated through redundancy (active-active for zero-downtime, active-passive for simpler consistency), automatic health checks with failover, geographic distribution across AZs and regions, and fallback paths for external dependencies — with the database being the most critical SPOF to eliminate via synchronous replication and automatic promotion, the load balancer eliminated by using cloud managed services or keepalived VIP, and the often-overlooked human SPOF eliminated by runbooks, automation, and blameless postmortems — and the practical interview answer is to enumerate every layer from DNS through storage, identify which components are single points, and explain the specific redundancy mechanism for each.**


# Stateless vs Stateful Services — Deep Dive for System Design Interviews

## Start With a Real Feeling

Imagine two different types of customer service.

**Stateless is a fast food cashier.** Every time you walk up, they don't remember you. "Hi, what can I get you?" They need all context in that one interaction. No memory of your previous orders. But here's the power: it doesn't matter which cashier you go to — any of the 5 cashiers can serve you just as well. One gets sick, you go to the next one. They hire 10 more on a busy day, no problem. Each cashier is interchangeable.

**Stateful is your family doctor.** She knows your entire medical history. She remembers that you're allergic to penicillin, that you had that knee surgery, that you've been managing your blood pressure. She gives infinitely better personalized care because of this history. But: you specifically need Dr. Sharma. If she's on vacation, the replacement doctor has no context. If the clinic adds a new doctor, they can't help you until you've built up history with them.

These aren't just metaphors — they describe exactly how services behave in distributed systems.

---

## Precise Definitions

### Stateless Service

A service where **each request contains all information needed to process it**. The service holds no memory of previous requests. Two consecutive requests from the same client are completely independent.

```
Request 1: POST /bookings
  Headers: Authorization: Bearer <JWT with userId, permissions>
  Body: { "eventId": 123, "seatId": "A5" }
  
  Service processes using ONLY what's in this request.
  No memory of any previous request.

Request 2: GET /bookings/456
  Headers: Authorization: Bearer <JWT with userId, permissions>
  
  Service processes using ONLY what's in this request.
  Has no idea Request 1 ever happened.
```

### Stateful Service

A service that **maintains information about past interactions** (state) that influences how it processes future requests.

```
Connection 1 established: TCP socket to database
  State stored: connection object, open transactions, cursor positions

Request 1: BEGIN TRANSACTION
  State stored: transaction in progress, locks held, undo log

Request 2: UPDATE seats SET status='booked' WHERE id='A5'
  Processed using stored state: inside transaction from Request 1
  
Request 3: COMMIT
  Processed using stored state: commits transaction from Request 1
  
Disconnect: connection closed, state cleaned up

If any other connection had tried to process Request 2:
  They wouldn't know about the transaction from Request 1
  Data integrity would be broken
  Must be same connection = stateful service requirement
```

---

## Where State Lives — The Core Distinction

The confusion often comes from conflating the service with where state is stored.

```
Misconception:
  "Our service stores user sessions → it's stateful"

Reality:
  Service stores user sessions in Redis → STATELESS SERVICE
  (state is external, any instance can access it)
  
  Service stores user sessions in local memory → STATEFUL SERVICE
  (state is internal, only this instance has it)

The question is: WHERE does the state live?
  In the service itself → stateful service
  In external storage (DB, cache, queue) → stateless service
  (with externalized state)
```

```
Stateless service (state externalized):
              ┌──────────────────┐
Request ─────►│  App Instance 1  │──► Redis (sessions)
              └──────────────────┘    PostgreSQL (data)
              ┌──────────────────┐
Request ─────►│  App Instance 2  │──► Redis (sessions)
              └──────────────────┘    PostgreSQL (data)
              ┌──────────────────┐
Request ─────►│  App Instance 3  │──► Redis (sessions)
              └──────────────────┘    PostgreSQL (data)

Any instance handles any request. All read from same Redis/PostgreSQL.

Stateful service (state internalized):
              ┌──────────────────────────┐
Request ─────►│  Instance 1              │
              │  In-memory: user sessions│
              │  Local: WebSocket conns  │
              └──────────────────────────┘
              ┌──────────────────────────┐
Request ─────►│  Instance 2              │
              │  In-memory: user sessions│
              │  Local: WebSocket conns  │ (different users pinned here)
              └──────────────────────────┘

Request from User A MUST go to Instance 1 (their session is there).
Request from User B MUST go to Instance 2.
```

## Stateless Services — Deep Dive

### The key enabler: JWT and token-based auth

The most common way to make auth stateless is to put all necessary context into the request itself.

```
Traditional session auth (stateful):
  Login → Server stores session in memory → Gives client session ID cookie
  
  Next request: client sends session ID
  Server must look up session ID in memory
  → This memory is local to ONE server instance
  → Next request must go to SAME server instance
  → Sticky sessions required → not stateless

JWT auth (stateless):
  Login → Server creates signed JWT containing userId, roles, expiry
  JWT stored on client (localStorage, cookie)
  
  Next request: client sends JWT
  Server validates signature (cryptographic check, no lookup needed)
  Server extracts userId, roles from JWT
  → No server-side session storage needed
  → Any server instance can validate any JWT
  → Fully stateless
```

```csharp
// Stateless JWT validation — no database lookup, no shared memory
public class JwtAuthMiddleware
{
    private readonly string _secretKey;

    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        string? token = context.Request.Headers["Authorization"]
            .FirstOrDefault()
            ?.Replace("Bearer ", "");

        if (token != null)
        {
            // Validate and decode — entirely from the token itself
            var principal = ValidateToken(token);
            if (principal != null)
            {
                context.User = principal;
                // No lookup needed — userId, roles all in the token
            }
        }

        await next(context);
    }

    private ClaimsPrincipal? ValidateToken(string token)
    {
        try
        {
            var handler = new JwtSecurityTokenHandler();
            var parameters = new TokenValidationParameters
            {
                ValidateIssuerSigningKey = true,
                IssuerSigningKey = new SymmetricSecurityKey(
                    Encoding.UTF8.GetBytes(_secretKey)
                ),
                ValidateLifetime = true,
                ValidateIssuer = false,
                ValidateAudience = false
            };

            return handler.ValidateToken(token, parameters, out _);
        }
        catch
        {
            return null; // invalid token
        }
    }
}
```

### Why statelessness enables everything you want

**Horizontal scaling — trivial:**
```
Low traffic:
  1 app instance
  
High traffic (Taylor Swift sale):
  100 app instances
  All identical, all stateless
  Load balancer distributes evenly
  No sticky routing needed
  Auto-scaling group adds/removes instances in seconds
  
This is impossible with stateful instances unless you
synchronize state across all 100 instances in real time.
```

**Zero-downtime deployments:**
```
Rolling deployment (stateless):
  Take Instance 1 out of load balancer
  Deploy new code to Instance 1
  Put Instance 1 back in load balancer
  Repeat for Instance 2, 3, 4...
  
  In-flight requests? Handled by Instance 2, 3, 4 during deployment.
  No session loss. Zero downtime.

Rolling deployment (stateful with local sessions):
  Take Instance 1 out of load balancer
  20 users had sessions on Instance 1 → their sessions are gone
  They're logged out → bad user experience
  
  You need session migration or sticky sessions with draining.
  Much more complex.
```

**Fault tolerance:**
```
Instance 1 crashes (stateless):
  Load balancer routes to Instance 2, 3, 4
  Users see brief 1-2 second blip if in-flight request fails
  Next request goes to another instance seamlessly
  User experience: barely noticeable

Instance 1 crashes (stateful, local sessions):
  20 users had sessions here → all logged out
  In-progress operations lost
  Data potentially corrupted if crashed mid-operation
  User experience: disruptive
```

### What must be externalized for statelessness

```
User sessions:
  Local memory → stateful ❌
  Redis with session ID in cookie → stateless ✅

Shopping cart / booking in progress:
  Local memory → stateful ❌
  Redis with cart data, referenced by session token → stateless ✅

File uploads in progress:
  Local temp directory → stateful ❌
  S3 multipart upload → stateless ✅

Background job state:
  Local queue in memory → stateful ❌
  Message queue (SQS, Kafka) → stateless ✅

Rate limiting counters:
  Local counter → stateful (doesn't work across instances) ❌
  Redis counter → stateless ✅
```

## Stateful Services — Deep Dive

### Why some things MUST be stateful

**WebSocket connections:**
```
User opens WebSocket for live seat updates:
  TCP connection established to Instance 1
  Instance 1 holds this connection object
  Instance 1 receives seat availability updates from Kafka
  Instance 1 pushes updates to user via this connection

Next message from same user:
  MUST go to Instance 1 (connection is there)
  If routed to Instance 2 → connection not found → error

This is inherently stateful.
The connection object CANNOT be externalized.
TCP connections are bound to a specific process.
```

**Database connections:**
```
PostgreSQL connection pool:
  Each app instance maintains N connections to DB
  These connections are stateful resources
    - Open TCP connection to PostgreSQL
    - Authentication state
    - Current transaction (if any)
    - Prepared statement cache
  
  You can't "externalize" a TCP connection to another process.
  Connection pools are inherently stateful.
  
This is why you can't have unlimited app instances without
connection pooling proxy (PgBouncer) — each instance needs
its own connections, and DB has connection limits.
```

**Distributed locks:**
```
Seat lock during checkout:
  User has lock on seat A5
  Lock stored in Redis with their userId and 10-minute TTL
  
  If we externalize the lock management to Redis → stateless service ✅
  But the lock itself is a stateful resource tracked in Redis
  
The service is stateless, but it's managing a stateful resource
(the lock) in external storage.
```

**Gaming and real-time systems:**
```
Multiplayer game session:
  Game state: positions, scores, health of all 10 players
  Updated 60 times per second
  All 10 players must see consistent state
  
  State is in memory of the game server process
  All 10 player connections go to same game server
  Cannot be distributed across instances
  Inherently stateful
  
  Scale by: shard players into separate game rooms
  Each room is on one server, multiple servers exist
  = stateful sharding, not stateless scaling
```

### Handling stateful service failures

The hard part about stateful services: when they fail, state is lost.

```csharp
// Stateful WebSocket service with reconnection handling
public class StatefulWebSocketService
{
    // In-memory state — inherently local to this instance
    private readonly ConcurrentDictionary<string, WebSocket> _connections = new();
    private readonly ConcurrentDictionary<string, string> _userEventRooms = new();

    public async Task HandleConnectionAsync(
        string userId,
        string eventId,
        WebSocket ws)
    {
        _connections[userId] = ws;
        _userEventRooms[userId] = eventId;

        // Reconnection: send any updates missed while disconnected
        var lastSeen = await GetLastSeenTimestampAsync(userId);
        if (lastSeen.HasValue)
        {
            var missed = await GetSeatUpdatesSinceAsync(eventId, lastSeen.Value);
            foreach (var update in missed)
                await ws.SendAsync(Serialize(update), WebSocketMessageType.Text, true, default);
        }

        // Store connection establishment time for future reconnections
        await StoreConnectionTimeAsync(userId, DateTime.UtcNow);

        await ListenAsync(userId, ws);
    }

    private async Task ListenAsync(string userId, WebSocket ws)
    {
        var buffer = new byte[4096];
        try
        {
            while (ws.State == WebSocketState.Open)
            {
                var result = await ws.ReceiveAsync(buffer, CancellationToken.None);
                if (result.MessageType == WebSocketMessageType.Close) break;
                await HandleMessageAsync(userId, buffer[..result.Count]);
            }
        }
        finally
        {
            _connections.TryRemove(userId, out _);
            _userEventRooms.TryRemove(userId, out _);
        }
    }

    // Scale across instances using Redis Pub/Sub for cross-instance messaging
    public async Task BroadcastToEventAsync(string eventId, SeatUpdate update)
    {
        // Publish to Redis channel — ALL instances subscribed to this event receive it
        await _redis.PublishAsync($"event:{eventId}", Serialize(update));
    }

    // Each instance subscribes to events where it has connections
    private async Task SubscribeToEventAsync(string eventId)
    {
        await _redis.SubscribeAsync($"event:{eventId}", async (channel, message) =>
        {
            // Find users in this event on THIS instance
            var usersInEvent = _userEventRooms
                .Where(kvp => kvp.Value == eventId)
                .Select(kvp => kvp.Key);

            var update = Deserialize<SeatUpdate>(message);

            foreach (var userId in usersInEvent)
            {
                if (_connections.TryGetValue(userId, out var ws) &&
                    ws.State == WebSocketState.Open)
                    await ws.SendAsync(Serialize(update),
                        WebSocketMessageType.Text, true, default);
            }
        });
    }
}
```

## The Spectrum — Most Systems Have Both

Real systems aren't purely one or the other. They mix based on the nature of each component.

```
AXS Architecture Layer by Layer:

STATELESS (externalize state):
  ┌────────────────────────────────────────────────┐
  │  REST API Servers (Booking, Events, Users)     │
  │  • Auth via JWT (no server-side session)       │
  │  • All data in PostgreSQL                      │
  │  • All cache in Redis                          │
  │  • Scale horizontally, no sticky sessions      │
  └────────────────────────────────────────────────┘

STATEFUL (local state):
  ┌────────────────────────────────────────────────┐
  │  WebSocket Server (real-time seat updates)     │
  │  • TCP connections held locally                │
  │  • Redis Pub/Sub for cross-instance messaging  │
  │  • Sticky routing via load balancer            │
  └────────────────────────────────────────────────┘

STATEFUL + SHARDED (data tier):
  ┌────────────────────────────────────────────────┐
  │  PostgreSQL (source of truth)                  │
  │  • Connections are stateful                    │
  │  • Transactions are stateful                   │
  │  • Scaled by replication, not horizontal clone │
  └────────────────────────────────────────────────┘
  ┌────────────────────────────────────────────────┐
  │  Redis Cluster (cache + locks + sessions)      │
  │  • Each key pinned to a shard                  │
  │  • Stateful per shard                          │
  │  • Scaled by consistent hashing across shards  │
  └────────────────────────────────────────────────┘
  ┌────────────────────────────────────────────────┐
  │  Kafka (event stream)                          │
  │  • Partition assignment is stateful            │
  │  • Consumer group offsets are stateful         │
  │  • Scaled by adding partitions                 │
  └────────────────────────────────────────────────┘
```

## Stateless vs Stateful Comparison

| Dimension | Stateless | Stateful |
|---|---|---|
| Horizontal scaling | Trivial (add instances) | Complex (shard or sync state) |
| Fault tolerance | High (any instance replaces another) | Low (failure loses local state) |
| Load balancing | Any algorithm (round robin, etc.) | Sticky sessions required |
| Deployment | Rolling with zero downtime | Requires session drain or migration |
| Memory usage | Low per instance | Higher (holds state) |
| Latency | Slightly higher (external state lookup) | Lower (state in memory) |
| Consistency | Easier (all see same external state) | Hard (sync across instances) |
| Examples | REST APIs, batch processors | WebSockets, DBs, game servers |

## Session Management — The Classic Stateless vs Stateful Problem

This deserves its own section because it's the most commonly asked about tradeoff.

### Option 1: Server-side session (stateful)

```csharp
// Stateful: session stored in server memory
public class StatefulSessionMiddleware
{
    // In-memory dictionary — only this instance has it
    private static readonly Dictionary<string, UserSession> _sessions = new();

    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        string? sessionId = context.Request.Cookies["session_id"];

        if (sessionId != null && _sessions.TryGetValue(sessionId, out var session))
        {
            context.Items["UserId"] = session.UserId;
            context.Items["Cart"] = session.Cart;
            // Problem: only works if request hits THIS instance
        }

        await next(context);
    }
}
```

Requires sticky sessions. Instance crashes → all sessions lost.

### Option 2: JWT (stateless)

```csharp
// Stateless: all context in the token
public class StatelessJwtMiddleware
{
    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        var token = ExtractToken(context);
        if (token != null)
        {
            // Cryptographic verification — no lookup
            var claims = ValidateAndDecode(token);
            context.Items["UserId"] = claims.UserId;
            context.Items["Roles"] = claims.Roles;
            // Works on ANY instance — no shared state needed
        }
        await next(context);
    }
}
```

Fully stateless. But: can't revoke tokens before expiry (JWT is self-contained).

### Option 3: External session store (stateless service, external state)

```csharp
// Stateless service, but state lives in Redis
public class ExternalizedSessionMiddleware
{
    private readonly IDatabase _redis;

    public async Task InvokeAsync(HttpContext context, RequestDelegate next)
    {
        string? sessionId = context.Request.Cookies["session_id"];

        if (sessionId != null)
        {
            // Lookup in Redis — works from any instance
            var sessionData = await _redis.StringGetAsync($"session:{sessionId}");
            if (sessionData.HasValue)
            {
                var session = JsonSerializer.Deserialize<UserSession>(sessionData!);
                context.Items["Session"] = session;

                // Slide TTL (keep session alive on activity)
                await _redis.KeyExpireAsync(
                    $"session:{sessionId}",
                    TimeSpan.FromHours(24)
                );
            }
        }

        await next(context);

        // Persist any session changes back to Redis
        if (context.Items["Session"] is UserSession updated)
        {
            await _redis.StringSetAsync(
                $"session:{sessionId}",
                JsonSerializer.Serialize(updated),
                TimeSpan.FromHours(24)
            );
        }
    }
}
```

Service is stateless (any instance handles any request). State is external (Redis). Best of both worlds — except Redis is now a potential SPOF.

> *"Walk me through the stateful vs stateless decisions in the AXS booking platform."*

```
Stateless decisions:

Booking API, Event API, User API:
  → Stateless REST services
  → JWT authentication (no server-side sessions)
  → All data in PostgreSQL (externalized)
  → All caching in Redis (externalized)
  → Can scale from 3 to 300 instances in 2 minutes during flash sale
  → Rolling deploy with zero downtime
  → Any instance handles any request

Seat reservation workflow:
  → Stateless service, but manages stateful resources in Redis
  → Soft lock stored in Redis: "seat:A5:event:123" → userId, TTL 10min
  → Any instance can create, check, or release the lock
  → Lock survives instance crashes (it's in Redis)

Stateful decisions:

WebSocket connections (live seat map):
  → Inherently stateful: TCP connection held by specific instance
  → Load balancer uses sticky sessions (hash of userId → consistent instance)
  → Redis Pub/Sub for cross-instance broadcasting
  → Instance crashes: client reconnects, WebSocket re-establishes
  → Client handles reconnection with exponential backoff

Database connection pool:
  → Each app instance maintains 20 connections to PostgreSQL
  → Connections are stateful (tied to process)
  → Managed via Npgsql connection pool
  → PgBouncer proxy manages total connections to DB (limits SPOF risk)

Explicit tradeoffs made:
  1. JWT chosen over server sessions: allows horizontal scaling
     Cost: can't revoke individual sessions instantly
     Mitigation: short JWT TTL (15 min) + refresh token pattern

  2. Seat locks in Redis chosen over in-memory:
     Allows any booking service instance to check any lock
     Cost: Redis becomes critical dependency
     Mitigation: Redis Sentinel for HA, graceful degradation to DB locks

  3. WebSocket server kept stateful (not moved to external):
     TCP connections cannot be externalized
     Mitigation: Redis Pub/Sub, client reconnection, sticky LB routing
```

### Question 1: Implement session externalization — convert stateful to stateless

```csharp
public interface ISessionStore
{
    Task<T?> GetAsync<T>(string sessionId, string key);
    Task SetAsync<T>(string sessionId, string key, T value, TimeSpan? ttl = null);
    Task RemoveAsync(string sessionId, string key);
    Task DestroyAsync(string sessionId);
}

public class RedisSessionStore : ISessionStore
{
    private readonly IDatabase _redis;
    private readonly TimeSpan _defaultTtl = TimeSpan.FromHours(24);

    public RedisSessionStore(IDatabase redis) => _redis = redis;

    private string FieldKey(string sessionId, string key) => $"session:{sessionId}:{key}";

    public async Task<T?> GetAsync<T>(string sessionId, string key)
    {
        var value = await _redis.StringGetAsync(FieldKey(sessionId, key));
        if (!value.HasValue) return default;
        return JsonSerializer.Deserialize<T>(value!);
    }

    public async Task SetAsync<T>(string sessionId, string key, T value, TimeSpan? ttl = null)
    {
        var json = JsonSerializer.Serialize(value);
        await _redis.StringSetAsync(
            FieldKey(sessionId, key),
            json,
            ttl ?? _defaultTtl
        );
    }

    public async Task RemoveAsync(string sessionId, string key)
        => await _redis.KeyDeleteAsync(FieldKey(sessionId, key));

    public async Task DestroyAsync(string sessionId)
    {
        // Delete all keys for this session
        var pattern = $"session:{sessionId}:*";
        var keys = _redis.Multiplexer.GetServer("localhost:6379")
            .Keys(pattern: pattern);
        foreach (var key in keys)
            await _redis.KeyDeleteAsync(key);
    }
}
```

### Question 2: Implement sticky session routing

> "Some requests must go to the same server instance. Build routing that pins users to instances."

```csharp
public class StickySessionLoadBalancer
{
    private readonly List<ServerInstance> _instances;
    private readonly IDatabase _redis;
    private readonly TimeSpan _stickyTtl;

    public StickySessionLoadBalancer(
        List<ServerInstance> instances,
        IDatabase redis,
        TimeSpan? stickyTtl = null)
    {
        _instances = instances;
        _redis = redis;
        _stickyTtl = stickyTtl ?? TimeSpan.FromHours(1);
    }

    public async Task<ServerInstance> GetInstanceAsync(string userId)
    {
        string stickyKey = $"sticky:{userId}";

        // Check if user is pinned to a specific instance
        var pinnedInstanceId = await _redis.StringGetAsync(stickyKey);
        if (pinnedInstanceId.HasValue)
        {
            var pinned = _instances.FirstOrDefault(
                i => i.Id == pinnedInstanceId.ToString() && i.IsHealthy
            );

            if (pinned != null)
            {
                // Refresh TTL on access
                await _redis.KeyExpireAsync(stickyKey, _stickyTtl);
                return pinned;
            }

            // Pinned instance is gone (crashed/removed) — re-assign
            await _redis.KeyDeleteAsync(stickyKey);
        }

        // New user or pinned instance unavailable — assign to least-loaded healthy instance
        var target = _instances
            .Where(i => i.IsHealthy)
            .OrderBy(i => i.ConnectionCount)
            .FirstOrDefault()
            ?? throw new NoHealthyInstanceException();

        // Pin this user to this instance
        await _redis.StringSetAsync(stickyKey, target.Id, _stickyTtl);

        return target;
    }

    public async Task HandleInstanceFailureAsync(string instanceId)
    {
        var instance = _instances.FirstOrDefault(i => i.Id == instanceId);
        if (instance != null) instance.IsHealthy = false;

        // Find all users pinned to failed instance
        // In production: scan Redis for keys matching "sticky:*" with value = instanceId
        // Then re-assign them to healthy instances
        // This is the reconnection wave — handle gracefully
        Console.WriteLine($"Instance {instanceId} failed. Users will be re-pinned on next request.");
    }
}
```

```csharp
// Fixed: distributed rate limiter using Redis
public class DistributedRateLimiter
{
    private readonly IDatabase _redis;
    private readonly int _maxRequests;
    private readonly TimeSpan _window;

    public DistributedRateLimiter(
        IDatabase redis,
        int maxRequests = 100,
        TimeSpan? window = null)
    {
        _redis = redis;
        _maxRequests = maxRequests;
        _window = window ?? TimeSpan.FromMinutes(1);
    }

    // Sliding window using Redis sorted set
    public async Task<RateLimitResult> CheckAsync(string userId)
    {
        string key = $"ratelimit:{userId}";
        var now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        var windowStart = now - (long)_window.TotalMilliseconds;

        // Atomic Lua script: remove old entries + count + conditionally add
        const string luaScript = @"
            -- Remove entries outside window
            redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', ARGV[1])
            
            -- Count current requests in window
            local count = redis.call('ZCARD', KEYS[1])
            
            if count < tonumber(ARGV[3]) then
                -- Under limit: add this request
                redis.call('ZADD', KEYS[1], ARGV[2], ARGV[2])
                redis.call('EXPIRE', KEYS[1], ARGV[4])
                return {1, count + 1}  -- allowed, new count
            else
                return {0, count}  -- rejected, current count
            end";

        var result = (RedisResult[])await _redis.ScriptEvaluateAsync(
            luaScript,
            new RedisKey[] { key },
            new RedisValue[]
            {
                windowStart,                          // ARGV[1]: window start
                now,                                  // ARGV[2]: current timestamp
                _maxRequests,                         // ARGV[3]: limit
                (long)_window.TotalSeconds + 1        // ARGV[4]: key TTL
            }
        );

        bool allowed = (long)result[0] == 1;
        long currentCount = (long)result[1];
        long remaining = Math.Max(0, _maxRequests - currentCount);

        return new RateLimitResult
        {
            Allowed = allowed,
            CurrentCount = (int)currentCount,
            Remaining = (int)remaining,
            ResetAt = DateTimeOffset.UtcNow.Add(_window)
        };
    }
}

public record RateLimitResult
{
    public bool Allowed { get; init; }
    public int CurrentCount { get; init; }
    public int Remaining { get; init; }
    public DateTimeOffset ResetAt { get; init; }
}

// Middleware usage
public class RateLimitMiddleware
{
    private readonly RequestDelegate _next;
    private readonly DistributedRateLimiter _limiter;

    public async Task InvokeAsync(HttpContext context)
    {
        string userId = context.User.FindFirstValue(ClaimTypes.NameIdentifier)
            ?? context.Connection.RemoteIpAddress?.ToString()
            ?? "anonymous";

        var result = await _limiter.CheckAsync(userId);

        // Add standard rate limit headers
        context.Response.Headers["X-RateLimit-Limit"] = _limiter.MaxRequests.ToString();
        context.Response.Headers["X-RateLimit-Remaining"] = result.Remaining.ToString();
        context.Response.Headers["X-RateLimit-Reset"] = result.ResetAt.ToUnixTimeSeconds().ToString();

        if (!result.Allowed)
        {
            context.Response.StatusCode = 429;
            context.Response.Headers["Retry-After"] = "60";
            await context.Response.WriteAsJsonAsync(new
            {
                error = "Rate limit exceeded",
                retryAfter = 60
            });
            return;
        }

        await _next(context);
    }
}
```

### Question 4: Convert a stateful counter to stateless with CRDT

> "Multiple service instances need to count events. Each instance has its own count. Merge them without coordination."

```csharp
// G-Counter CRDT: each instance increments its own counter
// Merge: take max of each instance's count
// Total: sum of all maximums
// No coordination needed — mathematically conflict-free

public class StatelessDistributedCounter
{
    // This instance's ID — unique per service instance
    private readonly string _instanceId;
    private long _localCount = 0;

    // Shared store for reading other instances' counts
    private readonly IDatabase _redis;

    public StatelessDistributedCounter(string instanceId, IDatabase redis)
    {
        _instanceId = instanceId;
        _redis = redis;
    }

    // Increment this instance's local count — no coordination
    public async Task IncrementAsync(string counterName, long amount = 1)
    {
        Interlocked.Add(ref _localCount, amount);

        // Periodically flush to Redis (every 100 increments or 5 seconds)
        if (_localCount % 100 == 0)
            await FlushToRedisAsync(counterName);
    }

    // Flush local count to Redis
    public async Task FlushToRedisAsync(string counterName)
    {
        await _redis.StringSetAsync(
            $"counter:{counterName}:{_instanceId}",
            _localCount,
            TimeSpan.FromMinutes(10) // TTL in case instance dies
        );
    }

    // Read total across all instances — eventual consistency
    public async Task<long> GetTotalAsync(string counterName)
    {
        // Find all instance counts for this counter
        // In production: use Redis SCAN or maintain instance registry
        var instanceKeys = await GetInstanceKeysAsync(counterName);

        long total = 0;
        foreach (var key in instanceKeys)
        {
            var value = await _redis.StringGetAsync(key);
            if (value.HasValue)
                total += (long)value;
        }

        return total;
    }

    // Merge counts from all instances
    // Each instance independently arrives at same total — no coordination
    public async Task<long> MergeAsync(string counterName)
    {
        await FlushToRedisAsync(counterName);
        return await GetTotalAsync(counterName);
    }

    private async Task<List<string>> GetInstanceKeysAsync(string counterName)
    {
        var server = _redis.Multiplexer.GetServer(
            _redis.Multiplexer.GetEndPoints().First()
        );
        var keys = server.Keys(pattern: $"counter:{counterName}:*");
        return keys.Select(k => k.ToString()).ToList();
    }
}

// Example: view counter per event
// Instance 1 gets 1,000 views → local count: 1,000
// Instance 2 gets 800 views → local count: 800
// Instance 3 gets 1,200 views → local count: 1,200

// Total: 1,000 + 800 + 1,200 = 3,000 views
// No instance needed to coordinate with others
// Each increments independently — fully stateless behavior
```

## One-Line Summary for Interviews

> **Stateless services hold no local state between requests — all context comes from the request itself (JWT, request body) and all persistent state lives in external stores (Redis, PostgreSQL) — enabling horizontal scaling by making every instance identical and interchangeable, zero-downtime rolling deploys, and trivial fault tolerance where any instance can replace any other; stateful services hold state locally (WebSocket connections, database transaction context, in-memory game state) which bounds them to specific instances, requiring sticky routing, careful failover handling, and state migration strategies — the practical design principle is to make your application tier stateless by externalizing all state while accepting that your data tier (databases, caches, queues) is inherently stateful and must be scaled through replication and sharding rather than horizontal cloning.**

# Rate Limiting — Deep Dive for System Design Interviews

## Start With a Real Feeling

You run a popular coffee shop. Most customers are regulars who order politely. But occasionally someone walks in and orders 500 coffees in one minute — either trying to resell them, stress-testing your kitchen, or just being disruptive.

Without a policy, your kitchen collapses. Every real customer waits forever or gets nothing.

So you post a sign: **"Maximum 10 orders per customer per hour."**

Now the disruptive customer gets cut off. Your kitchen handles real customers smoothly. The sign is your rate limit.

In software: users, clients, or attackers can send thousands of requests per second to your API. Without rate limiting, your servers collapse under load. Legitimate users get degraded service because bad actors (or buggy clients) consume all capacity.

Rate limiting says: **"You get N requests per time window. Beyond that, you wait."**

---

## Why Rate Limiting Exists — The Four Motivations

**1. Protection from abuse**
```
Without rate limiting:
  Attacker writes script: while(true) POST /bookings with stolen cards
  1,000 requests/second → fraud detection overwhelmed → real bookings blocked

With rate limiting:
  After 100 requests/minute from IP 1.2.3.4 → return 429
  Attacker slows to 100/minute → fraud detection catches them easily
```

**2. Fair resource distribution**
```
Without rate limiting:
  Heavy user consumes 90% of API capacity
  Other 10,000 users share the remaining 10%
  Heavy user experience: great
  Everyone else: terrible

With rate limiting:
  Every user gets fair share of capacity
  Heavy user rate limited → capacity freed for others
```

**3. Cost control**
```
Your API calls a paid service (OpenAI, Google Maps, Stripe)
Each call costs money
Bug in client code causes infinite retry loop
Without limit: $50,000 bill in one night

With rate limiting:
  Per-user limit → runaway client capped
  Per-API-key limit → vendor cost predictable
```

**4. Tier enforcement**
```
Free tier:    100 requests/day
Pro tier:     10,000 requests/day
Enterprise:   1,000,000 requests/day

Rate limiting enforces the business model.
Users exceeding free tier get 429 → prompted to upgrade.
```

## The Algorithms — Six Ways to Count Requests

### Algorithm 1: Fixed Window Counter

Divide time into fixed windows. Count requests per window. Reset at window boundary.

```
Window: 60 seconds (reset at :00, 1:00, 2:00...)
Limit: 100 requests per window

Timeline:
  12:00:00 → request #1
  12:00:30 → request #100 (limit reached)
  12:00:45 → request #101 → REJECTED (429)
  12:01:00 → window resets → request #1 again (allowed)
```

**The boundary attack problem:**

```
Limit: 100/minute, window resets at :00

Attack:
  12:00:55 → send 100 requests (at end of window)
  12:01:00 → window resets
  12:01:05 → send 100 more requests (new window)
  
  In 10 seconds: 200 requests processed
  = 2x the intended rate
  Fixed window is exploitable at boundaries
```

```csharp
public class FixedWindowRateLimiter
{
    private readonly IDatabase _redis;
    private readonly int _limit;
    private readonly TimeSpan _window;

    public async Task<bool> IsAllowedAsync(string clientId)
    {
        // Window key based on current time bucket
        long windowStart = DateTimeOffset.UtcNow.ToUnixTimeSeconds()
            / (long)_window.TotalSeconds;
        string key = $"ratelimit:fixed:{clientId}:{windowStart}";

        var count = await _redis.StringIncrementAsync(key);

        if (count == 1)
            await _redis.KeyExpireAsync(key, _window * 2);

        return count <= _limit;
    }
}
```

**Pros:** Simple. Low memory (one counter per client per window).
**Cons:** Boundary attack allows 2x burst. Traffic spike at window boundary.

### Algorithm 2: Sliding Window Log

Store timestamp of every request. Count requests in the past N seconds.

```
Limit: 5 requests per 60 seconds

T=10: Request arrives. Log: [10]
T=20: Request arrives. Log: [10, 20]
T=30: Request arrives. Log: [10, 20, 30]
T=40: Request arrives. Log: [10, 20, 30, 40]
T=50: Request arrives. Log: [10, 20, 30, 40, 50] ← 5 requests, at limit
T=55: Request arrives. Count in [0, 55] = 5 → REJECT

T=70: T=10 entry is now outside 60-second window
      Log: [20, 30, 40, 50] → only 4 in window
T=70: Request arrives → allowed. Log: [20, 30, 40, 50, 70]
```

```csharp
public class SlidingWindowLogLimiter
{
    private readonly IDatabase _redis;
    private readonly int _limit;
    private readonly TimeSpan _window;

    public async Task<bool> IsAllowedAsync(string clientId)
    {
        string key = $"ratelimit:log:{clientId}";
        long now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        long windowStart = now - (long)_window.TotalMilliseconds;

        const string lua = @"
            -- Remove entries outside window
            redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', ARGV[1])
            
            -- Count current entries
            local count = redis.call('ZCARD', KEYS[1])
            
            if count < tonumber(ARGV[3]) then
                -- Add this request
                redis.call('ZADD', KEYS[1], ARGV[2], ARGV[2])
                redis.call('EXPIRE', KEYS[1], ARGV[4])
                return 1  -- allowed
            end
            return 0  -- rejected
        ";

        var result = (long)await _redis.ScriptEvaluateAsync(
            lua,
            new RedisKey[] { key },
            new RedisValue[]
            {
                windowStart,
                now,
                _limit,
                (long)_window.TotalSeconds + 1
            }
        );

        return result == 1;
    }
}
```

**Pros:** Perfectly accurate. No boundary attack. Smooth rate limiting.
**Cons:** High memory — stores every request timestamp. 1M users × 1,000 requests/hour = 1 billion entries.

### Algorithm 3: Sliding Window Counter

Hybrid approach. Approximate sliding window using two fixed windows.

```
Current window count + weighted previous window count ≤ limit

Current time: 12:01:45 (75% through the 12:01 window)
Previous window (12:00): 80 requests
Current window (12:01): 40 requests (so far)

Weight of previous window = 1 - 0.75 = 0.25 (25% still counts)

Effective count = 40 + (80 × 0.25) = 40 + 20 = 60

Limit is 100 → allowed (60 < 100)
```

```csharp
public class SlidingWindowCounterLimiter
{
    private readonly IDatabase _redis;
    private readonly int _limit;
    private readonly TimeSpan _window;

    public async Task<bool> IsAllowedAsync(string clientId)
    {
        long windowSizeMs = (long)_window.TotalMilliseconds;
        long now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        long currentWindowStart = now - (now % windowSizeMs);
        long previousWindowStart = currentWindowStart - windowSizeMs;

        string currentKey = $"ratelimit:sw:{clientId}:{currentWindowStart}";
        string previousKey = $"ratelimit:sw:{clientId}:{previousWindowStart}";

        // Get both counts atomically
        var currentCount = (long?)await _redis.StringGetAsync(currentKey) ?? 0;
        var previousCount = (long?)await _redis.StringGetAsync(previousKey) ?? 0;

        // How far through current window are we?
        double elapsed = now - currentWindowStart;
        double windowFraction = elapsed / windowSizeMs;

        // Weight previous window by remaining fraction
        double effectiveCount = currentCount + previousCount * (1 - windowFraction);

        if (effectiveCount >= _limit)
            return false;

        // Increment current window
        await _redis.StringIncrementAsync(currentKey);
        await _redis.KeyExpireAsync(currentKey, _window * 2);

        return true;
    }
}
```

**Pros:** Low memory (two counters per client). Good approximation of sliding window. No boundary attacks.
**Cons:** Approximate (assumes even distribution within previous window). Error up to ~0.003% of limit in worst case — acceptable for most uses.

### Algorithm 4: Token Bucket

Tokens accumulate over time up to a maximum. Each request costs one token. Allows bursts up to bucket capacity.

```
Bucket capacity: 10 tokens (burst limit)
Refill rate: 2 tokens/second (sustained rate)

T=0:  Bucket full: [■■■■■■■■■■] (10 tokens)
T=0:  Request → consume 1 → [■■■■■■■■■ ] (9 tokens)
T=0:  Request → consume 1 → [■■■■■■■■  ] (8 tokens)
...
T=5:  10 requests in 5 seconds → [          ] (0 tokens)
T=5:  Request arrives → no tokens → REJECTED

T=5.5: 2 × 0.5 seconds × 2 tokens/sec = 1 token refilled → [■         ]
T=5.5: Request → consume 1 → [          ] (0 tokens)

Sustained rate: 2 requests/second (matches refill rate)
Burst capacity: 10 requests at once (bucket capacity)
```

```csharp
public class TokenBucketLimiter
{
    private readonly IDatabase _redis;
    private readonly double _capacity;      // max tokens
    private readonly double _refillRate;    // tokens per second

    public async Task<bool> IsAllowedAsync(string clientId, double cost = 1)
    {
        string key = $"ratelimit:token:{clientId}";

        const string lua = @"
            local tokens_key = KEYS[1]
            local last_refill_key = KEYS[1] .. ':time'
            
            local capacity = tonumber(ARGV[1])
            local refill_rate = tonumber(ARGV[2])
            local now = tonumber(ARGV[3])
            local cost = tonumber(ARGV[4])
            
            -- Get current state
            local tokens = tonumber(redis.call('get', tokens_key)) or capacity
            local last_refill = tonumber(redis.call('get', last_refill_key)) or now
            
            -- Calculate tokens to add based on time elapsed
            local elapsed = math.max(0, now - last_refill)
            local new_tokens = math.min(capacity, tokens + elapsed * refill_rate)
            
            if new_tokens < cost then
                return {0, new_tokens}  -- rejected
            end
            
            -- Consume tokens
            new_tokens = new_tokens - cost
            redis.call('set', tokens_key, new_tokens)
            redis.call('set', last_refill_key, now)
            redis.call('expire', tokens_key, 3600)
            redis.call('expire', last_refill_key, 3600)
            
            return {1, new_tokens}  -- allowed, tokens remaining
        ";

        var result = (RedisResult[])await _redis.ScriptEvaluateAsync(
            lua,
            new RedisKey[] { key },
            new RedisValue[]
            {
                _capacity,
                _refillRate,
                DateTimeOffset.UtcNow.ToUnixTimeMilliseconds() / 1000.0,
                cost
            }
        );

        bool allowed = (long)result[0] == 1;
        double remaining = (double)result[1];

        return allowed;
    }
}
```

**Pros:** Handles bursts naturally (bucket fills up during quiet periods). Smooth rate limiting. Intuitive to reason about.
**Cons:** Two state variables per client (tokens + last refill time). Slightly more complex than counter approaches.

### Algorithm 5: Leaky Bucket

Requests enter a queue (the bucket). They're processed at a fixed rate (the leak). If the bucket overflows, reject the request.

```
Bucket size: 10 (queue capacity)
Leak rate: 2 requests/second (processing rate)

T=0:  5 requests arrive → bucket: [■■■■■] (5 queued)
T=0:  3 more requests arrive → bucket: [■■■■■■■■] (8 queued)
T=0:  5 more requests arrive → bucket full at 10 → 3 rejected

Processing: 2 requests drained per second
T=1: bucket → [■■■■■■] (2 drained, 6 remaining)
```

Leaky bucket enforces **perfectly uniform output rate** regardless of input burst pattern.

```csharp
public class LeakyBucketLimiter
{
    private readonly Queue<TaskCompletionSource<bool>> _queue = new();
    private readonly int _bucketSize;
    private readonly TimeSpan _leakInterval; // time between processing each request
    private readonly object _lock = new();
    private bool _leaking = false;

    public LeakyBucketLimiter(int bucketSize, int requestsPerSecond)
    {
        _bucketSize = bucketSize;
        _leakInterval = TimeSpan.FromSeconds(1.0 / requestsPerSecond);
    }

    public Task<bool> IsAllowedAsync()
    {
        lock (_lock)
        {
            if (_queue.Count >= _bucketSize)
                return Task.FromResult(false); // bucket overflow → reject

            var tcs = new TaskCompletionSource<bool>();
            _queue.Enqueue(tcs);

            if (!_leaking)
            {
                _leaking = true;
                _ = Task.Run(LeakAsync);
            }

            return tcs.Task;
        }
    }

    private async Task LeakAsync()
    {
        while (true)
        {
            await Task.Delay(_leakInterval);

            TaskCompletionSource<bool>? next;
            lock (_lock)
            {
                if (!_queue.TryDequeue(out next))
                {
                    _leaking = false;
                    return;
                }
            }

            next.SetResult(true); // allow the queued request to proceed
        }
    }
}
```

**Pros:** Perfectly smooth output rate. Protects downstream from any burst.
**Cons:** Adds latency (requests wait in queue). Queue requires memory. Less intuitive burst handling than token bucket.

### Algorithm 6: Sliding Window Counter (Redis-native)

Production-grade implementation combining the best properties.

```csharp
public class ProductionRateLimiter
{
    private readonly IDatabase _redis;
    private readonly RateLimitConfig _config;

    public async Task<RateLimitResult> CheckAsync(
        string identifier,
        string operation = "default")
    {
        var config = _config.GetConfig(operation);
        string key = $"rl:{operation}:{identifier}";
        long now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        long windowMs = (long)config.Window.TotalMilliseconds;
        long windowStart = now - windowMs;

        const string lua = @"
            local key = KEYS[1]
            local window_start = ARGV[1]
            local now = ARGV[2]
            local limit = tonumber(ARGV[3])
            local window_seconds = tonumber(ARGV[4])
            
            -- Atomic: remove expired, count, conditionally add
            redis.call('ZREMRANGEBYSCORE', key, '-inf', window_start)
            local count = redis.call('ZCARD', key)
            
            local remaining = limit - count
            local allowed = 0
            
            if count < limit then
                redis.call('ZADD', key, now, now .. '-' .. math.random(1000000))
                redis.call('EXPIRE', key, window_seconds)
                allowed = 1
                remaining = remaining - 1
            end
            
            -- Calculate reset time (oldest entry + window duration)
            local oldest = redis.call('ZRANGE', key, 0, 0, 'WITHSCORES')
            local reset = now + (window_seconds * 1000)
            if #oldest > 0 then
                reset = tonumber(oldest[2]) + (window_seconds * 1000)
            end
            
            return {allowed, count + (1 - allowed), remaining, reset}
        ";

        var result = (RedisResult[])await _redis.ScriptEvaluateAsync(
            lua,
            new RedisKey[] { key },
            new RedisValue[]
            {
                windowStart,
                now,
                config.Limit,
                (long)config.Window.TotalSeconds
            }
        );

        bool allowed = (long)result[0] == 1;
        int currentCount = (int)(long)result[1];
        int remaining = (int)(long)result[2];
        long resetMs = (long)result[3];

        return new RateLimitResult
        {
            Allowed = allowed,
            Limit = config.Limit,
            Remaining = Math.Max(0, remaining),
            CurrentCount = currentCount,
            ResetAt = DateTimeOffset.FromUnixTimeMilliseconds(resetMs),
            RetryAfter = allowed ? null : TimeSpan.FromMilliseconds(resetMs - now)
        };
    }
}

public record RateLimitResult
{
    public bool Allowed { get; init; }
    public int Limit { get; init; }
    public int Remaining { get; init; }
    public int CurrentCount { get; init; }
    public DateTimeOffset ResetAt { get; init; }
    public TimeSpan? RetryAfter { get; init; }
}
```

## Where to Place Rate Limiting — The Layers

```
Layer 1: At the edge (CDN/WAF)
  Cloudflare, AWS WAF
  Rate limit by IP before traffic hits your infrastructure
  Zero infrastructure cost, but coarse-grained control
  Best for: DDoS, volumetric attacks

Layer 2: API Gateway
  Kong, AWS API Gateway, custom Nginx
  Rate limit by API key or IP
  Before traffic hits application servers
  Best for: API product tiers, coarse per-client limits

Layer 3: Application middleware (most flexible)
  In your ASP.NET Core, Express, etc. middleware
  Access to user context (authenticated user ID, tier, roles)
  Per-endpoint, per-user, per-tier control
  Best for: business logic rate limiting

Layer 4: Service-specific limits
  Within specific services (booking service, payment service)
  Domain-specific rules: "max 5 booking attempts per event per user"
  Best for: business rule enforcement

Production recommendation: ALL FOUR LAYERS
  Edge: blocks DDoS
  Gateway: enforces API key quotas
  Middleware: enforces per-user per-endpoint limits
  Service: enforces business rules
```

```csharp
public class RateLimitAwareClient
{
    public async Task<T> CallWithBackoffAsync<T>(
        Func<Task<HttpResponseMessage>> apiCall,
        int maxRetries = 5)
    {
        int retryCount = 0;
        while (retryCount < maxRetries)
        {
            var response = await apiCall();

            if (response.StatusCode != HttpStatusCode.TooManyRequests)
                return await response.Content.ReadFromJsonAsync<T>()
                    ?? throw new Exception("Null response");

            // Respect Retry-After header
            var retryAfter = response.Headers.RetryAfter?.Delta
                ?? TimeSpan.FromSeconds(Math.Pow(2, retryCount) + Random.Shared.Next(0, 1000) / 1000.0);

            Console.WriteLine($"Rate limited. Waiting {retryAfter.TotalSeconds:F1}s before retry {retryCount + 1}");
            await Task.Delay(retryAfter);
            retryCount++;
        }
        throw new RateLimitExceededException($"Rate limit exceeded after {maxRetries} retries");
    }
}
```

### Question 1: Implement all five algorithms

> "Implement a rate limiter. Which algorithm would you choose?"

The fixed window implementation was shown above. Here's a clean comparison of all approaches in a single interface:

```csharp
public interface IRateLimiter
{
    Task<bool> IsAllowedAsync(string clientId);
}

// In-memory token bucket — for single-server, no Redis
public class InMemoryTokenBucket : IRateLimiter
{
    private readonly ConcurrentDictionary<string, BucketState> _buckets = new();
    private readonly double _capacity;
    private readonly double _refillPerMs;

    public InMemoryTokenBucket(double capacity, double refillPerSecond)
    {
        _capacity = capacity;
        _refillPerMs = refillPerSecond / 1000.0;
    }

    public Task<bool> IsAllowedAsync(string clientId)
    {
        long now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();

        var bucket = _buckets.GetOrAdd(clientId, _ => new BucketState
        {
            Tokens = _capacity,
            LastRefillMs = now
        });

        lock (bucket)
        {
            // Refill tokens based on elapsed time
            long elapsed = now - bucket.LastRefillMs;
            bucket.Tokens = Math.Min(_capacity, bucket.Tokens + elapsed * _refillPerMs);
            bucket.LastRefillMs = now;

            if (bucket.Tokens < 1.0)
                return Task.FromResult(false);

            bucket.Tokens -= 1.0;
            return Task.FromResult(true);
        }
    }
}

public class BucketState
{
    public double Tokens { get; set; }
    public long LastRefillMs { get; set; }
}
```

### Question 2: Design a distributed rate limiter using Redis

Already covered in the sliding window implementation. Key points to emphasize:

```csharp
// The Lua script atomicity is critical — explain WHY
// Without atomic script, race condition:
//   Thread 1: ZCARD = 99 (below limit 100)
//   Thread 2: ZCARD = 99 (below limit 100)
//   Thread 1: ZADD (now 100)
//   Thread 2: ZADD (now 101) ← exceeded limit!
//
// With Lua: Redis executes ZCARD + ZADD atomically
// No other commands can interleave
// Thread safety guaranteed

// Also explain the sorted set choice:
// Key: userId
// Score: timestamp (milliseconds)
// Value: unique timestamp (prevents dedup of simultaneous requests)
// ZREMRANGEBYSCORE: O(log N) cleanup of old entries
// ZCARD: O(1) count
```

### Question 3: LeetCode 1721-adjacent — Design Hit Counter

> "Design a system to count how many hits occurred in the past 5 minutes."

```csharp
public class HitCounter
{
    // Array of 300 buckets (one per second for 5 minutes)
    private readonly int[] _hits = new int[300];
    private readonly int[] _timestamps = new int[300];
    private readonly object _lock = new();

    // Record a hit at given timestamp (Unix seconds)
    public void Hit(int timestamp)
    {
        lock (_lock)
        {
            int bucket = timestamp % 300;

            if (_timestamps[bucket] != timestamp)
            {
                // New time period — reset this bucket
                _timestamps[bucket] = timestamp;
                _hits[bucket] = 0;
            }

            _hits[bucket]++;
        }
    }

    // Count hits in past 5 minutes (300 seconds)
    public int GetHits(int timestamp)
    {
        lock (_lock)
        {
            int total = 0;
            for (int i = 0; i < 300; i++)
            {
                // Only count buckets within the 5-minute window
                if (timestamp - _timestamps[i] < 300)
                    total += _hits[i];
            }
            return total;
        }
    }
}

// This is essentially a fixed-window rate limiter with 1-second granularity
// Each array position = one second's bucket
// Circular array naturally overwrites old data
// O(300) = O(1) for both Hit and GetHits
```

```csharp
public class RateLimitMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ProductionRateLimiter _limiter;
    private readonly RateLimitOptions _options;

    public RateLimitMiddleware(
        RequestDelegate next,
        ProductionRateLimiter limiter,
        RateLimitOptions options)
    {
        _next = next;
        _limiter = limiter;
        _options = options;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // Determine identifier (user ID if authenticated, IP otherwise)
        string identifier = context.User.Identity?.IsAuthenticated == true
            ? $"user:{context.User.FindFirstValue(ClaimTypes.NameIdentifier)}"
            : $"ip:{context.Connection.RemoteIpAddress}";

        // Determine operation (endpoint + method)
        string operation = $"{context.Request.Method}:{context.Request.Path}";

        var result = await _limiter.CheckAsync(identifier, operation);

        // Always add rate limit headers
        context.Response.Headers["X-RateLimit-Limit"] = result.Limit.ToString();
        context.Response.Headers["X-RateLimit-Remaining"] = result.Remaining.ToString();
        context.Response.Headers["X-RateLimit-Reset"] =
            result.ResetAt.ToUnixTimeSeconds().ToString();

        if (!result.Allowed)
        {
            context.Response.StatusCode = StatusCodes.Status429TooManyRequests;

            if (result.RetryAfter.HasValue)
                context.Response.Headers["Retry-After"] =
                    ((int)result.RetryAfter.Value.TotalSeconds).ToString();

            await context.Response.WriteAsJsonAsync(new
            {
                error = "rate_limit_exceeded",
                message = $"Rate limit of {result.Limit} requests per window exceeded.",
                limit = result.Limit,
                remaining = 0,
                reset_at = result.ResetAt,
                retry_after_seconds = result.RetryAfter?.TotalSeconds
            });

            return; // Do not call next middleware
        }

        await _next(context);
    }
}
```

### Question 5: Multi-tier rate limiting

> "Implement different rate limits for free vs pro vs enterprise users."

```csharp
public class TieredRateLimiter
{
    private readonly IDatabase _redis;

    private static readonly Dictionary<string, TierConfig> _tiers = new()
    {
        ["free"] = new TierConfig(
            RequestsPerMinute: 60,
            RequestsPerDay: 1_000,
            BookingsPerDay: 5
        ),
        ["pro"] = new TierConfig(
            RequestsPerMinute: 600,
            RequestsPerDay: 100_000,
            BookingsPerDay: 100
        ),
        ["enterprise"] = new TierConfig(
            RequestsPerMinute: int.MaxValue, // unlimited
            RequestsPerDay: int.MaxValue,
            BookingsPerDay: int.MaxValue
        )
    };

    public async Task<TieredRateLimitResult> CheckAsync(
        string userId,
        string userTier,
        string operation)
    {
        if (!_tiers.TryGetValue(userTier, out var config))
            config = _tiers["free"];

        // Enterprise: skip rate limiting entirely
        if (config.RequestsPerMinute == int.MaxValue)
            return TieredRateLimitResult.Allowed(int.MaxValue, int.MaxValue);

        var tasks = new List<Task<(bool allowed, int remaining, string limitType)>>();

        // Check per-minute limit
        tasks.Add(CheckLimitAsync(
            $"rl:{userId}:minute",
            config.RequestsPerMinute,
            TimeSpan.FromMinutes(1),
            "per_minute"
        ));

        // Check per-day limit
        tasks.Add(CheckLimitAsync(
            $"rl:{userId}:day",
            config.RequestsPerDay,
            TimeSpan.FromDays(1),
            "per_day"
        ));

        // Check operation-specific limit
        if (operation == "booking")
        {
            tasks.Add(CheckLimitAsync(
                $"rl:{userId}:booking:day",
                config.BookingsPerDay,
                TimeSpan.FromDays(1),
                "bookings_per_day"
            ));
        }

        var results = await Task.WhenAll(tasks);

        // User is blocked if ANY limit is exceeded
        var failedLimit = results.FirstOrDefault(r => !r.allowed);
        if (failedLimit != default)
        {
            return TieredRateLimitResult.Blocked(
                limitType: failedLimit.limitType,
                tier: userTier,
                upgradeMessage: userTier == "free"
                    ? "Upgrade to Pro for 10x higher limits"
                    : null
            );
        }

        int mostConstrainedRemaining = results.Min(r => r.remaining);
        return TieredRateLimitResult.Allowed(
            remaining: mostConstrainedRemaining,
            limit: config.RequestsPerMinute
        );
    }

    private async Task<(bool allowed, int remaining, string limitType)> CheckLimitAsync(
        string key,
        int limit,
        TimeSpan window,
        string limitType)
    {
        long now = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        long windowStart = now - (long)window.TotalMilliseconds;

        // Sliding window with sorted set
        const string lua = @"
            redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', ARGV[1])
            local count = redis.call('ZCARD', KEYS[1])
            if count < tonumber(ARGV[3]) then
                redis.call('ZADD', KEYS[1], ARGV[2], ARGV[2])
                redis.call('EXPIRE', KEYS[1], ARGV[4])
                return {1, tonumber(ARGV[3]) - count - 1}
            end
            return {0, 0}
        ";

        var result = (RedisResult[])await _redis.ScriptEvaluateAsync(
            lua,
            new RedisKey[] { key },
            new RedisValue[]
            {
                windowStart,
                now,
                limit,
                (long)window.TotalSeconds + 1
            }
        );

        return ((long)result[0] == 1, (int)(long)result[1], limitType);
    }
}

public record TierConfig(int RequestsPerMinute, int RequestsPerDay, int BookingsPerDay);

public record TieredRateLimitResult
{
    public bool Allowed { get; private init; }
    public int Remaining { get; private init; }
    public int Limit { get; private init; }
    public string? LimitType { get; private init; }
    public string? Tier { get; private init; }
    public string? UpgradeMessage { get; private init; }

    public static TieredRateLimitResult Allowed(int remaining, int limit) =>
        new() { Allowed = true, Remaining = remaining, Limit = limit };

    public static TieredRateLimitResult Blocked(
        string limitType, string tier, string? upgradeMessage) =>
        new()
        {
            Allowed = false,
            LimitType = limitType,
            Tier = tier,
            UpgradeMessage = upgradeMessage
        };
}
```

## Algorithm Comparison

| Algorithm | Memory | Burst Handling | Boundary Attack | Best For |
|---|---|---|---|---|
| Fixed Window | O(1) | Allows 2× burst at boundary | Vulnerable | Simple cases, internal APIs |
| Sliding Log | O(requests) | Perfect | Immune | Low-traffic, accuracy critical |
| Sliding Counter | O(1) | ~Accurate | ~Immune | Production (best balance) |
| Token Bucket | O(1) | Configurable | Immune | User-facing APIs, burst tolerance |
| Leaky Bucket | O(queue) | Smooths bursts | Immune | Protecting downstream services |

## One-Line Summary for Interviews

> **Rate limiting protects services from overload and abuse by counting requests per client per time window and rejecting excess with HTTP 429 — implemented via fixed window counters (simple but boundary-exploitable), sliding window logs (accurate but memory-heavy), sliding window counters (best balance of accuracy and memory), or token buckets (best for handling legitimate bursts) — all stored in Redis using atomic Lua scripts to prevent race conditions across distributed servers — applied at multiple layers from CDN edge (IP-based volumetric protection) through API gateway (API key quotas) to application middleware (per-user per-endpoint limits based on authenticated identity and business tier) — with proper Retry-After and X-RateLimit headers so well-behaved clients can implement exponential backoff rather than hammering a 429 wall.**
