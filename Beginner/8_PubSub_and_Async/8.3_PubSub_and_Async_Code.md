## Q1. Implement an In-Memory Pub/Sub Broker

> Publishers send messages to named topics. All subscribers of that topic receive a copy. Subscribers can unsubscribe at any time. The publisher has no knowledge of who is subscribed.

### Key Difference from a Message Queue
```
Message Queue:  one message → one consumer (competing consumers)
Pub/Sub:        one message → ALL subscribers (fan-out)

Queue analogy:  task list — one worker picks up each task
Pub/Sub analogy: radio broadcast — every listener hears every transmission
```

```csharp
using System;
using System.Collections.Generic;

public class PubSubBroker
{
    // topic → list of (subscriberId, handler)
    private readonly Dictionary<string, List<(string Id, Action<string, object> Handler)>>
        _subscribers = new();
    private readonly object _lock = new();

    public void Subscribe(string topic, string subscriberId, Action<string, object> handler)
    {
        lock (_lock)
        {
            if (!_subscribers.ContainsKey(topic))
                _subscribers[topic] = new();

            // Prevent duplicate subscriptions
            if (_subscribers[topic].Any(s => s.Id == subscriberId))
            {
                Console.WriteLine($"[BROKER] {subscriberId} already subscribed to '{topic}'");
                return;
            }

            _subscribers[topic].Add((subscriberId, handler));
            Console.WriteLine($"[BROKER] {subscriberId} subscribed to '{topic}'  " +
                              $"(total: {_subscribers[topic].Count})");
        }
    }

    public void Unsubscribe(string topic, string subscriberId)
    {
        lock (_lock)
        {
            if (!_subscribers.TryGetValue(topic, out var subs)) return;
            int removed = subs.RemoveAll(s => s.Id == subscriberId);
            if (removed > 0)
                Console.WriteLine($"[BROKER] {subscriberId} unsubscribed from '{topic}'");
        }
    }

    // Fan-out: deliver to ALL subscribers of the topic
    public int Publish(string topic, object message)
    {
        List<(string Id, Action<string, object> Handler)> snapshot;

        lock (_lock)
        {
            if (!_subscribers.TryGetValue(topic, out var subs) || subs.Count == 0)
            {
                Console.WriteLine($"[BROKER] Published to '{topic}' — no subscribers");
                return 0;
            }
            snapshot = new List<(string, Action<string, object>)>(subs);
        }

        Console.WriteLine($"[BROKER] Publishing to '{topic}' → {snapshot.Count} subscriber(s)");

        foreach (var (id, handler) in snapshot)
        {
            try   { handler(topic, message); }
            catch (Exception ex)
            { Console.WriteLine($"  [ERROR] {id} handler threw: {ex.Message}"); }
        }

        return snapshot.Count;
    }

    public void PrintTopics()
    {
        lock (_lock)
        {
            Console.WriteLine("\n[BROKER] Active topics:");
            foreach (var (topic, subs) in _subscribers)
                Console.WriteLine($"  '{topic}': [{string.Join(", ", subs.Select(s => s.Id))}]");
        }
    }
}

// Usage — e-commerce order placed event
var broker = new PubSubBroker();

// Each service subscribes on startup
broker.Subscribe("order.placed", "email-service", (topic, msg) =>
    Console.WriteLine($"  [email-service]     Sending confirmation for: {msg}"));

broker.Subscribe("order.placed", "inventory-service", (topic, msg) =>
    Console.WriteLine($"  [inventory-service] Reserving stock for: {msg}"));

broker.Subscribe("order.placed", "analytics-service", (topic, msg) =>
    Console.WriteLine($"  [analytics-service] Recording sale: {msg}"));

broker.Subscribe("order.placed", "fraud-service", (topic, msg) =>
    Console.WriteLine($"  [fraud-service]     Running fraud check on: {msg}"));

broker.PrintTopics();

Console.WriteLine("\n=== Order Service publishes (knows nothing about subscribers) ===");
broker.Publish("order.placed", new { OrderId = 1001, Total = 99.99, UserId = 42 });

Console.WriteLine("\n=== Adding new service requires zero changes to Order Service ===");
broker.Subscribe("order.placed", "loyalty-service", (topic, msg) =>
    Console.WriteLine($"  [loyalty-service]   Adding points for: {msg}"));

broker.Publish("order.placed", new { OrderId = 1002, Total = 49.99, UserId = 7 });

Console.WriteLine("\n=== Fraud service goes down — unsubscribes ===");
broker.Unsubscribe("order.placed", "fraud-service");
broker.Publish("order.placed", new { OrderId = 1003, Total = 199.99, UserId = 15 });

// Output:
// [BROKER] email-service subscribed to 'order.placed'     (total: 1)
// [BROKER] inventory-service subscribed to 'order.placed' (total: 2)
// [BROKER] analytics-service subscribed to 'order.placed' (total: 3)
// [BROKER] fraud-service subscribed to 'order.placed'     (total: 4)
//
// === Order Service publishes ===
// [BROKER] Publishing to 'order.placed' → 4 subscriber(s)
//   [email-service]     Sending confirmation for: { OrderId = 1001, Total = 99.99 ... }
//   [inventory-service] Reserving stock for: ...
//   [analytics-service] Recording sale: ...
//   [fraud-service]     Running fraud check on: ...
```

### Why Pub/Sub Enables Open/Closed Systems
```
Without Pub/Sub:
  Order Service:
    send_confirmation_email(order)
    decrement_inventory(order)
    record_analytics(order)   ← must modify Order Service every time
    run_fraud_check(order)      a new service needs order events

With Pub/Sub:
  Order Service:
    broker.Publish("order.placed", order)  ← never changes

  New Loyalty Service just subscribes — Order Service is untouched.
  This is the Open/Closed Principle at the system level.
```

**Complexity:** Subscribe O(1), Unsubscribe O(n), Publish O(subscribers) — n = subscribers per topic

## Q2. Implement the Fan-Out Pattern (SNS + SQS Style)

> One published message fans out to multiple isolated queues — one per subscriber service. Each service processes from its own queue independently. A slow or crashed service doesn't block others.

### Why Fan-Out + Isolated Queues Beats Direct Pub/Sub
```
Pure Pub/Sub problem:
  Broker pushes to all subscribers simultaneously
  If Email Service is slow → it blocks the broker
  If Inventory Service crashes → it loses messages

Fan-Out + Queue solution:
  Broker delivers to each service's queue immediately (fast)
  Each service drains its own queue at its own pace
  Crashed service: messages buffer in queue → no loss
  Slow service:    its queue grows → others unaffected

  [SNS Topic] → [SQS Queue: email]     → [Email Workers]   (can scale independently)
              → [SQS Queue: inventory] → [Inventory Worker]
              → [SQS Queue: analytics] → [Analytics Worker]
```

```csharp
public class FanOutBroker
{
    // Each subscriber service gets its own isolated queue
    private readonly Dictionary<string, Queue<object>> _queues = new();
    private readonly object _lock = new();

    public void AddSubscriberQueue(string serviceName)
    {
        lock (_lock)
        {
            _queues[serviceName] = new Queue<object>();
            Console.WriteLine($"[FAN-OUT] Queue created for: {serviceName}");
        }
    }

    // Publish: copies message into EVERY subscriber queue (fan-out)
    public void Publish(string topic, object message)
    {
        lock (_lock)
        {
            Console.WriteLine($"[FAN-OUT] '{topic}' → fanning out to {_queues.Count} queue(s)");
            foreach (var (service, queue) in _queues)
            {
                queue.Enqueue(message);
                Console.WriteLine($"  → Enqueued in [{service}] queue  (depth: {queue.Count})");
            }
        }
    }

    // Each service drains its own queue independently
    public object? Consume(string serviceName)
    {
        lock (_lock)
        {
            if (!_queues.TryGetValue(serviceName, out var queue) || queue.Count == 0)
                return null;
            return queue.Dequeue();
        }
    }

    public int QueueDepth(string serviceName)
    {
        lock (_lock)
        {
            return _queues.TryGetValue(serviceName, out var q) ? q.Count : 0;
        }
    }

    public void PrintStatus()
    {
        lock (_lock)
        {
            Console.WriteLine("\n[FAN-OUT] Queue depths:");
            foreach (var (svc, q) in _queues)
                Console.WriteLine($"  {svc,-20}: {q.Count} message(s) pending");
        }
    }
}

// Usage — order events fan out to three service queues
var fanOut = new FanOutBroker();
fanOut.AddSubscriberQueue("email-service");
fanOut.AddSubscriberQueue("inventory-service");
fanOut.AddSubscriberQueue("analytics-service");

Console.WriteLine("\n=== 3 orders published ===");
fanOut.Publish("order.placed", new { OrderId = 1001, Total = 99.99 });
fanOut.Publish("order.placed", new { OrderId = 1002, Total = 49.99 });
fanOut.Publish("order.placed", new { OrderId = 1003, Total = 149.99 });

fanOut.PrintStatus();
// email-service:    3 messages pending
// inventory-service: 3 messages pending
// analytics-service: 3 messages pending

Console.WriteLine("\n=== email-service processes its queue ===");
while (true)
{
    var msg = fanOut.Consume("email-service");
    if (msg == null) break;
    Console.WriteLine($"  [email-service] Processed: {msg}");
}

Console.WriteLine("\n=== analytics-service crashes — queue holds its messages ===");
// analytics-service doesn't consume — messages buffer safely
Console.WriteLine($"  analytics-service queue depth: {fanOut.QueueDepth("analytics-service")}");
Console.WriteLine($"  inventory-service queue depth: {fanOut.QueueDepth("inventory-service")}");
Console.WriteLine("  → inventory-service is unaffected by analytics crash");

Console.WriteLine("\n=== analytics-service recovers, drains backlog ===");
while (true)
{
    var msg = fanOut.Consume("analytics-service");
    if (msg == null) break;
    Console.WriteLine($"  [analytics-service] Recovered: {msg}");
}

// Key insight:
// Each service's queue acts as a buffer — no message loss, no cross-service blocking
```

### Fan-Out vs Pure Pub/Sub — When to Use Each
```
Use Pure Pub/Sub when:
  → All subscribers are fast and reliable (internal in-process handlers)
  → Message loss on subscriber crash is acceptable
  → Low volume, real-time: stock tickers, live dashboards

Use Fan-Out + Queues (SNS+SQS) when:
  → Subscribers have different processing speeds
  → A crashed subscriber must not lose messages
  → Subscribers need independent scaling (add email workers without touching others)
  → High volume: order processing, notifications, ETL pipelines
  → This is the standard pattern in production AWS architectures
```

**Complexity:** Publish O(subscribers), Consume O(1), Space O(subscribers × messages)

## Q3. Implement Kafka-Style Consumer Groups

> A topic has multiple consumer groups. Each group receives ALL messages (like pub/sub). Within a group, each message goes to exactly ONE instance (like a queue). This gives both fan-out across services and load-sharing within a service.

### The Consumer Group Model
```
Topic: order.placed  [msg1, msg2, msg3, msg4, msg5, msg6]

Consumer Group A (Email Service — 2 instances):
  Instance A1 → gets msg1, msg3, msg5  (load shared within group)
  Instance A2 → gets msg2, msg4, msg6

Consumer Group B (Analytics Service — 1 instance):
  Instance B1 → gets msg1, msg2, msg3, msg4, msg5, msg6  (gets ALL)

→ Groups are independent: adding Group C doesn't affect A or B
→ Within Group A: messages are distributed (not duplicated)
→ Across Groups: messages are duplicated (each group gets all)
```

```csharp
public class KafkaStyleTopic
{
    private readonly List<object> _log = new(); // append-only message log
    private readonly object _lock = new();

    // group → (instances, next offset per instance — round-robin)
    private readonly Dictionary<string, (List<string> Instances, int GroupOffset)>
        _groups = new();

    public void Publish(object message)
    {
        lock (_lock)
        {
            _log.Add(message);
            Console.WriteLine($"[TOPIC] Published offset={_log.Count - 1}: {message}");
        }
    }

    public void RegisterGroup(string groupName, List<string> instanceIds)
    {
        lock (_lock)
        {
            _groups[groupName] = (new List<string>(instanceIds), groupOffset: 0);
            Console.WriteLine($"[GROUP] '{groupName}' registered with " +
                              $"{instanceIds.Count} instance(s): [{string.Join(", ", instanceIds)}]");
        }
    }

    // Each group polls independently. Within the group, messages are round-robined
    // across instances. Returns null when group is caught up.
    public (string? InstanceId, object? Message) Poll(string groupName)
    {
        lock (_lock)
        {
            if (!_groups.TryGetValue(groupName, out var group)) return (null, null);

            if (group.GroupOffset >= _log.Count) return (null, null); // caught up

            object message     = _log[group.GroupOffset];
            string instanceId  = group.Instances[group.GroupOffset % group.Instances.Count];

            _groups[groupName] = (group.Instances, group.GroupOffset + 1);
            return (instanceId, message);
        }
    }

    // Drain all pending messages for a group
    public void DrainGroup(string groupName)
    {
        Console.WriteLine($"\n[DRAIN] Group '{groupName}' consuming:");
        while (true)
        {
            var (instanceId, message) = Poll(groupName);
            if (message == null) break;
            Console.WriteLine($"  [{instanceId}] consumed: {message}");
        }
        Console.WriteLine($"  Group '{groupName}' caught up.");
    }
}

// Usage
var topic = new KafkaStyleTopic();

// Two consumer groups — each gets ALL messages
topic.RegisterGroup("email-service",    new List<string> { "email-1", "email-2" });
topic.RegisterGroup("analytics-service", new List<string> { "analytics-1" });

Console.WriteLine("\n=== Publishing 6 orders ===");
for (int i = 1; i <= 6; i++)
    topic.Publish(new { OrderId = 1000 + i, Total = i * 10.0 });

Console.WriteLine();
// Each group drains independently
topic.DrainGroup("email-service");
topic.DrainGroup("analytics-service");

Console.WriteLine("\n=== 3 more orders published ===");
for (int i = 7; i <= 9; i++)
    topic.Publish(new { OrderId = 1000 + i, Total = i * 10.0 });

// email-service has 2 workers sharing the load
// analytics-service gets all 3 by itself
topic.DrainGroup("email-service");
topic.DrainGroup("analytics-service");

// Output:
// [GROUP] 'email-service'     registered with 2 instance(s): [email-1, email-2]
// [GROUP] 'analytics-service' registered with 1 instance(s): [analytics-1]
//
// === Publishing 6 orders ===
// [TOPIC] Published offset=0: { OrderId = 1001, Total = 10 }
// ...
//
// [DRAIN] Group 'email-service' consuming:
//   [email-1]     consumed: { OrderId = 1001 }   ← instance 1 gets odd offsets
//   [email-2]     consumed: { OrderId = 1002 }   ← instance 2 gets even offsets
//   [email-1]     consumed: { OrderId = 1003 }
//   [email-2]     consumed: { OrderId = 1004 }
//   [email-1]     consumed: { OrderId = 1005 }
//   [email-2]     consumed: { OrderId = 1006 }
//   Group 'email-service' caught up.
//
// [DRAIN] Group 'analytics-service' consuming:
//   [analytics-1] consumed: { OrderId = 1001 }   ← single instance gets all
//   [analytics-1] consumed: { OrderId = 1002 }
//   ...all 6 messages...
//   Group 'analytics-service' caught up.
```

### What Real Kafka Adds On Top
```
1. Partitions — parallelism within a topic:
   Topic has N partitions → N consumers in a group can work in parallel
   partition key = order.user_id → all orders from same user → same partition
   → Ordering guaranteed per user, parallelism across users

2. Offsets — each group tracks its own read position:
   Group A at offset 500 → processing recent messages
   Group B at offset 0   → replaying all history (backfill)
   → Groups are completely independent; no coordination needed

3. Retention — messages not deleted after consumption:
   Stored on disk for configurable period (days/weeks/forever)
   New consumer group can replay from beginning
   → Used for audit logs, event sourcing, ML feature pipelines

4. Consumer group rebalance — when instances join/leave:
   Partitions are redistributed across available instances
   Kafka coordinates this automatically
```

**Complexity:** Publish O(1), Poll O(1), DrainGroup O(messages) — append-only log makes publish always O(1)

## Complexity Summary

| Pattern | Publish | Subscribe/Consume | Fan-out | Key Property |
|---------|---------|-------------------|---------|--------------|
| Pub/Sub Broker | O(subscribers) | O(1) subscribe | All get copy | Publisher decoupled from subscribers |
| Fan-Out + Queues | O(subscribers) | O(1) per consume | Isolated queues | Crashed service loses no messages |
| Consumer Groups | O(1) append | O(1) poll | Per-group full copy | Load-share within group, fan-out across groups |

## What Interviewers Test

```
1. Pub/Sub vs Queue — one sentence difference?
   → Queue: competing consumers, one gets the message
   → Pub/Sub: all subscribers get a copy
   → Mnemonic: Queue = task list, Pub/Sub = radio broadcast

2. Why use fan-out queues instead of direct pub/sub?
   → Isolation: slow/crashed subscriber doesn't affect others
   → Buffering: messages survive subscriber downtime
   → Independent scaling: add email workers without touching analytics

3. Consumer groups — why is it both pub/sub AND queue?
   → Across groups: every group gets all messages (pub/sub)
   → Within a group: each message goes to one instance (queue)
   → This is Kafka's core insight — the unified model

4. What happens if a subscriber is slow in direct pub/sub?
   → Broker blocks waiting to deliver → other subscribers delayed
   → Fix: async delivery (goroutines/threads) or fan-out to queues

5. How does Kafka guarantee ordering?
   → Partition key routes related messages to the same partition
   → Within a partition, messages are strictly ordered
   → One consumer per partition per group → no concurrent access to same partition

6. What is the difference between push and pull delivery?
   → Push: broker sends to subscriber (low latency, subscriber must keep up)
   → Pull: subscriber polls broker (backpressure, subscriber controls rate)
   → Kafka uses pull → consumers never overwhelmed by broker
   → SNS uses push → subscribers must handle the rate
```
