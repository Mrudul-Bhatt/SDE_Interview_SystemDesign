# Consistent Hashing — Deep Dive for System Design Interviews

## The Problem It Solves

Imagine you have a distributed cache with **N servers**. The naive approach to routing keys is:

```
server = hash(key) % N
```

This works great — until you **add or remove a server**.

When N changes to N±1, **almost every key remaps to a different server**. In a cache with 1 million keys and 10 servers, removing one server invalidates ~90% of cached data. Every cache miss now hits your database. This is a **cache stampede** — catastrophic for high-traffic systems like AXS during a flash sale.

---

## The Core Idea

Consistent hashing maps both **servers and keys onto the same circular ring** (0 to 2³²-1).

```
                    0
                    │
          ┌─────────┴─────────┐
          │                   │
   2²³    │       Ring        │  2⁸
          │                   │
          └─────────┬─────────┘
                  2³¹
```

**Assignment rule:** A key is assigned to the **first server encountered clockwise** on the ring.

---

## Step-by-Step Mechanics

### 1. Place servers on the ring
Hash each server's IP/name → a position on the ring.

```
hash("Server-A") → position 12
hash("Server-B") → position 145
hash("Server-C") → position 230
```

```
         0
         │
    C(230)      A(12)
         \      /
          ──────
         /      \
    B(145)
```

### 2. Place keys on the ring
```
hash("ticket:show:123") → position 50  → goes to Server-B (next clockwise)
hash("ticket:show:456") → position 180 → goes to Server-C
hash("user:session:789") → position 5  → goes to Server-A
```

### 3. Adding a server
Add Server-D at position 100.
- **Only keys between 12 and 100** (previously on Server-B) move to Server-D.
- Everything else is **untouched**.

### 4. Removing a server
Remove Server-B at position 145.
- Its keys now go to Server-C (next clockwise).
- Everything else is **untouched**.

> **Key insight:** Only `K/N` keys are remapped on average, not all of them.

---

## The Virtual Nodes Problem

Naive consistent hashing has a flaw: **uneven distribution**.

If servers land at positions 12, 14, 16, they each cover wildly different arc sizes → one server handles 80% of keys, others handle 5%.

### Solution: Virtual Nodes (vnodes)

Each physical server gets **multiple positions** on the ring:

```csharp
// Each server maps to V virtual nodes
for (int i = 0; i < virtualNodeCount; i++)
{
    string vnodeKey = $"{serverIp}#vnode-{i}";
    int position = Hash(vnodeKey);
    ring.Add(position, serverIp);  // SortedDictionary
}
```

With 150 virtual nodes per server:
- Distribution becomes **statistically even**
- A failed server's load spreads **across all remaining servers**, not just one neighbor
- Servers with more capacity get **more virtual nodes** (weighted distribution)

```
// Powerful server gets 3x the load
ring.AddVirtualNodes("Server-A-HighMem", vnodes: 450);
ring.AddVirtualNodes("Server-B-Normal", vnodes: 150);
```

---

## C# Implementation (Interview-Ready)

```csharp
public class ConsistentHashRing
{
    // Sorted so we can do clockwise lookup efficiently
    private readonly SortedDictionary<int, string> _ring = new();
    private readonly int _virtualNodes;
    private readonly HashAlgorithm _hasher = MD5.Create();

    public ConsistentHashRing(int virtualNodes = 150)
    {
        _virtualNodes = virtualNodes;
    }

    private int Hash(string key)
    {
        var bytes = _hasher.ComputeHash(Encoding.UTF8.GetBytes(key));
        return Math.Abs(BitConverter.ToInt32(bytes, 0));
    }

    public void AddServer(string server)
    {
        for (int i = 0; i < _virtualNodes; i++)
        {
            int pos = Hash($"{server}#vnode-{i}");
            _ring[pos] = server;
        }
    }

    public void RemoveServer(string server)
    {
        for (int i = 0; i < _virtualNodes; i++)
        {
            int pos = Hash($"{server}#vnode-{i}");
            _ring.Remove(pos);
        }
    }

    public string GetServer(string key)
    {
        if (_ring.Count == 0) throw new InvalidOperationException("No servers");

        int keyPos = Hash(key);

        // Find the first server at or after keyPos (clockwise)
        var entry = _ring.FirstOrDefault(kvp => kvp.Key >= keyPos);

        // Wrap around the ring if needed
        return entry.Value ?? _ring.First().Value;
    }
}
```

**Usage:**
```csharp
var ring = new ConsistentHashRing(virtualNodes: 150);
ring.AddServer("redis-1:6379");
ring.AddServer("redis-2:6379");
ring.AddServer("redis-3:6379");

string server = ring.GetServer("ticket:event:AXS-Lakers-2025");
// → Always routes to the same server for this key
```

## Naive Hashing vs Consistent Hashing

| Scenario                    | `hash(key) % N`        | Consistent Hashing |
| --------------------------- | ---------------------- | ------------------ |
| Add 1 server (10→11)        | ~91% keys remap        | ~9% keys remap     |
| Remove 1 server (10→9)      | ~90% keys remap        | ~10% keys remap    |
| Load distribution           | Even (if keys uniform) | Even with vnodes   |
| Weighted capacity           | Hard                   | Easy (more vnodes) |
| Cache invalidation on scale | Catastrophic           | Minimal            |

## AXS Interview Angle: Ticket Booking Cache

> *"How do you shard your Redis cache for event-seat locking across 10 nodes?"*

```
Problem:
- 50,000 events, each with up to 80,000 seats
- Soft locks stored in Redis (seat → userId, TTL 10 min)
- Need to scale Redis horizontally

Naive sharding → adding a Redis node during flash sale
  = 90% of seat locks remapped = locks appear GONE
  = double-booking risk under load

Consistent hashing → adding a Redis node
  = only ~10% of seat locks migrate
  = most locks remain valid
  = no phantom seat availability
```

**The key insight to say in the interview:**

> "We use consistent hashing so that scaling our Redis cluster during a flash sale doesn't invalidate existing soft locks. If a seat lock disappears due to rehashing, two users could both think a seat is available — causing double booking. Consistent hashing limits migration to ~1/N of keys when a node is added."

---

## Edge Cases to Mention

1. **Hot keys / celebrity events**: One event gets 1M req/s → all land on one node. Fix: prefix key with a random shard suffix + fan-out reads.

2. **Node failure**: If a Redis node dies, its virtual nodes are removed, load shifts smoothly to neighbors. But data is gone unless you have replication.

3. **Replication with consistent hashing**: Walk the ring clockwise to find the next 2 servers after primary → replicate there. That's how Cassandra handles replication factor.

4. **Hash function choice**: MD5 is fine for distribution. Avoid Java's default `hashCode()` — it's not consistent across JVM restarts.

---

## One-Line Summary for Interviews

> **Consistent hashing maps both servers and keys onto a circular ring so that adding/removing nodes only remaps ~K/N keys instead of all keys — critical for zero-disruption scaling of caches and databases.**
>

# Let's Build the Intuition from Scratch

## Forget computers for a second. Think about a Clock.

A clock face goes from 0 to 11 (or 0 to 359 degrees). It's a **circle**.

Now imagine a much bigger clock — one that goes from **0 to 1,000**.

That's your ring. Nothing more.

---

## Step 1: What is "Hashing" here?

A hash function is just a **black box that converts any string into a number**.

```
hash("Server-A")        →  150
hash("Server-B")        →  400
hash("Server-C")        →  750
hash("seat:row5:col3")  →  300
hash("seat:row1:col1")  →  600
hash("seat:vip:front")  →   50
```

You put in a name. You get out a number between 0 and 1000. That's it.
That number is the **position** on the clock.

---

## Step 2: Place Servers on the Clock

Take each server, hash its name, place it at that position on the clock face.

```
         0
         │
    ┌────┴────┐
    │         │
750 C         A 150
    │  clock  │
    └────┬────┘
         │
        400
         B
```

Servers A, B, C are now sitting at positions on the clock. **That's all "mapping a server to the ring" means** — computing a number for it and drawing it on the circle.

---

## Step 3: Place a Key on the Clock

Now you want to cache `"seat:row5:col3"`. Its hash is **300**.

Draw it on the clock too:

```
         0
         │
    ┌────┴────┐
    │         │
750 C         A 150
    │         │
    │   300   │
    └────B────┘
        400
```

The key lands at 300. Server B is at 400.

**Rule: Walk clockwise from the key until you hit a server. That's where the key lives.**

From 300, walking clockwise → first server you hit is **B at 400**.

So `"seat:row5:col3"` is stored on **Server B**. Done.

---

## Step 4: Try All Three Keys

```
"seat:vip:front"  → position  50  → walk clockwise → hit A at 150  → Server A
"seat:row5:col3"  → position 300  → walk clockwise → hit B at 400  → Server B
"seat:row1:col1"  → position 600  → walk clockwise → hit C at 750  → Server C
```

No math. No formulas. Just: **"where am I on the clock, and who's the next server clockwise?"**

---

## Step 5: NOW See Why It's Powerful

### Old way (`hash % N`): Add a server → disaster

You had 3 servers. Every key used `position % 3`.
Now you add a 4th server. Every key must use `position % 4`.
**Different number → different remainder → almost every key moves to a different server.**

It's like renumbering every seat in a stadium mid-show.

---

### Consistent hashing: Add a server → almost nothing changes

Add **Server D** at position **350**.

```
         0
         │
    ┌────┴────┐
    │         │
750 C         A 150
    │         │
    │  300 350│
    └────DB───┘
        400
```

Now re-ask: where does `"seat:row5:col3"` (position 300) go?

Walk clockwise from 300 → first server hit is **D at 350**. (Previously it was B.)

**Only keys between 150 and 350 moved.** Everything else? Completely untouched.

`"seat:vip:front"` (50) → still A.
`"seat:row1:col1"` (600) → still C.

Only the keys that were "between A and D" shifted. That's roughly **1/4 of keys**, not all of them.

---

## The Clock Analogy, Complete

| Clock concept                      | System design concept            |
| ---------------------------------- | -------------------------------- |
| Clock face (0–1000)                | The hash ring                    |
| Placing a hand at 3 o'clock        | Hashing a server name → position |
| Placing a sticky note at 2 o'clock | Hashing a key → position         |
| "Which hand is next clockwise?"    | "Which server owns this key?"    |
| Adding a new hand at 2:30          | Adding a new server              |
| Only notes between 2:00–2:30 shift | Only ~1/N keys remapped          |

---

## Why Does This Matter for AXS?

During a Lakers flash sale, your Redis cluster is under insane load. You need to add a new Redis node **right now**.

- **Old way**: Adding the node scrambles 90% of your seat locks. Users who locked Row 5 Seat 3 suddenly look unlocked. Two people buy the same seat. Chaos.
- **Consistent hashing**: Only ~10% of locks shift. The rest stay exactly where they were. The flash sale continues safely.

---

## Quick Intuition Check

> You have servers at positions 100, 400, 700 on a 0–1000 ring.
> A key hashes to position 650.
> Which server gets it?

Walk clockwise from 650 → next server is at **700**. ✅

> Now you remove that server (700). Where does the key go?

Walk clockwise from 650 → next server is now **100** (wrap around the clock). ✅

No formula. Just follow the clock.

That's the core question. But it appears in different forms depending on the interview level:

These are the ones worth knowing:

**2. Design a load balancer** — variations on LC 519, 528

---

## What They're Actually Testing

When they ask you to implement it, they want to see:

| Thing they watch for                                            | Why                                     |
| --------------------------------------------------------------- | --------------------------------------- |
| You use a `SortedDictionary` / sorted structure                 | Shows you understand clockwise lookup   |
| You use `BinarySearch` or `UpperBound` for lookup               | Shows O(log N) awareness, not O(N)      |
| You handle the **wraparound** (key > all servers → go to first) | Shows you truly understand the ring     |
| You add virtual nodes                                           | Shows production-level thinking         |
| You pick a good hash function                                   | Shows awareness of distribution quality |

---

## The Wraparound — Where Most People Fail

This is the single most common bug in interviews:

```csharp
public string GetServer(string key)
{
    int keyPos = Hash(key);

    // Find first server at or AFTER keyPos
    var entry = _ring.FirstOrDefault(kvp => kvp.Key >= keyPos);

    // ⚠️ What if keyPos = 980 and all servers are at 100, 400, 700?
    // FirstOrDefault returns null — but the answer is Server at 100 (wrap around!)

    return entry.Value ?? _ring.First().Value; // ← this handles the wraparound
}
```

If you handle wraparound cleanly in your code without being prompted, you stand out.

Yes — just this one problem, implemented cleanly from scratch:

1. First without virtual nodes (~15 lines)
2. Then add virtual nodes (~10 more lines)
3. Then make sure wraparound works with a test case

# Vertical vs Horizontal Scaling

## Start With a Real Feeling

Imagine you run a tea stall. One day a cricket match ends nearby and 500 people rush in.

**Vertical scaling** = you work faster. You brew tea at 2x speed, get a bigger stove, hire one more helper at the same counter.

**Horizontal scaling** = you open more counters. Same stall, but now 5 counters running in parallel, each serving a different queue.

Both handle the rush. But they fail differently and cost differently. That's the entire topic.

---

## Vertical Scaling (Scale Up)

### What it means
Make your **single machine more powerful**.

```
Before:          After:
┌─────────┐      ┌─────────────┐
│ 8 cores │  →   │  64 cores   │
│ 16 GB   │      │  512 GB RAM │
│ 1 TB    │      │  10 TB SSD  │
└─────────┘      └─────────────┘
```

### Real examples
- Upgrading your SQL Server from 16 GB RAM to 256 GB RAM
- Moving from a t2.medium to an r6g.16xlarge on AWS
- Adding more CPU to your monolith app server

### When it works well
- Your app is a **single-threaded or stateful monolith** (hard to distribute)
- You need **ACID transactions** — one powerful DB is simpler than distributed transactions
- **Short-term fix** during an unexpected traffic spike
- Legacy systems that can't be redesigned

### The hard ceiling problem

```
Traffic →  ████████████████████████  (keeps growing)
Biggest
machine →  ──────────────────────── (hard limit exists)
           You hit it. Game over.
```

There is a physical limit to how big one machine can get. And the cost curve is brutal — doubling performance often costs 5-10x more money.

### Failure mode
Single point of failure. **One machine goes down → everything goes down.** No redundancy.

---

## Horizontal Scaling (Scale Out)

### What it means
Add **more machines** of the same size, distribute the load across them.

```
Before:             After:
                    ┌──────────┐
                    │ Server 1 │
┌──────────┐        ├──────────┤
│ Server 1 │   →   │ Server 2 │  ← Load Balancer distributes
└──────────┘        ├──────────┤
                    │ Server 3 │
                    └──────────┘
```

### Real examples
- Adding more Redis nodes to your cache cluster
- Running 10 instances of your booking service behind a load balancer
- Sharding your database across multiple machines
- CDN edge servers — hundreds of machines globally serving static assets

### When it works well
- **Stateless services** — each request can go to any server
- You need **infinite theoretical scale** (just keep adding nodes)
- High availability — one node dies, others absorb the load
- **Read-heavy workloads** — add read replicas

### The new complexity you inherit

Horizontal scaling is not free. You now have to think about:

```
Problem 1: Session state
  User logs in on Server 1.
  Next request goes to Server 2.
  Server 2 doesn't know who they are. ← broken

Problem 2: Cache consistency
  Server 1 caches seat availability: "Row 5 available"
  Server 2 caches the same: "Row 5 available"
  Both allow booking. Double booking. ← catastrophic

Problem 3: Distributed transactions
  Debit happens on Node A.
  Credit on Node B.
  Node B crashes mid-transaction. ← money lost

Problem 4: Routing
  Which server handles which request?
  How do you route without overloading one? ← consistent hashing
```

Every one of these is a system design interview topic in its own right.

---

## Side-by-Side Comparison

| Dimension        | Vertical               | Horizontal                     |
| ---------------- | ---------------------- | ------------------------------ |
| How              | Bigger machine         | More machines                  |
| Cost curve       | Exponential (painful)  | Linear (predictable)           |
| Ceiling          | Hard physical limit    | Theoretically infinite         |
| Failure          | Total (SPOF)           | Partial (graceful degradation) |
| Complexity       | Simple                 | High (distributed systems)     |
| State management | Easy (one machine)     | Hard (shared state problem)    |
| Best for         | Databases, legacy apps | Stateless services, caches     |
| Example          | Bigger SQL Server      | Redis Cluster, microservices   |

---

## The Dirty Secret: You Always Do Both

No real system uses only one. The pattern is:

1. **Start vertical** — simplest, fastest to ship
2. **Hit the ceiling** — cost explodes or machine limit reached
3. **Go horizontal** — redesign for distribution
4. **Scale each horizontal node vertically too** — each node should be reasonably sized

```
         Load Balancer
              │
     ┌────────┼────────┐
     ▼        ▼        ▼
  [Node 1] [Node 2] [Node 3]   ← horizontal (3 nodes)
  32 cores  32 cores  32 cores  ← vertical (each node is powerful)
  128 GB    128 GB    128 GB
```

AWS calls this "right-sizing" — each node is vertically sized correctly, then you scale out horizontally.

---

## The Database Scaling Problem

This is where it gets hard — and where interviewers love to go deep.

**Stateless app servers** → horizontally scale easily. Just add more behind a load balancer.

**Databases** → painful to horizontally scale because of state.

The progression interviewers expect you to know:

```
Stage 1: One DB, vertical scaling
  → Works until ~10K req/s

Stage 2: Read replicas (horizontal reads)
  Primary handles writes.
  Replicas handle reads.
  → Works until write volume explodes

Stage 3: Sharding (horizontal writes)
  Split data across multiple DBs by key.
  User IDs 1–1M → DB1
  User IDs 1M–2M → DB2
  → Routing via consistent hashing
  → Cross-shard queries become painful

Stage 4: Distributed DB
  CockroachDB, Cassandra, DynamoDB
  → Built for horizontal scale natively
```

---

## AXS Interview Angle

> *"Your ticketing platform needs to handle a sudden 50x traffic spike when Taylor Swift tickets go on sale. How do you scale?"*

**The answer they want to hear:**

```
App layer (stateless):
  → Horizontal scaling — spin up 20 more booking service instances
    behind the load balancer. Each request is independent.
    Auto-scaling group on AWS triggers when CPU > 70%.

Session / soft locks (Redis):
  → Redis Cluster with consistent hashing.
    Add nodes under load without invalidating existing seat locks.

Database (stateful):
  → Vertical scale the primary write DB first (fastest to do).
    Add read replicas for seat availability queries (90% of traffic is reads).
    Writes (actual bookings) go to primary only.

Queue:
  → Decouple booking requests via SQS/Kafka.
    Users enter a virtual waiting room.
    Backend processes at a controlled rate.
    This protects the DB from a write stampede.
```

The interviewer wants to see you **distinguish which layer scales which way and why** — not just say "add more servers."

---

## Does Any Coding Question Come From This?

Directly — **almost never.** Scaling is an architectural decision, not an algorithm. You won't get "implement vertical scaling" as a coding problem.

But it shows up **indirectly** in three ways:

**1. As the motivation question before a design**
> "Why can't we just use one big server?" — you explain vertical scaling limits, then design the distributed solution.

**2. Load balancer implementation**
> "Implement a round-robin / least-connections load balancer" — this is the code-level manifestation of horizontal scaling.

```csharp
public class RoundRobinBalancer
{
    private readonly List<string> _servers;
    private int _current = 0;

    public string GetNext()
    {
        var server = _servers[_current % _servers.Count];
        Interlocked.Increment(ref _current); // thread-safe
        return server;
    }
}
```

**3. Consistent hashing** — which you already know — is the coding question that *enables* horizontal scaling of caches and databases.

---

## One-Line Summary for Interviews

> **Vertical scaling makes one machine stronger — simple but has a hard ceiling and a single point of failure. Horizontal scaling adds more machines — infinitely scalable but introduces distributed systems complexity. Stateless layers scale horizontally with ease; databases are where horizontal scaling gets hard and requires sharding, replication, and consistent routing.**



# Sharding — Deep Dive for System Design Interviews

## Start With a Real Feeling

You run a library. It has 10 million books. One librarian. One building.

Finding any book takes forever. Adding new books is getting impossible. The building is full.

**Option 1 (Vertical):** Build a taller building. Hire a stronger librarian. Has a ceiling.

**Option 2 (Sharding):** Open 10 libraries across the city. Books A–C go to Library 1. D–F go to Library 2. And so on.

Now each library is small, fast, and manageable. And you can open an 11th library anytime.

**That's sharding.** Split your data across multiple databases so each one handles a fraction of the load.

---

## Why One Database Eventually Breaks

Before sharding makes sense, you need to feel *why* a single DB fails.

```
Single DB under growing load:

Week 1:   ██░░░░░░░░  10% capacity  → fast, happy
Week 10:  █████░░░░░  50% capacity  → acceptable
Week 20:  ████████░░  80% capacity  → queries slowing
Week 30:  ██████████  100% capacity → timeouts, failures
Week 31:  ██████████🔥 → database catches fire
```

The symptoms before total failure:
- **Slow queries** — table scans on 500M rows take seconds
- **Lock contention** — writes block reads, reads block writes
- **Index size** — indexes no longer fit in RAM, spill to disk
- **Replication lag** — replica can't keep up with primary's write volume
- **Backup time** — taking 6 hours to back up one DB is dangerous

You've vertically scaled as far as you can. Read replicas helped with reads. But **writes are still all hitting one machine**. That's the wall sharding breaks through.

---

## What Sharding Actually Is

Sharding = **splitting rows of a table across multiple databases**, based on a rule.

```
Before sharding (1 DB, 9 rows):
┌──────────────────────────┐
│ users table              │
│ id=1, name=Alice         │
│ id=2, name=Bob           │
│ id=3, name=Carol         │
│ id=4, name=Dave          │
│ id=5, name=Eve           │
│ id=6, name=Frank         │
│ id=7, name=Grace         │
│ id=8, name=Heidi         │
│ id=9, name=Ivan          │
└──────────────────────────┘

After sharding (3 DBs, 3 rows each):
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  Shard 1     │  │  Shard 2     │  │  Shard 3     │
│ id=1, Alice  │  │ id=4, Dave   │  │ id=7, Grace  │
│ id=2, Bob    │  │ id=5, Eve    │  │ id=8, Heidi  │
│ id=3, Carol  │  │ id=6, Frank  │  │ id=9, Ivan   │
└──────────────┘  └──────────────┘  └──────────────┘
```

Same data. Same schema. Just split across machines. Each DB only knows about its own rows.

---

## The Shard Key — The Most Important Decision

The **shard key** is the column you use to decide which shard a row goes to.

Every other problem in sharding flows from this one choice. A bad shard key poisons the entire system.

### How routing works

```
Request: "Get user 5"
          ↓
    Routing Layer
    (shard_id = user_id % 3)
    (5 % 3 = 2 → Shard 2)
          ↓
       Shard 2
    → returns Eve
```

The routing layer — sometimes called a **shard router** or **application-level logic** — computes which shard owns the data, then queries only that shard.

---

## The 3 Sharding Strategies

### Strategy 1: Hash-Based Sharding

```
shard = hash(user_id) % num_shards
```

```
user_id=1  → hash=394  → 394 % 3 = 1 → Shard 1
user_id=2  → hash=856  → 856 % 3 = 2 → Shard 2
user_id=3  → hash=112  → 112 % 3 = 0 → Shard 0
```

**Good:** Even distribution. No hotspots.
**Bad:** Adding a shard remaps almost everything. (Sound familiar? → use consistent hashing to fix this.)

---

### Strategy 2: Range-Based Sharding

Split by value ranges.

```
user_id  1 – 1,000,000   → Shard 1
user_id  1M – 2,000,000  → Shard 2
user_id  2M – 3,000,000  → Shard 3
```

Or for AXS by date:
```
Events Jan–Mar 2025  → Shard 1
Events Apr–Jun 2025  → Shard 2
Events Jul–Sep 2025  → Shard 3
```

**Good:** Range queries are fast — all nearby data lives together.
**Bad:** Hotspots. If most users have IDs under 1M, Shard 1 is overwhelmed while others are empty.

---

### Strategy 3: Directory-Based Sharding

A lookup table explicitly maps keys to shards.

```
┌──────────────────────────┐
│   Shard Directory        │
│  user_id → shard_id      │
│  Alice   → Shard 2       │
│  Bob     → Shard 1       │
│  Carol   → Shard 3       │
└──────────────────────────┘
```

**Good:** Full flexibility. Can move individual users between shards.
**Bad:** The directory itself becomes a bottleneck and single point of failure.

## The Hotspot Problem — Where Sharding Goes Wrong

This is the trap interviewers love. You shard, but distribution is uneven.

### Example: Sharding by event_id at AXS

```
Shard 1: Taylor Swift concert    → 2,000,000 requests/hour
Shard 2: Local jazz show         → 200 requests/hour
Shard 3: Regional football game  → 5,000 requests/hour
```

Shard 1 is melting. Shards 2 and 3 are bored. You sharded but didn't actually solve the problem for the hot data.

### How to fix hotspots

**Fix 1: Add a random suffix to the shard key**
```
Instead of: shard = hash(event_id) % N
Use:        shard = hash(event_id + random(0,10)) % N
```
Spreads one hot event across 10 shards. But now reads must query all 10 and merge. Tradeoff.

**Fix 2: Dedicated shard for celebrity events**
Detect high-demand events ahead of time. Give them their own isolated shard with extra capacity.

**Fix 3: Cache aggressively in front of the hot shard**
Redis cache absorbs 95% of reads. Only cache misses hit the DB shard.

---

## Cross-Shard Queries — The Painful Part

Single-shard queries are fast. Cross-shard queries are where sharding hurts.

### Example: "Find all users who booked events in March"

```
// Before sharding — one query
SELECT * FROM bookings WHERE month = 'March';

// After sharding — you must ask every shard
Query Shard 1 → partial results
Query Shard 2 → partial results
Query Shard 3 → partial results
             ↓
      Merge in application
             ↓
      Return to user
```

This is called **scatter-gather**. It works but:
- Latency = slowest shard (not average)
- Network overhead multiplies
- Application logic gets complex

### How to reduce cross-shard queries

**Design your shard key around your most common query.**

If 90% of queries are "get all bookings for user X" → shard by `user_id`. All of a user's data lives on one shard. Most queries never cross shards.

---

## Resharding — When You Need More Shards

You started with 3 shards. Now you need 6. This is called **resharding** and it's painful.

```
Old:
Shard 1: user_id % 3 == 0   (users 3, 6, 9, 12...)
Shard 2: user_id % 3 == 1   (users 1, 4, 7, 10...)
Shard 3: user_id % 3 == 2   (users 2, 5, 8, 11...)

New (6 shards):
Shard 1: user_id % 6 == 0
Shard 2: user_id % 6 == 1
... etc
```

Almost every row needs to move. During migration:
- You can't take the DB offline (live system)
- Reads might hit the old location
- Writes might go to the new location
- You need **dual-write + backfill** strategies

This is why **consistent hashing matters for sharding** — it minimizes data movement when you add shards, just like it did for Redis nodes.

---

## Complete Picture: Sharding + Replication Together

Sharding alone doesn't give you fault tolerance. Each shard is still a single point of failure. Real systems combine both:

```
                    Load Balancer
                         │
               ┌─────────┴──────────┐
               ▼                    ▼
          Shard Router         Shard Router
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌────────┐   ← 3 shards (horizontal)
│Shard 1 │ │Shard 2 │ │Shard 3 │
│Primary │ │Primary │ │Primary │
└───┬────┘ └───┬────┘ └───┬────┘
    │          │           │
┌───▼────┐ ┌──▼─────┐ ┌───▼────┐   ← replicas per shard
│Replica │ │Replica │ │Replica │     (fault tolerance)
└────────┘ └────────┘ └────────┘
```

Each shard has its own primary + replica(s). You get both horizontal scale AND fault tolerance.

## AXS Interview Angle

> *"How would you shard the bookings database for AXS handling 100M bookings a year?"*

**The answer:**

```
Shard key choice: user_id

Reasoning:
  Most queries are "show my bookings" → scoped to one user.
  Sharding by user_id means 95% of queries hit exactly one shard.
  Cross-shard queries (e.g. "all bookings for event X") are rare
  and can be handled by a separate event-centric read model (CQRS).

Strategy: Consistent hash on user_id → 8 shards

Hotspot handling:
  Celebrity events don't affect this — bookings are spread by
  user_id, not event_id. User A and User B booking the same
  Taylor Swift concert land on different shards.

Resharding plan:
  Start with 8 shards but leave capacity.
  Use consistent hashing so adding shard 9 only moves ~11% of data.
  Backfill with dual-write during live migration.

Read scaling:
  Add one read replica per shard.
  Seat availability queries (read-heavy) go to replicas.
  Actual booking writes go to shard primaries only.
```

### Question 2: Scatter-gather query merger

> "You queried all 5 shards for bookings in March. Each returned a sorted list. Merge them into one sorted result."

This is **merge K sorted arrays** — a direct LeetCode problem (LC 23).

```csharp
// Each shard returns a sorted list of bookings by date
// Merge all into one sorted list efficiently

public List<Booking> MergeShardResults(List<List<Booking>> shardResults)
{
    // Min-heap ordered by booking date
    var heap = new PriorityQueue<(Booking booking, int shardIdx, int itemIdx), DateTime>();

    // Seed with first item from each shard
    for (int i = 0; i < shardResults.Count; i++)
    {
        if (shardResults[i].Count > 0)
        {
            var b = shardResults[i][0];
            heap.Enqueue((b, i, 0), b.Date);
        }
    }

    var result = new List<Booking>();

    while (heap.Count > 0)
    {
        var (booking, shardIdx, itemIdx) = heap.Dequeue();
        result.Add(booking);

        // Pull next item from the same shard
        int nextIdx = itemIdx + 1;
        if (nextIdx < shardResults[shardIdx].Count)
        {
            var next = shardResults[shardIdx][nextIdx];
            heap.Enqueue((next, shardIdx, nextIdx), next.Date);
        }
    }

    return result;
}
```

**This is a real interview question.** The consistent hashing question routes data *in*. The merge K sorted arrays question pulls data *out* across shards.

### Question 3: Detect a hotspot shard

> "Given a log of (user_id, timestamp) requests, find which shard is overloaded."

```csharp
public int FindHotShard(List<int> userIds, int numShards, double threshold)
{
    var shardCounts = new int[numShards];

    foreach (var userId in userIds)
        shardCounts[Math.Abs(userId % numShards)]++;

    double avg = userIds.Count / (double)numShards;

    for (int i = 0; i < numShards; i++)
        if (shardCounts[i] > avg * threshold) // e.g. 2x average = hotspot
            return i;

    return -1; // no hotspot
}
```

## One-Line Summary for Interviews

> **Sharding splits your database rows across multiple machines using a shard key, so each machine handles a fraction of the data and writes — the shard key choice determines everything: pick it wrong and you get hotspots and painful cross-shard queries; pick it right and most queries hit exactly one shard and you scale horizontally forever.**
