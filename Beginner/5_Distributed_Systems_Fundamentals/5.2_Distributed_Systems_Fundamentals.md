# CAP Theorem — Deep Dive for System Design Interviews

## Start With a Real Feeling

You run a bank with two branches — one in Mumbai, one in Delhi. Both branches share account data.

Priya walks into the Mumbai branch and deposits ₹10,000. One second later, Rahul walks into the Delhi branch asking "what's Priya's balance?"

The network cable connecting Mumbai and Delhi just snapped. What do you do?

**Option A:** Tell Rahul "Sorry, our system is temporarily unavailable. Try again later."
— Delhi refuses to answer until it can confirm the correct balance with Mumbai.

**Option B:** Tell Rahul the last known balance (₹0, before Priya's deposit).
— Delhi gives an answer, but it might be wrong.

**There is no Option C.** You cannot give Rahul an instantly correct answer AND stay available when the network is broken. You must choose one to sacrifice.

That forced choice — between correctness and availability when the network fails — is the CAP theorem.

---

## What CAP Theorem States

In any distributed system, you can only guarantee **two of three** properties simultaneously:

```
              Consistency
              (C)
              /\
             /  \
            /    \
           /      \
          /________\
    Availability    Partition Tolerance
    (A)             (P)
```

**C — Consistency:**
Every read receives the most recent write or an error. All nodes see the same data at the same time. No stale reads.

**A — Availability:**
Every request receives a response (not an error). The system keeps responding even if some nodes are down.

**P — Partition Tolerance:**
The system keeps operating even when network partitions occur (nodes can't communicate with each other).

---

## Why P Is Not Optional

This is the most important insight in CAP theorem — and what most people get wrong initially.

A **network partition** means some nodes cannot talk to other nodes. Messages are lost or delayed indefinitely.

```
Normal:                     During partition:
Node A ←────────→ Node B    Node A ✗────────✗ Node B
(communicate freely)        (cannot communicate)
```

In any real distributed system running across multiple machines — even within one data center — network partitions happen. Cables fail. Switches crash. Network congestion causes timeouts. AWS availability zones lose connectivity to each other.

**You cannot build a distributed system that is immune to network partitions.** The network is unreliable by nature. Partition Tolerance is not a design choice — it's a physical reality you must plan for.

So the real choice is:

```
CAP simplified:
  When a partition occurs, do you want:
  
  CP → Consistency + Partition Tolerance
       "I'll stop answering rather than risk giving wrong data"
  
  AP → Availability + Partition Tolerance  
       "I'll keep answering even if data might be stale"
```

CA (Consistency + Availability without Partition Tolerance) is theoretically possible only in a single-node system that never partitions — which isn't really "distributed."

---

## CP Systems — Consistency Over Availability

### The behavior

When a partition occurs, CP systems **refuse to answer** rather than risk returning stale data. They sacrifice availability to maintain correctness.

```
Normal operation:
  Client → Node A → Node B (replicates) → "Write confirmed"
  Client → Any node → Returns latest value ✅

During partition (Node A and B can't communicate):
  Client → Node A → "ERROR: Cannot guarantee consistency" ❌
  
  Node A refuses to serve reads/writes until partition heals.
  Better to return an error than return wrong data.
```

### When CP is the right choice

**Money and inventory — where wrong answers are catastrophic**

```
Banking transaction:
  "Debit $500 from account" 
  If two nodes process this simultaneously without coordination:
  Account could go to -$500 when only $300 was available.
  CP prevents this — refuse the transaction if nodes can't agree.

Seat booking:
  "Is seat A5 available?"
  Wrong answer = double booking = two people show up for one seat.
  CP refuses to answer rather than risk saying "yes" when it's already sold.

Distributed lock:
  "Is this resource locked?"
  Wrong answer = two processes enter a critical section simultaneously.
  CP is mandatory — correctness is the entire point of a lock.
```

### Real CP systems

**ZooKeeper:**
Used for distributed coordination, leader election, distributed locks. Uses Zab consensus protocol. During partition — minority partition nodes stop serving requests. Only the majority partition continues.

```
ZooKeeper cluster: 5 nodes
Partition splits into: [3 nodes] ✗ [2 nodes]

[3 nodes] — quorum met → continue serving ✅
[2 nodes] — no quorum → stop serving, return errors ❌

Correctness maintained: only one partition can have quorum.
No split-brain. No conflicting writes.
```

**HBase:**
Distributed database built on HDFS. Uses ZooKeeper for coordination. During partition — prefers consistency, may become unavailable.

**Traditional SQL databases (with synchronous replication):**
```
Primary write → must replicate to replica before acknowledging
If replica unreachable (partition) → write fails
Consistent? Yes. Available? No, during the partition.
```

**etcd:**
Key-value store used by Kubernetes. Uses Raft consensus. Minority partitions stop serving. Kubernetes control plane goes down but existing workloads keep running.

---

## AP Systems — Availability Over Consistency

### The behavior

When a partition occurs, AP systems **keep responding** even if responses might be stale. They sacrifice consistency to maintain availability.

```
Normal operation:
  Write → Node A → replicates to Node B → both nodes consistent

During partition (replication breaks):
  Write → Node A → cannot replicate to Node B
  Node A: updated data ✅
  Node B: stale data ⚠️ (but still serving requests)
  
  Client reads from Node B → gets stale data
  System is "available" but not "consistent"
  
When partition heals:
  Node A and B synchronize → eventually consistent ✅
```

### When AP is the right choice

**User-facing features where availability beats perfect accuracy**

```
Social media "likes" count:
  Post shows 10,432 likes on one server, 10,419 on another.
  13 like difference for 30 seconds?
  Nobody cares. Availability >> Consistency here.

Product availability display:
  "142 items in stock" vs "139 items in stock" across nodes.
  Close enough. Show the product. Take the order.
  Exact count enforced at checkout (CP layer).

DNS:
  You update your domain's IP address.
  Some users still hit old IP for 48 hours (TTL expiry).
  DNS is deliberately AP — availability of resolution >> consistency.
  Eventually consistent by design.

Shopping cart:
  Amazon famously uses AP for carts.
  Two devices add items simultaneously during partition?
  Both sets of items merge. Cart might have duplicates.
  Better than losing items or showing "service unavailable".
```

### Real AP systems

**Cassandra:**
Designed explicitly for AP. Uses "eventual consistency" with tunable consistency levels.

```
Cassandra consistency levels:
  ONE:    Write to 1 node, return success immediately
          → Fastest, least consistent
  QUORUM: Write to majority of nodes before success
          → Balanced
  ALL:    Write to ALL nodes before success  
          → Slowest, most consistent (but now it's CP-like)

During partition:
  Cassandra keeps serving even if some nodes are unreachable.
  Nodes that were partitioned sync when they reconnect.
  → Available, eventually consistent
```

**DynamoDB (default):**
Eventually consistent reads by default. Strongly consistent reads available (at higher cost and CP-like behavior).

**CouchDB:**
Built for offline-first applications. Devices sync when connected, work independently when not. Merge conflicts resolved by application logic.

**DNS:**
Classic AP system. Updates propagate eventually. Old data served during propagation. System never goes down just because one nameserver is unreachable.

## The Spectrum — Not Binary

Here's what the interview answer sounds like for most senior candidates:

> "CAP isn't actually a binary choice — it's a spectrum. Most modern systems are tunable. Cassandra lets you set consistency level per query. DynamoDB lets you choose eventually consistent vs strongly consistent reads. The real skill is identifying which operations in your system need CP and which can tolerate AP."

### Cassandra's tunable consistency

```
// Read user's shopping cart (AP is fine — stale data acceptable)
var cart = await cassandra.ExecuteAsync(
    new SimpleStatement("SELECT * FROM carts WHERE user_id = ?", userId)
        .SetConsistencyLevel(ConsistencyLevel.One)  // AP: fast, possibly stale
);

// Decrement inventory (CP required — wrong answer = oversell)
var result = await cassandra.ExecuteAsync(
    new SimpleStatement(
        "UPDATE inventory SET count = count - 1 WHERE product_id = ? IF count > 0",
        productId
    ).SetConsistencyLevel(ConsistencyLevel.Quorum)  // CP-like: must agree
);
```

Same database. Same cluster. Different consistency per operation based on what the business requires.

## PACELC — The Extension You Should Know

CAP only describes what happens during a partition. **PACELC** extends it to also describe the latency vs consistency tradeoff during normal operation.

```
PACELC:
  If Partition: choose between Availability and Consistency
  Else (normal operation): choose between Latency and Consistency
```

```
System         | Partition choice | Normal choice
─────────────────────────────────────────────────
Cassandra      | AP               | EL (low latency, eventual)
DynamoDB       | AP               | EL (low latency, eventual)
HBase          | CP               | EC (higher latency, consistent)
MySQL (async)  | CP               | EC (higher latency, consistent)
ZooKeeper      | CP               | EC (higher latency, consistent)
```

This is more useful in practice. Most of the time there are no partitions — but there's always a tradeoff between how fast you respond and how consistent the response is.

Mentioning PACELC in an interview without being prompted signals senior-level thinking.

## Consensus Algorithms — How CP Systems Work

CP systems need a way for nodes to **agree** on a value even when some nodes are unreachable. This is the **consensus problem**.

### Raft — the readable one

Raft is the consensus algorithm used by etcd, CockroachDB, and TiKV. It's designed to be understandable.

```
Roles:
  Leader:    One node handles all writes, coordinates replication
  Follower:  Passively receives replication from leader
  Candidate: Running for election to become leader

Normal operation:
  Client → Leader
  Leader writes to its log
  Leader replicates to Followers
  Majority of followers acknowledge
  Leader commits, responds to client

Leader fails (partition):
  Followers detect missing heartbeat
  Followers become candidates, start election
  Candidate requests votes from all nodes
  First to get majority votes becomes new leader
  
Partition: [Leader + 2 followers] ✗ [2 followers]
  Old partition: [2 followers] — no majority → stop serving
  New partition: [old leader + 2] — has majority → continue
  Only ONE partition can have majority → no split-brain
```

### Paxos — the original (harder to understand)

Paxos is the theoretical foundation. ZooKeeper's Zab protocol is based on Paxos. Rarely implemented from scratch — use Raft if you need consensus.

## AXS Interview Angle

> *"You're designing AXS's booking system. How do you apply CAP theorem to different parts of the system?"*

```
Seat inventory and booking (CP — consistency mandatory):
  Wrong answer = double booking = fraud, unhappy customers.
  Use PostgreSQL with synchronous replication.
  During partition: writes fail with error. Better than double booking.
  Users see "temporarily unavailable" — acceptable for booking flow.

Seat availability display (AP — staleness acceptable):
  The seat map showing red/green seats can be 1-2 seconds stale.
  Seeing a "available" seat that was just bought?
  → User tries to book it → gets error → tries another seat.
  Minor UX issue, not a data integrity issue.
  Use Cassandra or Redis with eventual consistency.
  During partition: keep showing (possibly stale) seat map.

User session data (AP — availability critical):
  User must be able to browse even if one session server is down.
  Slight session staleness is fine.
  Use Redis Cluster with AP behavior.

Payment processing (CP — money, no stale reads):
  Must confirm funds available before charging.
  Must not double-charge.
  Third-party payment processor (Stripe) handles this.
  Our integration: synchronous, at-most-once, CP behavior.

Analytics and dashboards (AP — approximate is fine):
  "How many tickets sold in the last hour?"
  Answer of 10,432 vs 10,419 is fine.
  Use Cassandra. During partition: approximate data served.
  Eventually reconciled when partition heals.

Distributed locks for seat locking (CP — correctness mandatory):
  "Did I get this lock?" must have a definitive answer.
  Use Redis with Redlock or ZooKeeper.
  During partition: lock acquisition fails. Seat not locked.
  Better than two users both thinking they have the lock.
```

### Question 1: Implement a simple consensus — leader election

> "Implement a leader election system where exactly one node becomes leader."

```csharp
public class LeaderElection
{
    private readonly IDatabase _redis;
    private readonly string _nodeId;
    private readonly string _lockKey = "leader:lock";
    private readonly TimeSpan _ttl = TimeSpan.FromSeconds(10);
    private bool _isLeader = false;

    public LeaderElection(IDatabase redis, string nodeId)
    {
        _redis = redis;
        _nodeId = nodeId;
    }

    // Try to become leader — atomic SET NX
    public async Task<bool> TryBecomeLeaderAsync()
    {
        // SET leader:lock <nodeId> NX EX 10
        // Only succeeds if key doesn't exist (no current leader)
        bool acquired = await _redis.StringSetAsync(
            _lockKey,
            _nodeId,
            _ttl,
            When.NotExists  // NX — atomic, no race condition
        );

        _isLeader = acquired;
        return acquired;
    }

    // Current leader must renew its lease before TTL expires
    // Prevents stale leaders from holding the lock after crashing
    public async Task<bool> RenewLeaseAsync()
    {
        if (!_isLeader) return false;

        // Only renew if WE are still the leader
        // Use Lua script for atomic check-and-set
        const string lua = @"
            if redis.call('get', KEYS[1]) == ARGV[1] then
                return redis.call('expire', KEYS[1], ARGV[2])
            else
                return 0
            end";

        var result = (long)await _redis.ScriptEvaluateAsync(
            lua,
            new RedisKey[] { _lockKey },
            new RedisValue[] { _nodeId, (long)_ttl.TotalSeconds }
        );

        if (result == 0) _isLeader = false; // lost leadership
        return result == 1;
    }

    // Leader heartbeat loop
    public async Task RunLeaderLoopAsync(CancellationToken cancellationToken)
    {
        while (!cancellationToken.IsCancellationRequested)
        {
            if (!_isLeader)
            {
                // Not leader — try to become one
                await TryBecomeLeaderAsync();
            }
            else
            {
                // Leader — renew lease before it expires
                bool stillLeader = await RenewLeaseAsync();
                if (stillLeader)
                    await DoLeaderWork(); // only leader does this work
            }

            // Check every 3 seconds (TTL is 10s — 3 heartbeats to recover)
            await Task.Delay(TimeSpan.FromSeconds(3), cancellationToken);
        }
    }

    private async Task DoLeaderWork()
    {
        Console.WriteLine($"Node {_nodeId} is leader — processing tasks");
        await Task.CompletedTask;
    }

    public bool IsLeader => _isLeader;
}
```

**What this demonstrates:** Understanding that CP requires consensus — only one node can be leader. The atomic SET NX prevents split-brain. TTL prevents a crashed leader from holding the lock forever. This is exactly how Redis-based leader election works in production.

### Question 2: Implement eventual consistency with conflict resolution

> "Two nodes accept writes during a partition. When they reconnect, resolve the conflict."

```csharp
public class EventuallyConsistentStore
{
    // Last-Write-Wins using vector clocks for conflict detection
    private readonly Dictionary<string, VersionedValue> _store = new();
    private readonly string _nodeId;

    public EventuallyConsistentStore(string nodeId)
    {
        _nodeId = nodeId;
    }

    public void Write(string key, string value)
    {
        var existing = _store.GetValueOrDefault(key);
        var newVersion = (existing?.Version ?? 0) + 1;

        _store[key] = new VersionedValue
        {
            Value = value,
            Version = newVersion,
            NodeId = _nodeId,
            Timestamp = DateTime.UtcNow
        };
    }

    public string? Read(string key)
        => _store.TryGetValue(key, out var v) ? v.Value : null;

    // Called when partition heals — merge state from another node
    public List<string> Merge(EventuallyConsistentStore other)
    {
        var conflicts = new List<string>();

        foreach (var (key, otherValue) in other._store)
        {
            if (!_store.TryGetValue(key, out var myValue))
            {
                // Key only exists on other node — take it
                _store[key] = otherValue;
                continue;
            }

            if (otherValue.Version > myValue.Version)
            {
                // Other node has newer version — take it (LWW)
                _store[key] = otherValue;
            }
            else if (otherValue.Version == myValue.Version
                     && otherValue.NodeId != myValue.NodeId)
            {
                // Same version, different nodes — TRUE CONFLICT
                conflicts.Add(key);

                // Strategy 1: Last-Write-Wins by timestamp
                if (otherValue.Timestamp > myValue.Timestamp)
                    _store[key] = otherValue;

                // Strategy 2: Could merge values (CRDT approach)
                // Strategy 3: Could surface to application for resolution
            }
            // else: my version is newer — keep mine
        }

        return conflicts; // report conflicts for application to handle
    }
}

public class VersionedValue
{
    public string Value { get; set; } = "";
    public long Version { get; set; }
    public string NodeId { get; set; } = "";
    public DateTime Timestamp { get; set; }
}

// Simulation: partition then merge
var nodeA = new EventuallyConsistentStore("A");
var nodeB = new EventuallyConsistentStore("B");

// Normal operation — both in sync
nodeA.Write("seat:A5", "available");
nodeB.Write("seat:A5", "available");

// --- PARTITION OCCURS ---

// Both nodes accept writes independently
nodeA.Write("seat:A5", "locked:user-42");  // User 42 locks on Node A
nodeB.Write("seat:A5", "locked:user-87");  // User 87 locks on Node B

// --- PARTITION HEALS ---

var conflicts = nodeA.Merge(nodeB);
// conflicts = ["seat:A5"] — both tried to lock same seat!
// Application must resolve: cancel one booking, notify user
Console.WriteLine($"Conflicts to resolve: {string.Join(", ", conflicts)}");
```

**What this demonstrates:** You understand why AP systems need conflict resolution strategies. Last-Write-Wins is the simplest. CRDTs (Conflict-free Replicated Data Types) are more sophisticated. This is exactly what happens in Cassandra during partition reconciliation.

### Question 3: Implement a quorum-based read/write system

> "Implement reads and writes that require majority agreement — the core of CP systems."

```csharp
public class QuorumStore
{
    private readonly List<NodeStore> _nodes;
    private readonly int _quorum;

    public QuorumStore(int nodeCount)
    {
        _nodes = Enumerable.Range(0, nodeCount)
            .Select(i => new NodeStore($"node-{i}"))
            .ToList();

        _quorum = (nodeCount / 2) + 1; // majority
    }

    // Write succeeds only if quorum of nodes acknowledge
    public async Task<bool> WriteAsync(string key, string value, long version)
    {
        var writeTasks = _nodes.Select(n => n.WriteAsync(key, value, version));
        var results = await Task.WhenAll(writeTasks);

        int successCount = results.Count(r => r);

        if (successCount >= _quorum)
        {
            Console.WriteLine($"Write quorum met ({successCount}/{_nodes.Count})");
            return true;
        }

        // Not enough nodes acknowledged — CP behavior: report failure
        Console.WriteLine($"Write failed — only {successCount}/{_nodes.Count} acknowledged");
        return false;
    }

    // Read from quorum, return most recent version seen
    public async Task<string?> ReadAsync(string key)
    {
        var readTasks = _nodes.Select(n => n.ReadAsync(key));
        var results = await Task.WhenAll(readTasks);

        var validResults = results.Where(r => r != null).ToList();

        if (validResults.Count < _quorum)
        {
            // Can't reach quorum — CP: return error rather than stale data
            throw new Exception("Cannot achieve read quorum — system unavailable");
        }

        // Return highest version among quorum responses
        return validResults
            .OrderByDescending(r => r!.Version)
            .First()!.Value;
    }
}

public class NodeStore
{
    private readonly Dictionary<string, (string value, long version)> _data = new();
    private bool _isAvailable = true; // simulate node failures

    public string NodeId { get; }

    public NodeStore(string nodeId) => NodeId = nodeId;

    public void SetAvailable(bool available) => _isAvailable = available;

    public Task<bool> WriteAsync(string key, string value, long version)
    {
        if (!_isAvailable) return Task.FromResult(false);
        _data[key] = (value, version);
        return Task.FromResult(true);
    }

    public Task<(string Value, long Version)?> ReadAsync(string key)
    {
        if (!_isAvailable) return Task.FromResult<(string, long)?>(null);
        return Task.FromResult(
            _data.TryGetValue(key, out var entry)
                ? ((string, long)?)(entry.value, entry.version)
                : null
        );
    }
}

// Demo: quorum behavior during partial failure
var store = new QuorumStore(nodeCount: 5); // quorum = 3

// Normal: all 5 nodes available
await store.WriteAsync("seat:A5", "available", version: 1);

// Simulate partition: 2 nodes go down
store._nodes[3].SetAvailable(false);
store._nodes[4].SetAvailable(false);

// Write still succeeds — 3/5 nodes (quorum met)
bool success = await store.WriteAsync("seat:A5", "locked:user-42", version: 2);
Console.WriteLine($"Write with 3/5 nodes: {success}"); // true

// Simulate worse partition: 3 nodes down (majority gone)
store._nodes[2].SetAvailable(false);

// Write fails — only 2/5 nodes (below quorum)
success = await store.WriteAsync("seat:A5", "booked", version: 3);
Console.WriteLine($"Write with 2/5 nodes: {success}"); // false — CP behavior
```

**What this demonstrates:** Quorum-based systems are CP — they refuse to operate when they can't reach a majority. This prevents split-brain (two minority partitions both accepting writes). The tradeoff: minority partitions are unavailable, which is exactly what CAP predicts.

---

## One-Line Summary for Interviews

> **CAP theorem states that distributed systems must choose between Consistency and Availability when network partitions occur — since partitions are inevitable in real distributed systems, the practical choice is CP (refuse requests rather than risk wrong answers, used for money, locks, and inventory) versus AP (keep responding with possibly stale data, used for caches, social feeds, and analytics) — and the senior-level insight is that this isn't binary: modern systems like Cassandra let you tune consistency per operation, different parts of the same system make different CAP choices, and PACELC extends CAP to also capture the latency-consistency tradeoff during normal operation.**


# Replication — Deep Dive for System Design Interviews

## Start With a Real Feeling

You're a newspaper editor with one printing press. Every morning, 10,000 people want the paper. One press can print 1,000 papers per hour. You can't serve everyone. And if the press breaks down — nobody gets a paper today.

**Solution:** Build 5 identical printing presses. Now you can print 5,000 papers per hour. And if one breaks down, the other four keep printing.

That's replication. **Keep identical copies of your data on multiple machines** so that:
1. More machines can serve more requests (scale reads)
2. If one machine dies, others take over (fault tolerance)

But now you have a new problem: when a story is updated, you need to update all 5 presses. What if press 3 gets the update but press 4 doesn't yet? Someone reads the old story from press 4. Someone reads the new story from press 3. Same newspaper, different content. That inconsistency is the core challenge of replication.

---

## Why Replication Exists — The Two Motivations

### Motivation 1: Fault Tolerance (High Availability)

```
Single node:
  Server dies → service down → everyone affected
  MTTR (mean time to recover) = 30 minutes
  Monthly downtime = potentially hours

With replication (3 nodes):
  Node 1 dies → Node 2 and 3 keep serving
  Automatic failover → downtime = seconds
  Monthly downtime = near zero
```

### Motivation 2: Read Scalability

```
Single node handling all reads:
  10,000 reads/second → node at 100% CPU → queries slow

With 5 read replicas:
  10,000 reads/second distributed across 6 nodes
  Each node handles ~1,667 reads/second → plenty of headroom
  Add more replicas = linear read scale
```

Notice: replication helps reads. It doesn't help write throughput — all writes still go to one place (the primary). For write scalability, you need sharding. This distinction is critical to state in interviews.

---

## Single-Leader Replication (Master-Replica)

This is the most common replication strategy. One node accepts writes. Others replicate from it.

### The architecture

```
                    Writes
                      │
                      ▼
              ┌───────────────┐
              │    Primary    │  ← only node accepting writes
              │   (Leader)    │
              └───────┬───────┘
                      │ replication stream
          ┌───────────┼───────────┐
          ▼           ▼           ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ Replica 1│ │ Replica 2│ │ Replica 3│  ← read-only
    └──────────┘ └──────────┘ └──────────┘
          │           │           │
          └───────────┴───────────┘
                    Reads
```

### How replication actually happens

The primary maintains a **replication log** (Write-Ahead Log or binlog). Every write is recorded here first. Replicas consume this log and apply the same operations to their own data.

```
Primary WAL:
  LSN 1001: INSERT INTO bookings VALUES (...)
  LSN 1002: UPDATE seats SET status='booked' WHERE id='A5'
  LSN 1003: UPDATE events SET seats_available=seats_available-1

Replica receives WAL stream:
  Applies LSN 1001 → replica has same booking
  Applies LSN 1002 → replica has seat marked booked
  Applies LSN 1003 → replica has decremented count

Replica is now identical to primary.
```

### Synchronous vs Asynchronous Replication

This is one of the most important distinctions in this topic.

**Synchronous Replication:**

```
Client                Primary              Replica
  │                      │                    │
  │── WRITE ────────────→│                    │
  │                      │── replicate ──────→│
  │                      │                    │ (applies write)
  │                      │←─ ACK ─────────────│
  │←─ SUCCESS ───────────│
  │   (only after replica confirmed)
```

Primary waits for replica to confirm before telling client "success."

```
Guarantee: If primary dies right after committing,
           replica has the data. No data loss.

Cost:      Every write waits for network round-trip to replica.
           Write latency = primary processing + network + replica processing
           If replica is slow or unreachable → primary writes block.
```

**Asynchronous Replication:**

```
Client                Primary              Replica
  │                      │                    │
  │── WRITE ────────────→│                    │
  │←─ SUCCESS ───────────│                    │
  │                      │── replicate ──────→│ (happens later)
  │                      │                    │ (applies write)
```

Primary commits and responds immediately. Replication happens in background.

```
Guarantee: Fast writes. Primary doesn't wait for replica.

Risk:      Primary crashes AFTER commit but BEFORE replica gets the write.
           That write is lost. Replica becomes new primary → data loss.

Replication lag: Replica might be seconds or minutes behind primary.
                 Reads from replica might return stale data.
```

**Semi-synchronous (PostgreSQL synchronous_standby_names):**

```
Hybrid approach:
  Wait for at least ONE replica to confirm (not all).
  
  Primary + 3 replicas:
    At least 1 replica must confirm → success
    Other 2 replicas catch up asynchronously
    
  Balance: One guaranteed copy immediately,
           without waiting for all replicas.
```

### The Replication Lag Problem

With async replication, replicas are always slightly behind:

```
T=0:  Primary processes booking for seat A5
T=0:  Primary responds "Booking Confirmed" to user
T=0+100ms: Replica receives and applies the booking

T=0+50ms: Another user reads seat A5 status from replica
           → Sees "available" (replica hasn't caught up)
           → Shows seat as available when it's actually booked
           → Race condition leads to double-booking attempt
```

**Solutions for replication lag:**

```
1. Read-your-writes consistency:
   After a write, route that user's reads to the primary
   for a short window (e.g., 60 seconds).
   
   if (user.lastWriteTime > DateTime.UtcNow.AddSeconds(-60))
       return primaryDb.Read(query);
   else
       return replicaDb.Read(query);

2. Monotonic reads:
   Each user always reads from the same replica.
   They might see stale data, but never go backwards in time.
   route = hash(userId) % replicaCount  // sticky routing

3. Strongly consistent reads:
   For critical reads, always hit primary.
   SELECT * FROM seats WHERE id='A5' /* FOR SHARE on primary */
   Slower but always correct.
```

### Failover — What Happens When Primary Dies

```
Normal:
  Primary ← all writes
  Replica 1, 2, 3 ← reads

Primary dies:
  Step 1: Detect failure (heartbeat timeout — typically 30s)
  Step 2: Elect new primary (usually most up-to-date replica)
  Step 3: Redirect all writes to new primary
  Step 4: Other replicas start replicating from new primary
  Step 5: Old primary recovers — rejoins as replica

Issues during failover:
  Write gap: 30s of writes may be lost (async replication)
  Split-brain: old primary recovers and both think they're primary
  Clients: must discover new primary address
```

**Split-brain prevention:**

```
STONITH: "Shoot The Other Node In The Head"
  When a new primary is elected, it sends a STONITH command
  to the old primary (via out-of-band management interface)
  forcing it to power off.
  Drastic but prevents two primaries accepting writes simultaneously.

Fencing:
  New primary invalidates old primary's lease/token.
  Old primary's writes are rejected by quorum.
```

## Multi-Leader Replication (Multi-Master)

Multiple nodes accept writes simultaneously. Each is both a leader and a follower to the others.

### The architecture

```
       Region: US                    Region: EU
  ┌──────────────────┐         ┌──────────────────┐
  │   Leader US      │←───────→│   Leader EU      │
  │ (accepts writes) │ sync    │ (accepts writes) │
  └────────┬─────────┘         └────────┬─────────┘
           │                            │
    ┌──────┴──────┐              ┌──────┴──────┐
    │  Replica 1  │              │  Replica 3  │
    │  Replica 2  │              │  Replica 4  │
    └─────────────┘              └─────────────┘
```

### When multi-leader makes sense

```
Multi-datacenter setup:
  Users in US write to US leader (low latency, ~5ms)
  Users in EU write to EU leader (low latency, ~5ms)
  Both leaders sync with each other (high latency, ~100ms, acceptable)
  
  vs single-leader:
  EU users write to US leader → 100ms write latency on every write
  → terrible UX

Offline-capable apps (Google Docs, Notion):
  Each device is a leader while offline
  Sync when reconnected
  Conflicts resolved by CRDT or last-write-wins

Collaborative editing:
  Multiple users editing same document simultaneously
  Each edit accepted locally, merged globally
```

### The write conflict problem — multi-leader's core challenge

```
US Leader: User A changes seat price from $250 to $300
EU Leader: User B changes same seat price from $250 to $200

Both are valid writes on their respective leaders.
When leaders sync: CONFLICT.
Which price is correct? $300 or $200?
```

**Conflict resolution strategies:**

**1. Last Write Wins (LWW):**
```csharp
// Each write carries a timestamp
// Higher timestamp wins on conflict
if (remoteWrite.Timestamp > localWrite.Timestamp)
    store[key] = remoteWrite.Value;

// Problem: clock skew between servers
// Server A clock is 100ms ahead of Server B
// B's "later" write appears earlier by timestamp
// LWW can lose data silently
```

**2. Application-level resolution:**
```csharp
public string ResolveConflict(Write local, Write remote, string conflictType)
{
    return conflictType switch
    {
        "price"   => Math.Min(local.Value, remote.Value).ToString(), // use lower price
        "count"   => (local.Delta + remote.Delta).ToString(),        // sum deltas
        "status"  => MergeStatuses(local.Value, remote.Value),       // domain logic
        _         => remote.Timestamp > local.Timestamp              // LWW fallback
                         ? remote.Value : local.Value
    };
}
```

**3. CRDTs (Conflict-free Replicated Data Types):**
```
Special data structures designed so conflicts resolve automatically
without coordination.

Examples:
  G-Counter: only increments, merge = take max of each node's count
  2P-Set: add-only and remove-only sets, merged deterministically
  LWW-Register: last-write-wins register with timestamp
  
AXS use case:
  Seat view count: G-Counter CRDT
  Each node increments its own counter
  Merged view count = sum of all node counts
  No conflicts possible — mathematically guaranteed
```

## Leaderless Replication (Dynamo-Style)

No designated leader. Any node can accept writes. Reads and writes go to multiple nodes simultaneously.

### The architecture

```
                Client
              /   |   \
             /    |    \
            ▼     ▼     ▼
         Node1  Node2  Node3   ← all equal, all accept reads/writes
            \    |    /
             \   |   /
              ▼  ▼  ▼
              Quorum (W + R > N)
```

### Quorum reads and writes

```
N = total replicas (e.g., 3)
W = nodes that must confirm write (e.g., 2)
R = nodes that must respond to read (e.g., 2)

Rule: W + R > N guarantees at least one node in read set
      has the latest write.

With N=3, W=2, R=2:
  Write to 2/3 nodes ✓
  Read from 2/3 nodes ✓
  At least 1 node overlap → always see latest write

Tunable consistency:
  Fast, weak consistency:  W=1, R=1 (AP behavior)
  Strong consistency:      W=3, R=3 (CP behavior, all nodes must agree)
  Balanced:                W=2, R=2 (default Cassandra)
```

### Read repair and anti-entropy

Even with quorum, nodes can drift out of sync. Two mechanisms fix this:

**Read repair:**
```
Client reads from 3 nodes (quorum = 2):
  Node 1 returns: {value: "booked", version: 5}
  Node 2 returns: {value: "available", version: 3}  ← stale!
  Node 3 returns: {value: "booked", version: 5}

Client takes highest version → returns "booked"

Read repair: client (or coordinator) writes version 5 back to Node 2
→ Node 2 is now up to date
→ Next read from Node 2 returns correct data
```

**Anti-entropy process:**
```
Background process runs constantly:
  Compares data between nodes using Merkle trees
  Identifies diverged ranges
  Syncs diverged data

Merkle tree:
  Hash each row → hash ranges → hash of hashes (tree)
  Compare root hashes → if equal, nodes are in sync
  If different → traverse tree to find diverged ranges
  Only sync the diverged portions (not entire dataset)
```

### Question 1: Implement a primary-replica routing layer *(most direct)*

> "Build a database connection router that sends writes to primary and reads to replicas with health checking."

```csharp
public class ReplicaAwareDbRouter
{
    private readonly string _primaryConnectionString;
    private readonly List<ReplicaNode> _replicas;
    private readonly TimeSpan _maxAcceptableLag;
    private int _roundRobinIndex = 0;

    public ReplicaAwareDbRouter(
        string primaryCs,
        List<string> replicaConnectionStrings,
        TimeSpan maxAcceptableLag)
    {
        _primaryConnectionString = primaryCs;
        _maxAcceptableLag = maxAcceptableLag;
        _replicas = replicaConnectionStrings
            .Select(cs => new ReplicaNode(cs))
            .ToList();

        // Start health checking in background
        _ = Task.Run(MonitorReplicaHealthAsync);
    }

    // All writes go to primary
    public IDbConnection GetWriteConnection()
        => new NpgsqlConnection(_primaryConnectionString);

    // Reads go to healthy, fresh replicas
    public IDbConnection GetReadConnection(bool requireStrongConsistency = false)
    {
        if (requireStrongConsistency)
            return GetWriteConnection(); // primary for strong consistency

        var healthyReplicas = _replicas
            .Where(r => r.IsHealthy && r.ReplicationLag <= _maxAcceptableLag)
            .ToList();

        if (healthyReplicas.Count == 0)
        {
            // All replicas unhealthy or too stale → fall back to primary
            return GetWriteConnection();
        }

        // Round-robin across healthy replicas
        int idx = Math.Abs(Interlocked.Increment(ref _roundRobinIndex));
        var replica = healthyReplicas[idx % healthyReplicas.Count];
        return new NpgsqlConnection(replica.ConnectionString);
    }

    // Read-your-writes: after a write, use primary for this user's reads
    public IDbConnection GetReadConnectionForUser(
        string userId,
        DateTime? lastWriteTime)
    {
        if (lastWriteTime.HasValue &&
            DateTime.UtcNow - lastWriteTime.Value < TimeSpan.FromSeconds(5))
        {
            // User just wrote — use primary to avoid reading stale data
            return GetWriteConnection();
        }

        return GetReadConnection();
    }

    private async Task MonitorReplicaHealthAsync()
    {
        while (true)
        {
            foreach (var replica in _replicas)
            {
                try
                {
                    using var conn = new NpgsqlConnection(replica.ConnectionString);
                    await conn.OpenAsync();

                    // Check replication lag via PostgreSQL system view
                    using var cmd = new NpgsqlCommand(
                        @"SELECT EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp()))
                          AS lag_seconds",
                        conn
                    );

                    var lagSeconds = (double)(await cmd.ExecuteScalarAsync() ?? 999);
                    replica.ReplicationLag = TimeSpan.FromSeconds(lagSeconds);
                    replica.IsHealthy = true;
                }
                catch
                {
                    replica.IsHealthy = false;
                    replica.ReplicationLag = TimeSpan.MaxValue;
                }
            }

            await Task.Delay(TimeSpan.FromSeconds(5));
        }
    }
}

public class ReplicaNode
{
    public string ConnectionString { get; }
    public bool IsHealthy { get; set; } = true;
    public TimeSpan ReplicationLag { get; set; } = TimeSpan.Zero;

    public ReplicaNode(string connectionString)
    {
        ConnectionString = connectionString;
    }
}
```

### Question 2: Implement conflict resolution for multi-leader

> "Two datacenters both accepted a write to the same seat. Resolve the conflict."

```csharp
public class ConflictResolver
{
    // Vector clock: tracks version per node
    // {nodeId → version} — detects causality
    public ResolvedWrite Resolve(Write local, Write remote)
    {
        var comparison = CompareVectorClocks(
            local.VectorClock,
            remote.VectorClock
        );

        return comparison switch
        {
            VectorClockComparison.LocalNewer =>
                new ResolvedWrite(local.Value, local.VectorClock),

            VectorClockComparison.RemoteNewer =>
                new ResolvedWrite(remote.Value, remote.VectorClock),

            VectorClockComparison.Concurrent =>
                // True conflict — happened simultaneously, no causal order
                ResolveConcurrentConflict(local, remote),

            _ => new ResolvedWrite(local.Value, local.VectorClock)
        };
    }

    private ResolvedWrite ResolveConcurrentConflict(Write local, Write remote)
    {
        // Merge vector clocks (take max of each component)
        var mergedClock = MergeVectorClocks(local.VectorClock, remote.VectorClock);

        // Domain-specific resolution
        if (local.Field == "seat_status")
        {
            // For seat status: "booked" beats "available" (stricter wins)
            string winner = (local.Value == "booked" || remote.Value == "booked")
                ? "booked"
                : "available";
            return new ResolvedWrite(winner, mergedClock);
        }

        if (local.Field == "price")
        {
            // For price: take lower (customer-friendly, auditable)
            decimal localPrice = decimal.Parse(local.Value);
            decimal remotePrice = decimal.Parse(remote.Value);
            return new ResolvedWrite(
                Math.Min(localPrice, remotePrice).ToString(),
                mergedClock
            );
        }

        // Default: last-write-wins by wall clock
        return local.Timestamp >= remote.Timestamp
            ? new ResolvedWrite(local.Value, mergedClock)
            : new ResolvedWrite(remote.Value, mergedClock);
    }

    private VectorClockComparison CompareVectorClocks(
        Dictionary<string, long> local,
        Dictionary<string, long> remote)
    {
        bool localHasNewer = false;
        bool remoteHasNewer = false;

        var allNodes = local.Keys.Union(remote.Keys);

        foreach (var node in allNodes)
        {
            long localVersion = local.GetValueOrDefault(node, 0);
            long remoteVersion = remote.GetValueOrDefault(node, 0);

            if (localVersion > remoteVersion) localHasNewer = true;
            if (remoteVersion > localVersion) remoteHasNewer = true;
        }

        if (localHasNewer && !remoteHasNewer) return VectorClockComparison.LocalNewer;
        if (remoteHasNewer && !localHasNewer) return VectorClockComparison.RemoteNewer;
        if (localHasNewer && remoteHasNewer)  return VectorClockComparison.Concurrent;
        return VectorClockComparison.Equal;
    }

    private Dictionary<string, long> MergeVectorClocks(
        Dictionary<string, long> a,
        Dictionary<string, long> b)
    {
        var merged = new Dictionary<string, long>(a);
        foreach (var (node, version) in b)
            merged[node] = Math.Max(merged.GetValueOrDefault(node, 0), version);
        return merged;
    }
}

public enum VectorClockComparison { LocalNewer, RemoteNewer, Concurrent, Equal }

public record Write(
    string Field,
    string Value,
    DateTime Timestamp,
    Dictionary<string, long> VectorClock
);

public record ResolvedWrite(string Value, Dictionary<string, long> VectorClock);
```

### Question 3: Implement anti-entropy with Merkle trees *(advanced)*

> "Efficiently detect which data diverged between two replicas without comparing every row."

```csharp
public class MerkleTree
{
    private readonly Dictionary<string, string> _data;
    private readonly int _buckets;

    // Divide keyspace into buckets, hash each bucket's contents
    public MerkleTree(Dictionary<string, string> data, int buckets = 16)
    {
        _data = data;
        _buckets = buckets;
    }

    // Build tree: hash of hashes
    public string[] GetLeafHashes()
    {
        var leaves = new string[_buckets];

        for (int i = 0; i < _buckets; i++)
        {
            // Which keys fall in this bucket?
            var bucketData = _data
                .Where(kvp => GetBucket(kvp.Key) == i)
                .OrderBy(kvp => kvp.Key)
                .Select(kvp => $"{kvp.Key}:{kvp.Value}");

            leaves[i] = ComputeHash(string.Join(",", bucketData));
        }

        return leaves;
    }

    public string GetRootHash()
    {
        var leaves = GetLeafHashes();
        return ComputeHash(string.Join(",", leaves));
    }

    // Find which buckets differ between this node and another
    public List<int> FindDivergentBuckets(string[] otherLeafHashes)
    {
        var myLeaves = GetLeafHashes();
        var divergent = new List<int>();

        for (int i = 0; i < _buckets; i++)
            if (myLeaves[i] != otherLeafHashes[i])
                divergent.Add(i);

        return divergent;
    }

    // Get all keys in a specific bucket (to sync that bucket)
    public Dictionary<string, string> GetBucketData(int bucket)
        => _data.Where(kvp => GetBucket(kvp.Key) == bucket)
                .ToDictionary(kvp => kvp.Key, kvp => kvp.Value);

    private int GetBucket(string key)
    {
        using var sha = SHA256.Create();
        var hash = sha.ComputeHash(Encoding.UTF8.GetBytes(key));
        return Math.Abs(BitConverter.ToInt32(hash, 0)) % _buckets;
    }

    private string ComputeHash(string input)
    {
        using var sha = SHA256.Create();
        var bytes = sha.ComputeHash(Encoding.UTF8.GetBytes(input));
        return Convert.ToHexString(bytes);
    }
}

public class AntiEntropySync
{
    // Sync Node A with Node B — only transfer divergent data
    public async Task<SyncResult> SyncAsync(
        MerkleTree nodeA,
        MerkleTree nodeB,
        Dictionary<string, string> nodeAData,
        Dictionary<string, string> nodeBData)
    {
        // Step 1: Compare root hashes — if equal, nothing to do
        if (nodeA.GetRootHash() == nodeB.GetRootHash())
            return new SyncResult(0, 0);

        // Step 2: Find which buckets differ (O(buckets) not O(rows))
        var divergentBuckets = nodeA.FindDivergentBuckets(nodeB.GetLeafHashes());

        int rowsSynced = 0;
        int conflicts = 0;

        // Step 3: For each divergent bucket, sync only those rows
        foreach (int bucket in divergentBuckets)
        {
            var aBucketData = nodeA.GetBucketData(bucket);
            var bBucketData = nodeB.GetBucketData(bucket);

            // Find rows that differ
            foreach (var (key, bValue) in bBucketData)
            {
                if (!aBucketData.TryGetValue(key, out var aValue))
                {
                    // Key only on B — copy to A
                    nodeAData[key] = bValue;
                    rowsSynced++;
                }
                else if (aValue != bValue)
                {
                    // Both have key but different value — conflict
                    conflicts++;
                    // Resolve: last-write-wins or application logic
                    nodeAData[key] = bValue; // simplified: B wins
                    rowsSynced++;
                }
            }
        }

        return new SyncResult(rowsSynced, conflicts);
    }
}

public record SyncResult(int RowsSynced, int Conflicts);
```

**Why Merkle trees matter for interviews:** Without them, checking if two replicas are in sync requires comparing every single row — O(N). Merkle trees reduce this to O(divergent buckets) — typically O(1) when replicas are nearly in sync. Cassandra, Bitcoin, and Git all use Merkle trees for exactly this reason.

## One-Line Summary for Interviews

> **Replication keeps identical copies of data on multiple machines for two reasons — fault tolerance so the system survives node failures, and read scalability so multiple machines serve read traffic — single-leader is the most common pattern where one primary accepts all writes and replicas follow its WAL log, with synchronous replication trading write latency for zero data loss and asynchronous trading durability for speed; multi-leader allows writes from multiple locations but requires conflict resolution; leaderless uses quorum writes and reads to guarantee consistency without a single point of failure; and replication lag — the delay between primary writes and replica visibility — is the source of subtle consistency bugs that require read-your-writes, monotonic reads, or routing critical reads to the primary.**

# Eventual Consistency vs Strong Consistency — Deep Dive for System Design Interviews

## Start With a Real Feeling

You and a friend are working on a shared Google Doc from different cities.

**Strong consistency** is like a whiteboard in a meeting room. When you write something, everyone in the room sees it immediately. No delay. Everyone always sees the same thing at the same moment.

**Eventual consistency** is like editing a shared document where changes sync over the network. You type something. Your friend's screen updates a second later. During that one second, you and your friend see different versions of the document. But given enough time with no new edits — everyone converges to the same content.

That gap — between "write happened" and "everyone sees it" — is the core of this topic. Sometimes that gap is unacceptable. Sometimes it's perfectly fine. Knowing which is which is the senior engineering judgment interviewers are testing.

---

## The Root Problem — Distributed Systems Have No Shared Clock

On a single machine, reads always see the latest write. There's one copy of data, one clock, one order of operations. Easy.

On multiple machines across a network:

```
User writes "seat A5 = booked" → Node A (New York)
                                     │
                              network hop (50ms)
                                     │
                                     ▼
                              Node B (London)
                              (still shows "seat A5 = available")

During those 50ms:
  Any read from Node A → "booked" ✅
  Any read from Node B → "available" ❌ (stale)
```

You cannot make data appear on multiple machines simultaneously. Physics prevents it. The question is: how do you handle this unavoidable delay?

**Strong consistency:** Make readers wait until all nodes agree.
**Eventual consistency:** Let readers get whatever the local node has. Trust that nodes will sync up eventually.

---

## Strong Consistency — Everyone Agrees Before Answering

### The guarantee

After a write completes, every subsequent read from any node returns that write's value. No stale reads. Ever. The system behaves as if there's only one copy of the data.

```
Timeline:
T=0: Write "seat A5 = booked" → system accepts it
T=1: Read from Node A → "booked" ✅
T=1: Read from Node B → "booked" ✅
T=1: Read from Node C → "booked" ✅

All reads see the write immediately. Regardless of which node.
```

### How it's achieved — the cost

Strong consistency requires **coordination** before responding. Before any node answers a read or accepts a write, a majority of nodes must agree on the current state.

**Option 1: Synchronous replication**

```
Client → write "seat A5 = booked" → Primary Node
Primary Node → replicate to Node B → wait for ACK
Primary Node → replicate to Node C → wait for ACK
All nodes confirmed → Primary responds "success" to client

Now ANY node can answer correctly.
Cost: write latency = primary processing + network roundtrip to all replicas
```

**Option 2: Consensus protocol (Raft/Paxos)**

```
Client → write → Leader
Leader → propose to all followers
Majority of followers ACK
Leader commits, responds to client

Any follower can serve reads after commit.
Cost: 2 network roundtrips minimum before write completes.
```

**Option 3: Read from primary only**

```
All writes → primary
All reads → primary (never read from replicas)

Trivially consistent — one node is the source of truth.
Cost: primary is a bottleneck. No read scaling. Single point of failure.
```

### When strong consistency is non-negotiable

```
Financial transactions:
  Account balance must be correct.
  "Is there $500 in this account?" cannot have a stale answer.
  Strong consistency required.

Seat/ticket booking:
  "Is seat A5 available?" must be accurate.
  Stale "yes" → double booking → two people at one seat.
  Strong consistency required.

Distributed locks:
  "Does service A hold the lock?" must be definitive.
  Stale answer → two services in critical section simultaneously.
  Strong consistency mandatory.

Inventory management:
  "Do we have 1 item in stock?" must be accurate.
  Stale "yes" → overselling → angry customers.
  Strong consistency required.

Leader election:
  "Who is the current leader?" must have one answer.
  Two nodes thinking they're both leader → split-brain.
  Strong consistency mandatory.
```

### The cost you pay

```
Latency:
  Every write must wait for network roundtrips.
  Adding nodes → more network hops → higher write latency.
  Geographic distribution → 100ms+ cross-region roundtrips on every write.

Availability:
  If coordination fails (partition), writes block.
  CP systems prefer errors over wrong answers.
  "Service unavailable" during network issues.

Throughput:
  Cannot process next write until current one is globally committed.
  Sequential bottleneck on write path.
  Horizontal write scaling requires sharding, not just replication.
```

## Eventual Consistency — Fast Now, Correct Later

### The guarantee

If no new writes occur, all nodes will eventually converge to the same value. During the convergence window — which can be milliseconds or minutes — different nodes might return different values.

```
Timeline:
T=0:   Write "seat A5 = booked" → Node A accepts it
T=0:   Node A → "booked" ✅
T=0:   Node B → "available" ⚠️ (hasn't received update yet)
T=0:   Node C → "available" ⚠️ (hasn't received update yet)

T=50ms: Node B receives replication → updates to "booked"
T=50ms: Node B → "booked" ✅
T=50ms: Node C → "available" ⚠️ (still stale)

T=80ms: Node C receives replication → updates to "booked"
T=80ms: Node C → "booked" ✅

All nodes converged. System is now consistent.
The "eventual" happened in 80ms.
```

### Why it exists — the tradeoffs gained

```
Latency:
  Write to one node → return immediately.
  No waiting for remote ACKs.
  Replication happens asynchronously in background.
  Write latency: 1-2ms regardless of cluster size.

Availability:
  Each node serves requests independently.
  Network partition → nodes keep serving local data.
  AP behavior: always available, possibly stale.

Geographic scale:
  Write to US datacenter → instantly confirmed.
  European replica updates in background (100ms later).
  No European user waits for cross-Atlantic roundtrip on writes.

Throughput:
  Writes don't serialize on consensus.
  Each node accepts writes at its own pace.
  Horizontal write scaling possible (with conflict resolution).
```

### When eventual consistency is acceptable

```
Social media feeds:
  Post published → appears on followers' feeds within seconds.
  Seeing a post 2 seconds late? Nobody cares.
  Seeing a slightly stale like count? Totally fine.

Product catalog:
  Price updated in US datacenter.
  European users see old price for 100ms.
  Acceptable — not going to cause data corruption.

DNS:
  Record updated → propagates over hours.
  Deliberately eventually consistent — highly available lookup
  matters more than instant propagation.

Analytics and metrics:
  "How many users are active right now?"
  Answer of 10,432 vs 10,419 across replicas is fine.
  Approximate real-time is good enough.

User profile / preferences:
  User changes their notification settings.
  One server has old setting for 200ms.
  Email sent to wrong preference for one notification?
  Annoying, not catastrophic.

Shopping recommendations:
  "You might also like..." based on recent purchases.
  Slightly stale recommendations? User won't notice.
  Low stakes. AP behavior correct choice.

Content delivery (CDN):
  Static assets cached at edge nodes.
  Asset updated → propagates to edges over minutes.
  Eventual consistency by design. Highly available.
```

### Eventual consistency failure modes — when it goes wrong

```
Scenario 1: User reads own write
  User updates profile photo.
  Reads profile from a different replica.
  Sees old photo. Thinks update failed. Updates again.
  Now has duplicate event in database.

Scenario 2: Lost update
  User A reads balance: $500 (from Node A)
  User B reads balance: $500 (from Node B, still stale)
  User A withdraws $300 → writes $200 to Node A
  User B withdraws $300 → writes $200 to Node B
  Both writes replicate → conflict
  Last-write-wins: one $300 withdrawal is silently lost
  Bank loses $300.

Scenario 3: Monotonicity violation
  User sees message thread: [msg1, msg2, msg3]
  Refreshes → different replica → sees: [msg1, msg3]
  msg2 temporarily "disappeared" (replica behind primary)
  User is confused — data appears to go backwards.
```

## The Spectrum — Not Binary

This is the senior-level insight. Consistency is not a binary choice. It's a dial.

```
Weak ◄─────────────────────────────────────────────► Strong
  │                                                      │
  │    Eventual    Read-your-    Monotonic    Linearizable│
  │   Consistency    writes       reads      Serializble  │
  │                                                       │
 Fast                                                   Slow
 Available                                          Consistent
```

### The consistency models on the spectrum

**Eventual consistency (weakest):**
All nodes converge eventually. No guarantees about when or what intermediate reads see. Maximum availability and performance.

**Read-your-writes consistency:**
After you write something, your subsequent reads always see that write. Other users might see stale data, but YOU never see stale data for your own writes.

```csharp
// Implementation: session affinity
public string Read(string userId, string key)
{
    var lastWriteTime = _userWriteTimes.GetValueOrDefault(userId);

    if (lastWriteTime.HasValue &&
        DateTime.UtcNow - lastWriteTime.Value < TimeSpan.FromSeconds(5))
    {
        // User recently wrote — read from primary (see own write)
        return _primary.Get(key);
    }

    return _replica.Get(key); // stale reads OK for other users
}
```

**Monotonic reads:**
If you've seen a value at version V, you'll never see a value older than V. Reads don't go backwards in time. No "message disappeared" scenarios.

```csharp
// Implementation: track highest version seen per user
public (string value, long version) Read(string userId, string key)
{
    var minVersion = _userSeenVersions.GetValueOrDefault(userId, 0);

    // Find a replica with version >= what user has already seen
    var freshReplica = _replicas.FirstOrDefault(r =>
        r.GetVersion(key) >= minVersion
    );

    if (freshReplica == null)
        return _primary.Read(key); // fall back to primary

    var result = freshReplica.Read(key);
    _userSeenVersions[userId] = Math.Max(minVersion, result.version);
    return result;
}
```

**Causal consistency:**
Writes that are causally related appear in causal order. "If A causes B, everyone sees A before B." Concurrent writes can appear in any order.

```
User A posts: "Anyone want coffee?"     ← write W1
User B replies: "Yes please!"          ← write W2 (caused by W1)

Causal consistency guarantees:
  Anyone who sees W2 has already seen W1.
  You never see "Yes please!" without the original question.

But:
  User C posts: "It's raining today"   ← write W3 (unrelated to W1, W2)
  W1, W2, W3 may appear in any order for users who saw none of them first.
```

**Linearizability (strong consistency):**
The system appears as if there's one copy of data. Every operation takes effect instantaneously at some point between its start and completion. The strongest single-object guarantee.

```
Timeline:
Write[A5=booked] starts ──────────────────── ends
                              ↑
                    appears to happen here

Read[A5] starts ────── ends
                  ↑
          sees "booked" (write happened before this read ended)
```

**Serializability (strongest — transaction-level):**
All transactions appear to execute one at a time in some serial order. Even if they ran concurrently, the result is as if they were sequential. The gold standard for ACID databases.

## AXS Interview Angle

> *"How do you decide between eventual and strong consistency for different parts of the AXS booking platform?"*

```
Booking transaction (strongly consistent — mandatory):
  "Is seat A5 available?"
  → Must be strongly consistent.

  → Stale answer = double booking = fraud, chargebacks, legal liability.
  → Cost of extra latency (10-20ms): completely acceptable.

Seat availability map display (eventually consistent — intentional):
  The visual seat map showing green/red seats for 70,000 seats.

  → Serving from async Redis replicas with ~200ms replication lag.
  → If seat A5 shows as "available" for 200ms after someone books it:
    User clicks it → tries to book → gets "seat just taken, try another".
    Minor UX friction. Not data corruption.
  → Benefit: Redis replicas absorb 95% of reads without hitting PostgreSQL.
  → Tradeoff explicitly accepted.

User session data (read-your-writes — minimum required):
  User logs in, updates email preference.
```

### Question 1: Implement read-your-writes consistency *(most practical)*

> "Users should always see their own writes even though we use eventually consistent replicas."

```csharp
public class ReadYourWritesStore
{
    private readonly IDatabase _primary;
    private readonly List<IDatabase> _replicas;
    // userId → timestamp of their last write
    private readonly IDatabase _writeTracker; // use Redis for distributed state

    private readonly TimeSpan _sessionWindow = TimeSpan.FromSeconds(10);

    public async Task WriteAsync(string userId, string key, string value)
    {
        // Write to primary
        await _primary.SetAsync(key, value);

        // Record that this user just wrote
        await _writeTracker.SetAsync(
            $"last_write:{userId}",
            DateTime.UtcNow.ToString("O"),
            _sessionWindow
        );
    }

    public async Task<string?> ReadAsync(string userId, string key)
    {
        // Check if this user recently wrote
        var lastWriteStr = await _writeTracker.GetAsync($"last_write:{userId}");

        if (lastWriteStr != null)
        {
            var lastWrite = DateTime.Parse(lastWriteStr);
            if (DateTime.UtcNow - lastWrite < _sessionWindow)
            {
                // User recently wrote — route to primary for consistency
                return await _primary.GetAsync(key);
            }
        }

        // User hasn't written recently — replica is fine
        var replica = GetLeastLoadedReplica();
        return await replica.GetAsync(key);
    }

    // Monotonic reads — user never sees data older than what they've seen
    public async Task<(string? value, long version)> ReadMonotonicAsync(
        string userId,
        string key)
    {
        // Get highest version this user has previously seen
        var seenVersionStr = await _writeTracker.GetAsync($"seen_version:{userId}:{key}");
        long minVersion = seenVersionStr != null ? long.Parse(seenVersionStr) : 0;

        // Find replica with version >= what user has seen
        foreach (var replica in _replicas)
        {
            var (value, version) = await replica.GetWithVersionAsync(key);

            if (version >= minVersion)
            {
                // Update highest seen version for this user+key
                await _writeTracker.SetAsync(
                    $"seen_version:{userId}:{key}",
                    version.ToString(),
                    TimeSpan.FromHours(1)
                );
                return (value, version);
            }
        }

        // No replica fresh enough — use primary
        var result = await _primary.GetWithVersionAsync(key);
        await _writeTracker.SetAsync(
            $"seen_version:{userId}:{key}",
            result.version.ToString(),
            TimeSpan.FromHours(1)
        );
        return result;
    }

    private IDatabase GetLeastLoadedReplica()
        => _replicas[Random.Shared.Next(_replicas.Count)];
}
```

### Question 2: Implement a CRDT counter *(eventually consistent without conflicts)*

> "Multiple nodes increment a view counter independently. Merge without conflicts."

```csharp
// G-Counter: Grow-only counter CRDT
// Each node maintains its own count. Total = sum of all.
// Merging = taking max of each node's count.
// No conflicts possible. Mathematically guaranteed.

public class GCounter
{
    // nodeId → count on that node
    private readonly Dictionary<string, long> _counts;
    private readonly string _nodeId;

    public GCounter(string nodeId)
    {
        _nodeId = nodeId;
        _counts = new Dictionary<string, long> { [nodeId] = 0 };
    }

    // Only this node can increment its own count
    public void Increment(long amount = 1)
    {
        _counts[_nodeId] = _counts.GetValueOrDefault(_nodeId, 0) + amount;
    }

    // Total = sum across all nodes
    public long Value => _counts.Values.Sum();

    // Merge with another node's counter — take max of each component
    // This is commutative, associative, idempotent — CRDT properties
    public GCounter Merge(GCounter other)
    {
        var merged = new GCounter(_nodeId);

        var allNodes = _counts.Keys.Union(other._counts.Keys);
        foreach (var node in allNodes)
        {
            merged._counts[node] = Math.Max(
                _counts.GetValueOrDefault(node, 0),
                other._counts.GetValueOrDefault(node, 0)
            );
        }

        return merged;
    }

    public Dictionary<string, long> GetState() =>
        new Dictionary<string, long>(_counts);
}

// Simulation: two nodes increment independently during partition
var nodeA = new GCounter("node-A");
var nodeB = new GCounter("node-B");

// Normal: both in sync (counts both at 0)

// ── PARTITION ──
// Node A increments (e.g., from US users)
nodeA.Increment(1000);
nodeA.Increment(500);

// Node B increments independently (e.g., from EU users)
nodeB.Increment(750);

Console.WriteLine($"Node A sees: {nodeA.Value}"); // 1500
Console.WriteLine($"Node B sees: {nodeB.Value}"); // 750
// Different! But no conflict — each node only knows its own increments.

// ── PARTITION HEALS ──
var merged = nodeA.Merge(nodeB);
Console.WriteLine($"Merged: {merged.Value}"); // 2250 — correct total!

// Merge is idempotent — merging twice gives same result
var doubleMerged = merged.Merge(nodeB);
Console.WriteLine($"Double merged: {doubleMerged.Value}"); // still 2250
```

### Question 3: Implement a version vector for causal consistency

> "Track which writes causally depend on which other writes."

```csharp
public class VersionVector
{
    // nodeId → logical clock value
    private readonly Dictionary<string, long> _clock;

    public VersionVector()
    {
        _clock = new Dictionary<string, long>();
    }

    private VersionVector(Dictionary<string, long> clock)
    {
        _clock = new Dictionary<string, long>(clock);
    }

    // Increment this node's clock on write
    public VersionVector Increment(string nodeId)
    {
        var next = new Dictionary<string, long>(_clock);
        next[nodeId] = next.GetValueOrDefault(nodeId, 0) + 1;
        return new VersionVector(next);
    }

    // Merge: take max of each component
    public VersionVector Merge(VersionVector other)
    {
        var merged = new Dictionary<string, long>(_clock);
        foreach (var (node, version) in other._clock)
            merged[node] = Math.Max(merged.GetValueOrDefault(node, 0), version);
        return new VersionVector(merged);
    }

    // Does this vector causally precede other?
    // "this happened before other"
    public bool HappensBefore(VersionVector other)
    {
        // All components of this ≤ other, and at least one is strictly less
        bool allLeOrEqual = _clock.All(kvp =>
            other._clock.GetValueOrDefault(kvp.Key, 0) >= kvp.Value);
        bool atLeastOneStrictlyLess = _clock.Any(kvp =>
            other._clock.GetValueOrDefault(kvp.Key, 0) > kvp.Value);

        return allLeOrEqual && atLeastOneStrictlyLess;
    }

    // Are these concurrent? (neither happens before the other)
    public bool IsConcurrentWith(VersionVector other)
        => !HappensBefore(other) && !other.HappensBefore(this) && !Equals(other);

    public override string ToString() =>
        "{" + string.Join(", ", _clock.Select(kvp => $"{kvp.Key}:{kvp.Value}")) + "}";
}

public class CausallyCon­sistentStore
{
    // key → (value, version vector)
    private readonly Dictionary<string, (string value, VersionVector vv)> _store = new();
    private VersionVector _nodeVector;
    private readonly string _nodeId;

    public CausallyConsistentStore(string nodeId)
    {
        _nodeId = nodeId;
        _nodeVector = new VersionVector();
    }

    public (string value, VersionVector vv) Write(
        string key,
        string value,
        VersionVector clientVector)
    {
        // Merge client's knowledge with our own
        _nodeVector = _nodeVector.Merge(clientVector).Increment(_nodeId);

        _store[key] = (value, _nodeVector);
        return (value, _nodeVector);
    }

    public (string? value, VersionVector vv) Read(string key)
    {
        if (_store.TryGetValue(key, out var entry))
            return entry;
        return (null, new VersionVector());
    }

    // Receive replicated write — check for causal violation
    public ReplicationResult Replicate(
        string key,
        string value,
        VersionVector incomingVV)
    {
        if (!_store.TryGetValue(key, out var existing))
        {
            _store[key] = (value, incomingVV);
            return ReplicationResult.Applied;
        }

        if (incomingVV.HappensBefore(existing.vv))
            return ReplicationResult.Ignored; // we already have newer

        if (existing.vv.HappensBefore(incomingVV))
        {
            _store[key] = (value, incomingVV); // incoming is newer
            return ReplicationResult.Applied;
        }

        // Concurrent writes — conflict!
        return ReplicationResult.Conflict;
    }
}

public enum ReplicationResult { Applied, Ignored, Conflict }

// Demo: causal consistency in action
var nodeA = new CausallyConsistentStore("A");
var nodeB = new CausallyConsistentStore("B");

// User posts question
var (_, vv1) = nodeA.Write("post:1", "Anyone want coffee?", new VersionVector());
Console.WriteLine($"Post written with vector: {vv1}"); // {A:1}

// Reply is causally dependent on the post
// Client passes vv1 to show "my reply depends on seeing the post"
var (_, vv2) = nodeB.Write("post:2", "Yes please!", vv1);
Console.WriteLine($"Reply written with vector: {vv2}"); // {A:1, B:1}

// Replicate post to nodeB
var result = nodeB.Replicate("post:1", "Anyone want coffee?", vv1);
Console.WriteLine($"Post replication: {result}"); // Applied

// Any node receiving the reply KNOWS it must have the post first
// vv2 = {A:1, B:1} tells the system:
//   "This write depends on A:1 (the post) and B:1 (this node's own clock)"
// If post hasn't arrived yet → hold the reply until post arrives
Console.WriteLine($"Reply depends on post: {vv1.HappensBefore(vv2)}"); // true
```

### Question 4: Implement a linearizable compare-and-swap *(strong consistency primitive)*

> "Implement an atomic compare-and-swap that works correctly across distributed nodes."

```csharp
public class LinearizableStore
{
    // Simulate a strongly consistent store using quorum
    private readonly List<NodeState> _nodes;
    private readonly int _quorum;

    public LinearizableStore(int nodeCount = 3)
    {
        _nodes = Enumerable.Range(0, nodeCount)
            .Select(_ => new NodeState())
            .ToList();
        _quorum = (nodeCount / 2) + 1;
    }

    // Atomic compare-and-swap — linearizable
    // Only swaps if current value == expected value
    public async Task<CasResult> CompareAndSwapAsync(
        string key,
        string? expectedValue,
        string newValue)
    {
        // Phase 1: Read current value from quorum
        var readTasks = _nodes.Select(n => n.ReadAsync(key));
        var reads = await Task.WhenAll(readTasks);
        var validReads = reads.Where(r => r != null).ToList();

        if (validReads.Count < _quorum)
            return CasResult.QuorumFailure; // can't determine current value

        // Use highest-versioned value among quorum responses
        var current = validReads
            .OrderByDescending(r => r!.Version)
            .First();

        // Check expected value
        if (current?.Value != expectedValue)
            return CasResult.Mismatch; // current != expected, CAS fails

        long newVersion = (current?.Version ?? 0) + 1;

        // Phase 2: Write new value to quorum
        var writeTasks = _nodes.Select(n =>
            n.ConditionalWriteAsync(key, newValue, newVersion, current?.Version ?? -1)
        );
        var writeResults = await Task.WhenAll(writeTasks);
        int successCount = writeResults.Count(r => r);

        if (successCount < _quorum)
        {
            // Partial write — must repair (read-repair or rollback)
            await RepairAsync(key, expectedValue, current?.Version ?? -1);
            return CasResult.Failure;
        }

        return CasResult.Success;
    }

    private async Task RepairAsync(string key, string? value, long version)
    {
        // Restore old value on nodes that may have accepted the write
        await Task.WhenAll(_nodes.Select(n =>
            n.ConditionalWriteAsync(key, value, version, version + 1)
        ));
    }
}

public class NodeState
{
    private readonly Dictionary<string, (string? value, long version)> _data = new();
    private readonly SemaphoreSlim _lock = new(1, 1);

    public async Task<(string? Value, long Version)?> ReadAsync(string key)
    {
        await _lock.WaitAsync();
        try
        {
            return _data.TryGetValue(key, out var entry)
                ? entry
                : (null, -1);
        }
        finally { _lock.Release(); }
    }

    public async Task<bool> ConditionalWriteAsync(
        string key,
        string? newValue,
        long newVersion,
        long expectedVersion)
    {
        await _lock.WaitAsync();
        try
        {
            var current = _data.GetValueOrDefault(key, (null, -1));

            // Only write if current version matches expected
            if (current.Item2 != expectedVersion)
                return false;

            _data[key] = (newValue, newVersion);
            return true;
        }
        finally { _lock.Release(); }
    }
}

public enum CasResult { Success, Mismatch, Failure, QuorumFailure }

// Demo: linearizable seat booking
var store = new LinearizableStore(nodeCount: 3);

// Two users simultaneously try to book seat A5
var task1 = store.CompareAndSwapAsync("seat:A5", "available", "booked:user-42");
var task2 = store.CompareAndSwapAsync("seat:A5", "available", "booked:user-87");

var results = await Task.WhenAll(task1, task2);
Console.WriteLine($"User 42 result: {results[0]}"); // One gets Success
Console.WriteLine($"User 87 result: {results[1]}"); // Other gets Mismatch

// Exactly one user booked the seat. Linearizability guaranteed.
```

## One-Line Summary for Interviews

> **Strong consistency guarantees every read sees the latest write regardless of which node handles it — achieved through synchronous replication, quorum coordination, or reading from primary only, at the cost of higher latency and reduced availability during partitions — while eventual consistency lets nodes return whatever they have locally and converge to agreement asynchronously, achieving lower latency and higher availability at the cost of stale reads and the need for conflict resolution; the practical senior-level answer is that these aren't binary choices but a spectrum — read-your-writes, monotonic reads, and causal consistency sit in between — and modern systems like Cassandra and DynamoDB let you tune consistency per operation so you can use strong consistency where money and inventory demand it and eventual consistency where speed and scale matter more than perfect accuracy.**
