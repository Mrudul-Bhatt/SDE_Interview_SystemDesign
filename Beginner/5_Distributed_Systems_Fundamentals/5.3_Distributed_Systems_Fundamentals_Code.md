## Q1. Implement a Quorum Read/Write System

> Given N replicas, a write succeeds if W nodes confirm it. A read succeeds if R nodes respond. Strong consistency is guaranteed when `W + R > N`. Implement both modes and show the difference.

### The Rule
```
N = total replicas
W = write quorum (nodes that must confirm a write)
R = read quorum (nodes that must respond to a read)

W + R > N  →  at least one node in the read set has the latest write
             →  strong consistency guaranteed

Examples with N=3:
  Eventual:  W=1, R=1  → fast, stale reads possible
  Quorum:    W=2, R=2  → W+R=4 > 3, strong consistency
  Strong:    W=3, R=1  → all nodes confirm, any single read is current
```

```csharp
using System;
using System.Collections.Generic;
using System.Linq;

public class Replica
{
    public string Id       { get; init; } = "";
    public bool IsHealthy  { get; set; } = true;

    private readonly Dictionary<string, (string Value, long Timestamp)> _data = new();

    public bool Write(string key, string value, long timestamp)
    {
        if (!IsHealthy) return false;
        _data[key] = (value, timestamp);
        return true;
    }

    public (string? Value, long Timestamp) Read(string key)
    {
        if (!IsHealthy) return (null, -1);
        return _data.TryGetValue(key, out var entry) ? entry : (null, -1);
    }
}

public class QuorumStore
{
    private readonly List<Replica> _replicas;
    private readonly int _w; // write quorum
    private readonly int _r; // read quorum
    private readonly int _n; // total replicas

    public QuorumStore(int n, int w, int r)
    {
        _n = n; _w = w; _r = r;
        _replicas = Enumerable.Range(1, n)
            .Select(i => new Replica { Id = $"node_{i}" })
            .ToList();

        Console.WriteLine($"[CONFIG] N={n}, W={w}, R={r}  " +
                          $"→ Strong consistency: {(w + r > n ? "YES" : "NO")} (W+R={w+r} {(w+r > n ? ">" : "≤")} N={n})");
    }

    public bool Write(string key, string value)
    {
        long timestamp = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        int confirmed = 0;

        foreach (var replica in _replicas)
        {
            if (replica.Write(key, value, timestamp))
                confirmed++;
        }

        bool success = confirmed >= _w;
        Console.WriteLine($"[WRITE] key={key} value={value}  " +
                          $"confirmed={confirmed}/{_n}  required={_w}  " +
                          $"→ {(success ? "SUCCESS" : "FAILED")}");
        return success;
    }

    public string? Read(string key)
    {
        var responses = new List<(string NodeId, string? Value, long Timestamp)>();

        foreach (var replica in _replicas)
        {
            var (value, ts) = replica.Read(key);
            if (value != null)
                responses.Add((replica.Id, value, ts));
        }

        if (responses.Count < _r)
        {
            Console.WriteLine($"[READ]  key={key}  responses={responses.Count}/{_n}  " +
                              $"required={_r}  → FAILED (not enough nodes responded)");
            return null;
        }

        // Return value with the highest timestamp (most recent write)
        var latest = responses.OrderByDescending(r => r.Timestamp).First();
        Console.WriteLine($"[READ]  key={key}  responses={responses.Count}/{_n}  " +
                          $"required={_r}  value={latest.Value} (from {latest.NodeId})  → SUCCESS");
        return latest.Value;
    }

    public void MarkDown(int nodeIndex) =>
        _replicas[nodeIndex].IsHealthy = false;

    public void MarkUp(int nodeIndex) =>
        _replicas[nodeIndex].IsHealthy = true;
}

// --- Demo 1: Eventual consistency (W=1, R=1) ---
Console.WriteLine("=== Eventual Consistency (N=3, W=1, R=1) ===");
var eventual = new QuorumStore(n: 3, w: 1, r: 1);
eventual.Write("user:1", "Alice");
eventual.Read("user:1");

// Take 2 nodes down — read still works from 1 node (but may be stale)
eventual.MarkDown(1);
eventual.MarkDown(2);
eventual.Read("user:1"); // succeeds from node_1

Console.WriteLine();

// --- Demo 2: Strong consistency (W=2, R=2, W+R=4 > N=3) ---
Console.WriteLine("=== Strong Consistency (N=3, W=2, R=2) ===");
var strong = new QuorumStore(n: 3, w: 2, r: 2);
strong.Write("balance", "1000");
strong.Read("balance");

// Take 2 nodes down — write fails (can't reach W=2), read fails (can't reach R=2)
strong.MarkDown(1);
strong.MarkDown(2);
strong.Write("balance", "500");  // FAILED — only 1 node up, need W=2
strong.Read("balance");          // FAILED — only 1 node up, need R=2
// → System rejects rather than risk inconsistency (CP behavior)

// Output:
// === Eventual Consistency (N=3, W=1, R=1) ===
// [CONFIG] N=3, W=1, R=1  → Strong consistency: NO (W+R=2 ≤ N=3)
// [WRITE] key=user:1 value=Alice  confirmed=3/3  required=1  → SUCCESS
// [READ]  key=user:1  responses=3/3  required=1  value=Alice  → SUCCESS
// [READ]  key=user:1  responses=1/3  required=1  value=Alice  → SUCCESS
//
// === Strong Consistency (N=3, W=2, R=2) ===
// [CONFIG] N=3, W=2, R=2  → Strong consistency: YES (W+R=4 > N=3)
// [WRITE] key=balance value=1000  confirmed=3/3  required=2  → SUCCESS
// [READ]  key=balance  responses=3/3  required=2  value=1000  → SUCCESS
// [WRITE] key=balance value=500   confirmed=1/3  required=2  → FAILED
// [READ]  key=balance  responses=1/3  required=2  → FAILED (not enough nodes responded)
```

### When to Use Each Quorum Setting
```
Payment / booking:   W=N, R=1   → all nodes confirm write, reads are always fresh
                     W=2, R=2   → balanced: tolerates 1 failure, still consistent

Social feed:         W=1, R=1   → fastest, stale reads acceptable
                     W=1, R=2   → slightly more consistent reads, still fast writes

Analytics counters:  W=1, R=1   → losing a count increment is acceptable
```

**Complexity:** Write O(N), Read O(N) — must contact all replicas; quorum is about how many must respond

## Q2. Implement Last Write Wins (LWW) Conflict Resolution

> In master-master replication, two nodes can accept conflicting writes to the same key simultaneously. Implement Last Write Wins using timestamps, detect conflicts, and show the risk of clock skew.

```csharp
public class LWWRegister
{
    public string NodeId { get; init; } = "";

    private readonly Dictionary<string, (string Value, long Timestamp, string OriginNode)> _data = new();

    public void Write(string key, string value, long? timestampOverride = null)
    {
        long ts = timestampOverride ?? DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        _data[key] = (value, ts, NodeId);
        Console.WriteLine($"[{NodeId}] WRITE key={key} value={value} ts={ts}");
    }

    // Merge incoming write from another node — LWW resolution
    public void Merge(string key, string value, long timestamp, string fromNode)
    {
        if (_data.TryGetValue(key, out var existing))
        {
            if (timestamp > existing.Timestamp)
            {
                Console.WriteLine($"[{NodeId}] MERGE key={key}: '{existing.Value}'(ts={existing.Timestamp}) " +
                                  $"← '{value}'(ts={timestamp} from {fromNode})  → ACCEPTED (newer)");
                _data[key] = (value, timestamp, fromNode);
            }
            else
            {
                Console.WriteLine($"[{NodeId}] MERGE key={key}: '{existing.Value}'(ts={existing.Timestamp}) " +
                                  $"← '{value}'(ts={timestamp} from {fromNode})  → REJECTED (older)");
            }
        }
        else
        {
            _data[key] = (value, timestamp, fromNode);
            Console.WriteLine($"[{NodeId}] MERGE key={key}: no local value → ACCEPTED '{value}'");
        }
    }

    public string? Get(string key) =>
        _data.TryGetValue(key, out var entry) ? entry.Value : null;

    public void ReplicateTo(LWWRegister other, string key)
    {
        if (_data.TryGetValue(key, out var entry))
            other.Merge(key, entry.Value, entry.Timestamp, NodeId);
    }

    public void PrintState() =>
        Console.WriteLine($"[{NodeId}] State: " +
            string.Join(", ", _data.Select(kv =>
                $"{kv.Key}={kv.Value.Value}(ts={kv.Value.Timestamp})")));
}

// --- Scenario 1: No conflict — one node writes, replicates ---
Console.WriteLine("=== Scenario 1: Normal replication (no conflict) ===");
var nodeA = new LWWRegister { NodeId = "MasterA" };
var nodeB = new LWWRegister { NodeId = "MasterB" };

nodeA.Write("email", "alice@old.com", timestampOverride: 1000);
nodeA.ReplicateTo(nodeB, "email");
nodeB.PrintState();

Console.WriteLine();

// --- Scenario 2: Conflict — concurrent writes on both masters ---
Console.WriteLine("=== Scenario 2: Concurrent conflict (LWW resolution) ===");
var nodeC = new LWWRegister { NodeId = "MasterA" };
var nodeD = new LWWRegister { NodeId = "MasterB" };

// Simultaneous writes at t=2000 and t=2050 (network partition)
nodeC.Write("email", "alice@version1.com", timestampOverride: 2000);
nodeD.Write("email", "alice@version2.com", timestampOverride: 2050);

Console.WriteLine("--- Partition heals, nodes replicate ---");
nodeC.ReplicateTo(nodeD, "email"); // MasterA sends ts=2000 to MasterB
nodeD.ReplicateTo(nodeC, "email"); // MasterB sends ts=2050 to MasterA

Console.WriteLine("--- Final state (should converge) ---");
nodeC.PrintState(); // → alice@version2.com (ts=2050 won)
nodeD.PrintState(); // → alice@version2.com (ts=2050 won)

Console.WriteLine();

// --- Scenario 3: Clock skew danger ---
Console.WriteLine("=== Scenario 3: Clock skew — wrong winner ===");
var nodeE = new LWWRegister { NodeId = "MasterA" };
var nodeF = new LWWRegister { NodeId = "MasterB" };

// User writes "version2" AFTER "version1" — but MasterA's clock is 200ms behind
nodeE.Write("email", "alice@version1.com", timestampOverride: 3000); // written at real t=3200
nodeF.Write("email", "alice@version2.com", timestampOverride: 3100); // written at real t=3100 (actually earlier!)

Console.WriteLine("--- Replication with skewed clocks ---");
nodeE.ReplicateTo(nodeF, "email");
nodeF.ReplicateTo(nodeE, "email");

nodeE.PrintState(); // → alice@version1.com (ts=3000 < ts=3100... but wrong winner!)
nodeF.PrintState(); // → alice@version2.com (LOST — the newer real-world write lost)
Console.WriteLine("⚠ Clock skew caused data loss: version2 (the actual latest) was overwritten");
```

### LWW Trade-offs and Alternatives
```
Last Write Wins (LWW):
  ✓ Simple to implement
  ✓ Always converges — no permanent conflicts
  ✗ Can silently discard writes (the lost update problem)
  ✗ Vulnerable to clock skew — NTP sync must be <1ms
  Used by: Cassandra (default), Redis (AOF conflicts)

Alternatives:
  CRDT (Conflict-free Replicated Data Types):
    → Data structures designed to merge without conflicts
    → Example: G-Counter (grow-only counter) — increment is always safe
    → Used by: Redis CRDT, Riak, collaborative editors

  Vector Clocks:
    → Track causal history instead of wall-clock time
    → Can detect "truly concurrent" vs "causally ordered" writes
    → Used by: Amazon DynamoDB (behind the scenes), Riak

  Application-defined merge:
    → Developer writes custom conflict handler
    → Example: shopping cart — merge both carts, deduplicate items
```

**Complexity:** Write O(1), Merge O(1), Replicate O(1) per key

## Q3. Simulate Replication Lag and Read-Your-Writes Consistency

> Model a master-slave system where writes go to the master and reads go to replicas. Show stale reads caused by replication lag, then fix it with the read-your-writes pattern.

```csharp
public class ReplicationLagSimulator
{
    private readonly Dictionary<string, string> _master = new();
    private readonly Dictionary<string, string> _replica = new();

    // Track which keys a client wrote — for read-your-writes routing
    private readonly HashSet<string> _recentWrites = new();

    private int _pendingReplicationCount = 0;

    public void Write(string key, string value, string clientId = "client")
    {
        _master[key] = value;
        _recentWrites.Add(key); // remember this client wrote this key
        _pendingReplicationCount++;

        Console.WriteLine($"[MASTER]  WRITE key={key} value='{value}'  " +
                          $"(replica lag: {_pendingReplicationCount} pending)");
    }

    // Simulate replica catching up asynchronously
    public void ReplicateAll()
    {
        foreach (var (key, value) in _master)
            _replica[key] = value;

        Console.WriteLine($"[REPLICA] Caught up — {_pendingReplicationCount} write(s) replicated");
        _pendingReplicationCount = 0;
    }

    // Naive read — always goes to replica (may be stale)
    public string? ReadFromReplica(string key)
    {
        _replica.TryGetValue(key, out var value);
        string lag = _pendingReplicationCount > 0 ? " ⚠ STALE" : "";
        Console.WriteLine($"[REPLICA] READ key={key} → '{value ?? "null"}'{lag}");
        return value;
    }

    // Read-your-writes — route to master if this client recently wrote the key
    public string? ReadWithConsistency(string key)
    {
        if (_recentWrites.Contains(key))
        {
            _master.TryGetValue(key, out var fresh);
            Console.WriteLine($"[MASTER]  READ key={key} → '{fresh}' (read-your-writes routing)");
            return fresh;
        }

        return ReadFromReplica(key);
    }

    // Clear the write tracking after a TTL (real systems use a timer)
    public void ClearWriteTracking() => _recentWrites.Clear();
}

// --- Scenario 1: Stale read without fix ---
Console.WriteLine("=== Scenario 1: Stale read (no consistency fix) ===");
var sim = new ReplicationLagSimulator();

sim.Write("user:1:avatar", "avatar_v1.png");
// Replica hasn't caught up yet
Console.WriteLine("User immediately reads their profile page:");
sim.ReadFromReplica("user:1:avatar"); // ⚠ STALE — returns null or old value

Console.WriteLine("\nUser refreshes after 50ms (replica caught up):");
sim.ReplicateAll();
sim.ReadFromReplica("user:1:avatar"); // now fresh

Console.WriteLine();

// --- Scenario 2: Read-your-writes fix ---
Console.WriteLine("=== Scenario 2: Read-your-writes consistency ===");
var sim2 = new ReplicationLagSimulator();

sim2.Write("user:1:avatar", "avatar_v2.png");

Console.WriteLine("User reads their own profile page immediately after upload:");
sim2.ReadWithConsistency("user:1:avatar"); // → routed to master, sees fresh data

Console.WriteLine("Other user reads the same profile (replica is fine for them):");
sim2.ClearWriteTracking(); // simulate: this is a different client session
sim2.ReadWithConsistency("user:1:avatar"); // → goes to replica (may be stale, acceptable)

Console.WriteLine("\nReplica catches up:");
sim2.ReplicateAll();
sim2.ReadWithConsistency("user:1:avatar"); // → now reads from replica (fresh)

// Output:
// === Scenario 1: Stale read (no consistency fix) ===
// [MASTER]  WRITE key=user:1:avatar value='avatar_v1.png'  (replica lag: 1 pending)
// User immediately reads their profile page:
// [REPLICA] READ key=user:1:avatar → 'null' ⚠ STALE
// User refreshes after 50ms (replica caught up):
// [REPLICA] Caught up — 1 write(s) replicated
// [REPLICA] READ key=user:1:avatar → 'avatar_v1.png'
//
// === Scenario 2: Read-your-writes consistency ===
// [MASTER]  WRITE key=user:1:avatar value='avatar_v2.png'  (replica lag: 1 pending)
// User reads their own profile page immediately after upload:
// [MASTER]  READ key=user:1:avatar → 'avatar_v2.png' (read-your-writes routing)
// Other user reads the same profile (replica is fine for them):
// [REPLICA] READ key=user:1:avatar → 'null' ⚠ STALE
// Replica catches up:
// [REPLICA] Caught up — 1 write(s) replicated
// [REPLICA] READ key=user:1:avatar → 'avatar_v2.png'
```

### The Four Consistency Guarantees (Know These for Interviews)
```
1. Eventual Consistency (weakest):
   → All nodes converge eventually if no new writes
   → Stale reads possible at any time
   → Example: DNS, Cassandra default

2. Read-Your-Writes:
   → After you write, you always see your own write
   → Others may still see stale data
   → Example: Route user's own reads to master after a write
   → Used by: Facebook (sticky routing), PostgreSQL read replicas

3. Monotonic Read:
   → Once you read a value, you never read an older one
   → Route a client session to the same replica consistently
   → Example: Stick user to replica R2 for the session lifetime

4. Strong / Linearizable (strongest):
   → Every read sees the latest write globally
   → Requires quorum (W+R>N) or single-master routing
   → Example: ZooKeeper, etcd, Google Spanner
   → Cost: higher latency, lower availability

Rule of thumb for interviews:
  Money / inventory / auth tokens → Strong consistency
  Profile views / feeds / counts  → Eventual + read-your-writes is enough
```

## What Interviewers Test

```
1. Quorum — can you derive the W+R>N rule?
   → With N=3, W=2, R=2: any read set of 2 overlaps any write set of 2
   → Overlap guarantees at least 1 node has the latest write
   → Forgetting to pick highest timestamp across R responses is a bug

2. LWW — what is the lost update problem?
   → Two concurrent writes → only one survives
   → The discarded write is silently lost — no error thrown
   → Only safe when last write is always "most correct" (counters: NO, email: maybe)

3. LWW — what breaks with clock skew?
   → NTP sync is not perfect — clocks can drift ±100ms
   → A "later" timestamp may actually be from an earlier real-world write
   → Fix: use Hybrid Logical Clocks (HLC) or vector clocks

4. Replication lag — what is the read-your-writes pattern?
   → After a write, route that client's reads to master for a short window
   → Common implementation: sticky session or write token with TTL

5. CAP in code — what happens when nodes go down in quorum store?
   → Eventual (W=1,R=1): still serves reads/writes with 1 node up
   → Strong (W=2,R=2): refuses both reads and writes with only 1 node up
   → Strong system trades availability for correctness (CP behavior)
```
