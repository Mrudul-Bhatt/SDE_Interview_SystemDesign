## Q1. Implement Indexed vs Non-Indexed Table Lookup

> Show the performance difference between a full table scan (no index) and a B-Tree index lookup. A `SortedDictionary` in C# is a balanced BST — identical to how real databases implement B-Tree indexes.

```csharp
using System;
using System.Collections.Generic;
using System.Diagnostics;

public class TableWithoutIndex
{
    private readonly List<Dictionary<string, object>> _rows = new();

    public void Insert(Dictionary<string, object> row) => _rows.Add(row);

    // Full table scan — O(n)
    public List<Dictionary<string, object>> FindByEmail(string email)
    {
        var results = new List<Dictionary<string, object>>();
        foreach (var row in _rows)
            if (row.TryGetValue("email", out var val) && val.Equals(email))
                results.Add(row);
        return results;
    }

    public int RowCount => _rows.Count;
}

public class TableWithIndex
{
    private readonly List<Dictionary<string, object>> _rows = new();
    // B-Tree index: email → list of row positions
    private readonly SortedDictionary<string, List<int>> _emailIndex = new();

    public void Insert(Dictionary<string, object> row)
    {
        int rowId = _rows.Count;
        _rows.Add(row);

        // Update index on every insert — the write overhead of indexing
        if (row.TryGetValue("email", out var email) && email is string emailStr)
        {
            if (!_emailIndex.ContainsKey(emailStr))
                _emailIndex[emailStr] = new List<int>();
            _emailIndex[emailStr].Add(rowId);
        }
    }

    // Index lookup — O(log n)
    public List<Dictionary<string, object>> FindByEmail(string email)
    {
        if (!_emailIndex.TryGetValue(email, out var rowIds))
            return new List<Dictionary<string, object>>();

        var results = new List<Dictionary<string, object>>();
        foreach (var id in rowIds)
            results.Add(_rows[id]);
        return results;
    }

    public int RowCount => _rows.Count;
}

// Usage — seed 1 million rows, compare lookup time
var noIndex = new TableWithoutIndex();
var withIndex = new TableWithIndex();

Console.WriteLine("Seeding 500,000 rows...");
for (int i = 0; i < 500_000; i++)
{
    var row = new Dictionary<string, object>
    {
        ["id"]    = i,
        ["name"]  = $"User{i}",
        ["email"] = $"user{i}@example.com"
    };
    noIndex.Insert(row);
    withIndex.Insert(row);
}

string target = "user499999@example.com"; // last row — worst case for scan

var sw = Stopwatch.StartNew();
var r1 = noIndex.FindByEmail(target);
sw.Stop();
Console.WriteLine($"Without index: {sw.ElapsedMilliseconds}ms  (found {r1.Count} row)");

sw.Restart();
var r2 = withIndex.FindByEmail(target);
sw.Stop();
Console.WriteLine($"With index:    {sw.ElapsedMilliseconds}ms  (found {r2.Count} row)");

// Expected output:
// Without index: ~80ms   (scanned all 500k rows)
// With index:    ~0ms    (SortedDictionary lookup, O(log n))
```

### What This Demonstrates
```
Without index:  O(n) — must read every row
With index:     O(log n) — SortedDictionary is a balanced BST
                           same structure real databases use (B-Tree)

At 500,000 rows:
  Scan:  ~500,000 comparisons
  Index: ~19 comparisons (log₂ 500,000 ≈ 19)

Trade-off: every Insert must update the index too
  → Read-heavy tables: add indexes
  → Write-heavy tables: fewer indexes (each index = extra write cost)
```

**Complexity:** Insert O(log n) with index vs O(1) without | FindByEmail O(log n) with index vs O(n) without

## Q2. Implement a Write-Ahead Log (WAL) with Commit and Rollback

> Simulate ACID Atomicity and Durability. All changes are written to a log first. On commit, changes are applied. On rollback (or crash), the log is replayed in reverse to undo partial changes.

```csharp
public class WALDatabase
{
    // Represents the committed database state
    private readonly Dictionary<string, decimal> _committed = new();

    // Write-Ahead Log: records every operation before applying it
    private readonly List<(string Op, string Key, decimal OldVal, decimal NewVal)> _log = new();

    private bool _inTransaction = false;
    // Staged changes not yet committed
    private readonly Dictionary<string, decimal> _staged = new();

    public void BeginTransaction()
    {
        if (_inTransaction) throw new InvalidOperationException("Transaction already in progress");
        _inTransaction = true;
        _staged.Clear();
        _log.Clear();
        Console.WriteLine("[WAL] BEGIN TRANSACTION");
    }

    public void Set(string key, decimal value)
    {
        if (!_inTransaction) throw new InvalidOperationException("No active transaction");

        decimal oldVal = _committed.GetValueOrDefault(key, 0);
        // Write to log BEFORE changing state (Write-Ahead)
        _log.Add(("SET", key, oldVal, value));
        _staged[key] = value;
        Console.WriteLine($"[WAL] LOG: SET {key} = {value}  (was {oldVal})");
    }

    public void Commit()
    {
        if (!_inTransaction) throw new InvalidOperationException("No active transaction");

        // Apply all staged changes to committed state
        foreach (var (key, value) in _staged)
            _committed[key] = value;

        _log.Clear();
        _staged.Clear();
        _inTransaction = false;
        Console.WriteLine("[WAL] COMMIT — changes applied");
    }

    public void Rollback()
    {
        if (!_inTransaction) throw new InvalidOperationException("No active transaction");

        // Replay log in reverse to undo changes
        Console.WriteLine("[WAL] ROLLBACK — undoing changes:");
        for (int i = _log.Count - 1; i >= 0; i--)
        {
            var (op, key, oldVal, _) = _log[i];
            Console.WriteLine($"  Undo {op} {key}: restoring to {oldVal}");
            // Staged changes are discarded — committed state unchanged
        }

        _log.Clear();
        _staged.Clear();
        _inTransaction = false;
    }

    public decimal Get(string key) =>
        _inTransaction && _staged.TryGetValue(key, out var staged)
            ? staged
            : _committed.GetValueOrDefault(key, 0);

    public void PrintState()
    {
        Console.WriteLine("  Committed state:");
        foreach (var (k, v) in _committed)
            Console.WriteLine($"    {k} = {v:C}");
    }
}

// Usage — bank transfer (classic ACID example)
var db = new WALDatabase();

// Seed initial balances
db.BeginTransaction();
db.Set("alice", 1000m);
db.Set("bob", 500m);
db.Commit();

Console.WriteLine("\n=== Before transfer ===");
db.PrintState();

// Successful transfer
Console.WriteLine("\n=== Transfer $300: Alice → Bob ===");
db.BeginTransaction();
db.Set("alice", db.Get("alice") - 300m); // alice: 1000 → 700
db.Set("bob",   db.Get("bob")   + 300m); // bob:    500 → 800
db.Commit();
db.PrintState();

// Failed transfer — system crash simulation
Console.WriteLine("\n=== Transfer $400: Alice → Bob (FAILS mid-way) ===");
db.BeginTransaction();
db.Set("alice", db.Get("alice") - 400m); // logged but not committed
// Simulate crash / validation failure before bob's credit
Console.WriteLine("  [CRASH] Power failure! Rolling back...");
db.Rollback();

Console.WriteLine("\n=== After rollback ===");
db.PrintState();
// Alice still has $700 — atomicity preserved

// Output:
// Before transfer:   alice=$1,000  bob=$500
// After transfer:    alice=$700    bob=$800
// After rollback:    alice=$700    bob=$800  (unchanged — rollback worked)
```

### How Real Databases Implement WAL
```
1. BEGIN TRANSACTION
2. Write changes to WAL log file on disk (durable)
3. COMMIT:
   → Flush WAL to disk
   → Apply changes to data pages
   → Mark transaction complete in log

On crash recovery:
   → Scan WAL log
   → Re-apply committed transactions (REDO)
   → Roll back incomplete transactions (UNDO)

This is exactly how PostgreSQL, MySQL InnoDB, and SQLite work.
```

**Complexity:** Commit O(n), Rollback O(n) — n = operations in transaction

## Q3. Implement an In-Memory Message Queue with Dead Letter Queue

> Build a queue where producers push messages and consumers process them. Failed messages are retried up to N times, then moved to a Dead Letter Queue (DLQ) for inspection.

```csharp
public class MessageQueue<T>
{
    private class QueueMessage
    {
        public T Payload         { get; init; } = default!;
        public int AttemptCount  { get; set; } = 0;
        public string MessageId  { get; init; } = Guid.NewGuid().ToString()[..8];
    }

    private readonly Queue<QueueMessage> _main = new();
    private readonly Queue<QueueMessage> _dlq  = new();
    private readonly int _maxAttempts;

    public int MainCount => _main.Count;
    public int DLQCount  => _dlq.Count;

    public MessageQueue(int maxAttempts = 3)
    {
        _maxAttempts = maxAttempts;
    }

    public void Enqueue(T payload)
    {
        var msg = new QueueMessage { Payload = payload };
        _main.Enqueue(msg);
        Console.WriteLine($"[ENQUEUE] id={msg.MessageId}  queue size={_main.Count}");
    }

    // Consumer calls this. Handler returns true (success) or false (failure).
    public bool Process(Func<T, bool> handler)
    {
        if (_main.Count == 0) return false;

        var msg = _main.Dequeue();
        msg.AttemptCount++;

        Console.Write($"[PROCESS] id={msg.MessageId}  attempt={msg.AttemptCount}/{_maxAttempts}  ");

        bool success = handler(msg.Payload);

        if (success)
        {
            Console.WriteLine("→ SUCCESS (acknowledged)");
        }
        else if (msg.AttemptCount < _maxAttempts)
        {
            Console.WriteLine("→ FAILED  (requeued for retry)");
            _main.Enqueue(msg); // requeue — at-least-once delivery
        }
        else
        {
            Console.WriteLine("→ FAILED  (max retries exceeded → DLQ)");
            _dlq.Enqueue(msg); // give up — move to dead letter queue
        }

        return success;
    }

    public void InspectDLQ()
    {
        Console.WriteLine($"\n[DLQ] {_dlq.Count} unprocessable message(s):");
        foreach (var msg in _dlq)
            Console.WriteLine($"  id={msg.MessageId}  attempts={msg.AttemptCount}  payload={msg.Payload}");
    }

    // Replay DLQ back to main queue after fixing the bug
    public int ReplayDLQ()
    {
        int count = _dlq.Count;
        while (_dlq.Count > 0)
        {
            var msg = _dlq.Dequeue();
            msg.AttemptCount = 0; // reset attempts
            _main.Enqueue(msg);
        }
        Console.WriteLine($"[REPLAY] {count} message(s) moved from DLQ → main queue");
        return count;
    }
}

// Usage — order processing queue
var queue = new MessageQueue<string>(maxAttempts: 3);

queue.Enqueue("order:1001");
queue.Enqueue("order:1002");
queue.Enqueue("order:1003"); // this one will fail

Console.WriteLine();
int callCount = 0;

// Process all messages — order:1003 simulates a broken handler
while (queue.MainCount > 0)
{
    queue.Process(payload =>
    {
        if (payload == "order:1003")
        {
            callCount++;
            return false; // always fails
        }
        return true; // success
    });
}

queue.InspectDLQ();

// Simulate: engineer fixes the bug, replays DLQ
Console.WriteLine("\n[FIX] Bug fixed. Replaying DLQ...");
queue.ReplayDLQ();
queue.Process(_ => true); // now succeeds

// Output:
// [ENQUEUE] id=a1b2c3d4  queue size=1
// [ENQUEUE] id=e5f6g7h8  queue size=2
// [ENQUEUE] id=i9j0k1l2  queue size=3
//
// [PROCESS] id=a1b2c3d4  attempt=1/3  → SUCCESS (acknowledged)
// [PROCESS] id=e5f6g7h8  attempt=1/3  → SUCCESS (acknowledged)
// [PROCESS] id=i9j0k1l2  attempt=1/3  → FAILED  (requeued for retry)
// [PROCESS] id=i9j0k1l2  attempt=2/3  → FAILED  (requeued for retry)
// [PROCESS] id=i9j0k1l2  attempt=3/3  → FAILED  (max retries exceeded → DLQ)
//
// [DLQ] 1 unprocessable message(s):
//   id=i9j0k1l2  attempts=3  payload=order:1003
```

### At-Least-Once vs Exactly-Once Delivery
```
At-least-once (this implementation):
  → Message retried until acknowledged
  → Consumer may process the same message twice (on crash before ACK)
  → Consumers must be idempotent: processing twice = same result as once
  → Used by: SQS, RabbitMQ default mode

Exactly-once:
  → Much harder — requires distributed transaction or dedup ID
  → Consumer tracks processed message IDs and skips duplicates
  → Used by: Kafka with transactional producers (v0.11+)

Idempotency example:
  // Safe — INSERT OR IGNORE checks if order already processed
  INSERT OR IGNORE INTO processed_orders (order_id) VALUES ('order:1003');
  IF rows_affected == 0: return; // already done, skip
```

**Complexity:** Enqueue O(1), Process O(1), ReplayDLQ O(dlq size)

## Q4. Implement an In-Memory Document Store

> Model a simple MongoDB-like store where documents are JSON objects. Support insert, find by field value, and update. Demonstrates the document model vs relational tables.

```csharp
public class DocumentStore
{
    private readonly Dictionary<string, Dictionary<string, object>> _store = new();
    private int _nextId = 1;

    public string Insert(Dictionary<string, object> doc)
    {
        string id = $"doc_{_nextId++}";
        doc["_id"] = id;
        _store[id] = new Dictionary<string, object>(doc);
        Console.WriteLine($"[INSERT] id={id}");
        return id;
    }

    public Dictionary<string, object>? FindById(string id) =>
        _store.TryGetValue(id, out var doc) ? doc : null;

    // Scan all documents and match on a single field — O(n)
    public List<Dictionary<string, object>> FindByField(string field, object value)
    {
        var results = new List<Dictionary<string, object>>();
        foreach (var doc in _store.Values)
            if (doc.TryGetValue(field, out var val) && val.Equals(value))
                results.Add(doc);
        return results;
    }

    // Flexible update — add/change any field without schema migration
    public bool Update(string id, Dictionary<string, object> fields)
    {
        if (!_store.TryGetValue(id, out var doc)) return false;
        foreach (var (key, value) in fields)
            doc[key] = value;
        Console.WriteLine($"[UPDATE] id={id}  fields=[{string.Join(", ", fields.Keys)}]");
        return true;
    }

    public bool Delete(string id)
    {
        bool removed = _store.Remove(id);
        if (removed) Console.WriteLine($"[DELETE] id={id}");
        return removed;
    }

    public void PrintAll()
    {
        Console.WriteLine($"\n[STORE] {_store.Count} document(s):");
        foreach (var (id, doc) in _store)
        {
            Console.Write($"  {id}: {{ ");
            Console.Write(string.Join(", ", doc.Select(kv => $"{kv.Key}: {kv.Value}")));
            Console.WriteLine(" }");
        }
    }
}

// Usage — user profile store (like MongoDB users collection)
var store = new DocumentStore();

// v1 schema — basic fields
string id1 = store.Insert(new Dictionary<string, object>
{
    ["name"]  = "Alice",
    ["email"] = "alice@example.com",
    ["role"]  = "admin"
});

string id2 = store.Insert(new Dictionary<string, object>
{
    ["name"]  = "Bob",
    ["email"] = "bob@example.com",
    ["role"]  = "user"
});

string id3 = store.Insert(new Dictionary<string, object>
{
    ["name"]  = "Charlie",
    ["email"] = "charlie@example.com",
    ["role"]  = "user"
});

// Schema-free update — add new field without ALTER TABLE
store.Update(id1, new Dictionary<string, object>
{
    ["preferences"] = new { theme = "dark", language = "en" },
    ["last_login"]  = DateTime.UtcNow.ToString("yyyy-MM-dd")
});

// Another document with completely different fields — no problem
store.Insert(new Dictionary<string, object>
{
    ["name"]       = "ServiceAccount",
    ["api_key"]    = "sk-1234",
    ["rate_limit"] = 1000
});

store.PrintAll();

// Find all users with role = "user"
Console.WriteLine("\n[FIND] role = 'user':");
var users = store.FindByField("role", "user");
foreach (var u in users)
    Console.WriteLine($"  {u["name"]} ({u["email"]})");

// Output:
// [STORE] 4 document(s):
//   doc_1: { name: Alice, email: alice@example.com, role: admin, _id: doc_1, preferences: ..., last_login: 2026-04-19 }
//   doc_2: { name: Bob, email: bob@example.com, role: user, _id: doc_2 }
//   doc_3: { name: Charlie, email: charlie@example.com, role: user, _id: doc_3 }
//   doc_4: { name: ServiceAccount, api_key: sk-1234, rate_limit: 1000, _id: doc_4 }
//
// [FIND] role = 'user':
//   Bob (bob@example.com)
//   Charlie (charlie@example.com)
```

### SQL vs Document Store Trade-offs
```
Adding a new field in SQL:
  ALTER TABLE users ADD COLUMN last_login DATETIME;
  → Locks the table during migration (minutes on large tables)
  → All existing rows get NULL for the new column

Adding a new field in Document Store:
  store.Update(id1, new { last_login = "2026-04-19" });
  → Only the updated document has the field
  → Other documents unchanged — no migration needed

When SQL wins:
  → Query across multiple related tables: JOINs are powerful
  → Data integrity: schema prevents storing wrong-shaped data
  → Complex aggregations: GROUP BY, window functions

When Document Store wins:
  → Rapidly evolving schema (early-stage product)
  → Each "row" has a different shape (user profiles, product catalogs)
  → Nested data naturally stored together (order + line items in one doc)
```

**Complexity:** Insert O(1), FindById O(1), FindByField O(n) without index, Update O(fields)

## Complexity Summary

| Operation | Without Index | With Index | Notes |
|-----------|--------------|------------|-------|
| Table scan (FindByEmail) | O(n) | O(log n) | Index = SortedDictionary / B-Tree |
| Insert with index | O(1) | O(log n) | Index write overhead |
| WAL Commit / Rollback | O(ops) | O(ops) | n = operations in transaction |
| Queue Enqueue / Process | O(1) | O(1) | DLQ move is also O(1) |
| Document FindByField | O(n) | O(log n) | With secondary index |

## What Interviewers Test

```
1. Indexing — why is SortedDictionary O(log n)?
   → Balanced BST — same structure as database B-Tree
   → log₂(1,000,000) ≈ 20 comparisons vs 500,000 for full scan
   → Write trade-off: every insert updates all indexes

2. WAL — what happens if the server crashes after logging but before commit?
   → On restart, DB scans WAL
   → Incomplete transactions (no COMMIT marker) → UNDO
   → Completed transactions → REDO (re-apply to data pages)

3. Message Queue — what is idempotency and why does it matter?
   → At-least-once delivery means the same message may arrive twice
   → Idempotent handler: processing twice = same outcome as once
   → Pattern: check if already processed before acting

4. DLQ — what do you do with messages in the DLQ?
   → Alert on-call engineer (CloudWatch alarm on DLQ depth)
   → Inspect payload to find bug
   → Fix bug, then replay DLQ to main queue

5. Document Store — what is the main downside of no schema?
   → No enforcement — "name" could be a string or an int depending on caller
   → Application must validate data shape
   → SQL schema = database enforces correctness for you
```
