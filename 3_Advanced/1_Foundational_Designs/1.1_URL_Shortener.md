# URL Shortener (bit.ly)

---

## Problem Statement

Long URLs are unwieldy in tweets, SMS, print, and QR codes. A URL shortener maps a long URL to a short, memorable code and redirects users transparently.

```
User pastes:   https://www.example.com/products/electronics/laptops?brand=dell&model=xps-15&color=black&ref=homepage
Service gives: https://sho.rt/2bX9kP

On visit:      GET sho.rt/2bX9kP  →  302 Redirect  →  original URL
```

---

## Requirements

### Functional
```
1. Shorten a long URL → unique short code (7 characters)
2. Redirect short code → original URL (< 10ms p99)
3. Optional: custom alias  ("sho.rt/my-brand" instead of "sho.rt/2bX9kP")
4. Optional: URL expiry (TTL in days)
5. Analytics: click count, clicks by country, clicks over time
6. Delete / deactivate a short URL
```

### Non-Functional
```
Scale:        100M new URLs/day  (high write)
              1B redirects/day   (10:1 read:write)
Availability: 99.99% (4 nines) — redirect failures = broken links everywhere
Latency:      Redirect < 10ms p99 (users expect instant)
Durability:   Short URLs must never silently disappear
Storage:      5 years retention by default
```

---

## Capacity Estimation

```
Writes:
  100M URLs/day ÷ 86,400s = ~1,160 writes/sec (peak ~3×: 3,500/s)

Reads:
  1B redirects/day ÷ 86,400s = ~11,600 reads/sec (peak ~35,000/s)

Storage (5 years):
  100M * 365 * 5 = 182.5B URLs
  Per record: ~500 bytes (code + long URL + metadata)
  Total: 182.5B * 500B ≈ 91 TB

Cache:
  80% traffic hits 20% of daily URLs (Pareto)
  20% of 100M = 20M URLs * 500B ≈ 10 GB RAM
  → fits in a single Redis node; easily sharded

Bandwidth (reads):
  35,000 req/s * 500B = ~17 MB/s → negligible
```

## API Design

```
POST /shorten
  Body: { "url": "https://...", "alias": "my-brand", "ttlDays": 30 }
  Response 201: { "shortUrl": "https://sho.rt/2bX9kP", "code": "2bX9kP", "expiresAt": "..." }
  Response 409: alias already taken
  Response 400: invalid URL / alias format

GET /{code}
  Response 302: Location: https://original-url.com   (temporary redirect)
  Response 301: Location: https://original-url.com   (permanent — browser caches)
  Response 404: code not found
  Response 410: URL expired (Gone)

GET /stats/{code}
  Response 200: { "clicks": 1234, "byCountry": {...}, "createdAt": "..." }

DELETE /{code}
  Response 204: deactivated
  Response 403: not your URL
```

## Core Algorithm: Base62 + Auto-Increment ID

### Why not MD5 / SHA hash?
```
MD5("https://example.com") = "9f6290f4436f3f3f..." → take first 7 chars = "9f6290f"

Problems:
  → Collisions: different URLs can map to same 7-char prefix
  → Collision handling adds complexity (check DB, re-hash with salt)
  → Non-deterministic: same URL hashed twice = same short code
    (good for dedup but means you need to check before every insert)
```

### Why Base62 + counter? (industry standard: bit.ly, TinyURL)
```
Auto-increment ID:  10001, 10002, 10003 ...
Base62 encode:       "2Bx", "2By", "2Bz" ...

Base62 alphabet:  0-9 (10) + a-z (26) + A-Z (26) = 62 symbols
7 characters:     62^7 = 3.52 trillion unique codes
At 100M/day:      3.52T ÷ 100M = 35,200 days ≈ 96 years of unique IDs

Advantages:
  → Zero collision by construction (each ID is unique)
  → Decode(Encode(id)) = id → bidirectional
  → Short: 7 chars vs 32-char MD5
  → Predictable length growth
```

### ID Generation at Scale
```
Single DB auto-increment:
  Works at low scale; bottleneck at high write rate
  
Distributed ID with range pre-allocation (Flickr-style):
  Each server claims a range: Server A gets 1–1000, Server B gets 1001–2000
  Atomic: UPDATE id_generator SET next_id = next_id + 1000 WHERE type='url'
  → No coordination per URL; range claimed once per 1000 URLs
  
Redis INCR:
  INCR url_counter → atomic, ~100K ops/sec per Redis instance
  Shard across Redis nodes if needed
  
Snowflake-style (Twitter):
  64-bit ID: timestamp(41) + datacenter(5) + worker(5) + sequence(12)
  Globally unique, time-ordered, no coordination
```

## Database Design

```sql
urls (
  id           BIGINT        PRIMARY KEY,   -- auto-increment
  short_code   VARCHAR(10)   UNIQUE NOT NULL,
  long_url     TEXT          NOT NULL,
  user_id      BIGINT,                      -- nullable for anonymous
  created_at   TIMESTAMP     DEFAULT NOW(),
  expires_at   TIMESTAMP,                   -- NULL = never expires
  is_active    BOOLEAN       DEFAULT TRUE,
  click_count  BIGINT        DEFAULT 0
)

-- Access pattern: almost always by short_code, never by id
CREATE INDEX idx_urls_short_code ON urls(short_code);

-- For analytics (separate table, append-only)
clicks (
  id           BIGINT        PRIMARY KEY,
  short_code   VARCHAR(10)   NOT NULL,
  clicked_at   TIMESTAMP     DEFAULT NOW(),
  country      CHAR(2),
  referrer     TEXT,
  user_agent   TEXT
)
CREATE INDEX idx_clicks_code_time ON clicks(short_code, clicked_at);
```

### Database Choice
```
Write path (create URL):
  Needs uniqueness constraint on short_code → relational DB (PostgreSQL)
  Or: distributed KV store (DynamoDB) with conditional write (IF NOT EXISTS)

Read path (redirect):
  Pure key lookup: short_code → long_url
  → Both SQL and NoSQL work; NoSQL is faster for pure lookups
  → In practice: PostgreSQL with read replicas handles 35K reads/sec easily

Separation:
  Write: Primary PostgreSQL
  Read:  Read replicas (3–5x fan-out) + Redis cache (handles 99% of traffic)
  Analytics: ClickHouse / BigQuery (append-only, time-series queries)
```

## Caching Strategy

```
Access pattern: 80% of clicks on 20% of URLs (Pareto)
Cache: Redis, key = short_code, value = long_url, TTL = 24h

Read path with cache:
  1. Check Redis: GET short_code
  2a. HIT  → return long_url (< 1ms, 99% of traffic)
  2b. MISS → query PostgreSQL, store in Redis, return

Cache invalidation:
  URL deleted/deactivated → DEL short_code from Redis
  URL expired → let TTL expire naturally (no action needed)
  URL updated → DEL then re-SET (write-through)

Cache sizing:
  10 GB RAM → 10B ÷ 500B = 20M entries
  = 20% of daily new URLs → covers peak traffic

Eviction policy: LRU (allkeys-lru)
  → Least recently accessed URLs evicted first
  → Guarantees hot URLs stay cached

CDN caching (for 301 permanent redirects):
  Browser/CDN caches 301 → no request reaches server for repeat visitors
  Trade-off: can't track clicks or revoke without cache-busting
  → bit.ly uses 301 to reduce load but tracks clicks server-side first
```

## Analytics

```
Naive approach: UPDATE urls SET click_count = click_count + 1 WHERE code = ?
  Problem: write amplification — every redirect causes a DB write
  At 35K reads/sec → 35K writes/sec → DB thrash

Better: Write-behind (async)
  1. Redirect returns immediately (fast path)
  2. Publish click event to Kafka/SQS: {code, timestamp, country, referrer}
  3. Consumer aggregates and flushes to ClickHouse every 5 seconds
  4. Dashboard queries ClickHouse (not the URL DB)

In-memory buffering:
  Redis INCR click:{code} → atomic, cheap counter per code
  Background job flushes Redis counters → persistent storage every 60s
  Loss risk: at most 60 seconds of counts on Redis failure (acceptable)

Real-time dashboard:
  Kafka → ClickHouse (columnar, time-series queries)
  Query: "clicks per hour for code X in last 7 days" → sub-second on ClickHouse
```

## Custom Aliases

```
"sho.rt/my-brand" instead of "sho.rt/2bX9kP"

Constraints:
  Max length: 20 chars
  Allowed: [a-zA-Z0-9_-]
  Reserved: "api", "admin", "stats", "health", "login" (block list)
  Uniqueness: same DB unique constraint

Flow:
  1. Validate format (regex) and not in blocklist
  2. INSERT with short_code = alias (instead of Base62(id))
  3. If duplicate → 409 Conflict

Namespace collision:
  "sho.rt/2bX9kP" — auto-generated codes are 7 chars, Base62 only
  Custom aliases can be longer (up to 20)
  → Prefix or length difference avoids overlap with auto-generated codes
```

## URL Expiry

```
Store expires_at timestamp in DB.

On redirect:
  IF now() > expires_at → return 410 Gone
  IF is_active = false → return 410 Gone

Background cleanup job:
  Every hour: DELETE FROM urls WHERE expires_at < NOW() AND expires_at IS NOT NULL
  Run during off-peak; batch deletes (LIMIT 10000 per iteration)

Cache TTL alignment:
  SET short_code longUrl EX <seconds_until_expiry>
  → Cache expires at same time as DB record (no stale cache serving dead URLs)
```

## Redirect Type: 301 vs 302

```
301 Permanent Redirect:
  Browser caches permanently → no future requests for this URL
  Pro: reduces server load dramatically
  Con: can't track clicks (browser skips server), can't revoke easily

302 Temporary Redirect:
  Browser always asks server → every click hits your server
  Pro: accurate analytics, can revoke/change destination anytime
  Con: more server load

Industry choice:
  bit.ly uses 301 (load reduction) but instruments clicks server-side
    before issuing redirect — logs click then 302s internally
  For analytics-heavy use cases: 302
  For CDN-friendly static links: 301
```

## Scaling Strategy

```
Layer 1 — Cache (handles ~99% of read traffic):
  Redis cluster, 10GB per shard, LRU eviction
  Horizontal: add shards as data grows

Layer 2 — Application servers:
  Stateless; scale horizontally behind load balancer
  Write servers separate from read servers (different scale needs)

Layer 3 — Database:
  PostgreSQL primary for writes (1,160/s is very manageable)
  3–5 read replicas for redirect lookups
  Partition urls table by short_code prefix if > 10TB

Layer 4 — ID generation:
  Redis INCR on dedicated counter service, or Snowflake worker per app server

Layer 5 — Analytics:
  Kafka for click event stream
  ClickHouse for time-series analytics queries
  S3 for raw event archive (compliance, replay)

Layer 6 — CDN:
  Static assets and API responses for popular URLs
  Edge PoPs reduce latency for global users
```

## Common Pitfalls

```
1. Hash collisions with MD5/SHA truncation
   → Use auto-increment + Base62; guaranteed unique

2. 301 caching breaking analytics
   → Use 302 if clicks matter; use 301 only for static, uncounted links

3. Cache serving expired URLs
   → Align Redis TTL with expires_at; delete cache key on deactivation

4. Hotspot on ID generator
   → Pre-allocate ranges per server; don't serialize every URL creation
      through one DB row

5. No rate limiting on POST /shorten
   → Spam bots create millions of short URLs pointing to malicious sites
   → Rate limit by IP + API key; scan long URLs with Google SafeBrowsing API

6. Predictable sequential codes (security)
   → 2bX, 2bY, 2bZ → attacker enumerates all URLs
   → Shuffle the Base62 alphabet or add random offset to ID before encoding
   → Or use a hash of (id + secret salt) instead of raw ID

7. Single point of failure on redirect
   → Redirect path must be multi-region; a dead redirect = broken links globally
   → Deploy read replicas + Redis in multiple regions; write to primary only
```

## Interview Answer Framework

1. **Requirements:** 100M writes/day, 1B redirects/day, 99.99% availability, < 10ms redirect
2. **Algorithm:** auto-increment ID → Base62 encode → 7-char code; 62^7 = 3.5T codes → 96 years
3. **Write path:** POST → validate → Redis INCR → Base62(id) → INSERT DB → return shortUrl
4. **Read path:** GET /code → Redis lookup → HIT: 302 redirect; MISS: DB read → cache → redirect
5. **Cache:** Redis LRU, 10GB covers 20% of URLs = 80% of traffic; TTL = min(24h, expires_at)
6. **Database:** PostgreSQL primary + 3 read replicas; index on short_code; analytics to ClickHouse
7. **301 vs 302:** 302 for analytics accuracy; 301 for cache-friendly static links
8. **Custom aliases:** same unique constraint; validate format + blocklist; max 20 chars
9. **Expiry:** expires_at in DB; check on redirect; return 410; cache TTL = seconds_until_expiry
10. **Scale:** stateless app servers + load balancer; Redis cluster; read replicas; Kafka for async analytics
