# Search Autocomplete (Typeahead)

---

## What Are We Building?

You know how when you type "appl" into Google, it instantly shows "apple", "apple watch",
"application" — before you even finish typing? That's autocomplete (also called typeahead).

```
User types: "appl"
System shows (within 100ms):
  1. apple          ← searched 10 million times
  2. apple watch    ← searched 8 million times
  3. apple store    ← searched 6 million times
  4. apple id       ← searched 4 million times
  5. application    ← searched 3 million times
```

The challenge: do this in under 100ms for millions of users typing at the same time.

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Return the top 5–10 best completions for whatever prefix the user has typed so far
2. Show most-searched completions first (not alphabetical)
3. Update suggestions when new trends emerge (e.g. a new iPhone announcement)
4. Hide offensive or banned words
5. Handle multi-word searches ("new york ti" → "new york times")
```

### Non-Functional (scale and speed)
```
Users:    10 million people use this per day
Traffic:  Each user types ~5 keystrokes per search, ~10 searches per session
          10M users × 10 searches × 5 keystrokes = 500M requests per day
          → That's about 5,800 requests every single second (peak: ~17,000/s)

Speed:    Must respond in under 100ms — humans notice lag above that
Storage:  About 1 billion unique search terms × 30 characters each ≈ 30 GB
          → This fits in RAM on a single powerful server
```

---

## The Core Idea: How Do We Find Completions Fast?

### Option 1: Just Store Every Prefix (Simple but Doesn't Scale)

You could build a giant dictionary:
```
"a"     → [amazon, apple, airbnb, ...]
"ap"    → [apple, application, app store, ...]
"app"   → [apple, application, app store, ...]
"appl"  → [apple, application, ...]
"apple" → [apple, apple watch, apple store, ...]
```

**Problem:** 1 billion words × average 30 characters per word = 30 billion dictionary entries.
That's way too much memory. ❌

---

### Option 2: Trie — The Right Answer

A **Trie** (pronounced "try") is a tree where each node represents one character.
Words that share a starting sequence share the same path in the tree.

Think of it like a phone tree menu:
```
Press A for...
  Press P for...
    Press P for...
      Press L for...
        Press E → "apple" ✓
               → Press space, then W for "apple watch" ✓
               → Press space, then S for "apple store" ✓
      Press T for → "apt" ✓
```

This way "apple" and "application" SHARE the nodes for A → P → P → L.
You don't store duplicate prefixes. Much more efficient. ✅

**Visual:**
```
root
 └── a
      └── p
           └── p
                └── l
                     └── e  ← "apple" ends here
                     |    └── (space)
                     |         └── w → a → t → c → h  ← "apple watch"
                     |         └── s → t → o → r → e  ← "apple store"
                     └── i → c → a → t → i → o → n    ← "application"
```

To find all completions for "appl": walk down A → P → P → L, then collect
everything below that node. ✅

---

## The Big Problem With a Basic Trie

Even with a trie, finding completions for "appl" means:
1. Walk to the "l" node — fast ✅
2. Search the ENTIRE subtree below "l" to collect all words — could be millions of words ❌
3. Sort them by frequency and take the top 5 — even slower ❌

**Solution: Pre-compute the top 5 at every single node.**

Instead of searching when someone types, compute the answers in advance and store them
directly in each node.

```
Each trie node stores:
  - The character it represents
  - Links to child nodes (next characters)
  - topK: the top 5 most-searched completions in this subtree ← KEY IDEA
```

Example:
```
Node "appl" stores topK = [
  ("apple",       10M searches),
  ("apple watch",  8M searches),
  ("apple store",  6M searches),
  ("apple id",     4M searches),
  ("application",  3M searches),
]
```

Now when a user types "appl":
- Walk to the "l" node → instant (4 steps)
- Read its topK list → instant (already computed)
- Return the 5 results → done!

**Lookup time: O(L) where L = number of characters typed. For "appl" = 4 steps. Always fast.**

---

## How Do We Keep topK Updated?

Every time a word's frequency changes (e.g., "apple watch" gets searched 1 more time):
1. Walk from root down to that word's leaf node
2. At EVERY node along the way, update its topK list

```
"apple watch" gets more popular:
  Update root.topK
  Update root['a'].topK
  Update root['a']['p'].topK
  Update root['a']['p']['p'].topK
  Update root['a']['p']['p']['l'].topK
  Update root['a']['p']['p']['l']['e'].topK
  ... all the way to the leaf
```

This is done in a background process (not while users are searching), so
it doesn't slow down lookups.

---

## Ranking: Why Does "apple" Come Before "application"?

The simplest rule: **show the most-searched term first.**

```
Basic ranking: rank = how many times this term was searched in the last 30 days
  "apple"        → 10 million searches → rank 1
  "apple watch"  → 8 million searches  → rank 2
  "application"  → 3 million searches  → rank 5
```

More sophisticated ranking adds:
```
Freshness boost: recent surges bubble up faster
  e.g., "iPhone 16" suddenly searched 100× more this week → boost it to the top

Personalization: your own past searches rank higher for you
  If YOU always search "apple developer", it appears higher in YOUR autocomplete
  (stored in a separate user history, not in the trie itself)

Trending boost: if a term's search rate is growing fast, boost it × 2
```

**For an interview: "frequency, with recency decay" is a complete answer.**

## How Does the System Learn What to Suggest?

The trie isn't manually built — it learns from what real users search.

```
Step 1: Log every search
  User searches "apple watch" → record {term="apple watch", timestamp, userId, region}

Step 2: Aggregate logs every hour (batch job)
  Count: "apple watch" was searched 8 million times in the last 7 days
  Count: "apple store" was searched 6 million times
  Output: list of (term, frequency) pairs

Step 3: Build a brand-new trie from scratch (offline, not on the live system)
  Read all (term, frequency) pairs
  Insert each term into the new trie, computing topK at every node
  This new trie is built on a separate server, not the one serving users

Step 4: Swap the new trie into production atomically
  Like swapping a restaurant menu — one moment it's the old one, next it's the new one
  No downtime, no half-updated state
```

This is called a **blue-green deployment** of the trie.
The result: search trends from the last hour are reflected in suggestions every ~1 hour.

## Caching: The Speed Booster

Even with a fast trie, 17,000 requests/second is a lot. Here's the key insight:

**Most users type the same popular prefixes.** "a", "ap", "app", "appl" are typed
millions of times per hour. We don't need to look them up in the trie every time.

```
Cache (Redis — a super-fast in-memory store):
  Key:   "appl"
  Value: ["apple", "apple watch", "apple store", "apple id", "application"]
  TTL:   10 minutes (then refresh from trie)
```

With a cache of the top 1 million popular prefixes:
- **95% of requests** are answered from cache (< 1ms)
- Only **5% of requests** actually need to touch the trie server

```
Without cache: trie server handles 17,000 req/s
With cache:    trie server handles 850 req/s   (5% of 17,000)
               → trie server is 20× less loaded
```

**Cache invalidation:** when the trie is rebuilt hourly, flush the cache.
There's a brief 30-second period where everything misses and hits the trie directly —
acceptable tradeoff.

---

## Debouncing: Don't Send a Request on Every Keystroke

If a user types "apple" (5 characters), without any throttling, 5 separate requests fire:
```
User types: a  → request 1
            ap → request 2
            app → request 3
            appl → request 4
            apple → request 5
```

**Debouncing** means: wait until the user pauses typing (100–200ms) before sending.

```
User types: a, p, p, l, e  (fast typing, no pauses)
→ Only ONE request sent for "apple"
→ 80% fewer requests
```

Also: if a request for "app" is still in-flight when the user types "appl", cancel the
first request — we don't need "app" results anymore.

## Filtering Offensive Words

We don't want banned words appearing as suggestions.

```
Two layers of protection:

1. At insert time (when building the trie):
   → Skip any term on the blocklist
   → It never enters the trie at all

2. At query time (when serving results):
   → Double-check the top-K results against the blocklist
   → Filter out any that slipped through (blocklist may have been updated since last rebuild)
```

The blocklist is maintained by a human review team. When someone reports an offensive
completion, a reviewer adds it to the blocklist and triggers a trie rebuild.

## Scaling to Google-Level Traffic

### One Server (Simple Systems)

The entire trie (~100 GB) fits in RAM on a single powerful server.
Add a Redis cache in front. This handles millions of users.

**Problem:** if that one server goes down, autocomplete is dead for everyone (single point of failure).

### Multiple Servers (Google Scale)

**Sharding by prefix:** split the trie into parts, each on a different server.

```
Server A handles: prefixes starting with a, b, c, d, e, f
Server B handles: prefixes starting with g, h, i, j, k, l, m
Server C handles: prefixes starting with n, o, p, q, r, s
Server D handles: prefixes starting with t, u, v, w, x, y, z

User types "ap" → router sends to Server A
User types "ne" → router sends to Server C
```

**Problem with alphabetical sharding:** letters like "e" and "s" are far more common
than "x" and "z", so Server A and C get overloaded.

**Better solution:** assign prefix ranges based on actual traffic, not alphabet.
A routing table (stored in ZooKeeper/etcd — a distributed config store) tells
each client which server to ask for which prefix.

**Read replicas:** each shard has 3 copies. All 3 can answer queries.
When the trie is rebuilt, the primary copy is updated, and the replicas sync automatically.

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ No debouncing
   → 5× unnecessary requests per word typed, kills your servers
   ✅ Always debounce at 100–150ms; cancel previous in-flight request

❌ Rebuilding the trie on every single search
   → At 5,800 searches/sec, you'd have 5,800 trie rebuilds/sec — impossible
   ✅ Batch-aggregate logs hourly; patch only trending terms every 5 minutes

❌ Storing user history inside the trie
   → One trie per user = billions of tries = impossible memory requirement
   ✅ Keep global trie; blend with user history separately at serve time

❌ Cache TTL too long
   → After a trie rebuild, cache still serves stale (old) results
   ✅ Flush cache when trie is swapped; or align TTL with rebuild schedule

❌ Forgetting to filter at insert time
   → Offensive terms enter the trie; you only catch them at query time (too late for edge cases)
   ✅ Block at both insert time AND query time for defense in depth

❌ One trie server, no replicas
   → Single point of failure; also a hot spot under high traffic
   ✅ At least 3 read replicas per shard

❌ Treating multi-word prefixes as separate words
   → "new york ti" breaks if trie only stores single words
   ✅ Index the entire phrase "new york times" as one key in the trie
```

## How It All Fits Together (Full Architecture)

```
User's browser
  │
  │  Types "appl" (after 100ms debounce)
  ▼
Load Balancer
  │
  ▼
API Server
  │
  ├─── Check Redis cache for "appl" ──→ HIT (95% of time) → return instantly
  │
  └─── Cache MISS (5% of time):
         │
         ▼
       Trie Server (the right shard for "a...")
         │  Walk: root → a → p → p → l
         │  Read: node.topK = [apple, apple watch, apple store, apple id, application]
         ▼
       Return results → store in Redis cache → return to user

Background (runs every hour):
  Search logs → Spark aggregation job → new trie built offline → swap into production → flush cache
```

## Tools Used in Real Systems

```
Trie / prefix index:
  Custom in-memory trie    What Google/Bing build; most control
  Redis ZRANGEBYLEX        Built-in sorted set that supports prefix queries
  Elasticsearch            Good for typo-tolerant search; overkill for pure prefix

Log aggregation:
  Apache Spark / Kafka     Batch and streaming aggregation of search events

Caching:
  Redis                    Prefix → top-K completions, TTL 10 minutes

Routing metadata:
  ZooKeeper / etcd         Stores which trie shard owns which prefix range
```

## How to Answer This in an Interview (Step by Step)

```
1. State the goal:
   "Return top-5 completions for a typed prefix in under 100ms at 10M DAU scale."

2. Choose the data structure:
   "I'd use a Trie with pre-computed topK at every node. This makes lookup O(L)
   where L is the number of characters typed — no subtree traversal needed."

3. Explain insert:
   "When a term's frequency changes, we walk root-to-leaf and update each node's
   topK list. O(L × K) per update, done in a background job, not on live queries."

4. Explain ranking:
   "Sort topK by frequency × recency decay. Personalization is blended at serve
   time from a separate user history store — not stored in the trie."

5. Explain the data pipeline:
   "Search logs → hourly Spark aggregation → new trie built offline → hot-swapped
   into production. Top-100 trending terms get incremental patches every 5 minutes."

6. Add the cache:
   "Redis caches prefix → top-K results with a 10-minute TTL. With 1M cached
   prefixes, 95% of traffic never hits the trie. Cache is flushed on trie rebuild."

7. Add client optimizations:
   "Debounce at 100ms after last keystroke. Cache seen prefixes in the browser
   for the session. Together these cut server requests by ~60-80%."

8. Talk about filtering:
   "Blocklist is applied at insert time (terms never enter the trie) and re-checked
   at query time as a safety net."

9. Scale it out:
   "Shard the trie by prefix range across 4+ servers, 3 read replicas each.
   A routing table in ZooKeeper tells clients which shard to hit."
```
