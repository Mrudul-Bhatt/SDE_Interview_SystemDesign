# Search Autocomplete (Typeahead)

---

## Problem Statement

As a user types in a search box, the system returns up to 5–10 suggested completions ranked by relevance — within 100ms of each keystroke.

```
User types: "appl"
System returns:
  1. apple          (freq: 10M)
  2. apple watch    (freq: 8M)
  3. apple store    (freq: 6M)
  4. apple id       (freq: 4M)
  5. application    (freq: 3M)

User types one more char: "apple "
System returns:
  1. apple watch    (freq: 8M)
  2. apple store    (freq: 6M)
  3. apple id       (freq: 4M)
  4. apple iphone   (freq: 2M)
  5. apple support  (freq: 1M)
```

---

## Requirements

### Functional
```
1. Return top-K (5–10) completions for a given prefix
2. Results ranked by relevance (frequency, recency, personalization)
3. Results update as new search trends emerge (near-real-time, not instant)
4. Filter offensive / blocklisted terms
5. Support multi-word prefixes ("new york ti" → "new york times")
6. Optional: personalized results (your past searches weighted higher)
```

### Non-Functional
```
Scale:     10M DAU; each session ≈ 10 searches × avg 5 keystrokes = 50 prefix queries/session
           → 10M × 50 = 500M prefix queries/day → ~5,800 req/s (peak 3× = ~17,000/s)
Latency:   < 100ms end-to-end (user perceives lag above 100ms)
           Trie lookup target: < 5ms; cache hit: < 1ms
Freshness: top results reflect last hour's trends (not last year's)
Storage:   1B unique search terms × avg 30 bytes = ~30 GB for trie (fits in RAM)
```

---

## Capacity Estimation

```
Unique search terms: ~1B (Google-scale)
Average term length: 30 chars
Trie storage (rough):
  Each node: char key + children map + topK list
  ≈ 100 bytes/node × 1B nodes = ~100 GB (distributed across machines)
  Cached top-K per node: 5 completions × 50 bytes = 250 bytes/node

Top-K cache (Redis):
  Popular prefixes (top 1M) × avg 5 results × 100 bytes ≈ 500 MB
  → Fits in single Redis node; 95%+ of traffic served from here

Query rate:
  5,800 req/s average; peak ~17,000 req/s
  → 3 API servers + Redis handles comfortably
```

## Core Data Structure: Trie

### Why Trie over alternatives?

```
Hash map (prefix → completions):
  Store every possible prefix: "a", "ap", "app", "appl", "apple" → O(1) lookup
  Problem: storage explosion — 1B terms × 30 avg length = 30B prefix entries
  → Impractical at scale

Inverted index (Elasticsearch-style):
  Edge n-gram tokenizer: "apple" → ["a","ap","app","appl","apple"]
  Good for fuzzy/typo-tolerant search
  Heavy: index is 5–10× term data size
  Better for full-text search than pure prefix autocomplete

Trie:
  Shared prefixes share nodes: "apple" and "application" share "appl" prefix
  Prefix search = walk O(L) nodes; L = prefix length
  Storage efficient for many words with common prefixes
  → Standard choice for prefix autocomplete
```

### Trie Node Structure

```
TrieNode {
  children:  Map<char, TrieNode>   // edges to next characters
  isEndOfWord: bool                // marks a complete term
  frequency:   int                 // search frequency (if end of word)
  topK:        List<Completion>    // ← KEY OPTIMIZATION: pre-computed top-K
                                   //   for ALL words in this subtree
}
```

### Naive vs Optimized Prefix Lookup

```
Naive approach:
  1. Walk to prefix node: O(L)
  2. DFS/BFS entire subtree to collect all completions: O(subtree size)
  3. Sort by frequency, take top K: O(N log N)
  Total: O(L + N log N) — too slow for a 1B-term trie

Optimized (pre-computed topK at every node):
  At insert time: walk root → leaf; at each node, update its topK list
  At query time:  walk to prefix node → return topK directly
  Total: O(L) for lookup; O(L × K) for insert (update K-sized list per level)

  Tradeoff: more memory (topK stored at every node) for much faster reads
  → Correct choice: reads >> writes for autocomplete
```

### Top-K Maintenance

```
At each node, maintain a sorted list of top-K completions in its subtree:

Insert "apple watch" with freq=150:
  root.topK         ← update with ("apple watch", 150)
  root['a'].topK    ← update
  root['a']['p'].topK ← update
  ...
  root['a']['p']['p']['l']['e'][' ']['w']['a']['t']['c']['h'].topK ← update

Each update: O(K) — remove old entry if exists, insert, re-sort, trim to K
Total insert cost: O(L × K) where L=term length, K=5 completions

Result: Search("appl") = root['a']['p']['p']['l'].topK → O(L), no traversal needed
```

## Ranking Strategy

```
Simple frequency-based (baseline):
  rank = search_count_last_30_days
  "apple" beats "application" if searched more often

Temporal decay (freshness):
  rank = frequency × e^(-λ × days_since_last_search)
  Recent surges in search (e.g., "iPhone 16") bubble up quickly

Personalization (user-specific):
  rank = global_frequency × 0.7 + user_frequency × 0.3
  Your own past searches weighted up

Boost signals:
  Trending: search rate increasing > 10×/hour → boost rank × 2
  Entity: known brand/location/person → floor rank at top-5 globally

In practice (Google/Bing):
  ML model: features = global freq, recency, user history, geo, device
  Served from pre-computed per-user top-K (personalised index)
  For interviews: frequency + recency is sufficient
```

## Data Collection Pipeline

```
How do we learn what to suggest?

1. Search logs:
   User searches "apple watch" → log {term, userId, timestamp, region}

2. Aggregation (batch, hourly):
   MapReduce / Spark job:
     GROUP BY term
     COUNT searches in last 7 days
     Output: (term, frequency) pairs

3. Trie builder:
   Read aggregated (term, freq) pairs
   Build new trie in memory
   Validate, then atomically swap into production
   → Blue-green trie swap: new trie built offline, hot-swapped in

4. Real-time updates (optional):
   Kafka stream of search events
   Lightweight counter service increments per-term count
   Every 5 minutes: flush hot terms, patch trie for top movers only
   → No full rebuild needed for incremental updates

Freshness vs consistency:
  Full rebuild: accurate but slow (1–2 hours)
  Incremental patch: fast but complex
  Google uses incremental; smaller systems use full hourly rebuild
```

## Caching Layer

```
Problem: 500M queries/day, but top-1000 prefixes account for 80% of traffic
  → Cache prefix → top-K results in Redis

Cache key:   prefix (e.g., "appl")
Cache value: JSON list of top-K completions
TTL:         10 minutes (balance freshness vs load)

Cache hit rate:
  With 1M cached prefixes, hit rate ~95%
  Only 5% of queries touch the trie server

Cache invalidation:
  When trie is rebuilt (hourly), flush all cache
  → Brief 100% miss rate for ~30 seconds while cache re-warms
  → OR: rebuild trie first, then selectively invalidate changed prefixes

Client-side cache:
  Browser caches completions for previously typed prefixes in session
  User types "appl" → result cached; backspace to "app" → cached; re-type "l" → instant
  → ~40% of prefix queries never leave the browser
```

---

## Debouncing (Client Side)

```
Problem: user types "a","ap","app","appl","apple" — 5 requests for one word

Debounce: wait N ms after last keystroke before sending request
  N = 100–200ms (Google uses ~100ms)
  User typing fast → only last prefix sent
  User pauses → sends immediately

Cancel in-flight request: if user keeps typing, cancel previous XHR

Result: 5 keystrokes → 1–2 actual API requests (60–80% reduction)
```

## Filtering Offensive Terms

```
Blocklist approach:
  Maintain set of banned terms
  At insert time: skip blocked terms → they never enter trie
  At query time: filter results that match blocklist patterns

Dynamic blocklist:
  Human review queue: reported completions → reviewed → added to blocklist
  Trie must be rebuilt / patched after blocklist update

Safe-search mode:
  Per-user setting: children's accounts get stricter blocklist
  Trie has multiple variants: safe / standard (or filter at query time)
```

## Scaling the Trie

```
Single server:
  100 GB trie fits in RAM on a single machine (128 GB server)
  Sufficient for medium scale; single point of failure

Distributed trie (for Google-scale):
  Shard by first 1–2 characters:
    Shard A: a-f
    Shard B: g-m
    Shard C: n-s
    Shard D: t-z
  Load balancer routes "ap" → Shard A, "ne" → Shard C

  Problem: hot shards (e, s are more common than x, z)
  Solution: shard by consistent hash of prefix, not alphabetically

Read replicas:
  Each shard has 3 replicas (read-heavy workload)
  Writes (trie rebuild) go to primary, replicated asynchronously

ZooKeeper:
  Tracks which shard owns which prefix range
  Clients discover correct shard on startup, cache routing table
```

## Common Pitfalls

```
1. No debouncing on client → 5× unnecessary API calls per word typed
   → Always debounce at 100–150ms; cancel in-flight requests

2. Rebuilding trie on every search event
   → Trie insert is O(L × K); at 5,800 req/s = 5,800 trie writes/sec → too expensive
   → Batch-aggregate logs; rebuild hourly or patch incrementally every 5 min

3. High-cardinality user IDs in trie for personalization
   → Don't put userId in trie → one trie per user is impossible
   → Separate user profile store; blend global trie result with user history at serve time

4. Cache TTL too long → stale results after trie rebuild
   → Align TTL with rebuild frequency; or flush cache on rebuild completion

5. Not filtering at insert time → offensive terms appear in completions
   → Apply blocklist at insert; re-filter at query time as safety net

6. Single trie server → SPOF and hot spot
   → Read replicas; distribute trie shards across machines

7. Multi-word prefix not handled
   → "new york ti" must work; trie keys are full multi-word strings, not single words
   → Index the entire phrase as one trie key
```

## Real-World Tools

```
Trie / prefix index:
  Redis ZRANGEBYLEX     Sorted sets with lex range; native prefix search
  Elasticsearch         Edge n-gram tokenizer; typo-tolerant; overkill for pure prefix
  Custom in-memory trie Best control; what Google/Bing build internally

Aggregation:
  Apache Spark / Flink  Batch/streaming aggregation of search logs
  Kafka                 Stream of search events → aggregator consumers

Caching:
  Redis                 Prefix → completions; TTL 10 min
  Memcached             Alternative; no built-in data structures

Hosting:
  ZooKeeper / etcd      Shard routing metadata
  Kubernetes            Auto-scaling trie servers
```

## Interview Answer Framework

1. **Requirements:** top-K completions per prefix, < 100ms, 10M DAU, freshness within 1 hour
2. **Data structure:** Trie with pre-computed topK at every node → O(L) lookup, not O(L + subtree)
3. **Insert optimization:** on insert, walk root → leaf updating each node's topK list; O(L × K) per insert
4. **Ranking:** frequency × recency decay; personalization as a blend at serve time, not in trie
5. **Data pipeline:** search logs → Kafka → hourly Spark aggregation → trie builder → hot-swap into prod
6. **Caching:** Redis, prefix → top-K JSON, TTL 10 min; 95% cache hit rate; flush on trie rebuild
7. **Debouncing:** client waits 100ms after last keystroke; client-side session cache for already-seen prefixes
8. **Filtering:** blocklist applied at insert time; re-checked at serve time as safety net
9. **Scaling:** shard trie by prefix range; 3 read replicas per shard; ZooKeeper for routing
10. **Freshness:** full rebuild hourly; real-time patch for top-100 trending terms every 5 minutes
