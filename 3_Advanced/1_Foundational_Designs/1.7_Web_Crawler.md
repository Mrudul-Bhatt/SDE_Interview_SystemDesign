# Web Crawler

---

## Problem Statement

A web crawler (spider) systematically browses the web, downloading pages and discovering new URLs so a search engine (or archiver, price monitor, etc.) can index content.

```
Seed URLs → Crawler → Fetches pages → Extracts links → Adds to queue → Repeat

Use cases:
  Search engines   Index billions of pages (Google, Bing)
  Price monitors   Track product prices across e-commerce sites
  Web archiving    Wayback Machine snapshots
  SEO tools        Audit site structure and broken links
  Data pipelines   Scrape structured data at scale
```

---

## Requirements

### Functional
```
1. Crawl billions of URLs starting from seed URLs
2. Extract all links from each page; add undiscovered URLs to frontier
3. Respect robots.txt (crawl delay, disallowed paths)
4. Deduplicate: never crawl the same URL twice
5. Prioritize important pages (higher PageRank / fresher content)
6. Store raw HTML + metadata (URL, timestamp, status code)
7. Re-crawl pages periodically (freshness)
```

### Non-Functional
```
Scale:        1B pages; web grows ~10% per year
Throughput:   ~1B pages ÷ 30 days = ~400 pages/sec sustained
Politeness:   ≥ 1 request/sec per domain (most sites require this)
Storage:      1B pages × 500 KB avg = 500 TB (compressed ~100 TB)
Freshness:    News pages re-crawled hourly; static pages monthly
Robustness:   Single worker failure must not lose discovered URLs
```

---

## Capacity Estimation

```
Pages to crawl:  1B
Average page:    500 KB HTML
Total storage:   500 TB raw (100 TB compressed at 5:1 ratio)
Crawl rate:      400 pages/sec

Workers needed:
  1 worker can crawl ~3 pages/sec (fetch 200ms + parse 50ms + politeness delay)
  400 pages/sec ÷ 3 = ~135 worker machines

DNS lookups:
  Each new domain → DNS query (50–200ms)
  Cache DNS per domain for 10 min → huge throughput gain

URL Frontier size:
  Discovered but unvisited URLs in memory/disk
  Estimate: ~500M URLs in queue × 100 bytes = 50 GB
  → Frontier must be on disk (Redis or distributed queue), not pure RAM
```

## URL Frontier (Heart of the Crawler)

The frontier manages which URLs to crawl next, in what order, and how fast.

### Two-Level Queue Design

```
Level 1: Front Queues (by priority)
  HIGH   ──┐
  MEDIUM ──┼──▶  URL Selector (weighted round-robin)
  LOW    ──┘

Level 2: Back Queues (one per domain — for politeness)
  example.com  queue: [/page1, /page2, /page3]
  news.com     queue: [/article/1, /article/2]
  blog.org     queue: [/post/1]

URL Selector:
  1. Pick URL from front queue (weighted by priority)
  2. Determine its domain
  3. Route to that domain's back queue

Back Queue Selector:
  1. Find a domain whose last-crawl time ≥ politeness delay ago
  2. Pop next URL from that domain's queue
  3. If all domains too recent → wait (back-pressure)
```

### Why Separate Front/Back Queues?

```
Without back queues:
  Front queue has 1000 URLs from example.com at top
  Crawler hits example.com 1000 times in 1 second → DoS → IP banned

With back queues:
  All example.com URLs go to one back queue
  Selector enforces ≥1s gap between pops from that queue
  Meanwhile, 50 other domain queues are drained normally
  → Politeness enforced per domain without slowing other domains
```

## Bloom Filter — Deduplication at Scale

```
Problem: 1B URLs seen; need to check "have I crawled this before?"
  Hash set: 1B × 100 bytes = 100 GB RAM → too expensive
  Bloom filter: 1B items, 1% false-positive rate → ~1.2 GB RAM (12× cheaper)

How it works:
  Bit array of m bits; k hash functions
  Insert(url):  set bits at h1(url)%m, h2(url)%m, ..., hk(url)%m
  Check(url):   if ALL k bits are set → "probably seen" (may be false positive)
                if ANY bit is 0       → "definitely NOT seen"

False positive: Bloom says "seen" but URL is actually new → miss a page
  At 1% FP rate: 1% of 1B new URLs (10M pages) skipped → acceptable
  No false negatives: if Bloom says "new", it IS new → no duplicate crawls

Parameters (for 1B items, 1% FP rate):
  m = 9.6B bits ≈ 1.2 GB
  k = 7 hash functions
  Formula: m = -n×ln(p) / (ln2)²  where n=items, p=FP rate
```

## robots.txt

```
Standard crawl etiquette (legal compliance in most jurisdictions):

User-agent: *
Disallow: /admin
Disallow: /private
Crawl-delay: 2

Rules:
  Disallow → skip those paths entirely
  Crawl-delay → override politeness delay with site's requested value
  Allow → whitelist within a Disallowed parent

Caching robots.txt:
  Fetch once per domain → cache for 24h
  Per-domain cache key: robots:{domain}
  Cost: 1 robots.txt fetch per domain (not per page)
```

## Spider Traps

```
Infinite URL spaces that would crawl forever:
  Calendar:  /calendar/2024/01 → /2024/02 → ... → /9999/12
  Sessions:  /page?session_id=abc123xyz (unique per visitor)
  Pagination: /search?page=1&q=x → page=2 → ... → page=999999

Detection strategies:
  1. URL depth limit:    max 5 hops from a seed URL
  2. Per-domain limit:   max 100K pages per domain
  3. URL normalization:  strip session params, lowercase, canonical form
  4. Repeated path:      /a/a/a/a/a → detect repeated segments
  5. Fingerprinting:     if page content == previously seen page → stop
```

---

## URL Normalization

```
Same page, many URL spellings → must map to one canonical form

Steps:
  1. Lowercase scheme and host:   HTTP://EXAMPLE.COM → http://example.com
  2. Remove default port:         http://example.com:80/ → http://example.com/
  3. Remove fragment:             /page#section → /page
  4. Sort query params:           ?b=2&a=1 → ?a=1&b=2
  5. Remove tracking params:      ?utm_source=email&id=1 → ?id=1
  6. Resolve relative URLs:       /about → https://example.com/about
  7. Remove trailing slash:       /about/ → /about (site-dependent)

Result: "http://example.com/page?a=1#top" and
        "HTTP://Example.COM:80/page?a=1&utm_source=x#bottom"
        → both normalize to "http://example.com/page?a=1"
        → Bloom filter sees the same URL → deduplication works
```

## DNS Caching

```
Problem: every new domain = 1 DNS lookup (50–200ms round trip)
  400 pages/sec × new domains → thousands of DNS lookups/sec

Solution: per-domain DNS cache
  Key:   domain name
  Value: IP address
  TTL:   10 minutes (match actual DNS TTL)
  Size:  ~10M distinct domains × 20 bytes = 200 MB

Without DNS cache:
  400 req/sec, 50ms DNS each = 20 seconds of DNS wait/sec → bottleneck

With DNS cache:
  ~95% cache hit after warm-up → DNS cost negligible
```

## Freshness — Re-crawl Strategy

```
Not all pages change at the same rate:

  News article published today:  re-crawl hourly
  Blog post from 2020:           re-crawl monthly
  Product price page:            re-crawl daily
  Static documentation:          re-crawl quarterly

Strategies:
  Fixed interval:    re-crawl every N days (simple, ignores content)
  Change detection:  compare content hash; if unchanged → backoff
    Unchanged 1×  → next crawl in 2× original interval
    Unchanged 5×  → next crawl in 32× interval (capped at 30 days)
    Changed       → reset to base interval (e.g., 1 day)

HTTP hints:
  Last-Modified header → use If-Modified-Since on re-crawl → 304 Not Modified
  ETag header          → use If-None-Match on re-crawl
  Cache-Control        → max-age tells you minimum wait time
```

## Distributed Crawling

```
Single machine: 3 pages/sec → too slow for the web
Solution: N crawler workers coordinated by a shared frontier

Coordination challenge:
  If two workers both pop example.com URLs simultaneously
  → Both hit example.com at same time → politeness violated

Solution: shard frontier by domain
  Hash(domain) % N_workers → worker assignment
  Worker 3 owns all example.com URLs → no cross-worker politeness issue
  Coordinator (ZooKeeper) tracks which worker owns which domain

Failure handling:
  Worker crashes → its domain URLs must be reassigned
  Frontier stored in Redis/Kafka (not RAM) → survives worker restarts
  Heartbeat: worker pings coordinator every 30s; no ping → reassign domains

Scale:
  100 workers × 3 pages/sec = 300 pages/sec (needs ~450 for 1B/month)
  Add workers linearly; ZooKeeper redistributes domain ownership
```

## Common Pitfalls

```
1. Not respecting robots.txt
   → IP banned by sites; legal issues in some jurisdictions
   → Always fetch and cache robots.txt before crawling a domain

2. No politeness delay
   → Flood origin servers → DoS → IP banned
   → Enforce minimum 1s (or Crawl-delay value) per domain

3. Keeping URL frontier in RAM only
   → Worker crash loses all discovered URLs
   → Persist frontier to Redis or distributed queue

4. Not normalizing URLs
   → /page and /page?utm=x counted as different → crawled twice
   → Apply full normalization pipeline before Bloom filter check

5. No depth limit
   → Spider traps crawl indefinitely
   → Hard limit on depth (5–10 hops from seed)

6. Fetching robots.txt on every request
   → N pages on example.com → N robots.txt fetches
   → Cache robots.txt per domain for 24h

7. Single Bloom filter node
   → All workers share one filter → network bottleneck
   → Partition Bloom filter; use probabilistic counting across workers
     or use domain-sharded approach (worker owns domain's Bloom slice)
```

## Interview Answer Framework

1. **Purpose:** download + index web pages; components = frontier, fetcher, parser, dedup filter, store
2. **URL Frontier:** two-level — front queues (by priority) + back queues (per domain for politeness)
3. **Politeness:** ≥1s between requests to same domain; respect Crawl-delay in robots.txt; back queues enforce this without blocking other domains
4. **Deduplication:** Bloom filter — 1B URLs at 1% FP rate = 1.2 GB vs 100 GB for a hash set; false positives acceptable, no false negatives
5. **robots.txt:** fetch once per domain, cache 24h; skip disallowed paths; use Crawl-delay value
6. **Spider traps:** depth limit (max 5 hops), per-domain page cap, URL normalization to strip session params
7. **URL normalization:** lowercase, remove fragments/tracking params, sort query params, resolve relative → canonical form before dedup check
8. **DNS caching:** per-domain cache, 10min TTL; eliminates 95% of DNS lookup latency
9. **Distributed:** shard frontier by hash(domain) → each worker owns a domain set → no cross-worker politeness conflict; ZooKeeper coordinates
10. **Freshness:** change-rate-based interval; exponential backoff for unchanged pages; HTTP 304 / ETag for efficient re-crawls
