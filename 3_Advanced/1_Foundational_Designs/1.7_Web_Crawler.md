# Web Crawler

---

## What Are We Building?

Imagine you're Google and you want to index every page on the internet.
You can't manually visit billions of websites. You need a program that does it automatically.

A **web crawler** (also called a spider or bot) is that program. It:
1. Visits a web page
2. Reads the HTML
3. Finds all the links on that page
4. Visits those links
5. Repeats forever

```
Start with a few pages (called "seeds"):
  https://wikipedia.org
  https://bbc.com

Visit wikipedia.org → find 500 links → add them to the to-do list
Visit bbc.com       → find 300 links → add them to the to-do list
Visit the first link from wikipedia.org → find more links → add those too
... keep going until you've visited billions of pages
```

**What is it used for?**
```
Google/Bing:    Crawl the web so they can build a searchable index
Wayback Machine: Save snapshots of websites over time
Price trackers: Check Amazon/eBay prices every hour
SEO tools:      Find broken links on your own website
Data pipelines: Collect structured data at scale
```

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Start from a few "seed" URLs and discover the entire web from there
2. From each page, extract all links and add new ones to the to-do list
3. Respect rules that websites publish about what crawlers can/can't access
4. Never crawl the same page twice (deduplicate)
5. Crawl important or frequently-changing pages first
6. Save the raw HTML of each page plus metadata (URL, time, status)
7. Re-crawl pages periodically so the index stays fresh
```

### Non-Functional (scale and reliability)
```
Scale:       1 billion pages total; the web grows ~10% per year
Speed:       Crawl ~400 pages every second to finish 1 billion in 30 days
Politeness:  Don't hammer any single website more than once per second
Storage:     1 billion pages × 500 KB each = 500 TB (compresses to ~100 TB)
Freshness:   News pages re-crawled every hour; old static pages re-crawled monthly
Reliability: If one crawler machine crashes, don't lose the list of discovered URLs
```

---

## How Much Do We Need?

Let's work out the numbers:

```
Pages to crawl:  1 billion
Each page:       ~500 KB of HTML
Total storage:   500 TB raw HTML (about 100 TB after compression)

Speed needed:
  1 billion pages in 30 days
  = 1,000,000,000 ÷ (30 × 24 × 3600 seconds)
  = ~400 pages per second

Machines needed:
  One machine can crawl ~3 pages/second
  (fetching the page takes ~200ms, parsing takes ~50ms, then we wait to be polite)
  400 pages/sec ÷ 3 pages/machine = ~135 machines

To-do list size:
  Pages discovered but not yet crawled: ~500 million URLs
  Each URL takes ~100 bytes
  500 million × 100 bytes = 50 GB
  → This is too big to keep in RAM alone; needs to be stored on disk
```

## The To-Do List (URL Frontier): The Heart of the Crawler

The **URL Frontier** is the list of pages we still need to visit. It sounds simple,
but managing it well is the hardest part of building a crawler.

### The Two Problems It Must Solve

**Problem 1: What order should we crawl in?**
Not all pages are equally important. A CNN article about breaking news is more
valuable to crawl than someone's personal blog post from 2008.

**Problem 2: How do we avoid hammering the same website too fast?**
If we have 1000 links from wikipedia.org and we visit them all in 1 second, we've
made 1000 requests per second to one server. That's basically a cyberattack (called a DoS).
We need to space out requests to the same website.

### The Solution: Two Layers of Queues

```
LAYER 1: Priority Queues (decides WHAT to crawl next)
  HIGH priority queue   ← news sites, popular pages
  MEDIUM priority queue ← normal websites
  LOW priority queue    ← obscure pages, rarely-updated content

LAYER 2: Per-Domain Queues (decides WHEN to crawl)
  wikipedia.org queue: [/page1, /page2, /page3, ...]
  bbc.com queue:       [/article/1, /article/2, ...]
  blog.org queue:      [/post/1, /post/2, ...]
```

**How the selector works:**
```
Step 1: Pick a URL from a priority queue (high priority first)
Step 2: Find out which website it belongs to (e.g., "this is from bbc.com")
Step 3: Put it in bbc.com's per-domain queue
Step 4: Only pull from bbc.com's queue when at least 1 second has passed
        since the last request to bbc.com
Step 5: Meanwhile, pull freely from other domains' queues
```

**Why this is clever:**
```
Without per-domain queues:
  We have 1000 bbc.com URLs at the top of the list
  → We fire 1000 requests to bbc.com in 1 second
  → bbc.com thinks we're attacking them → they block our IP

With per-domain queues:
  bbc.com gets at most 1 request per second
  Meanwhile, we're crawling wikipedia.org, cnn.com, etc. at the same time
  → We're still fast overall, but polite to each individual website
```

## Deduplication: Never Crawl the Same Page Twice

As the crawler runs, it discovers billions of URLs. Before adding a new URL to the
to-do list, we must check: "Have we already crawled this page?"

### The Naive Solution (Too Slow)

Store every seen URL in a hash set (like a Python `set` or C# `HashSet`).

```
Problem: 1 billion URLs × 100 bytes each = 100 GB of RAM just for this set
→ Too expensive. A server typically has 16-64 GB of RAM.
```

### The Smart Solution: Bloom Filter

A **Bloom filter** is a clever data structure that answers "have I seen this before?"
using a tiny fraction of the memory.

**Think of it like this — a simplified analogy:**

Imagine you have 100 light switches (all OFF to start). When you see a URL,
you flip 3 specific switches ON based on the URL's content.

To check if you've seen a URL:
- Look at those same 3 switches
- If ALL 3 are ON → you've probably seen it before
- If ANY is OFF → you definitely haven't seen it

```
How much memory does this save?
  Hash set:     1 billion URLs × 100 bytes = 100 GB
  Bloom filter: 1 billion URLs at 1% error rate = 1.2 GB
  → 80× smaller!
```

**The catch: false positives**
The Bloom filter can sometimes say "I've seen this" when it actually hasn't.
(Multiple URLs can flip the same switches.)

```
At 1% false positive rate:
  1% of NEW pages get incorrectly skipped → we miss 10 million pages out of 1 billion
  → Acceptable! We're still crawling 99% of pages.

Crucially: it NEVER has false negatives.
  If the Bloom filter says "this is new", it IS definitely new.
  → We never crawl the same page twice. ✅
```

## Respecting Website Rules: robots.txt

Almost every major website has a file called `robots.txt` at its root URL.
It's a set of instructions that tells crawlers what they're allowed to access.

**Example `robots.txt` for example.com:**
```
User-agent: *             ← applies to all crawlers
Disallow: /admin          ← don't crawl anything starting with /admin
Disallow: /private        ← don't crawl anything starting with /private
Crawl-delay: 2            ← wait 2 seconds between requests
```

**What we do with it:**
```
Before crawling any page on example.com:
  1. Fetch https://example.com/robots.txt  (just once per domain)
  2. Read the rules
  3. Save them in memory for 24 hours (no need to fetch again)
  4. Check every URL against the rules before crawling it

If a URL is disallowed → skip it, don't even try
If Crawl-delay = 2 → wait 2 seconds between requests to that site
```

**Why does this matter?**
```
Legally: ignoring robots.txt can get you sued in some countries
Practically: if you ignore it, the website will block your IP address
Ethically: websites spend money to run servers — be a good citizen
```

## Spider Traps: Websites That Could Crawl Forever

Some websites (accidentally or deliberately) generate infinite pages.
If your crawler falls into one, it'll crawl forever and never finish.

**Examples:**
```
Calendar website:
  /calendar/2024/01 links to /calendar/2024/02 links to /calendar/2024/03 ...
  → Infinite months, infinite pages

Session-based URLs:
  /shop?session=abc123 → new session ID for every visitor
  → Billions of "different" URLs that all show the same shop

Infinite pagination:
  /search?page=1 → /search?page=2 → /search?page=3 → /search?page=999999
```

**How we detect and escape them:**

```
1. Depth limit:
   Only follow links up to 5 hops away from a seed URL.
   Seed → page1 → page2 → page3 → page4 → page5 → STOP
   A calendar going 5,000 levels deep hits this limit immediately.

2. Per-domain page cap:
   Never crawl more than 100,000 pages from a single domain.
   Even if a site has infinite pages, we stop after 100K.

3. URL normalization (strip junk from URLs):
   /shop?session=abc123&id=1 → normalized to /shop?id=1
   /shop?session=xyz456&id=1 → also normalized to /shop?id=1
   → They look the same to the crawler → deduplicated → only crawled once.

4. Repeated path detection:
   /a/b/a/b/a/b/a looks like a loop → skip it
```

## URL Normalization: Making the Same Page Look the Same

The same web page can be reached with many different URLs:

```
http://EXAMPLE.COM/page?b=2&a=1#top
https://example.com/page?a=1&b=2
https://example.com/page?a=1&b=2&utm_source=twitter
https://example.com/page/

All four URLs show the EXACT SAME PAGE.
```

If we don't normalize, we'd crawl this page 4 times and waste resources.

**Normalization steps:**
```
1. Lowercase everything:
   HTTP://EXAMPLE.COM → http://example.com

2. Remove the default port:
   http://example.com:80/page → http://example.com/page
   (port 80 is the default for http, so it's redundant)

3. Remove the fragment (#anchor):
   /page#section → /page
   (the #section part is only used by the browser, the server ignores it)

4. Sort query parameters alphabetically:
   ?b=2&a=1 → ?a=1&b=2
   (same parameters, same page — order shouldn't matter)

5. Remove tracking parameters:
   ?a=1&utm_source=twitter → ?a=1
   (utm_source is just for analytics, doesn't change the page content)

6. Resolve relative URLs:
   The link "/about" on https://example.com → https://example.com/about

After all these steps, all four URLs above become:
   http://example.com/page?a=1

→ The Bloom filter sees one URL, not four. No duplicate crawls. ✅
```

## DNS Caching: A Hidden Performance Problem

When your crawler wants to visit `bbc.com`, before it can even connect, it must look up
what IP address `bbc.com` lives at. This is called a **DNS lookup**.

```
Problem:
  DNS lookup takes 50–200 milliseconds
  We're crawling 400 pages/second
  If every page needs a DNS lookup: 400 × 150ms = 60 seconds of DNS waiting per second
  → The crawler would be completely stuck waiting for DNS!

Solution: DNS cache
  The first time we visit bbc.com, we look up its IP: 151.101.0.81
  We save that: "bbc.com → 151.101.0.81" for 10 minutes
  Every bbc.com page we crawl in the next 10 minutes uses the cached IP → instant

Result:
  ~95% of DNS lookups are cache hits after warm-up
  DNS is no longer a bottleneck
```

## Keeping Pages Fresh: When to Re-Crawl?

The web changes constantly. If we crawled a news site last month, the articles are outdated.
If we crawled a dictionary page last year, it's probably still the same.

**Different pages change at different rates:**
```
CNN breaking news page:          changes every few minutes → re-crawl hourly
Amazon product price:            changes daily → re-crawl daily
Wikipedia article on history:    changes occasionally → re-crawl weekly
Old blog post from 2015:         rarely changes → re-crawl monthly
```

**How to decide when to re-crawl:**
```
Smart approach — exponential backoff:
  Crawl a page. Save a "content fingerprint" (hash of the HTML).
  Next day: crawl again. Compare fingerprints.
    → Page changed: re-crawl again tomorrow
    → Page unchanged: wait 2 days before next re-crawl

  Keep doubling the interval every time the page is unchanged:
    Unchanged 1 time  → wait 2 days
    Unchanged 2 times → wait 4 days
    Unchanged 3 times → wait 8 days
    Unchanged 5 times → wait 30 days (cap here)

  Page changes at any point → reset back to daily crawl

Efficient check using HTTP:
  When re-crawling, ask the server: "has this page changed since last time?"
  Server responds: "No" (HTTP 304) → don't even download the HTML again
  → Saves bandwidth for pages that haven't changed
```

## Running Multiple Crawlers in Parallel

One machine crawling 3 pages/second is too slow for the whole web.
We need ~135 machines working together.

**The challenge:**
```
If two machines both have bbc.com URLs:
  Machine A and Machine B both try to crawl bbc.com at the same second
  → bbc.com gets 2× the requests → politeness violated
```

**The solution: assign each website to exactly one machine**
```
When a new URL is discovered for bbc.com:
  Hash("bbc.com") → always assigns to Machine #7
  Machine #7 is responsible for ALL bbc.com pages
  → Only Machine #7 ever talks to bbc.com → politeness automatically maintained

If Machine #7 crashes:
  A coordinator (ZooKeeper) notices it stopped sending heartbeats
  Redistributes bbc.com (and Machine #7's other websites) to remaining machines
  The URL list was saved to disk (not RAM), so no discovered URLs are lost
```

**How the machines stay coordinated:**
```
Every 30 seconds, each machine sends a heartbeat: "I'm still alive"
If a machine misses 3 heartbeats → coordinator marks it as dead
Coordinator reassigns its work to other machines
```

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ Ignoring robots.txt
   → Website blocks your IP; legal issues in some countries
   ✅ Always fetch robots.txt first; cache it for 24 hours per domain

❌ No politeness delay
   → You're accidentally DDoS-ing websites → they block you
   ✅ Wait at least 1 second (or whatever Crawl-delay says) between same-domain requests

❌ Keeping the to-do list only in RAM
   → Machine crashes → lose all discovered but unvisited URLs → have to start over
   ✅ Store the frontier on disk (Redis or a message queue like Kafka)

❌ Not normalizing URLs before deduplication
   → /page and /page?utm_source=twitter look different → crawled twice → wasted work
   ✅ Normalize first, then check the Bloom filter

❌ No depth limit
   → Crawler falls into a calendar spider trap and crawls year 9999
   ✅ Stop following links more than 5-10 hops from a seed URL

❌ Fetching robots.txt for every single page request
   → 1 million bbc.com pages = 1 million robots.txt fetches
   ✅ Fetch robots.txt once per domain and cache for 24 hours

❌ Forgetting that the same page has many URL spellings
   → Crawl the same page 4 times because URLs look slightly different
   ✅ Normalize every URL before adding to the Bloom filter
```

## How It All Fits Together (Full Flow)

```
─── Starting up ─────────────────────────────────────────────────────────

Load seed URLs into the to-do list:
  wikipedia.org, bbc.com, cnn.com, amazon.com, ...

─── Main crawl loop (runs forever) ──────────────────────────────────────

Step 1: Pick the next URL from the to-do list
        (high priority pages first; respect per-domain timing)

Step 2: Check robots.txt for that domain
        → Disallowed? Skip this URL, go to next one

Step 3: Download the page
        GET https://bbc.com/article/123
        → Get back HTML content

Step 4: Save the HTML to storage (S3)
        {url, html, timestamp, status_code}

Step 5: Parse the HTML, find all links on the page
        <a href="/article/456"> → found new URL: bbc.com/article/456
        <a href="https://reuters.com/story/1"> → found new URL: reuters.com/story/1

Step 6: For each discovered link:
        a. Normalize it (lowercase, remove tracking params, etc.)
        b. Check Bloom filter: "Have we seen this before?"
           → Already seen: throw it away
           → New: add to the Bloom filter, add to the to-do list

Step 7: Go back to Step 1

─── Background jobs ─────────────────────────────────────────────────────

Re-crawl scheduler: looks at pages crawled > N days ago → puts them back in queue
```

## How to Answer This in an Interview (Step by Step)

```
1. Explain what a crawler does:
   "A crawler starts from seed URLs, downloads pages, extracts all links,
   and adds new links to a to-do list. It repeats this until it's crawled
   the entire web. The main challenges are scale, politeness, and deduplication."

2. Explain the URL Frontier:
   "I'd use a two-level queue. Front queues rank pages by importance (high/medium/low).
   Back queues are one per domain, and we only pop from each domain queue at most
   once per second. This way we're fast overall but polite to every individual site."

3. Explain deduplication:
   "With 1 billion URLs, a hash set would need 100 GB of RAM. Instead I'd use a
   Bloom filter — same 1 billion URLs needs only 1.2 GB. A 1% false positive rate
   means we might skip 10 million pages, which is acceptable. It never has false
   negatives, so we never crawl the same page twice."

4. Explain robots.txt:
   "Before crawling any page on a domain, fetch robots.txt once and cache it for
   24 hours. Skip any disallowed paths. Use the Crawl-delay value to override
   our default 1-second politeness delay."

5. Explain spider traps:
   "Limit crawl depth to 5 hops from a seed URL. Cap pages per domain at 100K.
   Normalize URLs to strip session IDs and tracking parameters so dynamically
   generated URLs collapse into a single canonical form."

6. Explain URL normalization:
   "Lowercase, remove default ports, strip fragments, sort query params, remove
   tracking params, resolve relative URLs. This ensures the same page always
   maps to one URL before the Bloom filter checks it."

7. Explain freshness:
   "Re-crawl frequency depends on change rate. Use exponential backoff: if a page
   is unchanged, double the wait time before checking again, up to 30 days.
   Use HTTP 304 / ETag to avoid downloading pages that haven't changed."

8. Explain distributed crawling:
   "Shard the URL frontier by domain: hash(domain) % N_workers assigns each
   website to one machine. That machine handles all pages for that domain, so
   politeness is automatically maintained without cross-machine coordination.
   A coordinator (ZooKeeper) handles reassignment when a machine crashes."
```
