# Video Streaming (YouTube / Netflix)

---

## Problem Statement

Design a system that stores, processes, and serves video content to hundreds of millions of users globally — like YouTube or Netflix. Users upload raw video; the platform transcodes it into multiple resolutions; viewers stream it on-demand with low startup latency and smooth playback.

```
Use cases:
  YouTube      User-generated content, live streaming, 500 hours uploaded/min
  Netflix      Premium VOD, globally distributed, 15% of world internet traffic
  TikTok       Short-form mobile video, vertical format, algorithmic feed
  Twitch       Live gaming streams, chat, VOD archive
  Coursera     Long-form educational video, transcripts, subtitles
```

---

## Requirements

### Functional
```
1. Users upload video files (mp4, mov, avi, etc.)
2. System transcodes to multiple resolutions: 360p, 480p, 720p, 1080p, 4K
3. Users stream videos with adaptive bitrate (ABR) based on network conditions
4. Search and browse videos by title, tag, category
5. View count, likes, comments, subscriptions
6. Resume playback from last position
7. Subtitles / closed captions
```

### Non-Functional
```
Scale:       2B active users (YouTube); 800M videos total
             500 hours of video uploaded per minute
             1B hours watched per day
Upload:      P99 upload complete < 60 minutes for a 2GB file
Latency:     Video starts playing < 2 seconds from click
             Seek to any position < 1 second
Durability:  Videos stored with 11 nines durability (S3)
Availability: 99.99% uptime for streaming; upload can tolerate brief delays
CDN:         Videos served from edge, not origin, for 95%+ of playback
```

## Upload Pipeline

```
Raw Upload Flow:
  1. Client splits video into 5MB chunks (resumable upload)
  2. Upload service receives chunks, writes to raw video blob store (S3)
  3. Upload service publishes "upload-complete" event to message queue (Kafka)
  4. Transcoding workers pick up event, begin async processing

Why chunks?
  - Resume interrupted uploads (mobile networks, large files)
  - Parallel upload of chunks to different servers
  - Client can verify each chunk independently (MD5 checksum)

Resumable upload protocol (TUS / S3 multipart):
  POST /upload/init        → returns upload_id + pre-signed chunk URLs
  PUT  /upload/{id}/chunk/{n}  → upload each chunk
  POST /upload/{id}/complete   → assemble + verify; trigger transcoding

Storage layout:
  raw/    {video_id}/original.mp4        ← original file
  hls/    {video_id}/360p/seg000.ts      ← HLS segments
  hls/    {video_id}/720p/seg000.ts
  hls/    {video_id}/manifest.m3u8       ← master playlist
  thumb/  {video_id}/thumb_{t}.jpg       ← thumbnail frames
```

## Transcoding

```
Why transcode?
  - Users upload 4K 60fps H.265; viewer on 3G can't play that
  - Multiple resolutions = adaptive bitrate switching
  - Standard format (H.264/H.265) works across all devices

Transcoding pipeline:
  Input:  raw original (any codec, any resolution)
  Output: multiple renditions (codec, resolution, bitrate)

  Renditions:
    360p   @ 400 kbps   (mobile, poor network)
    480p   @ 800 kbps
    720p   @ 2500 kbps  (default for most users)
    1080p  @ 5000 kbps
    4K     @ 16000 kbps (optional, premium)

  Worker fleet (GPU-enabled):
    - Pull job from queue
    - Download raw video from S3
    - Transcode all renditions in parallel (FFmpeg)
    - Split into 6-second HLS segments (.ts files)
    - Write segments to CDN origin (S3 + CloudFront)
    - Write master M3U8 playlist (lists all renditions + bandwidths)
    - Mark video as "ready" in metadata DB

HLS playlist structure:
  #EXTM3U
  #EXT-X-VERSION:3
  #EXT-X-STREAM-INF:BANDWIDTH=2500000,RESOLUTION=1280x720
  720p/index.m3u8
  #EXT-X-STREAM-INF:BANDWIDTH=800000,RESOLUTION=854x480
  480p/index.m3u8

Transcoding time (rough estimates):
  1-hour video → ~30 min on 4-core CPU per rendition
  Parallel renditions on GPU farm: ~10 min total
  YouTube uses 1000s of transcoders to handle 500 hrs/min
```

## Adaptive Bitrate Streaming (ABR)

```
Problem: user's bandwidth fluctuates. Fixed quality = buffering or waste.

Solution: HLS / DASH — split video into small segments; client picks quality per segment.

How it works:
  1. Client fetches master playlist (manifest.m3u8) — lists all renditions
  2. Client starts with 720p (default)
  3. After each segment download, client measures download speed
     - Speed > 5 Mbps → try 1080p next segment
     - Speed < 1 Mbps → drop to 480p next segment
  4. Buffer target: keep 15–30 seconds buffered ahead

Segment size choice:
  2-second segments: fine-grained switching, more requests
  6-second segments: balance (Netflix/YouTube default)
  10-second segments: fewer requests, slower switching

Client ABR algorithm (simplified):
  throughput = bytes_downloaded / download_time
  next_bitrate = max rendition where bitrate < throughput * 0.8  // 80% safety margin
  if buffer < 5s: emergency drop to lowest bitrate
```

## Storage Strategy

```
Raw video (original uploads):
  Storage: S3 (Glacier after 30 days if no re-transcode needed)
  Retention: permanent (original quality preserved)
  Size: ~1 GB per minute of 4K video → petabytes total

Transcoded segments:
  Storage: S3 (CDN origin)
  Replication: cross-region (us-east, eu-west, ap-southeast)
  Cache: CDN edge caches popular segments; long-tail content served from origin

Thumbnails:
  Generated at multiple timestamps during transcoding (FFmpeg -vframes)
  Stored in S3 as JPEG; served via CDN with very high TTL

Metadata DB (Cassandra):
  video_id (UUID)
  uploader_id
  title, description, tags[]
  status (UPLOADING | TRANSCODING | READY | DELETED)
  duration_seconds
  view_count (counter column — approximate)
  created_at

User watch history (Cassandra):
  partition key: user_id
  clustering key: watched_at DESC
  video_id, position_seconds, completed (bool)
  → enables "resume from position" and recommendation signals
```

## CDN Strategy

```
Popular videos (top 1%):
  - Fully cached at all CDN edge nodes
  - Cache TTL: 24 hours (segments are immutable once created)
  - HIT rate: 95%+ for popular content

Long-tail videos:
  - Not pre-seeded; fetched from origin on first request at each edge
  - Cache when requested; LRU eviction
  - Some edges may always miss → served from regional origin cache

Pre-seeding for new releases:
  - Netflix "Open Connect" appliances pushed to ISPs the day before
  - YouTube pushes viral/trending videos to edges proactively

CDN URL pattern:
  https://cdn.youtube.com/{video_id}/720p/seg042.ts
  Segments are content-addressed (immutable) → infinite cache TTL safe

Geo-routing:
  DNS returns nearest CDN PoP
  User in Tokyo → Tokyo edge → Singapore origin → S3 us-east (fallback)
```

## View Count & Analytics

```
Challenge: YouTube gets millions of concurrent views; naive DB increment would melt

Solution: buffered counting
  1. Player emits "heartbeat" event every 30 seconds to analytics service
  2. Analytics service writes to Kafka (high-throughput, no DB involved)
  3. Stream processor (Flink/Kafka Streams) aggregates counts per video per minute
  4. Batch writer flushes to Cassandra counter column every 60 seconds
  5. Cassandra atomic COUNTER UPDATE avoids race conditions

View count rules (YouTube):
  - 1 view per user per 24-hour session for short videos
  - Bots filtered by IP rate limit + user-agent + playback duration
  - "Official" count reconciled nightly via batch job against raw Kafka log

Real-time analytics (trending):
  - Kafka → Flink → ClickHouse (columnar store for fast aggregation)
  - "Trending" = views in last hour + view velocity (acceleration)
  - Trending score = views_1h * recency_weight / (age_hours + 1)^gravity
```

## Search

```
Indexing:
  When video is READY, worker publishes metadata to Elasticsearch
  Index fields: title (analyzed), description, tags, channel, category, language

Query types:
  Full-text: "funny cats 2024" → BM25 ranking
  Filter: category=Sports AND duration<5min
  Autocomplete: prefix match on title (edge n-gram tokenizer)

Ranking signals (beyond text relevance):
  - View count (log-normalized)
  - CTR from search results (click-through rate)
  - Watch time % (finished videos rank higher)
  - Recency (freshness boost for last 7 days)
  - Channel authority score
```

## Resume Playback

```
On pause / seek / close:
  Client sends: PUT /watch-progress/{videoId} { position_seconds: 342 }
  Written to: user_watch_history table (Cassandra)

On video open:
  Client fetches: GET /watch-progress/{videoId}
  Returns: { position_seconds: 342, total_duration: 1800 }
  Player seeks to position before starting ABR

Sync behavior:
  Update debounced: client sends position every 10 seconds while playing
  Not critical path: async write, no blocking
```

## Pitfalls

```
1. Single transcoding queue → head-of-line blocking
   Problem: one giant 4K upload blocks all short clips behind it
   Fix: priority queues — separate queues for <1min, 1–10min, >10min videos

2. CDN cache invalidation on re-transcode
   Problem: fixed-quality bug discovered after millions of views
   Fix: new segments get new filenames (content hash or version suffix);
        old segments expire naturally; update manifest to point to new segments

3. View count fraud / bot traffic
   Problem: inflated view counts by bots (affects monetization)
   Fix: verify-first pipeline — hold view for 48h, validate playback
        duration, unique IP, user-agent; only then confirm count

4. Hot video cold start
   Problem: new video goes viral → origin flooded before CDN caches it
   Fix: pre-warm CDN on upload completion for all trending channels;
        origin rate-limiting with queue for non-cached requests

5. Transcoding failure loses video
   Problem: transcoders crash mid-job; original lost from hot tier
   Fix: keep original in S3 permanently; transcoding is idempotent;
        on failure, message returns to queue; dead-letter queue after 3 retries

6. Player stuck buffering at 720p when user has 100Mbps
   Problem: client algorithm too conservative / fetches manifest too infrequently
   Fix: probe with a small 1080p segment first; measure speed; commit to quality

7. Seeking requires downloading from segment start
   Problem: user seeks to 45:00 in a 2-hour movie; must find the right segment
   Fix: HLS index maps segment → byte offset; player fetches segment 450 directly
        (each 6s segment is independent; no dependencies on earlier segments)
```

## Real-World Systems

```
YouTube:  Custom transcoding pipeline (Borg/GKE); custom ABR player (YT player);
          Vitess (MySQL sharding) for metadata; BigTable for watch history;
          Monarch for metrics; global anycast CDN

Netflix:  Open Connect CDN appliances at ISPs; 3 AWS regions for failover;
          Chaos Monkey; custom ABR algo (BOLA — buffer occupancy based);
          Hollow for in-memory datasets; Cassandra for viewing history

TikTok:  Short segments (1-3s) for faster start; aggressive prefetch of next video;
          ByteGrid CDN; HEVC encoding for 50% size reduction vs H.264

Twitch:   Live ingest via RTMP; Edge Transcoding Nodes (ETN) at each PoP;
          HLS with 2s segments for ~6s latency; VOD stored post-stream
```

## Interview Answer Framework

1. **Upload**: chunked resumable upload → S3 raw store → Kafka event → transcoder workers
2. **Transcoding**: GPU fleet, FFmpeg, all renditions in parallel, output HLS segments to S3
3. **ABR**: HLS manifest lists renditions; client measures throughput; switches quality per 6s segment
4. **CDN**: immutable segments → infinite TTL; 95%+ hit rate for popular; long-tail from origin
5. **Metadata**: Cassandra for video metadata + watch history; MySQL for user accounts
6. **View count**: buffered via Kafka → batch flush to Cassandra counter; bot filtering
7. **Search**: Elasticsearch; ranked by text relevance + CTR + watch time + recency
8. **Resume**: watch position stored per user+video; seek on re-open
9. **Scale**: 500 hrs/min upload → parallel transcoder fleet; 1B watch hrs/day → CDN absorbs load
10. **Pitfalls**: single transcoding queue, CDN cold start on viral, view fraud, segment seeking
