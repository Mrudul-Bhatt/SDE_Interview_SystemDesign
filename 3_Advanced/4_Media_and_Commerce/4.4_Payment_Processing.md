# Payment Processing (like Stripe or PayPal)

---

## What Are We Building?

You click "Buy" on an online store. You see the spinner for two seconds. Then "Payment successful!"
$29.99 leaves your card and arrives in the merchant's bank account two days later.

That spinner is hiding a lot of work:

```
Click "Buy"
    │
    ▼
Card details → tokenized → fraud check → bank approval → ledger entry → webhook to merchant
                                              │
                                              ▼
                                  Money actually moves 1-2 days later
                                  through banking networks
```

We're building a system like **Stripe** or **PayPal** — the layer between online businesses
and the banking system.

**Why this is harder than it looks:**
- Networks are unreliable. If the bank approves the charge but your network drops the response,
  you might charge the customer **twice**.
- Money is regulated. You can't lose it. You can't double-count it. Auditors will check.
- Card data is heavily restricted. PCI compliance has strict rules about how you store card numbers.
- Fraudsters are everywhere. About 1% of online card transactions are fraudulent attempts.
- You have to settle with banks at the end of every day and prove your books match theirs.

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Accept payment methods: credit/debit card, bank transfer (ACH/SEPA), wallets (Apple Pay, etc.)
2. Three-step lifecycle: authorize → capture → settle
3. Refunds: full or partial
4. Never double-charge a customer, even if a request is retried
5. Notify merchants (via webhooks) when payment status changes
6. Detect and block fraud
7. Handle multiple currencies; convert with fair FX rates
8. Reconcile internal ledger with bank statements daily
```

### Non-Functional (scale and reliability)
```
Scale:        Stripe handles ~300M API calls/day worldwide
Latency:      99% of authorizations finish in under 3 seconds (limited by card network)
Consistency:  Financial data must be STRONGLY consistent — no lost money, no duplicates
Durability:   Every ledger entry must survive any node failure (synchronous writes)
Uptime:       99.999% (5 nines) — downtime literally = lost revenue for merchants
Compliance:   PCI DSS Level 1 (never store raw card numbers)
              SOX (audit trail for every dollar moved)
```

## The #1 Problem: Don't Double-Charge

This is the single most important problem in payments.

```
Customer hits "Buy" — request goes to your server.
Your server forwards it to the bank.
The bank approves the $50 charge.
The bank's response is sent back...
... but the network drops it before it reaches your server.

Your server doesn't know what happened.
The customer's app sees an error and shows "Try again?"
The customer clicks Try again.

Now your server sends a SECOND request to the bank.
The bank approves ANOTHER $50 charge.

The customer just paid $100 for a $50 purchase.
```

This is a disaster. Even one instance erodes trust in your platform.

### The Solution: Idempotency Keys

Every payment request includes a **client-generated UUID** called an idempotency key.

```
Customer's first click:
  POST /charges
  Header: Idempotency-Key: abc-123-xyz
  Body: { amount: 5000, card_token: "tok_visa_4242" }

Server logic:
  1. Look up idempotency key in Redis.
  2. Found? → return the SAME response as last time. Don't touch the bank.
  3. Not found? → process normally. Save result with the key.

If the network drops the response, the customer's app retries with the SAME key:
  POST /charges
  Header: Idempotency-Key: abc-123-xyz   ← SAME key
  Body: { amount: 5000, card_token: "tok_visa_4242" }

Server: "I've seen this key. Here's the same response I sent before."
   → No second charge.
```

**Rules:**
- Same key + same params → return cached response
- Same key + different params → reject with conflict error
- TTL on key: 24 hours (after that, treat as a new request)

**Why not detect duplicates by amount + customer?** Because two legitimate $10 charges 5 seconds
apart could happen (refilling something twice). Only the idempotency key captures the user's
**intent** — "this is the same request I just tried."

## The Three-Step Payment Lifecycle

A payment isn't a single event. It's a state machine:

```
PENDING ──▶ AUTHORIZED ──▶ CAPTURED ──▶ SETTLED
              │                │
              │           [refund]──▶ REFUNDED
              │
        [void/cancel]
```

### Step 1: Authorize (Put a "Hold" on the Money)

The customer's bank locks the amount on their card. No money has moved yet — it's like a
**reservation**.

```
Real-world example: you check into a hotel.
  → Hotel runs a $500 authorization on your card on Day 1
  → Card shows the $500 as "held" but not actually charged
  → If you check out without incident, hotel CAPTURES $300 (your actual bill)
  → The other $200 falls off the hold within 7 days
```

Authorizations expire after about 7 days if not captured.

### Step 2: Capture (Actually Charge)

This triggers the real money movement. You can capture less than authorized (hotel example),
but never more.

```
Auth $500 → Capture $300 → The bank now considers $300 owed
```

### Step 3: Settle (Money Actually Arrives)

Banks don't move money instantly. They batch everything at end of day and settle overnight
("T+1" — one business day later, sometimes 2-3 for international).

```
Day 1: Customer captured for $300
Day 2 morning: Bank settlement runs → $300 minus fees arrives in merchant's account
```

### Refunds

A refund creates a NEW transaction that reverses a previous capture.

```
Original: Customer paid $100 to Merchant
Refund:   Merchant paid $100 back to Customer (via a new credit transaction)

You never modify or delete the original entry. You add an offsetting one.
This is non-negotiable for audit compliance.
```

## PCI Compliance: Don't Touch the Card Number

**PCI DSS** is the credit card industry's security standard. The most important rule:

> Never store raw card numbers in your database.

Why? If your database leaks, you've handed millions of cards to attackers. The fines and lawsuits
would destroy your company. PCI Level 1 is the strictest tier — required for processing >6M
transactions/year.

### The Solution: Tokenization

You have a separate, isolated service called the **Card Vault**.

```
Step 1: Customer's browser sends the card number → CARD VAULT (not your main server!)
Step 2: Vault encrypts and stores the card. Returns an opaque token: "tok_abc123"
Step 3: Your payment service stores ONLY the token in your DB.
Step 4: When charging: payment service sends token to Vault.
        Vault retrieves the real card number and forwards to the bank.
        Your payment service never touches the actual card data.
```

The Vault is isolated:
- Separate network segment with strict firewall rules
- All access logged for PCI audit
- Card numbers encrypted with AES-256
- Encryption keys managed by an HSM (Hardware Security Module — a tamper-resistant chip)

### Even Better: Stripe.js Approach

Stripe took it further: the card number is encrypted **in the customer's browser** using
Stripe's public key. Your server never sees the card number at all — not even briefly.

```
Browser → "stripe.js" library → encrypted card → Stripe Vault → token
                                                                  │
                                                                  ▼
                                            Token sent to YOUR server
```

This reduces your PCI scope dramatically — you have to comply with fewer requirements because
the sensitive data never touches your infrastructure.

## The Double-Entry Ledger (How Accountants Track Money)

This is borrowed from 700 years of accounting tradition. Every transaction is recorded as
**two entries**: a **debit** to one account and a **credit** to another.

> Sum of all debits = Sum of all credits. Always. If they don't match, something is broken.

Let's trace a $100 payment where Stripe takes a 2.9% fee:

```
Customer pays $100 to Merchant. Stripe's fee is $2.90.

When authorized (money held but not moved):
  Debit  Customer Account:   $100.00   (customer now "owes" us)
  Credit Suspense Account:   $100.00   (held in limbo)

When captured (charge confirmed):
  Debit  Suspense Account:   $100.00
  Credit Merchant Account:    $97.10   (merchant earns this minus our fee)
  Credit Platform Revenue:     $2.90   (our fee)

When settled (bank wire goes through):
  Debit  Merchant Account:    $97.10
  Credit Bank Settlement:     $97.10   (wired to merchant's real bank)
```

At every step, debits = credits. The ledger is **immutable and append-only** — you never
modify or delete entries. To "fix" a wrong entry, you add a reversal entry.

**Why this is bulletproof:** at any point in time, you can sum all entries by account and get
the exact balance. You can replay history. You can audit. If the math doesn't work, you know
immediately that something is wrong.

## Fraud Detection: Stopping the Bad Guys

Every transaction is scored before authorization. There are two layers.

### Layer 1: Rule-Based (Fast, Deterministic)

These are simple yes/no checks that run in milliseconds:

```
✗ Card BIN (first 6 digits) in known-stolen list → BLOCK
✗ Customer's IP in high-risk country → REVIEW
✗ Amount > 10× the customer's average → REVIEW
✗ 5+ failed attempts from same card in 10 min → BLOCK
✗ Same card used from US and Russia within 1 hour ("velocity check") → BLOCK
✗ Billing address doesn't match card's address (AVS) → REVIEW
✗ CVV doesn't match → BLOCK
```

### Layer 2: Machine Learning (Smarter, Slower)

For the gray areas, an ML model looks at hundreds of features:
- Amount, time of day, merchant category
- Device fingerprint (which browser, screen size, time zone)
- Customer's historical behavior
- Network graph: is this card connected to other suspicious cards?

The model outputs a score 0–100:
```
< 60  → ALLOW
60-80 → REVIEW (queue for human or hold for 24h)
> 80  → BLOCK
```

When chargebacks (disputed transactions) come back from the bank, they feed back into the
model as labeled training data.

### Important Rule: Don't Tell the Customer Why You Declined

If a declined fraudster knows exactly which rule tripped them, they can tune their next attempt
to evade it. So the customer-facing message is always generic:

```
"We couldn't process your payment. Please try again or contact your bank."
```

The actual reason is logged internally for the ops team.

## Webhooks: Notifying the Merchant

When payment status changes, you have to tell the merchant. They subscribed a URL like
`https://acme-shop.com/payment-webhook`.

```
Webhook payload:
{
  "event": "payment.captured",
  "payment_id": "pay_abc123",
  "amount": 10000,        // cents
  "currency": "USD",
  "status": "CAPTURED",
  "created_at": "2024-01-15T19:00:00Z"
}
```

### Security: HMAC Signing

How does the merchant know this webhook is really from you (and not an attacker forging events)?

```
You compute: signature = HMAC-SHA256(merchant_secret, payload)
You send it in header: X-Webhook-Signature: abc123...

Merchant verifies: HMAC-SHA256(their_secret, received_payload) == header_signature
```

Only you and the merchant know the shared secret, so an attacker can't forge a valid signature.

### Delivery: At-Least-Once with Retries

Merchants' servers go down sometimes. You can't drop the webhook just because their server is
returning errors right now.

```
Step 1: Write webhook event to DB IMMEDIATELY (before trying to send).
Step 2: Attempt HTTP POST to merchant URL (10 second timeout).
Step 3: On failure (timeout or non-2xx), retry with exponential backoff:
        10s → 1min → 5min → 30min → 2h → 12h → 24h
Step 4: After 7 retries, mark as FAILED and surface in merchant dashboard.
Step 5: Merchant can manually replay any webhook from the dashboard.
```

Because we deliver at-least-once (and sometimes the network drops responses), the merchant
might receive the same webhook twice. So every event includes a unique `event_id` — the merchant
checks: "have I already processed this event_id?" before acting.

## Reconciliation: Did the Books Balance?

Every night, you compare your internal ledger with the bank's settlement report:

```
Step 1: Fetch the bank settlement CSV (e.g., from your acquiring bank's API).
Step 2: Fetch all your ledger entries for the same period.
Step 3: Match each bank entry to a ledger entry by a reference ID.
Step 4: Investigate discrepancies:
        ✗ In bank but not in ledger → missing capture; trigger investigation
        ✗ In ledger but not in bank → pending settlement; expected next day
        ✗ Amount mismatch → fee miscalculation or FX rounding error
Step 5: Auto-resolve common discrepancies. Alert ops on anomalies.
```

Why this matters: small errors (an extra penny here, a missing capture there) accumulate.
If you don't catch them early, by month's end you've lost track of thousands of dollars.
Auditors require proof that your books match reality.

## Multi-Currency: Charging in Yen, Settling in USD

When a US merchant sells to a Japanese customer:

```
Customer sees: ¥3,000 (their currency)
Merchant gets: $20.00 (their currency)

Steps:
1. At quote time: lock the FX rate (¥150 = $1.00) for 30 seconds.
2. Customer agrees → process the charge at the LOCKED rate.
3. Record BOTH on the payment:
     customer_amount:   3000 JPY
     merchant_amount:   2000 cents USD
     fx_rate_locked:    0.00667
4. Always round to the currency's minor unit (USD = cents, JPY = whole yen).
5. Use banker's rounding (round half to even) to prevent systematic bias.
```

**Why lock the rate?** Otherwise the rate could drift between when the customer agreed to a price
and when the charge actually processes. The customer would be surprised when they see a slightly
different amount on their statement.

## How It All Fits Together

```
─── Customer hits "Buy" ────────────────────────────────────────────

Step 1: Browser sends card details DIRECTLY to Card Vault
        (via Stripe.js — bypasses your server)
        ← Vault returns token "tok_xyz"

Step 2: Browser sends to YOUR Payment API:
        POST /charges
        Header: Idempotency-Key: abc-123
        Body: { amount: 5000, card_token: "tok_xyz" }

Step 3: Payment API:
        ├─ Check Idempotency Store. Not seen → proceed.
        ├─ Validate request, check API key.
        └─ Forward to Fraud Service.

Step 4: Fraud Service:
        ├─ Run rule checks (instant).
        ├─ Run ML model. Score = 45 → ALLOW.
        └─ Return: proceed.

Step 5: Payment Engine:
        ├─ Create payment record: status = PENDING.
        ├─ Write ledger entries (debit customer, credit suspense).
        ├─ Send to Card Network (Visa/MC) via acquiring bank.
        ├─ Wait for response (1-3 seconds).
        ├─ Bank approves → status = AUTHORIZED.
        └─ Save result with idempotency key.

Step 6: Webhook Service:
        ├─ Write "payment.authorized" event to DB.
        └─ Async: POST to merchant's webhook URL (with HMAC signature).

Step 7: Later (typically immediately):
        ├─ Merchant calls /capture API.
        ├─ Payment Engine: status PENDING → CAPTURED.
        └─ Ledger: debit suspense, credit merchant, credit revenue.

Step 8: Overnight settlement:
        ├─ Batch all captures from the day.
        ├─ Send wire instructions to acquiring bank.
        ├─ Bank moves money to merchant's bank.
        └─ Status → SETTLED.

Step 9: Next morning:
        Reconciliation job matches your ledger ↔ bank statement.
```

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ No idempotency key → double charges on every network hiccup
   → Customer pays twice; trust destroyed
   ✅ Mandatory idempotency key on all mutating endpoints; 24h TTL

❌ Storing raw card numbers in your main database
   → PCI violation; if breached, lawsuits and fines bankrupt you
   ✅ Tokenize via isolated Card Vault; never let main DB see raw PANs

❌ Modifying or deleting old ledger entries
   → Breaks the audit trail; impossible to reconstruct history
   ✅ Ledger is immutable, append-only; corrections happen via reversal entries

❌ Fire-and-forget webhook delivery
   → Merchant URL down for 2 minutes → webhook lost → they miss a capture event
   ✅ Write event to DB first; retry queue with exponential backoff

❌ Synchronous authorization on the checkout page
   → 3 seconds of waiting → user abandons cart
   ✅ Optimistic UI: show "Processing..." → confirm via webhook or poll

❌ Telling the customer the precise fraud reason
   → "Failed AVS check" tells fraudsters which check to bypass next
   ✅ Generic message to customer; detailed reason only in internal logs

❌ Refund without checking remaining capturable amount
   → Issue $200 refund on $100 capture → negative balance → corrupt ledger
   ✅ Always verify: refund_amount ≤ captured_amount - already_refunded

❌ Race condition: two concurrent capture requests on same auth
   → Both succeed → total captured > authorized
   ✅ Optimistic locking with version column; or single-writer per payment_id

❌ FX rate fluctuating between quote and charge
   → Customer sees ¥3000, gets billed ¥3050 → confusing surprise
   ✅ Lock FX rate at quote time; store on payment record

❌ No reconciliation
   → Small errors accumulate; by month-end you've lost track of $50K
   ✅ Nightly automated reconciliation; alert on discrepancies > $1
```

## Real Systems at a Glance

```
Stripe:    State machine per "PaymentIntent"; Postgres + CockroachDB for global;
           Card Vault isolated from main service; webhooks via SQS+Lambda;
           Radar (ML+rules) for fraud

           Venmo as P2P wallet (no card interchange)

           Cash App for P2P + Square Banking
```

## How to Answer This in an Interview (Step by Step)

```
1. State the goal:
   "Payment processing: charge cards, move money, never double-charge, comply with
   PCI and SOX, reconcile with banks daily. The hardest part is exactly-once semantics
   across an unreliable network."

2. Lead with idempotency:
   "Every mutating request carries a client-generated idempotency key. Server checks
   if it has seen the key; if so, returns the cached response. This protects against
   network retries causing double charges."

3. Walk through the lifecycle:
   "Three states: AUTHORIZED (bank places a hold), CAPTURED (charge confirmed),
   SETTLED (money actually arrives at merchant's bank, usually T+1). Refunds create
   reversal entries — we never modify old ledger entries."

4. Explain PCI tokenization:
   "Raw card numbers never enter the main payment service. A separate Card Vault,
   in its own network segment, holds encrypted cards. The main service stores only
   opaque tokens. Even better: with Stripe.js, the card is encrypted in the browser
   and never touches our infrastructure."

5. Describe the ledger:
   "Double-entry accounting — every transaction is a debit + credit pair, sum to zero.
   Append-only and immutable. This guarantees auditability and makes errors immediately
   visible (debits ≠ credits)."

6. Fraud:
   "Two layers — rule-based checks (BIN blocklist, velocity, AVS, CVV) run instantly,
   followed by an ML model for gray areas. Block / review / allow. Never reveal the
   specific reason to the customer."

7. Webhooks:
   "Write the event to a durable store first, then attempt HTTP delivery with HMAC
   signature. Retry with exponential backoff (10s, 1min, 5min, ...). Merchants
   dedupe by event_id since delivery is at-least-once."

8. Reconciliation:
   "Nightly job matches our ledger to the bank's settlement file. Investigate
   mismatches: missing entries, amount drift, fee math errors. Auto-resolve common
   patterns, alert ops on anomalies."

9. Multi-currency:
   "Lock the FX rate at quote time, store it on the payment record. Round to the
   currency's minor unit using banker's rounding to avoid systematic bias."

10. Cite the main pitfalls:
    "Always idempotency keys. Never store raw cards. Immutable ledger. Write-first
    webhooks with retry. Optimistic UI for slow authorizations. Generic error
    messages so we don't help fraudsters."
```
