# Payment Processing (Stripe / PayPal)

---

## Problem Statement

Design a payment processing system that charges customers, moves money between accounts, and integrates with banks and card networks — like Stripe, PayPal, or an internal payments platform for an e-commerce site. The system must never double-charge, must reconcile with external parties, and must handle failures gracefully across distributed components.

```
Use cases:
  Stripe         Developer-facing payments API; card, ACH, SEPA
  PayPal         Wallet + card processing; buyer/seller flows
  Square         In-person POS + online payments
  Shopify Pay    E-commerce checkout flows
  Uber Payments  Ride billing, driver payouts, dispute handling
```

---

## Requirements

### Functional
```
1. Accept payment methods: credit/debit card, bank transfer (ACH/SEPA), digital wallets
2. Payment lifecycle: authorize → capture → settle
3. Refunds: full or partial, any time after capture
4. Idempotency: retrying the same request never double-charges
5. Webhooks: notify merchants of payment status changes
6. Fraud detection: score transactions, block suspicious ones
7. Multi-currency: charge in any currency, settle in merchant's currency
8. Reconciliation: end-of-day match internal ledger with bank statements
```

### Non-Functional
```
Scale:       Stripe processes ~1M transactions/day per merchant at peak;
             globally ~300M API calls/day
Latency:     P99 authorization < 3 seconds (card network roundtrip)
             Webhook delivery < 60 seconds
Consistency: Financial data must be strongly consistent (no lost money)
Durability:  Every ledger entry must survive node failures (synchronous DB writes)
Availability: 99.999% (5 nines) — downtime = lost revenue
Compliance:  PCI DSS Level 1 (never store raw card numbers)
             SOX compliance (audit trail for all money movement)
```

## The Core Challenge: Exactly-Once Money Movement

```
Problem: networks are unreliable. A charge request may:
  - Succeed at the bank but timeout before response reaches your server
  - Be retried by the client → double charge

Solution: idempotency keys
  Every payment request carries a client-generated idempotency_key (UUID).
  Before processing:
    1. Check idempotency store: have we seen this key before?
    2. YES → return the stored result (no re-processing)
    3. NO  → process, store result atomically, return result

  The idempotency key + merchant_id pair is unique.
  Same key + different params → error (conflict).
  TTL on key: 24 hours (after that, treat as new request).

Why not just check duplicate amounts?
  Two legitimate $10.00 charges from the same customer in 5 seconds is valid.
  Only the idempotency key captures the "same request" intent.
```

## Payment Lifecycle

```
State machine:

  PENDING ──[authorize]──▶ AUTHORIZED ──[capture]──▶ CAPTURED ──[settle]──▶ SETTLED
     │                          │                        │
     │                    [cancel/void]            [refund]──▶ REFUNDED
     │                          │
     └──[fraud block]──▶ BLOCKED
     └──[auth fail]──▶ FAILED

Authorize:
  Lock funds on the customer's card (pre-authorization).
  Card network places a "hold" for the amount.
  No money moves yet.
  Typical hold: 7 days. After that, authorization expires.

Capture:
  Trigger actual money movement.
  Can capture less than authorized (e.g., hotel pre-auth vs actual stay).
  Must happen before auth expires.

Settle:
  Bank-to-bank transfer completes. Money in merchant's account.
  Happens in batches (overnight ACH, or T+1 for cards).

Refund:
  Reverse a capture (full or partial).
  Creates a new credit transaction in the ledger.
  Takes 3–7 business days to reach customer's bank.

Auth-only (no capture):
  Used for pre-authorization checks.
  Hotels, rental cars: pre-auth on check-in, capture on check-out.
```

## PCI DSS and Card Tokenization

```
PCI DSS Level 1: strictest standard for card data handling.
Rule: never store raw Primary Account Number (PAN) in your database.

Tokenization:
  1. Client sends card number to Card Vault (isolated, PCI-scoped service)
  2. Vault stores encrypted card → returns opaque token (e.g., "tok_abc123")
  3. Your payment service stores only the token
  4. On charge: send token to Vault → Vault retrieves real PAN → sends to network

  Token is useless to an attacker (no PAN in your DB)
  Vault is the only system that touches real card data

Alternatives:
  Client-side encryption (Stripe.js): card number encrypted in browser,
    server never sees it → even simpler PCI scope reduction
  Apple Pay / Google Pay: device-generated token, single-use

Encryption at rest:
  Vault uses HSM (Hardware Security Module) for key management
  Card data encrypted with AES-256; keys rotated quarterly
  All access to Vault is logged for PCI audit
```

## Double-Entry Ledger

```
Every financial transaction is recorded as a pair of entries:
  Debit one account, Credit another.
  Sum of all debits = Sum of all credits (accounting identity).

Accounts:
  Customer account      (liability for us — we owe them)
  Merchant account      (liability — we owe merchant)
  Platform revenue      (asset — our fees)
  Suspense account      (temporary holds during processing)
  Bank settlement acc   (maps to actual bank account)

Example: $100 payment from customer to merchant (Stripe takes 2.9%):

  Entry 1: Debit  Customer Account   $100.00  (customer owes us)
  Entry 2: Credit Suspense Account   $100.00  (held during auth)

  On capture:
  Entry 3: Debit  Suspense Account   $100.00
  Entry 4: Credit Merchant Account    $97.10  (after 2.9% fee)
  Entry 5: Credit Platform Revenue    $2.90

  On settlement (T+1):
  Entry 6: Debit  Merchant Account   $97.10
  Entry 7: Credit Bank Settlement    $97.10  (wire to merchant's bank)

Properties:
  Immutable: entries are never deleted or modified
  Append-only: new entries for corrections (reversal entries)
  Auditable: full history of every dollar movement
```

## Fraud Detection

```
Every transaction is scored before authorization.

Rule-based checks (fast, deterministic):
  - Card BIN (first 6 digits) in blocked list
  - IP address in high-risk country list
  - Transaction amount > 10× customer's average
  - 5+ failed attempts from same card in 10 minutes
  - Card used in two countries within 1 hour (velocity)
  - Billing address mismatch (AVS check)
  - CVV mismatch

ML scoring (async, used for review queue):
  - Feature vector: amount, merchant category, time-of-day,
    device fingerprint, historical behavior, network graph
  - Score 0–100: > 80 = block, 60–80 = review, < 60 = allow
  - Feedback loop: disputes/chargebacks → retrain model

Decisions:
  ALLOW   → proceed to authorization
  REVIEW  → hold for manual review (up to 24h delay)
  BLOCK   → decline immediately with generic error

Note: never tell the customer exactly WHY declined
  (would help fraudsters tune their attacks)
```

## Webhook Delivery

```
After each payment state change, notify the merchant's registered URL.

Webhook payload:
  {
    "event": "payment.captured",
    "payment_id": "pay_abc123",
    "amount": 10000,  // cents
    "currency": "USD",
    "status": "CAPTURED",
    "created_at": "2024-01-15T19:00:00Z"
  }

Security: HMAC-SHA256 signature in X-Webhook-Signature header
  merchant can verify: HMAC(payload, merchant_secret) == header

Delivery guarantees (at-least-once):
  1. Write webhook event to DB before delivering
  2. Attempt HTTP POST to merchant URL (timeout: 10s)
  3. On failure (timeout / non-2xx): retry with exponential backoff
     Retry schedule: 10s, 1min, 5min, 30min, 2h, 12h, 24h
  4. After 7 retries: mark as FAILED, alert merchant in dashboard
  5. Merchant can manually replay any webhook from dashboard

Idempotency for merchants:
  Each webhook has a unique event_id
  Merchant should check: have I processed this event_id before?
  (at-least-once delivery means duplicates are possible)
```

## Reconciliation

```
End-of-day process to verify no money was lost or duplicated.

Steps:
  1. Fetch bank settlement report (CSV from acquiring bank)
  2. Fetch internal ledger entries for same settlement period
  3. Match each bank entry to a ledger entry by reference_id
  4. Discrepancies:
     - In bank but not in ledger → missing capture; trigger investigation
     - In ledger but not in bank → pending settlement; expect next day
     - Amount mismatch → fee calculation error or FX rounding issue
  5. Generate reconciliation report
  6. Auto-resolve common discrepancies; alert ops team for anomalies

Why this matters:
  Card networks charge interchange fees; these must match ledger
  FX conversion rates may differ from expected → rounding errors accumulate
  Human error in manual adjustments → audit trail critical
```

## Multi-Currency

```
Charge in customer's currency; settle in merchant's currency.

FX conversion:
  1. Get exchange rate at payment time (lock rate for 30s during checkout)
  2. Record both: customer_amount + customer_currency, merchant_amount + merchant_currency
  3. FX rate stored immutably on the payment record (for reconciliation)

Rounding rules:
  Always round to currency's minor unit (USD → cents, JPY → yen)
  Use banker's rounding (round half to even) to avoid systematic bias

DCC (Dynamic Currency Conversion):
  Offer cardholder option to pay in their home currency
  Stripe / bank provides FX rate (usually worse than interbank rate)
  Merchant gets slightly higher revenue; customer convenience
```

## Pitfalls

```
1. No idempotency key → double charges on retry
   Problem: network timeout → client retries → two charges
   Fix: mandatory idempotency_key on all mutating endpoints;
        check before processing; return cached result if seen

2. Synchronous authorization blocking checkout
   Problem: card network roundtrip adds 1–3s to checkout latency
   Fix: async authorization with optimistic display ("Payment processing…");
        most auths succeed; failure shown via webhook / polling

3. Lost webhook delivery (fire-and-forget HTTP call)
   Problem: merchant URL down; webhook lost; merchant misses capture event
   Fix: write webhook to DB first; async delivery worker; retry queue

4. Refund without checking capture amount
   Problem: refund $200 on a $100 capture → negative balance
   Fix: ledger-level check; refund_amount ≤ remaining_capturable_amount

5. Race condition: two captures for same authorization
   Problem: concurrent requests both capture; total > authorized amount
   Fix: optimistic locking on payment record (version column);
        or serialized state machine (one writer per payment_id)

6. Card vault breach exposing PANs
   Problem: attacker gets card vault DB → full PANs
   Fix: encrypt with HSM; vault in separate network segment;
        rotate encryption keys; use client-side tokenization (Stripe.js)

7. FX rate drift between quote and charge
   Problem: user sees $9.99 at quote; charged $10.03 due to rate change
   Fix: lock FX rate at quote time (store in session); charge using locked rate;
        absorb small slippage; re-quote if session expires

8. Settlement failure → merchant not paid
   Problem: bank API down on settlement day; batch fails silently
   Fix: idempotent settlement batch; retry failed entries;
        monitoring + alerting on settlement lag > 24h
```

## Real-World Systems

```
Stripe:     State machine per payment intent; Postgres with CockroachDB for global;
            Card Vault separate from charge service; webhooks via SQS + Lambda;
            Radar for fraud (ML + rules engine)

PayPal:     Three-tier: user wallet, merchant wallet, external bank;
            Braintree (acquired) for card processing;
            Venmo (P2P wallet, no interchange)

Square:     In-person POS + cloud bridge; end-to-end encryption on reader;
            Cash App (P2P) + Square Banking

Adyen:      Global acquirer (owns the acquiring bank license);
            Single platform across 200+ countries;
            RevenueAccelerate (revenue optimization + fraud)
```

## Interview Answer Framework

1. **Core problem:** exactly-once money movement; network unreliability → idempotency keys
2. **Payment states:** PENDING → AUTHORIZED → CAPTURED → SETTLED; refund creates reversal entry
3. **Idempotency:** every mutating call has idempotency_key; check store first; return cached result
4. **PCI DSS:** tokenize PANs in isolated Card Vault; never store raw card numbers in main DB
5. **Ledger:** double-entry; every payment = debit + credit; immutable append-only; sum(D)=sum(C)
6. **Fraud:** rule engine (fast, deterministic) + ML score (async, review queue); never reveal reason for decline
7. **Webhooks:** write-first, then deliver; retry with backoff; HMAC-signed; at-least-once
8. **Reconciliation:** nightly match of ledger vs bank statement; auto-resolve common discrepancies
9. **Multi-currency:** lock FX rate at quote; store both amounts; banker's rounding
10. **Pitfalls:** double charge, lost webhook, over-refund, concurrent capture race, vault breach
