## 2. Payment State Machine

```mermaid
stateDiagram-v2
    [*] --> PENDING : create payment intent

    PENDING --> AUTHORIZED : authorize OK (card network hold)
    PENDING --> FAILED : auth declined / timeout
    PENDING --> BLOCKED : fraud score > threshold

    AUTHORIZED --> CAPTURED : capture (triggers money movement)
    AUTHORIZED --> CANCELLED : void / cancel before capture
    AUTHORIZED --> FAILED : auth expires (> 7 days)

    CAPTURED --> SETTLED : bank settlement completes (T+1)
    CAPTURED --> REFUNDED : full refund
    CAPTURED --> PARTIALLY_REFUNDED : partial refund

    SETTLED --> REFUNDED : full refund post-settlement
    SETTLED --> PARTIALLY_REFUNDED : partial refund post-settlement

    REFUNDED --> [*]
    PARTIALLY_REFUNDED --> REFUNDED : remaining amount refunded
    SETTLED --> [*]
    FAILED --> [*]
    BLOCKED --> [*]
    CANCELLED --> [*]
```

## 3. Idempotency — Prevent Double Charge

```mermaid
sequenceDiagram
    participant C  as Client
    participant API as Payment API
    participant IS  as Idempotency Store (Redis)
    participant PE  as Payment Engine
    participant DB  as Ledger DB

    C->>+API: POST /charge {amount:100, key:"idem-xyz"}

    API->>+IS: GET idem-xyz
    IS-->>-API: NOT FOUND

    API->>+PE: Authorize $100 (card network)
    PE->>DB: Write AUTHORIZED entry
    PE-->>-API: {payment_id: "pay_abc", status: AUTHORIZED}

    API->>IS: SET idem-xyz → {pay_abc, AUTHORIZED} TTL=24h
    API-->>-C: 200 {payment_id: pay_abc, status: AUTHORIZED}

    Note over C,DB: Network timeout — client retries

    C->>+API: POST /charge {amount:100, key:"idem-xyz"} (retry)
    API->>+IS: GET idem-xyz
    IS-->>-API: HIT → {pay_abc, AUTHORIZED}
    API-->>-C: 200 {payment_id: pay_abc, status: AUTHORIZED}

    Note over C,DB: Same response — customer charged exactly once ✓
```

## 4. Double-Entry Ledger

```mermaid
flowchart TB
    subgraph PaymentFlow["$100 Payment: Customer → Merchant (2.9% fee)"]
        direction TB

        subgraph Step1["Step 1: Authorization (hold funds)"]
            E1["DEBIT  Customer Account    $100.00"]
            E2["CREDIT Suspense Account   $100.00"]
            E1 -.->|"paired entry"| E2
        end

        subgraph Step2["Step 2: Capture (move money)"]
            E3["DEBIT  Suspense Account   $100.00"]
            E4["CREDIT Merchant Account    $97.10"]
            E5["CREDIT Platform Revenue     $2.90"]
            E3 -.->|"3-way split"| E4
            E3 -.-> E5
        end

        subgraph Step3["Step 3: Settlement (T+1 wire)"]
            E6["DEBIT  Merchant Account   $97.10"]
            E7["CREDIT Bank Settlement    $97.10"]
            E6 -.->|"paired entry"| E7
        end

        subgraph Invariant["Invariant (always holds)"]
            INV["Σ DEBITS = Σ CREDITS\n$100 + $100 + $97.10 = $100 + $97.10 + $2.90 + $97.10"]
        end
    end

    Step1 --> Step2 --> Step3
    Step3 --> Invariant

    note["Entries are IMMUTABLE — corrections via reversal entries\nAppend-only → full audit trail preserved"]
    style note fill:#fff9c4
```

## 5. Fraud Detection Pipeline

```mermaid
flowchart LR
    subgraph Input["Payment Request"]
        REQ["amount, card_token\ncustomer_id, IP\nmerchant, device_fp"]
    end

    subgraph Rules["Rule Engine (sync, < 50ms)"]
        R1["BIN in blocked list?"]
        R2["IP in high-risk country?"]
        R3["Velocity: 5+ fails / 10min?"]
        R4["Two countries in 1hr?"]
        R5["AVS / CVV mismatch?"]
    end

    subgraph ML["ML Scorer (async, 100ms)"]
        FEAT["Feature extraction\n(historical behavior,\nnetwork graph, time)"]
        MODEL["Gradient Boosted Model\n→ score 0–100"]
    end

    subgraph Decision["Decision"]
        ALLOW["score < 60\n→ ALLOW"]
        REVIEW["60 ≤ score < 80\n→ REVIEW (manual)"]
        BLOCK["score ≥ 80\n→ BLOCK (decline)"]
    end

    REQ --> Rules --> ML
    Rules -->|"hard block"| BLOCK
    ML --> FEAT --> MODEL --> ALLOW & REVIEW & BLOCK

    ALLOW -->|"proceed"| AUTH["Authorization"]
    REVIEW -->|"hold 24h"| QUEUE["Review Queue"]
    BLOCK -->|"generic error"| DECLINE["Decline (no reason given)"]
```

## 6. Webhook Delivery with Retry

```mermaid
sequenceDiagram
    participant PE  as Payment Engine
    participant WDB as Webhook DB
    participant WW  as Webhook Worker
    participant M   as Merchant URL

    PE->>WDB: INSERT webhook_event {id, payload, status=PENDING}
    Note over PE,WDB: Write-first — event survives worker crash

    WW->>WDB: Poll PENDING events
    WDB-->>WW: webhook_event {id, payload, merchant_url}

    WW->>WW: Sign payload: HMAC-SHA256(payload, merchant_secret)
    WW->>+M: POST /webhook {payload, X-Signature: hmac}

    alt 2xx response
        M-->>-WW: 200 OK
        WW->>WDB: UPDATE status=DELIVERED, delivered_at=now
    else timeout / 5xx
        M-->>-WW: 503 / timeout
        WW->>WDB: UPDATE attempt_count++, next_retry=now+backoff
        Note over WW,WDB: Backoff: 10s → 1min → 5min → 30min → 2h → 12h → 24h
    end

    Note over WW,M: After 7 failures → status=FAILED\nMerchant alerted; can replay from dashboard
```

## 7. Refund Flow

```mermaid
sequenceDiagram
    participant M   as Merchant
    participant API as Payment API
    participant PE  as Payment Engine
    participant LD  as Ledger Service
    participant NET as Card Network

    M->>+API: POST /refund {payment_id, amount:50.00, key:"ref-xyz"}

    API->>PE: Validate: amount ≤ remaining_captured?
    Note over PE: remaining = captured - already_refunded\n$100.00 - $0 = $100.00 ≥ $50.00 ✓

    PE->>+NET: Refund request (using original auth reference)
    NET-->>-PE: Refund accepted, ref_id=REF123

    PE->>LD: Write reversal entries:
    Note over LD: DEBIT  Merchant Account  $50.00\nCREDIT Customer Account $50.00\n(fee reversal: DEBIT Revenue $1.45, CREDIT Merchant $1.45)

    PE->>PE: Update payment status → PARTIALLY_REFUNDED
    PE-->>-API: {refund_id: ref_123, status: REFUNDED, amount: 50.00}
    API-->>M: 200 OK

    Note over NET,M: Customer sees credit in 3–7 business days\n(bank processing time, not our system)
```

## 8. Reconciliation — Nightly Batch

```mermaid
flowchart LR
    subgraph Sources["Data Sources"]
        BANK_RPT["Bank Settlement Report\n(CSV, from acquiring bank)\nref_id, amount, settled_at"]
        LEDGER_RPT["Internal Ledger\n(settlement entries for same day)\nref_id, amount, payment_id"]
    end

    subgraph Matching["Matching Engine"]
        LOAD["Load both datasets\ninto memory / temp table"]
        JOIN["Full outer join on ref_id"]
        MATCH["Matched: both sides present\n+ amounts agree → OK"]
        MISS_BANK["In bank only\n→ investigate: missing capture?"]
        MISS_LEDGER["In ledger only\n→ pending settlement\n(retry tomorrow)"]
        MISMATCH["Amount differs\n→ fee / FX rounding?\nFlag for manual review"]
    end

    subgraph Output["Output"]
        REPORT["Reconciliation Report\n(stored in audit DB)"]
        ALERTS["Ops Alerts\n(PagerDuty for anomalies)"]
        AUTO["Auto-resolve common cases\n(known rounding diff < $0.01)"]
    end

    BANK_RPT & LEDGER_RPT --> LOAD --> JOIN
    JOIN --> MATCH & MISS_BANK & MISS_LEDGER & MISMATCH
    MATCH --> REPORT
    MISS_BANK & MISMATCH --> ALERTS
    MISS_LEDGER --> AUTO
    AUTO --> REPORT
```
