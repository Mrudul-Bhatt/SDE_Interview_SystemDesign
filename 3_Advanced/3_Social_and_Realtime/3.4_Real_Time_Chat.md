# Real-Time Chat (like WhatsApp or Slack)

---

## What Are We Building?

You open WhatsApp and type "hey, want to grab dinner?" to your friend Bob. Half a second later,
Bob sees the message pop up on his phone — even though his phone is in his pocket on the other
side of the world.

```
Alice's phone                              Bob's phone
     │                                          │
     │  "hey, want to grab dinner?"             │
     │  ───────────────────────────▶            │
     │                                          │  ← appears in < 100ms
     │                            ◀───────────── │
     │  ✓✓ (Bob received)                       │
     │                            ◀───────────── │
     │  ✓✓ blue (Bob read it)                   │
```

That's real-time chat. Sounds simple, but doing this for **2 billion users sending 100 billion
messages per day** requires careful engineering.

**The big problems we have to solve:**
- Deliver messages instantly to users who are online
- Queue messages for users who are offline; deliver when they reconnect
- Send a push notification to offline users so they know to open the app
- Show delivery status: sent → delivered → read
- Keep messages in order (even when network is jittery)
- Show who's online ("last seen 5 minutes ago")
- Handle group chats with hundreds of members

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Send a message to one person or a group
2. Real-time delivery if recipient is online
3. Queue messages for offline recipients; deliver on reconnect
4. Push notification ("ping!") for offline users
5. Delivery receipts: ✓ sent → ✓✓ delivered → ✓✓ blue (read)
6. Message history with pagination ("load older messages")
7. Presence: online/offline + "last seen" timestamp
8. Group chats with up to 500 members
```

### Non-Functional (scale and reliability)
```
Users:       2 billion total
Messages:    100 billion per day = ~1.15 million per second on average
Latency:     End-to-end delivery in under 100ms for online users
Order:       Messages must appear in the order they were sent
Durability:  Never lose a message, even if a server crashes mid-send
Uptime:      99.99% — people expect chat to always work
Retention:   Keep messages for 7 years (WhatsApp policy)
```

## How Much Hardware?

Before building anything, engineers estimate **how big the system needs to be**. This is called
"capacity planning" or "back-of-the-envelope estimation." Most systems are sized by how many
*requests* arrive — but chat has a unique cost that none of the others do: it must keep a **live
connection open for every online user, all day long.** So our questions are:

1. **How many messages**, and how much storage do they need?
2. **How many servers to hold all the open connections?** (this is chat's defining cost)
3. **How much extra work does group delivery add?**

Let's work through them, slowly.

---

### Question 1: How many messages, and how much storage?

```
Step 1 — Messages per second:
  100 billion messages/day ÷ 86,400 seconds/day ≈ 1.15 million messages/sec

Step 2 — Storage per day (chat messages are tiny — just text):
  1.15M/sec × ~100 bytes each × 86,400 sec ≈ 10 TB/day

Step 3 — We keep messages for 7 years (WhatsApp's policy):
  10 TB/day × 365 days × 7 years ≈ 25 PB (petabytes) total
```

**Worth noticing:** unlike a message queue (which deletes data after a few days), chat keeps your
history essentially forever, so storage genuinely **piles up** to ~25 PB. But it's *cheap* to store —
text is tiny and old messages are rarely read, so they sit on inexpensive disk. Storage is a big
number here, but it is **not** the hard part. The hard part is next.

---

### Question 2: How many servers to hold the open connections? (chat's defining cost)

Here's what makes real-time chat different from almost every other system. A normal web request
arrives, gets answered in a few milliseconds, and the connection closes. But for chat, the server
can't close the connection — it has to **keep it open** so it can instantly *push* a new message
to your phone the moment one arrives. Every online user is holding one of these live connections
open the entire time the app is open.

```
Step 1 — How many users are online at once?
  2 billion total users, ~10% online at any moment ≈ 200 million users online

Step 2 — Each online user holds one open connection (a "WebSocket"):
  → 200 million simultaneous open connections to keep alive

Step 3 — How many can one server hold?
  ~50,000 open connections per server (an idle connection is cheap to keep,
  but memory and OS limits cap how many one machine can babysit)

Step 4 — So how many chat servers?
  200,000,000 ÷ 50,000 = 4,000 chat servers
```

**The key insight: chat is *connection-bound*, not request-bound.** We need 4,000 servers not
because 1.15M messages/sec is hard to process (it isn't), but because *holding 200 million doors
open at once* is what eats the hardware. This is a completely different sizing driver from the
other systems — and it's *why* the rest of this doc obsesses over WebSockets, which server holds
which connection, and what happens when a connection-holding server crashes.

> Takeaway: count chat servers by *concurrent open connections* (~50K each), not by messages/sec.

---

### Question 3: How much extra work does group delivery add?

One person sending one message doesn't mean one delivery — in a group, that message has to be
pushed to *everyone* in the group. So delivery work is *bigger* than send work.

```
Average group size: ~10 members
1.15M messages sent/sec × 10 recipients each = 11.5M delivery events/sec
```

**What this tells us:** the system actually does ~**10× more delivery work than sending work** —
sending 1.15M messages/sec turns into 11.5M "push this to a phone" events/sec. This is the same
"fan-out" multiplication we saw in the social feed, and it's why delivery (finding each recipient's
server and pushing down their connection) is the busiest path in the system.

---

### The numbers to remember

```
~1.15M messages/sec    → the send rate (the easy part)
~25 PB storage         → big but cheap; history kept ~forever, rarely read
~4,000 chat servers    → the hard part: holding 200M open connections (~50K each)
~11.5M deliveries/sec  → group fan-out: 1 send → ~10 pushes
```

The big idea connecting them: chat is defined by **persistent connections**, not request volume.
Keeping 200 million live connections open is what sets the server count, and group fan-out
multiplies every send into many deliveries — so the whole design is about routing pushes to the
right open connection, fast.

## The Transport Question: How Does the Server Push to Your Phone?

The browser-side classic — HTTP — only works one direction: the client asks the server. The
server can't just call the client whenever it wants. So how do we get real-time push?

### Option 1: Short Polling (Bad)

```
Every 5 seconds, the client asks: "any new messages?"
  → Up to 5 seconds of delay (feels laggy)
  → Massive wasted requests when nothing's happening (battery drain, server load)
```

### Option 2: Long Polling (Better)

```
Client sends a request and the server holds it open.
When a new message arrives → server responds immediately → client sends another request.
  → Low latency
  → Works through all firewalls (it's just HTTP)
  → But: one request per message → still high overhead
```

### Option 3: WebSocket (The Right Answer)

A WebSocket is a **persistent two-way connection** between client and server. After an initial
HTTP handshake, both sides can send messages any time, in either direction, with very little
overhead (no HTTP headers per message).

```
Alice's phone ────[persistent TCP connection]────▶ Chat Server 1
              ◀────[messages flow both ways]──────
```

WebSockets win for production chat:
- Lowest latency (no per-message HTTP overhead)
- Bidirectional (server can push without client asking)
- Efficient (one connection per device, not one per message)

**The trade-off:** these connections are long-lived. The chat server has to remember which
connections it's holding, and if it crashes, the client has to reconnect.

## The Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                  Real-Time Chat System                           │
│                                                                  │
│  Alice's phone ──WebSocket──▶ Chat Server 1                     │
│  Bob's phone   ──WebSocket──▶ Chat Server 2                     │
│  Carol's phone ──WebSocket──▶ Chat Server 1                     │
│                                                                  │
│                    │                                             │
│                    ▼                                             │
│         ┌─────────────────────────┐                              │
│         │  Message Bus            │  ← Redis Pub/Sub or Kafka    │
│         │  (cross-server routing) │                              │
│         └─────────────────────────┘                              │
│                                                                  │
│  Supporting services:                                            │
│    Message Store: Cassandra (all messages, durable)              │
│    Connection Registry: Redis (userId → server)                  │
│    Presence: Redis with TTL (online/offline)                     │
│    Push Notifications: APNs (iOS) + FCM (Android)                │
└─────────────────────────────────────────────────────────────────┘
```

## How a Message Flows (Both Users Online)

This is the magic moment — Alice sends, Bob receives — and there's a lot happening behind the scenes.

```
Step 1: Alice's phone sends "Hello Bob!" over its WebSocket to Chat Server 1.

Step 2: Chat Server 1:
   a. Assigns a unique message ID (a Snowflake ID — globally unique, time-sortable)
   b. Persists the message to Cassandra FIRST (durability before delivery)
   c. Looks up: where is Bob right now?
      → Asks the Connection Registry (Redis): "user:bob is on Chat Server 2"

Step 3: Chat Server 1 publishes the message to a "channel" on the Message Bus:
   channel = "user:bob"
   payload = "Hello Bob! (msg id 12345)"

Step 4: Chat Server 2 is subscribed to that channel (because Bob is connected to it).
   It receives the message from the Message Bus.

Step 5: Chat Server 2 pushes the message down Bob's WebSocket to his phone.

Step 6: Bob's phone sends back: "got it!" (a delivery ACK)
   → Server updates message status to "Delivered"
   → Sends a "Delivered" event back to Alice → her UI shows ✓✓

Total time: about 50–100ms.
```

**Why persist first?** Because if we deliver and then crash before saving, the message would be
gone from history even though Bob saw it. The rule: **durability before delivery, always.**

---

## What If Bob Is Offline?

```
Step 1–2: Same as before — Chat Server 1 receives Alice's message and stores it in Cassandra.

Step 3: Chat Server 1 publishes to the Message Bus channel "user:bob"...
        but nobody is subscribed (Bob isn't connected anywhere).

Step 4: Chat Server 1 checks the Presence service: "is Bob online?" → No.
        → Send a push notification via APNs (iOS) or FCM (Android):
          "Alice sent you a message"
        → Bob's phone gets a notification on the lock screen.

Step 5: Bob opens the app. His phone connects a new WebSocket to (say) Chat Server 3.
        Chat Server 3 registers Bob in the Connection Registry.
        It then asks the Message Store: "any messages Bob hasn't seen yet?"
        It pulls all undelivered messages and pushes them down the new WebSocket.
        Bob's UI shows: "Alice: Hello Bob!"
```

## The Cross-Server Routing Problem

Here's a subtle but important question: when Alice sends a message, her phone is connected to
Chat Server 1. Bob's phone is connected to Chat Server 2. **How does Server 1 know where to send
the message?**

Server 1 doesn't even know Bob exists — Bob isn't connected to it!

### The Connection Registry (Redis)

Every time a user connects, we record where they are:

```
On connect:    SET user_server:bob = "ChatServer2"  (with 5-minute TTL refresh)
On disconnect: DEL user_server:bob
```

Now Server 1 can look up: `GET user_server:bob` → "ChatServer2" → "forward via the bus."

### The Message Bus (Redis Pub/Sub or Kafka)

Once Server 1 knows Bob is on Server 2, how does it actually send the message there? It could
open a direct TCP connection, but at 4,000 servers that's 16 million connections.

Instead, every server subscribes to channels for the users it's holding:

```
Chat Server 2 subscribes to: "user:bob", "user:dave", "user:eve", ...
Chat Server 1 just publishes to "user:bob"
The message bus routes it to whichever servers subscribed.
```

This is much cleaner. Server 1 doesn't need to know about Server 2 at all — it just publishes
to the right channel and trusts the bus to deliver.

**Redis Pub/Sub vs Kafka:**
- **Redis Pub/Sub**: super fast (sub-millisecond), but fire-and-forget. If the target server
  is restarting at that exact moment, the message is dropped from the bus (but it's still safe
  in Cassandra — backlog drain on reconnect will deliver it).
- **Kafka**: durable, even if the destination server is down. More reliable but adds latency.

WhatsApp uses something closer to Pub/Sub (low latency) and relies on the backlog drain for safety.

## Storing Messages (Cassandra)

Every message gets persisted to Cassandra. Why Cassandra?

```
Schema:
  Table: messages
    PRIMARY KEY: (chat_id, sent_at DESC, message_id)
    Columns: sender_id, content, delivery_status, media_url

  All messages for one chat live in ONE partition, sorted by time (newest first).
```

**Why this is great for chat:**
- "Get the last 50 messages in this chat" → one fast partition scan
- "Load older messages before time T" → range scan within the partition
- No single leader bottleneck (Cassandra is a ring of equals; can write to any node)
- Handles 1M+ writes/sec natively

### Snowflake Message IDs

Each message ID is a 64-bit number combining:
- **41 bits**: timestamp in milliseconds since some epoch
- **10 bits**: machine ID (which server generated this)
- **12 bits**: sequence number within that millisecond

```
This means:
  - Globally unique (no central counter needed)
  - Sortable by time (just sort by the number)
  - Embedded timestamp (you can read "when was this sent" from the ID alone)
```

## Delivery Receipts (The Three Checkmarks)

WhatsApp shows three states for every message:

```
✓     Sent       — the server received your message and persisted it
✓✓    Delivered  — the recipient's device received it
✓✓    Read       — the recipient opened the chat and saw it
(blue)
```

How each one happens:

```
Sent:      Server returned "OK" after writing to Cassandra → client shows ✓
Delivered: Recipient's phone confirms receipt via WebSocket ACK → ✓✓
Read:      Recipient opens the chat → app sends "read" event → ✓✓ turns blue
```

### Group Chats Are Trickier

In a 10-person group, what does "Delivered" mean? When all 10 received it? When any one did?

```
WhatsApp's choice:
  ✓✓ shown when ALL members have received it
  ✓✓ blue shown when ALL members have read it

This requires tracking delivery status per (message, member) pair —
a separate table:
  receipts(message_id, user_id, status)
```

## Presence: How Do We Know You're Online?

We can't trust "user connected a WebSocket" as the only signal. What if their phone is connected
but the network died silently? We need a heartbeat.

```
Every 15 seconds, the client sends a heartbeat over the WebSocket.
Server runs: SET presence:bob = "online" EX 30   (TTL of 30 seconds)

If the TTL expires (no heartbeat for 30+ seconds) → Bob is considered offline.

To check if someone is online:
  EXISTS presence:bob  → 1 = online, 0 = offline
```

**Why TTL instead of "set offline on disconnect"?** Because phones lose network silently all the
time. A clean disconnect is rare. The TTL approach handles dead connections automatically.

### Last Seen

```
On every heartbeat: also update last_seen:bob = current_timestamp
When friend opens Bob's profile: read last_seen:bob → "Last seen 5 minutes ago"
```

WhatsApp lets you hide "last seen" from privacy settings — a flag on the user profile that
the read query checks.

## Group Chats

A group is just a set of user IDs:

```
group_members:family = {alice, bob, carol, dave}
```

When Alice sends a message to the family group:

```
Step 1: Store the message ONCE in Cassandra (one row, attached to chat_id = "group:family")
Step 2: For each member except Alice:
          - Online? Push to their WebSocket (via the message bus)
          - Offline? Send push notification + store in their undelivered queue
```

### Large Groups

What if the group has 500 members and Alice sends a message? That's 500 deliveries.

```
Done synchronously: takes 500 × 1ms = 500ms before we say "sent" to Alice — too slow.

Better: fan out asynchronously.
  - Persist the message
  - Return "sent" to Alice IMMEDIATELY (her UI shows ✓)
  - Push to the message bus once per recipient
  - Each chat server delivers to its local connections
```

Alice's UI feels instant. Bob sees the message within 50ms. Other members within a few hundred ms.

## Sharing Photos and Videos

You can't send a 5 MB photo through the chat server — at 1 million messages/sec, that's 5 TB/sec
of data through chat servers. Impossible.

```
The trick: media goes around the chat servers, not through them.

Step 1: Client asks Media Service: "I want to upload a photo, give me a URL"
Step 2: Media Service generates a presigned S3 URL (valid for 5 minutes):
        https://chat-media.s3.../upload-token-xyz
Step 3: Client uploads the photo DIRECTLY to S3 using that URL
        (bypasses chat servers entirely)
Step 4: Client sends a normal chat message with just the URL:
        { content: "", media_url: "https://cdn.../photo-xyz.jpg" }
Step 5: Recipients download the photo from the CDN
```

The chat server only handles tiny text-and-link messages. The big bytes flow through S3 + CDN,
which are built for it.

## Message Ordering: Why It's Not As Simple As It Looks

Alice sends "A" then "B" in quick succession. Due to network jitter, the server receives "B" first.
Bob would see "B" before "A" — wrong!

Several solutions, used in combination:

### Snowflake Timestamps as the Primary Sort

Since Snowflake IDs include a millisecond timestamp, just sort messages by ID. This handles 99%
of cases — clock skew on the server is small.

### Client Sequence Numbers as the Tiebreaker

If two messages happen in the same millisecond, the client's sequence number breaks the tie.
The client labels each message:

```
{ chat_id: "alice-bob", client_seq: 1, content: "A" }
{ chat_id: "alice-bob", client_seq: 2, content: "B" }
```

Even if "B" arrives first, the server (and recipient) sort by client_seq.

### Server-Side Sequence (Strictest, Slowest)

For absolute ordering, use a central counter per chat:
```
INCR chat_seq:alice-bob  → 1, 2, 3, 4...
```

This guarantees a single global order but adds a small bottleneck per chat. WhatsApp uses the
Snowflake-plus-client-seq approach — fast and "good enough."

              │

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ Delivering before persisting
   → Server delivers to Bob, crashes before writing to DB → message lost from history
   ✅ Always persist to Cassandra FIRST, then deliver

❌ Storing WebSocket connection state only in server memory
   → Server crashes → all 50K connections drop their context → mess
   ✅ Store userId → serverId mapping in Redis; clients auto-reconnect with backoff

❌ Synchronous fan-out for large groups
   → 500-member group send takes 500ms → sender's UI feels slow
   ✅ Persist once, return "sent" immediately, fan out async via message bus

❌ Push notification reveals message content
   → "Alice: my password is xyz" visible on a stranger's lock screen
   ✅ Notification says "New message from Alice"; app fetches content on open

❌ Thundering herd on reconnect after outage
   → 50K clients all reconnect to the same recovered server at the same moment → overload again
   ✅ Exponential backoff with random jitter on the client

❌ Presence via polling
   → 200M users polling 100 contacts every 30s = 600M req/sec
   ✅ Subscribe to presence updates for contacts (push), don't poll

❌ Single sequence counter per chat
   → All messages for that chat serialize through one counter → bottleneck
   ✅ Use Snowflake IDs + client sequence number; reserve server counters for strict-ordering chats

❌ Embedding media bytes in messages
   → 5 MB photo × 1M msg/sec → 5 TB/sec through chat servers → impossible
   ✅ Presigned S3 upload; message carries just a URL

❌ Trusting "WebSocket connected" as the presence signal
   → Phones lose network silently all the time; ghost users appear online
   ✅ Heartbeat with TTL — if no heartbeat in 30s, treat as offline
```

## How to Answer This in an Interview (Step by Step)

```
1. State the scale:
   "2 billion users, 100B messages/day, ~1.15M messages/second. End-to-end delivery
   in under 100ms for online users. Messages must never be lost."

2. Choose the transport:
   "WebSockets — persistent bidirectional TCP. Lowest latency, server can push any
   time. About 50K connections per server, so 4,000 servers for 200M concurrent users.
   Need sticky load balancing so a user stays on the same server."

3. Walk through the send flow:
   "Client sends over WebSocket. Server assigns a Snowflake ID, persists to Cassandra
   FIRST for durability, then looks up the recipient's server in a Redis registry and
   publishes to a message bus channel. The destination server picks it up and pushes
   down its WebSocket."

4. Cross-server routing:
   "Each user's server is recorded in Redis (userId → serverId). Servers subscribe
   to channels for their connected users. This means publishers don't need to know
   about other servers individually — just publish to the right channel."

5. Offline delivery:
   "If no server subscribed → user is offline. Send push notification via APNs/FCM.
   Message stays in Cassandra. When user reconnects, server drains the backlog of
   undelivered messages."

6. Storage:
   "Cassandra partitioned by chat_id, clustered by sent_at descending. Reads like
   'last 50 messages' or 'older than time T' are single-partition scans. Snowflake
   IDs are time-sortable and globally unique."

7. Delivery receipts:
   "Three states: Sent (server persisted), Delivered (recipient device ACKed), Read
   (recipient opened the chat). Stored per (message, user) for group chats."

8. Presence:
   "Heartbeat every 15s with Redis TTL of 30s. If TTL expires without a refresh,
   user is offline. Last seen is stored on disconnect or TTL expiry."

9. Group chats:
   "Persist once with chat_id = group:family. Fan out asynchronously through the
   message bus to all members — don't block the sender on 500 individual pushes."

10. Media:
    "Bytes never go through chat servers. Client gets a presigned S3 URL, uploads
    directly. Message contains only the URL. Recipients download from CDN."

11. Ordering:
    "Snowflake IDs sort by time; client sequence numbers as a tiebreaker. For most
    chat workloads this is enough — no need for a global per-chat counter."
```
