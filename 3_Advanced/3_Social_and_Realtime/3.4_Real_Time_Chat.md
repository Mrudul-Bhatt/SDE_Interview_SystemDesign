# Real-Time Chat

---

## Problem Statement

Design a real-time messaging system where users can send and receive messages instantly, see delivery and read receipts, and get messages even when they were offline — at the scale of WhatsApp (100B messages/day) or Slack.

```
Use cases:
  1:1 chat         Direct messages between two users (WhatsApp, iMessage)
  Group chat       Messages delivered to all members (family group, Slack channel)
  Channels         Broadcast to subscribers (Telegram channels, Discord)
  Typing indicator Live "Alice is typing..." signals
  Presence         Online / offline / last seen status
  Media sharing    Images, videos, documents (separate upload flow)
```

---

## Requirements

### Functional
```
1. Send a message to a user or group
2. Messages delivered in real-time to online recipients
3. Messages queued and delivered when offline recipients reconnect
4. Push notifications for offline users
5. Delivery receipts: Sent → Delivered → Read (WhatsApp double-checkmark)
6. Message history: retrieve past messages with pagination
7. Presence: online status and last-seen timestamp
8. Group chat: up to 500 members per group
```

### Non-Functional
```
Scale:       2B users, 100B messages/day (WhatsApp scale)
Throughput:  ~1.15M messages/sec sustained
Latency:     < 100ms end-to-end for online recipients
Ordering:    Messages in a chat appear in the order sent
Durability:  Messages never lost (even if server crashes mid-send)
Availability: 99.99% — people expect chat to always work
```

---

## Capacity Estimation

```
Messages/sec:     100B/day ÷ 86400 = 1.15M/sec
Avg message size: 100 bytes (text only)
Storage/day:      1.15M × 100B × 86400 = 10 TB/day
Retention:        7 years for WhatsApp → 10 TB × 365 × 7 ≈ 25 PB

WebSocket connections:
  2B users; ~10% online simultaneously = 200M connections
  1 WebSocket server handles ~50K connections → 4000 servers
  (Each connection is persistent TCP; much cheaper than HTTP)

Message fan-out for group chat:
  1.15M msgs/sec; avg group size = 10 members
  → Up to 11.5M delivery events/sec
```

## Real-Time Transport: WebSocket vs Alternatives

```
HTTP Short Polling:
  Client polls every 5s: "any new messages?"
  PRO:  Simple, works everywhere
  CON:  High latency (up to 5s), massive wasted requests, poor battery life

HTTP Long Polling:
  Client sends request; server holds it open until a message arrives
  PRO:  Lower latency than short poll; works with proxies/firewalls
  CON:  One request per message; connection overhead; timeout management

Server-Sent Events (SSE):
  Server pushes events over HTTP; client reads stream
  PRO:  Simple API; auto-reconnect built in
  CON:  One direction only (client → server still needs separate HTTP)

WebSocket (chosen):
  Full-duplex TCP connection; both sides send anytime
  PRO:  Lowest latency; bidirectional; efficient (no HTTP headers per message)
  CON:  Long-lived connections require more server resources; needs sticky routing

→ Production: WebSocket for real-time delivery; HTTP fallback for restricted networks
```

## Message Delivery Flow

### Online Recipient
```
1. Alice sends message to Bob (both online)
   Client A → WebSocket → Chat Server 1

2. Chat Server 1:
   a. Assign message ID (Snowflake or UUID)
   b. Persist to Message Store (Cassandra) — durability first
   c. Look up Bob's server: registry says "Bob on Chat Server 2"
   d. Publish to Message Bus: channel=user:bob, payload=message

3. Message Bus routes to Chat Server 2 (subscribed to user:bob channel)

4. Chat Server 2:
   a. Finds Bob's WebSocket connection
   b. Pushes message to Bob's client
   c. Bob's client sends "delivered" ACK

5. Chat Server 2 updates delivery status → "Delivered" event back to Alice
```

### Offline Recipient
```
1-2: Same as above (persist + publish to bus)

3. Message Bus: no server subscribed for Bob (he's offline)

4. Chat Server 1 checks presence: Bob is offline
   → Call Push Notification Service (APNs for iOS, FCM for Android)
   → Send push: "New message from Alice"

5. When Bob comes online:
   a. Connects WebSocket → Chat Server 3
   b. Server 3 registers Bob in user registry
   c. Fetches undelivered messages from Message Store
   d. Delivers them in order
   e. Sends "Delivered" receipts for all
```

## Cross-Server Routing (The Key Scalability Challenge)

```
Problem: Alice (Server 1) sends to Bob (Server 2) — how does Server 1 know where Bob is?

Solution: User Connection Registry (Redis)
  On connect:   SET user_server:{userId} serverId EX 300 (5 min TTL)
  On message:   GET user_server:{userId} → serverId → forward to that server
  On disconnect: DEL user_server:{userId}

Message routing via Redis Pub/Sub:
  Each server subscribes to a set of user channels
  Server 1 publishes to channel: user:{bobId}
  Server 2 (subscribed to user:{bobId}) receives it
  Server 2 pushes to Bob's WebSocket

Alternative: Kafka partitioned by userId
  More durable than Redis Pub/Sub; handles server crashes
  Message not lost if destination server is restarting
```

## Message Storage (Cassandra)

```
Why Cassandra for chat?
  High write throughput (no leader bottleneck)
  Time-series natural: clustering by timestamp
  Wide rows: one row per chat_id, all messages as columns

Schema:
  Table: messages
    PRIMARY KEY: (chat_id, sent_at DESC, message_id)
    Columns: sender_id, content, message_type, delivery_status, media_url

  Table: user_chats  (which chats does a user belong to?)
    PRIMARY KEY: (user_id, last_activity DESC)
    Columns: chat_id, other_user_id (or group_name), unread_count

Read patterns:
  "Get last 50 messages in chat X" → single partition scan, O(1)
  "List all chats for user Y" → single partition scan
  "Load messages before timestamp T" → ZREVRANGEBYSCORE equivalent via clustering key

Message IDs:
  Snowflake IDs: time-sortable, globally unique, no coordination
  Useful as tiebreaker when two messages have same millisecond timestamp
```

## Delivery Receipts

```
States: SENT → DELIVERED → READ

Sent:       Server received and persisted the message
Delivered:  Recipient's device received the message
Read:       Recipient opened the conversation (read the message)

Implementation:
  Client ACK on receive → server updates status → notify sender
  Read receipt: client sends READ event when user opens chat
  Group chats: DELIVERED when ALL members receive; READ when ANY member reads
    (WhatsApp: per-member read status tracked separately)

Storage:
  Table: message_receipts
    PRIMARY KEY: (message_id, user_id)
    Columns: status (delivered/read), updated_at

  Or: embed as counters in message row (simpler, less precise)
```

## Presence (Online / Last Seen)

```
Approach: Redis TTL heartbeat
  Client sends heartbeat every 15s: SET presence:{userId} online EX 30
  If TTL expires (no heartbeat for 30s) → user considered offline
  "Last seen" = timestamp stored when TTL expires or explicit disconnect

Implementation:
  On heartbeat: SET presence:{userId} {timestamp} EX 30
  On disconnect: SET last_seen:{userId} {now}; DEL presence:{userId}
  Read online: EXISTS presence:{userId}
  Read last_seen: GET last_seen:{userId}

Privacy setting: users can hide last-seen (WhatsApp privacy setting)
  → Store flag in user profile; apply at read time
```

## Group Chat

```
Group members stored as a set:
  SADD group_members:{groupId} userId1 userId2 userId3
  SCARD group_members:{groupId}  → member count

Fan-out on send:
  1. Store message once (one write to Cassandra)
  2. For each member:
     a. Online → push via WebSocket
     b. Offline → push notification

Large groups (500+ members):
  Fan-out to 500 WebSocket connections per message
  At 1M msgs/sec × 500 members = 500M delivery events/sec → needs care
  Solution: tiered fan-out — server delivers to local connections first,
  then async fan-out to other servers via message bus
```

## Media Sharing

```
Why not embed media in message?
  100 KB photo × 1.15M msg/sec → 115 GB/sec through chat servers → impossible

Solution: presigned URL upload
  1. Client requests upload URL from Media Service
  2. Media Service generates S3 presigned URL (valid 5 min)
  3. Client uploads directly to S3 (bypasses chat servers)
  4. Client sends chat message with media_url reference (tiny)
  5. Recipients download from CDN (S3 + CloudFront)

Flow:
  Client → GET /media/upload-url → {presigned_url, media_id}
  Client → PUT {presigned_url} (file upload to S3)
  Client → SEND message {content="", media_url="cdn.../media_id"}
```

## Message Ordering

```
Problem: Alice sends msg A, then msg B; server receives B first due to network jitter
  → Bob sees B before A — wrong order

Solutions:
  Client sequence number: client assigns monotonic seq per chat
    Receiver buffers out-of-order messages; displays in seq order

  Server-side ordering: server assigns sequence number per chat
    Requires a per-chat counter (Redis INCR or DB sequence)
    Serializes all writes for a chat → can be a bottleneck at high volume

  Hybrid (WhatsApp/iMessage):
    Snowflake message ID (time-sortable) as primary sort key
    Close-in-time ties resolved by sender's client sequence
    → No single lock; acceptable ordering for normal latency
```

## Pitfalls

```
1. Storing WebSocket state in chat server memory only
   Problem: server crash loses all connection state; clients must reconnect
   Fix: connection registry in Redis; clients auto-reconnect with exponential backoff

2. Fan-out to large groups blocks the send path
   Problem: sending to 500-member group takes 500ms if done synchronously
   Fix: async fan-out via message bus; return "sent" to sender immediately

3. Not persisting before delivering
   Problem: server delivers to online user but crashes before writing to DB
   → Message lost from history even though recipient saw it
   Fix: always write to Cassandra first, then deliver

4. Push notification reveals message content
   Problem: "You have a new message: {content}" visible on lock screen
   Fix: push notification = "you have 1 new message" (no content); app fetches on open

5. Thundering herd on reconnect
   Problem: server outage → 50K clients reconnect simultaneously → overload
   Fix: exponential backoff with jitter on client reconnect

6. Presence polling overload
   Problem: 200M users each polling 100 contacts every 30s = 600M req/sec
   Fix: presence by subscription (subscribe to updates for contacts you care about, not polling)

7. Message ordering with multiple devices
   Problem: user has phone + laptop; sequence numbers diverge per device
   Fix: server-side sequence per chat; all devices sync from server as source of truth
```

## Real-World Systems

```
WhatsApp:    Erlang (high-concurrency); XMPP protocol; Mnesia + Cassandra
Slack:       WebSocket + Kafka; Vitess (MySQL sharding); channels as ordered logs
Discord:     Cassandra for message history; Elixir + Rust for gateway; 250K msg/sec
iMessage:    APNs (Apple Push Notification Service); end-to-end encryption
Signal:      Signal Protocol (double ratchet E2E encryption); sealed sender
```

## Interview Answer Framework

1. **Transport:** WebSocket (full-duplex, low latency); HTTP long poll fallback; sticky LB
2. **Message flow:** persist first → publish to message bus → deliver online → push offline
3. **Cross-server routing:** Redis user→server registry; Redis pub/sub or Kafka for inter-server
4. **Storage:** Cassandra partitioned by chat_id + time-clustered; Snowflake IDs for ordering
5. **Delivery receipts:** SENT (stored) → DELIVERED (device ACK) → READ (user opened chat)
6. **Offline:** push notification (APNs/FCM) on offline; deliver buffered messages on reconnect
7. **Group chat:** store once; fan-out to N members; async for large groups
8. **Presence:** Redis TTL heartbeat (SET EX 30); last_seen on disconnect
9. **Media:** presigned S3 upload; message carries URL reference only
10. **Ordering:** Snowflake IDs (time-sortable) + client sequence; server seq number for strict ordering
