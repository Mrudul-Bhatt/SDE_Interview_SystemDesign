# Collaborative Document Editing

---

## Problem Statement

Design a system where multiple users edit the same document simultaneously and see each other's changes in real time — like Google Docs, Notion, or Figma. Every user's screen must converge to the same content even when edits arrive out of order or conflict.

```
Use cases:
  Google Docs    Text editing for millions of concurrent documents
  Notion         Block-based pages, databases, embedded content
  Figma          Real-time design collaboration (vector graphics)
  VS Code Live   Code collaboration with shared cursors and terminals
  Confluence     Team wikis with concurrent editing
```

---

## Requirements

### Functional
```
1. Multiple users edit a document simultaneously
2. All clients see each other's changes in real time (< 200ms latency)
3. No edit is lost: concurrent edits from different users both survive
4. All clients converge to the same document state
5. Cursor positions of other users are visible (colored carets)
6. Undo/redo works correctly even with concurrent edits
7. Document persisted durably; offline edits sync on reconnect
```

### Non-Functional
```
Scale:       1B documents (Google Docs); 10M concurrent active documents
Latency:     Local edit appears instantly (optimistic local apply)
             Server-confirmed version < 200ms
Consistency: Eventual — all clients converge; no single "ground truth" lock
Durability:  Operations logged; document reconstructible from operation history
```

---

## The Core Problem: Concurrent Edits

```
Initial document: "Hello"
                   01234   (positions)

Alice types 'X' at position 5 → "HelloX"
Bob   types 'Y' at position 5 → "HelloY"

Both sent to server simultaneously. Naïve apply:
  Server applies Alice first: "HelloX"
  Server applies Bob (pos=5):  "HelloXY"  ← but Bob intended to append after "Hello"

Correct result: "HelloXY" or "HelloYX" depending on tie-break — either is valid
but BOTH clients must see the SAME result.

The challenge: transform each operation to account for concurrent operations
               applied before it, so all clients converge.
```

---

## Two Approaches: OT vs CRDT

### Operational Transformation (OT)
```
Used by: Google Docs (Jupiter protocol)

Idea: Every edit is an "operation" (Insert or Delete at a position).
      When operations arrive out of order, transform them against
      the concurrent operations that were already applied.

Core function: Transform(op1, op2) → op1'
  op1' has the same intent as op1, but adjusted for op2 being applied first.

Example:
  Doc = "Hello"
  Client A: Insert(pos=5, char='X')   → sends to server
  Client B: Insert(pos=2, char='Z')   → sends to server concurrently

  Server receives A first, applies: "HelloX"
  Server receives B (pos=2). Must transform B against A:
    A was Insert at pos=5; B is Insert at pos=2.
    B's pos=2 < A's pos=5 → A doesn't affect B's position.
    Transformed B = Insert(pos=2, char='Z') unchanged.
  Server applies transformed B: "HeZlloX"

  Server broadcasts transformed ops to all clients.
  Client A already has "HelloX". Receives B transformed → applies Insert(pos=2, 'Z') → "HeZlloX" ✓
  Client B already has "HeZllo". Receives A transformed → Insert must shift right because B was before A's pos.
    Transform A against B: B inserted at pos=2, which is < A's pos=5 → A shifts to pos=6.
    Client B applies Insert(pos=6, 'X') to "HeZllo" → "HeZlloX" ✓

All clients converge to "HeZlloX".

Transform rules (Insert vs Insert):
  Transform(Insert(p1, c1), Insert(p2, c2)):
    if p2 < p1           → new_p1 = p1 + 1  (concurrent insert shifted our position right)
    if p2 > p1           → new_p1 = p1       (concurrent insert was after us, no shift)
    if p2 == p1          → tie-break by clientId (alphabetical) to get deterministic order

Transform rules (Insert vs Delete):
  Transform(Insert(p1, c1), Delete(p2)):
    if p2 < p1           → new_p1 = p1 - 1  (concurrent delete freed a slot before us)
    else                 → new_p1 = p1

Transform rules (Delete vs Insert):
  Transform(Delete(p1), Insert(p2, c2)):
    if p2 <= p1          → new_p1 = p1 + 1
    else                 → new_p1 = p1

Transform rules (Delete vs Delete):
  Transform(Delete(p1), Delete(p2)):
    if p2 < p1           → new_p1 = p1 - 1
    if p2 == p1          → NoOp (character already gone)
    if p2 > p1           → new_p1 = p1

Limitations:
  OT is complex to implement correctly (especially 3+ concurrent clients)
  Requires a central server to serialize and transform all ops
  Hard to scale beyond a single authoritative server
```

### CRDT (Conflict-free Replicated Data Type)
```
Used by: Figma (design elements), Automerge, Yjs, Atom Teletype

Idea: Give every character a globally unique, immutable ID.
      Position is defined by ID relationships, not integer indices.
      Concurrent inserts can be merged deterministically without a server.

Structure (RGA — Replicated Growable Array):
  Each character: { id: (clientId, seqNum), value: 'H', deleted: false }
  Characters linked in a list: head → c1 → c2 → c3 → ... → tail

Insert(after: charId, value: 'X', id: (alice, 7)):
  → Insert new char node with id=(alice,7) after the node with charId
  → If two clients insert after the same charId: tie-break by clientId

Delete(charId):
  → Mark character as deleted=true (tombstone)
  → Character stays in list (for future position calculations)
  → Rendered invisibly

Properties:
  Commutative: apply ops in any order → same result
  Idempotent: applying same op twice → same result
  No transformation needed: just merge
  Scales to P2P: no central server required for conflict resolution

Trade-off:
  Tombstones accumulate → list grows over time (GC needed)
  Higher memory than plain string for large documents
```

## Architecture

```
┌────────────────────────────────────────────────────────────────────┐
│                  Collaborative Editing System                       │
│                                                                     │
│  Client A ──WS──▶ ┌──────────────────────┐                         │
│  Client B ──WS──▶ │  Collab Server        │                         │
│  Client C ──WS──▶ │  - Op queue per doc   │                         │
│                   │  - OT transformer     │                         │
│                   │  - Version tracking   │                         │
│                   └──────────┬───────────┘                          │
│                              │                                      │
│               ┌──────────────┼──────────────┐                       │
│               ▼              ▼              ▼                       │
│         ┌──────────┐  ┌──────────┐  ┌──────────┐                   │
│         │ Op Log   │  │ Doc Store│  │ Presence │                   │
│         │(Cassandra│  │(snapshot │  │ (Redis)  │                   │
│         │ per doc) │  │+ ops)    │  │cursors   │                   │
│         └──────────┘  └──────────┘  └──────────┘                   │
└────────────────────────────────────────────────────────────────────┘

Client-side flow:
  1. User types → apply locally (instant feedback = optimistic UI)
  2. Send op to server with client's current version number
  3. Server transforms op → applies to server doc → broadcasts to others
  4. Client receives others' ops → transforms against pending local ops → applies
```

## Operation Log & Versioning

```
Every operation stored durably:
  doc_id:     "doc:abc123"
  version:    42  (monotonically increasing per document)
  client_id:  "alice-session-7"
  op_type:    INSERT | DELETE
  position:   5
  character:  'X'  (for INSERT)
  timestamp:  2024-01-15T19:00:00Z

Document reconstruction:
  Start from snapshot at version N
  Replay all ops from version N+1 to latest
  → Any version of the document is recoverable

Snapshots:
  Full document text saved periodically (e.g., every 1000 ops)
  Avoids replaying 100K operations on every load
  Stored in S3 or document DB

Load document:
  1. Fetch latest snapshot (version N)
  2. Apply ops from N+1 to latest
  3. Send snapshot + recent ops to client
  4. Client renders current state; joins live update stream
```

## Cursor Synchronization

```
Each client broadcasts their cursor position to others:
  { clientId: "alice", docId: "doc:abc", position: 47, color: "#4285F4" }

Server broadcasts to all co-editors in the document room.

Challenge: cursor positions must be transformed when remote ops arrive.
  Alice's cursor is at position 20.
  Bob inserts a character at position 5.
  Alice's cursor shifts to position 21 (same character, just shifted).

Implementation:
  Treat cursor as a "retain" operation — transform it like an insert position.
  Client applies cursor transforms whenever a remote op arrives.

Storage:
  Cursor positions are ephemeral (Redis, TTL=5s)
  Lost on disconnect → cursor disappears from other clients
```

## Undo/Redo in Collaborative Context

```
Problem: simple stack-based undo breaks with concurrent edits.
  Alice types 'X', Bob types 'Y', Alice undoes → what gets undone?

Solution: selective undo
  Each undo is a new operation (invert of the original op, transformed to current state)
  Undo of Insert(pos=5,'X') = Delete(pos=5') where pos=5' is transformed against
    all ops that happened after the original insert

  → Undo never "undoes" another user's work
  → Undo is commutative with concurrent edits

Implementation:
  Each client maintains its own undo stack of its OWN operations (not others')
  Undo applies the inverse, transformed against the current document state
  This is called "local undo" — the standard in Google Docs
```

## Pitfalls

```
1. Applying OT transform incorrectly (the "tombstone problem")
   Problem: Delete-then-Insert at same position → wrong position after transform
   Fix: thorough test suite for all 4 op-pair combinations; use a proven library (Automerge, ShareDB)

2. Client version out of sync
   Problem: client sends op at version 5; server is at version 8; transform against 3 ops
   Fix: server stores all ops; client sends version; server replays transforms from that version

3. No operation log (only storing final text)
   Problem: cannot reconstruct history; cannot undo; cannot replay to join mid-session
   Fix: append-only op log is the source of truth; snapshots are just caches

4. Giant documents with no snapshots
   Problem: loading a 100K-op doc requires replaying all ops on every client open
   Fix: periodic snapshots; only replay delta since last snapshot

5. Network partition (client offline)
   Problem: client edits while offline; reconnects; 500 ops need to be reconciled
   Fix: client queues ops with version; on reconnect, server batch-transforms all queued ops

6. Presence not cleaned up on disconnect
   Problem: ghost cursors remain visible after user leaves
   Fix: Redis TTL on cursor keys; WebSocket disconnect event triggers cleanup

7. Race condition: two servers handle same document
   Problem: if two Collab Servers receive ops for the same doc, they each transform independently
   Fix: route all ops for a document to the same server (consistent hashing by doc_id)
        or use a single-writer lock per document in Redis
```

## Real-World Systems

```
Google Docs:   Jupiter OT protocol; single authoritative server per doc shard
Figma:         CRDT for design tree; WebSocket; Rust server
Notion:        Block-based structure; each block is a CRDT unit
Yjs:           Open-source CRDT library; used by many editors
Automerge:     CRDT with JSON data model; works offline
ShareDB:       OT server library (Node.js); used by Quill, CodeMirror integrations
```

## Interview Answer Framework

1. **Core problem:** concurrent edits conflict; need merge strategy; all clients must converge
2. **OT vs CRDT:** OT = transform operations server-side (Google Docs); CRDT = conflict-free merge without server (Figma, Yjs); OT is simpler to reason about; CRDT scales better
3. **OT transform rules:** Insert vs Insert (shift position if before), Insert vs Delete, Delete vs Delete (NoOp if same pos)
4. **Architecture:** WebSocket per client; collab server serializes ops; transform → apply → broadcast
5. **Operation log:** every op stored with version; document = snapshot + replay ops; enables history/undo
6. **Cursor sync:** broadcast cursor position; transform position when remote op arrives; ephemeral in Redis
7. **Undo:** selective local undo = inverse op transformed against current state; never undoes others' work
8. **Offline:** queue ops locally; on reconnect, server batch-transforms all queued ops from client's version
9. **Scale:** shard documents by doc_id; one server owns each document to serialize transforms; snapshot every N ops
10. **Pitfalls:** no op log, giant doc without snapshots, ghost cursors, version mismatch, multiple servers for same doc
