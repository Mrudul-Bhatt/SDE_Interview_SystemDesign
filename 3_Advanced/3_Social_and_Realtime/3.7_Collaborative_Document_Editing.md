# Collaborative Document Editing (like Google Docs)

---

## What Are We Building?

You and three coworkers open the same Google Doc. You all start typing at the same time.

```
Alice types in paragraph 1: "Welcome to..."
Bob fixes a typo in paragraph 3.
Carol deletes a sentence in paragraph 5.
Dave moves his cursor around, highlighting his selection in green.

All four of you see each other's edits appear in real time, in your own browser,
within about 100 milliseconds. Nobody's edit is lost. Nobody overwrites anybody.
Everyone ends up looking at the exact same document.
```

This is **collaborative editing**, and it's surprisingly hard. The core challenge:

> What happens when two people edit the same spot at the exact same moment?

If we're not careful, one edit overwrites the other. Or worse — different users end up
with different versions of the document, and they never reconverge.

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Multiple users edit the same document simultaneously
2. Everyone sees each other's changes in real time (< 200ms)
3. No edit is ever lost — concurrent edits both survive
4. Every client converges to the same final document state
5. Show other users' cursors and selections (colored carets)
6. Undo/redo works correctly even when others are editing
7. Documents are saved durably; offline edits sync on reconnect
```

### Non-Functional (scale and reliability)
```
Scale:        1 billion documents (Google Docs scale)
              ~10 million actively being edited at any moment
Latency:      Your own typing appears INSTANTLY (no waiting for server)
              Other people's edits appear within 200ms
Consistency:  All clients eventually converge to the same state
Durability:   Every operation logged; document recoverable from history
```

---

## The Core Problem: Two People Editing the Same Spot

Let's see exactly how things go wrong if we're naïve about this.

```
Starting document:  "Hello"
                     01234   ← positions

Alice (in her browser): types 'X' at position 5  → her doc shows "HelloX"
Bob   (in his browser): types 'Y' at position 5  → his doc shows "HelloY"

Both clients send their edit to the server at the same moment:
  Alice's op:  Insert(position=5, char='X')
  Bob's op:    Insert(position=5, char='Y')

If the server just applies them in arrival order:
  Server applies Alice → "HelloX"
  Server applies Bob (at pos=5) → "HelXloY"  ❌ wrong!
```

What went wrong? Bob's "position 5" was based on the original document "Hello". After Alice's
insert, that position no longer refers to the same place. We need to **adjust Bob's operation**
based on what Alice did.

The correct result is either "HelloXY" or "HelloYX" — but **both Alice and Bob must end up
seeing the same result**, even if it isn't the one they originally intended.

There are two main strategies for solving this: **OT** and **CRDT**.

---

## Strategy 1: Operational Transformation (OT) — Google Docs' Approach

The idea: every edit is an "operation" (Insert or Delete at a position). When operations
arrive out of order, the server **transforms** each one to account for the operations that
were already applied.

### The Transform Function

Think of it like adjusting directions after a road closure:
- You were told: "turn right at the 5th intersection"
- But one new street was just added between intersections 2 and 3
- Now your "5th intersection" should actually be the 6th
- The transform function does this adjustment automatically

### Example: Two Inserts at Different Positions

```
Starting doc:  "Hello"

Alice: Insert(position=5, char='X')  → her local view becomes "HelloX"
Bob:   Insert(position=2, char='Z')  → his local view becomes "HeZllo"

Server receives Alice's op first → applies it: server doc = "HelloX"

Server receives Bob's op (position=2). Need to transform it against Alice's op:
  Alice inserted at position 5. Bob inserts at position 2.
  Bob's position 2 is BEFORE Alice's position 5 → Alice's insert doesn't shift Bob.
  → Transformed Bob's op = Insert(position=2, 'Z') unchanged.

Server applies transformed Bob: "HeZlloX"

Server broadcasts the transformed ops back to all clients:
  Alice gets Bob's op. Her doc is "HelloX". She applies Insert(pos=2, 'Z') → "HeZlloX" ✓
  Bob gets Alice's op. His doc is "HeZllo". But wait — Alice's "position 5" was in the
    ORIGINAL doc, before Bob's Z. After Bob's insert, that position needs to shift right.
    Transformed Alice's op = Insert(position=6, 'X')
  Bob applies it → "HeZlloX" ✓

All three views converge to "HeZlloX". ✓
```

### The Transform Rules

```
Two inserts:
  If concurrent insert was at a position BEFORE mine → my position shifts right by 1
  If concurrent insert was AFTER mine                → no change
  If at the same position                            → break the tie by client ID

Insert vs Delete:
  If the delete was BEFORE my insert → my position shifts LEFT by 1 (a slot was freed)
  Otherwise                          → no change

Delete vs Delete at the same position:
  → NoOp (the character is already gone; deleting it again would delete the WRONG character)
```

### The Downsides of OT

```
✅ Conceptually clean: insert/delete operations are easy to reason about
❌ The transform rules get tricky with 3+ simultaneous clients
❌ Requires a CENTRAL SERVER to serialize and transform all ops (single point of coordination)
❌ Implementing it correctly is famously hard — most attempts have bugs
```

Google Docs uses OT (their Jupiter protocol). It works, but Google had to put a lot of
engineering into making it correct.

## Strategy 2: CRDT (Conflict-Free Replicated Data Type) — Figma's Approach

CRDTs take a completely different angle: **don't use positions at all.**

### The Idea

Instead of representing the document as a string with integer positions, represent it as a
linked list where every character has a **globally unique ID**.

```
Document "Hello" stored as a linked list:

head → [id: (alice,1), val: 'H'] → [id: (alice,2), val: 'e'] → [id: (alice,3), val: 'l']
     → [id: (alice,4), val: 'l'] → [id: (alice,5), val: 'o'] → tail
```

Each character ID is `(clientId, sequenceNumber)` — unique forever, never reused.

### Insert

"Insert character after this specific ID":
```
Alice does: Insert "X" after the node with id (alice,5)
  → new node: [id: (alice,6), val: 'X']
  → linked in after (alice,5)

This op carries the ID of the character it's inserted AFTER — not a position.
So when other clients receive it, they know exactly where to put it,
regardless of what else has changed.
```

### Concurrent Inserts at the Same Spot

What if Alice and Bob both insert after the same character?

```
Both do: Insert "X" / "Y" after (alice,5)

Both ops eventually arrive at every client. We need a deterministic rule:
  → Break the tie by clientId (alphabetical or numerical)
  → "alice" < "bob", so alice's X comes first

Final order: ...(alice,5), X, Y...

Every client applies the same tie-break → every client gets the same result. ✓
```

### Delete

Delete just marks the character as a **tombstone** instead of removing it:
```
Delete (alice,5) → mark the node as deleted=true; don't remove from the list

Why keep it? Because future operations might reference it ("insert after this ID").
If we removed it, those operations would have nowhere to anchor.
```

### Why CRDTs Are Powerful

```
✅ Commutative: operations can be applied in any order, result is the same
✅ Idempotent: applying the same op twice doesn't change anything
✅ No central server needed for conflict resolution — works peer-to-peer
✅ Works offline naturally — apply local edits, sync the ops later, everything merges

❌ Tombstones never get removed, so the data structure grows over time
❌ More memory than a plain string
❌ Implementing position-stable IDs correctly is tricky
```

Figma uses CRDTs because their design canvas needs to work seamlessly even with brief
network drops. Yjs and Automerge are popular open-source CRDT libraries.

## The Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│              Collaborative Editing System                         │
│                                                                   │
│  Alice's browser ──WebSocket──▶                                  │
│  Bob's browser   ──WebSocket──▶ ┌────────────────────────┐       │
│  Carol's browser ──WebSocket──▶ │   Collab Server         │       │
│                                  │   - per-doc op queue    │       │
│                                  │   - OT transformer      │       │
│                                  │   - version tracking    │       │
│                                  └──────────┬─────────────┘       │
│                                             │                     │
│              ┌──────────────────┬───────────┴────────────┐        │
│              ▼                  ▼                        ▼        │
│        ┌──────────┐      ┌──────────┐           ┌──────────┐      │
│        │ Op Log   │      │ Snapshot │           │ Presence │      │
│        │ Cassandra│      │ Store    │           │ (Redis)  │      │
│        │ per doc  │      │ (S3)     │           │ cursors  │      │
│        └──────────┘      └──────────┘           └──────────┘      │
└──────────────────────────────────────────────────────────────────┘
```

### The Client-Side Flow (Optimistic UI)

This is the trick that makes typing feel instant even though it has to round-trip through a server.

```
1. You type 'X'.
2. Your browser applies it LOCALLY, immediately — you see 'X' in 0ms.
3. Your browser sends the op to the server in the background.
4. The server may transform your op (if other people sent ops at the same time)
   and broadcasts the transformed version.
5. When you get the server's response, you adjust if needed (rare).
```

You never wait for the server. This is called **optimistic application**.

When the server's response arrives, your local version is updated. If you typed 10 characters
before the server caught up, those 10 are kept in a "pending" queue. The server transforms
incoming remote ops against your pending ops so nothing gets misaligned.

## The Operation Log: How Documents Are Stored

**Surprise:** the document's "text" isn't actually the source of truth. The **log of operations is.**

```
Every op stored in Cassandra:
  doc_id:    "doc:abc123"
  version:   42
  client_id: "alice-session-7"
  op_type:   INSERT
  position:  5
  character: 'X'
  timestamp: 2024-01-15T19:00:00Z
```

The document text is derivable: start from empty, replay all ops in order → current state.

### Snapshots: So We Don't Replay 100,000 Ops Every Time

If a doc has 100K ops, loading it would be slow.

```
Every 1000 ops, save a snapshot:
  doc_id:    "doc:abc123"
  version:   40,000
  text:      "Welcome to our Q1 planning document..."

Loading the doc:
  1. Fetch the latest snapshot (e.g., at version 40,000)
  2. Replay ops 40,001 onward (at most ~1000 ops) → current state
  3. Send to client
  4. Client joins the live stream
```

Snapshots are just an optimization. If they're lost, you can always rebuild from the op log.
The op log is sacred.

## Showing Other Users' Cursors

When Alice's cursor is at position 47, other clients see a colored caret there labeled "Alice".

```
Alice broadcasts: { clientId: "alice", docId: "doc:abc", position: 47, color: "#4285F4" }
Server broadcasts to all other clients in the room.
Each client renders Alice's caret at her position.
```

### Cursors Must Be Transformed Too

If Bob inserts a character at position 5, Alice's cursor (at position 47) should shift to 48 —
it's still pointing at the same letter, just one slot to the right.

```
On each remote op:
  If the op's position < my cursor position → shift my cursor right by 1 (for inserts)
  Treat the cursor like a "retain" operation in OT — it transforms the same way.
```

### Cursors Are Ephemeral

```
Stored in Redis with 5-second TTL.
If your WebSocket drops, your cursor disappears from everyone's view after a few seconds.
No need to persist cursors to a database — they have no historical value.
```

## Undo and Redo (Surprisingly Subtle)

Naïve undo: pop the last action off a stack and reverse it. **This breaks horribly in collaboration.**

```
Alice types 'X'.
Bob types 'Y'.
Alice hits Ctrl+Z.

Naïve undo: pop the LAST op → that's Bob's 'Y' → undo Bob's 'Y'.
But Alice shouldn't be able to undo Bob's work!
```

### Selective Undo (What Google Docs Does)

Each client keeps an undo stack of **only its own operations**. When you hit Ctrl+Z:

```
1. Find your most recent op (e.g., Insert(pos=5, 'X'))
2. Compute the inverse: Delete(pos=5)
3. TRANSFORM that inverse against all operations that happened AFTER yours
   (your 'X' might no longer be at position 5 — Bob may have inserted before it)
4. Apply the transformed undo as a new operation
```

Your undo affects only your own contribution. Bob's 'Y' is never touched.
Redo is the same idea — invert your last undo and reapply.

## What Happens When Someone Goes Offline?

```
Alice's WiFi drops. She keeps typing for 5 minutes — 200 ops queued locally.

Eventually her connection returns:
  1. Her client reconnects to the WebSocket
  2. Server sees her at version 5,000; current version is 5,150 (others edited too)
  3. Client sends its 200 queued ops, tagged with version 5,000
  4. Server transforms each one against ops 5,001–5,150 → applies them
  5. Server broadcasts the transformed ops to everyone (including Alice)
  6. Alice's client receives ops 5,001–5,350 → applies what's new
  7. Everyone converges
```

This is where CRDTs shine — they handle offline naturally because operations carry their own
context (which character ID to insert after) rather than depending on a position that might
have shifted.

## How It All Fits Together

```
─── A user types a character ───────────────────────────────────────

Alice presses 'X' at position 5:
       │
       ▼
   Browser applies locally: "HelloX"   (0ms — instant feedback)
       │
       ├─ Stores op in pending queue
       └─ Sends to server over WebSocket:
              { op: Insert(5, 'X'), based_on_version: 42 }

   Collab Server (for doc:abc):
       ├─ Look up current server version: 45 (3 ops happened since Alice's base)
       ├─ Transform Alice's op against ops 43, 44, 45
       ├─ Apply transformed op → server doc updated → new version 46
       ├─ Append to Cassandra op log
       └─ Broadcast transformed op to Bob, Carol, etc.

   Each remote client:
       ├─ Receives the transformed op
       ├─ Transforms it against their own pending local ops
       ├─ Applies it
       └─ Updates cursor positions as needed

─── Loading a document ─────────────────────────────────────────────

Client requests doc:abc:
       │
       ▼
   Server:
       ├─ Fetch latest snapshot (version 40,000)
       ├─ Replay ops 40,001 to current (say 40,847)
       └─ Return: snapshot + recent ops + current version

   Client renders → joins WebSocket → live updates start streaming
```

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ Storing only the final text (no operation log)
   → Can't reconstruct history, can't do undo, can't replay for late joiners
   ✅ The op log is the source of truth; document text is derivable from it

❌ No snapshots; replaying full op log on every load
   → 100K-op doc takes 30 seconds to load
   ✅ Snapshot every 1000 ops; replay only the delta

❌ Implementing OT transform rules from scratch
   → Subtle bugs cause divergence with 3+ concurrent clients
   ✅ Use a battle-tested library (ShareDB, Yjs, Automerge) unless you have very specific needs

❌ Two servers handling ops for the same document
   → They transform independently → divergence
   ✅ Route all ops for one document to a single server (consistent hash by doc_id)

❌ Waiting for server confirmation before showing your own typing
   → Feels laggy and broken
   ✅ Optimistic local apply — show the edit immediately, reconcile when the server responds

❌ Not handling client-version mismatch on reconnect
   → Offline client sends ops based on version 5000; server is at 5150 → applied at wrong positions
   ✅ Server transforms each queued op against all ops since the client's base version

❌ Cursors persisted to the database
   → Pointless overhead; ghost cursors from disconnected users
   ✅ Cursors in Redis with short TTL; cleaned up automatically on disconnect

❌ Undoing other users' edits
   → User accidentally erases their coworker's paragraph
   ✅ Selective undo — each user's undo stack only contains their OWN operations

❌ CRDT tombstones grow forever
   → Document with millions of edits eventually has gigabytes of dead tombstones
   ✅ Background garbage collection that removes tombstones once no live op references them
```

## Real Systems at a Glance

```
Google Docs:   OT via the Jupiter protocol; one authoritative server per doc shard
Figma:         CRDT for design tree (positions, sizes, properties); Rust server; WebSocket
Notion:        Block-based — each block (paragraph, heading) is a CRDT unit
Yjs:           Open-source CRDT library used by many editors
Automerge:     CRDT with a JSON-like data model; designed for offline-first apps
ShareDB:       OT server library (Node.js); used with Quill, CodeMirror
VS Code Live:  Microsoft's collaboration framework; OT-based, with shared cursors and terminals
```

## How to Answer This in an Interview (Step by Step)

```
1. State the core problem:
   "Multiple users editing the same document simultaneously. We need every client to
   converge to the same state, even when edits arrive out of order. Two people typing
   at the same position must both have their edits preserved."

2. Compare the two approaches:
   "There are two strategies. Operational Transformation (OT) — used by Google Docs —
   transforms each operation against concurrent ones to adjust positions. CRDTs — used
   by Figma — give every character a unique ID so that ordering is intrinsic, and
   concurrent edits merge automatically without a central server."

3. Walk through OT with an example:
   "If Alice inserts at position 5 and Bob inserts at position 2 concurrently, when
   the server applies Bob's op AFTER Alice's, it doesn't shift because Bob's position
   was before Alice's. But when Alice's op reaches Bob, it must shift right by 1 to
   account for Bob's earlier insert. The transform function makes everyone converge."

4. Explain optimistic UI:
   "On the client, edits are applied LOCALLY immediately — the user never waits for
   the server. The op is sent in the background, transformed by the server, and
   broadcast. The client maintains a 'pending' queue to reconcile with incoming
   remote ops."

5. Describe the data model:
   "The source of truth is an append-only operation log per document, stored in
   Cassandra. The document text is derivable by replaying ops. Snapshots every 1000
   ops let us load fast without replaying everything."

6. Describe the architecture:
   "WebSocket per client. A 'collab server' owns each document (consistent hashing
   by doc_id) — that single server serializes and transforms all ops for one doc,
   then broadcasts them to other clients in the room."

7. Explain cursor sync:
   "Each client broadcasts its cursor position. Remote ops shift other users' cursors
   in the same way they shift insert positions. Cursors are ephemeral — stored in
   Redis with a 5-second TTL."

8. Explain undo:
   "Each user has their own undo stack containing only their own ops. Undo computes
   the inverse op, transformed against all subsequent ops, and applies it as a new
   operation. Nobody can undo someone else's work."

9. Offline editing:
   "Client queues ops locally with its base version. On reconnect, the server batch-
   transforms each queued op against everything that happened in the meantime. CRDTs
   handle this more gracefully than OT because they don't depend on positions."

10. Cite the main pitfalls:
    "Two servers handling the same doc (avoid with consistent hashing). No op log
    (store ops, derive text). Naïve undo (use selective undo). Implementing OT from
    scratch (use a battle-tested library)."
```
