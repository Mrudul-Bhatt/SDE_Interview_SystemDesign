# Social Media Feed

---

## Problem Statement

Design a news feed system that shows a user a stream of posts from people they follow, ranked by recency or relevance — the core product experience of Twitter/X, Instagram, Facebook, and LinkedIn.

```
Use cases:
  Home timeline     Posts from accounts you follow (Twitter/X, Instagram)
  News feed         Posts + shares + reactions (Facebook)
  Company feed      Articles from your network (LinkedIn)
  Explore feed      Posts from accounts you don't follow (trending content)
```

---

## Requirements

### Functional
```
1. Users can follow / unfollow other users
2. Users can create posts (text, image URL, video URL)
3. Users can like / comment on posts
4. GET /feed returns the user's personalized feed (paginated)
5. Feed contains posts from followed accounts
6. Feed supports both chronological and ranked (algorithmic) ordering
7. New posts appear in followers' feeds within seconds
```

### Non-Functional
```
Scale:       300M users, 200M DAU (Twitter-scale)
Posts:       500M posts/day = ~5800 posts/sec peak
Feed reads:  Timeline opened ~8 times/day per DAU = 18.5K reads/sec
Latency:     Feed load < 200ms (p99)
Consistency: Eventual — slight delay in new posts appearing is acceptable
Storage:     500M posts/day × 1 KB = 500 GB/day (keep 90 days = 45 TB)
```

---

## Capacity Estimation

```
Posts per second:   500M/day ÷ 86400s = ~5800/sec
Feed reads/sec:     200M DAU × 8 views/day ÷ 86400s = ~18500/sec

Fan-out cost:
  Average followers: ~200 per user
  5800 posts/sec × 200 followers = 1.16M feed-writes/sec
  → Feed cache must handle 1.16M writes/sec + 18.5K reads/sec

Celebrity problem:
  Elon Musk: 170M followers
  1 tweet → 170M feed-write operations → 170M cache writes
  At 1M writes/sec → takes 170 seconds to fan out → unacceptable
  Solution: hybrid fan-out (skip celebrities at write time, pull at read time)

Feed cache size per user:
  Store last 1000 post IDs × 8 bytes = 8 KB per user
  200M users × 8 KB = 1.6 TB total feed cache (Redis cluster)
```

---

## The Fan-out Problem

```
When Alice (500 followers) posts:
  Fan-out on Write (Push model):
    Post stored → write postId into each follower's feed cache immediately
    PRO:  feed reads are instant (pre-computed)
    CON:  write amplification: 500 followers = 500 cache writes per post
          Celebrities with 50M followers make this unworkable

  Fan-out on Read (Pull model):
    Post stored → do nothing
    When Bob opens feed: pull latest posts from every account Bob follows
    PRO:  no write amplification
    CON:  read-time scatter-gather: if Bob follows 500 people → 500 DB reads
          → High read latency

  Hybrid (production approach):
    Regular users (< 10M followers): fan-out on write → pre-computed feed
    Celebrity users (≥ 10M followers): skip fan-out on write
    At read time: merge pre-computed feed + fresh pull from celebrity accounts
    → Bounded write amplification + fast reads for most users
```

## Follow Graph Storage

```
Data model:
  followers:{userId}  → Set of user IDs who follow this user
  following:{userId}  → Set of user IDs this user follows

Redis implementation:
  SADD followers:alice bob         # bob follows alice
  SADD following:bob   alice       # bob follows alice (symmetric)
  SCARD followers:alice            # follower count (O(1))
  SMEMBERS followers:alice         # all followers (O(N))

For very large follower lists (celebrities):
  Paginate via SSCAN cursor — don't load 50M members at once
  Store in sharded Redis: followers:{userId}:{shard}

Cassandra alternative (for large graphs):
  Table: follows (follower_id, followee_id, created_at) PK=(follower_id, followee_id)
  Table: followers (followee_id, follower_id, created_at) PK=(followee_id, follower_id)
  → Both directions indexed for efficient lookup
```

## Feed Storage (Redis Sorted Set)

```
Per-user feed stored as a Redis sorted set:
  Key:   feed:{userId}
  Score: Unix timestamp (or rank score for algorithmic feed)
  Value: postId

Operations:
  Add to feed:      ZADD feed:bob 1700000123 "post:xyz"     O(log N)
  Read feed:        ZREVRANGE feed:bob 0 19               (top 20)  O(log N + K)
  Cursor read:      ZREVRANGEBYSCORE feed:bob MAX cursor LIMIT 0 20
  Trim to 1K:       ZREMRANGEBYRANK feed:bob 0 -1001       (remove oldest)
  Remove on unfollow: ZREM feed:bob "post:xyz"

Why sorted set?
  Score = insertion order / rank → efficient range queries
  ZREVRANGE = most recent first = O(log N + K)
  Trim keeps memory bounded even for active users
```

## Post Data Model

```
Post table (Cassandra — wide rows by author):
  partition key: author_id
  clustering key: created_at DESC, post_id
  columns: content, media_url, like_count, comment_count, post_type

Cassandra advantages for this workload:
  High write throughput (no single-leader bottleneck)
  Time-series naturally modeled with clustering key
  Wide rows: "get recent 20 posts by user" = single partition scan

Post IDs:
  Use Snowflake IDs (64-bit):
    41 bits: timestamp (ms since epoch) → sortable by time
    10 bits: machine ID
    12 bits: sequence
  → Globally unique, time-sortable, no coordination needed

Like / Comment counters:
  Cassandra counter columns for like_count
  Or: separate likes table + eventual counter (avoid hot counter row)
```

## Ranking Algorithm

```
Chronological:
  Score = post.created_at (Unix timestamp)
  Simplest; no machine learning; Twitter "Latest" tab

Algorithmic (engagement-based decay):
  Score = (likes × 2 + comments × 3 + shares × 5) / (age_hours + 1)^gravity
  gravity = 1.8 (HackerNews-style decay)
  + affinity bonus: posts from close friends rank higher
  + freshness bonus: very recent posts get a floor boost

Production ranking pipeline:
  1. Candidate generation: fetch last 200 posts from follow graph
  2. Feature extraction: engagement rate, author affinity, post type
  3. Ranking model: lightweight gradient boosting (not deep learning for feeds)
  4. Filtering: remove seen posts, blocked accounts, spam
  5. Return top 20
```

## Pagination — Cursor-Based

```
Problem with offset pagination:
  GET /feed?offset=20&limit=20
  New posts arrive → items shift → page 2 shows duplicates or skips items

Cursor-based pagination:
  GET /feed?limit=20                      → first page
  Response includes: next_cursor = "1700000100"  (timestamp of last item)
  GET /feed?before=1700000100&limit=20   → second page
  → Stable: new posts don't affect cursor position

Implementation:
  cursor = score (timestamp ticks) of last returned post
  Query: ZREVRANGEBYSCORE feed:userId (cursor -inf) LIMIT 0 20
  Return next cursor = score of last item in this page
```

## Handling Celebrities (The N+1 Problem)

```
Threshold: users with > 10M followers are "celebrities"

Write path for celebrity post:
  1. Store post in Post DB
  2. Add to celebrity's own post list (NOT fan-out to followers)
  3. Mark author as celebrity in cache

Read path (when user opens feed):
  1. Load pre-computed feed from Redis (posts from non-celebrity follows)
  2. For each celebrity the user follows:
       Pull latest 10 posts from celebrity's post list
  3. Merge all posts → rank → return top 20

Cost analysis:
  If user follows 2 celebrities + 498 regular users:
    Pre-computed feed: O(1) Redis read
    Celebrity pull: 2 × 10 = 20 DB reads (bounded by number of celeb follows)
  → p99 feed load: pre-computed fetch + ~20 small reads = fast
```

---

## Unfollow Handling

```
Simple approach:
  Don't retroactively remove posts from feed cache
  Posts from unfollowed user remain until they age out of the 1000-post window
  → Eventual consistency; stale posts disappear within hours

Strict approach (Twitter):
  On unfollow: asynchronously scan feed cache, remove posts by that author
  Expensive: O(feed_size) scan per unfollow; done as background job
  → Immediate "clean" feed; better user experience

New follow backfill:
  When Bob follows Alice:
  Fan-out service pulls Alice's last 20 posts and adds to Bob's feed
  → Bob immediately sees Alice's recent content
```

## Pitfalls

```
1. No celebrity handling
   Problem: celebrity posts fan-out to 50M feeds → system overloaded
   Fix: threshold-based hybrid fan-out; pull celebrities at read time

2. Offset-based pagination
   Problem: new posts cause page drift → duplicates or skipped content
   Fix: cursor-based pagination using post timestamp / score

3. Unbounded feed cache
   Problem: active users accumulate millions of feed entries → Redis OOM
   Fix: cap at 1000 entries per feed; trim on each write (ZREMRANGEBYRANK)

4. Following thousands of accounts (fan-in read problem)
   Problem: user follows 5000 accounts; read-time pull from all → high latency
   Fix: hybrid approach; cap follows at a reasonable limit (e.g., 10K)

5. Thundering herd on celebrity post
   Problem: celebrity posts → millions request feed simultaneously
   Fix: local cache on feed servers (TTL=30s); serve stale for seconds

6. Like counts showing stale values
   Problem: multiple regions update count independently → diverge
   Fix: eventual consistency for counts; display "1.2M likes" not exact number

7. All posts in single Cassandra partition
   Problem: very prolific user has billions of posts → hot partition
   Fix: time-bucket partition key: (author_id, year_month)
```

## Real-World Systems

```
Instagram feed:   Fan-out on write; Redis sorted sets; algorithmic ranking
Twitter timeline: Hybrid fan-out; Flock (follow graph store); Earlybird search
Facebook:         EdgeRank / News Feed Ranking; heavy ML ranking pipeline
LinkedIn:         Viral actions (likes/comments) trigger re-distribution
```

## Interview Answer Framework

1. **Fan-out models:** write (push) vs read (pull) vs hybrid; choose hybrid for celebrity problem
2. **Feed store:** Redis sorted set per user; score = timestamp or rank; ZREVRANGE for O(1) feed read
3. **Post store:** Cassandra partitioned by author_id + time-clustered; Snowflake IDs for time-sortability
4. **Celebrity threshold:** skip fan-out for mega-accounts; merge on read (bounded cost)
5. **Ranking:** chronological (score = time) or algorithmic (engagement decay formula)
6. **Pagination:** cursor-based (not offset); stable under concurrent new posts
7. **Unfollow:** lazy eviction (posts age out) vs async cleanup job; trade-off freshness vs cost
8. **Follow backfill:** on new follow, seed feed with followee's recent N posts
9. **Scalability:** feed cache = Redis cluster; post DB = Cassandra; follow graph = sharded Redis
10. **Pitfalls:** celebrity problem, unbounded cache, offset pagination drift, hot partition
