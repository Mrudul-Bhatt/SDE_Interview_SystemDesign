# Social Media Feed (like Twitter or Instagram)

---

## What Are We Building?

When you open Twitter, Instagram, or Facebook, you see a stream of posts from the people you follow.

```
Bob opens the app and sees:
  [Alice]   "Just got back from Hawaii! 🌺"      2 minutes ago
  [Carol]   "New blog post: ..."                  15 minutes ago
  [Dave]    "Look at this sunset 🌅 [photo]"     1 hour ago
  [Alice]   "Anyone want to grab coffee?"         3 hours ago
  ...
```

That stream is called a **feed** (or "timeline"). It's personalized — Bob and Carol see completely
different feeds because they follow different people.

**The challenge:** show the right posts, in the right order, fast (under 200ms), to 200 million
active users — including some celebrities with 170 million followers each.

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Follow / unfollow other users
2. Create posts (text, image, video)
3. Like and comment on posts
4. GET /feed → return personalized feed, page by page
5. Feed contains posts from people you follow
6. Two ordering modes: chronological (newest first) or "ranked" (algorithmic)
7. New posts should appear in followers' feeds within seconds
```

### Non-Functional (scale and reliability)
```
Users:      300 million total, 200 million active per day (Twitter-scale)
Posts:      500 million posts per day = about 5,800 posts/second on average
Feed views: Each active user opens the app ~8 times/day = 18,500 reads/sec
Speed:      Feed must load in under 200ms (99% of the time)
Freshness:  A few seconds of delay before a new post appears is acceptable
Storage:    500M posts/day × 1 KB each = 500 GB/day; keep 90 days = 45 TB
```

---

## The Core Question: How Do We Build a Personalized Feed?

The whole challenge boils down to one question:

> When Bob opens the app, how do we quickly figure out which posts to show him?

There are two basic approaches. Each has a serious problem. The real answer combines them.

---

### Approach 1: Fan-Out on Write (Push Model)

**The idea:** when someone posts, immediately deliver a copy of that post into every follower's feed.

```
Alice posts "Hello world!" and Alice has 500 followers.
     │
     ▼
We write the post ID into 500 follower feeds RIGHT NOW.

Bob's feed cache:        ["hello world!", ...previous posts]
Carol's feed cache:      ["hello world!", ...previous posts]
... and 498 more
```

**When Bob opens the app:**
```
Just read Bob's feed cache → return the top 20 posts. Done. Super fast (1ms).
```

**The problem:** what if Alice has 50 million followers?

```
Alice posts → write to 50,000,000 feed caches → takes a long time
At 1 million writes/sec, that's 50 seconds per post.
Followers wait 50 seconds to see Alice's tweet → unacceptable.

Elon Musk has 170M followers. One tweet would take ~3 minutes to fully fan out.
```

This is called the **celebrity problem**. Push works great for normal users, terrible for celebrities.

---

### Approach 2: Fan-Out on Read (Pull Model)

**The idea:** when someone posts, do nothing. When a user opens the app, scan all the accounts
they follow and pull the latest posts.

```
Alice posts "Hello world!" → just stored in Alice's posts list. No fan-out.

Bob opens the app:
  1. Look up everyone Bob follows (500 accounts)
  2. For each, pull their latest 5 posts
  3. Merge all 2,500 posts, sort by time, return top 20
```

**The problem:** Bob has to wait while we do 500 separate database lookups.
This is called **scatter-gather** and it's slow.

```
500 follow lookups × 5ms each = 2.5 seconds. Way over the 200ms budget.
```

---

### The Real Answer: Hybrid Fan-Out

Use BOTH approaches, depending on who's posting.

```
For REGULAR users (< 10M followers):
  Use push (fan-out on write).
  Their post lands in all follower feeds quickly.
  Total work: at most 10M writes — manageable.

For CELEBRITIES (≥ 10M followers):
  Skip the push. Just store their post in their own posts list.
  When a follower opens the app, pull the celebrity's latest posts at read time.

Reading Bob's feed:
  1. Read Bob's pre-computed feed cache (fast — the 498 regular accounts he follows)
  2. For each celebrity Bob follows (typically 1–5), pull their latest 5 posts
  3. Merge → rank → return top 20
```

**Why this works:**
- 99.9% of users aren't celebrities, so most posts are pushed normally (super fast reads)
- The few celebrity accounts are handled at read time, but you only follow a handful of them,
  so it's just a few extra pulls — not 500.

## How Much Hardware?

Before building anything, engineers estimate **how big the system needs to be**. This is called
"capacity planning" or "back-of-the-envelope estimation." The goal is to answer two questions:

1. **How much work per second** must the system handle? (so we know how many servers to buy)
2. **How much memory/storage** do we need? (so we know how big the database/cache must be)

Let's work through both, slowly.

---

### Question 1: How much *writing* work does posting create?

You might think "5,800 posts per second isn't that much." But here's the catch with the push model:
**one post turns into many writes**, because we copy that post into every follower's feed.

```
Step 1 — How many posts per second?
  500 million posts/day ÷ 86,400 seconds/day ≈ 5,800 posts/sec

Step 2 — How many followers does an average person have?
  About 200 (most people are not celebrities)

Step 3 — So how many feed-cache writes per second?
  Each post is copied to ~200 follower feeds, so:
  5,800 posts/sec × 200 followers = 1,160,000 writes/sec
```

**The key insight:** posting 5,800 times per second creates **1.16 million writes per second** —
roughly **200× more work** than the number of posts. This multiplication is called
**fan-out amplification**, and it's the single most important number in this whole design.
It's *why* we can't push celebrity posts (170M followers × even a few posts = billions of writes).

> Takeaway: the system isn't sized by "how many people post" — it's sized by
> "how many followers those posts reach." Reads and writes explode through the follow graph.

---

### Question 2: How much memory does the feed cache need?

The feed cache (in Redis) stores each user's ready-to-show feed. Let's figure out how big it gets.

```
Step 1 — How much memory does ONE user's feed take?
  We keep the newest 1,000 post IDs per user.
  Each post ID is a number that takes 8 bytes of memory.
  1,000 IDs × 8 bytes = 8,000 bytes ≈ 8 KB per user

Step 2 — How much for ALL users?
  200 million active users × 8 KB each = 1,600,000,000 KB ≈ 1.6 TB
```

**What this tells us:** we need about **1.6 terabytes of fast memory** just for the feeds.
That's too much for a single machine, so we spread it across many Redis servers working
together (a "Redis cluster"). 1.6 TB sounds huge, but notice we're only storing tiny post *IDs*,
not the full posts — the actual text, images, and like counts live in Cassandra (a cheaper,
disk-based database). Storing only IDs is what keeps this number from being 100× bigger.

---

### The two numbers to remember

```
~1.16 million feed writes/sec   → drives how many servers we need (the work)
~1.6 TB of feed cache           → drives how much Redis memory we need (the storage)
```

Everything else in the design (the celebrity threshold, the 1,000-post cap, storing IDs not
content) exists to keep these two numbers manageable.

## Storing the Follow Graph

The "follow graph" is just the data that records **who follows whom**. Think of it like two
address books that must always agree with each other.

A "follow" relationship points in one direction (Bob follows Alice, but Alice might not follow
Bob back). The surprising part: to make the app fast, we store that *same* relationship **twice** —
once from each person's point of view.

```
For each user we keep TWO lists:

  followers:{alice}  = everyone who follows Alice   (Alice's "fans")
  following:{bob}    = everyone Bob follows          (Bob's "subscriptions")

So when Bob follows Alice, we update BOTH lists at once:
  → add "bob"   to followers:alice   (Alice gained a follower)
  → add "alice" to following:bob     (Bob added a subscription)
```

**Why store the same fact twice?** Because the two halves of the app ask *opposite* questions, and
each list answers one of them instantly:

```
When Alice POSTS → "Who do I deliver this to?"  → read followers:alice   (her fans)
When Bob OPENS the app → "Whose posts do I want?" → read following:bob   (his subscriptions)
```

If we stored only one direction, answering the other question would mean scanning *every user in
the system* to find the matches — far too slow. Keeping both lists is a classic trade: we spend a
little extra storage (and one extra write when someone follows) to make both reads instant.

**The celebrity wrinkle:** for normal people these lists are small — a few hundred to a few
thousand names. But a celebrity's `followers` list can have 50+ million names, which is too big to
keep on a single server. So for those, we **split the list across many servers** (called
"sharding") — the same way you'd split one impossibly-thick address book into many volumes.

## Storing the Feed Itself (Redis Sorted Set)

We said each user's ready-to-show feed lives in **Redis**. Redis is a database that keeps
everything in memory (RAM), which makes it extremely fast — perfect for something we read on every
app open. Redis offers a few data types; the one we use here is the **sorted set**.

**What's a sorted set?** It's a list of items where each item carries a number called a **score**,
and Redis *automatically keeps the list ordered by that score* at all times. You never sort it
yourself — add an item and it slots into the right place instantly. For a feed, we use each post's
**timestamp as the score**, so the newest posts are always on top, ready to read.

```
Bob's feed (a sorted set):
  Score (the post's time)   Item (the post ID)
  1700000300                post:abc      ← highest score = newest = top of feed
  1700000245                post:xyz
  1700000100                post:def
  1700000050                post:ghi      ← lowest score = oldest
```

(Those big numbers are "Unix timestamps" — seconds since 1970. A bigger number just means a more
recent time.)

**The three things we do with it** (the `Z...` names are just Redis's commands for sorted sets):

```
Add a new post to Bob's feed:
  ZADD feed:bob 1700000400 "post:new"
  → inserts the post, automatically placed in time order

Read the newest 20 posts (what Bob sees when he opens the app):
  ZREVRANGE feed:bob 0 19
  → "REV" = reverse = highest score first = newest first; "0 19" = the first 20

Trim the feed so it never exceeds 1,000 posts:
  ZREMRANGEBYRANK feed:bob 0 -1001
  → deletes everything except the newest 1,000, run after each add
```

**How fast are these?** Very. Adding or reading uses an operation that scales as **O(log N)** —
a piece of Big-O notation that, in plain terms, means "stays fast even when the set is huge." If
the feed grows from 1,000 to 1,000,000 items, the work only grows a tiny bit, not a million times.
That's exactly the property you want for something hit on every app open.

**Why cap each feed at 1,000 posts?** Two reasons. First, nobody scrolls past ~100 posts in a
sitting, so 1,000 is far more than enough for any real scroll. Second — and more importantly — an
active user could accumulate *millions* of posts in their feed over the years, and storing that for
200 million users would blow through Redis's memory. Capping at 1,000 (trimming the oldest on every
add) keeps each user's feed small and bounds the total memory (this is where that 1.6 TB figure
from the hardware section comes from). If a user somehow scrolls past 1,000, we fall back to
fetching older posts from the slower database.

## Storing the Posts Themselves

Remember: the feed cache only holds tiny post **IDs**, not the actual posts. The real content — the
text, the image link, the like count — lives in a separate, bigger database. We use **Cassandra**
for this.

**What's Cassandra?** It's a database designed to spread data across many servers and handle a
huge volume of writes without slowing down. Unlike a traditional database that funnels everything
through one main server, Cassandra has no single bottleneck — which is exactly what we need when
5,800 posts arrive every second.

When you design a Cassandra table, the two most important decisions are the **partition key** and
the **clustering key**. They sound technical, but the idea is simple:

```
Cassandra table: posts

  Partition key:   author_id
     → decides WHICH server a post lives on. All of one author's posts are
       grouped together on the same server. (Think: one folder per author.)

  Clustering key:  created_at DESC
     → decides the ORDER of posts inside that folder. "DESC" = newest first.

  Columns (the actual data):  content, media_url, like_count, comment_count
```

Putting those two together gives us exactly the lookup we need most:

```
"Give me Alice's 20 most recent posts"
  → go straight to Alice's folder (partition key = alice)
  → it's already sorted newest-first (clustering key)
  → just grab the top 20. One fast read, no scanning, no sorting.
```

This is why Cassandra fits so well — and it's precisely the lookup the **celebrity pull** needs
(remember: we don't pre-push celebrity posts; we fetch their recent ones at read time). Grouping an
author's posts together and pre-sorting them by time makes that fetch a single quick operation.
(The term for "all of one author's posts stored together in one sorted group" is a **wide row** —
just a name for this layout.)

**Where do post IDs come from? The Snowflake trick.** Every post needs a unique ID. The naive way —
a single counter that hands out 1, 2, 3, ... — fails at scale, because every server would have to
ask that one counter for the next number (a bottleneck, and a single point of failure). Instead we
use a **Snowflake ID**: a 64-bit number built by gluing together three pieces —

```
[ timestamp ]  +  [ which machine generated it ]  +  [ a counter on that machine ]
```

Because it starts with a timestamp, Snowflake IDs are **automatically sortable by time** (a bigger
ID = a newer post). And because each machine includes its own ID and counter, **any server can
mint unique IDs on its own** — no central counter to bottleneck. So we get globally-unique,
time-ordered IDs for free, generated anywhere.

## Ranking: Why "Best Posts First" Is Hard

The simplest order is **chronological**: newest first. Twitter's "Latest" tab uses this.
Easy: just sort by timestamp.

But most apps rank "algorithmically" — show the most engaging posts first, even if they're slightly older.

### A Simple Ranking Formula

```
score = (likes × 2 + comments × 3 + shares × 5) / (age_hours + 1)^1.8
```

Let's break this down:
- **Engagement**: shares count more than comments, comments more than likes (people work harder to share)
- **Age decay**: divide by age to make older posts rank lower. The `^1.8` makes recency matter more
- **+1**: prevents dividing by zero when a post is brand new

```
A 1-hour-old post with 100 likes:
  score = (100 × 2) / (1 + 1)^1.8 = 200 / 3.48 = 57.4

A 24-hour-old post with 1000 likes:
  score = (1000 × 2) / (24 + 1)^1.8 = 2000 / 380 = 5.3

→ The newer, less popular post wins. Recency dominates.
```

This is the same formula used by Hacker News.

### Real Production Ranking

Big platforms use machine learning, but the pipeline is similar:
```
1. Candidate generation: grab the last 200 posts from accounts you follow
2. Feature extraction: how often do you interact with this author? Are you usually online now?
3. Rank with a model (boosted trees, not deep learning — needs to be very fast)
4. Filter out: posts you've already seen, blocked accounts, flagged spam
5. Return top 20
```

## Pagination: How "Load More" Should Work

When you scroll down, the app loads the next batch of posts. There are two ways to do this.

### Offset Pagination (Bad)

```
GET /feed?offset=20&limit=20   → get posts 21–40
```

**Problem:** new posts are constantly arriving at the top of the feed. By the time you scroll,
the offsets have shifted.

```
You opened the app, saw posts at offsets 0–19.
Alice posts a new one → it's now at offset 0, shifting everything down.
You scroll, asking for offsets 20–39.
But what was at offset 20 is now at offset 21 → you see it again (duplicate).
```

### Cursor Pagination (Good)

Instead of "give me posts 21–40," say "give me posts older than the last one I saw."

```
GET /feed?limit=20                          → returns first 20 posts
Response includes: next_cursor = 1700000100  (timestamp of the oldest post you got)
GET /feed?before=1700000100&limit=20         → returns posts older than that
```

Even if 50 new posts arrive at the top, the cursor still points to the same spot in time.
No duplicates, no skips.

## What Happens When You Unfollow?

When Bob unfollows Alice, do we immediately remove Alice's posts from Bob's feed cache?

**Lazy approach (simpler):**
```
Do nothing. Alice's posts stay in Bob's feed until they age out (max 1,000 posts).
Within hours, they naturally disappear as new posts push them out.
```

**Strict approach (Twitter-style):**
```
Background job scans Bob's feed cache and removes posts authored by Alice.
Better UX (no awkward old posts) but more expensive.
```

Most apps use the lazy approach — users rarely notice.

### What About a New Follow?

When Bob follows Alice, we want her recent posts to appear in his feed immediately.

```
On follow:
  Fetch Alice's last 20 posts.
  Insert them into Bob's feed cache.
  → Bob refreshes and sees Alice's content right away.
```

## How It All Fits Together

```
─── Posting a tweet ─────────────────────────────────────────────────

Alice taps "Post" → "Hello!"
       │
       ▼
   Post Service
       ├─ Generate Snowflake post ID
       ├─ Store post in Cassandra
       └─ Send to Fan-Out Service
              │
              ▼
       Is Alice a celebrity (≥ 10M followers)?
              │
       ┌──────┴──────┐
       │             │
   NO (regular)   YES (celebrity)
       │             │
       ▼             ▼
   Push post ID    Do nothing extra
   to all 500      (post is stored in Alice's
   follower feeds  posts list — pull at read time)

─── Reading a feed ──────────────────────────────────────────────────

Bob opens the app:
       │
       ▼
   Feed Service
       ├─ Read pre-computed feed from Redis (top 100 IDs)
       ├─ For each celebrity Bob follows (say 2):
       │     Pull their last 5 posts directly from Cassandra
       ├─ Merge all post IDs
       ├─ Fetch post details (content, like count) from Cassandra
       ├─ Rank by score formula
       └─ Return top 20 to Bob with a cursor for the next page
```

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ No celebrity handling
   → Pushing one Elon Musk tweet to 170M feeds takes minutes; system melts
   ✅ Hybrid fan-out with a threshold (e.g., > 10M followers → pull at read time)

❌ Offset-based pagination
   → New posts arrive → users see duplicates or skip posts when scrolling
   ✅ Cursor-based pagination using timestamps

❌ Unbounded feed cache
   → Active user has been on the app for years; cache has millions of entries; Redis runs out
   ✅ Cap each feed at 1,000 entries; trim on every write

❌ Pulling exact like counts on every feed view
   → Reading like_count from a hot row in Cassandra → write/read contention
   ✅ Eventually consistent counters; show "1.2M likes" not "1,234,567 likes"

❌ One Cassandra partition per user, forever
   → Prolific user has billions of posts in one partition → hot partition, slow scans
   ✅ Time-bucket the partition key: (author_id, year_month) instead of just author_id

❌ Following thousands of accounts with no limit
   → User follows 50,000 accounts → read-time pull is infeasible
   ✅ Cap follows at something reasonable (Twitter limits this); rely on hybrid fan-out

❌ Recomputing the ranking score on every read
   → Expensive ML model run for every feed open → slow and costly
   ✅ Pre-compute score at write/post time; store it as the sorted set score in Redis
```

## Real Systems at a Glance

```
Instagram:   Fan-out on write; Redis sorted sets; algorithmic ranking
Twitter:     Hybrid fan-out; "Flock" follow graph store; Earlybird for search
Facebook:    News Feed Ranking — heavy ML pipeline (EdgeRank successors)
LinkedIn:    Viral re-distribution (when a post is liked, push to liker's network)
```

## How to Answer This in an Interview (Step by Step)

```
1. State the scale:
   "200M daily active users, 500M posts a day, 18K feed reads a second.
   Feed must load in under 200ms. Eventual consistency is acceptable."

2. Frame the core problem:
   "The key question is: when a user opens the app, how do we get their feed quickly?
   There are two approaches — push on write or pull on read — and the right answer
   is a hybrid that handles celebrities differently."

3. Explain hybrid fan-out:
   "For regular users (< 10M followers), push the post ID to every follower's feed cache
   on write. For celebrities, skip the fan-out and pull their latest posts at read time.
   This gives bounded write work for celebrities and instant reads for everyone else."

4. Describe the feed cache:
   "Redis sorted set per user, scored by timestamp or rank score. Top 1,000 entries
   per user, trimmed on every write. ZREVRANGE for fast top-K reads."

5. Describe post storage:
   "Cassandra partitioned by author_id, clustered by created_at descending. Wide rows
   give us 'last N posts by this author' in a single partition scan. Snowflake IDs
   are globally unique and time-sortable."

6. Explain ranking:
   "Either chronological (sort by timestamp) or algorithmic with an engagement/decay
   formula like (likes×2 + comments×3 + shares×5) / (age_hours+1)^1.8. Real systems
   use ML, but the structure is the same — score each candidate, take the top 20."

7. Pagination:
   "Cursor-based, not offset. The cursor is the timestamp/score of the last post returned.
   New posts arriving at the top don't shift the cursor, so no duplicates or skips."

8. Handle edge cases:
   "On unfollow, lazy-evict — let posts age out naturally. On new follow, backfill the
   followee's last 20 posts. For prolific authors, time-bucket the Cassandra partition
   key to avoid hot rows."

9. Cite the main pitfalls:
   "Celebrity problem (use hybrid fan-out), offset pagination drift (use cursors),
   unbounded cache (trim to 1000), hot Cassandra partitions (time-bucket the key)."
```
