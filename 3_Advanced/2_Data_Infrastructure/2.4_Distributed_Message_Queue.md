# Distributed Message Queue (like Kafka)

---

## What Are We Building?

Imagine you run an online store. When a customer places an order, lots of things need to happen:
- Charge their credit card
- Email them a receipt
- Update the inventory
- Notify the warehouse to ship
- Update analytics dashboards

If the order service tried to do all of this itself, it would be slow and fragile. If the email server
goes down, do we fail the whole order? No.

A **message queue** is like a post office between services. The order service drops a letter saying
"new order #42 placed" into the post office and walks away. Five different services come pick up
copies of that letter and each do their own job at their own pace.

```
Order placed
     │
     ▼
┌──────────────────────────┐
│   Message Queue (Kafka)  │   ← the "post office"
│   topic: "orders"        │
└──────────────────────────┘
     │       │       │       │
     ▼       ▼       ▼       ▼
  Payment  Email  Shipping  Analytics
  service  svc    service   service
```

The order service finishes in 5ms. The other services do their work in the background.
If the email server is down for 10 minutes, no problem — the message waits for it.

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Producers (senders) can post messages to named "topics" (like channels)
2. Consumers (receivers) can subscribe to topics and read messages in order
3. Multiple consumers can share the work (a "consumer group")
4. Messages stick around for N days (so a service that goes offline doesn't lose data)
5. Consumers track their own position (so they can replay from any point in history)
6. Split each topic across multiple servers (so we can scale)
7. Guarantee: a message is delivered AT LEAST once (configurable to "exactly once")
```

### Non-Functional (scale and reliability)
```
Volume:    1 million messages PER SECOND written, 5 million per second read
Speed:     99% of writes finish in under 5ms, reads in under 10ms
Safety:    Messages survive even if a server crashes (3 copies of everything)
Retention: Keep messages for 7 days by default
Order:     Guaranteed in order within one partition; not across partitions (more on this later)
Storage:   Petabytes of stored events (1 PB = 1,000 TB)
```

## How Much Hardware Do We Need?

Before building anything, engineers estimate **how big the system needs to be**. This is called
"capacity planning" or "back-of-the-envelope estimation." Every system has one thing that hurts
the most; for a message queue it's the **relentless firehose of writes** — a million messages
every second, never stopping. So our questions are:

1. **How much data flows through**, and how much piles up? (the firehose → total storage)
2. **How many servers (brokers)** to keep up with the write speed? (the real bottleneck)
3. **How many partitions** so the work runs in parallel?

Let's work through them, slowly.

---

### Question 1: How much data flows through?

A message queue is a river, not a lake — data pours in constantly, and (because we only keep it
for 7 days) old data flows out the other end. Let's measure the river.

```
Step 1 — Messages per day:
  1 million messages/sec × 86,400 seconds/day ≈ 86.4 billion messages/day

Step 2 — Turn that into data per day:
  86.4 billion × ~1 KB each ≈ 86 TB of new data every day

Step 3 — We keep 3 copies of everything (replication, for safety):
  86 TB × 3 = 258 TB actually written to disk per day

Step 4 — We keep messages for 7 days, then delete them:
  258 TB × 7 days ≈ 1.8 PB (petabytes) stored at any moment
```

**The key insight:** notice the **7-day retention cap** is what keeps storage from growing
forever. Without it, 258 TB/day would pile up endlessly. Because old messages are deleted, the
total parks at ~1.8 PB instead of climbing without limit. (1 PB = 1,000 TB.)

---

### Question 2: How many servers (brokers)? — the real bottleneck

Here's what makes a message queue different from a database: **the limit isn't how much data you
store, it's how fast you can write it to disk.** A million messages a second is a torrent, and a
single disk can only absorb so much per second.

```
Step 1 — How fast is data arriving?
  1 million messages/sec × 1 KB = 1 GB/sec pouring in

Step 2 — Each message is written 3 times (the 3 copies):
  1 GB/sec × 3 = 3 GB/sec of total write load

Step 3 — How fast can one server write to disk?
  About 500 MB/sec (when writing sequentially — see below)

Step 4 — So how many servers (called "brokers") just to keep up?
  3 GB/sec ÷ 500 MB/sec ≈ 6 brokers minimum, just for the writes
```

**The key insight: this system is *throughput-bound*, not *size-bound*.** We size the cluster by
**write speed** (how many GB/sec it must absorb), not by total data. That's the opposite of the
Distributed KV Store, where sheer *size* forced the design. The one trick that makes 500 MB/sec
per server possible is that the queue only ever **appends** to the end of a file (never edits the
middle) — sequential disk writes are ~100× faster than random ones, which is the whole reason a
message queue can keep up with a firehose at all.

> Takeaway: count servers by *write bandwidth* here, not by storage. Throughput is the enemy.

---

### Question 3: How many partitions? (running in parallel)

Six brokers can keep up *in total*, but only if the work is actually spread across them. You can't
pour a million messages/sec into a single file on a single server — one server can't take it. The
fix is to **split each topic into partitions**, each living on a different broker, all writing at
once.

```
How fast can ONE partition go?
  About 100,000 messages/sec (one partition = one append-only file on one broker)

So how many partitions to reach 1 million/sec?
  1,000,000 ÷ 100,000 = 10 partitions per topic
```

**What this tells us:** a partition is the **unit of parallelism**. One partition tops out around
100K msg/sec, so to hit our 1M/sec target we split the topic into at least 10 partitions running
side by side. More partitions = more parallelism = more throughput (and more consumers can read at
once too). This is *why* the next section is all about topics and partitions — they're not just
organization, they're how the queue scales past one server's limits.

---

### The numbers to remember

```
~1.8 PB stored        → huge, but CAPPED by 7-day retention (a river, not a lake)
~6 brokers minimum    → sized by WRITE SPEED (3 GB/sec), not by storage size
~10 partitions/topic  → the parallelism needed to reach 1M messages/sec
```

The big idea connecting them: a message queue is defined by **throughput, not size**. The firehose
of writes is the enemy, sequential appends are what let one server keep up, and partitions are how
we run many servers in parallel to absorb the full flood.

## Core Concept #1: Topics, Partitions, and Offsets

### A Topic Is Like a Filing Cabinet

A **topic** is just a named channel. You might have topics like `"orders"`, `"user_clicks"`,
or `"payments"`. Producers write to topics; consumers read from topics.

### A Partition Is One Drawer in That Cabinet

A topic isn't stored in one place — it's split into **partitions**. Think of each partition as a
separate drawer in the filing cabinet, on a different server.

```
Topic "orders" has 3 partitions:

Partition 0 (lives on Server A):  [msg, msg, msg, msg, ...]
Partition 1 (lives on Server B):  [msg, msg, msg, msg, ...]
Partition 2 (lives on Server C):  [msg, msg, msg, msg, ...]
```

Why split? Because one server can't handle 1 million messages per second. With 10 partitions
on 10 servers, each only handles 100K — much more manageable.

### An Offset Is the Position in One Drawer

Each partition is an **append-only log** — new messages are added at the end. Each message
gets a sequential number called an **offset**.

```
Partition 0:
   offset:   0    1    2    3    4    5    6 ...
   msg:    "A"  "B"  "C"  "D"  "E"  "F"  "G"

Reading partition 0 from offset 3:
  → returns "D", "E", "F", "G", ...
```

**Key insight:** offsets only make sense within ONE partition.
"Offset 5 in partition 0" and "offset 5 in partition 1" are completely different messages.

### Why Doesn't Order Work Across Partitions?

If you have 3 partitions and you want to read "the next 10 messages in order," there's no
global order. Partition 0's offset 5 might have happened AFTER partition 1's offset 7.

**Ordering is guaranteed within ONE partition only.**

If you need messages for the same user/order/entity to stay in order, send them all to the
SAME partition. How? By using a **key**.

```
Producer sends: { key: "user42", value: "clicked button" }

Routing: hash("user42") % 3 = 1  → goes to partition 1
All future messages with key "user42" → also partition 1 → ordered for user42
```

Messages without a key get distributed round-robin across partitions (no ordering guarantee,
maximum spread).

## Core Concept #2: Consumer Groups (Sharing the Work)

Consumers don't read individually — they form **groups**. Each group reads the topic independently,
and within a group, the partitions are split among consumers.

```
Topic "orders" has 3 partitions. Two groups want to read it.

Group A — "analytics" (2 consumers):
  Consumer A1 reads partitions 0 and 1
  Consumer A2 reads partition 2

Group B — "billing" (3 consumers):
  Consumer B1 reads partition 0
  Consumer B2 reads partition 1
  Consumer B3 reads partition 2
```

**Two important rules:**

```
✅ One partition is read by AT MOST ONE consumer in a group at a time.
   (Two consumers in the same group can't both read partition 0 — they'd duplicate work.)

✅ One consumer can read MULTIPLE partitions.
   (If you have 3 partitions and only 1 consumer, that 1 consumer handles all 3.)
```

**Adding more consumers = more parallelism, up to a limit.** If your topic has 3 partitions,
the 4th consumer in a group just sits idle (nothing to assign). So pick your partition count
based on how much parallelism you'll ever want.

**Two groups read the same topic independently.** The analytics service reading messages
doesn't affect the billing service. Each group has its OWN bookmark (committed offset).

---

## Core Concept #3: Consumer Lag (Are We Keeping Up?)

```
consumer_lag = latest_message_offset - consumer's_current_offset
```

If new messages are flowing in faster than a consumer can process them, lag grows.

```
Partition 0 has 1,000,000 messages.
Consumer is at offset 850,000.
Lag = 1,000,000 - 850,000 = 150,000 messages behind.

This is the main health metric for any consumer.
If lag > 100K and growing → time to add more consumers or optimize processing.
```

## How a Producer Sends a Message

```
Step 1: Producer wraps the message — { key, value, headers }

Step 2: Decide which partition to send to:
   If key is given:    partition = hash(key) % numPartitions
   If key is null:     round-robin across partitions

Step 3: Send to the LEADER of that partition.
   Each partition has one "leader" server that handles all writes.
   (The other replicas are "followers" that copy from the leader.)

Step 4: Wait for an acknowledgment:
   acks=0  → "I don't care, just send it" (fastest, might lose data)
   acks=1  → "Wait for the leader to confirm" (default; data lost if leader dies before copying)
   acks=-1 → "Wait for ALL copies to confirm" (slowest, safest)
```

### Batching: The Speed Booster

Producers don't send each message immediately. They buffer messages and send them in batches.

```
Without batching:
  Send 1 message → wait for round-trip → send next → wait → ...
  At 5ms per round-trip: 200 messages/sec max per producer

With batching (linger.ms=5, batch.size=16KB):
  Buffer messages for up to 5ms, then send 100+ in one batch
  → 10× higher throughput, costs 5ms of extra latency
```

Almost always worth the trade.

## Replication: Keeping 3 Copies

Each partition has multiple replicas across servers, just like in a key-value store.

```
Partition 0:
  Server A — LEADER     (handles all reads and writes)
  Server B — Follower   (copies from leader; standby)
  Server C — Follower   (copies from leader; standby)
```

### ISR: "Who's Actually Caught Up?"

The replicas don't always keep up perfectly. If Server B is slow (network issue, garbage collection),
it might be 5 seconds behind. We don't want to wait for slow followers on every write.

**In-Sync Replicas (ISR):** the set of followers that ARE currently caught up.

```
Leader at offset 1000.
Server B at offset 999 → in ISR (only 1 behind, that's fine)
Server C at offset 800 → NOT in ISR (too far behind, treat as broken)

When producer sends with acks=-1:
   → Wait for the LEADER + all ISR members to confirm
   → Don't wait for Server C (it's out of sync, would block forever)
```

If Server C catches up later, it rejoins the ISR. If the leader dies, a new leader is picked from
the ISR (NEVER from a non-ISR replica — that would lose data).

## Delivery Guarantees: How Many Times Will a Message Be Processed?

This is the biggest gotcha in message queues. There are three levels:

### At-Most-Once (Might Lose Messages, Never Duplicate)

```
Producer: acks=0 (fire and forget)
Consumer: bookmark the offset BEFORE processing the message
```

What can go wrong:
- Producer fails to deliver → message lost
- Consumer bookmarks offset, then crashes before processing → message lost

Use when loss is acceptable: metrics, debug logs, click tracking where one missed event doesn't matter.

### At-Least-Once (The Default; Might Duplicate)

```
Producer: acks=-1 with retries
Consumer: process the message FIRST, THEN bookmark the offset
```

What can go wrong:
- Consumer processes the message, then crashes BEFORE bookmarking the offset
- When the consumer restarts, it sees no bookmark → reads the same message again → processes it twice

Use when you can handle duplicates. The consumer code should be **idempotent** —
"add $10 to the bill" is bad (could happen twice), "set the bill to $50" is good (safe to repeat).

### Exactly-Once (The Holy Grail; Slower)

Kafka can achieve this with **transactions**:
- Producer sends with a unique ID + sequence number → broker rejects duplicates
- Consumer wraps "process message + commit offset" in one atomic transaction

Use for money: payments, inventory deduction. Slower and more complex, but you sleep better.

## How Messages Are Stored on Disk

This is what makes Kafka so fast — the storage trick.

### Append-Only Log Segments

Each partition is stored as a series of files on disk:

```
00000000000000000000.log  ← messages 0–999
00000000000000001000.log  ← messages 1000–1999
00000000000000002000.log  ← messages 2000–...  (currently being written)
```

Each file is paired with an `.index` file (a sparse map of "offset → file position") so we can
seek to any offset without reading the whole file.

### Sequential I/O Is the Key

The reason Kafka can write 1 million messages per second is **all writes are sequential**.

```
Imagine a vinyl record. The needle is at position X.
  Random write: lift needle, move to position 12,000, drop, write, lift, move to 47,000...
  Sequential write: needle stays in place, vinyl rolls under it.

Sequential disk writes are ~100× faster than random writes.
```

Plus, modern OSes cache recent file writes in RAM (the **page cache**). Most reads — from
consumers caught up with the live stream — never touch disk at all. They read straight from RAM.

### Zero-Copy: Reading Without Copying

Normally, sending a file to a consumer involves: disk → OS buffer → app buffer → network buffer.
That's 3 copies of the same bytes.

Kafka uses the **sendfile()** system call: disk → network card, with the OS managing the transfer
directly. No copies into user space. This is part of why Kafka can saturate a 10 Gbps network link.

## Log Compaction: Keeping Only the Latest

By default, old messages are deleted after 7 days. But sometimes you want a topic that acts like
a permanent table — keep only the LATEST value for each key.

```
Topic "user_profiles" with log compaction:

Before compaction:
  key=user1, value="email: alice@old.com"   (offset 5)
  key=user2, value="email: bob@bob.com"     (offset 6)
  key=user1, value="email: alice@new.com"   (offset 100)
  key=user1, value="email: alice@newer.com" (offset 500)

After compaction:
  key=user2, value="email: bob@bob.com"     (offset 6)
  key=user1, value="email: alice@newer.com" (offset 500)

Only the LATEST value for each key is kept. Older values are deleted.
```

A consumer that reads from offset 0 will see every key exactly once with its current value.
This is perfect for use cases like:
- Configuration changes (only the latest config matters)
- User profile updates (only the current email matters)
- Database change capture (CDC — bring a downstream system up to date)

To delete a key entirely, write a message with that key and a `null` value. This is a **tombstone**.
After compaction, the tombstone (and the key) are eventually removed.

## Consumer Rebalancing: When the Team Changes

What happens when a new consumer joins a group, or an existing one crashes?

```
Before: Group A has 2 consumers, topic has 3 partitions
  Consumer A1 ← partitions 0, 1
  Consumer A2 ← partition 2

New consumer A3 joins the group:
  → REBALANCE happens
  → Coordinator reassigns partitions
After:
  Consumer A1 ← partition 0
  Consumer A2 ← partition 1
  Consumer A3 ← partition 2
```

### The Old Way: Stop-the-World Rebalance

The original protocol: every consumer pauses, the coordinator reassigns everything, then consumers
resume. During this time (often 5–30 seconds), the consumer group stops processing entirely.

This is painful during a rolling deploy — every restart of a single consumer pauses the whole group.

### The Modern Way: Cooperative Rebalance

Only partitions that are MOVING get paused. If A1's assignment doesn't change, A1 keeps consuming.
Newer Kafka (≥ 2.4) uses this by default.

### Static Membership: Don't Rebalance for Brief Restarts

If a consumer has a stable ID and disconnects briefly (say, a rolling deploy), don't trigger a rebalance.
Wait a few seconds; if the consumer comes back with the same ID, give it back its old partitions.

## How It All Fits Together

```
─── Producer side ──────────────────────────────────────────────────

Order service: send({key: "order:42", value: "..."}) to topic "orders"
       │
       ├─ hash("order:42") % 10 = partition 3
       ├─ Look up: partition 3 leader is Broker 7
       └─ Send batch (5ms wait, 16KB capacity) to Broker 7

Broker 7:
       ├─ Append to active log segment (sequential disk write)
       ├─ Replicate to ISR followers (Broker 2, Broker 9)
       ├─ Wait for ISR acks (if acks=-1)
       └─ Confirm to producer

─── Consumer side ──────────────────────────────────────────────────

Billing service (consumer group "billing"):
       │
       ├─ Coordinator assigns partitions 0, 3, 7 to this consumer
       ├─ Poll partition 3 from last committed offset (say 14,500)
       ├─ Receive batch of 100 messages
       ├─ Process each (charge card, update DB)
       └─ Commit new offset (14,600) to Kafka

If consumer crashes mid-batch:
       Next consumer reads from 14,500 again → some messages reprocessed
       → consumer logic should be idempotent
```

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ Too few partitions
   → 10 consumers in a group but only 3 partitions → 7 consumers sit idle
   ✅ Plan partition count based on your MAX parallelism need (you can't shrink later)

❌ All messages going to one partition (null keys with bad distribution, or one super-active key)
   → One partition gets all the traffic, one consumer overwhelmed
   ✅ Choose keys with good cardinality (user_id, order_id)

❌ Committing offset BEFORE processing
   → Consumer crashes mid-process → message silently lost forever
   ✅ Always: process FIRST, commit offset AFTER

❌ Consumer takes too long between polls
   → Broker thinks consumer is dead → triggers unnecessary rebalance → outage
   ✅ Tune max.poll.interval.ms; reduce batch size if processing is slow

❌ Forgetting that messages get deleted after retention
   → Slow consumer falls 8 days behind → reads "offset out of range" → catastrophe
   ✅ Set retention based on max acceptable consumer lag; alert when lag > 50% of retention

❌ Assuming order across partitions
   → "Process events for user42 in order" but events are split across partitions → reordered
   ✅ Use the user_id as the key so all of user42's events land in the same partition

❌ Choosing exactly-once when at-least-once + idempotent processing would work
   → Exactly-once adds latency, complexity, transactional commit overhead
   ✅ Make the consumer idempotent (e.g., dedupe by message ID); use at-least-once

❌ Producer with acks=0 in production
   → Highest throughput but messages silently dropped on broker hiccups
   ✅ Use acks=-1 unless you genuinely don't care about losing data (metrics only)
```

## Real Systems at a Glance

```
Apache Kafka:    The standard. Self-hosted, super fast, hundreds of consumer groups
Amazon Kinesis:  AWS-managed equivalent; simpler but pricier and fewer features
Google Pub/Sub:  GCP-managed; auto-scales; topics partitioned automatically
Apache Pulsar:   Alternative with built-in tiered storage (offload old data to S3)
RabbitMQ:        Different model — true queue (push to consumers) rather than log; lower throughput
```

## How to Answer This in an Interview (Step by Step)

```
1. State the goal:
   "A message queue lets services produce events without knowing who consumes them, and
   consumers read at their own pace. We need 1M writes/sec, 7-day retention, ordering
   within a partition."

2. Explain the core model:
   "Topics are named channels; each topic is split into partitions for parallelism.
   Producers route messages by hash(key) % numPartitions — same key always goes to
   the same partition, so order is preserved per key."

3. Explain consumer groups:
   "Each partition is consumed by at most one consumer per group. Multiple groups
   read the same topic independently, each with its own offset. Adding consumers
   within a group scales parallelism up to the partition count."

4. Explain replication and ISR:
   "Each partition has 3 replicas. Writes go to the leader, which copies to followers.
   The set of followers caught up is the In-Sync Replica list. With acks=-1, the
   producer waits for ALL ISR members. If the leader fails, a new one is elected
   only from the ISR — never from a lagging replica."

5. Explain delivery guarantees:
   "At-least-once is the default — process before committing the offset. Combine with
   idempotent consumer logic to handle duplicates safely. Exactly-once is available
   via Kafka transactions for use cases like payments, but it costs latency."

6. Explain storage:
   "Each partition is an append-only log of files on disk. All writes are sequential
   (100× faster than random). The OS page cache serves recent reads from RAM.
   sendfile() does zero-copy network sends. This is why Kafka can saturate a 10G link."

7. Explain log compaction:
   "For topics where I want the latest value per key — like user profiles or config —
   log compaction keeps only the most recent message per key. Older entries are
   eventually deleted. Send a null value to delete a key (tombstone)."

8. Explain rebalancing:
   "When consumers join or leave, the group coordinator reassigns partitions. Modern
   cooperative rebalancing only pauses partitions actually being moved. Static
   membership prevents rebalances during brief restarts like rolling deploys."

9. Monitoring and pitfalls:
   "The key metric is consumer lag = latest offset minus committed offset. Alert when
   lag grows. Pre-size partitions based on max parallelism. Make consumers idempotent.
   Always process before committing."
```
