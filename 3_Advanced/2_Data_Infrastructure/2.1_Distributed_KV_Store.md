# Distributed Key-Value Store

---

## Problem Statement

A distributed key-value store provides low-latency read/write access to arbitrary data by spreading it across many nodes.
Think Redis, DynamoDB, Cassandra, Riak — systems that sacrifice some relational features for horizontal scale and high availability.

```
Use cases:
  Session storage     User session data (auth tokens, cart contents)
  Cache layer         Hot database results (avoid expensive SQL queries)
  Feature flags       Per-user configuration at low latency
  Rate limiting       Counters per IP/user (token bucket state)
  Distributed locks   Leader election, mutex
  Pub/Sub state       Message broker metadata
```

---

## Requirements

### Functional
```
1. Put(key, value)        Store a key-value pair
2. Get(key)               Retrieve value by key; return null if missing
3. Delete(key)            Remove a key
4. TTL support            Keys expire automatically after N seconds
5. Partitioning           Data spread across N nodes; each node owns a range
6. Replication            Each partition has R replicas for fault tolerance
7. Tunable consistency    Choose between strong/eventual per operation
```

### Non-Functional
```
Scale:       10B keys; 1KB avg value = ~10 TB total data
Throughput:  500K reads/sec, 100K writes/sec
Latency:     p99 read < 5ms, p99 write < 10ms
Durability:  Data survives node failure (replication factor ≥ 3)
Availability: 99.99% uptime (prefer AP over CP for most workloads)
```

## Capacity Estimation

```
Keys:          10B × 1KB avg = 10 TB
Nodes:         10 TB ÷ 500 GB/node = 20 data nodes
               With RF=3: 60 total node-partitions
Replication:   RF=3 → can tolerate 2 simultaneous node failures

Memory (hot tier):
  10% of data "hot" = 1 TB → ~14 cache nodes (64 GB RAM each)
  Bloom filter per partition: 1B keys × 1.2 bytes = 1.2 GB

Read throughput:
  500K reads/sec ÷ 20 nodes = 25K reads/node/sec → manageable
  With RF=3 and read from 1 replica → spread to 60 endpoints

Write throughput:
  100K writes/sec; W=2 quorum → each write touches 2 nodes
  200K node-writes/sec ÷ 60 nodes ≈ 3.3K writes/node/sec
```

## Consistent Hashing — Partitioning

```
Problem: with N nodes, adding/removing a node invalidates N/N+1 fraction
         of all keys → massive cache misses or data migrations

Solution: consistent hashing ring
  1. Hash each node to a point on a 0–2^32 ring
  2. Hash each key to a point on the ring
  3. Key belongs to the first node clockwise from its hash position

Virtual nodes (vnodes):
  Each physical node gets V virtual nodes (e.g., V=150)
  → Better load distribution
  → When a node fails, its load is distributed to all others (not just one)
  → When adding a node, it takes a few vnodes from many nodes

Example (3 nodes, ring positions):
  Node A: positions 10, 45, 120, 200, ...
  Node B: positions 25, 70, 155, 230, ...
  Node C: positions 5,  90, 175, 260, ...
  Key "user:123" hashes to position 80 → goes to Node B (position 90 next clockwise)
```

## Replication

```
Replication Factor (RF): typically RF=3
  Every key is stored on the primary node + RF-1 next nodes clockwise

Replication strategies:
  1. Synchronous (Strong consistency):
     Write to primary → wait for all RF replicas to ACK
     + Guarantees all reads see latest write
     − Higher write latency; less available under node failure

  2. Quorum (Tunable):
     W + R > RF  (typical: RF=3, W=2, R=2)
     W=2: write ACKed by 2 of 3 replicas
     R=2: read from 2 of 3 replicas; return latest version
     + Balances latency and consistency
     − Stale reads still possible within quorum

  3. Eventual (High availability):
     Write to primary only; async replication to replicas
     + Lowest latency writes, always available
     − Stale reads until convergence (seconds typically)

Conflict resolution (for eventual consistency):
  Last-Write-Wins (LWW):    highest timestamp wins; simple but loses updates
  Vector clocks:            track causality per client; reconcile diverged versions
  CRDTs:                    conflict-free replicated data types (counters, sets)
```

## Write Path — LSM Tree

```
Why LSM (Log-Structured Merge-Tree)?
  Random writes to disk are slow (HDD: ~100 IOPS; SSD: ~10K IOPS)
  LSM converts random writes → sequential appends → fast

Step 1: WAL (Write-Ahead Log)
  Append operation to WAL on disk immediately
  Crash-safe: replay WAL to recover MemTable after restart

Step 2: MemTable
  Write to in-memory sorted structure (Red-Black tree / skip list)
  Serves all reads for recently written keys

Step 3: Flush to SSTable
  When MemTable hits size limit (e.g., 64 MB) → flush to immutable SSTable on disk
  SSTable: sorted file with an index at the end for binary search

Step 4: Compaction
  Background process merges SSTables
  Removes overwritten values, expired TTL keys (tombstones)
  Levels: L0 (newest), L1, L2, L3 (oldest, most compacted)

Read path:
  1. Check MemTable (O(log n))
  2. Check Bloom filter → skip SSTable if key definitely absent
  3. Binary search L0 SSTables (newest first)
  4. Binary search L1, L2, ... until found
  Bloom filter eliminates ~99% of unnecessary SSTable disk reads
```

## Read Path — Consistency Levels

```
Operation: GET("user:42")
  R=1:  Fastest; read from nearest replica; may be stale
  R=2:  Read from 2 of 3 replicas; take highest version
  R=3:  Read all 3; guaranteed fresh (linearizable)

Read repair:
  During quorum read, if replicas disagree on version
  → Return highest version to client
  → Async: update stale replicas with latest value

Hinted handoff:
  Node A is down; writes intended for A go to a "hint" node
  When A recovers, the hint node forwards buffered writes to A
  Prevents data loss without blocking writes
```

## Failure Detection — Gossip Protocol

```
Problem: how do 100 nodes detect which peers are alive/dead?
  Centralized heartbeat: single coordinator = bottleneck + SPOF

Solution: gossip protocol (epidemic broadcast)
  1. Every T seconds, each node randomly picks a few peers
  2. Exchanges state vector: {node_id → (status, version, last_seen)}
  3. Merges received state with own state (higher version wins)
  4. Propagates changes exponentially: log(N) rounds to reach all nodes

Failure detection:
  If last_seen(node) > threshold (e.g., 3×T = 9s) → suspect failure
  Suspicion confirmed after additional rounds → mark DOWN
  ZooKeeper / etcd can complement for strong leader elections
```

## CAP Theorem — Design Trade-offs

```
CAP: a distributed system can guarantee only 2 of 3:
  C — Consistency:   every read returns the latest write
  A — Availability:  every request gets a response (no timeout)
  P — Partition tolerance: system works despite network splits

KV stores choose:
  CP (prefer consistency):
    Zookeeper, HBase, etcd
    Under network partition: some nodes refuse requests rather than serve stale data
    Use case: distributed locks, config management

  AP (prefer availability):
    Cassandra, DynamoDB, Riak, CouchDB
    Under network partition: nodes serve stale data; resolve conflicts later
    Use case: session store, shopping cart, social feed

PACELC extension:
  Even without partition (normal operation):
    Choose between Latency (L) vs Consistency (C)
  DynamoDB: PA/EL (available+low-latency by default)
  Spanner: PC/EC (consistent but higher latency)
```

## Slab Allocator (Memory Management)

```
Problem: naive malloc/free causes heap fragmentation
  A 100-byte value request gets 128-byte allocation
  Freed slot can only be reused by same-size request
  After millions of operations: lots of unusable gaps

Solution: slab allocator (Memcached model)
  Pre-allocate "slabs" of fixed-size chunks
  Slab classes: 64B, 128B, 256B, 512B, 1KB, 2KB, ...
  Each key+value rounded up to nearest slab class
  Free list per slab class: O(1) alloc/free

Trade-off: ~50% memory waste on average; eliminated fragmentation
```

## Pitfalls

```
1. Hot partitions (hotspot keys)
   Problem: "trending" key gets 90% of traffic → one node overwhelmed
   Fix: local shard caching for hot keys; key suffix randomization for writes

2. No TTL expiration sweep
   Problem: expired keys linger, consuming memory
   Fix: lazy expiration (check on read) + active sweep (background thread)

3. Clock skew in Last-Write-Wins
   Problem: two servers disagree on current time → wrong winner
   Fix: use logical clocks (Lamport timestamps) or vector clocks instead

4. Thundering herd on cache miss
   Problem: popular key expires; 10K clients simultaneously hit database
   Fix: probabilistic early expiration; read-lock + single filler pattern

5. Compaction blocking I/O
   Problem: background compaction saturates disk bandwidth → high p99 latency
   Fix: I/O throttling for compaction; keep L0 small; use separate disk for WAL

6. Unbalanced vnodes
   Problem: random hash of node IDs → some nodes get 2× load
   Fix: deterministic vnode placement; periodic rebalancing

7. Stale reads after failover
   Problem: replica promoted to primary; not fully caught up
   Fix: wait for replica lag = 0 before promotion (or use R=RF reads during failover)
```

## Interview Answer Framework

1. **Data model:** key → value (opaque bytes); support TTL, atomic compare-and-swap (CAS)
2. **Partitioning:** consistent hashing ring with virtual nodes; avoids mass rehashing on node changes
3. **Replication:** RF=3, quorum W=2/R=2; tunable per operation; hinted handoff for unavailable nodes
4. **Write path:** WAL → MemTable → SSTable (LSM tree); sequential I/O beats random writes by 100×
5. **Read path:** MemTable → Bloom filter → SSTable levels; read repair for stale replicas
6. **Failure detection:** gossip protocol; O(log N) rounds to converge; hinted handoff preserves writes
7. **Consistency model:** AP by default (eventual); strong consistency via R=W=RF (slower)
8. **CAP choice:** AP for session/cache workloads; CP for config/lock workloads
9. **Hot keys:** local cache + randomized write sharding; TTL jitter to spread expiry bursts
10. **Memory:** slab allocator avoids fragmentation; Bloom filter avoids unnecessary disk reads
