# Distributed Key-Value Store (like Redis or DynamoDB)

---

## What Are We Building?

A **key-value store** is the simplest possible database: you give it a key, it gives you a value back.

```
store.Put("session:user42", "{name: Alice, cart: [item1, item2]}")
store.Get("session:user42")  →  "{name: Alice, cart: [item1, item2]}"
store.Delete("session:user42")
```

Think of it like a giant dictionary — or a locker room where each locker has a number (the key)
and you put whatever you want inside (the value).

**What it's used for:**
```
Session storage    → "is user42 logged in? what's in their cart?"
Rate limiting      → "how many API calls has this IP made in the last minute?"
Feature flags      → "is the new checkout button enabled for this user?"
Caching            → "store the result of this expensive database query for 60 seconds"
Leaderboards       → "what are the top 10 scores right now?"
```

The twist: we need this to handle **10 billion keys** and **500,000 reads per second**.
That's too much for one server. So we split the data across many servers — that's the "distributed" part.

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Put(key, value)   — store data
2. Get(key)          — retrieve data; return "not found" if the key doesn't exist
3. Delete(key)       — remove data
4. TTL support       — "this key expires in 60 seconds" (auto-delete)
5. Spread data       — across many servers so no single server holds everything
6. Backup copies     — keep 3 copies of each piece of data (so if a server dies, data is safe)
7. Consistency knob  — choose between "always fresh data" vs "faster but possibly slightly stale"
```

### Non-Functional (scale and reliability)
```
Scale:       10 billion keys, ~1 KB each = about 10 TB of data
Speed:       500,000 reads/second, 100,000 writes/second
Latency:     99% of reads finish in under 5ms; 99% of writes in under 10ms
Durability:  Data survives if any server dies (keep 3 copies)
Uptime:      99.99% (less than 1 hour of downtime per year)
```

## How Much Hardware Do We Need?

Before building anything, engineers estimate **how big the system needs to be**. This is called
"capacity planning" or "back-of-the-envelope estimation." For most systems the first question is
"how much traffic?" — but this one is different. Here the *amount of data itself* is the problem,
and it's what forces the entire distributed design. So our questions are:

1. **How many servers just to hold the data?** (it won't fit on one machine)
2. **How many copies do we keep** so nothing is lost when a server dies?
3. **Can those servers handle the traffic**, and what about hot keys?

Let's work through them, slowly.

---

### Question 1: How many servers just to *hold* the data?

This is the insight that defines the whole system: **the data is far too big for any single
machine.** Watch how big it gets.

```
Step 1 — Total data size:
  10 billion keys × ~1 KB each = 10 TB of data

Step 2 — How much fits on one server?
  A reasonable data server holds ~500 GB.

Step 3 — So how many servers just to store it once?
  10,000 GB ÷ 500 GB per server = 20 servers
```

**The key insight:** 10 TB can't live on one box, so we *must* chop the data into pieces and
spread them across ~20 servers. This splitting is called **sharding**, and it's the reason this is
a "distributed" key-value store at all. (How we decide which key goes to which server — without
overloading one — is the consistent-hashing problem covered in the next section.)

> Takeaway: it's *size*, not speed, that forces this design. The data simply doesn't fit on one
> machine, so it has to be split.

---

### Question 2: How many copies do we keep?

If we store each piece of data on exactly one server, then when that server dies (and servers
*always* eventually die), we permanently lose 1/20th of all our data. Unacceptable. So we keep
**multiple copies** of everything on different servers — this is called **replication**.

```
Replication factor = 3  (the industry-standard "keep 3 copies")
  20 servers of data × 3 copies = 60 total servers
```

**The key insight:** replication trades **3× the hardware** for **durability** — now any single
server (even two) can die and every piece of data still survives on another machine. The "3" isn't
magic; it's the common sweet spot between cost and safety. This is also what lets us promise
99.99% uptime: a dead server just means readers switch to one of its copies.

---

### Question 3: Can those servers handle the traffic (and hot keys)?

We sized the cluster for *storage*. Now let's **double-check** it can also handle the *read load* —
and deal with the reality that some keys are wildly more popular than others.

```
Throughput check — does 60 servers handle 500,000 reads/sec?
  500,000 reads/sec ÷ 60 servers ≈ 8,300 reads/sec per server
  → that's a comfortable load for one server → we're fine ✓

Hot keys — the 10% of data that gets 90% of the traffic:
  10% of 10 TB = 1 TB of "popular" data we want in fast memory (RAM)
  Each cache server has ~64 GB RAM:
  1,000 GB ÷ 64 GB ≈ 16 cache servers
```

**What this tells us:** two things. First, the 60 servers we bought for *storage* turn out to
comfortably handle the *read traffic* too (8,300/sec each is easy) — so **storage was the binding
constraint**, not throughput. Second, traffic is never spread evenly: a small slice of "hot" keys
(a viral item, a celebrity's profile) gets most of the requests, so we put that ~1 TB in a
dedicated **in-memory cache** (~16 servers) to keep those reads blazing fast and protect the data
servers from being hammered.

---

### The numbers to remember

```
~20 servers     → just to STORE 10 TB (size forces sharding)
~60 servers     → after keeping 3 copies (replication buys durability)
~8,300 reads/sec each → comfortable, so storage—not traffic—is the limit
~16 cache servers → hold the hot 10% in RAM for speed
```

The big idea connecting them: unlike most systems, this one is driven by **data size, not request
rate**. The data won't fit on one machine (→ shard across 20), losing a machine can't lose data
(→ replicate to 60), and a hot minority of keys gets cached in RAM for speed.

## The Big Problem: How Do We Split Data Across 20 Servers?

### Approach 1: Alphabetical Split (Bad)

Assign letters to servers: Server 1 gets keys starting with A–E, Server 2 gets F–J, etc.

**Problem:** The letter "S" is far more common than "X". Server 1 gets overloaded, Server 5 sits idle.

Also, if we add a 21st server, we have to redistribute ALL the data — massive work.

---

### Approach 2: Consistent Hashing (The Right Answer)

Here's the key insight: imagine a **clock face** with numbers 0 to 4 billion around the edge.

```
Step 1: Place each server on the clock face by hashing its name.
  hash("Server1") = 450 million  → Server 1 is at position 450M
  hash("Server2") = 1.2 billion  → Server 2 is at position 1.2B
  hash("Server3") = 2.8 billion  → Server 3 is at position 2.8B

Step 2: Place each key on the clock face by hashing it.
  hash("session:user42") = 800 million  → this key is at position 800M

Step 3: Walk clockwise until you hit a server. That server owns this key.
  Position 800M, walking clockwise... hit Server 2 at 1.2B.
  → "session:user42" belongs to Server 2.
```

**Why this is great:** if Server 2 goes down, its keys just "roll over" to Server 3 — the next one clockwise.
Only the keys that were on Server 2 need to move. All other servers are unaffected.

Compare this to alphabetical split: if we add one server, we'd need to rematch every key to its new letter range.
With consistent hashing, adding one server only moves ~5% of keys (just the portion it takes from its neighbor).

**Virtual nodes — solving uneven load:**

Even with hashing, some servers might randomly land closer together on the clock face and get more keys.

Fix: give each physical server **150 spots** on the clock face instead of 1.

```
Physical Server 1 gets virtual spots at: 45, 320, 890, 1200, ... (150 spread-out positions)
Physical Server 2 gets virtual spots at: 78, 415, 950, 1350, ... (150 different positions)
```

Now every server owns about 1/20th of the clock face on average, regardless of where they landed.

## Replication: Keeping 3 Copies of Everything

Once we know which server owns a key (its "primary"), we also store copies on the next 2 servers
clockwise on the ring.

```
"session:user42" belongs to Server 2.
Copies also go to: Server 3 (next clockwise) and Server 4 (one more clockwise)

So if Server 2 crashes, Server 3 or Server 4 can answer reads.
If Server 2 AND Server 3 both crash simultaneously, Server 4 still has it.
```

This is **replication factor 3** — you can lose 2 servers and still not lose any data.

---

## How Consistent Does the Data Need to Be?

When you have 3 copies of data on 3 different servers, they can drift apart.
The question is: how many copies does a write need to update before we say "success"?

```
W = number of copies that must confirm a write before we return "OK" to the client
R = number of copies we read from (and take the freshest version)
RF = replication factor (total copies) = 3

The rule: if W + R > RF, you're guaranteed to always read your own writes.
```

**Three modes:**

```
Fast writes (Eventual Consistency): W=1, R=1
  Write goes to 1 server, reads from 1 server.
  + Lowest latency, always available even if 2 servers are down
  - You might read a slightly stale value for a few seconds after a write
  Good for: shopping cart, session data, social feed

Balanced (Quorum): W=2, R=2
  Write must land on 2 of 3 servers before returning "OK".
  Reads come from 2 servers; take the freshest version.
  + Good balance of speed and freshness
  - Writes are slower because you're waiting for 2 confirmations
  Good for: most applications

Strongest (Read Everything): W=3, R=3
  Every read hits all 3 servers; you're guaranteed the freshest data.
  + Strongest possible guarantee
  - Slowest; if any server is down, reads fail
  Good for: financial transactions, config values
```

**The simple way to remember it:** W=2, R=2 with RF=3 is like a vote. If you write to 2 servers and
read from 2 servers, at least 1 server has BOTH the latest write AND the read request — so you'll
always get the latest data.

## Write Path: How Data Is Stored (The LSM Tree)

Storing data efficiently on disk is harder than it sounds.

**The naive approach:** just write each key directly to disk wherever it belongs.

**Problem:** every write jumps to a random location on disk. Hard disks are slow at random jumps —
think of a record player needle jumping all over a vinyl. Sequential writes (writing to one place at
the end of a file) are 100× faster.

**The solution: LSM Tree (Log-Structured Merge Tree)**

Think of it like how you'd take notes during a meeting:

```
Step 1: STICKY NOTE (Write-Ahead Log / WAL)
  The moment a write comes in, scribble it immediately on a sticky note.
  This is purely for crash recovery — if the power dies, you can replay the sticky notes.
  This is a sequential append, so it's very fast.

Step 2: NOTEBOOK (MemTable)
  Also write it to an in-memory sorted notebook.
  This notebook is what you actually read from for recent writes.
  "In-memory" means it's in RAM — extremely fast, no disk involved.
  The notebook is kept sorted alphabetically by key (so lookups are fast).

Step 3: FILING CABINET (SSTable on disk)
  When the notebook fills up (say, 64 MB), copy it to a permanent file on disk.
  This file is called an SSTable (Sorted String Table).
  It's immutable — you never modify it, only create new ones.
  Creating it is one big sequential write — fast!

Step 4: ARCHIVING (Compaction)
  Over time, many SSTables accumulate.
  A background process merges old SSTables together, removing outdated values.
  (If you wrote key "user:42" three times, only the latest value matters.)
```

**Read path:**

To find a key, you check in order from freshest to oldest:

```
1. Check the MemTable (notebook)           → if found here, return immediately
2. Check each SSTable, newest first        → but searching 100 SSTables is slow...
```

This is where the **Bloom filter** saves us.

---

## Bloom Filter: Should We Even Look?

Before searching an SSTable file on disk, we ask: "is this key probably in this file?"

A Bloom filter is a small piece of memory that answers this question in less than 1ms.

Think of it like a checklist with 100 checkboxes. When a key is added to an SSTable:
- Run it through 3 different hash functions → get 3 numbers between 1 and 100
- Check those 3 boxes

To check if a key "might be" in the SSTable:
- Run the key through the same 3 hash functions
- If ANY of the 3 boxes is unchecked → key is DEFINITELY NOT in this SSTable → skip it

```
Result of Bloom filter check:
  All 3 boxes checked   → "maybe" in this file (search it)
  Any box unchecked     → "definitely not" in this file (skip it)
```

The Bloom filter can say "maybe" even when the key isn't there (false positive — we do a wasted search).
But it NEVER says "not here" when the key actually is there (no false negatives).

**Why this matters:**
Without Bloom filter: reading 1 key might require checking 50 SSTable files on disk → slow.
With Bloom filter: checking 1 million keys in a Bloom filter takes 1.2 GB of RAM, and it skips 99% of
unnecessary disk reads.

## Read Repair: Keeping Copies in Sync

When you read with quorum (R=2), you ask 2 servers and take the freshest version.

What if those 2 servers disagree?

```
Server 2 says: user42.name = "Alice"   (version 5)
Server 3 says: user42.name = "Alicia"  (version 7)

→ Return version 7 to the client (latest wins)
→ Async: tell Server 2 "hey, you're behind — here's the latest value"
```

Server 2 is now caught up. The next read will be consistent.
This is called **read repair** — stale copies fix themselves automatically as part of normal reads.

---

## Hinted Handoff: What If a Server Is Down During a Write?

If Server 2 is down when a write comes in, but we still need W=2 confirmations:

```
Normal target: Server 2 (down) + Server 3 (up)
We can only write to Server 3.
Instead, we ask Server 4: "hold this write as a hint for Server 2"
Server 4 stores: {intended_for: Server2, key: "user:42", value: "Alice"}

When Server 2 comes back online:
Server 4 sees it, forwards the hint, Server 2 catches up.
```

This prevents data loss without blocking writes when a server is temporarily unavailable.

## Failure Detection: How Do 20 Servers Know Who Is Alive?

**Naive approach:** one central "heartbeat server" that everyone checks in with.
Problem: what if THAT server goes down? Or what if 20 servers all ping it every second? (20,000 pings/sec)

**Better approach: Gossip Protocol**

Think of it like how rumors spread through an office:

```
Step 1: Every few seconds, each server randomly picks 2–3 other servers.
Step 2: They exchange their "what I know about everyone" table.
Step 3: Each server updates its table with any newer information it just heard.
Step 4: Repeat.
```

If Server 5 goes down, Server 8 notices (no heartbeat in 9 seconds) → tells Servers 2 and 11 →
they tell 4 others → within about 10 "gossip rounds," all 20 servers know Server 5 is down.

The math: in 20 servers, information reaches everyone in about log₂(20) ≈ 4–5 rounds.
Each round takes 1–2 seconds. So the whole network knows about a failure within ~10 seconds.

## CAP Theorem: The Fundamental Trade-Off

In a distributed system, you can only fully guarantee 2 of these 3 things:

```
C — Consistency:         every read returns the most recent write
A — Availability:        every request gets a response (never hangs)
P — Partition tolerance: system keeps working even if servers can't talk to each other
```

Network splits (partitions) happen in the real world — a cable gets cut, a router fails.
So you MUST handle partitions. That means the real choice is **C vs A** during a network split:

```
Choose Consistency (C): if two servers can't agree on the latest value, REFUSE the request.
  → User gets an error, but data is never stale.
  → Used by: ZooKeeper, etcd (for distributed locks, config — where wrong data is dangerous)

Choose Availability (A): if servers can't talk, answer with whatever you have, even if slightly stale.
  → User always gets a response, but might be slightly out of date.
  → Used by: Cassandra, DynamoDB (for sessions, carts — where a 1-second stale read is fine)
```

**For most key-value stores: choose A.** A shopping cart being 2 seconds stale is acceptable.
A distributed lock returning stale data would be catastrophic.

## Hot Keys: When One Key Gets All the Traffic

Imagine a sudden tweet from a celebrity goes viral. Suddenly millions of people are looking up
the same key: `"post:tweet:99999"`.

All those requests go to one server (the one that owns that key). That server gets overwhelmed
while all other servers sit idle.

**Fix:**

```
Option 1: Local copies on every server
  When a key is "hot," copy it to all 20 servers temporarily.
  Reads are spread across all 20 servers.
  Writes still go to primary, which broadcasts the update.

Option 2: Randomized key suffix
  Instead of writing "product:iphone16" once, write it as:
    "product:iphone16:r3"
    "product:iphone16:r7"
    "product:iphone16:r14"
  (10 random copies on 10 different servers)
  Reads pick a random suffix → spread across 10 servers
  Writes go to all 10 → slightly more work, but no single hot spot
```

## TTL Expiry: How Keys Auto-Delete

Some keys should expire automatically:

```
store.Put("rate_limit:ip:1.2.3.4", "47 requests", TTL: 60 seconds)
```

After 60 seconds, this key should disappear.

**Two strategies (both are used):**

```
Lazy expiry:
  When someone asks for the key, check if it's expired.
  If yes, delete it right now and return "not found."
  + Simple; no background work
  - Expired keys can linger in memory until someone asks for them

Active sweep (background thread):
  Every second, pick 20 random keys and check if they're expired. Delete if so.
  + Memory is reclaimed promptly
  - Uses a tiny bit of CPU continuously
```

Redis uses both — lazy eviction plus an active sweep every 100ms.

## How It All Fits Together

```
Client wants: GET("session:user42")
                    │
                    ▼
             Coordinator Node
             ├─ hash("session:user42") = position 800M on the ring
             ├─ Find owner: Server 2 (position 1.2B, next clockwise)
             └─ Also find replicas: Server 3, Server 4
                    │
          ┌─────────┼─────────┐
          ▼         ▼         ▼
       Server2   Server3   Server4
       (primary) (replica) (replica)
          │
          ▼  (Read path on Server 2)
       1. Check MemTable → not there (key was written 2 hours ago)
       2. Check Bloom filter for each SSTable → "definitely not" in 8 of 10
       3. Search 2 remaining SSTables → found in SSTable #3
       4. Return value
                    │
             Take freshest version from R=2 servers
             If Server 3 is stale → async read repair
                    │
                    ▼
             Return to client  (< 5ms total)
```

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ Not handling hot keys
   → One key gets 1 million reads/second → one server crashes
   ✅ Detect hot keys and scatter copies across many servers

❌ Forgetting Bloom filters
   → Every read checks every SSTable file → disk I/O is 100× too slow
   ✅ Bloom filter skips 99% of SSTable files before touching disk

❌ Last-Write-Wins with wall clock time
   → Server A thinks it's 12:00:01, Server B thinks it's 12:00:00
   → Server A's write beats Server B's even if B's was actually later
   ✅ Use logical clocks (Lamport timestamps) or vector clocks that aren't affected by clock skew

❌ TTL keys that never get cleaned up
   → Expired keys pile up in memory; system runs out of RAM
   ✅ Use both lazy expiry (on read) AND an active background sweep

❌ Choosing CP for a session store
   → If 2 servers can't talk to each other, ALL session reads fail
   ✅ Session data should be AP — a slightly stale session is fine; an error is not

❌ All replicas on the same rack / availability zone
   → Power goes out in Datacenter A → you lose all 3 copies
   ✅ Spread replicas across 3 different physical locations

❌ No compaction, just accumulating SSTables forever
   → More SSTables to search = reads get slower every day
   ✅ Background compaction merges SSTables and removes overwritten/expired entries
```

## Real Systems at a Glance

```
Redis:      Single server (in-memory), extremely fast, often used as a cache layer
DynamoDB:   Amazon's managed distributed KV store; auto-scales; uses eventual consistency by default
Cassandra:  Wide-column, ring-based gossip, tunable consistency; great for time-series data
etcd:       Small, strongly consistent (CP); used inside Kubernetes for cluster config
```

## How to Answer This in an Interview (Step by Step)

```
1. State what it is:
   "A key-value store is the simplest database — store a value, retrieve it by key.
   Distributed means spreading data across many servers since 10 billion keys won't fit on one."

2. Explain partitioning:
   "I'd use consistent hashing — imagine a clock face where each server and each key gets
   a position. A key goes to the nearest server clockwise. Adding servers only moves ~5%
   of keys. Virtual nodes (150 per server) prevent uneven load."

3. Explain replication:
   "Replication factor of 3 — each key is stored on the primary and the next 2 servers
   clockwise. You can lose 2 servers and still not lose any data."

4. Explain consistency:
   "Tunable consistency: W+R>RF guarantees fresh reads. W=2, R=2 with RF=3 is the
   sweet spot — fast enough, consistent enough. Eventual (W=1) for maximum speed."

5. Explain the write path:
   "LSM tree: every write goes to a WAL (crash recovery), then the in-memory MemTable.
   When MemTable fills up, flush it to an immutable SSTable on disk.
   This converts random writes into sequential writes — 100× faster."

6. Explain the read path:
   "Check MemTable first. Then use Bloom filters to skip SSTables where the key
   definitely isn't — eliminates 99% of disk reads. Background read repair keeps
   replicas in sync."

7. Explain failure detection:
   "Gossip protocol — each server shares health info with random peers every few seconds.
   Failures propagate to all 20 servers in about 10 seconds. No single point of failure."

8. Explain CAP:
   "I'd choose AP — availability over consistency. A session read that's 2 seconds stale
   is fine. A session read that returns an error is not. Use CP only for locks or config."

9. Handle hot keys:
   "Detect hot keys by tracking read counts. Once a key passes a threshold, scatter copies
   across all servers and spread reads evenly."

10. Handle TTL:
    "Lazy expiry on every read, plus a background sweep thread checking random keys every
    100ms. Both together ensure expired memory is reclaimed promptly."
```
