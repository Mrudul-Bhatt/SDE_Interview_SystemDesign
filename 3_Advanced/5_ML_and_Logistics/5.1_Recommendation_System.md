# Recommendation System (Netflix / YouTube / Amazon)

---

## Problem Statement

Design a system that recommends relevant items to users — like Netflix movie suggestions, YouTube video feed, Amazon product recommendations, or Spotify Discover Weekly. The system must handle millions of users and items, serve recommendations in milliseconds, and improve over time from user feedback.

```
Use cases:
  Netflix      "Because you watched…" — movie and show recommendations
  YouTube      Homepage feed, Up Next, search results ranking
  Amazon       "Customers also bought…", personalized homepage
  Spotify      Discover Weekly, Daily Mix, Radio
  TikTok       For You Page — hyper-personalized short video feed
  LinkedIn     People you may know, jobs, content feed
```

---

## Requirements

### Functional
```
1. Personalized recommendations per user (homepage feed, related items)
2. Handle new users (cold start) — no interaction history available
3. Handle new items (cold start) — no ratings yet
4. Real-time signal incorporation (what user just clicked/watched)
5. Diverse recommendations (not all the same genre/type)
6. Explainability: "Recommended because you watched X"
```

### Non-Functional
```
Scale:       300M users × 100M items = 30 trillion user-item pairs (too large to store)
             Serve recommendations in < 100ms (online serving)
             Retrain model daily or hourly on latest interactions
Freshness:   New items must surface within hours (not days)
Diversity:   Top-10 list should not be all the same category
Coverage:    Long-tail items (not just popular ones) should get exposure
```

## Two Fundamental Approaches

### Collaborative Filtering (CF)
```
Idea: users who agreed in the past will agree in the future.
      No need to understand the item — just find similar users/items.

User-based CF:
  "Users similar to you also liked X."
  1. Find K users most similar to target user (cosine/Pearson similarity)
  2. Take items they liked that target user hasn't seen
  3. Aggregate scores weighted by similarity

Item-based CF:
  "Because you liked X, you may like Y (similar items)."
  1. For each item target user liked, find K similar items
  2. Items are similar if many users co-rated them similarly
  3. Score = Σ(similarity(liked_item, candidate) × user_rating)

  Item-based is preferred in practice:
    - Item similarity is stable (recompute weekly)
    - User similarity changes as users add new ratings
    - Scales better: item-item matrix is n_items × n_items (manageable)

Similarity metrics:
  Cosine:    cos(u,v) = (u·v) / (||u|| × ||v||)  — ignores rating scale
  Pearson:   correlation coefficient — mean-centered cosine
  Jaccard:   |A∩B| / |A∪B|  — good for binary (liked/not liked)
  Adjusted:  mean-center each user's ratings first (removes rating bias)

Limitations:
  Sparsity: most user-item pairs unrated (> 99.9% sparse)
  Cold start: no recommendations for new users or items
  Scalability: computing all pairwise similarities is O(n²)
  Popularity bias: popular items dominate recommendations
```

### Content-Based Filtering
```
Idea: recommend items similar to what you've liked, based on item features.

Steps:
  1. Build item feature vector (genre, director, cast, tags, description TF-IDF)
  2. Build user profile = weighted average of features of items they liked
  3. Score: cosine_similarity(user_profile, candidate_item)

Advantages:
  - Solves new item cold start (features available immediately)
  - Explainable: "Recommended because you like Action movies"
  - No other users' data needed (privacy-friendly)

Limitations:
  - Requires good item metadata
  - Filter bubble: only recommends more of the same
  - Can't discover serendipitous items outside user's known taste
```

## Matrix Factorization

```
Core insight: approximate the sparse R matrix as a product of two dense matrices.

  R ≈ U × V^T

  R: user-item rating matrix  [n_users × n_items]  (mostly unknown)
  U: user latent factor matrix [n_users × k]        (k = embedding dim, e.g. 64)
  V: item latent factor matrix [n_items × k]

  Predicted rating for user u, item i:
    r̂(u,i) = U[u] · V[i]   (dot product of k-dimensional vectors)

  Loss: minimize Σ_{(u,i) observed} (r(u,i) - U[u]·V[i])² + λ(||U||² + ||V||²)
    (L2 regularization prevents overfitting)

Training algorithms:
  SGD (Stochastic Gradient Descent):
    For each observed (u, i, r):
      error = r - U[u]·V[i]
      U[u] += η * (error * V[i] - λ * U[u])
      V[i] += η * (error * U[u] - λ * V[i])

  ALS (Alternating Least Squares):
    Fix V, solve for U analytically → closed form per user
    Fix U, solve for V analytically → closed form per item
    Alternates until convergence
    Better for implicit feedback (plays, clicks, purchases)

Implicit vs Explicit feedback:
  Explicit: star ratings (1–5) — rare, biased (people rate extremes)
  Implicit: clicks, watch time, purchases — abundant, noisy
    Positive: user interacted with item
    Negative: not interacted ≠ disliked (may not have seen it)
    Confidence weighting: c(u,i) = 1 + α * interaction_count

SVD / Funk SVD:
  Classic matrix factorization (Netflix Prize winner)
  Add bias terms: r̂(u,i) = μ + b_u + b_i + U[u]·V[i]
  μ = global mean, b_u = user bias, b_i = item bias
  Handles "users who always rate high" and "items always rated low"
```

## Two-Tower Architecture (YouTube / TikTok)

```
Modern approach for large-scale retrieval.

Architecture:
  User Tower:  user_id embedding + context features → user vector (d=256)
  Item Tower:  item_id embedding + content features → item vector (d=256)

  Score(u, i) = cosine_similarity(user_vector, item_vector)

Training: contrastive learning
  Positive pairs: (user, item they interacted with)
  Negative pairs: random items user didn't interact with (in-batch negatives)
  Loss: maximize score of positive, minimize score of negatives

Why two towers?
  Pre-compute all item vectors offline → store in vector index (FAISS)
  At serve time: compute user vector → ANN search in item space → top-K candidates
  ANN search in 256-dim space over 100M items: < 10ms (HNSW index)

Compared to MF:
  Two-tower can incorporate rich features (text, image, context)
  MF is ID-based only (no features)
  Two-tower handles cold-start better (item features available)
```

## Retrieval + Ranking Pipeline

```
Two-stage pipeline (used by Google, Netflix, YouTube, TikTok):

Stage 1 — Retrieval (Candidate Generation):
  Goal: reduce billions of items to top-500 candidates in < 50ms
  Methods:
    - Two-tower ANN search: user vector → nearest item vectors
    - Item-based CF: items similar to user's recent interactions
    - Trending/popular items: always include some fresh content
    - User's followed channels/artists: direct subscription items
  Output: ~500 diverse candidates

Stage 2 — Ranking:
  Goal: score each candidate with rich features; pick top-10–50
  Features:
    - User-item relevance score (from retrieval)
    - Item freshness (time since publish)
    - Item popularity (CTR, engagement rate)
    - User context (device, time of day, location)
    - Historical engagement (did user finish similar items?)
    - Position bias correction (propensity score weighting)
  Model: gradient boosted trees (XGBoost) or small neural net
  Output: ranked list of ~50 items

Stage 3 — Post-processing:
  Diversity filter: at most 2 items per category in top-10
  Business rules: boost promoted content, filter blocked content
  A/B experiment allocation: serve variant based on user's bucket

Why separate retrieval and ranking?
  Can't run complex ranking model on billions of items
  Retrieval must be fast (ANN) but can be less precise
  Ranking can use expensive features but only on ~500 items
```

## Cold Start Problem

```
New User Cold Start:
  Problem: no interaction history → no personalized signal
  Solutions:
    1. Popularity-based: show trending / top-rated items globally
    2. Onboarding flow: ask user to rate 5–10 items or select interests
    3. Demographic fallback: use age/location/device to select cohort model
    4. Session-based: immediately incorporate what user clicks during session
       (even first 3 clicks can drive personalization)
    5. Cross-domain: if user logged in via Google, use search/Gmail signals
       (with consent)

New Item Cold Start:
  Problem: no ratings/clicks → item never gets recommended → never gets ratings
  Solutions:
    1. Content-based features: embed item using text/image → find similar items
    2. Exploration budget: force-include new items in X% of recommendation slots
    3. Bandit algorithms: Thompson sampling — explore new items,
       update confidence as data arrives
    4. Publisher boost: boost new items from popular creators for first 48h
    5. Pre-training on metadata: use item description embedding as initial V[i]
```

## Evaluation Metrics

```
Offline metrics (historical data):
  Precision@K:   fraction of top-K recommendations the user actually liked
                 = |{relevant items in top-K}| / K
  Recall@K:      fraction of all items user liked that appear in top-K
                 = |{relevant items in top-K}| / |{total relevant items}|
  NDCG@K:        Normalized Discounted Cumulative Gain
                 rewards relevant items ranked higher
                 DCG = Σ rel_i / log₂(i+1)  NDCG = DCG / IDCG (ideal)
  MAP@K:         Mean Average Precision — average of P@K across all users
  Coverage:      % of items that appear in at least one recommendation list
  Diversity:     average pairwise distance of items in recommendation list
  Serendipity:   how surprising yet relevant a recommendation is

Online metrics (A/B test):
  CTR:           click-through rate on recommendations
  Watch time:    total minutes watched from recommended videos
  Long-click:    user spent > threshold time on item (signals real engagement)
  Session depth: number of items consumed per session
  Retention:     7-day / 30-day return rate after seeing recommendations

Key insight: offline NDCG ≠ online CTR
  Always validate with A/B test before shipping
  Optimize for the business metric (watch time, revenue), not proxy metric
```

## Real-Time Signals

```
Session context (last N actions, updated in real time):
  User opens app → clicks 3 action movies → next recommendations shift toward action
  Implementation: store session as sliding window in Redis
                  session_key = {userId}:{sessionId}
                  value = [item_id_1, item_id_2, ...last_10]
                  TTL = 30 minutes

Feature freshness pipeline:
  Click event → Kafka → Feature Store (Redis) → Ranking model reads latest features

Two levels:
  Near-real-time (minutes): update user embedding after significant events
  Real-time (ms): include raw session items as features in ranking model
                  (no model retraining needed; features change, model static)
```

## Filter Bubble and Diversity

```
Filter bubble problem:
  User watches 3 cooking videos → system recommends only cooking forever
  → User's taste never expands → engagement eventually drops

Mitigation strategies:

  1. Exploration (ε-greedy):
     With probability ε (e.g., 5%), serve a random recommendation outside
     the predicted set. Collect signal. Exploit the signal later.

  2. Thompson Sampling (Bayesian bandits):
     Maintain Beta(α, β) distribution per item per user context.
     Sample from distribution → recommend highest sample.
     α increases on interaction; β increases on non-interaction.
     Naturally balances exploration vs exploitation.

  3. Diversity constraint in post-processing:
     Maximal Marginal Relevance (MMR):
       Select item that maximizes: λ * relevance - (1-λ) * max_similarity_to_selected
     Ensures selected items are both relevant and different from each other.

  4. Category diversity:
     Hard constraint: at most 2 items per category in top-10.

  5. Serendipity injection:
     1-2 slots reserved for items outside user's predicted taste
     but highly rated by similar users
```

## Pitfalls

```
1. Popularity bias
   Problem: popular items dominate recommendations; niche items never surface
   Fix: normalize scores by item popularity; reserve slots for long-tail items;
        add item novelty to ranking objective

2. Position bias in training data
   Problem: items shown in slot 1 get more clicks regardless of quality
   Fix: inverse propensity scoring (IPS); debias training labels by CTR of position
        propensity = P(click | shown at position i) — estimate from randomized expts

3. Data leakage
   Problem: training on future interactions to predict past → inflated offline metrics
   Fix: strict temporal split; train on t < T, evaluate on t ≥ T

4. Filter bubble / echo chamber
   Problem: only recommend what model predicts user likes → narrows taste
   Fix: diversity constraints, exploration budget, serendipity injection

5. Cold start with no onboarding
   Problem: new user gets generic popular recommendations; no personalization
   Fix: 3-question onboarding (genre/category preferences); incorporate
        first 3 session signals immediately (session-based recommendations)

6. Stale embeddings
   Problem: model trained yesterday; viral item from today not in embedding index
   Fix: near-real-time incremental embedding update every 1–4 hours;
        content-based fallback for items < 4 hours old

7. Training/serving skew
   Problem: features used in training differ from features available at serve time
   Fix: use the same Feature Store for both training and serving;
        log features at prediction time (log-and-wait pattern)

8. No A/B testing → can't measure real impact
   Problem: offline metrics improved but users churn more
   Fix: every model change goes through A/B test; guard rails: CTR, session length,
        7-day retention must not drop below threshold
```

## Real-World Systems

```
Netflix:   Two-stage (retrieval + ranking); separate models per context
           (homepage, similar items, continue watching);
           Causal inference for long-term impact measurement

YouTube:   Two-tower retrieval (Candidate Generation); deep neural ranking;
           Optimizes for watch time, not just CTR; position debiasing

Spotify:   Collaborative filtering + audio embeddings (Word2Vec on playlists);
           Discover Weekly = user history + collaborative signal;
           BaRT model for sequential session recommendations

Amazon:    Item2Vec for product embeddings; graph-based CF;
           "Frequently bought together" = co-purchase matrix;
           Real-time session context (what's in cart right now)

TikTok:    Heavy emphasis on real-time signals (every swipe, pause, replay);
           Fast feedback loop; doubling down on exploration for new users;
           Very short model retraining cycles (hours, not days)
```

## Interview Answer Framework

1. **Types:** collaborative filtering (user/item based), content-based, matrix factorization, two-tower neural
2. **Pipeline:** retrieval (billions → 500) + ranking (500 → 10) — separation required for latency
3. **CF mechanics:** item-item cosine similarity; score = Σ(sim(liked, candidate) × rating)
4. **Matrix factorization:** R ≈ U × V^T; learn via SGD; implicit feedback via confidence weighting
5. **Two-tower:** user + item towers → embeddings → ANN search (FAISS/HNSW); pre-compute item vectors
6. **Cold start:** new user → popularity + onboarding signals; new item → content features + exploration budget
7. **Diversity:** MMR, category constraints, ε-greedy exploration
8. **Metrics:** precision@K, NDCG@K offline; CTR, watch time, retention online; always A/B test
9. **Real-time:** session window in Redis → immediate personalization without retraining
10. **Pitfalls:** popularity bias, position bias, data leakage, filter bubble, training/serving skew
