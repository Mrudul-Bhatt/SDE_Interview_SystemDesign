# Recommendation System (like Netflix or YouTube)

---

## What Are We Building?

You open Netflix. Before you can even think, the homepage shows you "Top picks for you" with
8 movies that all look interesting. You watch one. You come back next week — completely different,
equally interesting suggestions.

```
Netflix homepage rows:
  Because you watched "Breaking Bad" →  [crime thrillers]
  Trending now →                        [what's popular today]
  Top picks for you →                   [your personal "best" guesses]
  Continue watching →                   [stuff you started but didn't finish]
```

Behind this is a **recommendation system** — software that, given **you**, picks ~10 things
out of 100 million that you're most likely to enjoy.

**The core challenge:**
- 300 million users × 100 million items = 30 trillion possible pairs
- We can't score every pair, and we have 100ms to respond
- We don't know what you like until you tell us (the cold start problem)
- We want to surprise you sometimes, not just show what you already love (diversity)
- We get better only when we learn from what you actually click on (feedback loops)

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Personalized recommendations on the homepage, "More Like This", search ranking
2. Handle NEW users (no history) — cold start
3. Handle NEW items (just uploaded) — cold start
4. React to what you just clicked, in real time
5. Variety — not 10 of the same genre
6. Explanations — "because you watched X"
```

### Non-Functional (scale and reliability)
```
Scale:      300M users × 100M items
Latency:    Recommendations served in under 100ms
Retraining: Models retrained at least daily (some platforms hourly)
Freshness:  A video uploaded today should be recommendable today
Coverage:   Long-tail items get exposure too, not just hits
```

---

## The Core Question: How Do We Predict What You'll Like?

There are two foundational ideas. Modern systems combine them, but understanding each
separately is essential.

---

## Approach 1: Collaborative Filtering

**The idea:** people who agreed in the past will agree in the future. You don't need to
understand the movie — you just need to find people similar to you.

### User-Based: "People Like You Also Watched..."

```
Step 1: Find users most similar to you (the ones who rated the same movies similarly).
Step 2: Look at what THOSE users also liked.
Step 3: Recommend the items they loved that you haven't seen yet.
```

```
You rated:        Inception ★★★★★, Dark Knight ★★★★★, Memento ★★★★
Bob rated:        Inception ★★★★★, Dark Knight ★★★★★, Memento ★★★★★, Tenet ★★★★★
Carol rated:      Inception ★★★, Twilight ★★★★★

Bob is similar to you (overlapping high ratings on the same titles).
Carol is not (very different taste).
Bob also loved Tenet → recommend Tenet to you.
```

### Item-Based: "Because You Watched X, You Might Like Y"

```
Step 1: For each item you liked, find OTHER items that similar users also like.
Step 2: Score each candidate by how strongly it relates to your liked items.

Two items are "similar" if many users rated them similarly.
"Inception" and "Tenet" → both highly rated by mostly the same people → similar.
```

**Why item-based is preferred in practice:**
- Item-item similarities are stable — you can compute them weekly
- User-user similarities change every time someone rates a new movie
- The item-item similarity matrix is `items × items`, which is more manageable than `users × users`

### The Big Limitation: Cold Start

Collaborative filtering only works once you have a rating history. A brand new user with zero
interactions? No predictions. A brand new movie with no ratings? It never gets recommended,
so it never gets ratings — a chicken-and-egg problem.

---

## Approach 2: Content-Based Filtering

**The idea:** look at the actual properties of items you've liked, then find similar items.

```
You watched: Inception, The Dark Knight, Memento
These items share: director = Christopher Nolan, genre = thriller, tone = cerebral

Build your "taste vector": features extracted from movies you liked, weighted by your ratings.

For each candidate movie, compute: cosine_similarity(your_taste, movie_features)
Recommend the highest-scoring ones.
```

**Advantages:**
- Works for NEW items immediately (we have their features the moment they're uploaded)
- Explainable: "we recommended this because you like Nolan films"
- Doesn't depend on other users' data (privacy-friendly)

**Limitations:**
- Needs good item metadata (genres, tags, descriptions)
- **Filter bubble**: you only see "more of the same"; never get surprising hits

## Approach 3: Matrix Factorization (The Netflix Prize Winner)

This is where collaborative filtering gets mathematical and powerful.

### The Setup

```
Imagine a giant table:
  rows = 300M users
  columns = 100M items
  cells = ratings (1-5 stars)

But 99.9% of cells are EMPTY (most users haven't seen most items).

R ≈ U × V^T

  R = the giant ratings table (mostly unknown)
  U = a SHORT, dense user table:  300M users × 64 numbers each
  V = a SHORT, dense item table:  100M items × 64 numbers each
```

Each user is represented by a **64-dimensional vector** — a list of 64 numbers that capture
their "taste." Each item is also a 64-dimensional vector — its "characteristics" (though we
don't know what each dimension means; the model discovers them).

### How Do We Predict?

```
predicted_rating(user u, item i) = U[u] · V[i]   (dot product)

If u = [0.8, 0.1, ..., 0.5] and i = [0.7, 0.3, ..., 0.4]
predicted rating = 0.8×0.7 + 0.1×0.3 + ... + 0.5×0.4
```

If the dot product is high, the user likely likes the item. Done.

### How Do We Learn U and V?

```
Start with random numbers. Then for every known rating in the data:
  Compute: predicted = U[u] · V[i]
  Compute: error = actual_rating - predicted
  Adjust U[u] and V[i] to reduce that error (stochastic gradient descent)

Repeat over millions of ratings → U and V converge to vectors that predict
unseen ratings remarkably well.
```

Why this is powerful: the 64 numbers learn to capture latent traits we never explicitly named —
"is it cerebral?", "does it have car chases?", "is it nostalgic?" The model figures it out
purely from rating patterns.

### Implicit vs Explicit Feedback

```
Explicit:  Star ratings (1-5). Rare, biased (people only rate when they LOVED or HATED).
Implicit:  Clicks, watch time, purchases. Abundant but noisy.

For implicit feedback we use a slightly different formula:
  Positive signal: user watched the movie → 1
  Negative signal: user didn't watch → could mean "didn't like it" OR "never saw it"
  
  Use a confidence weighting: c(u,i) = 1 + α × watch_duration
  → strong signal if they watched a lot, weak if they watched a little
```

YouTube and Netflix mostly use implicit feedback because explicit ratings are too sparse.

## Approach 4: Two-Tower Neural Networks (The Modern Standard)

This is what powers YouTube, TikTok, and modern Netflix. It generalizes matrix factorization
by replacing "user_id" and "item_id" with deep neural networks that can incorporate features.

```
   User Tower                          Item Tower
   ──────────                          ──────────
   user_id              ──┐            ┌──    item_id
   age, country         ──┤            ├──    title, description (text embedding)
   device, time of day  ──┤            ├──    category, tags
   recent items watched ──┤            ├──    upload time
                          │            │
                          ▼            ▼
                  user_vector       item_vector
                  (256 numbers)     (256 numbers)

   Score = cosine_similarity(user_vector, item_vector)
```

### Why "Two Towers"?

Because we can **pre-compute all item vectors offline**. We don't need to recompute them when
a user requests recommendations.

```
Offline (runs daily):
  For every item in the catalog → run the item tower → store 100M item vectors
  Build a FAISS index (Facebook's ANN library) over these vectors

Online (when user requests):
  Run user tower with current context → get user vector (1ms)
  Ask FAISS: "give me the 500 item vectors closest to this user vector" (10ms)
  Done — 500 candidates ready for ranking
```

ANN search ("Approximate Nearest Neighbor") searches 100M items in 10ms by using clever index
structures like HNSW (Hierarchical Navigable Small World). It's not exact, but it's 99% accurate
and 1000× faster than exact search.

### Why Two-Tower Beats Plain Matrix Factorization

Matrix factorization only knows IDs. Two-tower can use:
- Item text (we can embed descriptions)
- Item images (we can embed thumbnails)
- User context (what device, what time of day)
- Recent session activity

This handles cold start much better — a brand new item still has text + image features for
the item tower to use.

## The Two-Stage Pipeline (How Real Systems Actually Work)

You can't run a fancy ranking model on 100M items in 100ms. So real systems split into stages.

```
Stage 1 — RETRIEVAL (Candidate Generation):
  Goal: 100M items → 500 candidates
  Time budget: 50ms
  Methods:
    - Two-tower ANN search → ~300 candidates
    - Item-item CF on user's recent likes → ~100 candidates
    - Trending right now → ~50 candidates
    - Subscribed channels → ~50 candidates
  Cast a wide net. Approximate is fine.

Stage 2 — RANKING:
  Goal: 500 candidates → top 10
  Time budget: 30ms
  Model: gradient boosted trees (XGBoost) or a small neural net
  Features (per candidate):
    - Score from retrieval
    - Item freshness (hours since upload)
    - Item engagement (CTR, average watch %)
    - User context (device, time of day, current session)
    - Did the user finish similar items recently?
    - Position bias correction
  Output: top 10–50, sorted by predicted engagement

Stage 3 — POST-PROCESSING:
  Diversity: at most 2 items per genre in the top 10
  Business rules: boost paid promos, filter blocked content
  A/B experiment: which model variant to serve this user
```

**Why split?** Because the retrieval models need to be fast (10ms across 100M items), so they
have to be simple. The ranking model can be expensive — but only over 500 items, which is
manageable. We get the best of both worlds.

## The Cold Start Problem

The eternal recommendation problem: what do you do when you have no data?

### New User (Just Signed Up)

```
You don't know anything about them. Options:

1. Popular fallback: show globally trending / top-rated items
2. Onboarding survey: "Pick 5 movies you like" (Netflix does this)
3. Demographic priors: use age/country/device → show what similar demographics like
4. Real-time learning: even the first 3 clicks are massive signal
   → adapt within the session (Redis-cached session items as a feature)
5. Cross-domain: if they signed up via Google, leverage other signals (with consent)
```

### New Item (Just Uploaded)

```
Nobody's rated it yet. If you only use CF, it stays invisible forever.

1. Content features: embed its title/description/thumbnail with the item tower
   → it can be discovered even with zero ratings
2. Exploration budget: reserve, say, 5% of recommendation slots for new items
3. Multi-armed bandits (Thompson Sampling):
   - Each new item has uncertain potential
   - Sample from a probability distribution; show the high-uncertainty winners
   - Quickly learn whether the item is good or bad
4. Creator boost: trusted creators' new uploads get a 48-hour boost
```

## Evaluation: How Do We Know If It's Good?

### Offline Metrics (Run on Historical Data)

```
Precision@10:  Of the 10 items we recommended, how many did the user actually engage with?
Recall@10:     Of all items the user engaged with, what fraction was in our top 10?
NDCG@10:       Like precision, but with extra credit for putting good items higher
Coverage:      What fraction of items show up in at least one user's recommendations?
Diversity:     Are the 10 items varied, or all from the same category?
```

### Online Metrics (Run A/B Tests)

```
CTR:           Click-through rate on recommendations
Watch time:    Total minutes consumed
Long clicks:   Users spending > 30 seconds after clicking (real engagement, not bounces)
Session depth: Items consumed per session
Retention:     Do users come back 7 days later?
```

**Crucial insight:** offline metrics often don't match online metrics. A model with higher NDCG
might actually decrease user engagement (maybe it recommends "safe" choices people don't want).
**Always validate offline wins with an A/B test before shipping.**

## Real-Time Signals (The Killer Feature)

Models retrain daily, but user behavior changes every second. How do recommendations react
in real time?

```
You open the app at 8 PM.
You click 3 horror movies in a row.
By the 4th recommendation, the system should be showing more horror.

How?

1. Session window: last 10 clicks per user, stored in Redis (30-minute TTL)
2. The ranking model takes "recent_session_items" as a feature
3. No model retraining needed — the features change; the model stays static
4. Effect: within seconds of clicking, your recommendations shift toward similar content
```

This is the difference between feeling like a personalized service vs. a robot that thinks
it knows you better than you do.

## The Filter Bubble Problem

```
You watch 3 cooking videos → the system recommends only cooking → you never see anything
new → eventually you get bored and leave.
```

This is the **filter bubble**, and it's the silent killer of recommendation engagement.

### How We Break Out

```
1. ε-greedy exploration:
   5% of the time, recommend something OUTSIDE the predicted top. See what happens.
   If they engage → update the model with the new signal.

2. Maximal Marginal Relevance (MMR):
   When picking the top 10, don't just pick the 10 highest scores.
   Pick high-scoring items that are DIFFERENT from items already picked.
   
   score = λ × relevance - (1 - λ) × similarity_to_already_picked

3. Category constraints:
   "At most 2 items per genre in the top 10" — a hard rule.

4. Serendipity injection:
   Reserve 1-2 slots for "items beloved by users with similar but slightly different
   tastes" — what people one step removed from you love.
```

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ Popularity bias
   → Top movies always win; small films never surface
   ✅ Normalize scores by item popularity; reserve slots for long-tail items

❌ Position bias in training
   → Items shown in position 1 get more clicks regardless of quality, so the model
     learns "position 1 = good" instead of "good item = good"
   ✅ Inverse propensity scoring; randomized experiments to estimate position effects

❌ Data leakage in training
   → Predict yesterday's clicks using today's data → unrealistic offline metrics
   ✅ Strict temporal split: train on events before time T, evaluate after T

❌ Filter bubble / echo chamber
   → Users get trapped in their existing taste; eventually disengage
   ✅ Exploration budget, MMR diversity, serendipity injection

❌ Cold start with no plan
   → New users get the same generic popular content for weeks
   ✅ Onboarding survey + heavy session-based learning in first 5 clicks

❌ Stale embeddings
   → Viral video from this morning isn't in yesterday's index
   ✅ Incremental embedding refresh every 1-4 hours; content-based fallback for new items

❌ Training/serving skew
   → Features used in training don't exist at serve time (e.g., used a feature that's
     only computable in batch, but serving needs it real-time)
   ✅ Same feature store for training and serving; log served features for next training

❌ Optimizing the proxy, not the goal
   → Improve CTR by 10% but watch time drops 5% → users click then bounce
   ✅ Optimize the business metric (watch time, retention) directly; A/B test always

❌ One-model-fits-all
   → Same model for homepage, search, "more like this" → none of them excellent
   ✅ Separate models per surface; share embedding backbone, fine-tune per surface
```

## Real Systems at a Glance

```
Netflix:   Two-stage (retrieval + ranking); separate models per surface (homepage,
           similar, continue watching); heavy causal inference for long-term impact

YouTube:   Two-tower retrieval; deep neural ranking; optimizes watch time, not CTR;
           position debiasing is critical

Spotify:   Word2Vec on playlists (treat playlists as "sentences" of songs); Discover
           Weekly = collaborative + audio features; BaRT for session sequences

Amazon:    Item2Vec product embeddings; graph-based CF; "frequently bought together"
           from co-purchase matrices; real-time cart context

TikTok:    Real-time signals dominate (every swipe, pause, replay matters); extremely
           fast feedback loops; model retrains hourly; heavy exploration for new users
```

## How to Answer This in an Interview (Step by Step)

```
1. State the goal:
   "Personalized top-N recommendations from 100M items, served in under 100ms,
   that get better as users interact and that solve cold start for new users and items."

2. Outline the approaches:
   "Collaborative filtering looks at user behavior patterns. Content-based uses item
   features. Matrix factorization learns latent vectors for users and items. Two-tower
   networks are the modern generalization — they handle features and scale via ANN search."

3. Describe the two-stage pipeline:
   "You can't run a complex model on 100M items. Stage 1 retrieves ~500 candidates
   with fast methods (two-tower ANN search, item-CF, trending). Stage 2 ranks those
   500 with a heavier model (XGBoost) using rich features. Stage 3 applies diversity
   and business rules."

4. Explain matrix factorization:
   "R ≈ U × V^T — each user and item gets a 64-dim vector. Train via SGD or ALS to
   minimize squared error on known ratings. For implicit feedback, use confidence
   weighting based on engagement intensity."

5. Explain two-tower:
   "User and item have separate networks producing 256-dim vectors. Trained via
   contrastive learning. Item vectors are pre-computed offline and indexed in FAISS.
   At serve time, just compute user vector and ANN-search."

6. Cold start strategies:
   "New users: popularity fallback + onboarding + heavy real-time session learning.
   New items: content embeddings + exploration budget + Thompson sampling bandits."

7. Diversity:
   "MMR for marginal relevance, category quotas in post-processing, and ε-greedy
   exploration to prevent filter bubbles."

8. Real-time signals:
   "Recent session items stored in Redis with TTL. Ranking model takes them as
   features. Effect: recommendations shift within seconds of user action without
   any model retraining."

9. Evaluation:
   "Offline: precision@K, NDCG@K. Online: CTR, watch time, retention. Always A/B
   test before shipping — offline gains often don't translate to online gains."

10. Cite key pitfalls:
    "Popularity bias, position bias in training, filter bubble, training/serving
    skew, and optimizing proxies instead of the business metric."
```
