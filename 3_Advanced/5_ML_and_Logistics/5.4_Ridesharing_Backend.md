# Ridesharing Backend (like Uber or Lyft)

---

## What Are We Building?

You open Uber, tap "Where to?", confirm pickup, and 30 seconds later a car icon is heading
toward you on the map.

```
You tap Where to?
   ↓
You enter destination and confirm
   ↓
The app finds a driver
   ↓
You see "Sam is on the way • 3 min"
   ↓
The car icon moves toward you in real time
   ↓
Sam arrives. You ride. Payment auto-charges. Done.
```

A LOT happens behind that 30 seconds:
- Millions of drivers' GPS positions are continuously updating
- The system has to find the BEST nearby driver (closest, highest-rated, etc.)
- It has to claim that driver atomically — two riders can't get the same Sam
- It has to figure out an accurate ETA for Sam's arrival
- It has to price the trip (and surge if demand is high)
- It has to track the trip through pickup, transit, dropoff, and payment

**The scale:**
- Uber: 5+ million trips per day
- 10+ million active drivers globally
- Each driver sends GPS every 4 seconds → **2.5 million location writes per second**
- The match has to happen in under 3 seconds

---

## Requirements (What the System Must Do)

### Functional (features)
```
1. Rider requests a ride with pickup and dropoff
2. System finds nearby available drivers
3. Match + dispatch: offer the trip to the best driver
4. Real-time driver location tracking (GPS every 4 seconds)
5. ETA estimation: when will the driver arrive? when will the trip end?
6. Surge pricing when demand exceeds supply
7. Fare calculation: base + distance + time + surge multiplier
8. Trip lifecycle states: REQUESTED → DRIVER_ASSIGNED → IN_PROGRESS → COMPLETED
9. Ratings (rider rates driver, driver rates rider)
10. Cancellation with policies (fees apply after a grace period)
```

### Non-Functional (scale and reliability)
```
Scale:        5M trips/day; 10M+ drivers; 2.5M GPS updates/sec
Latency:      Match a driver in under 3 seconds; location writes under 100ms
Uptime:       99.99% — downtime = direct revenue loss
Consistency:  Driver assignment must be STRONG (no double-booking)
              Location updates can be eventual
Durability:   Trip records survive forever (billing, disputes, analytics)
```

## Core Subsystems

```
1. Location Service     — ingests driver GPS, stores in geospatial index
2. Dispatch Service     — finds the best driver and offers them the trip
3. Trip Service         — runs the state machine
4. Pricing Service      — base fare + surge calculation
5. ETA Service          — predicts arrival and trip duration
6. Payment Service      — charges the rider after completion
7. Notification Service — push notifications to both apps
```

We'll dive into the three hardest ones: **location tracking**, **dispatch matching**, and
**surge pricing**.

## Problem #1: Tracking 10 Million Drivers in Real Time

Every driver's app sends a GPS update every 4 seconds while online.

```
10M drivers × (1 update / 4 sec) = 2.5M GPS writes per second
```

That's an extreme write rate. Postgres or Cassandra would struggle. We need something built
for this: **Redis with a geospatial index**.

### Redis Geo Operations

```
Driver sends GPS update:
  GEOADD drivers:available:NYC -74.0 40.7 driver_42

  This stores driver_42 at longitude -74.0, latitude 40.7 in the NYC index.

Finding nearby drivers when a rider requests:
  GEORADIUS drivers:available:NYC -73.98 40.75 2 km ASC COUNT 20

  → Returns up to 20 drivers within 2 km of (lng=-73.98, lat=40.75),
    sorted by distance.
```

Redis can do this in under 5ms even with 10 million drivers indexed, because internally it
uses **geohashes** — a clever encoding that turns 2D coordinates into a 1D sortable string.

### Geohashes Explained Briefly

A geohash converts a lat/lng pair into a short string like `"dr5ru9"`. The cool property:
**locations with similar prefixes are close on the map**.

```
"dr5ru9" → ~150m × 150m tile somewhere in Manhattan
"dr5ru" → ~1.2km × 0.6km tile (one character shorter, bigger area)
"dr5r" → ~5km × 5km neighborhood
```

So "find all drivers in a 1km area" becomes "find all geohashes that share my 6-character prefix"
— a fast string range scan.

### Uber's H3: Better Geohashes Using Hexagons

Square cells have a problem: at cell boundaries, you have to query 9 cells (yours + 8 neighbors)
to avoid missing drivers right on the edge.

Uber invented **H3** — a hexagonal grid covering the entire Earth. Hexagons have only 6 neighbors,
all at equal distance, with no boundary discontinuities. Better for radius queries.

### Two Storage Paths for Locations

```
Real-time (Redis): used for "find nearby drivers" — fast, ephemeral
History (Cassandra): every GPS point logged via Kafka → for trip replay, analytics, ML
```

Redis answers "where is everyone right now?" Cassandra answers "where did Sam go on Tuesday?"

### TTL: Auto-Removing Dead Drivers

A driver's app might crash, lose signal, or drive into a tunnel. We don't want stale ghosts
in the index.

```
Every GPS update refreshes a TTL on the driver's entry.
If no update arrives in 30 seconds → entry expires → driver removed from index.
```

This is **eventual consistency for free** — the index self-cleans without explicit deletion.

## Problem #2: Matching a Driver in Under 3 Seconds

The dispatch algorithm is the heart of the system.

### Step 1: Find Candidates

```
GEORADIUS drivers:available:NYC <rider_lng> <rider_lat> 3 km ASC COUNT 20

Returns the 20 closest available drivers within 3km of the rider's pickup point.
```

### Step 2: Rank Them

Distance isn't everything. We score each candidate on multiple factors:

```
Score = ETA + driver_rating_bonus + acceptance_rate_bonus + ride_type_match

ETA dominates because riders care most about pickup time, but a 4.9-star driver
30 seconds farther away might still win over a 4.2-star driver right there.
```

### Step 3: Sequential Dispatch (NOT Broadcast)

Here's where it gets interesting. There are two ways to dispatch a trip offer:

```
BROADCAST: send the offer to all 20 drivers at once. First one to accept wins.
  ❌ Race conditions (two accept at the same instant)
  ❌ Spammy for drivers (they get pings constantly)
  ❌ Phantom trips if multiple drivers tap accept

SEQUENTIAL: offer to driver #1, wait 15 seconds. If decline/timeout → offer to #2.
  ✅ Clean driver experience (one offer at a time)
  ✅ Easy to atomically "claim" a driver during the 15s window
  ✅ Clear accountability — only one driver is "engaged" with this trip
```

Uber uses sequential dispatch.

### Step 4: Atomic Claim with Redis SET NX

The critical question: how do we ensure two simultaneous trip requests don't BOTH get driver Sam?

```
Redis command: SET driver:42:state "PENDING_OFFER:trip_abc" NX EX 15

  NX  = "only set if the key DOESN'T already exist"
  EX 15 = "auto-expire after 15 seconds"

Two riders' dispatch jobs both try this command simultaneously:
  → ONLY ONE wins (the SET returns OK)
  → The other gets back "nil" (failed, already claimed)
  → The loser moves on to driver #2 in its candidate list
```

This is the same trick a database uses for row locking, but Redis does it in 0.1ms.

If the driver accepts within 15 seconds, the state transitions to `ON_TRIP:trip_abc`. If they
decline or the 15s expires, the lock auto-releases — they're available again for someone else.

### Step 5: Retry With Wider Radius

If all 20 nearby drivers decline, expand the search:

```
First attempt: 3 km radius, 20 drivers
If exhausted: expand to 5 km
If exhausted: expand to 8 km
If still no driver: surge price upward (attract drivers) + retry
After 60 seconds total: give up → "no drivers available" to rider
```

## The Trip State Machine

Once a driver is assigned, the trip moves through a strict sequence of states.

```
   REQUESTED
       │
       ├─ accept ────────────▶ DRIVER_ASSIGNED
       ├─ cancel by rider ───▶ CANCELLED
       └─ no driver ─────────▶ FAILED
                                       │
                                       ├─ "Picked up" tapped ──▶ IN_PROGRESS
                                       └─ cancel ──────────────▶ CANCELLED (maybe fee)
                                                                       │
                                                                       ├─ "End trip" tapped ──▶ COMPLETED
                                                                       └─ incident reported ──▶ DISPUTED
```

Each transition has rules:
- `IN_PROGRESS` can only happen when the driver's GPS is within 100 m of the pickup point
  (anti-fraud — prevents drivers from marking pickup remotely)
- `COMPLETED` can only happen when the driver's GPS is at the dropoff
- Cancellation policy enforced at each cancel point

### Cancellation Policy

```
Rider cancels BEFORE driver assigned:    Free
Rider cancels WITHIN 2 min after assigned: Free
Rider cancels AFTER 2 min, driver en route: Cancellation fee
Driver cancels: Counts against their acceptance rate
```

The 2-minute grace period covers genuine mistakes. After that, the driver has invested
time/fuel, so a fee compensates them.

## Problem #3: Surge Pricing

Friday night, 11 PM, Times Square. Concert just ended. 500 people want a ride. There are 15
drivers nearby. What happens?

Without surge: 15 lucky people get rides; 485 stare at "searching for drivers" forever.

**With surge:** the price multiplies. This does two things:
- **Discourages low-priority demand** (some people decide to wait/walk instead)
- **Attracts more drivers** to the area (the surge zone shows up on the driver heatmap)

### The Formula

```
For each geohash zone (~5km × 5km):

  available_drivers = drivers in zone with status AVAILABLE (last 5 min)
  pending_requests  = trip requests in zone (last 5 min)

  ratio = pending_requests / max(available_drivers, 1)

  surge_multiplier = clamp(ratio, min=1.0, max=8.0)
```

Example:
```
Times Square, Friday 11 PM
Available drivers: 5
Pending requests:  40
ratio = 40 / 5 = 8
surge_multiplier = min(8, 8) = 8.0×

A normal $10 ride costs $80.
```

8× is the regulatory cap in most markets. The base fare gets multiplied, but distance and
time charges stay the same.

### Granularity and Refresh

```
Zone size:    Geohash-5 (~5km × 5km)
Refresh:      Every 1–5 minutes
Smoothing:    Exponential moving average (avoids flapping between 2× and 3× every minute)
```

The surge map shown to drivers updates in near-real-time so they can drive into hot zones.
The rider sees the multiplier BEFORE confirming the ride ("2.1× pricing in effect") — they
opt in knowingly.

### Surge Manipulation (and the Fix)

```
Naive attack: drivers in an area coordinate to go offline together → artificial surge appears
              → they come back online and grab the surged trips.

Detection: flag patterns where many drivers go offline in one zone right before a surge spike.
           Compare against normal patterns; reduce surge if manipulation suspected.
```

## ETA Estimation

Two ETAs are shown:
1. **Driver arrival ETA** (how long until Sam reaches the rider)
2. **Trip ETA** (how long the trip will take from pickup to dropoff)

### A Simple Model

```
straight_line_distance = haversine(point_a, point_b)   // km, "as the crow flies"
road_distance = straight_line_distance × 1.3            // roads aren't straight

eta_minutes = road_distance / avg_speed × 60

avg_speed depends on:
  - Time of day (rush hour vs late night)
  - Day of week
  - Type of road (urban vs highway)
  - Current real-time traffic conditions
```

### Production Reality

Uber's actual ETA system is a machine-learned model:
- Inputs: H3 cell of origin and destination, time of day, day of week, real-time traffic feed
- Trained on millions of historical trips
- Updated every few minutes with fresh data
- Different models for different cities (traffic patterns differ wildly)

### Live ETA Updates

The ETA shown to the rider should update if traffic changes en route. Every 30 seconds during
"driver en route," the system re-estimates and pushes the new ETA via WebSocket. If the ETA
jumps by more than 3 minutes, the rider gets a proactive notification.

## Data Model

```
Trip (Cassandra):
  trip_id, rider_id, driver_id, status
  pickup_lat, pickup_lng, dropoff_lat, dropoff_lng
  requested_at, assigned_at, started_at, completed_at
  distance_km, duration_min
  base_fare, surge_mult, final_fare
  driver_rating, rider_rating

Driver Profile (MySQL):
  driver_id, name, vehicle_info
  status (OFFLINE / AVAILABLE / ON_TRIP)
  avg_rating (last 500 trips)
  acceptance_rate (last 30 days)

Driver Real-Time State (Redis only — ephemeral):
  GEOADD drivers:available:NYC <lng> <lat> driver_42
  driver:42:state = "AVAILABLE" | "PENDING:trip_abc" | "ON_TRIP:trip_abc"
  TTL: 30 seconds (auto-cleanup if GPS stops)
```

**Why Redis for live state and Cassandra for history?**
- Redis: extreme throughput, sub-millisecond reads, but volatile
- Cassandra: high-throughput durable writes, but slower per-key access
- Use the right tool for each job: Redis = "now"; Cassandra = "always"

## How It All Fits Together

```
─── A rider books a trip ───────────────────────────────────────

Rider taps "Confirm Ride":
       │
       ▼
   Trip Service
       ├─ Pre-authorize rider's payment method
       ├─ Create trip in DB: status = REQUESTED
       └─ Hand to Dispatch Service

   Dispatch Service:
       ├─ GEORADIUS to find 20 nearest available drivers
       ├─ Rank by ETA + rating + acceptance rate
       └─ Try drivers sequentially:
              For each candidate:
                 SET driver:N:state "PENDING:trip_abc" NX EX 15
                 If success → push offer to that driver's app via WebSocket
                 Wait up to 15s for accept/decline
                 On accept → move to DRIVER_ASSIGNED, break
                 On decline/timeout → continue to next candidate

─── During the trip ────────────────────────────────────────────

Driver app every 4s:
       └─ GEOADD location update → also broadcast to rider's app

Rider app:
       └─ Receives live ETA updates every 30s

When driver arrives at pickup (GPS within 100m):
       └─ Driver taps "Picked up" → status: IN_PROGRESS

When driver reaches dropoff:
       └─ Driver taps "End trip" → status: COMPLETED
              ├─ Calculate final fare = base + distance + time + surge
              ├─ Charge pre-authorized payment
              ├─ Pay driver (minus platform fee) at end of day
              └─ Request ratings from both
```

## Common Mistakes to Avoid (Interview Pitfalls)

```
❌ No atomic driver claim
   → Two riders match the same driver simultaneously → phantom trips
   ✅ Redis SET NX EX with a 15-second TTL — only one winner per claim

❌ Broadcast dispatch (offer to all nearby drivers)
   → Race conditions, driver spam, accept-then-cancel after seeing trip details
   ✅ Sequential dispatch — one offer at a time, 15s window each

❌ Storing driver locations in a SQL database
   → 2.5M writes/sec swamps any normal DB
   ✅ Redis Geo for live state; Kafka → Cassandra for the immutable history log

❌ No TTL on driver entries
   → Crashed apps leave ghost drivers in the index → riders matched to phantom cars
   ✅ TTL on every GEO entry; refreshed by each GPS update; expires within 30s

❌ Allowing trip start without GPS proximity check
   → Drivers tap "Picked up" without actually being there → mileage fraud
   ✅ Require driver GPS within 100m of pickup before status transition

❌ Naive surge price flapping
   → Surge jumps 1× → 4× → 1× → 4× every minute → riders furious
   ✅ Exponential moving average; gradual ramp-up/down; smoothing

❌ No protection against surge manipulation
   → Drivers go offline together to fake demand spike
   ✅ Pattern detection on coordinated offline events; reduce surge if suspicious

❌ Payment failure handled after trip starts
   → Trip completes but card declines → free ride for fraudster
   ✅ Pre-authorize the card BEFORE assigning a driver

❌ Driver state inconsistency between Redis and DB
   → Redis says AVAILABLE, DB says ON_TRIP → double-bookings
   ✅ Redis is authoritative for real-time state; DB updated async; reconcile on reconnect

❌ No live ETA refresh
   → Rider sees "3 min" but driver is stuck in traffic for 15 → angry rider
   ✅ Push live ETA every 30s; proactive alert if ETA grows by > 3 min
```

## Real Systems at a Glance

```
Uber:    H3 hexagonal spatial index; Ringpop for consistent hashing of driver shards;
         Cadence (workflow engine) for trip lifecycle; Kafka for location events;
         Cassandra for trip storage; DISCO dispatch system; ETAS ML model

Lyft:    Heavy AWS stack; gRPC between services; real-time matching across regional clusters

DiDi:    Largest fleet (China); blends batch and real-time matching for very dense cities;
         food delivery shares much of the same infrastructure

DoorDash: Same matching primitives, but with restaurants as fixed origins instead of moving drivers
```

## How to Answer This in an Interview (Step by Step)

```
1. State the scale:
   "2.5 million GPS updates per second, 5 million trips per day, match in under 3 seconds.
   Two riders must NEVER get the same driver. Trips must persist forever for billing."

2. Walk through location tracking:
   "Driver apps push GPS every 4 seconds via WebSocket. Writes go to Redis Geo (real-time
   index) AND Kafka (durable log). Cassandra absorbs the Kafka stream for history.
   Redis entries have a 30-second TTL so dead drivers self-clean."

3. Explain geospatial indexing:
   "Geohash or H3 — encode lat/lng into a short string so similar locations are near in
   sort order. GEORADIUS finds nearby drivers in under 5ms even with 10M indexed."

4. Walk through dispatch:
   "GEORADIUS for 20 candidates. Rank by ETA + rating + acceptance rate. SEQUENTIAL
   offers — try driver #1, wait 15s, fall through. Atomic claim via Redis SET NX so two
   riders can't get the same driver. Expand radius if all decline."

5. Trip state machine:
   "REQUESTED → DRIVER_ASSIGNED → IN_PROGRESS → COMPLETED. Transitions guarded by GPS
   proximity (can't start trip unless driver is near pickup). Cancellation rules per state."

6. Surge pricing:
   "Per geohash zone, recompute every 1-5 min: ratio = pending_requests / available_drivers.
   Multiply that into the fare, capped at 8×. Use exponential moving average to avoid
   flapping. Rider sees multiplier before confirming."

7. ETA:
   "Haversine distance × road factor / expected speed by H3 cell × time of day, adjusted
   by real-time traffic. Live updates every 30s during en-route. Proactive notifications
   if ETA grows significantly."

8. Payment:
   "Pre-authorize card BEFORE assigning driver. Final charge on COMPLETED. If charge fails,
   trip enters a collections workflow."

9. Storage rationale:
   "Redis: real-time ephemeral state (locations, driver locks, surge). Cassandra: durable
   trip history (high write throughput, time-clustered). MySQL: user profiles. Kafka:
   event stream backbone."

10. Cite the main pitfalls:
    "Atomic driver claim via SET NX. TTL on geo entries to clean dead drivers. GPS-gated
    state transitions to prevent fraud. Smoothing on surge. Pre-auth payment before dispatch."
```
