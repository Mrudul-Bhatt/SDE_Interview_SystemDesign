# Ridesharing Backend (Uber / Lyft)

---

## Problem Statement

Design the backend for a ridesharing platform — like Uber or Lyft — that matches riders with nearby drivers in real time, tracks vehicles continuously, prices trips dynamically, and handles the full trip lifecycle from request to payment.

```
Use cases:
  Uber        On-demand rides, UberPool, UberEats delivery
  Lyft        Ridesharing, scooters, Lyft Line (shared rides)
  Ola         Indian market, auto-rickshaws, bikes, cabs
  DiDi        China, dominant rideshare + food delivery
  DoorDash    Delivery logistics (same matching / routing challenges)
```

---

## Requirements

### Functional
```
1. Rider requests a ride (pickup + dropoff location)
2. System finds nearby available drivers
3. Match + dispatch: offer trip to best driver; handle accept/decline
4. Real-time driver location tracking (update every 4 seconds)
5. ETA estimation for driver arrival and trip duration
6. Dynamic surge pricing based on supply/demand imbalance
7. Fare calculation: base + distance + time + surge multiplier
8. Trip lifecycle: REQUESTED → DRIVER_ASSIGNED → EN_ROUTE → IN_PROGRESS → COMPLETED
9. Rider and driver ratings after trip
10. Cancellation with policies
```

### Non-Functional
```
Scale:       Uber: 5M trips/day; 10M+ active drivers globally
             GPS updates: 10M drivers × 1 update/4s = 2.5M writes/sec
             Matching latency: driver found < 3 seconds
             Location update write latency: < 100ms
Availability: 99.99% — downtime directly loses revenue
Consistency:  Driver state must be consistent — cannot be assigned to two trips
              (strong consistency for driver assignment, eventual for location)
Durability:   Trip records persist for billing, disputes, and analytics
```

## Core Subsystems

```
1. Location Service    — ingests driver GPS; geospatial index
2. Dispatch Service    — finds + matches driver to rider
3. Trip Service        — manages trip state machine
4. Pricing Service     — surge calculation, fare computation
5. ETA Service         — estimates arrival and trip duration
6. Payment Service     — processes fare after completion
7. Notification Service — pushes updates to driver/rider apps
```

## Driver Location Tracking

```
Problem: 10M drivers each sending GPS every 4 seconds
         = 2.5M location writes per second

Each GPS update:
  { driver_id, lat, lng, heading, speed, timestamp }

Storage:
  Redis Geospatial (GEOADD):
    Key: "drivers:available:{city}"
    Members: driver_id
    Score: encoded lat/lng (Redis uses geohash internally)
    Operations: GEOADD (write), GEORADIUS (range query)

  Write path: Driver App → WebSocket → Location Service → Redis (GEOADD)
  Also write to Kafka → Cassandra (historical, for analytics + replay)

Geospatial indexing:
  Geohash: encode lat/lng to a short alphanumeric string
    Length 6 → ~1.2km × 0.6km precision
    Length 7 → ~153m × 153m precision
    Nearby search: query current cell + 8 neighbors (handles boundary effects)

  H3 (Uber's own): hexagonal grid, no boundary discontinuities
    Uniform cell size across globe; better for radius queries
    Resolution 9 → ~105m diameter hex cells

Redis GEORADIUS performance:
  Finds all drivers within N km: O(N + log M) where M = indexed members
  10M drivers: < 5ms for radius=2km query
```

## Matching / Dispatch Algorithm

```
Goal: assign the best driver to a rider's request in < 3 seconds.

Step 1: Find candidate drivers
  GEORADIUS "drivers:available:city" lat lng 3km ASC COUNT 20
  → Up to 20 nearest available drivers, sorted by distance

Step 2: Rank candidates
  Score = f(ETA, driver_rating, acceptance_rate, ride_type_preference)
  ETA is most important signal (rider wants fast pickup)

Step 3: Sequential dispatch (Uber's approach)
  Offer trip to #1 ranked driver
  Wait up to 15 seconds for accept/decline
  If no response / decline → try #2, then #3, etc.
  If all decline: expand radius and retry

Why sequential, not broadcast?
  Broadcast (send to all, first accept wins):
    - Creates race conditions and phantom trips
    - Drivers cancel after accepting if they don't like the trip
    - Creates noise/spam for drivers
  Sequential:
    - Clean driver experience (one offer at a time)
    - Atomically marks driver as "pending" during offer window
    - Easier to reason about consistency

Atomic driver assignment (Redis):
  SET driver:{id}:state "PENDING_OFFER:{trip_id}" NX EX 15
  NX = only set if key doesn't exist (atomic claim)
  EX 15 = auto-expire after 15s (driver timeout)
  Returns nil if already claimed → try next driver

Dispatch timeout (entire attempt):
  Max 60 seconds to find a driver (adjustable)
  After timeout → notify rider "no drivers available"
  Optionally retry with larger radius or different ride type
```

## Trip State Machine

```
States:
  REQUESTED       — rider has submitted request; looking for driver
  DRIVER_ASSIGNED — driver accepted; heading to pickup
  DRIVER_EN_ROUTE — driver confirmed on the way (optional granularity)
  IN_PROGRESS     — driver picked up rider; trip underway
  COMPLETED       — rider dropped off; payment processing
  CANCELLED       — cancelled by rider or driver before pickup
  FAILED          — no driver found, or payment failed

Transitions:
  REQUESTED      → DRIVER_ASSIGNED  : driver accepts offer
  REQUESTED      → CANCELLED        : rider cancels before assignment
  REQUESTED      → FAILED           : no driver found within timeout
  DRIVER_ASSIGNED → IN_PROGRESS     : driver taps "Pick up" (GPS at pickup)
  DRIVER_ASSIGNED → CANCELLED       : driver cancels (penalty applies)
  DRIVER_ASSIGNED → CANCELLED       : rider cancels (may incur fee if driver en route)
  IN_PROGRESS    → COMPLETED        : driver taps "End trip" (GPS at dropoff)
  IN_PROGRESS    → DISPUTED         : rider/driver reports incident

Cancellation policy:
  Rider cancels before driver assignment: free
  Rider cancels after driver assigned but before pickup:
    - If within 2 minutes: free
    - If after 2 minutes and driver > 50% of the way: cancellation fee
  Driver cancellation: counts against acceptance rate
```

## Surge Pricing

```
Problem: demand spikes (concerts, rain, rush hour) → not enough drivers
Solution: raise price to attract more drivers, reduce demand

Formula:
  supply_score   = available_drivers in zone (last 5 min)
  demand_score   = pending requests in zone (last 5 min)
  ratio          = demand_score / max(supply_score, 1)

  surge_multiplier = max(1.0, min(ratio, MAX_SURGE))
  MAX_SURGE: Uber caps at ~8× in most markets (capped by regulation)

  Example:
    Zone: Times Square, Friday 11pm
    Drivers available: 5
    Pending requests: 40
    ratio = 40/5 = 8
    surge_multiplier = min(8, 8) = 8× → fare is 8× base

Granularity:
  Surge zones: geohash prefix length 5 (~5km × 5km)
  Recomputed every 1–5 minutes
  Visualized as heatmap in driver app (attracts drivers to surge zones)

Surge communication:
  Rider sees surge warning before confirming: "2.1× pricing in effect"
  Driver sees surge heatmap overlay in their app
  Surge disappears when supply balances demand

Smoothing:
  Use exponential moving average of supply/demand to avoid flapping
  Gradual ramp-up/down rather than step changes
```

## ETA Estimation

```
Two ETAs to display:
  1. Driver arrival ETA (to pickup): distance to pickup / expected speed
  2. Trip ETA (pickup to dropoff): route distance / expected speed

Inputs:
  - Haversine distance (straight-line, adjusted by road factor ~1.3×)
  - Historical average speed for this route × time of day × day of week
  - Real-time traffic feed (Google Maps / Bing Maps API)
  - Current driver speed + heading (for fine-grained prediction)

Simple model:
  eta_minutes = (distance_km * road_factor) / avg_speed_kmh * 60
  road_factor ≈ 1.3 (straight-line → actual road distance)
  avg_speed_kmh varies: city = 25, highway = 80, rush hour = 15

Production:
  Uber H3 ETAs: machine-learned per H3 cell × time-of-day
  Graph routing: Dijkstra/A* on road network graph
  Online learning: update speed estimates every few minutes from trip data
```

## Data Model

```
Trip (Cassandra):
  trip_id         UUID
  rider_id        UUID
  driver_id       UUID
  status          ENUM
  pickup_lat/lng
  dropoff_lat/lng
  requested_at    TIMESTAMP
  assigned_at     TIMESTAMP
  started_at      TIMESTAMP
  completed_at    TIMESTAMP
  distance_km     DOUBLE
  duration_min    INT
  base_fare       DECIMAL
  surge_mult      DOUBLE
  final_fare      DECIMAL
  driver_rating   INT (1-5, after completion)
  rider_rating    INT (1-5)

Driver (MySQL/Cassandra):
  driver_id
  name, vehicle_info
  status          ENUM (OFFLINE, AVAILABLE, ON_TRIP)
  rating          DOUBLE (avg of last 500 ratings)
  acceptance_rate DOUBLE (accepted / offered, last 30 days)
  current_lat/lng (from Redis, not DB)

Driver location (Redis only — ephemeral):
  GEOADD drivers:available:{city} lng lat driver_id
  Key: driver:{id}:state → "AVAILABLE" | "PENDING:{trip_id}" | "ON_TRIP:{trip_id}"
  TTL: 30s (driver removed from index if stops sending GPS)
```

## Pitfalls

```
1. Race condition: two riders get same driver
   Problem: two dispatch jobs both query and assign the same driver
   Fix: atomic Redis SET NX to "claim" driver before assignment
        only one request wins the SET NX; other retries with next driver

2. Driver goes offline mid-dispatch
   Problem: driver selected but GPS stops (app crash, tunnel)
   Fix: Redis TTL on driver's geo entry (30s); stale drivers auto-removed
        Dispatch checks driver is still available after SET NX succeeds

3. GPS spoofing / phantom drivers
   Problem: driver app reports fake location to get better trips
   Fix: cross-validate with cell tower data; flag anomalous position jumps;
        trip cannot start unless driver GPS is within 100m of pickup

4. Surge manipulation by drivers
   Problem: drivers go offline to create artificial surge, then come back
   Fix: detect and flag surge manipulation patterns;
        compare driver offline duration against nearby surge events;
        reduce surge zone if offline rate > threshold

5. Payment failure after trip completes
   Problem: trip completed, fare calculated, payment fails
   Fix: rider has pre-authorized card; trip cannot start without valid payment;
        for failure: retry payment, then collections queue

6. ETA wildly wrong due to traffic
   Problem: rider shown 5min ETA; driver takes 20min due to accident
   Fix: real-time traffic feed; push live ETA updates every 30s during en-route;
        proactive notification if ETA increases > 3 min

7. No drivers available → rider frustration
   Problem: large radius search still returns no drivers
   Fix: surge pricing to attract drivers; show rider "X drivers near you"
        to set expectations; offer scheduled rides

8. Split-brain driver state (Redis + DB out of sync)
   Problem: driver state in Redis says AVAILABLE but DB says ON_TRIP
   Fix: Redis is authoritative for real-time state;
        DB is updated asynchronously; resolve discrepancy on reconnect
```

## Real-World Systems

```
Uber:     H3 hexagonal spatial index; Ringpop for consistent hashing;
          Cadence (workflow orchestration) for trip lifecycle;
          Kafka for location events; Cassandra for trips;
          DISCO dispatch system; ETAS ML model for ETA

Lyft:     Similar stack; heavy use of AWS; gRPC between services;
          Real-time matching via location service cluster

DiDi:     Largest rideshare fleet; batch + real-time matching;
          Orders-of-magnitude more complex in dense Chinese cities

General:  All use some form of geohash/H3 for spatial indexing;
          Redis for real-time ephemeral state (driver locations);
          Kafka for high-throughput event streaming;
          Cassandra/DynamoDB for durable trip storage
```

## Interview Answer Framework

1. **Scale:** 2.5M GPS writes/sec → Redis GEOADD; 5M trips/day → Cassandra
2. **Location:** drivers send GPS every 4s → Location Service → Redis Geo index; TTL auto-removes stale
3. **Matching:** GEORADIUS for candidates → rank by ETA → sequential dispatch with 15s timeout
4. **Atomic assignment:** Redis SET NX to prevent two riders getting same driver
5. **Trip state machine:** REQUESTED → DRIVER_ASSIGNED → IN_PROGRESS → COMPLETED
6. **Surge pricing:** demand/supply ratio per geohash zone, recomputed every 5 min, capped at 8×
7. **ETA:** Haversine × road factor / avg speed × traffic; ML model per H3 cell × time of day
8. **Fare:** base + per_km + per_min + surge; pre-authorized at request
9. **Cancellation:** free before assignment; fee if driver en route > 2 min
10. **Pitfalls:** race condition (SET NX), GPS spoofing, surge manipulation, stale driver state
