## 2. Driver Location Update and Indexing

```mermaid
sequenceDiagram
    participant D  as Driver App
    participant LS as Location Service
    participant R  as Redis (Geo Index)
    participant K  as Kafka

    loop Every 4 seconds
        D->>+LS: GPS {driver_id, lat, lng, speed, heading}

        LS->>R: GEOADD drivers:available:nyc lng lat driver_id
        Note over R: O(log N) insert\nAuto-removes stale drivers\n(TTL=30s refreshed each update)

        LS->>K: Publish location-event {driver_id, lat, lng, ts}
        LS-->>-D: 200 OK

        Note over K: Async consumers:\n- Cassandra (historical path)\n- Analytics (surge compute)\n- ETA model retraining
    end

    Note over LS,R: Driver goes offline → no more GPS updates\n→ Redis TTL expires after 30s\n→ Driver removed from available index automatically
```

## 3. Trip Dispatch — Sequential Offer Flow

```mermaid
sequenceDiagram
    participant R   as Rider
    participant TS  as Trip Service
    participant DS  as Dispatch Service
    participant RD  as Redis (Geo + State)
    participant D1  as Driver #1 (best match)
    participant D2  as Driver #2 (fallback)

    R->>+TS: POST /ride/request {pickup, dropoff}
    TS->>TS: Create trip REQUESTED, calculate surge

    TS->>+DS: Dispatch(trip)
    DS->>+RD: GEORADIUS drivers:available:nyc radius=3km COUNT=20
    RD-->>-DS: [driver1, driver2, driver3, ...] sorted by distance

    DS->>DS: Rank by ETA, rating, acceptance_rate
    DS->>RD: SET driver:d1:state "PENDING:trip123" NX EX 15
    Note over RD: NX = only set if not exists\nEX 15 = auto-expire in 15s

    DS->>+D1: Push offer: trip details, pickup, fare estimate
    alt Driver #1 accepts within 15s
        D1-->>-DS: ACCEPT trip123
        DS->>RD: SET driver:d1:state "ON_TRIP:trip123"
        DS-->>-TS: assigned driver=d1
        TS-->>R: Trip assigned, ETA=4 min
    else Driver #1 declines or times out
        D1-->>DS: DECLINE (or timeout)
        DS->>RD: DEL driver:d1:state (release claim)
        DS->>RD: SET driver:d2:state "PENDING:trip123" NX EX 15
        DS->>+D2: Push offer
        D2-->>-DS: ACCEPT trip123
        DS-->>TS: assigned driver=d2
        TS-->>R: Trip assigned (fallback driver), ETA=6 min
    end
```

## 4. Trip State Machine

```mermaid
stateDiagram-v2
    [*] --> REQUESTED : Rider submits request

    REQUESTED --> DRIVER_ASSIGNED : Driver accepts offer
    REQUESTED --> CANCELLED       : Rider cancels (free)
    REQUESTED --> FAILED          : No driver found in 60s

    DRIVER_ASSIGNED --> IN_PROGRESS  : Driver arrives, taps "Pick Up"
    DRIVER_ASSIGNED --> CANCELLED    : Rider cancels (fee if > 2min)\nor Driver cancels (penalty)

    IN_PROGRESS --> COMPLETED  : Driver taps "End Trip" at dropoff
    IN_PROGRESS --> DISPUTED   : Incident reported

    COMPLETED --> [*]
    FAILED    --> [*]
    CANCELLED --> [*]

    note1: DRIVER_ASSIGNED → IN_PROGRESS\nrequires GPS within 100m of pickup
    note2: COMPLETED triggers:\n- Fare charge\n- Rating prompt\n- Driver back to AVAILABLE
```

## 5. Surge Pricing — Supply/Demand Computation

```mermaid
flowchart TB
    subgraph Input["Inputs (per 5-min window, per geohash zone)"]
        SUP["Available drivers\nin zone = supply_count"]
        DEM["Pending ride requests\nin zone = demand_count"]
    end

    subgraph Compute["Surge Computation"]
        RATIO["ratio = demand_count / max(supply_count, 1)"]
        SURGE["surge_multiplier = clamp(ratio, 1.0, 8.0)"]
        SMOOTH["EMA smoothing:\nnew_surge = 0.7 × prev_surge + 0.3 × raw_surge\n(prevents flapping)"]
    end

    subgraph Output["Output"]
        RIDER["Rider sees:\n2.1× surge warning\nbefore confirming ride"]
        DRIVER["Driver sees:\nsurge heatmap in app\n→ attracted to high-surge zones"]
        FARE["Final fare:\nbase × distance × time × surge_multiplier"]
    end

    SUP & DEM --> RATIO --> SURGE --> SMOOTH
    SMOOTH --> RIDER & DRIVER & FARE

    example["Example:\nTimes Square 11pm Friday\nDrivers=5, Requests=40\nratio=8 → surge=8.0×"]
    style example fill:#fff9c4
```

## 6. Fare Calculation

```mermaid
flowchart LR
    subgraph Components["Fare Components"]
        BF["Base Fare\n$2.00\n(per trip, flat)"]
        DF["Distance Fare\n$1.25 / km\n× trip_distance_km"]
        TF["Time Fare\n$0.20 / min\n× trip_duration_min"]
        SF["Surge Multiplier\n1.0× – 8.0×"]
    end

    subgraph Calc["Calculation"]
        RAW["raw_fare = base + distance_fare + time_fare"]
        FINAL["final_fare = raw_fare × surge_multiplier"]
        MIN["final_fare = max(final_fare, minimum_fare=$5)"]
    end

    subgraph Example["Example Trip"]
        EX["distance=8km, duration=22min, surge=1.5×\nbase=2.00\ndistance=8×1.25=10.00\ntime=22×0.20=4.40\nraw=16.40\nfinal=16.40×1.5=$24.60"]
    end

    BF & DF & TF --> RAW --> FINAL --> MIN
    MIN --> Example
    SF --> FINAL
```

## 8. Atomic Driver Assignment — Preventing Double-Booking

```mermaid
flowchart TB
    subgraph Concurrent["Two Concurrent Dispatch Jobs (Trip A and Trip B)"]
        TA["Trip A dispatch\nfinds driver d1 available"]
        TB["Trip B dispatch\nfinds driver d1 available (same driver)"]
    end

    subgraph Redis["Redis Atomic Operation"]
        NX1["Trip A: SET driver:d1:state\n'PENDING:tripA' NX EX 15"]
        NX2["Trip B: SET driver:d1:state\n'PENDING:tripB' NX EX 15"]
        RESULT["Redis processes atomically:\nNX1 arrives first → SUCCESS (d1 claimed by Trip A)\nNX2 arrives second → NIL (key exists → FAIL)"]
    end

    subgraph Outcome["Outcome"]
        OK["Trip A → driver d1 assigned\n(proceeds with offer)"]
        RETRY["Trip B → d1 already taken\n→ try next driver d2"]
    end

    TA --> NX1 --> RESULT --> OK
    TB --> NX2 --> RESULT --> RETRY

    note["SET ... NX (Not eXists) is atomic in Redis\nNo two dispatch jobs can claim the same driver\nEX 15 = auto-release if driver doesn't respond\n(prevents stuck locks)"]
    style note fill:#e8f5e9
```
